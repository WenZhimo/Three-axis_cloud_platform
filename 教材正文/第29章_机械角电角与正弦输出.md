# 第29章 机械角、电角与正弦输出

## 1. 本章目标

第28章已经把 PID 输出限制在受约束的角度补偿范围内。本章继续回答一个更靠近执行器的问题：这些角度量如何进入电机相位输出。

本章目标是：

- 理解 `机械角与电角转换` 为什么是云台姿态角和电机定子相位之间的桥梁。
- 理解机械角、电角、定子电角和 PID 补偿角之间的区别。
- 理解项目如何用 `mechanical2electricalDegrees[]` 和 `electrical2mechanicalDegrees[]` 表示换算关系。
- 理解 `正弦查表快速三角函数` 的基本思想，以及项目中两套正弦表的边界。
- 能从 `computeMotorCommands()` 追踪到 `PWM_Motor_SetAngle()` 的角度输入，但不把本章误写成完整三相 PWM 硬件映射。

## 3. 问题背景

三轴云台的姿态角来自 IMU 和姿态解算，表示的是云台结构在空间中的机械姿态。例如 Pitch 轴抬起多少弧度，Yaw 轴转过多少弧度。

电机底层需要的不是“云台机械角应该是多少”，而是“定子三相磁场应该放在哪个电角位置”。如果直接把机械角当成电机相位角，电机磁场与转子位置就会错位，输出方向、输出幅值和闭环补偿都可能异常。

因此本项目的执行链路中间需要一层转换：

```text
姿态机械角
-> PID补偿角
-> 定子电角
-> 正弦表索引
-> 三相CCR目标值
```

第27章和第28章解决的是 PID 输出如何得到、如何限制。本章解决的是 PID 输出进入电角和正弦表之前的含义转换。

## 4. 核心概念

### 4.1 机械角

机械角描述真实机构转过的角度。

项目中 `sensors.margAttitude500Hz[ROLL]`、`sensors.margAttitude500Hz[PITCH]`、`sensors.margAttitude500Hz[YAW]` 属于姿态角来源。

它们先经过符号、归一化、误差计算和 PID，再参与电机输出。

机械角的单位在项目控制链中主要是弧度。

### 4.2 电角

电角描述电机定子磁场相位。对于多极电机，转子机械转一圈，电角会转多圈。一般关系可以写成：

```text
电角 = 机械角 * 极对数
```

更完整的形式是：

```text
theta_e = p * theta_m + theta_offset
omega_e = p * omega_m
```

其中 `theta_m` 是机械角，`theta_e` 是电角，`p` 是极对数，`theta_offset`
是安装、相序或标定引入的固定电角偏移。如果电机总极数为 `N_poles`，则通常有：

```text
p = N_poles / 2
```

因此，“14 极电机”对应的是 `7` 个极对，而不是 `14` 个极对。ST 的 PMSM/BLDC
电机控制资料也把 pole pairs 作为电角换算和电机参数识别中的关键输入，而不是只看总极数。

项目变量名使用 `mechanical2electricalDegrees[]`，但当前代码中的角度计算主要以弧度参与运算。因此阅读时不要被 `Degrees` 命名误导，本章按“机械角到电角的换算系数”理解它。

本项目里存在三层容易混淆的证据：

- `config.c` 中 `rollMotorPoles/pitchMotorPoles/yawMotorPoles` 默认都是 `14.0f`。
- `computeMotorCommands.c` 中 `mechanical2electricalDegrees[]` 初值都是 `7.0f`。
- Roll 后续 PID 分支会把 Roll 换算系数改为 `14.0f`，Pitch 某分支会改为 `20.0f`。

所以教材不能简单写成“项目一直按 14 极电机的 7 极对运行”。更准确的表述是：
初始数组值 `7.0f` 与 14 极电机的 7 极对关系一致；但当前运行分支存在额外重写，
这些重写是源码事实，不等于已经由电机规格书或实测标定证明的极对数。
其设计意图、实物相位效果和是否应回连到 `config.c` 的电机极数字段，仍需要【待验证】。

### 4.3 定子电角

定子电角是最终传给 `PWM_Motor_SetAngle()` 的角度参数。它不只是传感器当前角度，也可能叠加 PID 补偿、符号方向和固定偏移。

例如 Roll 后续 PID 分支中：

```text
stator_electrical_angle = current_electrical_angle + ROLL_STATOR_SIGN * pidCmd[ROLL]
```

这里 `pidCmd[ROLL]` 是电角方向上的补偿量，不是 PWM 占空比。

还要注意，源码里的 `*_STATOR_SIGN` 不能全部按“纯符号位”理解。
`ROLL_STATOR_SIGN = -1.0f`、`YAW_STATOR_SIGN = +1.0f` 确实主要表现为方向约定；
但 `PITCH_STATOR_SIGN = 2.0f`，已经不是简单的 `+1/-1`。
因此 Pitch 分支中的 `PITCH_STATOR_SIGN * pidCmd[PITCH]`
同时包含“方向约定”和“补偿增益”两层含义。
阅读公式时，更稳妥的说法是：这些常量参与形成定子电角补偿项，而不一定只是选择正反方向。

### 4.4 软件电角与真实转子电角边界

本章要特别区分两类“电角”。

第一类是电机控制理论中的转子电角，通常来自编码器、霍尔传感器、观测器或开环估计。
它描述转子磁场相对于定子的电气相位。

第二类是当前项目中的软件计算电角。它主要由 IMU 姿态角、符号常量、
`mechanical2electricalDegrees[]` 和固定偏移组合得到。例如：

```text
current_electrical_angle = roll_angle * mechanical2electricalDegrees[ROLL]
```

仓库内没有发现编码器、霍尔状态解码、相电流观测器或反电动势观测器参与当前主控制链。
因此，教材不能把 `current_electrical_angle` 写成“传感器实测转子电角”。
更准确的说法是：它是项目根据云台机械姿态和换算系数得到的电角控制变量。

这个边界很重要。若机械结构、传动关系、零位偏移或轴映射不准确，软件电角仍然能被计算出来，
但它未必等于真实电机需要的转子电角。相关相位正确性必须通过第30章通道映射和第33章硬件点测继续验证。

### 4.5 正弦查表快速三角函数

正弦查表是把一圈正弦值预先存入数组，运行时用角度换算成索引，再从数组中取近似正弦值。这样可以减少实时计算 `sinf()` 的开销。

当前项目中有两套相关实现：

- `Drivers/SRC/Src/fastTrig.c` 定义 `sinDataI16[SINARRAYSIZE]`，由 `initSinArray()` 生成 1024 点正弦表，`fastSin()` 用索引近似正弦。
- `Drivers/CustomDrivers/Src/drv_pwmMotors.c` 内部定义 `sinTable[SIN_TABLE_SIZE]`。
  它由 `GenerateSinTable()` 生成 720 点正弦表，`PWM_Motor_SetAngle()` 当前实际使用这张表生成三相 CCR 目标值。

因此，`fastTrig.c` 是项目中的查表示例和工具模块；当前主电机输出链路的正弦表证据在 `drv_pwmMotors.c`。

### 4.6 表分辨率与量化误差

正弦表不是连续函数，而是一圈电角的离散采样。

`drv_pwmMotors.c` 当前使用 720 点表：

```text
每点电角间隔 = 2π / 720 ≈ 0.00873 rad ≈ 0.5°
```

720 还有一个对三相输出很重要的性质：它能被 3 整除。
因此 `SIN_TABLE_SIZE / 3 = 240`，正好对应 120° 电角偏移；
`2 * SIN_TABLE_SIZE / 3 = 480`，对应 240° 电角偏移。
如果未来把表长改成不能被 3 整除的数，简单的整数索引偏移就不再精确表示 120°，
三相平衡误差会变成表长设计问题，而不只是查表量化问题。

`fastTrig.c` 的 1024 点表分辨率更细：

```text
每点电角间隔 = 2π / 1024 ≈ 0.00614 rad ≈ 0.3516°
```

表越大，角度量化误差越小，但占用 RAM 越多。当前两个表都是 16 位整数数组：

```text
drv_pwmMotors.c: 720 * 2 bytes = 1440 bytes
fastTrig.c:      1024 * 2 bytes = 2048 bytes
```

这说明查表法节省实时 `sinf()` 计算，但并不是零成本。它用 RAM 和索引量化换取运行时速度。

### 4.7 中点偏置与幅值

定时器 CCR 不能直接写入负数，所以三相正弦值不会直接作为 CCR。

`PWM_Motor_SetAngle()` 使用：

```text
half_period = (ARR + 1) / 2
current_amp = half_period * power_percent / 100
CCR = half_period + sin_value * current_amp
```

其中 `sin_value` 是查表正弦值缩放回 `[-1, 1]` 后的概念值。

因此三相 CCR 围绕 `half_period` 上下摆动。`power_percent` 控制摆动幅值，
不是控制相位角。相位角来自 `angle_rad`，幅值来自 `power_percent`。

从 STM32 PWM1 模式角度看，CCR 是定时器计数周期内的比较点。
本项目 TIM2/TIM3/TIM4 使用向上计数和 `TIM_OCMODE_PWM1`。
本地 LL 头文件对 PWM1 的描述是：向上计数时，`TIMx_CNT < TIMx_CCRy` 期间通道为有效状态。
因此在当前极性配置下，CCR 越大，有效电平持续时间越长。
本章关注的是软件如何计算 CCR 目标值；示波器上看到的实际电平、驱动器输入逻辑和相电压仍需硬件验证。

### 4.8 三相平衡与公共偏置

三相正弦输出不是三条互不相关的 PWM。

理想数学形式是：

```text
a = sin(theta)
b = sin(theta + 2π/3)
c = sin(theta + 4π/3)
```

它们满足：

```text
a + b + c = 0
```

因此，若只看正弦交流分量，三相总和为零。项目写入 CCR 时又给三相都加上同一个
`half_period`，所以理论上：

```text
CCR_A + CCR_B + CCR_C ≈ 3 * half_period
```

这个公共偏置让 CCR 始终落在非负计数范围内。真正随电角变化的是三相之间的差值，
而不是三相共同加上的 `half_period`。

ST 关于 S-PWM 的资料也把三相正弦 PWM 描述为三路相差 120 度、平均占空比在
50% 附近的输出。第29章用项目源码落地这个思想，但仍不把它等同为闭环电流控制。

### 4.9 轴名与物理接口边界

第29章只讲“角度如何变成三相目标值”，不把轴枚举注释直接当成硬件真相。

`drv_pwmMotors.h` 中 `MotorAxis_t` 有注释，`drv_pwmMotors.c` 的 `switch(axis)` 中也有
MOT0/MOT1/MOT2 注释。当前应以实际写入哪些 TIMx->CCR 通道作为硬证据。

例如 `computeMotorCommands.c` 的 Roll 控制路径调用了 `PWM_Motor_SetAngle(MOTOR_PITCH, ...)`。
这说明逻辑轴、枚举名和物理接口之间存在项目内映射关系，不能只凭名字推断。
具体 CCR 通道和物理接口映射留给第30章逐项展开。

## 5. 工作原理

### 5.1 从机械误差到电角误差

以 Roll 后续 PID 分支为例，项目先得到当前机械角：

```text
roll_angle = wrapToPif(ROLL_SENSOR_SIGN * sensors.margAttitude500Hz[ROLL])
```

再计算目标与当前的机械角误差：

```text
roll_error_mech = wrapToPif(rollTargetSlew - roll_angle)
```

随后进入机械角到电角的转换：

```text
current_electrical_angle = roll_angle * mechanical2electricalDegrees[ROLL]
target_electrical_angle = current_electrical_angle + roll_error_mech * mechanical2electricalDegrees[ROLL]
```

这一步的意义是：姿态误差发生在机械结构上，但电机相位补偿必须按电角尺度计算。

如果极对数系数 `p` 取错，电角误差会随机械角线性累积。设真实极对数为 `p_true`，
代码使用 `p_code`，则同一机械角 `theta_m` 下的电角偏差为：

```text
Δtheta_e = (p_code - p_true) * theta_m
```

这意味着误差不是一个固定偏置，而是会随着机械角转动持续放大。
例如真实极对数是 7，却按 14 计算时，机械转过 1 rad，电角会多算 7 rad。
这种错误不会只让方向“偏一点”，而是会让相位补偿持续跑偏，最终表现为
输出方向、转矩方向或控制稳定性异常【待验证】。

极对数错误、固定偏移错误和符号错误要分开看。

若极对数系数错误，误差随机械角线性变化：

```text
Δtheta_e = (p_code - p_true) * theta_m
```

若固定偏移错误，误差更像一个常量：

```text
Δtheta_e = theta_offset_code - theta_offset_true
```

若符号错误，则机械角变化方向被镜像：

```text
theta_e_code = -p * theta_m + theta_offset
```

这三类错误的调试方法不同。极对数错误要看不同机械角下相位误差是否继续累积；
固定偏移错误要看是否整体偏一个近似固定角度；符号错误要看正向扰动时电角是否反向变化。
仓库源码能证明当前使用了哪些系数、偏移和符号，不能单独证明哪一类与真实硬件完全匹配。

### 5.2 PID输出不是PWM占空比

`updatePID()` 返回后，项目把结果放入 `pidCmd[]`，再经过第28章讲过的限幅和速率限制。这个 `pidCmd[]` 仍然是角度补偿量。

Roll 分支中，`pidCmd[ROLL]` 加到当前电角上：

```text
current_electrical_angle + ROLL_STATOR_SIGN * pidCmd[ROLL]
```

Pitch 分支中，`pidCmd[PITCH]` 加到 `current_elec` 和 `pitch_elec_offset` 上：

```text
current_elec + pitch_elec_offset + PITCH_STATOR_SIGN * pidCmd[PITCH]
```

这里有一个容易被命名掩盖的细节。`computeMotorCommands.c` 把
`PITCH_STATOR_SIGN` 定义为 `2.0f`，所以 Pitch 的 PID 补偿不是一比一叠加到定子电角。
如果 `pidCmd[PITCH] = 0.3 rad`，实际进入叠加项的是：

```text
PITCH_STATOR_SIGN * pidCmd[PITCH] = 2.0 * 0.3 = 0.6 rad
```

因此第28章中 `PITCH_CMD_LIMIT_RAD` 和速率限制约束的是原始 `pidCmd[PITCH]`；
到了本章的定子电角合成阶段，还要再乘以 `PITCH_STATOR_SIGN`。
这会改变 Pitch 轴电角补偿的有效幅度和每次更新的有效变化量。
源码能证明这个乘法关系存在，但不能仅凭命名判断它是临时调参、方向修正还是有意的轴向增益设计，
其真实硬件效果仍需要结合响应测试验证。

Yaw 分支中，`pidCmd[YAW]` 与当前电角组合：

```text
-current_elec + YAW_STATOR_SIGN * pidCmd[YAW]
```

所以 PID 输出不是直接写入 TIM 的 CCR，也不是直接等于占空比。它先决定定子电角，再由 `PWM_Motor_SetAngle()` 转成正弦三相输出。

### 5.3 电角归一化

电角可以不断增加或出现负值，但正弦表只需要一圈内的位置。

`computeMotorCommands.c` 会在部分分支用 `wrapToPif()` 把角度限制在 `[-PI, PI]`。`PWM_Motor_SetAngle()` 内部又使用：

```text
normalized_angle = fmodf(angle_rad, 2π)
如果 normalized_angle < 0，则加 2π
```

这说明上层可以传入正负电角，底层会把它归一化到 `[0, 2π)` 后再查表。

### 5.4 从电角到正弦表索引

`PWM_Motor_SetAngle()` 中的索引计算为：

```text
index = normalized_angle * SIN_TABLE_SIZE / (2π)
```

`SIN_TABLE_SIZE` 在 `drv_pwmMotors.h` 中为 720。也就是说，一圈电角被分成 720 个采样点。角度越大，索引越靠后；角度绕回一圈后，索引重新回到 0 附近。

这里要优先相信宏定义和表达式，而不是只看源码注释。
`drv_pwmMotors.c` 在索引计算前的注释写着“0 ~ 359”，但当前 `SIN_TABLE_SIZE=720`，
`sinTable[]` 的有效索引范围实际是：

```text
0 <= index <= 719
```

几个边界点可以帮助理解这个范围：

```text
normalized_angle = 0          -> index = 0
normalized_angle = PI         -> index = 360
normalized_angle 接近 2PI     -> index 接近 719
normalized_angle = 2PI        -> 经过 fmodf 后回到 0
```

代码后面还有一层保护：

```text
if (index >= SIN_TABLE_SIZE) index = 0
```

这主要防止浮点舍入或边界输入导致越界。它不是把索引限制到 359，
也不是说明正弦表只有 360 点。阅读这类底层代码时，应把“数组长度、计算公式和越界保护”三者合起来判断真实范围。

### 5.5 三相相差120度

底层正弦输出需要 A/B/C 三相相差 120 度。`PWM_Motor_SetAngle()` 当前用表索引偏移实现：

```text
sin_a = sinTable[index]
sin_b = sinTable[(index + SIN_TABLE_SIZE / 3) % SIN_TABLE_SIZE]
sin_c = sinTable[(index + 2 * SIN_TABLE_SIZE / 3) % SIN_TABLE_SIZE]
```

本章只讲到“三相目标值来自同一电角的三个相位偏移”。至于这些值具体写入 TIM2/TIM3/TIM4 哪些 CCR 通道，属于第30章。

### 5.6 三相和与线电压视角

把第5.5节的三相写成连续形式：

```text
sA = sin(theta)
sB = sin(theta + 2π/3)
sC = sin(theta + 4π/3)
```

根据三角恒等式：

```text
sin(theta) + sin(theta + 2π/3) + sin(theta + 4π/3) = 0
```

项目的 CCR 公式可以写成：

```text
CCR_A = half_period + amp * sA
CCR_B = half_period + amp * sB
CCR_C = half_period + amp * sC
```

所以三相 CCR 的理论和为：

```text
CCR_A + CCR_B + CCR_C = 3 * half_period
```

再看两相之间的差值：

```text
CCR_A - CCR_B = amp * (sA - sB)
```

公共的 `half_period` 被抵消了。这说明中点偏置主要是为了适配定时器 CCR 的非负范围，
并不代表三相交流分量都增加了同一个有效力矩分量。

但这里仍然只能得到软件目标值。实际线电压、电流和力矩还取决于驱动器、电源、
绕组连接、死区、导通损耗和负载，仓库内没有这些实测证据。

### 5.7 从正弦值到CCR范围

`drv_pwmMotors.c` 中 `SIN_SCALE=32767`，表值近似表示：

```text
sin_q15 ≈ sin(angle) * 32767
```

写入 CCR 前，代码计算：

```text
ccr = half_period + (sin_q15 * current_amp) / SIN_SCALE
```

所以如果 `sin_q15 = +32767`，CCR 约等于：

```text
half_period + current_amp
```

如果 `sin_q15 = -32767`，CCR 约等于：

```text
half_period - current_amp
```

当前 TIM2/TIM3/TIM4 的 `Period=3599`，所以 `period=3600`，`half_period=1800`。
这里要把三个量分开：

```text
ARR寄存器语义上的周期顶点 = 3599
源码局部变量 period = ARR + 1 = 3600
占空比换算分母 = ARR + 1 = 3600
```

因此用 `ccr / (ARR + 1)` 估算占空比是合理的；但判断 CCR 是否超过比较寄存器的常见有效计数范围时，
要记得向上计数周期中的计数值通常是 `0..ARR`。当前源码正常功率路径下不会接近这个上限，
所以这个差异多数时候不可见；一旦进入溢出保护，就需要单独记录。

以 `power_percent=60` 为例：

```text
current_amp = 1800 * 0.60 = 1080
CCR范围约为 [720, 2880]
```

以 `power_percent=40` 为例：

```text
current_amp = 1800 * 0.40 = 720
CCR范围约为 [1080, 2520]
```

以 `power_percent=30` 为例：

```text
current_amp = 1800 * 0.30 = 540
CCR范围约为 [1260, 2340]
```

这些只是目标 CCR 数值范围，不等于真实相电流或真实力矩。真实电流还取决于驱动器、
电机绕组、电源、电感和 PWM 频率等硬件因素，仓库内没有直接测量证据。

还要注意，`PWM_Motor_SetAngle()` 用强制类型转换计算索引：

```text
index = (int)(normalized_angle * SIN_TABLE_SIZE / (2π))
```

这不是四舍五入，而是向下截断到当前表点。因此 720 点表的索引量化误差范围约为：

```text
0 <= angle_error < 2π / 720 ≈ 0.00873rad ≈ 0.5°
```

如果使用最近点四舍五入，最坏相位误差可约束到半个表步长；但当前源码没有这样做。
这是一种用更少计算换取速度的工程近似。

`sinTable[]` 的元素由：

```text
(int16_t)(sinf(rad) * SIN_SCALE)
```

得到，也存在幅值量化。单个表值的归一化量化单位约为：

```text
1 / 32767 ≈ 0.0000305
```

这个误差通常远小于索引相位误差，但它仍然说明查表不是连续正弦函数。

### 5.8 power_percent边界

`PWM_Motor_SetAngle()` 中有一段功率限制：

```text
if power_percent > 60.0:
    power_percent = 40.0
if power_percent < 0.0:
    power_percent = 0.0
```

注意这不是普通的“超过 60 就夹到 60”。源码实际行为是：大于 60 时回退到 40。

当前 `computeMotorCommands.c` 中常见调用包括：

```text
Roll相关路径: 40 或 60
Pitch路径:    60
Yaw路径:      30
```

因此本章可以记录当前功率参数来源，但不能把 `power_percent` 写成严格线性的开环力矩百分比。

这个边界还带来一个非单调跳变：

```text
power_percent = 60.0  -> 保持 60.0
power_percent = 60.1  -> 回退到 40.0
```

所以调试时如果从 60 附近继续增大参数，输出幅值反而会突然下降。
这可能是人为安全回退，也可能是临时调试策略；仓库源码只能证明当前行为，
不能证明它是最终硬件保护设计【待验证】。
它只是正弦 CCR 幅值缩放参数。

### 5.9 PWM载波与控制更新频率边界

第26章的实时控制循环大约以 500Hz 计算新的姿态、PID 和电角目标。
第12章和第30章已经说明，当前 TIM2/TIM3/TIM4 的 PWM 载波按工程配置约为 20kHz。

这两个频率不是一回事：

```text
控制更新: computeMotorCommands() 约每 2ms 产生一次新电角和CCR目标
PWM载波:  TIM2/TIM3/TIM4 硬件约每 50us 完成一个PWM周期
```

因此一次控制输出大约会持续多个 PWM 周期。若按 500Hz 和 20kHz 估算：

```text
20kHz / 500Hz = 40
```

也就是一次 `PWM_Motor_SetAngle()` 写入后的 CCR 目标，理论上会被硬件 PWM 重复作用约 40 个载波周期，直到下一次控制帧更新。
这个结论有助于理解“电角更新频率”和“PWM 开关频率”的差别，但真实引脚频率、CCR 生效时刻和跨定时器同步仍属于第30章与仓库外示波器验证范围。

## 6. STM32实现机制

本章涉及 STM32 的方式有三层。

第一，`Core/Src/main.c` 先调用 `MX_TIM3_Init()`、`MX_TIM2_Init()`、`MX_TIM4_Init()`，再调用 `PWM_Motor_Init()`。这说明 PWM 外设已经初始化后，电机驱动模块才生成正弦表并启动 PWM 通道。

第二，`PWM_Motor_Init()` 调用 `GenerateSinTable()`，并启动 TIM2、TIM3、TIM4 的 PWM 通道。它还把 TIM2、TIM3、TIM4 的计数器清零，用于减小跨定时器相位错位。完整相位同步属于第30章，本章只保留这个入口关系。

这里有两个初始化边界。`PWM_Motor_Init()` 如果发现 `isInitialized` 已经为真，会直接返回
`HAL_OK`，不会重新生成表或重新启动 PWM 通道。函数内部调用多次 `HAL_TIM_PWM_Start()`，
但当前代码没有逐个检查这些调用的返回值，最后仍把 `isInitialized` 置为 1 并返回
`HAL_OK`。因此 `isInitialized == 1` 能证明软件走过初始化路径，
不能单独证明所有 PWM 通道都已经在硬件上成功输出【待验证】。

相反，`PWM_Motor_SetAngle()` 如果发现 `isInitialized == 0`，会直接 `return`，
没有错误码、日志或 LED 提示。若调试时 `pidCmd[]` 和 `stator_electrical_angle`
都在变化，但 CCR 没变化，应把初始化门控作为第一类排查项。

第三，`PWM_Motor_SetAngle()` 根据电角和功率百分比计算 CCR 目标值。

`Core/Src/tim.c` 中 TIM2、TIM3、TIM4 当前 `Period` 都是 3599。

`PWM_Motor_SetAngle()` 使用 `htim3.Init.Period + 1` 作为周期基准，并假设三个定时器 ARR 一致。

这个假设当前有 `tim.c` 配置证据。

第四，正弦表在初始化阶段生成。`GenerateSinTable()` 使用 `sinf()` 生成 720 个表值，
这发生在 `PWM_Motor_Init()` 中，不在每个 500Hz 控制帧中重复执行。

第五，`PWM_Motor_SetAngle()` 运行时仍不是纯整数代码。它包含 `fmodf()`、浮点乘法、
表索引计算、整数乘除和 `__disable_irq()` 包围的 CCR 写入。
查表减少了实时 `sinf()` 成本，但没有消除全部浮点和临界区开销。
如果三轴每帧都调用输出函数，这部分仍应纳入第26章的 `executionTime500Hz` 预算观察。

从工程选型看，当前项目采用的是查表正弦电压输出，而不是 SVPWM 或完整 FOC。
源码中没有看到 Clarke/Park 变换、相电流 ADC 采样、电流 PI 环、SVPWM 扇区判断或母线电压补偿。
因此本章只能把当前方案描述为“按电角生成三相正弦 CCR 目标”，不能写成电流闭环矢量控制。

这种方案的优点是结构直接、调试变量少、适合先验证轴映射和相序。代价是没有电流反馈，
不能直接限制相电流，也不能用软件目标值证明力矩线性或效率最优。相关结论保持【待验证】。

第六，`PWM_Motor_SetAngle()` 使用直接寄存器写 `TIMx->CCRn`，不是 HAL 宏或 LL 函数。
这减少了函数调用开销，但也要求教材按当前源码逐项说明通道映射，不能仅根据 HAL 初始化推断。

## 8. 代码分析

### 8.1 换算系数数组

`computeMotorCommands.c` 定义：

```text
mechanical2electricalDegrees[3] = {7.0f, 7.0f, 7.0f}
electrical2mechanicalDegrees[3] = {1/7, 1/7, 1/7}
```

这两个数组表示机械角和电角之间的换算关系。`computeMotorCommands.h` 用 `extern` 暴露它们，说明它们是可被其他模块观察或使用的控制状态。

需要注意的是，Roll 后续 PID 分支会把 `mechanical2electricalDegrees[ROLL]` 改为 `14.0f`。

Pitch 在 `return_state == false` 分支会把 `mechanical2electricalDegrees[PITCH]` 改为 `20.0f`。Yaw 当前使用数组已有值，未看到同类分支内重写证据。

因此三轴当前换算系数不是完全统一的配置读取结果，而是包含初值和分支内写入。

这个边界对读者很重要：

- `config.c` 的 `rollMotorPoles/pitchMotorPoles/yawMotorPoles` 是配置字段。
- `mechanical2electricalDegrees[]` 的 `7.0f` 初值是运行前默认值。
- 分支内写成 `14.0f`、`20.0f` 是运行时覆盖。

所以本章只能说“源码中同时存在 7、14、20 三类电角系数证据”，不能直接把
`config.c` 的 14 极配置当成最终运行换算，也不能把初值 7 直接当成全局真值。
哪一个值对应当前真实电机极对数，需要结合电机铭牌、相序测试、外部标定或仓库外实测继续【待验证】。

### 8.2 Roll输出路径

Roll 有两个路径。

回中路径中，代码把 `rollTargetSlew` 乘以 `mechanical2electricalDegrees[ROLL]`，叠加 `ROLL_STATOR_SIGN` 和 `ROLL_ELEC_OFFSET` 后，传给 `PWM_Motor_SetAngle()`。

后续 PID 路径中，代码先计算 `current_electrical_angle` 和 `target_electrical_angle`，PID 输出限幅后，再生成：

```text
stator_electrical_angle = current_electrical_angle + ROLL_STATOR_SIGN * pidCmd[ROLL]
```

这条路径清楚体现了“机械角误差 -> 电角目标 -> PID电角补偿 -> 定子电角”。

### 8.3 Pitch输出路径

Pitch 先对传感器角乘 `PITCH_SENSOR_SIGN`，再用 `pointingCmd[PITCH]` 和 `pitch_angle` 执行 PID。随后代码根据 `return_state` 选择电角计算方向：

```text
return_state == true:  current_elec = pitch_angle * mechanical2electricalDegrees[PITCH]
return_state == false: current_elec = -pitch_angle * mechanical2electricalDegrees[PITCH]
```

最后叠加：

```text
pitch_elec_offset + PITCH_STATOR_SIGN * pidCmd[PITCH]
```

其中 `pitch_elec_offset = -1.25f` 是固定偏移。该偏移有源码证据，但缺少更高层说明来源，本章只按项目常量记录，不推测机械安装原因。
`PITCH_STATOR_SIGN = 2.0f` 也应按源码事实记录为“补偿系数”，而不是只写成符号方向。
调试 Pitch 轴时，建议同时记录三列：`pidCmd[PITCH]`、`PITCH_STATOR_SIGN * pidCmd[PITCH]`
和 `stator_electrical_angle`。如果只看 `pidCmd[PITCH]`，会低估最终叠加到定子电角上的补偿量。

### 8.4 Yaw输出路径

Yaw 先用 `wrapToPif()` 得到机械误差，再计算 `safe_target` 进入 PID。输出阶段：

```text
current_elec = yaw_angle * mechanical2electricalDegrees[YAW]
stator_electrical_angle = -current_elec + YAW_STATOR_SIGN * pidCmd[YAW]
```

Yaw 的当前电角前面带负号，说明该轴输出方向和传感器/电机安装方向之间存在符号约定。源码有这个符号证据，但本章不进一步猜测硬件安装原因。

### 8.5 fastTrig.c的查表函数

`fastTrig.c` 的流程是：

```text
initSinArray()
-> 生成 1024 点 sinDataI16[]
-> fastSin(x) 把 x 映射为索引
-> 返回缩放后的浮点正弦近似值
```

`fastSin()` 对负角使用 `-x` 查表再取负值。它说明了查表法的基本思想：把实时三角函数计算换成数组访问和缩放。

当前没有发现 `initSinArray()` 或 `fastSin()` 在主控制输出路径中被调用。因此调试电机输出时，不应只看 `fastTrig.c`，还必须看 `drv_pwmMotors.c`。

### 8.6 drv_pwmMotors.c的输出查表

`drv_pwmMotors.c` 的 `GenerateSinTable()` 生成 720 点 `sinTable[]`。`PWM_Motor_SetAngle()` 收到电角后：

1. 把电角归一化到 `[0, 2π)`。
2. 把电角映射为 `sinTable` 索引。
3. 取 A/B/C 三相相差 120 度的正弦值。
4. 按 `power_percent` 缩放幅值。
5. 加上 `half_period` 得到 CCR 目标值。

这就是本章“正弦输出”的核心。CCR 写入哪些定时器通道，留到第30章展开。

调试索引时，不要只看注释里的“0 ~ 359”。当前真实索引空间由
`SIN_TABLE_SIZE=720` 决定。若断点观察到 `index=360`，这不是异常，
而是 `normalized_angle≈π` 时的一圈中点；若观察到 `index=719`，表示电角已经接近一圈末端。
只有 `index < 0` 或 `index >= 720` 才越出当前表范围，源码中的保护会把后者回退到 0。
因此调试记录表应包含 `SIN_TABLE_SIZE`、`normalized_angle`、计算前浮点索引值、
强制转换后的 `index` 和保护后的最终索引。

调试时还可以增加一个平衡性检查：

```text
sin_a + sin_b + sin_c ≈ 0
ccr_val[0] + ccr_val[1] + ccr_val[2] ≈ 3 * half_period
```

由于查表、整数乘除和强制类型转换存在量化误差，上式不一定精确为 0 或精确等于
`3 * half_period`，但应该接近。如果偏差很大，应优先检查索引偏移、表长度是否能被 3 整除、
`SIN_SCALE` 是否一致，以及是否发生了 CCR 溢出保护。

`PWM_Motor_SetAngle()` 的溢出保护写法是：

```text
if ccr_val[i] > period:
    ccr_val[i] = period
if ccr_val[i] < 0:
    ccr_val[i] = 0
```

在当前配置里，`period = htim3.Init.Period + 1 = 3600`，而 `tim.c` 中 TIM2/TIM3/TIM4 的
`Period` 字段是 3599。也就是说，保护上限使用的是 `ARR + 1`，不是 `ARR` 本身。
在当前 `power_percent <= 60` 的正常计算范围内，`ccr_val[]` 大约落在 `[720, 2880]`，
这个保护通常不会触发；如果调试时真的观察到 `ccr_val[i]` 被改成 3600，
它更像是异常输入、表值/幅值异常或周期配置不一致的报警线索，而不是常规幅值控制路径。
该写法写入比较寄存器后的硬件占空比语义，需要结合第12章的 PWM 模式和示波器继续验证。

当前 `SIN_TABLE_SIZE=720` 正好满足三相偏移要求：

```text
SIN_TABLE_SIZE / 3 = 240
2 * SIN_TABLE_SIZE / 3 = 480
```

这使得 B 相和 C 相的索引偏移正好对应 120° 和 240°。
如果未来为了节省 RAM 或提高分辨率修改表长，必须重新确认表长是否能被 3 整除，
否则整数除法会让三相偏移出现结构性误差。

### 8.7 power_percent与CCR幅值

`PWM_Motor_SetAngle()` 的第三个参数不是 PID 输出，而是正弦幅值百分比。

源码中先处理边界：

```text
if power_percent > 60.0f:
    power_percent = 40.0f
if power_percent < 0.0f:
    power_percent = 0.0f
```

然后计算：

```text
current_amp = half_period * power_percent / 100
```

这说明 `power_percent` 控制的是三相 CCR 围绕 `half_period` 的摆动幅值。
由于源码对大于 60 的值回退到 40，调试时不能只按“百分比越大输出越大”线性理解。

尤其要注意 60 附近的跳变：

```text
60.0  -> current_amp = half_period * 0.60
60.1  -> power_percent 被改为 40.0
```

所以这不是饱和限幅，而是阈值回退。若调试者把功率从 60 慢慢调到 61，
软件目标幅值会从 60% 突然降到 40%。这类非单调行为必须记录在调参日志里，
否则容易误判为电机、驱动器或供电异常。

源码随后还有 CCR 溢出保护：

```text
if ccr_val[i] > period: ccr_val[i] = period
if ccr_val[i] < 0:      ccr_val[i] = 0
```

在当前 `power_percent <= 60` 的正常路径下，理论 CCR 范围不会触碰这个保护。
例如 `half_period=1800`、`current_amp=1080` 时，范围约为 `[720, 2880]`。
因此如果调试中观察到溢出保护触发，应优先检查 `Period`、`power_percent`、
`SIN_SCALE` 或表值是否异常，而不是把它当成常规限幅路径。

还要注意，当前代码把上限夹到 `period`，而 `ARR = period - 1`。
在当前正常幅值下不会触碰这个边界；若未来改大功率或改小周期导致触碰，
`CCR=ARR+1` 在 PWM1 模式下的实际输出语义需要结合 STM32 定时器行为和示波器验证【待验证】。

### 8.8 初始化门控与静默返回边界

`drv_pwmMotors.c` 使用静态变量 `isInitialized` 作为电机输出模块的软件门控。

初始化函数：

```text
if isInitialized: return HAL_OK
GenerateSinTable()
HAL_TIM_PWM_Start(...)
isInitialized = 1
return HAL_OK
```

输出函数：

```text
if !isInitialized: return
```

这带来三个调试边界：

- 重复调用 `PWM_Motor_Init()` 不会重新启动通道，也不会重新生成表。
- `PWM_Motor_SetAngle()` 在未初始化时静默返回，不会告诉上层调用失败。
- `HAL_TIM_PWM_Start()` 的返回值当前没有逐项检查，`HAL_OK` 不能单独证明每个通道都启动成功。

当前 `Core/Src/main.c` 的顺序是先初始化 TIM2/TIM3/TIM4，再调用 `PWM_Motor_Init()`，
然后进入 500Hz 控制循环。这个顺序有源码证据。
但调试硬件无输出时，仍应同时查看 `isInitialized`、PWM 通道启动返回值、
定时器寄存器和实际引脚波形，缺少外部测量时保持【待验证】。

### 8.9 枚举名、逻辑轴和物理接口

本章已经追踪到 `PWM_Motor_SetAngle(axis, angle_rad, power_percent)`，但不要把 `axis`
简单理解成逻辑姿态轴。

当前 `computeMotorCommands.c` 中：

```text
Roll控制路径调用 MOTOR_PITCH
Pitch控制路径调用 MOTOR_ROLL
Yaw控制路径调用 MOTOR_YAW
```

而 `drv_pwmMotors.c` 中实际根据 `axis` 写入不同 TIMx->CCRn。

这里还有一层注释漂移边界。`drv_pwmMotors.h` 中的枚举注释写成：

```text
MOTOR_ROLL  -> 对应 Mot0
MOTOR_PITCH -> 对应 Mot1
MOTOR_YAW   -> 对应 Mot2
```

但 `drv_pwmMotors.c::PWM_Motor_SetAngle()` 的 `switch(axis)` 注释和寄存器写入显示：

```text
case MOTOR_PITCH -> P轴 / 物理接口 MOT0 -> TIM3 CCR2/CCR3/CCR4
case MOTOR_ROLL  -> R轴 / 物理接口 MOT1 -> TIM3 CCR1 + TIM2 CCR4/CCR3
case MOTOR_YAW   -> Y轴 / 物理接口 MOT2 -> TIM4 CCR4 + TIM2 CCR2 + TIM4 CCR3
```

也就是说，枚举名、头文件注释、实现文件注释和实际 CCR 写入并不完全提供同一种映射叙事。
教材应把“写入哪个 TIMx->CCRn”作为硬证据，把注释只作为线索。
若要判断真实电机接口，应继续结合第30章的通道表和第33章的硬件点测，而不能仅凭
`MOTOR_ROLL` 或“对应 Mot0”这样的名字下结论。

因此应把三层名称分开：

```text
逻辑控制轴: ROLL/PITCH/YAW 姿态控制
电机枚举:   MOTOR_ROLL/MOTOR_PITCH/MOTOR_YAW
物理输出:   TIM2/TIM3/TIM4 的具体 CCR 通道
```

第29章只说明角度参数如何到达 `PWM_Motor_SetAngle()`。
第30章再审查电机枚举和物理 CCR 通道之间的映射。

### 8.10 PWM周期基准与CCR语义边界

`PWM_Motor_SetAngle()` 使用：

```text
period = htim3.Init.Period + 1
half_period = period / 2
```

当前 `Core/Src/tim.c` 中 TIM2、TIM3、TIM4 的 `Period` 都是 3599，
所以用 `htim3` 的周期作为统一基准在当前工程配置下是成立的。

但这是一条隐含约束。如果后续只修改 TIM2 或 TIM4 的 `Period`，
而 `PWM_Motor_SetAngle()` 仍按 TIM3 计算 `ccr_val[]`，跨定时器电机的三相幅值基准就可能不一致。
因此调试本章输出数值时，至少要同时核对：

```text
htim2.Init.Period
htim3.Init.Period
htim4.Init.Period
htim2.Init.Prescaler
htim3.Init.Prescaler
htim4.Init.Prescaler
```

CCR 的含义也要接回 PWM 模式。TIM2/TIM3/TIM4 当前配置为 `TIM_OCMODE_PWM1`。
在向上计数、PWM1 和当前极性下，可以简化理解为：

```text
duty ≈ CCR / (ARR + 1)
```

这解释了为什么 `half_period=1800` 对应约 50% 占空比。
但这一节仍然只说明比较值语义；最终引脚、电机驱动器和相电压表现需要第30章和第33章验证。

### 本节证据边界

本节只根据当前仓库说明文件、函数、宏、变量和调用关系。运行时频率、外部硬件表现、主机侧现象、传感器方向、电机响应或真实控制效果仍需调试记录、日志或仓库外实测证据；缺少证据时保持【待验证】。

### 8.11 构建产物证据边界

本章前面的分析主要来自源码。为了避免把“源码里写了”误读成“最终固件一定执行了”，还需要把 Debug 构建产物作为第二层证据。

当前 `Debug/Three-axis_cloud_platformV2.map` 可以证明以下符号已经进入最终镜像：

```text
.text.GenerateSinTable        0x08002608
.text.PWM_Motor_Init          0x080026ac
.text.PWM_Motor_SetAngle      0x08002748
.text.wrapToPif               0x080042f4
.text.computeMotorCommands    0x080043b4
.data.mechanical2electricalDegrees  0x20000020
.bss.sinTable                 0x20000744
.bss.isInitialized            0x20000ce4
```

这说明当前构建确实保留了角度归一化、控制输出、正弦表、PWM 初始化和 `mechanical2electricalDegrees[]`。
同时也要注意：`electrical2mechanicalDegrees[]` 在当前 `.map` 中只出现在地址为 `0x00000000` 的输入段记录，
没有看到与 `mechanical2electricalDegrees[]` 类似的最终 RAM 分配地址。
因此教材只能说源码定义了它，不能把当前 `.map` 写成“它已经作为运行时对象进入最终镜像”的证据。

当前 `Debug/Three-axis_cloud_platformV2.list` 可以继续证明更细的调用和指令边界：

- `PWM_Motor_Init()` 调用 `GenerateSinTable()`，并启动 TIM3、TIM2、TIM4 的 PWM 通道。
- `GenerateSinTable()` 的循环上界反汇编为 `cmp.w r3, #720`，并把 `sinf(rad) * SIN_SCALE` 写入 `sinTable[]`。
- `PWM_Motor_SetAngle()` 调用 `fmodf()` 做电角归一化，再把归一化角映射到 `SIN_TABLE_SIZE=720` 的索引。
- `PWM_Motor_SetAngle()` 用 `index + 240` 与 `index + 480` 生成三相 120° 偏移。
- Roll、Pitch、Yaw 分支在 `computeMotorCommands()` 中分别调用 `PWM_Motor_SetAngle()`。
- `PWM_Motor_SetAngle()` 最后直接写入 `TIM3->CCR2/3/4`、`TIM3->CCR1 + TIM2->CCR4/3`、`TIM4->CCR4 + TIM2->CCR2 + TIM4->CCR3`。

这些证据能证明“角度合成、查表、CCR 目标值计算和寄存器写入路径进入了当前 Debug 固件”。
它们不能证明真实转子电角已经测得，不能证明电机相序正确，不能证明电流、力矩、发热或示波器波形符合预期，
也不能证明 `power_percent` 与实际输出力矩成线性关系。上述结论仍需要第30章、第33章和仓库外实测记录继续验证；缺少证据时保持【待验证】。

## 9. 调试方法

第一步，观察机械角到电角换算。

- Roll 观察 `rollDiag.currentCtrlMechRad`、`rollDiag.currentElecRad`、`rollDiag.targetElecRad`。
  同时记录 `mechanical2electricalDegrees[ROLL]`。
- Pitch 观察 `pitch_angle`、`current_elec` 和 `mechanical2electricalDegrees[PITCH]`。
- Yaw 观察 `yaw_angle`、`current_elec` 和 `mechanical2electricalDegrees[YAW]`。
- 同时记录 `rollMotorPoles/pitchMotorPoles/yawMotorPoles`、`POLE_PAIRS` 和当前数组值，
  区分“配置字段”“常量”“运行时换算系数”。
- 如果要证明某个系数是真实极对数，需要电机规格、手动相位步进、相序测试或外部测量证据；
  仓库源码本身只能证明软件采用了哪个数值。

第二步，确认 PID 输出仍是角度补偿。

- 观察 `pidCmd[]` 和 `stator_electrical_angle` 的关系。
- 如果 `pidCmd[]` 变化但 CCR 没有按预期变化，应继续进入 `PWM_Motor_SetAngle()`，而不是把 `pidCmd[]` 当成占空比。

第三步，观察正弦表初始化。

- 在 `PWM_Motor_Init()` 后确认 `isInitialized` 为 1。
- 确认 `GenerateSinTable()` 已在初始化时执行。
- 记录每个 `HAL_TIM_PWM_Start()` 的返回值；当前源码没有逐项判断这些返回值。
- 若 `PWM_Motor_SetAngle()` 没有写 CCR，先确认是否因为 `isInitialized == 0` 静默返回。
- `fastTrig.c` 的 `sinDataI16[]` 只有在 `initSinArray()` 被调用后才有有效表值；当前主输出链路未见调用证据。

第四步，观察电角归一化和索引。

- 在 `PWM_Motor_SetAngle()` 中观察 `angle_rad`、`normalized_angle` 和 `index`。
- 当输入角为负时，确认 `normalized_angle` 被转换到 `[0, 2π)`。
- 当前 `SIN_TABLE_SIZE=720`，所以有效 `index` 范围是 0 到 719。源码注释中的“0 ~ 359”不能作为当前表长依据。
- 建议额外记录 `normalized_angle * SIN_TABLE_SIZE / (2π)` 的浮点中间值，区分“浮点位置”“截断后的整数索引”和“越界保护后的最终索引”。

第五步，观察正弦三相目标值。

- 观察 `sin_a`、`sin_b`、`sin_c` 是否来自相差 `SIN_TABLE_SIZE / 3` 的索引。
- 确认 `SIN_TABLE_SIZE` 是否能被 3 整除；当前 720 点表满足该条件。
- 观察 `ccr_val[]` 是否围绕 `half_period` 上下变化。
- 计算 `sin_a + sin_b + sin_c`，确认三相正弦分量是否近似平衡。
- 计算 `ccr_val[0] + ccr_val[1] + ccr_val[2]`，确认它是否接近 `3 * half_period`。
- 用 `ccr_val[i] / (ARR + 1)` 估算目标占空比，确认 CCR 没有越界到周期之外。
- 区分 `ARR=3599` 和局部变量 `period=3600`。若看到保护后 `ccr_val[i]=3600`，不要直接当成正常 100% 占空比结论，应继续核对定时器比较语义和实际波形。

第六步，观察幅值缩放。

- 观察 `power_percent`、`current_amp`、`half_period` 和 `ccr_val[]`。
- 分别测试 30、40、59.9、60、60.1 和大于 60 的输入值。
- 确认大于 60 时源码会把 `power_percent` 改为 40，而不是 60。
- 若 `ccr_val[]` 触发溢出保护，记录触发前的 `period`、`SIN_SCALE`、表值和幅值。

第七步，观察轴映射边界。

- 在 `computeMotorCommands.c` 中记录逻辑轴调用了哪个 `MOTOR_*` 枚举。
- 对照 `drv_pwmMotors.h` 的枚举注释和 `drv_pwmMotors.c` 的 `switch(axis)` 注释，记录是否存在不一致。
- 在 `drv_pwmMotors.c` 中记录该枚举进入哪个 `case`。
- 最终以 `TIMx->CCRn` 写入位置作为映射硬证据，注释只作为辅助线索。
- 具体 TIMx->CCRn 映射留到第30章，但第29章要先确认“逻辑轴名不等于物理接口名”。

第八步，观察软件电角边界。

- 记录 `current_electrical_angle`、`target_electrical_angle`、`stator_electrical_angle` 的来源。
- 标注它们是否来自 IMU 姿态角、PID 补偿、固定偏移、符号常量或真实转子传感器。
- 当前主链路未见编码器、霍尔解码或观测器证据，因此不要把软件电角写成实测转子电角。

第九步，观察控制更新与 PWM 载波的区别。

- 记录 `computeMotorCommands()` 的调用周期和 TIM2/TIM3/TIM4 的理论 PWM 频率。
- 按 `20kHz / 500Hz` 估算一次控制更新跨越多少个 PWM 周期。
- 若使用示波器验证，应区分 CCR 更新频率、PWM 载波频率和电机机械响应频率。

这些调试方法只定位到“电角和正弦输出目标值”。第30章再继续观察 TIM2/TIM3/TIM4 的具体 CCR 通道映射。

调试记录建议：

- 记录机械角来源、极对数配置、电角计算表达式、PID 补偿项和定子电角结果。
- 对正弦输出，应记录表初始化、角度归一化、索引、三相相位偏移和 `ccr_val[]`。
- 对三相平衡，应记录三相正弦和、三相 CCR 和，以及线间差值是否抵消公共偏置。
- 对幅值缩放，应记录 `power_percent`、`current_amp`、`half_period`、理论 CCR 范围和实际 `ccr_val[]`。
- 对轴映射，应记录逻辑轴、`MOTOR_*` 枚举和后续第30章要验证的物理通道。
- 对构建产物，应记录 `.map` 中函数、全局对象地址，以及 `.list` 中 `PWM_Motor_Init()`、`GenerateSinTable()`、`PWM_Motor_SetAngle()` 和三轴调用点的反汇编证据。
- `pidCmd[]` 仍是角度补偿记录项，不能写成占空比或电机实际转矩。
- 电角方向、相序正确性和实际转动效果需要第30章及仓库外实测验证，缺失时标记为【待验证】。

## 10. 常见问题

问题一：`mechanical2electricalDegrees[]` 名字里有 Degrees，是否表示角度单位是度。

不应这样理解。当前项目控制链中多数角度以弧度计算，`mechanical2electricalDegrees[]` 在代码中实际作为机械角到电角的倍数使用。变量名保留源码原名，教材解释按运行行为。

问题二：PID 输出是否直接控制 PWM 占空比。

不是。PID 输出进入 `pidCmd[]`，再参与 `stator_electrical_angle` 计算。占空比目标值由 `PWM_Motor_SetAngle()` 根据电角、正弦表和功率百分比计算。

问题三：`config.c` 中的 `rollMotorPoles`、`pitchMotorPoles`、`yawMotorPoles` 是否驱动当前换算。

当前没有发现 `computeMotorCommands.c` 读取这些字段参与换算的证据。它们是配置结构中的字段和默认值，但当前换算来源主要是 `mechanical2electricalDegrees[]` 的初值和分支内赋值。

相关边界：14 极电机不等于 14 极对。若字段含义是总极数，则极对数应为 7。
当前数组初值 `7.0f` 与这个关系一致，但 Roll 分支写 `14.0f`、Pitch 分支写 `20.0f`
属于运行时源码行为，不能直接当成电机物理极对数结论。是否为临时调参、安装补偿、
错误配置或特殊传动等原因，需要仓库外证据继续【待验证】。

问题四：`fastSin()` 是否就是当前电机输出的正弦函数。

当前证据不是。`fastSin()` 存在于 `fastTrig.c`，但主电机输出路径中的正弦表是 `drv_pwmMotors.c` 的内部 `sinTable[]`。

问题五：为什么电角需要归一化。

正弦表只表示一圈。电角可能为负，也可能超过一圈。归一化后才能稳定映射到 `[0, SIN_TABLE_SIZE)` 的有效索引范围。

问题五补充：为什么源码注释写“0 ~ 359”，教材却说当前有效索引是 0 到 719。

因为当前 `drv_pwmMotors.h` 定义 `SIN_TABLE_SIZE=720`，`GenerateSinTable()` 也按 720 次循环填充 `sinTable[]`。
`PWM_Motor_SetAngle()` 的索引公式使用的是 `(float)SIN_TABLE_SIZE`，不是固定常数 360。
所以当前运行边界应以宏和表达式为准。注释可能是早期 360 点表或角度习惯留下的文字，不能替代表长证据。

问题六：为什么三相索引相差 `SIN_TABLE_SIZE / 3`。

一圈分成三等份就是 120 度。三相正弦输出需要 A/B/C 三相相位相差 120 度，因此用三分之一表长作为索引偏移。

问题七：三相都加 `half_period`，是否会破坏三相平衡。

不会破坏软件目标值中的交流分量平衡。
`half_period` 是公共偏置，两相相减时会抵消。
它的作用是把正负正弦值平移到定时器可写的非负 CCR 范围。
真实电流是否平衡仍需外部测量【待验证】。

问题八：三相正弦表正确，是否就说明电机方向一定正确。

不能。
正弦表只能说明软件生成了相差 120 度的三相目标。
真实电机方向还受相线顺序、轴映射、机械安装、符号常量和电角偏置影响。
这些需要第30章的 CCR 映射和第33章的仓库外点测共同验证。

问题九：`power_percent` 是否等于电机力矩百分比。

不能直接这样理解。
它在源码中用于缩放正弦 CCR 幅值，但真实力矩还受电源、电机绕组、电流环、
驱动器和负载影响。当前项目没有电流闭环和力矩标定证据。

问题十：`power_percent` 超过 60 时会怎样。

源码不是夹到 60，而是回退到 40。
因此如果调试者传入 70，实际幅值会按 40 计算。
这属于源码边界，是否符合设计意图需要另行验证。

问题十一：为什么不能只看 `MotorAxis_t` 注释判断硬件接口。

因为当前逻辑轴调用的 `MOTOR_*` 枚举和 `drv_pwmMotors.c` 中实际写入的 CCR 通道之间存在映射层。
硬件接口结论必须以 `PWM_Motor_SetAngle()` 的 `switch(axis)` 和后续第30章通道表为准。

问题十二：`current_electrical_angle` 是否等于真实转子电角。

当前不能这样写。它是由 IMU 姿态角和换算系数推导出来的软件控制变量。
仓库内没有发现编码器、霍尔传感器解码、反电动势观测器或电流观测器参与主输出链路。
所以真实转子电角是否与它一致，需要硬件标定和相序验证【待验证】。

问题十三：为什么极对数错误和固定偏移错误不能混为一谈。

极对数错误会让电角误差随机械角变化而累积；固定偏移错误更像整体相位平移。
如果只在一个机械角位置测试，二者可能看起来都像“相位不对”。
因此调试时要在多个机械角位置记录软件电角和响应，而不是只调一个 offset。

问题十四：500Hz 控制循环和 20kHz PWM 载波是什么关系。

500Hz 是软件生成新电角和 CCR 目标的频率层级；20kHz 是定时器硬件重复输出 PWM 的载波频率层级。
一次控制更新大约跨越 40 个 PWM 周期。二者混淆会导致把“CCR 更新慢”误判为“PWM 频率低”。

问题十五：为什么 `power_percent=60.1` 可能比 `60.0` 输出更小。

因为当前源码不是把超过 60 的值夹到 60，而是回退到 40。
所以 `60.0` 会按 60% 幅值计算，而 `60.1` 会被改成 40%。
这是非单调边界，调参时必须记录函数内部实际使用的值。

问题十六：`isInitialized == 1` 是否说明所有 PWM 通道都输出正常。

不能。
它只说明 `PWM_Motor_Init()` 走到了设置软件标志的位置。
当前源码没有逐项检查 `HAL_TIM_PWM_Start()` 的返回值，也没有示波器证据。
因此通道是否真正输出仍需寄存器和硬件波形验证【待验证】。

问题十七：为什么正弦表长度最好能被 3 整除。

三相正弦需要 120° 和 240° 的相位偏移。
当前代码用 `SIN_TABLE_SIZE / 3` 和 `2 * SIN_TABLE_SIZE / 3` 做整数索引偏移。
720 能被 3 整除，所以偏移正好是 240 和 480 个表点。
若未来表长不能被 3 整除，三相偏移会出现结构性相位误差。

问题十八：如果枚举注释和 `switch(axis)` 注释不一致，应相信哪一个。

应优先相信实际寄存器写入路径。
注释可以帮助理解作者意图，但当前 `drv_pwmMotors.h` 的枚举注释和
`drv_pwmMotors.c` 的 `switch(axis)` 注释并不完全一致。
调试时应从 `computeMotorCommands.c` 的 `PWM_Motor_SetAngle()` 调用开始，
追到 `drv_pwmMotors.c` 中对应 `case`，再记录最终写入的 `TIMx->CCRn`。
真实物理接口还要用第30章通道表和第33章硬件点测验证。

## 13. 本章总结

本章把第28章的 PID 输出继续连接到电机相位输出前端。

当前项目中：

- `mechanical2electricalDegrees[]` 和 `electrical2mechanicalDegrees[]` 表示机械角与电角换算关系。
- 电角理论关系应按极对数计算，14 极电机通常对应 7 极对。
- 当前源码同时存在 7、14、20 三类换算系数证据，不能只凭一个数值断言真实极对数。
- Roll、Pitch、Yaw 的电角计算路径并不完全相同，存在不同的符号、偏移和分支写入。
- `pidCmd[]` 是角度补偿量，不是 PWM 占空比。
- `PWM_Motor_SetAngle()` 的输入 `angle_rad` 是电气角度。
- 当前 `current_electrical_angle` 是由 IMU 姿态角和换算系数推导的软件电角，不是实测转子电角。
- `PWM_Motor_SetAngle()` 内部把电角归一化到 `[0, 2π)`，再映射到 `SIN_TABLE_SIZE=720` 的正弦表索引。
- 当前 `sinTable[]` 的有效索引范围是 0 到 719；源码注释中的“0 ~ 359”与当前宏定义不一致，应以 `SIN_TABLE_SIZE` 和表达式为准。
- 720 点正弦表的电角分辨率约为 0.5°，占用约 1440 字节 RAM。
- 720 点表能被 3 整除，因此 `SIN_TABLE_SIZE / 3` 正好对应 120° 相位偏移。
- `SIN_SCALE=32767` 把正弦值以 16 位整数形式保存，再缩放回 CCR 幅值。
- 三相正弦分量理论和为 0，三相 CCR 理论和约为 `3 * half_period`。
- `half_period` 是公共偏置，两相差值会抵消它。
- 当前索引使用强制类型转换截断，720 点表的相位量化误差小于一个表步长。
- `power_percent` 控制 CCR 围绕 `half_period` 的摆动幅值，不等于真实力矩百分比。
- 当前源码中 `power_percent > 60` 会回退到 40，而不是夹到 60。
- 由于 `60.0` 保持 60、`60.1` 回退到 40，功率参数在阈值附近存在非单调跳变。
- 当前溢出保护把过大的 `ccr_val[i]` 夹到 `period=ARR+1`，而不是 `ARR`；正常路径通常不触发，触发时应视为需要排查的边界状态。
- `PWM_Motor_SetAngle()` 在 `isInitialized == 0` 时静默返回，不会报告 CCR 未写入。
- `PWM_Motor_Init()` 当前没有逐项检查 `HAL_TIM_PWM_Start()` 返回值，`isInitialized == 1` 不能单独证明硬件通道输出正常。
- 当前 TIM2/TIM3/TIM4 的 `Period` 都是 3599，`PWM_Motor_SetAngle()` 才能用 TIM3 周期作为统一 CCR 基准。
- 500Hz 控制更新和约 20kHz PWM 载波属于不同时间尺度，一次控制更新大约跨越 40 个 PWM 周期。
- `fastTrig.c` 提供 1024 点 `fastSin()` 查表实现，但当前主电机输出链路使用 `drv_pwmMotors.c` 内部 720 点 `sinTable[]`。
- `config.c` 中存在三轴电机极对数字段，但当前未发现它们参与 `computeMotorCommands.c` 的换算。
- 逻辑姿态轴、`MOTOR_*` 枚举和物理 CCR 通道之间存在映射层，不能只凭名字推断硬件输出。
- `drv_pwmMotors.h` 的枚举注释与 `drv_pwmMotors.c` 的 `switch(axis)` 注释存在映射叙事差异，应以实际 `TIMx->CCRn` 写入和硬件点测为准。
- 当前 `.map` 和 `.list` 能证明角度合成、正弦表、PWM 初始化、三轴调用点和 CCR 写入路径进入 Debug 固件，但不能证明真实转子电角、相序、电流、力矩或示波器波形正确。

本章边界是：只讲机械角、电角、正弦查表和 `PWM_Motor_SetAngle()` 的角度输入逻辑；不展开 TIM2/TIM3/TIM4 具体通道写入和三轴硬件映射。

本章保留十三个工程边界：

- 极数、极对数和运行时换算系数必须分开验证，不能把 `rollMotorPoles=14` 直接等同于当前电角系数。
- 正弦表和 CCR 公式只能证明目标占空比生成方式，不能证明真实相电流或力矩。
- 软件电角只是控制变量，不能替代转子电角传感器或硬件相位标定证据。
- `power_percent` 阈值回退是源码行为，是否是最终硬件保护策略仍需验证。
- `isInitialized` 是软件状态标志，不是 PWM 引脚波形或驱动器输出证据。
- `HAL_TIM_PWM_Start()` 返回值未逐项检查，初始化失败可能缺少直接日志证据。
- 表长若未来修改，必须重新验证三相 120° 索引偏移是否仍然成立。
- CCR 溢出保护在当前正常功率路径下通常不会触发；触发时还要注意保护上限是 `period=ARR+1`，不是 `ARR` 本身。
- 电角方向、相序和轴映射正确性需要第30章和第33章继续验证。
- 枚举名和源码注释只能作为线索，不能替代 `PWM_Motor_SetAngle()` 中的实际 CCR 写入证据。
- `POLE_PAIRS` 与 `rollMotorPoles/pitchMotorPoles/yawMotorPoles` 当前有定义证据，
  但缺少主控制路径读取证据。
- 当前没有 SVPWM、FOC、电流采样或电流闭环证据，不能把查表正弦输出写成矢量控制。
- `.map/.list` 属于构建产物证据，只能证明符号、对象和调用/写寄存器路径进入当前固件，不能替代硬件实测或运行时正确性证明。

下一章将进入 `多定时器三相电机驱动`，继续说明 `PWM_Motor_SetAngle()` 如何把三相 CCR 目标值写入不同定时器通道，并形成 Roll、Pitch、Yaw 的实际执行链。
