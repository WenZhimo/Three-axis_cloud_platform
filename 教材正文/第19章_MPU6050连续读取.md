## 3. 问题背景

MPU6050 初始化只是让芯片进入工作状态。云台控制真正需要的是周期性、稳定、可追踪的传感器输入。

项目面对三个工程问题：

- 什么时候读传感器。
- 在哪里读传感器。
- 读出的 14 字节如何进入项目数据结构。

如果在中断里直接执行 I2C 读取，可能拉长中断时间，并与主循环读取产生冲突。项目源码已经在 `Core/Src/stm32f1xx_it.c` 中留下说明：读 I2C 会耗时，并且中断中读、主函数中也读会造成总线冲突和读数为 0。因此项目采用“中断置位，主循环执行读取”的方式。

这就是本章的主线：中断产生节拍，主循环消费节拍，读取函数完成字节到原始量的转换。

## 4. 核心概念

### 4.1 500Hz帧触发标志

`Core/Inc/main.h` 定义：

```c
#define FRAME_COUNT   1000
#define COUNT_500HZ   2
```

`Core/Src/stm32f1xx_it.c` 的 `SysTick_Handler()` 中，当 `systemReady == true` 且 `mpu6050Calibrating == false` 时，`frameCounter` 递增。每当：

```c
frameCounter % COUNT_500HZ == 0
```

就设置：

```c
frame_500Hz = true;
```

这说明当前项目使用 SysTick 相关逻辑生成 500Hz 帧标志。该标志不是数据本身，而是通知主循环“可以执行一帧 500Hz 工作”。

### 4.2 主循环消费标志

`Core/Src/main.c` 的 `while (1)` 中检查：

```c
if (frame_500Hz)
{
    frame_500Hz = false;
    ...
    MPU6050_Read_And_Process();
}
```

主循环先清除标志，再计算 `dt500Hz`，然后调用 `MPU6050_Read_And_Process()`。

这是一种常见嵌入式结构：中断只做短操作，主循环做耗时操作。I2C 读取不放在 `SysTick_Handler()` 内部，可以降低中断阻塞风险。

这里还要拆出两个容易被忽略的实时语义。

第一，`frame_500Hz` 是布尔标志，不是事件计数器。若主循环在某一帧内处理过慢，SysTick
可能再次把它置为 `true`，但标志本身不会记录“积压了几帧”。因此它适合表达“至少有一帧待处理”，
不适合统计错过了多少个 500Hz 周期。是否发生丢帧，需要结合 `dt500Hz` 和
`executionTime500Hz` 观察。

第二，`frame_500Hz`、`systemReady`、`mpu6050Calibrating` 由中断和主循环共享，但当前
`main.c` / `main.h` 中 `frame_500Hz` 不是 `volatile` 声明。多数调试构建可能仍能按预期工作，
但从 C 语言和嵌入式并发边界看，跨中断共享标志通常应使用 `volatile` 或更明确的同步机制。
本轮不修改固件，只把它作为教材中的可见性风险和代码审查点记录；当前编译优化下是否触发问题仍需【待验证】。

还要分清 TIM6 和 500Hz 标志的角色。项目调用了 `MX_TIM6_Init()`，并在进入主循环前执行
`__HAL_TIM_SET_COUNTER(&htim6, 0)`，但第19章看到的 500Hz 触发证据来自
`SysTick_Handler()` 中的 `COUNT_500HZ` 分频。还要注意，当前 `micros()` 实现也没有读取
`TIM6->CNT`，而是用 `HAL_GetTick()` 和 `SysTick->VAL` 组合计算微秒时间。因此 TIM6 在本章只能记录为“已初始化且被清零过的定时器对象”，不能写成 500Hz 中断源，也不能写成当前 `micros()` 的直接计数来源。

### 4.3 14字节连续读取

`Drivers/CustomDrivers/Inc/mpu6050.h` 定义：

```c
#define REG_ACCEL_XOUT_H        0x3B
#define MPU_DATA_LENGTH         14
```

MPU6050 的连续输出区从加速度计 X 轴高字节开始，项目一次读取 14 字节：

- 加速度计 X/Y/Z：6 字节。
- 温度：2 字节。
- 陀螺仪 X/Y/Z：6 字节。

`MPU6050_Read_And_Process()` 调用：

```c
MPU_Read_Reg(REG_ACCEL_XOUT_H, I2C2_Buffer_Rx, MPU_DATA_LENGTH)
```

这就是本章正式知识点 `MPU6050连续数据读取` 的核心证据。

### 4.4 输出寄存器连续性

MPU6050 手册把加速度、温度、陀螺仪输出放在一段连续寄存器中：

| 十六进制地址 | 寄存器 | 缓冲区索引 | 数据含义 |
| --- | --- | --- | --- |
| `0x3B` | `ACCEL_XOUT_H` | `[0]` | X 轴加速度高字节 |
| `0x3C` | `ACCEL_XOUT_L` | `[1]` | X 轴加速度低字节 |
| `0x3D` | `ACCEL_YOUT_H` | `[2]` | Y 轴加速度高字节 |
| `0x3E` | `ACCEL_YOUT_L` | `[3]` | Y 轴加速度低字节 |
| `0x3F` | `ACCEL_ZOUT_H` | `[4]` | Z 轴加速度高字节 |
| `0x40` | `ACCEL_ZOUT_L` | `[5]` | Z 轴加速度低字节 |
| `0x41` | `TEMP_OUT_H` | `[6]` | 温度高字节 |
| `0x42` | `TEMP_OUT_L` | `[7]` | 温度低字节 |
| `0x43` | `GYRO_XOUT_H` | `[8]` | X 轴陀螺仪高字节 |
| `0x44` | `GYRO_XOUT_L` | `[9]` | X 轴陀螺仪低字节 |
| `0x45` | `GYRO_YOUT_H` | `[10]` | Y 轴陀螺仪高字节 |
| `0x46` | `GYRO_YOUT_L` | `[11]` | Y 轴陀螺仪低字节 |
| `0x47` | `GYRO_ZOUT_H` | `[12]` | Z 轴陀螺仪高字节 |
| `0x48` | `GYRO_ZOUT_L` | `[13]` | Z 轴陀螺仪低字节 |

这解释了为什么项目从 `0x3B` 开始一次读 14 字节，而不是分别读 6 次或 7 次。
连续读减少 I2C 事务次数，也降低不同轴来自不同读取时刻的风险。

连续读取的另一个意义是“寄存器地址自动递增”。主机只发送一次起始寄存器地址 `0x3B`，
后续每收到一个字节，MPU6050 内部读指针进入下一个输出寄存器。若把 14 字节拆成多次读取，
不仅总线开销更高，还会让加速度、温度和陀螺仪来自不同事务窗口；这些轴之间的同步程度会进一步下降。

还要把“连续读的一致性”和“样本新鲜度”分开。MPU6050 寄存器手册说明，突发读取有助于让一组传感器输出来自同一采样更新窗口；但当前项目没有读取 `INT_STATUS` 的 Data Ready 位，也没有启用 FIFO 或读取 FIFO 计数。因此本章可以说“一次 14 字节突发读比拆分多次读更有利于同窗口读取”，不能说“每次 500Hz 主循环都一定拿到了全新样本”。

### 4.5 高低字节合成

MPU6050 输出寄存器以高字节在前、低字节在后的方式排列。一个轴的 16 位有符号原始量可写成：

```text
raw = (int16_t)((high << 8) | low)
```

项目没有直接写这个表达式，而是通过联合体实现：

```c
rawAccel[XAXIS].bytes[1] = I2C2_Buffer_Rx[0];
rawAccel[XAXIS].bytes[0] = I2C2_Buffer_Rx[1];
```

这种写法依赖当前 Cortex-M3 小端存储约定：`bytes[0]` 对应低 8 位，`bytes[1]` 对应高 8 位。
所以本章要同时讲“传感器寄存器顺序”和“MCU 内存字节顺序”，不能只说“两个字节拼成一个 `int16_t`”。

## 5. 工作原理

第19章的数据路径可以拆成五步：

1. `SysTick_Handler()` 根据帧计数设置 `frame_500Hz`。
2. `main()` 主循环发现 `frame_500Hz == true` 后清除标志。
3. 主循环计算 `deltaTime500Hz` 和 `dt500Hz`。
4. 主循环调用 `MPU6050_Read_And_Process()` 执行 I2C 连续读取。
5. 读取函数把 14 字节写入 `rawAccel`、`rawMPU6050Temperature` 和 `rawGyro`。

读取函数内部又可以拆成四段：

- 使用 `MPU_Read_Reg()` 从 `REG_ACCEL_XOUT_H` 开始读取 14 字节。
- 如果读取失败，清零 `rawAccel` 和 `rawGyro` 并返回。
- 按高字节、低字节顺序填充联合体的 `bytes[1]` 和 `bytes[0]`。
- 把原始值送入方向矩阵处理，再写回 `rawAccel` 和 `rawGyro`。

本章重点是前三段。第四段涉及方向矩阵，属于第20章。

第18章已经推导出当前配置下 MPU6050 陀螺仪寄存器更新基准可按 800Hz 理解。
第19章看到的是主循环按 `COUNT_500HZ=2` 尝试以 500Hz 消费数据。

二者关系可以写成：

```text
传感器寄存器更新周期约为 1 / 800 = 1.25 ms
主循环读取周期目标约为 1 / 500 = 2.00 ms
每个主循环读取周期约跨过 800 / 500 = 1.6 个传感器更新周期
```

这意味着项目并不是读取每一个 MPU6050 内部更新样本，而是以 500Hz 从更高的寄存器更新流中取样。
但是当前源码没有使用 MPU6050 数据就绪中断，也没有读取中断状态寄存器来锁定采样相位。
因此“每次主循环读取是否严格对应全新样本”仍需运行日志或外部测量【待验证】。

## 6. STM32实现机制

STM32 侧的关键机制是：不要在中断里执行耗时 I2C 事务。

`SysTick_Handler()` 本身每 1 ms 进入一次。项目在其中维护 `frameCounter`，每两次置位一次 `frame_500Hz`。如果把 I2C 连续读取也放在中断里，会增加中断执行时间，并可能影响其他实时逻辑。

项目源码中的注释已经给出工程判断：中断中读 I2C 会耗时，且和主循环重复读取会造成总线冲突和读数为 0。因此当前实现把中断职责限制为置位标志。

从 I2C2 配置看，`Core/Src/i2c.c` 中 `hi2c2.Init.ClockSpeed = 100000`，也就是 100kHz 标准模式。
一次 14 字节连续读至少包括设备地址写、寄存器地址、设备地址读和 14 个数据字节。
按每字节 8 位数据加 1 位 ACK/NACK 粗略估算：

```text
(1 + 1 + 1 + 14) * 9 = 153 个 SCL 周期
153 / 100000 = 1.53 ms
```

这个 1.53ms 是总线位传输理论下限，不包含起始/停止条件、HAL 轮询等待、函数调用、上升沿余量和器件响应差异。
它说明 14 字节阻塞式 I2C 读取已经占用 2ms 控制帧的大部分预算，必须用 `executionTime500Hz`、DWT 或逻辑分析仪实测。

HAL 层还可以继续拆解。`MPU_Read_Reg()` 内部调用：

```c
HAL_I2C_Mem_Read(&hi2c2, MPU6050_I2C_ADDR, REG_ACCEL_XOUT_H,
                 I2C_MEMADD_SIZE_8BIT, I2C2_Buffer_Rx, 14, 100);
```

在 `STM32F1xx HAL I2C` 阻塞式实现中，`HAL_I2C_Mem_Read()` 不是简单“读 14 个字节”。
它会先等待总线空闲，锁定 I2C 句柄，把状态置为 `HAL_I2C_STATE_BUSY_RX`，然后执行内存读事务：

```text
START
7位设备地址 + Write
8位寄存器地址 0x3B
RESTART
7位设备地址 + Read
连续接收 14 字节
STOP
```

这解释了为什么同一个 I2C 句柄不适合在 SysTick 和主循环中同时访问。HAL 句柄内部有
`State`、`Mode`、`ErrorCode`、`XferSize` 等状态，阻塞式读期间它处于忙状态；重入访问要么被拒绝，
要么造成总线时序和数据流混乱。

还要注意超时参数。`MPU_Read_Reg()` 传入的超时是 `100 ms`，而 500Hz 控制帧目标周期是
`2 ms`。一旦总线异常导致阻塞等待接近超时，单次读取失败就可能跨过许多个 500Hz 帧。
这不是稳态性能，而是异常路径性能；但对“系统失联后是否还能及时保护”很重要，当前仓库没有实测保护记录，
应标记为【待验证】。

主循环拿到标志后执行：

- 时间戳测量：`currentTime = micros()`。
- 周期计算：`dt500Hz = deltaTime500Hz / 1000000.0f`。
- I2C 连续读取：`MPU6050_Read_And_Process()`。
- 后续处理：写入 `sensors.accel500Hz`、`sensors.gyro500Hz` 并调用姿态更新。

后续处理虽然出现在同一个 500Hz 块中，但不属于本章重点。第19章只确认原始数据如何进入这个块。

因此，本章的 STM32 结论不是“当前 500Hz 一定稳定成立”，而是：

- 源码意图是 SysTick 分频生成 500Hz 工作标志。
- 实际读取使用阻塞式 HAL I2C Mem 接口。
- 100kHz I2C 的 14 字节连续读理论总线时间已经接近 2ms 帧预算。
- 稳态是否守住 500Hz、异常时是否超时拖垮主循环，需要 `executionTime500Hz`、DWT 或逻辑分析仪证明。

## 7. 项目中的应用

本章涉及的项目证据文件包括：

- `Drivers/CustomDrivers/Src/mpu6050.c`
- `Drivers/CustomDrivers/Inc/mpu6050.h`
- `Core/Src/main.c`
- `Core/Src/stm32f1xx_it.c`
- `Core/Inc/main.h`

`Core/Inc/main.h` 提供 `FRAME_COUNT`、`COUNT_500HZ` 和 `frame_500Hz` 的声明。

`Core/Src/stm32f1xx_it.c` 在 `SysTick_Handler()` 中置位 `frame_500Hz`。

`Core/Src/main.c` 在主循环中清除该标志并调用读取函数。

`Drivers/CustomDrivers/Src/mpu6050.c` 中的 `MPU6050_Read_And_Process()` 是连续读取的实际执行点。

它先读取 14 字节缓冲区 `I2C2_Buffer_Rx[14]`。

随后函数填充 `rawAccel[3]`、`rawMPU6050Temperature`、`rawGyro[3]`。

读取完成后，`Core/Src/main.c` 立刻使用 `rawAccel` 和 `rawGyro` 计算 `sensors.accel500Hz` 和 `sensors.gyro500Hz`。这说明第19章是第21章物理量缩放和第23章姿态解算的输入前置。

## 8. 代码分析

### 8.1 SysTick只置位frame_500Hz

`Core/Src/stm32f1xx_it.c` 中的逻辑可以概括为：

```c
if ((systemReady == true) && (mpu6050Calibrating == false))
{
    frameCounter++;
    if ((frameCounter % COUNT_500HZ) == 0)
    {
        frame_500Hz = true;
    }
}
```

源码中原本有一段在中断里调用 `MPU6050_Read_And_Process()` 并复制 `accelData500Hz`、`gyroData500Hz` 的代码，但已经被注释掉。注释说明原因是 I2C 耗时，并且中断和主函数重复读取会造成总线冲突。

因此，`accelData500Hz` 和 `gyroData500Hz` 虽然在 `mpu6050.h` 中声明、在 `mpu6050.c` 中定义，但当前主线并未使用中断中的复制逻辑。教材不能把它们写成当前 500Hz 实时控制循环的数据来源。

#### 8.1.1 `accelData500Hz/gyroData500Hz`的构建证据边界

这两个数组的名字很像“500Hz 控制环的正式传感器数据”，但当前证据要分层看：

- 源码声明层：`Drivers/CustomDrivers/Inc/mpu6050.h` 声明 `extern int16_t accelData500Hz[3]` 和 `gyroData500Hz[3]`，`mpu6050.c` 也给出定义。
- 注释历史层：`Core/Src/stm32f1xx_it.c` 中给这两个数组赋值的代码位于注释块内，当前中断路径只保留 `frame_500Hz = true`。
- 构建产物层：`Debug/Three-axis_cloud_platformV2.map` 中 `accelData500Hz` 和 `gyroData500Hz` 只出现在 `0x00000000` 输入段记录里，没有像 `rawAccel`、`rawGyro` 那样获得最终 RAM 地址；`.list` 中主循环后续读取的是 `rawAccel/rawGyro` 并写入 `sensors.*500Hz`。

所以本章可以把 `accelData500Hz/gyroData500Hz` 写成“历史中断读取方案留下的未接入数组”，不能写成当前主控制路径的采样缓存。调试时如果只观察这两个数组，可能会误判 500Hz 数据没有更新；当前应优先观察 `rawAccel/rawGyro`、`sensors.accel500Hz/sensors.gyro500Hz` 以及 `MPU6050_Read_And_Process()` 的调用点。

`frame_500Hz` 还有一个隐藏语义：它只能表示“有一帧工作待主循环处理”，不能表示“有多少帧工作待处理”。
如果主循环某次执行超过 2ms，下一次 SysTick 到来时仍只是把这个布尔值设为 `true`。
等主循环重新进入 `if (frame_500Hz)` 时，它只会处理一次读取和一次姿态/控制更新。

因此，`dt500Hz` 能反映“这次处理距离上次处理过了多久”，但不能补回中间没有读取到的传感器样本。
例如一次异常耗时让两次处理间隔变成 4ms，`dt500Hz` 可以变大到约 `0.004s`，
后续积分可按较大的时间步长计算；但这仍然只有一次 I2C 读取，不等于补采了两帧 MPU6050 数据。

跨中断共享标志还涉及编译器可见性。当前 `frame_500Hz` 在 `main.c` 中定义为普通 `bool`，
在 `main.h` 中也以普通 `extern bool` 声明。严格的代码审查会把它标记为应考虑 `volatile` 的共享变量。
本教材只记录风险，不修改源码；若要证明当前编译产物无该问题，需要结合优化等级和反汇编【待验证】。

### 8.2 主循环计算dt并读取

`Core/Src/main.c` 在 500Hz 块中做三件与本章相关的事：

```c
currentTime = micros();
deltaTime500Hz = currentTime - previous500HzTime;
dt500Hz = (float)deltaTime500Hz / 1000000.0f;
MPU6050_Read_And_Process();
```

`dt500Hz` 后续会给姿态算法和控制算法使用，但本章不展开积分和控制意义。这里需要抓住一点：读取函数处在 500Hz 主循环块的开头，原始传感器数据是本帧后续处理的基础。

从时间语义看，本章要区分三种“时间”：

- `COUNT_500HZ` 期望周期：由 SysTick 分频得到，目标约 `2ms`。
- `dt500Hz` 实际周期：由 `micros()` 差值计算，反映主循环实际消费间隔。
- 传感器样本时间：由 MPU6050 内部寄存器更新和主循环读取相位共同决定，当前没有数据就绪中断证据。

这三个时间接近时，系统表现可能接近理想 500Hz；三者分离时，姿态积分、滤波和 PID 都会受到影响。
当前仓库只能证明软件计算了 `dt500Hz`，不能证明实际样本相位稳定。

还要注意，`currentTime = micros()` 出现在调用 `MPU6050_Read_And_Process()` 之前。它更适合作为本帧软件调度时间戳，而不是 MPU6050 内部采样瞬间的时间戳。当前 14 字节数据里没有样本序号，项目也没有为每次读取保存独立时间戳，所以无法仅凭仓库代码判断两次相邻读取是否读到了同一个传感器更新窗口。

### 8.3 I2C连续读取失败处理

`MPU6050_Read_And_Process()` 内部先定义：

```c
uint8_t I2C2_Buffer_Rx[14];
```

这个缓冲区是函数栈上的局部数组，不是持久采样缓存。函数返回后，后续代码能看到的是已经写入全局变量的 `rawAccel`、`rawMPU6050Temperature` 和 `rawGyro`，而不是 `I2C2_Buffer_Rx` 本身。因此如果要调试 14 字节原始突发读结果，应在 `MPU6050_Read_And_Process()` 内部断点观察，或显式增加日志/复制缓冲区；不能在函数返回后再把 `I2C2_Buffer_Rx` 当作可追踪的数据源。

然后执行：

```c
if (MPU_Read_Reg(REG_ACCEL_XOUT_H, I2C2_Buffer_Rx, MPU_DATA_LENGTH) != HAL_OK)
{
    memset(rawAccel, 0, sizeof(rawAccel));
    memset(rawGyro, 0, sizeof(rawGyro));
    return;
}
```

这说明读取失败时，项目没有沿用上一帧数据，而是清零加速度和陀螺仪原始值。这个行为会影响后续姿态算法和控制输出，调试时必须注意。

还要注意一个细节：失败路径只清零 `rawAccel` 和 `rawGyro`，没有清零 `rawMPU6050Temperature`。
如果后续流程在失败后继续使用温度值，那么它可能是上一帧或默认值。
当前 500Hz 主循环不直接调用 `MPU6050_ComputeTemError()`，但静态误差估计流程会调用温度补偿函数。
因此读取失败与温度数据新鲜度之间的关系应作为调试记录项，而不能默认温度同步更新。

失败清零的传播路径也要拆开看。`MPU6050_Read_And_Process()` 自身没有返回 `bool` 或错误码，
所以 `main.c` 无法在本帧直接判断“本次读取失败”。它会继续执行：

- 用清零后的 `rawAccel/rawGyro` 计算 `sensors.accel500Hz` 和 `sensors.gyro500Hz`。
- 调用 `MargAHRSupdate()`。
- 在启动收敛后调用 `computeMotorCommands(dt500Hz)`。

这意味着读取失败在当前主线中会被转换为一组看似合法的数值输入，而不是显式故障状态。
清零让问题更容易在数据上暴露，但也可能把一次通信故障变成姿态和控制输入突变。
是否需要故障计数、保留上一帧、进入保护状态或禁用电机，属于后续安全策略设计；
当前仓库没有这类闭环保护证据，保持【待验证】。

换句话说，失败路径的工程问题不只是“清零还是不清零”，还包括“上层是否知道失败”。当前函数返回类型是 `void`，主循环没有错误码、失败计数、连续失败阈值或降级状态可用。教材在这里要让读者看见接口边界：通信失败被折叠进全局数据值，而不是被显式传播成控制状态。

#### 8.3.1 HAL状态码与I2C错误码边界

`MPU_Read_Reg()` 的返回值类型是 `HAL_StatusTypeDef`。根据 `stm32f1xx_hal_def.h`，这个类型只区分四类总体结果：

```text
HAL_OK      = 0
HAL_ERROR   = 1
HAL_BUSY    = 2
HAL_TIMEOUT = 3
```

但 `MPU6050_Read_And_Process()` 并没有保存这个返回值，也没有把它传给主循环，而是只做 `!= HAL_OK` 判断。这意味着当前 500Hz 主线只能区分“本次 14 字节读取成功或失败”，不能从函数接口区分是总线忙、超时、地址无应答、仲裁丢失还是其它 I2C 错误。

更细的原因在 HAL I2C 句柄里。`stm32f1xx_hal_i2c.h` 为 `I2C_HandleTypeDef` 定义了 `State`、`Mode` 和 `ErrorCode` 字段，错误码包括 `HAL_I2C_ERROR_BERR`、`HAL_I2C_ERROR_ARLO`、`HAL_I2C_ERROR_AF`、`HAL_I2C_ERROR_OVR` 和 `HAL_I2C_ERROR_TIMEOUT` 等。`HAL_I2C_Mem_Read()` 内部会等待 `BUSY`、`RXNE`、`BTF` 等标志，并在部分超时路径设置 `HAL_I2C_ERROR_TIMEOUT`；如果 I2C 句柄本来不是 `HAL_I2C_STATE_READY`，函数还可能直接返回 `HAL_BUSY`。

因此，读取失败的调试记录至少应分三层写：

| 层级 | 可观察对象 | 能说明什么 | 不能说明什么 |
|---|---|---|---|
| 项目封装层 | `MPU_Read_Reg(...) != HAL_OK` | 本帧 14 字节读取没有按项目期望完成。 | 不能区分 NACK、BUSY、TIMEOUT 或其它错误来源。 |
| HAL状态层 | `HAL_StatusTypeDef` 返回值 | 能按 HAL 公共类型区分总体状态；在本读路径中至少可把 `HAL_BUSY` 与 `HAL_ERROR` 从非 OK 中拆出。 | `HAL_ERROR` 仍可能包含超时、应答失败或其它 I2C 细分原因。 |
| HAL句柄层 | `hi2c2.State`、`hi2c2.Mode`、`hi2c2.ErrorCode` | 能进一步定位状态机位置和错误类别。 | 仍不能单独证明外部上拉、电源、线缆或 MPU6050 内部状态正确。 |

`Debug/Three-axis_cloud_platformV2.list` 能证明 `MPU_Read_Reg()` 调用了 `HAL_I2C_Mem_Read()`，也能看到 `HAL_I2C_Mem_Read()` 包含多条 `return HAL_ERROR`、`return HAL_BUSY` 和 `return HAL_OK` 路径；`Debug/Drivers/STM32F1xx_HAL_Driver/Src/stm32f1xx_hal_i2c.su/.cyclo` 还能证明这些等待和分支进入当前 Debug 构建。但这些构建产物仍然只是静态证据，不能替代运行时对 `hi2c2.ErrorCode` 和 I2C 波形的观察【待验证】。

#### 8.3.2 读取函数接口返回值断点

把失败路径继续拆细后，可以得到一个很重要的接口结论：当前 `MPU6050_Read_And_Process()` 的调用者只能观察“函数已经返回”，不能直接观察“本次 14 字节突发读是否成功”。这是因为 `Drivers/CustomDrivers/Inc/mpu6050.h` 中声明的是：

```c
void MPU6050_Read_And_Process(void);
```

而 `Drivers/CustomDrivers/Src/mpu6050.c` 内部把 `MPU_Read_Reg(...) != HAL_OK` 消费掉以后，只通过 `memset(rawAccel, 0, sizeof(rawAccel))` 和 `memset(rawGyro, 0, sizeof(rawGyro))` 改写全局数据。`Core/Src/main.c` 的 500Hz 分支在 `MPU6050_Read_And_Process()` 返回后，继续读取 `rawAccel/rawGyro`，写入 `sensors.accel500Hz/sensors.gyro500Hz`，再调用 `MargAHRSupdate()`。因此，主循环看到的是数据对象的新状态，而不是一次显式的通信状态。

这个断点适合用“接口契约”来讲：

| 证据对象 | 当前项目能证明什么 | 不能证明什么 |
|---|---|---|
| `void MPU6050_Read_And_Process(void)` | 调用者没有返回值可接收。 | 不能从函数签名判断本帧读取成功或失败。 |
| `if (MPU_Read_Reg(...) != HAL_OK) { ... return; }` | HAL 读取失败在函数内部被折叠为清零 `rawAccel/rawGyro`。 | 不能把失败原因传播给主循环，也不能证明温度值同步失效。 |
| `main()` 中先调用读取函数，再换算 `sensors.*500Hz` | 失败后的清零值会继续进入后续数据流。 | 不能证明本帧 AHRS 输入来自一帧有效的 MPU6050 新样本。 |
| `.map` 中的 `.bss.rawAccel/.bss.rawGyro/.bss.rawMPU6050Temperature` | 这些全局对象确实进入最终 RAM 布局。 | 不能证明某次运行中对象内容来自成功读取。 |
| `.list` 中 `main()` 到 `MPU6050_Read_And_Process()` 的 `bl` 指令 | 当前构建保留了主循环调用路径。 | 不能证明每次 500Hz 分支都按期完成，也不能证明硬件总线无错误。 |

所以，本章不能把“存在连续读取函数”和“主循环每帧得到有效传感器样本”写成同一件事。更严谨的说法是：当前构建能证明 500Hz 主线会调用连续读取函数，并把读取函数留下的全局对象继续送入 AHRS；但“本帧样本有效”“本帧样本新鲜”“失败后已经进入保护策略”都需要额外运行时证据，当前保持【待验证】。

如果未来要把这个接口边界变成更可验证的工程接口，最小改动方向不是直接相信 `rawAccel/rawGyro` 的数值，而是让读取函数显式输出状态，例如返回 `bool`、保存 `lastReadStatus`、累计 `readFailCount`、记录成功时间戳，或在 `sensors` 中增加数据有效标志。当前仓库没有这些实现，本段只作为接口审查标准，不把它写成已完成的工程能力。

还要再向后推一层：清零 `rawAccel/rawGyro` 并不等于传给 AHRS 的输入全为 `0`。`Core/Src/main.c` 在读取函数返回后继续做偏置扣除和比例换算：

```c
sensors.accel500Hz[XAXIS] =
    ((float)rawAccel[XAXIS].value - accelTCBias[XAXIS]) * ACCEL_SCALE_FACTOR;
sensors.gyro500Hz[ROLL] =
    ((float)rawGyro[ROLL].value - gyroRTBias[ROLL] - gyroTCBias[ROLL]) * GYRO_SCALE_FACTOR;
```

因此在读取失败并清零原始量时，实际传播到 AHRS 的形式是：

```text
accel500Hz[i] = -accelTCBias[i] * ACCEL_SCALE_FACTOR
gyro500Hz[i]  = -(gyroRTBias[i] + gyroTCBias[i]) * GYRO_SCALE_FACTOR
```

`Drivers/CustomDrivers/Src/mpu6050.c` 中当前全局初值为 `accelTCBias = {1.0f, 0.0f, 0.0f}`、`gyroTCBias = {0.0f, 0.0f, 0.0f}`、`gyroRTBias = {0.0f, 0.0f, 0.0f}`，`Drivers/CustomDrivers/Inc/mpu6050.h` 中 `ACCEL_SCALE_FACTOR = 0.00119708f`、`GYRO_SCALE_FACTOR = 0.00026646f`。在这些初值尚未被温度补偿或运行校准改写时，一次读取失败会给 AHRS 一个约 `(-0.00119708, 0, 0)` 的加速度输入，而不是严格的 `(0, 0, 0)`。

再看 `Drivers/SRC/Src/MargAHRS.c`：`MargAHRSupdate()` 先把 NaN/Inf 输入置零，然后计算：

```c
norm = sqrt(ax*ax + ay*ay + az*az);

if (norm > 0.1f)
{
    ...
}
```

也就是说，读取失败后的偏置派生输入会继续进入 AHRS。若加速度范数不超过 `0.1f`，加速度校正分支被跳过；若某次温度补偿或校准后的偏置使范数超过阈值，AHRS 可能把这组由故障路径产生的有限数值当作可归一化的重力方向参与校正。无论加速度校正是否进入，后续四元数积分与归一化路径仍会继续执行。这个边界非常适合作为调试题：通信失败没有被上层显式识别，而是被转换成一组“数学上有限、语义上可疑”的姿态输入。

### 8.4 加速度和温度字节装载

读取成功后，项目按高字节、低字节填入联合体：

```c
rawAccel[XAXIS].bytes[1] = I2C2_Buffer_Rx[0];
rawAccel[XAXIS].bytes[0] = I2C2_Buffer_Rx[1];
```

加速度三轴对应：

- `I2C2_Buffer_Rx[0..1]` -> `rawAccel[XAXIS]`
- `I2C2_Buffer_Rx[2..3]` -> `rawAccel[YAXIS]`
- `I2C2_Buffer_Rx[4..5]` -> `rawAccel[ZAXIS]`

温度对应：

- `I2C2_Buffer_Rx[6..7]` -> `rawMPU6050Temperature`

需要注意，源码注释中有一处旧说明写着 `Buffer[0,1]` 赋给 `rawAccel[YAXIS]`，但实际代码赋给的是 `rawAccel[XAXIS]`。教材应以实际赋值代码为准，并把注释不一致作为待清理的可读性问题记录。

按统一公式看，X 轴加速度当前等价于：

```text
rawAccel[XAXIS].value =
    (int16_t)((I2C2_Buffer_Rx[0] << 8) | I2C2_Buffer_Rx[1])
```

Y、Z 轴同理，只是缓冲区索引分别后移 2 字节和 4 字节。
这一步仍然是“原始计数值”，不是 `g`、`m/s^2` 或姿态角。

#### 8.4.1 温度样本的消费边界

14 字节连续读取中间包含温度高低字节，这说明本次 I2C 事务确实把温度寄存器也带入了项目内存。`Debug/Three-axis_cloud_platformV2.list` 中能看到 `I2C2_Buffer_Rx[6]` 写入 `rawMPU6050Temperature.bytes[1]`、`I2C2_Buffer_Rx[7]` 写入 `bytes[0]`，`.map` 也显示 `rawMPU6050Temperature` 位于 `.bss`。

但温度样本和加速度/陀螺仪样本的后续消费路径不同：

| 数据 | 当前装载证据 | 当前直接消费证据 | 教学边界 |
|---|---|---|---|
| `rawAccel/rawGyro` | `MPU6050_Read_And_Process()` 从缓冲区装载并经过方向矩阵写回 | `main.c` 500Hz 分支立刻扣除偏置、乘缩放因子，写入 `sensors.accel500Hz/gyro500Hz` | 可以作为本章到 AHRS 输入链路的直接前置。 |
| `rawMPU6050Temperature` | `I2C2_Buffer_Rx[6..7]` 装入温度联合体 | `MPU6050_ComputeTemError()` 用 `temp = rawMPU6050Temperature.value / 340.0f + 36.53f` 计算温补偏置；`MPU6050_ComputeStaticError()` 会调用该函数 | 可以证明温度样本参与温补路径；不能证明当前 500Hz 主循环每帧都重新计算温补偏置。 |

因此，第19章可以说“14 字节突发读取同时获得加速度、温度和陀螺仪寄存器输出”，但不能把温度样本写成与 `sensors.accel500Hz/gyro500Hz` 同级的 500Hz 主线观测量。当前 `Core/Src/main.c` 的 500Hz 块没有直接调用 `MPU6050_ComputeTemError()`，温度补偿何时更新要沿着初始化静态误差流程和后续温补章节继续核对。

### 8.5 陀螺仪字节装载

陀螺仪三轴对应：

- `I2C2_Buffer_Rx[8..9]` -> `rawGyro[XAXIS]`
- `I2C2_Buffer_Rx[10..11]` -> `rawGyro[YAXIS]`
- `I2C2_Buffer_Rx[12..13]` -> `rawGyro[ZAXIS]`

源码注释特别强调：这里严格按物理轴 X、Y、Z 提取，不在这里擅自映射为 Roll/Pitch。
后续 `rawGyro[ROLL]`、`rawGyro[PITCH]`、`rawGyro[YAW]` 的使用依赖宏定义中
`ROLL=0`、`PITCH=1`、`YAW=2` 与三轴索引的对应关系。

### 8.6 方向矩阵调用边界

字节装载后，函数把 `rawAccel`、`rawGyro` 的 `value` 放入 `straightAccelData`、`straightGyroData`，然后调用：

```c
matrixMultiply(3, 3, 1, rotatedAccelData, orientationMatrix, straightAccelData);
matrixMultiply(3, 3, 1, rotatedGyroData, orientationMatrix, straightGyroData);
```

随后把旋转后的结果写回 `rawAccel` 和 `rawGyro`。

这说明 `MPU6050_Read_And_Process()` 不只是“读寄存器”，还包含安装方向矩阵处理。

矩阵的定义、`orientIMU()`、`orientationMatrix` 的具体含义属于第20章。本章只提醒读者：读完后的 `rawAccel/rawGyro` 已经不是刚从寄存器直接拼出来的值，而是经过当前方向矩阵处理后的值。

这里还有一个容易被 `(float)` 强转误导的细节。源码中的 `straightAccelData`、`rotatedAccelData`、`straightGyroData`、`rotatedGyroData` 都是 `int16_t` 数组，`matrixMultiply()` 的入参和输出也都是 `int16_t`。因此当前方向矩阵处理是整数矩阵乘以整数向量，主要用于轴交换和符号变换；那些 `(float)rawAccel[...]` 赋值在落入 `int16_t` 数组时不会保留浮点小数精度。若未来要支持非整数安装误差校准矩阵，需要同时修改矩阵数据类型、乘法函数和后续缩放边界。

### 8.6.1 调试观测窗口

同一个传感器样本在函数内会经过多个可观察窗口，调试时不能把这些窗口混成一个“原始值”：

| 观测窗口 | 观测位置 | 能证明什么 | 不能证明什么 |
|---|---|---|---|
| `I2C2_Buffer_Rx[0..13]` | `MPU_Read_Reg()` 返回 `HAL_OK` 后、字节装载前 | 本次 14 字节突发读返回的原始字节序列。 | 函数返回后该局部数组已经不可直接观察；也不能证明下一帧仍是新样本。 |
| `rawAccel/rawGyro` 字节装载后 | `bytes[1]/bytes[0]` 赋值之后、`matrixMultiply()` 之前 | 高低字节是否按 `0x3B..0x48` 输出区顺序写入联合体。 | 这只是旋转前的临时全局值，不是主循环后续使用的最终 `rawAccel/rawGyro`。 |
| `straight*` 与 `rotated*` 数组 | 两次 `matrixMultiply()` 调用前后 | 方向矩阵对整数三轴向量做了怎样的轴交换或符号变换。 | 当前数组为 `int16_t`，不能证明存在浮点安装误差校准能力。 |
| 函数返回后的 `rawAccel/rawGyro` | `for (axis = 0; axis < 3; axis++)` 写回之后 | 主循环随后用于偏置扣除和比例换算的值。 | 不能再直接反推 I2C 缓冲区原始字节，除非同时记录旋转前窗口。 |
| `sensors.accel500Hz/gyro500Hz` | `main.c` 中偏置扣除和缩放之后 | 进入 AHRS 与控制链路的物理量入口。 | 不能证明传感器安装方向、标定参数或样本新鲜度已经正确【待验证】。 |

因此，第19章的调试建议应明确断点位置：要查 I2C 字节顺序，就停在字节装载前后；要查方向矩阵影响，就停在 `matrixMultiply()` 前后；要查 AHRS 输入，就看 `main.c` 写入 `sensors.*500Hz` 之后。只在函数返回后观察 `rawAccel/rawGyro`，会漏掉旋转前的证据。

### 8.7 构建产物证据边界

源码能说明“设计意图”，构建产物还能补一层“最终固件确实包含这些符号和调用”的证据。

`Debug/Three-axis_cloud_platformV2.map` 中可以看到：

- `.text.MPU6050_Read_And_Process` 进入最终镜像，符号地址为 `0x08002dd4`。
- `.text.MPU_Read_Reg`、`.text.HAL_I2C_Mem_Read` 和 `.text.matrixMultiply` 进入最终镜像。
- `.bss.frame_500Hz`、`.bss.rawAccel`、`.bss.rawGyro`、`.bss.rawMPU6050Temperature` 进入最终 RAM 布局。

`Debug/Three-axis_cloud_platformV2.list` 中可以进一步看到：

- `main()` 的 500Hz 分支检查并清除 `frame_500Hz`，随后通过 `bl 8002dd4 <MPU6050_Read_And_Process>` 调用连续读取函数。
- `main()` 在读取函数返回后继续读取 `rawAccel/rawGyro`，并写入 `sensors.accel500Hz` 和 `sensors.gyro500Hz`。
- `SysTick_Handler()` 中存在 `frame_500Hz = true` 的置位路径；注释块中的中断内读取逻辑没有成为当前主线调用。
- `MPU_Read_Reg()` 内部调用 `HAL_I2C_Mem_Read()`，`MPU6050_Read_And_Process()` 调用 `MPU_Read_Reg(REG_ACCEL_XOUT_H, I2C2_Buffer_Rx, MPU_DATA_LENGTH)`。
- `MPU6050_Read_And_Process()` 包含读取失败清零路径、`I2C2_Buffer_Rx[0..13]` 到加速度/温度/陀螺仪原始量的装载路径，以及两次 `matrixMultiply()` 调用。
- `MPU6050_ComputeStaticError()` 也调用 `MPU6050_Read_And_Process()`，但这是初始化/静态误差路径，不能替代主循环 500Hz 读取证据。

`Debug/Drivers/CustomDrivers/Src/mpu6050.su` 还能看到当前构建下 `MPU6050_Read_And_Process()`
的静态栈用量估算为 64 字节，`MPU_Read_Reg()` 为 32 字节，`matrixMultiply()` 为 24 字节；同目录
`.cyclo` 文件记录三者圈复杂度分别为 3、1 和 5。这里的 64 字节只覆盖
`MPU6050_Read_And_Process()` 自身栈帧，不能把被调用的 `MPU_Read_Reg()`、`HAL_I2C_Mem_Read()`
和 `matrixMultiply()` 自动折叠成“总栈峰值”。

HAL I2C 层的构建产物也给出边界：`Debug/Drivers/STM32F1xx_HAL_Driver/Src/stm32f1xx_hal_i2c.su`
记录 `HAL_I2C_Mem_Read()` 静态栈 56 字节，`.cyclo` 记录其圈复杂度 21。相比项目封装函数，
HAL 阻塞读路径包含更多状态检查、标志等待、超时和错误返回分支；这能解释为什么“项目层只是读 14 字节”
不等于“运行时只是一段固定耗时的数组拷贝”。

因此，`.map`、`.list`、`.su` 和 `.cyclo` 能证明当前构建包含 500Hz 标志、连续读取函数、HAL I2C
读取函数、方向矩阵函数、原始数据全局对象、主循环到连续读取函数的调用路径，以及本次 Debug 构建下的
静态栈和圈复杂度估算。它们不能证明 MPU6050 硬件一定响应、14 字节总线事务每次都成功、每帧样本一定新鲜、
500Hz 时序长期稳定，也不能证明 AHRS 或电机控制结果正确；静态栈估算也不能替代运行时最大栈水位。
这些运行时结论仍需日志、调试器、DWT、逻辑分析仪或实测记录；缺少证据时保持【待验证】。

### 8.8 官方资料、源码与构建产物分层

第19章容易把三类证据混成一句“项目连续读取 MPU6050 数据”。更严格的拆法是先看官方资料，再看源码，
最后看构建产物。

第一层，TDK/InvenSense `MPU-6000 and MPU-6050 Register Map and Descriptions, Revision 4.2`
只能证明器件输出寄存器的排列规则：`ACCEL_XOUT_H` 从 `0x3B` 开始，随后依次是加速度低字节、
Y/Z 轴、温度高低字节和陀螺仪 X/Y/Z 轴，直到 `GYRO_ZOUT_L=0x48`。这能支持“一次从
`0x3B` 连续读取 14 字节覆盖加速度、温度和陀螺仪输出区”的教材结论。TDK Product Center 的
MPU-6050 页面只能补充器件级背景，例如它是 6 轴 MotionTracking 器件并通过数字接口输出数据；
它不能替代寄存器表。

第二层，本仓库源码证明项目实际选择了这条读取方案：`mpu6050.h` 定义
`REG_ACCEL_XOUT_H=0x3B` 和 `MPU_DATA_LENGTH=14`，`mpu6050.c` 在
`MPU6050_Read_And_Process()` 中把这两个宏传给 `MPU_Read_Reg()`，再把 `I2C2_Buffer_Rx[0..13]`
装入 `rawAccel`、`rawMPU6050Temperature` 和 `rawGyro`。`Core/Src/main.c` 证明主循环在
`frame_500Hz` 分支中调用该函数，并在返回后继续把 `rawAccel/rawGyro` 写入 `sensors.*500Hz`。

第三层，当前 Debug 构建产物证明这些源码路径没有停留在文本层面：`.map` 中有
`.text.MPU6050_Read_And_Process`、`.text.MPU_Read_Reg` 和 `.text.HAL_I2C_Mem_Read`；
`.list` 中能看到 `main()` 跳转到 `0x08002dd4 <MPU6050_Read_And_Process>`，随后读取函数调用
`MPU_Read_Reg(REG_ACCEL_XOUT_H, I2C2_Buffer_Rx, MPU_DATA_LENGTH)`，并保留失败清零、14 字节装载和
两次 `matrixMultiply()` 调用路径。

这三层合起来能证明“本项目当前构建试图按 MPU6050 官方输出寄存器布局执行 14 字节突发读取”。
但它们仍不能证明“每次读取都成功、每帧都新鲜、读数与运动一致、500Hz 长期稳定或控制效果正确”。
这些结论必须继续依赖 `HAL_StatusTypeDef`、`hi2c2.ErrorCode`、连续日志、DWT 计时、逻辑分析仪、
样本时间戳或 Data Ready/FIFO 证据；当前缺失时统一保持【待验证】。

#### 8.8.1 同步机制缺失的可复核证据

这里的“缺失”不是凭经验判断，而是可以用源码搜索和构建产物边界复查：

| 同步机制 | 应出现的最小项目证据 | 当前仓库可见结果 | 教材结论边界 |
|---|---|---|---|
| Data Ready 轮询 | `INT_STATUS` 寄存器宏、`DATA_RDY_INT` 位判断，或读取 `INT_STATUS` 的代码 | 在 `Core/Src`、`Drivers/CustomDrivers/Src` 和 `Drivers/CustomDrivers/Inc` 中未找到 `INT_STATUS`、`DATA_READY` 或 `DATA_RDY` 项目定义/调用 | 不能说每次 14 字节读取前都等待了 MPU6050 新样本 |
| Data Ready 外部中断 | MPU6050 INT 引脚的 GPIO/EXTI 配置、对应 `EXTI*_IRQHandler` 或 HAL GPIO EXTI 回调中的采样触发 | 只看到 CubeMX/HAL 层通用 `EXTI` 能力和 `gpio.c` 注释，没有看到与 MPU6050 INT 引脚绑定的项目逻辑 | 不能把通用 EXTI 支持写成 MPU6050 已经采用中断同步 |
| FIFO 读取 | `FIFO_EN`、`USER_CTRL`、`FIFO_COUNTH/L`、`FIFO_R_W` 等寄存器宏和 FIFO 读写路径 | 当前 MPU6050 自定义驱动没有这些宏和读写路径，连续读取仍从 `REG_ACCEL_XOUT_H` 开始 | 不能说项目使用 FIFO 缓冲、按样本计数取数或避免轮询漏样 |
| 每帧样本序号 | 递增样本计数、每次突发读独立时间戳，或把 Data Ready/FIFO 状态随样本一起保存 | 当前主线保存 `dt500Hz` 和处理耗时，但 14 字节输出区本身没有项目级样本序号 | 不能仅凭两次 `dt500Hz` 接近 2ms 推出两帧必然对应两个不同传感器更新 |

因此，第19章应把“当前主线采用 500Hz 轮询突发读”与“严格按 MPU6050 数据就绪事件取样”分开讲。前者有 `frame_500Hz`、`MPU6050_Read_And_Process()`、`HAL_I2C_Mem_Read()` 和构建产物证据；后者在当前仓库中没有足够实现证据，缺少时应保持【待验证】。

## 9. 调试方法

本节按统一调试结构组织：现象 -> 可能原因 -> 定位方法 -> 验证步骤 -> 解决方案 -> 经验总结。对第19章而言，核心是把 500Hz 触发、主循环消费、I2C 14 字节读取、联合体装载、方向矩阵处理、时间预算和样本新鲜度分开验证。

### 9.1 现象与可能原因

常见现象包括 `frame_500Hz` 不触发、主循环没有及时消费标志、`MPU_Read_Reg()` 返回非 `HAL_OK`、14 字节缓冲区不随姿态变化、`rawAccel/rawGyro` 与直接拼接结果不一致、500Hz 执行时间接近预算，或连续两帧数据无法证明来自 MPU6050 新样本。可能原因包括 `systemReady/mpu6050Calibrating` 门控未满足、布尔帧标志积压、I2C BUSY/超时/NACK、方向矩阵写回改变了原始拼接值、阻塞式 I2C 占用时间过长，或当前没有 Data Ready/FIFO 同步证据。

### 9.2 定位方法：节拍触发和主循环消费

第一层，检查节拍是否触发。

- 在 `Core/Src/stm32f1xx_it.c` 中观察 `frameCounter` 是否递增。
- 确认 `systemReady == true` 且 `mpu6050Calibrating == false`。
- 确认 `COUNT_500HZ` 为 2，`frame_500Hz` 是否被周期性置位。

第二层，检查主循环是否消费标志。

- 在 `Core/Src/main.c` 的 `if (frame_500Hz)` 内设置断点。
- 确认进入后 `frame_500Hz` 被清为 `false`。
- 观察 `deltaTime500Hz` 和 `dt500Hz` 是否在合理范围内。
- 若 `dt500Hz` 经常明显大于 `0.002s`，说明主循环实际消费周期已经偏离目标 500Hz。
- 若怀疑优化导致主循环看不到中断置位，应检查 `frame_500Hz` 是否需要 `volatile`，
  或结合反汇编确认每轮循环确实从内存重新读取该标志。

### 9.3 验证步骤：I2C读取和联合体装载

第三层，检查 I2C 读取结果。

- 在 `MPU6050_Read_And_Process()` 中观察 `MPU_Read_Reg()` 返回值。
- 若返回非 `HAL_OK`，检查第14章 I2C 配置和第18章 MPU6050 初始化是否通过。
- 同时记录 `hi2c2.State` 和 `hi2c2.ErrorCode`，区分 BUSY、超时、NACK 等不同原因。
- 若返回成功，观察 `I2C2_Buffer_Rx[0..13]` 是否随姿态变化。

第四层，检查联合体装载。

- 观察 `rawAccel[XAXIS].bytes[1]` 和 `bytes[0]` 是否对应缓冲区高低字节。
- 观察 `rawAccel[XAXIS].value` 是否随 `bytes` 改变。
- 对 `rawGyro` 做同样检查。

### 9.4 验证步骤：方向矩阵、时间预算和样本新鲜度

第五层，检查方向矩阵影响。

- 如果 `rawAccel/rawGyro` 与缓冲区直接拼接结果不一致，先不要判定读取错误，因为函数末尾会经过 `orientationMatrix` 处理并写回。
- 方向矩阵细节留到第20章验证。

第六层，检查时间预算。

- 观察 `executionTime500Hz` 是否明显接近 2000 微秒。
- 用 DWT 或逻辑分析仪单独测量 `MPU6050_Read_And_Process()` 的耗时。
- 若总线读取已经接近 1.5ms，应继续确认姿态更新、PID 和 10Hz 打印是否挤占剩余预算。
- 异常路径要单独记录，因为 `MPU_Read_Reg()` 的超时参数是 100ms，一次总线卡死可能跨越多个 500Hz 周期。

第七层，检查样本新鲜度。

- 当前没有数据就绪中断同步，读取相位由主循环消费时刻决定。
- 可记录连续两帧 `I2C2_Buffer_Rx[0..13]` 是否变化，以及 `dt500Hz` 是否稳定。
- 若缺少连续日志，不能断言 500Hz 每帧都取得全新传感器样本。
- 若要证明样本严格来自新一次 MPU6050 更新，应增加 `INT_STATUS` Data Ready 读取、外部中断触发记录、FIFO 读写记录或逻辑分析仪证据；当前仓库没有这些证据。

### 9.5 解决方案：调试记录

- 记录 500Hz 触发标志、主循环消费位置、I2C 读取返回值和 14 字节缓冲区快照。
- 记录 `frame_500Hz` 是否只是布尔积压标志，避免把它误读成帧计数器。
- 对联合体装载，应记录高低字节来源、`bytes[]` 顺序、`value` 结果和轴名。
- 读取失败记录应包含 HAL 状态、清零变量、后续数据流影响和是否触发保护路径。
- 若读取函数失败但 `main.c` 继续运行，应追踪 `sensors.accel500Hz`、`sensors.gyro500Hz`、
  `MargAHRSupdate()` 和 `computeMotorCommands()` 是否继续使用清零后的输入。
- 进一步追踪失败传播时，应同时记录 `accelTCBias`、`gyroRTBias`、`gyroTCBias`、换算后的 `sensors.*500Hz`、
  `MargAHRSupdate()` 内部 `norm`，以及 `norm > 0.1f` 分支是否进入。
- 若没有真实姿态动作、串口日志或调试截图，数据随运动变化的结论保持【待验证】。
- 对性能记录，应同时记录 I2C2 100kHz 配置、理论总线时间下限、单函数实测耗时和整帧 `executionTime500Hz`。
- 对样本记录，应区分 MPU6050 寄存器更新频率、主循环读取频率和 AHRS 实际消费频率。

### 9.6 经验总结

第19章的调试结论应按证据层级收敛：`frame_500Hz` 只能证明软件调度触发，`MPU_Read_Reg()` 返回 `HAL_OK` 只能证明一次 HAL 读事务成功，`rawAccel/rawGyro` 更新只能证明读取函数留下了新对象状态；样本是否全新、方向是否正确、AHRS 是否得到可信输入，还需要 Data Ready/FIFO、连续日志、方向矩阵和后续章节证据支撑。缺少运行记录时，数据新鲜度和真实姿态响应保持【待验证】。

## 10. 常见问题

问题一：为什么不直接在 `SysTick_Handler()` 里读取 MPU6050。

因为 I2C 连续读取是耗时操作。
源码注释已经说明，在中断中读 I2C 并且主函数也读，会造成总线冲突和读数为 0。
当前主线采用中断置位、主循环读取。
这也和第09章的结论一致：SysTick 负责产生节拍标志，耗时业务留给主循环消费。

问题二：`accelData500Hz` 和 `gyroData500Hz` 是否是当前主线数据来源。

不是。它们有声明和定义，但 `stm32f1xx_it.c` 中给它们赋值的代码处于注释块内。
当前主线数据来源是 `rawAccel/rawGyro`，
随后在 `main.c` 中写入 `sensors.accel500Hz` 和 `sensors.gyro500Hz`。

问题三：为什么读取 14 字节，而不是只读加速度和陀螺仪的 12 字节。

MPU6050 从 `REG_ACCEL_XOUT_H` 开始的连续区域中间包含温度寄存器。项目一次读取 14 字节，把加速度、温度和陀螺仪都带出来。温度后续可用于温度补偿章节。

问题三补充：为什么不分多次读每个轴。

分多次读会增加 I2C 起始、地址和寄存器地址阶段的开销，也会扩大各轴读数之间的时间间隔。
一次从 `ACCEL_XOUT_H` 开始连续读 14 字节，更符合 MPU6050 输出寄存器连续排列的设计。

问题四：源码注释说加速度 X/Y 有映射，但代码看起来没有交换，应该相信哪个。

应以实际赋值代码为准。当前代码将 `[0..1]` 赋给 `rawAccel[XAXIS]`，将 `[2..3]` 赋给 `rawAccel[YAXIS]`。旧注释与代码不一致，应记录为可读性问题。

问题五：`MPU6050_Read_And_Process()` 是否只做连续读取。

不是。它还调用 `matrixMultiply()` 做方向矩阵处理，并把处理后的结果写回 `rawAccel/rawGyro`。本章只记录这个边界，矩阵含义留到第20章。

问题六：连续读取成功是否就能证明传感器方向和物理单位正确。

不能。
连续读取只证明 I2C 读到了 14 字节并完成了解析入口。
方向是否符合云台控制坐标，要看第20章的方向矩阵。
物理单位是否正确，要看第21章的零偏扣除和缩放因子。
因此本章的结论边界是“读到并解析原始数据”，不是“控制输入已经可信”。

问题七：第18章说寄存器更新基准约 800Hz，本章为什么还是 500Hz 读取。

因为这是两个不同层次的频率。
800Hz 来自 MPU6050 当前 DLPF 和分频寄存器组合的寄存器更新推导。
500Hz 来自项目 `SysTick_Handler()` 每两个 1ms Tick 置位一次 `frame_500Hz`。
当前项目没有使用 MPU6050 数据就绪中断做严格同步，所以实际样本相位和是否丢样仍需【待验证】。

问题八：读取失败时清零是否一定比保留上一帧更好。

不能简单判断。
清零能让故障更显眼，避免旧数据伪装成正常输入。
但加速度或角速度突然归零也可能冲击姿态和控制算法。
保留上一帧能减少瞬态跳变，却可能掩盖传感器失联。
当前源码选择清零，本章只解释其后果，不把它评判为唯一正确方案。

问题八补充：清零原始量是否等于 AHRS 输入全为 0。

不一定。`main.c` 会在清零后的 `rawAccel/rawGyro` 上继续扣除偏置并乘以缩放因子。
因此失败路径传给 AHRS 的是偏置派生值：

```text
accel500Hz[i] = -accelTCBias[i] * ACCEL_SCALE_FACTOR
gyro500Hz[i]  = -(gyroRTBias[i] + gyroTCBias[i]) * GYRO_SCALE_FACTOR
```

这些值通常是有限数值，不会被 `MargAHRSupdate()` 的 NaN/Inf 清理逻辑直接归零。
是否参与加速度校正，还要看 `sqrt(ax*ax + ay*ay + az*az)` 是否大于 `0.1f`。

问题九：`frame_500Hz` 为什么最好不要理解为“每一帧都排队等待处理”。

因为它是 `bool`，不是递增计数器。
中断把它置为 `true`，主循环把它清为 `false`。
如果主循环处理过慢，中间多次置位不会累加成多个待处理事件。
所以它只能提示“至少有一次 500Hz 任务到期”，不能证明没有错过帧。
丢帧或超时应通过 `dt500Hz`、`executionTime500Hz` 和外部测量继续验证。

问题十：TIM6 初始化是否说明 500Hz 来自 TIM6 中断。

不能这样说。
当前第19章的 500Hz 证据在 `SysTick_Handler()`：`frameCounter % COUNT_500HZ == 0`
时置位 `frame_500Hz`。
当前 `main.c` 虽然在进入主循环前清零 `htim6`，但 `micros()` 函数体实际读取的是
`HAL_GetTick()` 和 `SysTick->VAL`，没有读取 `TIM6->CNT`。因此教材应写成“SysTick 分频触发
500Hz 标志，当前 `micros()` 由 HAL Tick 和 SysTick 当前计数共同估算”，而不是写成
“TIM6 产生 500Hz 读取中断”或“TIM6 直接支撑当前微秒时间戳”。

问题十一：为什么 `frame_500Hz` 的 `volatile` 问题只记录，不直接修改代码。

因为本轮目标是教材深度复查，不修改固件行为。
从代码审查角度，跨中断共享变量通常应考虑 `volatile` 或其他同步方式。
但是否在当前编译优化等级下形成实际故障，需要查看编译选项、反汇编和运行现象。
缺少这些证据时，本章把它标为工程风险和【待验证】项。

问题十二：连续读取成功是否等价于“样本一定新鲜”。

不等价。
连续读取成功只能证明主机从 `0x3B` 开始拿到了 14 个字节，并按当前代码完成了解析。
它不能单独证明本次数据相对于上一帧一定更新，也不能证明读取相位与 MPU6050 内部数据更新严格对齐。
当前项目没有 Data Ready 中断、没有 FIFO 计数、没有样本序号，也没有给每个突发读附独立时间戳。
所以样本新鲜度必须靠连续日志、Data Ready/FIFO 证据或外部测量继续验证。

## 13. 本章总结

本章完成了 MPU6050 连续读取主线分析。

项目不是在中断中直接读 MPU6050。
当前路径是在 `SysTick_Handler()` 中根据 `COUNT_500HZ` 置位 `frame_500Hz`，
然后由 `Core/Src/main.c` 主循环调用 `MPU6050_Read_And_Process()`。
读取函数从 `REG_ACCEL_XOUT_H` 开始连续读取 `MPU_DATA_LENGTH=14` 字节，
并将数据装入 `rawAccel`、`rawMPU6050Temperature`、`rawGyro`。

本章已经证明：

- 当前 500Hz 入口由 `Core/Src/stm32f1xx_it.c` 置位，主循环消费。
- 当前 I2C 连续读取在 `Drivers/CustomDrivers/Src/mpu6050.c` 中执行。
- 当前主线不使用中断注释块中的 `accelData500Hz/gyroData500Hz` 复制逻辑。
- `MPU6050_Read_And_Process()` 读取后还会进行方向矩阵处理，因此 `rawAccel/rawGyro` 是“读取并初步方向处理后的原始量”。
- `I2C2_Buffer_Rx[14]` 是读取函数内部的局部缓冲区，函数返回后主线依赖全局 `rawAccel/rawGyro` 承接结果。
- 当前 `.map/.list` 能证明连续读取函数、HAL I2C 读取函数、方向矩阵函数、原始数据对象和主循环调用路径已经进入最终构建。
- 当前 `.su/.cyclo` 能补充 `MPU6050_Read_And_Process()`、`MPU_Read_Reg()`、`matrixMultiply()`
  和 `HAL_I2C_Mem_Read()` 的静态栈用量与圈复杂度边界，但不能替代调用链叠加、运行时栈水位或总线耗时实测。
- MPU6050 输出寄存器从 `0x3B` 到 `0x48` 连续排列，项目一次读取 14 字节符合寄存器布局。
- 温度字节进入 `rawMPU6050Temperature`，并可被 `MPU6050_ComputeTemError()` 用于温补偏置计算；但当前 500Hz 主循环没有像消费 `rawAccel/rawGyro` 那样每帧直接消费温度样本。
- 官方寄存器表、项目源码和 Debug 构建产物只能共同证明“当前固件尝试按官方输出区布局执行突发读取”，不能替代硬件响应、样本新鲜度和控制效果实测。
- 当前 100kHz I2C 下 14 字节连续读的理论总线时间下限约为 1.53ms，真实耗时仍需【待验证】。
- 当前项目以 500Hz 主循环读取约 800Hz 寄存器更新流，但没有数据就绪中断同步证据。
- 当前项目没有 Data Ready、FIFO、样本序号或每次突发读时间戳证据，不能仅凭 14 字节读取成功断言每帧样本新鲜。
- `frame_500Hz` 是布尔触发标志，不是积压帧计数器；处理超时会体现在 `dt500Hz` 变大，
  但不会自动补回中间没有读取的传感器样本。
- 当前 `frame_500Hz` 是跨中断共享的普通 `bool`，从代码审查角度应记录 `volatile` 或同步机制风险；
  当前编译产物是否受影响仍需【待验证】。
- 当前 `micros()` 实现读取 `HAL_GetTick()` 和 `SysTick->VAL`，不是读取 TIM6；TIM6 初始化和清零不能作为 500Hz 触发或当前微秒时间戳来源的直接证据。
- `HAL_I2C_Mem_Read()` 的 14 字节读取包含地址写、寄存器地址写、重复起始、地址读和连续接收，
  不是单纯的数组拷贝。
- `MPU_Read_Reg()` 的 100ms 超时远大于 2ms 控制帧，异常路径可能跨越多个 500Hz 周期。
- `MPU6050_Read_And_Process()` 只按 `HAL_OK` / 非 `HAL_OK` 折叠读取结果；要区分 `HAL_BUSY`、`HAL_TIMEOUT`、应答失败或总线错误，必须继续观察 `HAL_StatusTypeDef`、`hi2c2.State` 和 `hi2c2.ErrorCode`。
- 读取失败时当前函数清零 `rawAccel/rawGyro` 但不返回错误状态，主循环仍会继续运行后续姿态和控制路径。
- 当前矩阵处理链路使用 `int16_t` 矩阵和向量，适合解释轴交换/符号翻转，不能被写成浮点精密安装误差校准。
- 读取失败后传给 AHRS 的输入不一定全为零，因为主循环还会执行偏置扣除和比例换算；故障会变成一组偏置派生的有限数值。
- `MargAHRSupdate()` 会用 `norm > 0.1f` 决定是否进入加速度校正，但四元数积分和归一化路径仍会继续执行。

目标回扣：

- 本章已经把 `MPU6050连续数据读取` 接到第14章 I2C 和第18章寄存器初始化之后。
- 本章已经定位 500Hz 数据入口和 14 字节连续读取路径。
- 本章只闭合“读取与原始量承接”目标，不替代第20章到第23章的方向、缩放和姿态判断。

本章保留以下边界：

- 不展开 `orientationMatrix` 和 `matrixMultiply()` 的数学意义。
- 不展开温度补偿、静态零偏和物理量缩放。
- 不展开 `MargAHRSupdate()` 和后续姿态解算。
- 不把 `.su/.cyclo` 静态分析结果写成运行时栈水位、I2C 总线耗时或 500Hz 周期稳定性证明。
