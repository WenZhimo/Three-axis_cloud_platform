# 第19章 MPU6050连续读取

## 1. 本章目标

本章分析 MPU6050 初始化成功之后，项目如何把 IMU 原始数据带入 500Hz 实时控制主线。

本章目标是：

- 理解 `MPU6050连续数据读取` 在知识链中的位置：它依赖第14章 `I2C主机通信` 和第18章 `MPU6050寄存器配置`。
- 找到 500Hz 数据入口：`Core/Src/stm32f1xx_it.c` 中只置位 `frame_500Hz`，`Core/Src/main.c` 中才真正调用 `MPU6050_Read_And_Process()`。
- 理解 `MPU6050_Read_And_Process()` 从 `REG_ACCEL_XOUT_H` 开始连续读取 `MPU_DATA_LENGTH=14` 字节。
- 理解 14 字节如何分别进入加速度、温度和陀螺仪原始量。
- 明确本章边界：只讲连续读取和原始数据承接，不展开安装方向矩阵、温度补偿、物理量缩放、姿态解算和 PID。

本章知识链为：

`知识点总表` 中的 `MPU6050连续数据读取`
-> `知识依赖图` 中依赖 `MPU6050寄存器配置`、`I2C主机通信`
-> `学习优先级` 中 P2 项目关键知识
-> `教学顺序` 第31项
-> `教材章节` 第19章。

## 2. 前置知识

本章正式前置知识包括：

- `I2C主机通信`
- `MPU6050寄存器配置`

第14章已经解释 STM32 通过 I2C 访问从设备寄存器。第18章已经解释 MPU6050 初始化如何确认设备 ID、唤醒设备并写入采样率、DLPF 和量程寄存器。

本章还会继续使用第17章的 `16位数据联合体解析`，因为连续读取到的是字节流，而项目最终要得到 `int16_t` 原始值。

需要注意，本章不会展开第20章的 IMU 安装方向矩阵，也不会展开第21章的标定和物理量缩放。源码中的读取函数已经调用了矩阵旋转，但本章只把它作为调用边界记录，不解释矩阵含义。

## 3. 问题背景

MPU6050 初始化只是让芯片进入工作状态。云台控制真正需要的是周期性、稳定、可追踪的传感器输入。

项目面对三个工程问题：

- 什么时候读传感器。
- 在哪里读传感器。
- 读出的 14 字节如何进入项目数据结构。

如果在中断里直接执行 I2C 读取，可能拉长中断时间，并与主循环读取产生冲突。项目源码已经在 `Core/Src/stm32f1xx_it.c` 中留下说明：读 I2C 会耗时，并且中断中读、主函数中也读会造成总线冲突和读数为 0。因此项目采用“中断置位，主循环执行读取”的方式。

这就是本章的主线：中断产生节拍，主循环消费节拍，读取函数完成字节到原始量的转换。

## 4. 核心概念

### 4.1 500Hz帧触发标志

`Core/Inc/main.h` 定义：

```c
#define FRAME_COUNT   1000
#define COUNT_500HZ   2
```

`Core/Src/stm32f1xx_it.c` 的 `SysTick_Handler()` 中，当 `systemReady == true` 且 `mpu6050Calibrating == false` 时，`frameCounter` 递增。每当：

```c
frameCounter % COUNT_500HZ == 0
```

就设置：

```c
frame_500Hz = true;
```

这说明当前项目使用 SysTick 相关逻辑生成 500Hz 帧标志。该标志不是数据本身，而是通知主循环“可以执行一帧 500Hz 工作”。

### 4.2 主循环消费标志

`Core/Src/main.c` 的 `while (1)` 中检查：

```c
if (frame_500Hz)
{
    frame_500Hz = false;
    ...
    MPU6050_Read_And_Process();
}
```

主循环先清除标志，再计算 `dt500Hz`，然后调用 `MPU6050_Read_And_Process()`。

这是一种常见嵌入式结构：中断只做短操作，主循环做耗时操作。I2C 读取不放在 `SysTick_Handler()` 内部，可以降低中断阻塞风险。

这里还要拆出两个容易被忽略的实时语义。

第一，`frame_500Hz` 是布尔标志，不是事件计数器。若主循环在某一帧内处理过慢，SysTick
可能再次把它置为 `true`，但标志本身不会记录“积压了几帧”。因此它适合表达“至少有一帧待处理”，
不适合统计错过了多少个 500Hz 周期。是否发生丢帧，需要结合 `dt500Hz` 和
`executionTime500Hz` 观察。

第二，`frame_500Hz`、`systemReady`、`mpu6050Calibrating` 由中断和主循环共享，但当前
`main.c` / `main.h` 中 `frame_500Hz` 不是 `volatile` 声明。多数调试构建可能仍能按预期工作，
但从 C 语言和嵌入式并发边界看，跨中断共享标志通常应使用 `volatile` 或更明确的同步机制。
本轮不修改固件，只把它作为教材中的可见性风险和代码审查点记录；当前编译优化下是否触发问题仍需【待验证】。

还要分清 TIM6 和 500Hz 标志的角色。项目调用了 `MX_TIM6_Init()`，并在进入主循环前执行
`__HAL_TIM_SET_COUNTER(&htim6, 0)`，但第19章看到的 500Hz 触发证据来自
`SysTick_Handler()` 中的 `COUNT_500HZ` 分频。TIM6 更像 `micros()` 微秒计时链路的一部分，
不能把它直接写成“500Hz 中断源”。

### 4.3 14字节连续读取

`Drivers/CustomDrivers/Inc/mpu6050.h` 定义：

```c
#define REG_ACCEL_XOUT_H        0x3B
#define MPU_DATA_LENGTH         14
```

MPU6050 的连续输出区从加速度计 X 轴高字节开始，项目一次读取 14 字节：

- 加速度计 X/Y/Z：6 字节。
- 温度：2 字节。
- 陀螺仪 X/Y/Z：6 字节。

`MPU6050_Read_And_Process()` 调用：

```c
MPU_Read_Reg(REG_ACCEL_XOUT_H, I2C2_Buffer_Rx, MPU_DATA_LENGTH)
```

这就是本章正式知识点 `MPU6050连续数据读取` 的核心证据。

### 4.4 输出寄存器连续性

MPU6050 手册把加速度、温度、陀螺仪输出放在一段连续寄存器中：

| 十六进制地址 | 寄存器 | 缓冲区索引 | 数据含义 |
| --- | --- | --- | --- |
| `0x3B` | `ACCEL_XOUT_H` | `[0]` | X 轴加速度高字节 |
| `0x3C` | `ACCEL_XOUT_L` | `[1]` | X 轴加速度低字节 |
| `0x3D` | `ACCEL_YOUT_H` | `[2]` | Y 轴加速度高字节 |
| `0x3E` | `ACCEL_YOUT_L` | `[3]` | Y 轴加速度低字节 |
| `0x3F` | `ACCEL_ZOUT_H` | `[4]` | Z 轴加速度高字节 |
| `0x40` | `ACCEL_ZOUT_L` | `[5]` | Z 轴加速度低字节 |
| `0x41` | `TEMP_OUT_H` | `[6]` | 温度高字节 |
| `0x42` | `TEMP_OUT_L` | `[7]` | 温度低字节 |
| `0x43` | `GYRO_XOUT_H` | `[8]` | X 轴陀螺仪高字节 |
| `0x44` | `GYRO_XOUT_L` | `[9]` | X 轴陀螺仪低字节 |
| `0x45` | `GYRO_YOUT_H` | `[10]` | Y 轴陀螺仪高字节 |
| `0x46` | `GYRO_YOUT_L` | `[11]` | Y 轴陀螺仪低字节 |
| `0x47` | `GYRO_ZOUT_H` | `[12]` | Z 轴陀螺仪高字节 |
| `0x48` | `GYRO_ZOUT_L` | `[13]` | Z 轴陀螺仪低字节 |

这解释了为什么项目从 `0x3B` 开始一次读 14 字节，而不是分别读 6 次或 7 次。
连续读减少 I2C 事务次数，也降低不同轴来自不同读取时刻的风险。

连续读取的另一个意义是“寄存器地址自动递增”。主机只发送一次起始寄存器地址 `0x3B`，
后续每收到一个字节，MPU6050 内部读指针进入下一个输出寄存器。若把 14 字节拆成多次读取，
不仅总线开销更高，还会让加速度、温度和陀螺仪来自不同事务窗口；这些轴之间的同步程度会进一步下降。

### 4.5 高低字节合成

MPU6050 输出寄存器以高字节在前、低字节在后的方式排列。一个轴的 16 位有符号原始量可写成：

```text
raw = (int16_t)((high << 8) | low)
```

项目没有直接写这个表达式，而是通过联合体实现：

```c
rawAccel[XAXIS].bytes[1] = I2C2_Buffer_Rx[0];
rawAccel[XAXIS].bytes[0] = I2C2_Buffer_Rx[1];
```

这种写法依赖当前 Cortex-M3 小端存储约定：`bytes[0]` 对应低 8 位，`bytes[1]` 对应高 8 位。
所以本章要同时讲“传感器寄存器顺序”和“MCU 内存字节顺序”，不能只说“两个字节拼成一个 `int16_t`”。

## 5. 工作原理

第19章的数据路径可以拆成五步：

1. `SysTick_Handler()` 根据帧计数设置 `frame_500Hz`。
2. `main()` 主循环发现 `frame_500Hz == true` 后清除标志。
3. 主循环计算 `deltaTime500Hz` 和 `dt500Hz`。
4. 主循环调用 `MPU6050_Read_And_Process()` 执行 I2C 连续读取。
5. 读取函数把 14 字节写入 `rawAccel`、`rawMPU6050Temperature` 和 `rawGyro`。

读取函数内部又可以拆成四段：

- 使用 `MPU_Read_Reg()` 从 `REG_ACCEL_XOUT_H` 开始读取 14 字节。
- 如果读取失败，清零 `rawAccel` 和 `rawGyro` 并返回。
- 按高字节、低字节顺序填充联合体的 `bytes[1]` 和 `bytes[0]`。
- 把原始值送入方向矩阵处理，再写回 `rawAccel` 和 `rawGyro`。

本章重点是前三段。第四段涉及方向矩阵，属于第20章。

第18章已经推导出当前配置下 MPU6050 陀螺仪寄存器更新基准可按 800Hz 理解。
第19章看到的是主循环按 `COUNT_500HZ=2` 尝试以 500Hz 消费数据。

二者关系可以写成：

```text
传感器寄存器更新周期约为 1 / 800 = 1.25 ms
主循环读取周期目标约为 1 / 500 = 2.00 ms
每个主循环读取周期约跨过 800 / 500 = 1.6 个传感器更新周期
```

这意味着项目并不是读取每一个 MPU6050 内部更新样本，而是以 500Hz 从更高的寄存器更新流中取样。
但是当前源码没有使用 MPU6050 数据就绪中断，也没有读取中断状态寄存器来锁定采样相位。
因此“每次主循环读取是否严格对应全新样本”仍需运行日志或外部测量【待验证】。

## 6. STM32实现机制

STM32 侧的关键机制是：不要在中断里执行耗时 I2C 事务。

`SysTick_Handler()` 本身每 1 ms 进入一次。项目在其中维护 `frameCounter`，每两次置位一次 `frame_500Hz`。如果把 I2C 连续读取也放在中断里，会增加中断执行时间，并可能影响其他实时逻辑。

项目源码中的注释已经给出工程判断：中断中读 I2C 会耗时，且和主循环重复读取会造成总线冲突和读数为 0。因此当前实现把中断职责限制为置位标志。

从 I2C2 配置看，`Core/Src/i2c.c` 中 `hi2c2.Init.ClockSpeed = 100000`，也就是 100kHz 标准模式。
一次 14 字节连续读至少包括设备地址写、寄存器地址、设备地址读和 14 个数据字节。
按每字节 8 位数据加 1 位 ACK/NACK 粗略估算：

```text
(1 + 1 + 1 + 14) * 9 = 153 个 SCL 周期
153 / 100000 = 1.53 ms
```

这个 1.53ms 是总线位传输理论下限，不包含起始/停止条件、HAL 轮询等待、函数调用、上升沿余量和器件响应差异。
它说明 14 字节阻塞式 I2C 读取已经占用 2ms 控制帧的大部分预算，必须用 `executionTime500Hz`、DWT 或逻辑分析仪实测。

HAL 层还可以继续拆解。`MPU_Read_Reg()` 内部调用：

```c
HAL_I2C_Mem_Read(&hi2c2, MPU6050_I2C_ADDR, REG_ACCEL_XOUT_H,
                 I2C_MEMADD_SIZE_8BIT, I2C2_Buffer_Rx, 14, 100);
```

在 `STM32F1xx HAL I2C` 阻塞式实现中，`HAL_I2C_Mem_Read()` 不是简单“读 14 个字节”。
它会先等待总线空闲，锁定 I2C 句柄，把状态置为 `HAL_I2C_STATE_BUSY_RX`，然后执行内存读事务：

```text
START
7位设备地址 + Write
8位寄存器地址 0x3B
RESTART
7位设备地址 + Read
连续接收 14 字节
STOP
```

这解释了为什么同一个 I2C 句柄不适合在 SysTick 和主循环中同时访问。HAL 句柄内部有
`State`、`Mode`、`ErrorCode`、`XferSize` 等状态，阻塞式读期间它处于忙状态；重入访问要么被拒绝，
要么造成总线时序和数据流混乱。

还要注意超时参数。`MPU_Read_Reg()` 传入的超时是 `100 ms`，而 500Hz 控制帧目标周期是
`2 ms`。一旦总线异常导致阻塞等待接近超时，单次读取失败就可能跨过许多个 500Hz 帧。
这不是稳态性能，而是异常路径性能；但对“系统失联后是否还能及时保护”很重要，当前仓库没有实测保护记录，
应标记为【待验证】。

主循环拿到标志后执行：

- 时间戳测量：`currentTime = micros()`。
- 周期计算：`dt500Hz = deltaTime500Hz / 1000000.0f`。
- I2C 连续读取：`MPU6050_Read_And_Process()`。
- 后续处理：写入 `sensors.accel500Hz`、`sensors.gyro500Hz` 并调用姿态更新。

后续处理虽然出现在同一个 500Hz 块中，但不属于本章重点。第19章只确认原始数据如何进入这个块。

因此，本章的 STM32 结论不是“当前 500Hz 一定稳定成立”，而是：

- 源码意图是 SysTick 分频生成 500Hz 工作标志。
- 实际读取使用阻塞式 HAL I2C Mem 接口。
- 100kHz I2C 的 14 字节连续读理论总线时间已经接近 2ms 帧预算。
- 稳态是否守住 500Hz、异常时是否超时拖垮主循环，需要 `executionTime500Hz`、DWT 或逻辑分析仪证明。

## 7. 项目中的应用

本章涉及的项目证据文件包括：

- `Drivers/CustomDrivers/Src/mpu6050.c`
- `Drivers/CustomDrivers/Inc/mpu6050.h`
- `Core/Src/main.c`
- `Core/Src/stm32f1xx_it.c`
- `Core/Inc/main.h`

`Core/Inc/main.h` 提供 `FRAME_COUNT`、`COUNT_500HZ` 和 `frame_500Hz` 的声明。

`Core/Src/stm32f1xx_it.c` 在 `SysTick_Handler()` 中置位 `frame_500Hz`。

`Core/Src/main.c` 在主循环中清除该标志并调用读取函数。

`Drivers/CustomDrivers/Src/mpu6050.c` 中的 `MPU6050_Read_And_Process()` 是连续读取的实际执行点。

它先读取 14 字节缓冲区 `I2C2_Buffer_Rx[14]`。

随后函数填充 `rawAccel[3]`、`rawMPU6050Temperature`、`rawGyro[3]`。

读取完成后，`Core/Src/main.c` 立刻使用 `rawAccel` 和 `rawGyro` 计算 `sensors.accel500Hz` 和 `sensors.gyro500Hz`。这说明第19章是第21章物理量缩放和第23章姿态解算的输入前置。

## 8. 代码分析

### 8.1 SysTick只置位frame_500Hz

`Core/Src/stm32f1xx_it.c` 中的逻辑可以概括为：

```c
if ((systemReady == true) && (mpu6050Calibrating == false))
{
    frameCounter++;
    if ((frameCounter % COUNT_500HZ) == 0)
    {
        frame_500Hz = true;
    }
}
```

源码中原本有一段在中断里调用 `MPU6050_Read_And_Process()` 并复制 `accelData500Hz`、`gyroData500Hz` 的代码，但已经被注释掉。注释说明原因是 I2C 耗时，并且中断和主函数重复读取会造成总线冲突。

因此，`accelData500Hz` 和 `gyroData500Hz` 虽然在 `mpu6050.h` 中声明、在 `mpu6050.c` 中定义，但当前主线并未使用中断中的复制逻辑。教材不能把它们写成当前 500Hz 实时控制循环的数据来源。

`frame_500Hz` 还有一个隐藏语义：它只能表示“有一帧工作待主循环处理”，不能表示“有多少帧工作待处理”。
如果主循环某次执行超过 2ms，下一次 SysTick 到来时仍只是把这个布尔值设为 `true`。
等主循环重新进入 `if (frame_500Hz)` 时，它只会处理一次读取和一次姿态/控制更新。

因此，`dt500Hz` 能反映“这次处理距离上次处理过了多久”，但不能补回中间没有读取到的传感器样本。
例如一次异常耗时让两次处理间隔变成 4ms，`dt500Hz` 可以变大到约 `0.004s`，
后续积分可按较大的时间步长计算；但这仍然只有一次 I2C 读取，不等于补采了两帧 MPU6050 数据。

跨中断共享标志还涉及编译器可见性。当前 `frame_500Hz` 在 `main.c` 中定义为普通 `bool`，
在 `main.h` 中也以普通 `extern bool` 声明。严格的代码审查会把它标记为应考虑 `volatile` 的共享变量。
本教材只记录风险，不修改源码；若要证明当前编译产物无该问题，需要结合优化等级和反汇编【待验证】。

### 8.2 主循环计算dt并读取

`Core/Src/main.c` 在 500Hz 块中做三件与本章相关的事：

```c
currentTime = micros();
deltaTime500Hz = currentTime - previous500HzTime;
dt500Hz = (float)deltaTime500Hz / 1000000.0f;
MPU6050_Read_And_Process();
```

`dt500Hz` 后续会给姿态算法和控制算法使用，但本章不展开积分和控制意义。这里需要抓住一点：读取函数处在 500Hz 主循环块的开头，原始传感器数据是本帧后续处理的基础。

从时间语义看，本章要区分三种“时间”：

- `COUNT_500HZ` 期望周期：由 SysTick 分频得到，目标约 `2ms`。
- `dt500Hz` 实际周期：由 `micros()` 差值计算，反映主循环实际消费间隔。
- 传感器样本时间：由 MPU6050 内部寄存器更新和主循环读取相位共同决定，当前没有数据就绪中断证据。

这三个时间接近时，系统表现可能接近理想 500Hz；三者分离时，姿态积分、滤波和 PID 都会受到影响。
当前仓库只能证明软件计算了 `dt500Hz`，不能证明实际样本相位稳定。

### 8.3 I2C连续读取失败处理

`MPU6050_Read_And_Process()` 内部先定义：

```c
uint8_t I2C2_Buffer_Rx[14];
```

然后执行：

```c
if (MPU_Read_Reg(REG_ACCEL_XOUT_H, I2C2_Buffer_Rx, MPU_DATA_LENGTH) != HAL_OK)
{
    memset(rawAccel, 0, sizeof(rawAccel));
    memset(rawGyro, 0, sizeof(rawGyro));
    return;
}
```

这说明读取失败时，项目没有沿用上一帧数据，而是清零加速度和陀螺仪原始值。这个行为会影响后续姿态算法和控制输出，调试时必须注意。

还要注意一个细节：失败路径只清零 `rawAccel` 和 `rawGyro`，没有清零 `rawMPU6050Temperature`。
如果后续流程在失败后继续使用温度值，那么它可能是上一帧或默认值。
当前 500Hz 主循环不直接调用 `MPU6050_ComputeTemError()`，但静态误差估计流程会调用温度补偿函数。
因此读取失败与温度数据新鲜度之间的关系应作为调试记录项，而不能默认温度同步更新。

失败清零的传播路径也要拆开看。`MPU6050_Read_And_Process()` 自身没有返回 `bool` 或错误码，
所以 `main.c` 无法在本帧直接判断“本次读取失败”。它会继续执行：

- 用清零后的 `rawAccel/rawGyro` 计算 `sensors.accel500Hz` 和 `sensors.gyro500Hz`。
- 调用 `MargAHRSupdate()`。
- 在启动收敛后调用 `computeMotorCommands(dt500Hz)`。

这意味着读取失败在当前主线中会被转换为一组看似合法的数值输入，而不是显式故障状态。
清零让问题更容易在数据上暴露，但也可能把一次通信故障变成姿态和控制输入突变。
是否需要故障计数、保留上一帧、进入保护状态或禁用电机，属于后续安全策略设计；
当前仓库没有这类闭环保护证据，保持【待验证】。

### 8.4 加速度和温度字节装载

读取成功后，项目按高字节、低字节填入联合体：

```c
rawAccel[XAXIS].bytes[1] = I2C2_Buffer_Rx[0];
rawAccel[XAXIS].bytes[0] = I2C2_Buffer_Rx[1];
```

加速度三轴对应：

- `I2C2_Buffer_Rx[0..1]` -> `rawAccel[XAXIS]`
- `I2C2_Buffer_Rx[2..3]` -> `rawAccel[YAXIS]`
- `I2C2_Buffer_Rx[4..5]` -> `rawAccel[ZAXIS]`

温度对应：

- `I2C2_Buffer_Rx[6..7]` -> `rawMPU6050Temperature`

需要注意，源码注释中有一处旧说明写着 `Buffer[0,1]` 赋给 `rawAccel[YAXIS]`，但实际代码赋给的是 `rawAccel[XAXIS]`。教材应以实际赋值代码为准，并把注释不一致作为待清理的可读性问题记录。

按统一公式看，X 轴加速度当前等价于：

```text
rawAccel[XAXIS].value =
    (int16_t)((I2C2_Buffer_Rx[0] << 8) | I2C2_Buffer_Rx[1])
```

Y、Z 轴同理，只是缓冲区索引分别后移 2 字节和 4 字节。
这一步仍然是“原始计数值”，不是 `g`、`m/s^2` 或姿态角。

### 8.5 陀螺仪字节装载

陀螺仪三轴对应：

- `I2C2_Buffer_Rx[8..9]` -> `rawGyro[XAXIS]`
- `I2C2_Buffer_Rx[10..11]` -> `rawGyro[YAXIS]`
- `I2C2_Buffer_Rx[12..13]` -> `rawGyro[ZAXIS]`

源码注释特别强调：这里严格按物理轴 X、Y、Z 提取，不在这里擅自映射为 Roll/Pitch。
后续 `rawGyro[ROLL]`、`rawGyro[PITCH]`、`rawGyro[YAW]` 的使用依赖宏定义中
`ROLL=0`、`PITCH=1`、`YAW=2` 与三轴索引的对应关系。

### 8.6 方向矩阵调用边界

字节装载后，函数把 `rawAccel`、`rawGyro` 的 `value` 放入 `straightAccelData`、`straightGyroData`，然后调用：

```c
matrixMultiply(3, 3, 1, rotatedAccelData, orientationMatrix, straightAccelData);
matrixMultiply(3, 3, 1, rotatedGyroData, orientationMatrix, straightGyroData);
```

随后把旋转后的结果写回 `rawAccel` 和 `rawGyro`。

这说明 `MPU6050_Read_And_Process()` 不只是“读寄存器”，还包含安装方向矩阵处理。

矩阵的定义、`orientIMU()`、`orientationMatrix` 的具体含义属于第20章。本章只提醒读者：读完后的 `rawAccel/rawGyro` 已经不是刚从寄存器直接拼出来的值，而是经过当前方向矩阵处理后的值。

### 本节证据边界

## 9. 调试方法

调试连续读取时，建议分层检查。

第一层，检查节拍是否触发。

- 在 `Core/Src/stm32f1xx_it.c` 中观察 `frameCounter` 是否递增。
- 确认 `systemReady == true` 且 `mpu6050Calibrating == false`。
- 确认 `COUNT_500HZ` 为 2，`frame_500Hz` 是否被周期性置位。

第二层，检查主循环是否消费标志。

- 在 `Core/Src/main.c` 的 `if (frame_500Hz)` 内设置断点。
- 确认进入后 `frame_500Hz` 被清为 `false`。
- 观察 `deltaTime500Hz` 和 `dt500Hz` 是否在合理范围内。
- 若 `dt500Hz` 经常明显大于 `0.002s`，说明主循环实际消费周期已经偏离目标 500Hz。
- 若怀疑优化导致主循环看不到中断置位，应检查 `frame_500Hz` 是否需要 `volatile`，
  或结合反汇编确认每轮循环确实从内存重新读取该标志。

第三层，检查 I2C 读取结果。

- 在 `MPU6050_Read_And_Process()` 中观察 `MPU_Read_Reg()` 返回值。
- 若返回非 `HAL_OK`，检查第14章 I2C 配置和第18章 MPU6050 初始化是否通过。
- 同时记录 `hi2c2.State` 和 `hi2c2.ErrorCode`，区分 BUSY、超时、NACK 等不同原因。
- 若返回成功，观察 `I2C2_Buffer_Rx[0..13]` 是否随姿态变化。

第四层，检查联合体装载。

- 观察 `rawAccel[XAXIS].bytes[1]` 和 `bytes[0]` 是否对应缓冲区高低字节。
- 观察 `rawAccel[XAXIS].value` 是否随 `bytes` 改变。
- 对 `rawGyro` 做同样检查。

第五层，检查方向矩阵影响。

- 如果 `rawAccel/rawGyro` 与缓冲区直接拼接结果不一致，先不要判定读取错误，因为函数末尾会经过 `orientationMatrix` 处理并写回。
- 方向矩阵细节留到第20章验证。

第六层，检查时间预算。

- 观察 `executionTime500Hz` 是否明显接近 2000 微秒。
- 用 DWT 或逻辑分析仪单独测量 `MPU6050_Read_And_Process()` 的耗时。
- 若总线读取已经接近 1.5ms，应继续确认姿态更新、PID 和 10Hz 打印是否挤占剩余预算。
- 异常路径要单独记录，因为 `MPU_Read_Reg()` 的超时参数是 100ms，一次总线卡死可能跨越多个 500Hz 周期。

第七层，检查样本新鲜度。

- 当前没有数据就绪中断同步，读取相位由主循环消费时刻决定。
- 可记录连续两帧 `I2C2_Buffer_Rx[0..13]` 是否变化，以及 `dt500Hz` 是否稳定。
- 若缺少连续日志，不能断言 500Hz 每帧都取得全新传感器样本。

调试记录建议：

- 记录 500Hz 触发标志、主循环消费位置、I2C 读取返回值和 14 字节缓冲区快照。
- 记录 `frame_500Hz` 是否只是布尔积压标志，避免把它误读成帧计数器。
- 对联合体装载，应记录高低字节来源、`bytes[]` 顺序、`value` 结果和轴名。
- 读取失败记录应包含 HAL 状态、清零变量、后续数据流影响和是否触发保护路径。
- 若读取函数失败但 `main.c` 继续运行，应追踪 `sensors.accel500Hz`、`sensors.gyro500Hz`、
  `MargAHRSupdate()` 和 `computeMotorCommands()` 是否继续使用清零后的输入。
- 若没有真实姿态动作、串口日志或调试截图，数据随运动变化的结论保持【待验证】。
- 对性能记录，应同时记录 I2C2 100kHz 配置、理论总线时间下限、单函数实测耗时和整帧 `executionTime500Hz`。
- 对样本记录，应区分 MPU6050 寄存器更新频率、主循环读取频率和 AHRS 实际消费频率。

## 10. 常见问题

问题一：为什么不直接在 `SysTick_Handler()` 里读取 MPU6050。

因为 I2C 连续读取是耗时操作。
源码注释已经说明，在中断中读 I2C 并且主函数也读，会造成总线冲突和读数为 0。
当前主线采用中断置位、主循环读取。
这也和第09章的结论一致：SysTick 负责产生节拍标志，耗时业务留给主循环消费。

问题二：`accelData500Hz` 和 `gyroData500Hz` 是否是当前主线数据来源。

不是。它们有声明和定义，但 `stm32f1xx_it.c` 中给它们赋值的代码处于注释块内。
当前主线数据来源是 `rawAccel/rawGyro`，
随后在 `main.c` 中写入 `sensors.accel500Hz` 和 `sensors.gyro500Hz`。

问题三：为什么读取 14 字节，而不是只读加速度和陀螺仪的 12 字节。

MPU6050 从 `REG_ACCEL_XOUT_H` 开始的连续区域中间包含温度寄存器。项目一次读取 14 字节，把加速度、温度和陀螺仪都带出来。温度后续可用于温度补偿章节。

问题三补充：为什么不分多次读每个轴。

分多次读会增加 I2C 起始、地址和寄存器地址阶段的开销，也会扩大各轴读数之间的时间间隔。
一次从 `ACCEL_XOUT_H` 开始连续读 14 字节，更符合 MPU6050 输出寄存器连续排列的设计。

问题四：源码注释说加速度 X/Y 有映射，但代码看起来没有交换，应该相信哪个。

应以实际赋值代码为准。当前代码将 `[0..1]` 赋给 `rawAccel[XAXIS]`，将 `[2..3]` 赋给 `rawAccel[YAXIS]`。旧注释与代码不一致，应记录为可读性问题。

问题五：`MPU6050_Read_And_Process()` 是否只做连续读取。

不是。它还调用 `matrixMultiply()` 做方向矩阵处理，并把处理后的结果写回 `rawAccel/rawGyro`。本章只记录这个边界，矩阵含义留到第20章。

问题六：连续读取成功是否就能证明传感器方向和物理单位正确。

不能。
连续读取只证明 I2C 读到了 14 字节并完成了解析入口。
方向是否符合云台控制坐标，要看第20章的方向矩阵。
物理单位是否正确，要看第21章的零偏扣除和缩放因子。
因此本章的结论边界是“读到并解析原始数据”，不是“控制输入已经可信”。

问题七：第18章说寄存器更新基准约 800Hz，本章为什么还是 500Hz 读取。

因为这是两个不同层次的频率。
800Hz 来自 MPU6050 当前 DLPF 和分频寄存器组合的寄存器更新推导。
500Hz 来自项目 `SysTick_Handler()` 每两个 1ms Tick 置位一次 `frame_500Hz`。
当前项目没有使用 MPU6050 数据就绪中断做严格同步，所以实际样本相位和是否丢样仍需【待验证】。

问题八：读取失败时清零是否一定比保留上一帧更好。

不能简单判断。
清零能让故障更显眼，避免旧数据伪装成正常输入。
但加速度或角速度突然归零也可能冲击姿态和控制算法。
保留上一帧能减少瞬态跳变，却可能掩盖传感器失联。
当前源码选择清零，本章只解释其后果，不把它评判为唯一正确方案。

问题九：`frame_500Hz` 为什么最好不要理解为“每一帧都排队等待处理”。

因为它是 `bool`，不是递增计数器。
中断把它置为 `true`，主循环把它清为 `false`。
如果主循环处理过慢，中间多次置位不会累加成多个待处理事件。
所以它只能提示“至少有一次 500Hz 任务到期”，不能证明没有错过帧。
丢帧或超时应通过 `dt500Hz`、`executionTime500Hz` 和外部测量继续验证。

问题十：TIM6 初始化是否说明 500Hz 来自 TIM6 中断。

不能这样说。
当前第19章的 500Hz 证据在 `SysTick_Handler()`：`frameCounter % COUNT_500HZ == 0`
时置位 `frame_500Hz`。
TIM6 在本项目里更直接的证据是用于 `micros()` 相关微秒计时，`main.c` 进入主循环前清零 TIM6 计数器。
因此教材应写成“SysTick 分频触发 500Hz 标志，TIM6 支撑微秒时间测量”，而不是写成
“TIM6 产生 500Hz 读取中断”。

问题十一：为什么 `frame_500Hz` 的 `volatile` 问题只记录，不直接修改代码。

因为本轮目标是教材深度复查，不修改固件行为。
从代码审查角度，跨中断共享变量通常应考虑 `volatile` 或其他同步方式。
但是否在当前编译优化等级下形成实际故障，需要查看编译选项、反汇编和运行现象。
缺少这些证据时，本章把它标为工程风险和【待验证】项。

## 13. 本章总结

本章完成了 MPU6050 连续读取主线分析。

项目不是在中断中直接读 MPU6050。
当前路径是在 `SysTick_Handler()` 中根据 `COUNT_500HZ` 置位 `frame_500Hz`，
然后由 `Core/Src/main.c` 主循环调用 `MPU6050_Read_And_Process()`。
读取函数从 `REG_ACCEL_XOUT_H` 开始连续读取 `MPU_DATA_LENGTH=14` 字节，
并将数据装入 `rawAccel`、`rawMPU6050Temperature`、`rawGyro`。

本章已经证明：

- 当前 500Hz 入口由 `Core/Src/stm32f1xx_it.c` 置位，主循环消费。
- 当前 I2C 连续读取在 `Drivers/CustomDrivers/Src/mpu6050.c` 中执行。
- 当前主线不使用中断注释块中的 `accelData500Hz/gyroData500Hz` 复制逻辑。
- `MPU6050_Read_And_Process()` 读取后还会进行方向矩阵处理，因此 `rawAccel/rawGyro` 是“读取并初步方向处理后的原始量”。
- MPU6050 输出寄存器从 `0x3B` 到 `0x48` 连续排列，项目一次读取 14 字节符合寄存器布局。
- 当前 100kHz I2C 下 14 字节连续读的理论总线时间下限约为 1.53ms，真实耗时仍需【待验证】。
- 当前项目以 500Hz 主循环读取约 800Hz 寄存器更新流，但没有数据就绪中断同步证据。
- `frame_500Hz` 是布尔触发标志，不是积压帧计数器；处理超时会体现在 `dt500Hz` 变大，
  但不会自动补回中间没有读取的传感器样本。
- 当前 `frame_500Hz` 是跨中断共享的普通 `bool`，从代码审查角度应记录 `volatile` 或同步机制风险；
  当前编译产物是否受影响仍需【待验证】。
- `HAL_I2C_Mem_Read()` 的 14 字节读取包含地址写、寄存器地址写、重复起始、地址读和连续接收，
  不是单纯的数组拷贝。
- `MPU_Read_Reg()` 的 100ms 超时远大于 2ms 控制帧，异常路径可能跨越多个 500Hz 周期。
- 读取失败时当前函数清零 `rawAccel/rawGyro` 但不返回错误状态，主循环仍会继续运行后续姿态和控制路径。
