# 第30章 多定时器三相电机驱动

## 1. 本章目标

第29章已经说明 PID 输出如何变成定子电角，以及定子电角如何进入正弦表。本章继续向下追踪：三相正弦目标值如何写入 STM32 的多个 PWM 定时器通道，并最终形成 Roll、Pitch、Yaw 三轴执行器输出。

本章目标是：

- 理解 `多定时器PWM相位同步` 为什么是本项目三相输出稳定性的前提。
- 理解 `三相正弦PWM电机驱动` 如何把电角、功率百分比和正弦表转换为 A/B/C 三相 CCR 目标值。
- 理解 `三轴电机硬件映射` 如何把 `MotorAxis_t` 逻辑轴映射到 TIM2、TIM3、TIM4 的具体 CCR 寄存器。
- 明确 `drv_pwmMotors.h` 中枚举注释与 `drv_pwmMotors.c` 中实际 `switch` 映射存在不一致，教材以实际 CCR 写入为准。
- 能从 `Core/Src/main.c` 的初始化顺序追踪到 `PWM_Motor_Init()`、`PWM_Motor_SetAngle()` 和 TIM 通道配置。

本章知识链为：

`知识点总表` 中的 `多定时器PWM相位同步`、`三相正弦PWM电机驱动`、`三轴电机硬件映射`

## 3. 问题背景

第29章讲到 `PWM_Motor_SetAngle()` 会得到三相 CCR 目标值。问题是，本项目并不是用一个定时器的三个通道驱动所有电机。

当前输出分布在多个定时器上：

- TIM3 提供 4 个 PWM 通道。
- TIM2 提供 3 个 PWM 通道。
- TIM4 提供 2 个 PWM 通道。

三个电机一共需要 9 个 PWM 输出，所以项目把 TIM2、TIM3、TIM4 组合起来使用。这带来三个工程问题。

第一，多个定时器必须使用一致的 PWM 周期，否则同一套 CCR 算法不能稳定映射到所有通道。

第二，跨定时器输出的相位要尽量对齐，否则同一个三相正弦目标写到不同定时器后，实际波形时间基准可能错开。

第三，逻辑轴和物理通道之间必须有清晰映射。否则控制层认为在驱动 Pitch，底层却可能写到另一个物理接口。

本章就是把这些问题在项目代码中逐一落地。

## 4. 核心概念

### 4.1 多定时器PWM相位同步

多定时器 PWM 相位同步是指多个独立定时器的计数起点尽量一致。

对于本项目，TIM2、TIM3、TIM4 都用于电机输出。如果它们的计数器起点差异很大，同一时刻写入的三相 CCR 目标值也可能对应不同的 PWM 周期位置。

当前项目没有使用 TIM 主从模式做硬件级同步。

`Core/Src/tim.c` 中 TIM2、TIM3、TIM4 的 `MasterSlaveMode` 都是 `TIM_MASTERSLAVEMODE_DISABLE`。

项目采用的是软件方式：启动所有 PWM 通道后，关闭全局中断，连续把 `TIM2->CNT`、`TIM3->CNT`、`TIM4->CNT` 清零。

因此教材应写成“减小相位错位”或“尽量对齐”，不能把它写成严格硬件同步。

### 4.2 三相正弦PWM电机驱动

三相正弦 PWM 的核心是：用 A/B/C 三个相差 120 度的正弦值，分别决定三个 PWM 通道的 CCR。

在本项目中：

```text
电角 angle_rad
-> normalized_angle
-> sinTable[index]
-> sin_a / sin_b / sin_c
-> ccr_val[0..2]
-> TIMx->CCRn
```

这里的 `ccr_val[]` 不是占空比百分数，而是具体写入 CCR 寄存器的比较值。

### 4.3 三轴电机硬件映射

三轴电机硬件映射回答的是：逻辑轴 `MOTOR_PITCH`、`MOTOR_ROLL`、`MOTOR_YAW` 分别写到哪些 TIM 通道。

当前应以 `Drivers/CustomDrivers/Src/drv_pwmMotors.c` 中 `PWM_Motor_SetAngle()` 的 `switch (axis)` 为准：

- `MOTOR_PITCH` -> TIM3 CH2 / TIM3 CH3 / TIM3 CH4
- `MOTOR_ROLL` -> TIM3 CH1 / TIM2 CH4 / TIM2 CH3
- `MOTOR_YAW` -> TIM4 CH4 / TIM2 CH2 / TIM4 CH3

`Drivers/CustomDrivers/Inc/drv_pwmMotors.h` 中 `MotorAxis_t` 的注释写成 `MOTOR_ROLL` 对应 Mot0、`MOTOR_PITCH` 对应 Mot1。

这个注释与 `drv_pwmMotors.c` 中的实际 CCR 写入注释不一致。本章按实际写寄存器路径分析，并把头文件注释列为需要注意的边界。

### 4.4 CCR比较值

CCR 是定时器输出比较寄存器。PWM1 模式下，计数器从 0 计到 ARR，CCR 决定输出翻转位置，从而决定占空比。

`Core/Src/tim.c` 中 TIM2、TIM3、TIM4 的 `Period` 都是 3599，所以一个 PWM 周期有 3600 个计数位置。`PWM_Motor_SetAngle()` 用：

```text
period = htim3.Init.Period + 1
half_period = period / 2
```

作为三相输出的中心。三相正弦值围绕 `half_period` 上下摆动。

### 4.5 PWM频率

PWM 频率由定时器输入时钟、PSC 和 ARR 决定：

```text
f_pwm = f_tim / ((PSC + 1) * (ARR + 1))
```

当前工程中，`Three-axis_cloud_platformV2.ioc` 记录：

```text
APB1TimFreq = 72MHz
```

TIM2、TIM3、TIM4 都在 APB1 定时器时钟域内，且源码配置为：

```text
PSC = 0
ARR = 3599
```

所以当前 PWM 频率可推导为：

```text
f_pwm = 72MHz / (1 * 3600) = 20kHz
```

20kHz 是当前配置下的理论 PWM 载波频率。真实引脚波形仍需示波器或逻辑分析仪验证。

### 4.6 初始Pulse与运行CCR

`Core/Src/tim.c` 中 `sConfigOC.Pulse = 500` 是 PWM 通道初始化时的初始比较值。
在当前 `ARR=3599`、PWM1 模式和高电平有效极性配置下，可以先按比较值比例估算为：

```text
duty_init = 500 / (3599 + 1) = 500 / 3600 ≈ 13.9%
```

所以它不是“0% 输出”，也不是第29章中的三相正弦目标值。它只是 PWM 通道启动后的初始比较点。

进入运行后，`PWM_Motor_SetAngle()` 会直接写：

```text
TIMx->CCRn = ccr_val[i]
```

因此教材分析运行时三相输出时，应以 `PWM_Motor_SetAngle()` 写入的 CCR 为准；
`Pulse=500` 只能说明初始化初值，不能代表闭环运行时的固定占空比。

### 4.7 预装载与瞬时写入边界

当前 TIM2、TIM3、TIM4 配置中 `AutoReloadPreload` 为 `DISABLE`。
这说明 ARR 自动重装载预装载关闭，但不能直接推断 CCR 通道预装载关闭。

本地 HAL 源码中，`HAL_TIM_PWM_ConfigChannel()` 会为 PWM 通道设置 `OCxPE`。
因此运行期写 `TIMx->CCRn` 时，应按“写入通道比较值的预装载路径”理解；
新比较值通常在对应定时器的更新事件进入有效比较路径。

这里有三个容易混淆的最小单元。

第一，软件写入时刻。`PWM_Motor_SetAngle()` 在临界区内执行 `TIMx->CCRn = ccr_val[i]`。
这个动作只说明 CPU 已经把新比较值写到对应定时器通道。

第二，PWM 正在比较的有效值。ST 的 AN4776 把预装载机制拆成 preload register
和 active/shadow register。启用预装载后，软件访问先进入 preload 实例，
正在输出的 PWM 周期仍使用 active 实例。

第三，更新事件。只有对应定时器产生 update event 时，preload 内容才转入 active 内容。
因此“写入 CCR”与“本周期立即改变输出比较点”不是同一件事。

但 TIM2、TIM3、TIM4 是三个独立定时器。当前没有发现跨定时器主从触发、
DMA burst 或统一更新事件提交的证据。因此临界区能降低软件写入被打断的风险，
不能证明多个定时器在同一计数点硬件同步装载新 CCR。

### 4.8 控制帧与PWM周期

本项目的控制循环约为 500Hz，对应控制帧周期：

```text
T_control = 1 / 500Hz = 2ms
```

第4.5节推导的 PWM 载波约为 20kHz，对应 PWM 周期：

```text
T_pwm = 1 / 20kHz = 50us
```

所以一个 500Hz 控制帧大约包含：

```text
T_control / T_pwm = 2ms / 50us = 40
```

个 PWM 周期。也就是说，`PWM_Motor_SetAngle()` 每次写入新的 `ccr_val[]` 后，
这些目标值会在随后的多个 PWM 周期内重复使用，直到下一次控制帧更新。

若通道 CCR 预装载启用，软件写入和有效比较值更新之间通常还会隔着对应定时器的
下一次 update event。因此单个定时器内的更新延迟可理解为“最多接近一个 PWM 周期”。
但跨 TIM2/TIM3/TIM4 时，是否在同一个 50us 周期边界一起生效，取决于三个定时器
的计数相位和同步机制。当前项目没有硬件主从同步证据，所以这仍应标为【待验证】。

## 5. 工作原理

### 5.1 初始化顺序

`Core/Src/main.c` 中，项目先初始化 TIM3、TIM2、TIM4：

```text
MX_TIM3_Init()
MX_TIM2_Init()
MX_TIM4_Init()
```

之后调用：

```text
PWM_Motor_Init()
```

这说明 `PWM_Motor_Init()` 不是配置定时器参数的函数，而是在 HAL 已经完成定时器初始化后，生成正弦表、启动 PWM 通道，并做计数器清零。

### 5.2 TIM2/TIM3/TIM4周期一致

`Core/Src/tim.c` 和 `Three-axis_cloud_platformV2.ioc` 都显示：

```text
TIM2 Prescaler = 0, Period = 3599
TIM3 Prescaler = 0, Period = 3599
TIM4 Prescaler = 0, Period = 3599
```

这使得 `PWM_Motor_SetAngle()` 可以用 `htim3.Init.Period + 1` 作为统一周期基准。若某个定时器的 Period 不一致，`ccr_val[]` 写入其他定时器时就可能出现比例错误。

结合当前时钟配置，还可以继续推导 PWM 载波频率。

`Core/Src/main.c::SystemClock_Config()` 设置 HSE 作为 PLL 源，PLL 乘 9，系统时钟为 72MHz。
APB1 分频为 2，`.ioc` 中记录 `APB1Freq=36MHz`、`APB1TimFreq=72MHz`。

因此 TIM2/TIM3/TIM4 的当前定时器输入时钟按 72MHz 分析：

```text
f_pwm = 72MHz / ((0 + 1) * (3599 + 1)) = 20kHz
```

这个推导解释了为什么三个定时器必须同时保持 `Prescaler=0` 和 `Period=3599`。
只改其中一个定时器，既会破坏三相幅值比例，也会破坏 PWM 载波频率一致性。

### 5.3 PWM通道启动

`PWM_Motor_Init()` 启动了 9 个 PWM 通道：

```text
TIM3 CH2 / CH3 / CH4
TIM3 CH1 / TIM2 CH4 / TIM2 CH3
TIM4 CH4 / TIM2 CH2 / TIM4 CH3
```

这些通道正好对应三个三相电机输出。函数没有检查每个 `HAL_TIM_PWM_Start()` 的返回值，所以如果某个通道启动失败，当前代码不会在这里直接报告错误。这是调试时需要注意的边界。

从本地 HAL 源码看，`HAL_TIM_PWM_Start()` 会调用 `TIM_CCxChannelCmd(..., TIM_CCx_ENABLE)`，
并在非触发从模式下使能定时器计数。因此它更接近“打开 PWM 硬件输出路径”，
而不是“只登记一个软件状态”。

还要注意 `HAL_TIM_PWM_Start()` 自身的状态前提。本地 `stm32f1xx_hal_tim.c`
先检查目标通道是否处于 `HAL_TIM_CHANNEL_STATE_READY`，如果不是 READY，就直接返回
`HAL_ERROR`；只有通过这个检查后，才会把通道状态置为 BUSY、使能 CCER 中的
`CCxE` 输出位，并在条件满足时使能定时器计数。也就是说，`HAL_TIM_PWM_Start()`
的 `HAL_OK` 表示这一条启动路径已经走到硬件输出使能步骤，而 `HAL_ERROR`
表示这一条路径在软件通道状态检查处就已经失败。

当前 `PWM_Motor_Init()` 没有保存这 9 个返回值，也没有把其中任何一次失败聚合到
自己的返回值。因此本函数最后的 `HAL_OK` 更准确地说是“函数执行到末尾”，
而不是“9 个 PWM 通道都已经被 HAL 证明启动成功”。这个边界会继续影响
`isInitialized` 的含义：`isInitialized=1` 只能说明本驱动的初始化流程已经放行
后续 `PWM_Motor_SetAngle()` 和 `PWM_Motor_Hardware_Diagnosis()`，不能反向证明
TIM2/TIM3/TIM4 的所有目标通道都已经产生了外部波形。

这也引出一个启动窗口。`Core/Src/main.c` 中 `PWM_Motor_Init()` 先于滤波器、
PID 状态、MPU6050 初始化和 `eepromConfig.pitchEnabled=false` 等控制使能关闭语句。
所以从 `PWM_Motor_Init()` 到后续软件控制真正接管某一路 CCR 之前，
PWM 硬件可能已经按初始化 `Pulse=500` 运行。控制轴使能关闭只能阻止
`computeMotorCommands()` 继续调用 `PWM_Motor_SetAngle()`，并不等价于调用
`HAL_TIM_PWM_Stop()` 或清零 CCR。

还要再拆出 `PWM_Motor_Init()` 内部的时间顺序。源码先连续调用 9 次
`HAL_TIM_PWM_Start()`，然后才进入临界区清零 `TIM2->CNT`、`TIM3->CNT`、`TIM4->CNT`。
因此启动阶段至少包含三个不同事件：

```text
事件1：各 PWM 通道被逐个启动，使用初始化 Pulse
事件2：TIM2/TIM3/TIM4 的 CNT 被软件清零，尽量对齐计数起点
事件3：首次 PWM_Motor_SetAngle() 写入运行期 ccr_val[]
```

这三个事件不能合并成一句“PWM 初始化完成”。如果调试开机瞬态，应分别记录
9 路 `HAL_TIM_PWM_Start()` 的顺序与返回值、CNT 清零前后的计数值、
以及第一次运行期 CCR 覆盖发生在哪个控制帧。这样才能判断观察到的波形来自初始化
`Pulse=500`，还是来自闭环接管后的三相正弦输出。

这不能被写成“电机一定会动作”。外部驱动器电源、使能脚、功率级输入逻辑、
相线连接和真实波形都需要仓库外实测证据，因此保持【待验证】。

### 5.4 软件计数器对齐

所有 PWM 通道启动后，项目执行：

```text
__disable_irq()
TIM2->CNT = 0
TIM3->CNT = 0
TIM4->CNT = 0
__enable_irq()
```

`__disable_irq()` 和 `__enable_irq()` 来自 CMSIS 层，用于形成很短的临界区。这样做的目的不是改变 PWM 参数，而是尽量让三个独立定时器从接近同一个计数位置开始运行。

因为这不是硬件主从同步，所以它不能证明长期无相位漂移，只能说明项目在初始化时做了软件对齐。

还要注意源码注释和教材结论之间的证据边界。`drv_pwmMotors.c` 的注释写到
连续清零只差 1 到 2 个 CPU 时钟周期，并称可保证三路 PWM 完全对齐。
这可以表达作者的调试意图，但教材不能把注释当成硬件同步证明。

从可验证事实看，当前代码能证明的是：

```text
关闭中断
-> CPU 连续写 TIM2->CNT、TIM3->CNT、TIM4->CNT
-> 恢复中断
```

它不能证明的是：

```text
三个定时器有共同触发源
三个定时器长期无相位漂移
三个定时器在同一个 update event 装载 CCR
外部引脚波形在示波器上完全重合
```

因此本章采用“减小初始相位错位”的表述。若要证明“完全对齐”，需要至少提供
定时器主从配置证据，或示波器/逻辑分析仪捕获的多通道波形证据【待验证】。

### 5.5 三相CCR目标值生成

`PWM_Motor_SetAngle()` 中，电角先被归一化到 `[0, 2π)`，再映射到 `SIN_TABLE_SIZE=720` 的正弦表索引。

随后取三相：

```text
sin_a = sinTable[index]
sin_b = sinTable[index + 1/3圈]
sin_c = sinTable[index + 2/3圈]
```

功率百分比转换为幅值：

```text
current_amp = half_period * power_percent / 100
```

最后得到：

```text
ccr_val[x] = half_period + sin_x * current_amp / SIN_SCALE
```

这意味着输出以 50% 占空比附近为中心，按正弦值上下偏移。

### 5.6 输出保护

`PWM_Motor_SetAngle()` 有两类保护。

第一，功率百分比限制：

```text
if power_percent > 60.0f: power_percent = 40.0f
if power_percent < 0.0f: power_percent = 0.0f
```

这里要注意，超过 60 时不是限制到 60，而是改成 40。教材按源码记录，不解释为常规饱和逻辑。

第二，CCR 溢出保护：

```text
ccr_val[i] > period -> period
ccr_val[i] < 0 -> 0
```

这防止异常幅值导致 CCR 超出定时器周期范围。

### 5.7 三相原子更新

写入 CCR 前，项目再次关闭全局中断：

```text
__disable_irq()
switch(axis) 写三个 CCR
__enable_irq()
```

这样做是为了让同一电机的三相 CCR 尽量在一个临界区内更新，减少 A/B/C 三相只更新一部分时被中断打断的风险。

但这里也要保留边界。关闭全局中断只能减少软件写寄存器被打断的风险，
不能保证三个不同定时器在硬件层面同一计数点同时装载新 CCR。

### 5.7.1 关中断不等于冻结定时器

`__disable_irq()` 的直接证据来自 CMSIS 内联实现和 `.list` 反汇编中的 `cpsid i`。
它说明 CPU 进入了屏蔽可屏蔽中断的状态，但源码没有停止 TIM2、TIM3、TIM4，也没有写入统一的门控位让三个计数器暂停。
因此这段临界区应拆成两个独立问题：

| 问题 | 当前证据 | 能证明什么 | 不能证明什么 |
|---|---|---|---|
| CPU 是否会被普通 IRQ 打断 | `__disable_irq()` / `__enable_irq()` 包围三次 CCR 写入，`.list` 保留 `cpsid i` | 写 CCR 的软件序列尽量连续执行 | 不能证明定时器计数器在这段时间停止 |
| TIM2/TIM3/TIM4 是否在同一硬件时刻更新 | `tim.c` 中三者周期一致，`PWM_Motor_Init()` 初始化阶段连续清零 CNT | 可以减小初始相位错位 | 不能证明运行期跨定时器共享同一个 update event |

所以，“三相原子更新”在本章中只表示 CPU 侧的三次寄存器写入被放在短临界区内。
它不是硬件锁存、不是定时器暂停，也不是跨 TIM 的同步提交。若要证明运行期三相在同一个 PWM 边界生效，
仍需要主从同步、统一更新事件、DMA 批量提交或外部逻辑分析仪证据【待验证】。

当前本地 HAL 会启用通道 `OCxPE`，但没有发现跨 TIM2/TIM3/TIM4 的
DMA burst、定时器主从触发或统一更新事件提交证据。
所以本章应把它称为“临界区内三相写入”，而不是“硬件同步更新”。

把这条链路按时间拆开，可以得到更准确的工程解释：

```text
CPU进入临界区
-> 连续写入三个 TIMx->CCRn
-> CPU退出临界区
-> 各自定时器等待自己的 update event
-> 各自通道把预装载值转为有效比较值
```

如果三个相都在同一个定时器上，例如 `MOTOR_PITCH` 使用 TIM3 CH2/CH3/CH4，
三相通常共享 TIM3 的 update event。若三相跨两个定时器，例如 Roll 和 Yaw，
则还要看 TIM2/TIM3/TIM4 的计数相位是否接近。当前软件清零 CNT 可以减小初始差异，
但不能替代主从同步或 DMA 批量提交。

#### 三相写入完成度证据边界

同一句“写入三相 PWM”在调试时至少要拆成四个证据层级，否则很容易把
CPU 层面的寄存器写入误读成外部电机已经同步响应：

| 层级 | 当前可用证据 | 能证明什么 | 不能证明什么 |
|---|---|---|---|
| CPU 写 CCR 完成 | `PWM_Motor_SetAngle()` 在 `__disable_irq()` / `__enable_irq()` 之间按 `axis` 连续写三个 `TIMx->CCRn`；`.list` 中保留 Pitch、Roll、Yaw 三组 CCR 写指令 | 当前 Debug 固件包含三相 CCR 写入路径；同一调用中三次写寄存器尽量不被普通中断打断 | 不能证明 CCR active compare 已经同一时刻改变，也不能证明无效 `axis` 有错误报告 |
| 预装载到有效比较值 | `HAL_TIM_PWM_ConfigChannel()` 设置 `TIM_CCMR1_OC1PE`、`OC2PE`、`TIM_CCMR2_OC3PE`、`OC4PE`；第 8.10 节已核对 `MasterSlaveMode = TIM_MASTERSLAVEMODE_DISABLE` | PWM 通道存在 CCR 预装载语义，新 CCR 通常要等各自定时器 update event 才进入有效比较值 | 不能证明 TIM2/TIM3/TIM4 的 update event 在同一计数点发生 |
| 外部引脚 PWM 波形 | `HAL_TIM_MspPostInit()` 配置 TIM2/TIM3/TIM4 相关 GPIO 为复用推挽输出；`.map` / `.list` 保留初始化和启动路径 | 固件包含把定时器通道连到引脚的配置路径 | 不能证明示波器上已经出现正确频率、占空比、相位和边沿质量 |
| 驱动器与电机响应 | 章节内只有相序映射、CCR 目标值和诊断函数旁证 | 可以建立待测项目清单，例如相线顺序、转向、力矩、电流、温升 | 不能证明真实负载下方向、力矩、噪声、发热或稳定性正确，仍为【待验证】 |

因此，`MOTOR_PITCH` 的三相都落在 TIM3 上时，证据链最多可以说三相共享同一个定时器的更新节奏；
`MOTOR_ROLL` 和 `MOTOR_YAW` 跨 TIM2/TIM3 或 TIM2/TIM4 时，只能说 CPU 在临界区内完成了三次写入。
是否在同一个 PWM 周期边界同时转入 active compare，需要主从同步、统一 update event、DMA 批量提交或外部逻辑分析仪记录继续证明。

结合 500Hz 控制帧和 20kHz PWM 载波，可以把运行期更新理解成：

```text
每 2ms 左右计算一次新电角
-> 生成一次新的 ccr_val[0..2]
-> 在临界区写入三个 CCR preload 路径
-> 最多等待约一个 50us PWM 周期进入对应 active compare
-> 约 40 个 PWM 周期后进入下一次控制更新
```

这说明 CCR 预装载带来的单周期延迟相对 2ms 控制帧通常很小，
但它对跨定时器三相是否同一周期生效仍然重要。
如果未来把控制频率提高到接近 PWM 频率，或者要求更严格的三相同步，
当前“CPU 临界区逐个写 CCR”的方案就需要重新评估。

这也是为什么本章只建议把 DMA、定时器主从和主动 update event 当作后续优化方向。
当前项目每个控制帧只写 9 个 CCR，软件路径简单、可读性强；
但它牺牲了硬件级同步提交能力和批量更新的可证明性。

## 6. STM32实现机制

本章涉及 STM32 的关键机制有六个。

第一，通用定时器 PWM。TIM2、TIM3、TIM4 都配置为 PWM 输出，使用 `TIM_OCMODE_PWM1`。第12章已经讲过 PWM 的基本含义，本章把它用于三相电机。

第二，GPIO 复用输出。`Core/Src/tim.c` 中 `HAL_TIM_MspPostInit()` 把 TIM 通道映射到对应 GPIO：

- TIM2 CH2 -> PA1
- TIM2 CH3 -> PA2
- TIM2 CH4 -> PA3
- TIM3 CH1 -> PA6
- TIM3 CH2 -> PA7
- TIM3 CH3 -> PB0
- TIM3 CH4 -> PB1
- TIM4 CH3 -> PB8
- TIM4 CH4 -> PB9

这些 PWM 引脚的 `GPIO_InitStruct.Speed` 当前都配置为 `GPIO_SPEED_FREQ_LOW`。
GPIO 速度配置不改变定时器计数频率，但会影响引脚边沿速度、上升/下降时间、
EMI 和外部驱动器看到的波形质量。20kHz 载波能否在实际硬件上形成足够干净的边沿，
需要示波器或逻辑分析仪验证；源码只能证明 GPIO 被配置为复用推挽低速输出。

第三，CMSIS 寄存器直接访问。项目没有通过 HAL 宏写 CCR，而是直接写：

```text
TIM3->CCR2 = ...
TIM2->CCR4 = ...
TIM4->CCR3 = ...
```

这种写法直接、开销低，但要求开发者非常清楚每个通道对应的轴和相。

第四，临界区。`__disable_irq()` / `__enable_irq()` 被用于初始化阶段的 CNT 清零和运行阶段的三相 CCR 写入，减少中断插入导致的瞬时不一致。

这里还要区分“直接关开中断”和“保存并恢复原中断状态”。
本地 CMSIS `cmsis_gcc.h` 中 `__disable_irq()` 对应 `cpsid i`，
`__enable_irq()` 对应 `cpsie i`；同一文件也提供 `__get_PRIMASK()` 和
`__set_PRIMASK()`。当前 `drv_pwmMotors.c` 直接调用 `__disable_irq()` 后再调用
`__enable_irq()`，没有保存进入临界区前的 PRIMASK 状态。
这在当前主循环直接调用场景下简单可读，但若未来在已经关中断的上下文里调用该函数，
函数末尾的 `__enable_irq()` 可能改变调用者原本的中断屏蔽状态。
因此它是“简单临界区”，不是嵌套安全的临界区封装。

第五，定时器资源组合。三轴三相输出一共需要 9 路 PWM。当前 TIM3 提供 4 路、TIM2 提供 3 路、TIM4 提供 2 路，
刚好组合出 9 路主电机输出。TIM8 虽然也有 PWM 初始化证据，但当前 `PWM_Motor_SetAngle()` 没有写 TIM8 的 CCR。

因此本章按当前主链路解释 TIM2/TIM3/TIM4，而不是把 TIM8 写入主电机输出链路。
是否未来改用 TIM8 或重排通道，属于后续硬件/代码重构问题。

第六，HAL 初始化与寄存器运行写入并存。TIM 参数、GPIO 复用和 PWM 模式由 Cube/HAL 生成代码初始化；
运行期的三相更新则直接写 CMSIS 外设寄存器。这个组合的优点是初始化清晰、运行写入开销低。
本地 HAL 在 `HAL_TIM_PWM_ConfigChannel()` 中还会设置通道 `OCxPE`。
代价是寄存器映射和预装载生效边界必须由开发者自己维护，
头文件注释与实现不一致时必须以写寄存器路径为准。

## 7. 项目中的应用

本章涉及的项目证据文件包括：

- `Core/Src/main.c`
- `Core/Src/tim.c`
- `Three-axis_cloud_platformV2.ioc`
- `Drivers/CustomDrivers/Src/drv_pwmMotors.c`
- `Drivers/CustomDrivers/Inc/drv_pwmMotors.h`
- `Drivers/SRC/Src/computeMotorCommands.c`

项目中的运行关系是：

```text
Core/Src/main.c
-> MX_TIM3_Init() / MX_TIM2_Init() / MX_TIM4_Init()
-> PWM_Motor_Init()
-> computeMotorCommands()
-> PWM_Motor_SetAngle(axis, angle_rad, power_percent)
-> TIMx->CCRn
```

`computeMotorCommands.c` 决定调用哪个逻辑轴：

- Roll 分支当前调用 `PWM_Motor_SetAngle(MOTOR_PITCH, ...)`。
- Pitch 分支当前调用 `PWM_Motor_SetAngle(MOTOR_ROLL, ...)`。
- Yaw 分支当前调用 `PWM_Motor_SetAngle(MOTOR_YAW, ...)`。

这说明控制轴和电机枚举之间存在项目特定映射。不能把 `rollEnabled`、
Roll 控制分支、`MOTOR_ROLL` 枚举和物理 Roll 电机直接当作同一层概念。
调试时应把链路拆成两级：

```text
控制分支名 -> 传入 PWM_Motor_SetAngle() 的 MotorAxis_t
MotorAxis_t -> drv_pwmMotors.c switch 分支实际写入的 TIMx->CCRn
```

按当前源码，这条二级映射是：

```text
Roll 控制分支  -> MOTOR_PITCH -> TIM3 CCR2/CCR3/CCR4
Pitch 控制分支 -> MOTOR_ROLL  -> TIM3 CCR1, TIM2 CCR4, TIM2 CCR3
Yaw 控制分支   -> MOTOR_YAW   -> TIM4 CCR4, TIM2 CCR2, TIM4 CCR3
```

所以“关闭 rollEnabled”只说明 Roll 控制分支不再运行，不等于
`MOTOR_ROLL` 对应的 TIM3 CCR1、TIM2 CCR4、TIM2 CCR3 不会被其它控制分支写入。
第31章会继续从系统架构角度解释三轴控制流，本章只记录底层执行器映射。

## 8. 代码分析

### 8.1 PWM_Motor_Init()

`PWM_Motor_Init()` 的输入是全局定时器句柄 `htim2`、`htim3`、`htim4`，输出是启动 PWM 通道并设置 `isInitialized = 1`。

函数的状态变化包括：

- 生成 `sinTable[]`。
- 启动 9 个 PWM 通道。
- 清零 TIM2/TIM3/TIM4 计数器。
- 将 `isInitialized` 置为 1。

如果 `isInitialized` 已经为 1，函数直接返回 `HAL_OK`。这避免重复启动。

风险点是：`HAL_TIM_PWM_Start()` 的返回值未逐个检查。若后续要增强诊断，可以考虑记录启动失败状态；当前教材只记录现状。

这里要把三个状态语义分开：

```text
HAL_TIM_PWM_Start() 单次返回 HAL_OK
    -> 这一通道通过 HAL 通道状态检查，并执行到 CCxE 使能路径

PWM_Motor_Init() 返回 HAL_OK
    -> 当前实现中函数执行到末尾，或已经初始化过后直接返回

isInitialized = 1
    -> 本驱动允许后续 SetAngle/Diagnosis 继续写 CCR
```

这三者不是同一层证明。尤其在当前实现中，如果第 4 路 `HAL_TIM_PWM_Start()`
失败，后面第 5 至第 9 路仍会继续调用；函数末尾仍会设置 `isInitialized=1`
并返回 `HAL_OK`。所以调试时不能只看 `PWM_Motor_Init()==HAL_OK`，还要逐项记录
9 次 HAL 返回值，并把它们和对应的 `TIMx->CCER` 输出使能位、GPIO 复用配置、
示波器波形放在同一张表里。

### 8.2 PWM_Motor_SetAngle()

`PWM_Motor_SetAngle()` 的输入是：

- `axis`：逻辑电机轴。
- `angle_rad`：电气角度，单位弧度。
- `power_percent`：功率百分比。

函数先检查 `isInitialized`。如果未初始化，直接返回，不写任何 CCR。

随后它完成：

```text
角度归一化
-> 正弦表索引
-> 三相正弦值
-> 幅值缩放
-> CCR范围保护
-> 按axis写入CCR
```

这里的 `axis` 决定写哪组三相通道，`angle_rad` 决定三相相位，`power_percent` 决定正弦幅值。

`switch(axis)` 当前没有 `default` 分支。如果传入的 `axis` 不是
`MOTOR_PITCH`、`MOTOR_ROLL` 或 `MOTOR_YAW`，函数仍会进入临界区，
但不会写任何 CCR，也不会返回错误或打印日志。`MotorAxis_t` 是枚举类型，
但 C 语言不能阻止运行时把越界整数强制传入该参数。
因此本章把 `axis` 合法性也作为调试边界：正常主线调用有固定枚举值证据，
但底层函数本身没有无效轴防护。

### 8.3 初始化Pulse与运行CCR

`Core/Src/tim.c` 对 TIM2/TIM3/TIM4 的 PWM 通道初始化都使用：

```text
sConfigOC.Pulse = 500
```

在 `ARR=3599` 时，初始占空比约为：

```text
500 / 3600 ≈ 13.9%
```

但运行期进入 `PWM_Motor_SetAngle()` 后，代码直接写入各个 `TIMx->CCRn`。
因此 `Pulse=500` 不是运行时三相正弦输出的固定占空比。

调试时如果刚初始化还没调用 `PWM_Motor_SetAngle()`，可能看到初始 Pulse 影响；
闭环运行后，应以当前 CCR 寄存器值和 `ccr_val[]` 为准。

更精细地说，初始化 Pulse 可能出现在 `HAL_TIM_PWM_Start()` 之后、首次运行期 CCR
覆盖之前。由于 `PWM_Motor_Init()` 是按通道逐个启动，再统一清零计数器，
不同通道“开始输出初始化比较值”的时刻并不完全相同。CNT 清零能重新拉近
TIM2/TIM3/TIM4 的计数相位，但它不会自动把初始化 `Pulse=500` 改成 0，
也不会替代后续 `PWM_Motor_SetAngle()` 的运行期写入。

再把 `Core/Src/main.c` 的启动顺序接进来，可以看到更细的边界：

```text
init_eepromConfig(true)
-> PWM_Motor_Init()
-> initFirstOrderFilter()
-> initPID()
-> MPU6050_Init()
-> eepromConfig.pitchEnabled = false
-> eepromConfig.rollEnabled = false
-> eepromConfig.yawEnabled = false
-> startup_delay_counter 达到 1000 后部分轴再打开
-> computeMotorCommands()
```

也就是说，PWM 通道启动、控制轴软件使能、闭环电角输出下发，是三件不同的事。
`eepromConfig.*Enabled=false` 影响的是 `computeMotorCommands()` 中对应分支是否执行；
它没有停止已经启动的 TIM 通道，也没有主动把 `TIMx->CCRn` 改成 0。
因此调试开机阶段时，应单独记录“PWM 载波是否已经启动”和“闭环控制是否已经接管”。
真实外部占空比、电机力矩或动作仍需示波器和硬件点测确认【待验证】。

### 8.4 PWM频率证据链

当前 PWM 频率推导需要三类证据：

```text
SystemClock_Config(): SYSCLK = 72MHz, APB1 = HCLK / 2
Three-axis_cloud_platformV2.ioc: APB1TimFreq = 72MHz
tim.c: PSC = 0, ARR = 3599
```

所以：

```text
f_pwm = 72MHz / ((0 + 1) * (3599 + 1)) = 20kHz
```

这个频率是根据工程配置得到的理论值。实际引脚 PWM 频率仍需要外部测量【待验证】。

### 8.5 Pitch物理接口映射

`MOTOR_PITCH` 分支写入：

```text
TIM3->CCR2 = ccr_val[0]
TIM3->CCR3 = ccr_val[1]
TIM3->CCR4 = ccr_val[2]
```

结合 `Core/Src/tim.c`，对应 GPIO 是：

```text
TIM3 CH2 -> PA7
TIM3 CH3 -> PB0
TIM3 CH4 -> PB1
```

这组三相全部在 TIM3 上。

### 8.6 Roll物理接口映射

`MOTOR_ROLL` 分支写入：

```text
TIM3->CCR1 = ccr_val[0]
TIM2->CCR4 = ccr_val[1]
TIM2->CCR3 = ccr_val[2]
```

对应 GPIO 是：

```text
TIM3 CH1 -> PA6
TIM2 CH4 -> PA3
TIM2 CH3 -> PA2
```

这组三相跨 TIM3 和 TIM2，因此更依赖多定时器周期一致和初始化阶段的计数器对齐。

### 8.7 Yaw物理接口映射

`MOTOR_YAW` 分支写入：

```text
TIM4->CCR4 = ccr_val[0]
TIM2->CCR2 = ccr_val[1]
TIM4->CCR3 = ccr_val[2]
```

对应 GPIO 是：

```text
TIM4 CH4 -> PB9
TIM2 CH2 -> PA1
TIM4 CH3 -> PB8
```

这组三相跨 TIM4 和 TIM2，也依赖多定时器协同。

### 8.8 头文件注释边界

`drv_pwmMotors.h` 中 `MotorAxis_t` 的枚举值是：

```text
MOTOR_ROLL = 0
MOTOR_PITCH = 1
MOTOR_YAW = 2
```

枚举值本身是有效的，但旁边的 Mot0/Mot1 注释与 `drv_pwmMotors.c` 的实际分支注释和 CCR 写入不一致。阅读和调试时，应以 `PWM_Motor_SetAngle()` 的 `switch` 分支为准。

本章不修改源码，只在教材中记录这个风险点。

### 8.9 HAL启动返回值边界

`PWM_Motor_Init()` 连续调用 9 次 `HAL_TIM_PWM_Start()`，但当前没有逐项检查返回值。

这意味着函数最后返回 `HAL_OK`，并不严格证明每一个 PWM 通道都成功启动。
从 HAL 实现路径看，失败可能发生在通道状态检查阶段：当
`TIM_CHANNEL_STATE_GET(htim, Channel)` 不是 `HAL_TIM_CHANNEL_STATE_READY` 时，
`HAL_TIM_PWM_Start()` 会返回 `HAL_ERROR`，不会继续执行 `TIM_CCxChannelCmd()`
使能该通道的 `CCxE` 位。当前项目代码没有捕获这个结果，所以失败通道不会在
`PWM_Motor_Init()` 的返回值中暴露出来。

这个边界可以拆成两类证据：

```text
软件放行证据：PWM_Motor_Init() 返回 HAL_OK，isInitialized=1
硬件启动证据：单路 HAL_TIM_PWM_Start() 返回 HAL_OK，CCER.CCxE=1，外部引脚有波形
```

第一类证据只能说明驱动层不再拦截后续 `PWM_Motor_SetAngle()` 调用；第二类证据
才接近“某一路 PWM 已经打开”。如果只记录第一类证据，就可能出现一种隐蔽现象：
控制算法持续写 `TIMx->CCRn`，软件变量也看起来正常变化，但某一路由于启动阶段失败，
实际引脚没有有效 PWM 输出。

如果某一路 PWM 没有波形，调试时应回到：

```text
HAL_TIM_PWM_Start() 返回值
TIMx->CCER 输出使能位
GPIO 复用配置
实际引脚波形
```

逐层验证。当前教材只记录源码现状，不把 `PWM_Motor_Init()` 返回 `HAL_OK` 写成九路 PWM 全部成功的强证明。

#### 8.9.1 调用点返回值丢弃的构建证据

`Debug/Three-axis_cloud_platformV2.list` 还能把这个边界推进到调用点机器码层面。
在 `PWM_Motor_Init()` 中，9 次 `HAL_TIM_PWM_Start()` 都表现为设置 `r1` 通道参数、
装入 `r0` 定时器句柄地址、执行 `bl 8009518 <HAL_TIM_PWM_Start>`。每次 `bl` 之后，
当前反汇编没有看到针对返回值的 `cmp`、`cbz`、`bne` 等判断，而是直接装载下一路通道参数；
第 9 次调用之后也直接进入 `cpsid i` 和 `TIM2/TIM3/TIM4` 计数器清零流程。

这说明“返回值未逐项检查”不是只来自源码阅读，而是在当前 Debug 构建产物中也保留为
“调用后继续执行”的控制流。它能证明当前固件不会因为某一路 `HAL_TIM_PWM_Start()`
返回 `HAL_ERROR` 而停止初始化流程；但它仍不能证明某一路确实失败，也不能证明外部波形缺失。
是否失败必须通过临时记录返回值、检查 `CCER.CCxE` 和示波器/逻辑分析仪继续验证【待验证】。

### 8.10 预装载和同步更新边界

`tim.c` 中 TIM2/TIM3/TIM4 的 `AutoReloadPreload` 为 `TIM_AUTORELOAD_PRELOAD_DISABLE`。
这只说明 ARR 预装载关闭，不说明 CCR 通道预装载关闭。

本地 `stm32f1xx_hal_tim.c` 的 `HAL_TIM_PWM_ConfigChannel()` 会设置：

```text
TIM_CCMR1_OC1PE
TIM_CCMR1_OC2PE
TIM_CCMR2_OC3PE
TIM_CCMR2_OC4PE
```

所以 PWM 通道存在 CCR 预装载路径。真正需要保留边界的是：
当前没有看到跨 TIM2/TIM3/TIM4 的统一更新事件、主从同步或 DMA burst 提交。

从寄存器观察角度，读者可以把本节拆成四个检查点：

```text
CCMR1/CCMR2 的 OCxPE 位：通道预装载是否启用
CR1 的 ARPE 位：ARR 自动重装载预装载是否启用
EGR 的 UG 位：是否由软件主动产生 update event
SMCR/CR2：是否配置主从触发或 TRGO 关联
```

当前源码只证明了 HAL 配置通道时启用了 `OCxPE`，以及定时器主从模式被禁用。
没有看到运行期写 `EGR.UG` 主动提交 CCR 的路径，也没有看到三定时器统一触发的配置。

因此，三相写入临界区只能降低软件打断风险，
不能等同于三个不同定时器在同一硬件计数点装载新 CCR。
如果未来要做到更严格同步，应考虑定时器主从、DMA 或统一更新事件机制，
并重新更新教材证据。

还有一个临界区嵌套边界。当前代码使用：

```text
__disable_irq()
写 CNT 或 CCR
__enable_irq()
```

它没有执行：

```text
primask = __get_PRIMASK()
__disable_irq()
...
__set_PRIMASK(primask)
```

所以它不会恢复调用前的中断屏蔽状态，而是无条件重新打开 IRQ。
当前主线从 500Hz 主循环调用 `PWM_Motor_SetAngle()`，没有发现它被放进另一个已关中断临界区的证据。
但如果未来把电机输出迁移到 ISR、RTOS 临界区或其它关中断区域，
就需要改成保存/恢复 PRIMASK 的写法，或者明确禁止嵌套调用【待验证】。

### 8.11 PWM_Motor_Hardware_Diagnosis()

`PWM_Motor_Hardware_Diagnosis()` 是底层点测函数，不是当前主控制路径的一部分。
当前搜索只发现它的声明和定义，没有发现 `Core/Src/main.c` 或
`computeMotorCommands()` 调用它。

还要注意，它不是“每调用一次就立即关闭所有通道并切换一步”。函数内部先检查：

```text
if current_tick - last_tick >= 1000
```

只有 1000ms 条件满足时，才会进入分支，先把 9 路 CCR 写成 `0`，
再根据 `step` 给某一相写入固定 `duty=1000`。
未到 1000ms 的调用不会改写 CCR。

当前函数中 Pitch 和 Yaw 的相位写入多处被注释掉，未注释的点测写入集中在 Roll 三相：

```text
step 0: TIM3->CCR1 = duty
step 1: TIM2->CCR4 = duty
step 2: TIM2->CCR3 = duty
```

因此它当前更像“Roll 单相轮询点测框架”，不是完整三轴自动诊断工具。
它还绕过 `PWM_Motor_SetAngle()`、正弦查表、`power_percent` 限制和 `MotorAxis_t`
映射，直接写 TIMx->CCRn。这个函数更适合第33章讨论电机硬件诊断；
本章只把它作为底层 CCR 映射的旁证，不把它写成主输出链路。

这里还要拆出“诊断旁证”和“被动观察”的差别。`PWM_Motor_Hardware_Diagnosis()`
不是只读取寄存器的观察函数；一旦进入 1000ms 到期分支，它会主动先把 9 路 CCR
写成 `0`，再把当前 step 对应的某一路写成 `1000`。因此如果临时把它接入主循环，
它会和 `computeMotorCommands()` / `PWM_Motor_SetAngle()` 争用同一批 CCR。

按当前源码，诊断函数未注释写入的是 Roll 物理输出组：

```text
TIM3->CCR1
TIM2->CCR4
TIM2->CCR3
```

而正常控制路径中，Pitch 分支调用 `PWM_Motor_SetAngle(MOTOR_ROLL, ...)`，
最终也会写这组三个寄存器。因此调试时不能只问“诊断函数有没有调用”，
还要记录同一控制周期内谁最后写了这些 CCR。最后写入者决定寄存器窗口看到的数值，
也决定下一次 update event 之后更可能出现在外部引脚上的比较值。

### 8.12 控制帧和PWM周期边界

第26章的 500Hz 控制循环和本章的 20kHz PWM 载波不是同一个频率层级。
当前项目大致是每 2ms 计算一次新的电机电角和 CCR 目标值，而 PWM 硬件每 50us
完成一个载波周期。

因此，调试时应区分三类时间：

```text
控制更新时间: computeMotorCommands() 进入 PWM_Motor_SetAngle()
软件写入时间: CPU 写 TIMx->CCRn 的具体时刻
硬件生效时间: 对应定时器 update event 后 active compare 使用新值
```

如果只在调试器中看到 `TIMx->CCRn` 已改变，只能说明寄存器可见值改变；
不能直接证明外部引脚已经在同一 PWM 周期全部切换到新比较点。
这一区别来自 ST 定时器的预装载和更新事件机制，也正是多定时器三相输出需要额外验证的原因。

当前 Debug 构建产物只能证明它的输入段曾参与链接并被记录，不能证明它保留在最终镜像、

## 9. 调试方法

第一步，确认初始化顺序。

- 在 `Core/Src/main.c` 中确认 `MX_TIM3_Init()`、`MX_TIM2_Init()`、`MX_TIM4_Init()` 先于 `PWM_Motor_Init()`。
- 在 `PWM_Motor_Init()` 结束后观察 `isInitialized` 是否为 1。

第二步，确认 TIM 周期一致。

- 观察 `htim2.Init.Period`、`htim3.Init.Period`、`htim4.Init.Period` 是否都为 3599。
- 观察 `htim2.Init.Prescaler`、`htim3.Init.Prescaler`、`htim4.Init.Prescaler` 是否都为 0。
- 结合 `.ioc` 中 `APB1TimFreq=72MHz`，计算理论 PWM 频率是否为 20kHz。
- 若使用示波器测得频率不同，继续检查时钟树、PSC、ARR 和实际引脚通道。

第三步，确认 GPIO 复用和输出速度。

- 观察 TIM2/TIM3/TIM4 对应 GPIO 是否为 `GPIO_MODE_AF_PP`。
- 观察 `GPIO_InitStruct.Speed` 是否为 `GPIO_SPEED_FREQ_LOW`。
- 若外部波形边沿过慢或驱动器识别异常，应结合负载、电平转换和示波器结果继续验证【待验证】。

第四步，确认计数器对齐入口。

- 在 `PWM_Motor_Init()` 中观察 `TIM2->CNT`、`TIM3->CNT`、`TIM4->CNT` 清零位置。
- 注意这只是初始化阶段的软件对齐，不是持续硬件同步。

第五步，确认三相 CCR 生成。

- 在 `PWM_Motor_SetAngle()` 中观察 `ccr_val[0]`、`ccr_val[1]`、`ccr_val[2]`。
- 确认它们围绕 `half_period` 上下变化，并且没有超出 `[0, period]`。

第六步，确认轴到通道映射。

- 调用 `PWM_Motor_SetAngle(MOTOR_PITCH, ...)` 时观察 TIM3 CCR2/CCR3/CCR4。
- 调用 `PWM_Motor_SetAngle(MOTOR_ROLL, ...)` 时观察 TIM3 CCR1、TIM2 CCR4、TIM2 CCR3。
- 调用 `PWM_Motor_SetAngle(MOTOR_YAW, ...)` 时观察 TIM4 CCR4、TIM2 CCR2、TIM4 CCR3。
- 如果临时传入越界 `axis` 做防御性测试，应确认当前函数不会写 CCR，也不会报告错误。

第七步，确认控制层调用的逻辑轴。

- 在 `computeMotorCommands.c` 中观察 Roll、Pitch、Yaw 分支分别传入的 `MotorAxis_t`。
- 把“控制分支名 -> `MotorAxis_t` -> 实际 TIMx->CCRn”拆成三列记录，不要把分支名直接写成物理输出组。
- 不要只根据 `drv_pwmMotors.h` 中的 Mot 注释判断物理接口。

第八步，区分初始化 Pulse 和运行 CCR。

- 刚初始化后观察 `sConfigOC.Pulse=500` 对应的初始 CCR。
- 调用 `PWM_Motor_SetAngle()` 后观察 `TIMx->CCRn` 是否被 `ccr_val[]` 覆盖。
- 不要把 `Pulse=500` 当作闭环运行时固定占空比。
- 沿 `main.c` 记录 `PWM_Motor_Init()`、三轴 `Enabled=false` 和首次控制接管的先后顺序。
- 确认软件使能关闭只是阻止对应分支继续写运行 CCR，不等于 PWM 通道已经停止。

第九步，检查 PWM 启动返回值。

- 临时记录 9 次 `HAL_TIM_PWM_Start()` 的返回状态。
- 同时记录 `PWM_Motor_Init()` 的最终返回值和 `isInitialized`，避免把驱动放行误读成九路硬件启动证明。
- 若某路无波形，先确认启动返回值，再查 CCER、GPIO 复用和外部测量。
- 当前源码没有逐项检查返回值，所以运行证据需要调试补充。

第十步，检查 CCR 预装载和更新事件。

- 观察 `TIMx->CCMR1/CCMR2` 中对应通道的 `OCxPE` 位。
- 观察 `TIMx->CR1` 中 `ARPE` 位，区分 ARR 预装载和 CCR 预装载。
- 搜索运行期是否写 `TIMx->EGR` 或调用 `HAL_TIM_GenerateEvent()`。
- 观察 `TIMx->CNT` 与 `TIMx->SR.UIF`，判断新 CCR 何时进入有效比较路径。

第十一步，检查临界区中断状态。

- 观察 `PWM_Motor_Init()` 和 `PWM_Motor_SetAngle()` 中的 `__disable_irq()` / `__enable_irq()`。
- 若未来从已关中断上下文调用，记录进入前后的 PRIMASK。
- 当前代码没有保存/恢复 PRIMASK，不能当作嵌套安全临界区。

第十二步，记录控制帧和 PWM 周期关系。

- 记录 500Hz 控制帧的 `dt`，计算约 `2ms`。
- 记录 20kHz PWM 周期，计算约 `50us`。
- 说明一次控制输出大约跨越 40 个 PWM 周期。
- 若外部测量显示更新抖动，应区分控制循环抖动、CCR 写入抖动和 PWM 周期边界。

第十三步，确认硬件点测函数边界。

- 搜索 `PWM_Motor_Hardware_Diagnosis()` 是否有主程序或主控制路径调用点。
- 记录它是否进入 `current_tick - last_tick >= 1000` 分支。
- 记录 Pitch/Yaw 写入是否仍被注释，当前未注释写入是否主要覆盖 Roll 三相。
- 区分诊断函数直接写 CCR 与主链路 `PWM_Motor_SetAngle()` 通过正弦表生成 CCR 的差异。
- 如果临时接入诊断函数，应记录它和 `computeMotorCommands()` 的调用先后顺序，
  并确认同一批 CCR 是否在下一次 500Hz 控制帧中被闭环输出覆盖。

调试记录建议：

- 记录 TIM2/TIM3/TIM4 的 PSC、ARR、启动通道、计数器清零和 `isInitialized`。
- 保留 `Debug/Three-axis_cloud_platformV2.map` 与 `Debug/Three-axis_cloud_platformV2.list` 中本章符号、调用和 CCR 写入证据的搜索记录。
- 记录理论 PWM 频率、外部测得 PWM 频率和测量设备条件。
- 对每个轴，应记录传入的 `MotorAxis_t`、三相 `ccr_val[]`、实际写入的 TIM/CCR 寄存器。
- 对异常输入，应记录无效 `axis` 是否导致无 CCR 写入且无错误报告。
- 对预装载边界，应记录 `OCxPE`、`ARPE`、`CNT`、`SR.UIF` 和是否存在主动 `EGR.UG`。
- 对临界区边界，应记录进入前后 PRIMASK，确认是否存在嵌套关中断调用。
- 对 GPIO 输出，应记录复用模式、输出速度、外部边沿质量和测量探头条件。
- 对时间尺度，应记录控制帧周期、PWM 载波周期、CCR 写入时刻和外部波形更新时刻。
- 对硬件点测函数，应记录是否有调用点、是否进入 1000ms 到期分支、实际未注释写入了哪些 CCR。
- 软件计数器清零只能证明初始化阶段对齐，不能证明持续硬件同步或相位无漂移。
- `Pulse=500`、`ccr_val[]` 和实际 `TIMx->CCRn` 应分别记录，避免混淆初始化值和运行值。
- 启动窗口内应分开记录 PWM 硬件通道、控制轴软件使能和首次闭环 CCR 覆盖时刻。
- 外部 PWM 波形、驱动响应、电机相序、转向和温升都属于仓库外实测证据。

## 10. 常见问题

问题一：TIM2、TIM3、TIM4 是否使用硬件主从同步。

当前证据不是。`Core/Src/tim.c` 中它们的 `MasterSlaveMode` 是禁用状态。项目通过启动后连续清零 CNT 的方式做初始化阶段的软件对齐。

问题二：为什么 `PWM_Motor_SetAngle()` 使用 `htim3.Init.Period + 1`。

当前 TIM2、TIM3、TIM4 的 Period 都是 3599，所以使用 TIM3 的 Period 作为统一基准可以成立。若后续改动某个定时器周期，需要同步检查这里。

问题三：`power_percent > 60` 时为什么变成 40。

这是当前源码行为。
教材不能把它解释成“限幅到 60”。
它更像一个保守降额处理，但具体调参意图证据不足，本章按源码记录。
在没有仓库外实测证据前，不能把这段逻辑描述成硬件保护结论。

问题四：头文件说 `MOTOR_ROLL` 对应 Mot0，为什么正文说 `MOTOR_PITCH` 写到 Mot0。

因为当前实际 `PWM_Motor_SetAngle()` 分支中，`MOTOR_PITCH` 写 TIM3 CH2/CH3/CH4，
并且源码注释称其为 P轴 -> MOT0。
头文件 Mot 注释与实现注释不一致，调试时以实际 CCR 写入为准。

问题五：为什么写 CCR 前要关闭中断。

同一电机的三相 CCR 最好一次性更新。如果更新 A 相后被中断打断，B/C 相延后写入，可能出现短暂的三相目标不一致。临界区可以降低这种风险。

问题六：TIM8 是否参与当前主电机输出。

当前主电机输出证据集中在 TIM2、TIM3、TIM4。TIM8 在 `Core/Src/tim.c` 中有 PWM 配置和中断配置，但本章主链路没有发现 `PWM_Motor_SetAngle()` 写 TIM8 CCR。

问题七：CCR 写入正确是否就表示电机一定会按预期转动。

不能。
CCR 写入只能证明 MCU 输出寄存器层面的目标值被更新。
电机是否响应、方向是否正确、三相是否接对、驱动能力是否足够，都需要仓库外点测。
因此本章结论停在“软件输出映射”，第33章再讨论下载后的硬件诊断。

问题八：当前 PWM 频率是多少。

根据当前工程配置，`APB1TimFreq=72MHz`、`PSC=0`、`ARR=3599`，
理论 PWM 频率为：

```text
72MHz / 3600 = 20kHz
```

这是配置推导值，不等于外部实测波形；真实引脚频率仍需测量【待验证】。

问题九：`Pulse=500` 是否就是运行时占空比。

不是。
它是 TIM 通道初始化时的初始比较值。运行后 `PWM_Motor_SetAngle()` 会直接写 `TIMx->CCRn`，
闭环输出应以当前 CCR 寄存器值为准。

问题十：控制轴使能关闭，是否等于 PWM 通道已经停止。

不是。
`eepromConfig.pitchEnabled`、`rollEnabled`、`yawEnabled` 控制的是
`computeMotorCommands()` 是否进入对应轴分支并继续调用 `PWM_Motor_SetAngle()`。
当前代码没有在这些标志关闭时调用 `HAL_TIM_PWM_Stop()`，也没有统一清零九路 CCR。
因此开机和延时接管阶段要把“PWM 已启动”和“闭环控制已使能”分开观察。

问题十一：`HAL_TIM_PWM_Start()` 都调用了，是否就证明九路 PWM 都启动成功。

当前源码没有逐项检查返回值，所以不能这样证明。
若要证明九路 PWM 都成功，需要记录返回值、输出使能位和实际引脚波形。

问题十二：`TIMx->CCRn` 已经写入，是否说明外部 PWM 波形已经切到新占空比。

不能直接这样说。
本地 HAL 的 PWM 配置会启用 `OCxPE`，写 CCR 后通常还要等对应定时器的 update event
把预装载值转为有效比较值。调试时应同时观察 `OCxPE`、`CNT`、`SR.UIF`
和是否存在主动 `EGR.UG`，再结合外部波形确认【待验证】。

问题十三：`PWM_Motor_SetAngle()` 传入无效 `axis` 会怎样。

当前 `switch(axis)` 没有 `default` 分支。
如果传入的值不是 `MOTOR_PITCH`、`MOTOR_ROLL` 或 `MOTOR_YAW`，
函数不会写任何 CCR，也不会返回错误。正常主链路传入的是固定枚举值，
但底层函数本身没有运行时轴参数校验。

问题十四：`__disable_irq()` / `__enable_irq()` 是否是嵌套安全临界区。

当前写法不是完整的嵌套安全封装。
它没有保存进入临界区前的 PRIMASK，而是在结束时直接 `__enable_irq()`。
若未来从已经关中断的上下文调用，可能改变调用者原本的中断状态。
需要嵌套安全时，应考虑 `__get_PRIMASK()` / `__set_PRIMASK()` 形式。

问题十五：GPIO 速度配置为 Low 会不会改变 PWM 频率。

不会改变定时器计数频率。
PWM 频率由定时器时钟、PSC 和 ARR 决定。
GPIO speed 影响的是引脚输出边沿速度和波形质量，实际边沿是否满足驱动器需求需要外部测量【待验证】。

问题十六：`PWM_Motor_Hardware_Diagnosis()` 是否能证明三轴硬件都已经完成点测。

不能。
当前只发现该函数的声明和定义，没有发现主程序或主控制路径调用证据。
函数内部也只有 1000ms 到期时才清零 CCR 并写入固定 `duty=1000`。
此外，当前 Pitch/Yaw 的单相写入多处被注释，未注释写入主要覆盖 Roll 三相。
因此它只能作为底层点测框架和映射旁证，不能写成完整三轴自动诊断结论。

如果把它临时接入主循环，还不能和闭环输出混在一起解释。诊断函数和
`computeMotorCommands()` 都可能写同一批 `TIMx->CCRn`；当前未注释的 Roll
物理点测通道还会被 Pitch 控制分支的 `PWM_Motor_SetAngle(MOTOR_ROLL, ...)`
覆盖。若示波器或寄存器窗口看到的值和预期不一致，应先确认最后写入者，
再判断是否存在硬件输出问题。

## 13. 本章总结

本章把第29章的电角和正弦表继续连接到 STM32 定时器输出。

当前项目中：

- TIM2、TIM3、TIM4 都用于主电机 PWM 输出。
- TIM2、TIM3、TIM4 当前都配置为 `Prescaler=0`、`Period=3599`。
- 结合 `.ioc` 中 `APB1TimFreq=72MHz`，当前理论 PWM 频率为 20kHz。
- `sConfigOC.Pulse=500` 是初始化初值，闭环运行时 CCR 由 `PWM_Motor_SetAngle()` 直接写入。
- 在当前 `ARR=3599` 下，`Pulse=500` 对应约 13.9% 的理论初始占空比，不等于 0% 输出。
- `PWM_Motor_Init()` 启动 9 个 PWM 通道，并通过临界区内连续清零 `TIM2->CNT`、`TIM3->CNT`、`TIM4->CNT` 减小初始相位错位。
- `PWM_Motor_Init()` 早于三轴软件使能关闭语句，启动窗口内应区分 PWM 硬件启动和闭环控制接管。
- 9 路 PWM 是逐个 `HAL_TIM_PWM_Start()` 后才清零 CNT；初始化 Pulse、CNT 对齐和首次运行 CCR 覆盖是三个不同时间点。
- 当前 `PWM_Motor_Init()` 没有逐项检查 9 次 `HAL_TIM_PWM_Start()` 的返回值。
- 当前 `PWM_Motor_Init()` 的 `HAL_OK` 和 `isInitialized=1` 只能证明驱动流程放行，不能证明 9 路 PWM 均已硬件启动成功。
- `PWM_Motor_SetAngle()` 根据电角、正弦表和功率百分比生成 `ccr_val[0..2]`。
- `PWM_Motor_SetAngle()` 的 `switch(axis)` 当前没有 `default` 分支，无效轴不会写 CCR，也不会报告错误。
- 控制分支名和 `MotorAxis_t` 枚举名不是一一同名映射：Roll 分支当前传入 `MOTOR_PITCH`，Pitch 分支当前传入 `MOTOR_ROLL`。
- `MOTOR_PITCH` 当前写 TIM3 CH2/CH3/CH4。
- `MOTOR_ROLL` 当前写 TIM3 CH1、TIM2 CH4、TIM2 CH3。
- `MOTOR_YAW` 当前写 TIM4 CH4、TIM2 CH2、TIM4 CH3。
- `drv_pwmMotors.h` 的 Mot 注释与 `drv_pwmMotors.c` 实际 CCR 写入存在不一致，教材以实际写寄存器路径为准。
- TIM2/TIM3/TIM4 的 PWM GPIO 当前配置为复用推挽低速输出。
- TIM8 当前不是 `PWM_Motor_SetAngle()` 的主输出目标。
- 当前临界区直接使用 `__disable_irq()` / `__enable_irq()`，没有保存/恢复 PRIMASK。
- HAL PWM 通道配置会启用 `OCxPE`，但当前没有跨定时器主从同步、DMA 批量更新或统一提交证据。
- `PWM_Motor_Hardware_Diagnosis()` 当前没有主程序调用证据，且未注释点测写入主要覆盖 Roll 三相；它是诊断旁证，不是主输出链路。
- `PWM_Motor_Hardware_Diagnosis()` 是主动写 CCR 的旁路诊断函数；若临时接入，必须确认它没有被 500Hz 闭环输出覆盖。
- 500Hz 控制帧约为 2ms，20kHz PWM 周期约为 50us，一次控制输出约跨越 40 个 PWM 周期。
- `TIMx->CCRn` 写入、预装载值转入 active compare、外部 PWM 波形改变、驱动器与电机响应，是四个需要分开验证的阶段。
- 当前 `.map` / `.list` 可以证明本章相关初始化函数、PWM 启动路径、CNT 清零、闭环调用和三轴 CCR 写入进入 Debug 固件。
- 当前 `.su` / `.cyclo` 可以证明 TIM 初始化、PWM 启动、通道配置和运行期 CCR 写入函数的静态栈与圈复杂度边界，但不能证明真实执行耗时或硬件输出同步。

本章边界：

- 本章证明软件层面的三相 CCR 写入映射，不证明真实电机相线顺序、方向或驱动能力正确。
- 控制分支名、枚举名、物理接口名和真实机械轴响应必须分开记录，不能用同名关系代替寄存器和实测证据。
- 临界区只能降低 CPU 写 CCR 被中断打断的风险，不能证明跨定时器硬件同步更新。
- 对跨 TIM 的 Roll/Yaw，临界区写入只能证明 CPU 完成三次 CCR store，不能证明三路比较值在同一 update event 生效。
- 当前临界区不是嵌套安全封装，未来若在已关中断上下文调用需要重新设计。
- 无效 `axis` 输入没有错误报告，底层函数缺少防御性分支。
- GPIO speed 只说明引脚输出配置，不证明外部边沿质量满足驱动器要求。
- `PWM_Motor_Hardware_Diagnosis()` 的存在不能证明硬件点测已经执行，也不能证明 Pitch/Yaw 已完成单相覆盖。
- 电机是否按预期响应，需要结合第33章下载后的仓库外点测记录验证。
- 20kHz PWM 频率是工程配置推导值，真实引脚波形频率仍需外部测量【待验证】。
- 构建产物证据只能证明符号、调用和指令路径进入当前 Debug 固件，不能证明上电执行顺序、九路 HAL 启动均成功、外部 PWM 波形或跨定时器硬件同步正确。
- 静态栈用量和圈复杂度只能作为资源审查入口，不能替代 DWT 计时、逻辑分析仪或真实负载下的同步更新验证。

下一章将进入 `三轴云台控制架构`，把传感器、姿态解算、500Hz 实时控制循环、PID 输出和本章的执行器链路合并成完整系统视图。
