## 3. 问题背景

500Hz 实时控制循环承担的是实时链路：

```text
读取传感器
-> 更新姿态
-> 计算控制量
-> 输出电机指令
```

但一个能运行的嵌入式控制系统，只靠高频链路还不够。它还需要回答这些问题：

- 开机后是否立刻允许电机参与控制。
- 某个轴是否已经完成回中，能否切换到后续控制状态。
- 调试信息多久输出一次，避免串口打印干扰实时任务。
- 姿态、时间步长、PID 输出一旦出现 `NaN` 或 `Inf`，应该在哪里处理。

本项目把这些问题分散在多个文件中：

- `Core/Src/main.c`
- `Drivers/SRC/Src/computeMotorCommands.c`
- `Drivers/SRC/Src/pid.c`
- `Drivers/SRC/Src/MargAHRS.c`

本章的任务是把它们整理成一条稳定性主线。

## 4. 核心概念

### 4.1 10Hz低频任务

`10Hz低频任务` 是主循环中每 100ms 执行一次的非实时任务入口。

项目中使用：

```text
HAL_GetTick() - last_print_tick >= 100
```

判断 100ms 是否到达。由于 `HAL_GetTick()` 依赖 SysTick 的 1ms 系统节拍，所以低频任务不是独立硬件中断，而是主循环中的轮询任务。

当前低频任务做两件事：

- 根据 `return_state_count` 和 `return_state_count_roll` 切换回中状态。
- 通过 `printf()` 输出 Roll 姿态和 Roll 回中计数。

这两件事的门控不同。源码中 `if (systemReady)` 只包住 Pitch/Roll 回中状态切换，
而 `printf("%.6f,%d\r\n", ...)` 位于这个 `if` 之后，仍在 100ms 轮询块内。
由于当前 `systemReady = true` 发生在进入 `while (1)` 之前，这个差异通常不容易被观察到；
但从架构教学上，必须区分“低频状态切换门控”和“低频打印路径”。

### 4.2 运行状态门控

`运行状态门控` 指用状态变量控制某段逻辑是否允许运行。

当前项目至少有四层门控：

- `systemReady`：控制 `SysTick_Handler()` 是否推进 500Hz 帧标志，也控制 10Hz 状态切换逻辑是否执行。
- `mpu6050Calibrating`：在 `SysTick_Handler()` 中与 `systemReady` 共同决定是否推进 500Hz 帧标志。
- `startup_delay_counter`：控制 AHRS 收敛等待和三轴使能时机。
- `eepromConfig.pitchEnabled`、`rollEnabled`、`yawEnabled`：控制 `computeMotorCommands()` 内部各轴分支是否运行。
- `return_state`、`return_state_roll`：控制 Pitch/Roll 处于回中路径还是后续控制路径。

这些门控并不等价。`systemReady` 面向系统初始化，`mpu6050Calibrating` 面向温度校准期间的帧暂停，
`startup_delay_counter` 面向开机等待，`eepromConfig.*Enabled` 面向轴级控制，
`return_state*` 面向轴内阶段切换。

还要注意当前主线边界：`main.c` 中 `mpu6050Calibration()` 调用处于注释状态。
因此 `mpu6050Calibrating` 是已实现的校准暂停门控，但当前主运行路径未看到温度校准被实际启动。

### 4.3 NaN与Inf防护

`NaN` 表示不是一个有效数值，`Inf` 表示无穷大。它们可能来自非法数学运算、异常传感器输入、过小时间步长导致的除法放大，或未被约束的中间量。

控制系统中，异常浮点值的危险不在于变量本身难看，而在于它会继续传播：

```text
异常姿态
-> 异常误差
-> 异常PID
-> 异常电角
-> 异常PWM输出
```

因此项目在姿态解算、PID 和分轴控制中多次检查 `isnan()` 和 `isinf()`。

### 4.4 时间触发与周期保证

`HAL_GetTick() - last_print_tick >= 100` 只能说明低频任务具备“到期触发条件”。
它不能单独证明任务严格每 100ms 执行一次。

原因是这个判断位于主循环中。只有当主循环运行到该判断语句时，
低频任务才有机会执行。如果前面的 500Hz 分支、I2C 读取或串口发送阻塞，
实际执行时间会晚于理论触发时间。

因此第32章要区分两句话：

```text
低频任务目标周期是100ms
低频任务实测间隔约为100ms【待验证】
```

第一句来自源码，第二句必须来自串口日志、GPIO翻转、逻辑分析仪或断点时间戳。

### 4.5 帧标志与丢帧合并

`frame_500Hz` 是布尔标志，不是递增队列。

`SysTick_Handler()` 到期时把它置为 true，主循环消费时再把它清为 false。
如果主循环正在执行 I2C 读取、串口打印或其他阻塞代码，多次 500Hz 触发可能都只表现为：

```text
frame_500Hz = true
```

也就是说，布尔标志能告诉主循环“至少有一个 500Hz 帧到期”，
但不能记录“阻塞期间累计错过了几个 500Hz 帧”。
这就是为什么系统级调试不能只看 `executionTime500Hz`。

`executionTime500Hz` 记录的是主循环开始消费某一帧后，500Hz 分支内部执行了多久。
`dt500Hz` 记录的是两次消费之间经过了多久。
如果主循环被 10Hz 打印或其他路径推迟，`executionTime500Hz` 可能仍然正常，
但 `dt500Hz` 会变大。

### 4.6 异常替代与安全停机

`异常替代` 是把异常输入或中间量替换为某个值，例如 0 或 0.002s。

`安全停机` 则要求控制输出进入确定的安全状态，例如停止更新电机、
关闭轴使能、清零 CCR 或关闭 PWM 输出。

当前项目中，多处代码体现的是异常替代：

```text
NaN姿态 -> 0
异常deltaT -> 0.002s
I2C读取失败 -> rawAccel/rawGyro清零
```

这些替代能阻断 `NaN` 或 `Inf` 的传播，但不自动等同于硬件安全停机。
教材必须保留这个边界，否则读者会误以为“有异常防护”就等于“电机一定安全”。

还要把三类执行层动作分开：

- 控制层停止更新：例如 `eepromConfig.*Enabled == false` 时，对应轴分支不再调用 `PWM_Motor_SetAngle()`。
- 控制量替代：例如把异常 PID 输出改成 0。
- 寄存器级关断：例如显式把相关 `TIMx->CCRn` 写成 0，或停止 PWM 通道。

这三者不是同义词。当前 `PWM_Motor_SetAngle()` 中，即使 `power_percent` 被限制到 0，
`current_amp` 变为 0 后三相 `ccr_val[]` 仍会等于 `half_period`，而不是全部等于 0。
因此“功率百分比为 0”“轴分支不执行”和“CCR 清零”在当前源码证据中是三个不同概念。
至于三相等占空比在实际驱动板上是否等效为无转矩或安全状态，需要结合硬件拓扑和仓库外实测，不能只凭函数名判断。

### 4.7 HAL_GetTick回绕与无符号减法

`HAL_GetTick()` 返回的是 `uint32_t`。本项目里的 `last_print_tick` 也是 `uint32_t`，
因此低频任务判断：

```text
HAL_GetTick() - last_print_tick >= 100
```

本质上是一次无符号整数差值比较。
HAL 源码中 `HAL_IncTick()` 会按 `uwTickFreq` 递增 `uwTick`，
而 `HAL_GetTick()` 直接返回这个 32 位计数值。
由于 `uwTickFreq` 默认是 1KHz，所以当前系统 tick 约为 1ms 一次。

这意味着，`HAL_GetTick()` 会在约 49.7 天后发生 32 位回绕：

```text
2^32 ms / 1000 / 60 / 60 / 24 ≈ 49.7 天
```

但只要比较阈值远小于 `2^31` 个 tick，`now - last` 这种写法通常仍能跨回绕工作。
本章的 100ms 轮询远小于这个边界，所以当前写法比直接比较绝对值更稳妥。

这也是工程里常见下面这种写法的原因：

```text
if (HAL_GetTick() - last >= interval)
```

而不是：

```text
if (HAL_GetTick() >= last + interval)
```

后者在 tick 回绕附近更容易出错。
不过这个结论只对无符号计数和足够小的间隔成立。
如果有人把时间戳改成有符号类型，或者把间隔设得接近回绕尺度，
结论就需要重新验证【待验证】。

## 5. 工作原理

### 5.1 低频任务为什么放在主循环

串口打印和状态切换不需要 500Hz 的频率。如果每个 2ms 控制帧都打印一次，会明显增加串口阻塞风险，也会让调试输出难以阅读。

项目选择用 100ms 轮询，是一种简单的分频方式：

```text
SysTick提供1ms时间基准
-> HAL_GetTick()返回毫秒计数
-> 主循环判断是否经过100ms
-> 执行低频状态逻辑和打印
```

这种方式的优点是实现简单，不需要新增调度器；代价是低频任务的实际执行时刻仍受主循环是否被阻塞影响。如果 500Hz 实时控制分支或串口发送耗时过长，10Hz 低频任务也会延后。

低频任务的实际触发间隔可以写成：

```text
T_actual = 100ms + T_delay
```

其中 `T_delay` 来自主循环走到 10Hz 判断之前的额外等待。
如果 500Hz 分支正在执行、I2C 阻塞读正在等待、或串口正在发送上一条日志，
`T_delay` 就可能增大。

由于代码在进入 10Hz 任务块后立刻使用当前 `HAL_GetTick()` 覆盖 `last_print_tick`，
下一轮 100ms 是从“本次低频任务开始执行的时刻”重新开始计时。
这让低频任务不会因为前面已经晚到 300ms 就连续补跑三次，但会产生周期漂移。

这种“不会补跑”的行为有好处，也有代价。

好处是，如果系统曾经卡住 350ms，恢复后不会立刻连续打印三四次，把主循环再次压住。
代价是，低频状态切换也会随实际执行时刻后移。
因此 `return_state_count >= 650` 只是“状态切换条件已经满足”，
真正翻转 `return_state` 仍要等主循环下一次运行到 10Hz 判断。

还要分清阻塞发生的位置。
如果阻塞发生在 10Hz 判断之前，当前写法会在恢复后只执行一次低频任务。
如果阻塞发生在 `last_print_tick` 已更新之后，例如本次 `printf()` 本身卡住超过 100ms，
那么函数返回后下一轮主循环可能很快再次满足：

```text
HAL_GetTick() - last_print_tick >= 100
```

所以低频任务的“不会补跑”不是无条件结论，而是与时间戳更新位置和阻塞位置有关。

### 5.2 启动门控的工作过程

项目初始化结束后先关闭三轴：

```text
pitchEnabled = false
rollEnabled = false
yawEnabled = false
```

进入 500Hz 实时控制分支后，系统先持续读取传感器并运行 `MargAHRSupdate()`。当：

```text
startup_delay_counter == 1000
```

时，代码执行：

```text
zeroPIDintegralError()
zeroPIDstates()
pitchEnabled = true
rollEnabled = false
yawEnabled = true
```

在 500Hz 下，1000 帧约等于 2 秒。这说明项目给 AHRS 一个收敛窗口，再打开 Pitch 和 Yaw。Roll 仍保持关闭，等待后续回中状态切换。

还要注意，`startup_delay_counter == 1000` 时的提示打印位于 500Hz 分支内部，而不是 10Hz 低频任务内部。
它只执行一次，但仍会被计入这一帧的 `executionTime500Hz`。
如果这次 `printf()` 阻塞时间较长，它可能让本帧耗时变大，并推迟主循环消费下一次 `frame_500Hz`。
因此调试启动过程时，不能只把串口阻塞风险理解为 10Hz 打印风险。

### 5.3 低频状态切换的工作过程

Pitch 回中计数在 500Hz 的 Pitch 分支中更新：

```text
fabs(sensors.margAttitude500Hz[PITCH]) < 0.3
```

满足时 `return_state_count` 增加，否则清零。

10Hz 低频任务再判断：

```text
return_state_count >= 650
pitchEnabled == true
```

如果满足，项目把 `return_state` 置为 false，并打开 Roll：

```text
rollEnabled = true
```

Roll 回中计数在 500Hz 的 Roll 分支中更新：

```text
fabsf(pointingCmd[ROLL] - sensors.margAttitude500Hz[ROLL]) < 0.2f
```

10Hz 低频任务再判断：

```text
return_state_count_roll >= 650
rollEnabled == true
```

如果满足，项目把 `return_state_roll` 置为 false。

因此，回中状态不是单一频率完成的：计数更新发生在 500Hz 实时控制分支，状态翻转发生在 10Hz 低频任务。

还要再拆一层证据边界。10Hz 低频任务提交的是状态变量，不是执行层输出：

```text
500Hz分支更新 return_state_count*
-> 10Hz分支提交 return_state*/rollEnabled
-> 后续500Hz分支重新读取门控
-> computeMotorCommands() 进入新的控制路径
-> PWM_Motor_SetAngle() 写入新的CCR组合
```

所以，`return_state = false`、`rollEnabled = true` 或 `return_state_roll = false`
只能证明状态提交已经发生，不能单独证明 PWM 输出在同一时刻已经改变。
如果 10Hz 状态提交发生在本轮 500Hz 分支之后，新的门控效果至少要等下一次被主循环消费的 500Hz 帧才能进入 `computeMotorCommands()`。
又因为 `frame_500Hz` 是布尔标志，主循环阻塞时多个理论帧可能合并为一次消费，
所以调试记录不能只写“第650帧立刻切换输出”，而应写清楚“计数达标、10Hz提交、下一次500Hz读取、首次CCR变化”四个时刻。

### 5.3.1 状态提交与输出生效边界

这一段逻辑可以拆成四层证据，避免把变量翻转误读成硬件输出已经变化。

| 阶段 | 触发位置 | 当前证据能证明什么 | 当前证据不能证明什么 |
|---|---|---|---|
| 计数达标 | `computeMotorCommands()` 的 500Hz Pitch/Roll 分支 | `return_state_count*` 已在对应角度窗口内连续累加 | 不能证明 `return_state*` 已经翻转，也不能证明 Roll 轴已经被打开 |
| 10Hz 提交 | `main.c` 的 100ms 低频块 | `return_state = false`、`rollEnabled = true` 或 `return_state_roll = false` 已被写入 | 不能证明本轮或上一轮 500Hz 分支已经重新读取这些新值 |
| 后续读取 | 下一次被主循环消费的 500Hz 分支 | `computeMotorCommands()` 有机会按新的 `eepromConfig.*Enabled` 与 `return_state*` 进入新路径 | 不能证明一定出现 CCR 变化，仍需看执行层是否调用 `PWM_Motor_SetAngle()` |
| 执行层变化 | `PWM_Motor_SetAngle()` 写入 TIM 比较寄存器路径 | 软件层已经把新的电角度和功率转换为 CCR 写入意图 | 不能证明外部电机已经运动，真实力矩、堵转和机械响应仍属 `【待验证】` |

因此，调试表最好同时记录 `HAL_GetTick()`、500Hz 帧消费时刻、`return_state*`、
`rollEnabled`、`PWM_Motor_SetAngle()` 调用次数以及关键 CCR 值。
这样才能把“状态机提交成功”和“执行层输出真的发生变化”分开验收。

### 5.4 异常值防护的分层

项目中的 `NaN与Inf防护` 不是一个集中函数，而是分层防护。

第一层在姿态解算入口。`MargAHRSupdate()` 会把异常的 `gx`、`gy`、`gz`、`ax`、`ay`、`az` 置为 0。

第二层在四元数归一化。若 `norm_q < 1e-10f` 或 `norm_q` 为 `NaN`，项目把四元数重置为单位姿态。

第三层在姿态输出。Roll、Pitch、Yaw 计算结果若为 `NaN`，被置为 0。

第四层在 PID。`updatePID()` 会把异常或超范围的 `deltaT` 回退为 0.002s，把异常 D 项、滤波 D 项和输出置为 0。

第五层在分轴控制。Roll、Pitch、Yaw 分支分别对角度或 PID 输出做异常检查，再进入限幅和电角合成。

这里有一个容易跳过的差异：PID 层显式检查 `deltaT`，但 `MargAHRSupdate()` 当前没有看到同样的
`dt` 范围回退。AHRS 中直接执行：

```text
halfT = dt * 0.5f
```

随后再通过四元数模长检查和姿态输出 `NaN` 清零兜底。
因此异常 `dt500Hz` 不是“只要进了 PID 就已经被全链路修好”。
它可能已经先影响 AHRS 积分，再在 PID 层被局部回退为 0.002s。
调试时应把 AHRS 的 `dt` 入口和 PID 的 `deltaT` 入口分开记录。

### 5.5 异常替代的传播路径

异常替代不是只影响当前函数。

例如 MPU6050 读取失败时，`MPU6050_Read_And_Process()` 会清零：

```text
rawAccel[]
rawGyro[]
```

随后 `main.c` 仍会把这些值缩放到：

```text
sensors.accel500Hz[]
sensors.gyro500Hz[]
```

注意，这一步不是简单地把 `raw` 零值复制成 `sensors` 零值。
`main.c` 还会执行零偏扣除：

```text
sensors = (raw - bias) * scale
```

因此读取失败后的 `raw` 清零，可能变成带零偏项的物理量输入。
这些输入仍会传给 `MargAHRSupdate()`。如果加速度向量模长低于阈值，
当前 AHRS 中的加速度校正分支不会执行；如果陀螺输入也接近零，
姿态可能保持上一状态，而不是自动进入安全状态。

因此读取失败后的“清零”是数据替代策略，不是自动退出 500Hz 控制链。
若要实现真正的传感器失效安全门控，还需要明确健康标志、连续失败计数、
轴使能关闭策略和 PWM 输出处理策略。

### 5.6 当前门控边界

项目中声明了：

```text
mpu_ok = false
```

并在 MPU6050 初始化成功或失败时打印信息。但当前证据没有看到 `mpu_ok` 被置为 true，也没有看到它参与 `computeMotorCommands()` 或 500Hz 实时控制分支门控。

因此，不能写成“MPU6050 初始化失败会自动阻止控制链路运行”。更稳妥的结论是：当前代码有初始化结果提示，但没有看到基于 `mpu_ok` 的运行门控证据。这是后续工程健壮性可以继续检查的点。

还要把 `Error_Handler()` 与运行期异常防护分开。项目中很多 HAL 初始化或配置失败路径会调用 `Error_Handler()`，
进入关闭中断后的无限循环；但 MPU6050 初始化失败路径当前只是打印失败提示，并继续走到后续初始化代码。
运行期 MPU 读取失败也没有调用 `Error_Handler()`，而是清零 raw 数据后返回。

所以 `Error_Handler()` 不是本项目所有异常的统一安全停机机制。
教材应把它写成“部分初始化/配置错误的停止路径”，而不是“传感器运行异常的通用保护路径”。

轴使能也不能直接写成 PWM 关断。当前 `computeMotorCommands()` 的三轴门控只决定是否进入对应轴分支；
如果某个轴未使能，该轴不会执行 PID、限幅、电角合成和 `PWM_Motor_SetAngle()`，
但源码中没有看到这个分支自动把该轴对应的 `TIMx->CCRn` 清零。

这还要结合初始化顺序理解。`tim.c` 中 TIM2、TIM3、TIM4 的相关 PWM 通道初始化时使用：

```text
ARR = 3599
Pulse = 500
```

若按边沿对齐 PWM1 的计数口径估算，初始比较值占周期计数的比例约为：

```text
500 / (3599 + 1) ≈ 13.9%
```

而 `main.c` 在 `init_eepromConfig(true)` 后调用 `PWM_Motor_Init()` 启动 PWM 通道，
再显式设置 `eepromConfig.pitchEnabled = false`、`rollEnabled = false`、`yawEnabled = false`。
这说明轴使能关闭主要是控制层停止闭环更新的门控；它不等价于“PWM 输出寄存器已经清零”。
当前源码中明确全通道写 0 的路径主要出现在 `PWM_Motor_Hardware_Diagnosis()` 的诊断分支，
且该函数不是当前主控制路径的一部分。

还要单独说明 `mpu6050Calibrating`。温度校准函数 `mpu6050Calibration()` 内部会：

```text
mpu6050Calibrating = true
...
mpu6050Calibrating = false
```

而 `SysTick_Handler()` 只有在 `mpu6050Calibrating == false` 时才产生 500Hz 帧标志。
因此，如果温度校准函数被实际调用，它会暂停常规 500Hz 控制帧生成。

不过这里还存在一个注释与实现不一致的边界：`main.c` 里的提示字符串写成“约 2 分钟”，
而 `mpu6050Calibration.c` 中实际调用的是 `HAL_Delay(10000)`，按代码字面量只等于约 10 秒。
因此，教材不能把“2 分钟校准暂停”写成当前源码已经证明的时长，只能写成“代码注释和实现存在不一致，需要实测或重构确认”的边界【待验证】。

但当前 `main.c` 中 `mpu6050Calibration()` 调用处于注释状态。
所以本章不能写成“开机一定会进入温度校准并暂停控制帧”。
更准确的说法是：校准暂停门控已经存在，但当前主线没有启用该校准流程。

### 5.7 串口打印耗时估算

USART3 当前配置为 115200 baud、8 位数据、无校验、1 位停止位。
常见 8N1 串口帧每个字符约占 10 bit：

```text
T_char = 10 / 115200 ≈ 86.8us
```

若一次 `printf()` 发送 20 个字符，仅线缆传输时间约为：

```text
T_20char = 20 * 86.8us ≈ 1.74ms
```

当前 `_write()` 对每个字符单独调用一次 `HAL_UART_Transmit()`。
所以真实 CPU 占用还包含每次 HAL 调用、等待 TXE/TC 标志和函数开销。

当前 10Hz 打印格式是：

```text
printf("%.6f,%d\r\n", sensors.margAttitude500Hz[ROLL], return_state_count_roll)
```

若输出类似：

```text
-1.234567,650\r\n
```

字符数约为 15 个，单看串口线速时间约为：

```text
15 * 86.8us ≈ 1.30ms
```

这还没有包含 HAL 轮询等待、函数调用和 `printf()` 浮点格式化开销。
若 UART 标志异常，`HAL_UART_Transmit()` 还会按传入的 `0xFFFF` 超时参数等待。
当前 `_write()` 是逐字符发送，所以最坏等待时间会随字符数放大。
这个 `0xFFFF`  timeout 参数在 HAL 阻塞发送路径中按 HAL tick 参与超时判断，
通常应按毫秒级预算理解。它不是说每个字符一定会等待这么久，
而是说明一旦 UART 状态标志不按预期变化，逐字符阻塞发送存在很大的最坏情况等待上界。

这解释了为什么打印即使放在 10Hz，也仍然需要纳入系统时间预算。
它不一定破坏每一帧 500Hz 控制，但可能推迟下一次 `frame_500Hz` 消费。

### 5.8 两个时间指标的诊断分工

`dt500Hz` 和 `executionTime500Hz` 不回答同一个问题。

`dt500Hz` 回答：

```text
两次500Hz分支消费之间隔了多久
```

`executionTime500Hz` 回答：

```text
本次500Hz分支内部执行花了多久
```

如果 `executionTime500Hz` 接近 2ms，说明 500Hz 分支本身已经吃满周期。
如果 `executionTime500Hz` 正常但 `dt500Hz` 偶尔远大于 0.002s，
更像是主循环在两次消费之间被低频打印、I2C 阻塞或其他代码推迟。

这一区分非常重要。只看 `executionTime500Hz`，可能漏掉布尔帧标志被合并后的调度延迟。

### 5.9 轮询比较的边界稳定性

第4.7节说明了 `HAL_GetTick() - last_print_tick` 为什么适合做 10Hz 轮询。
这里再补一个工程角度：`print_interval` 变量在源码中已经定义，
但当前判断仍直接写成字面量 100。

这不是功能错误，但它说明当前代码把“周期参数”分散成了两个来源：

```text
const uint32_t print_interval = 100
HAL_GetTick() - last_print_tick >= 100
```

若未来要改成可配置周期，最好把判断统一回 `print_interval`。
否则注释、常量和实际判断容易分叉。
当前教材只能把它写成“存在命名常量，但实际判断仍使用字面量”的可维护性边界，
不能写成已经完成参数化设计。

## 6. STM32实现机制

从 STM32 实现角度看，本章机制依赖五个基础资源。

第一，SysTick 和 HAL 毫秒计数。第09章已经确认 `HAL_GetTick()` 来自 HAL 毫秒时基；本章只使用它解释 10Hz 低频任务的 100ms 轮询入口，不重新展开 500Hz 实时控制循环的微秒时间测量。

在当前项目中，SysTick 还有一个状态门控用途：`SysTick_Handler()` 先调用 `HAL_IncTick()`，
然后只有在 `systemReady == true && mpu6050Calibrating == false` 时才推进 `frameCounter`。
这意味着 HAL 毫秒 tick 会继续走，但 500Hz 控制帧是否生成还受系统就绪和校准状态约束。

第二，USART3 阻塞发送。`Core/Src/usart.c` 中 `fputc()` 和 `_write()` 最终都调用：

```text
HAL_UART_Transmit(&huart3, ..., 0xFFFF)
```

这意味着 `printf()` 会通过 USART3 发送，并且当前实现是阻塞式发送。把打印放在 10Hz 低频任务中，可以降低它对 500Hz 实时控制路径的干扰。

还要注意，本工程同时存在两层重定向证据：

- `Core/Src/usart.c` 中定义了 `fputc()`，并在 `#ifdef __GNUC__` 下定义了 `_write()`。
- `Core/Src/syscalls.c` 中也有 weak `_write()`，它会调用 weak `__io_putchar()`。

链接时应以实际参与链接的强符号为准。当前 `usart.c` 中的 `_write()` 是更直接的证据，
因为它逐字符调用 `HAL_UART_Transmit(&huart3, ..., 1, 0xFFFF)`。

第三，C 标准数学异常检查。项目在多个 `.c` 文件中使用 `isnan()` 和 `isinf()` 判断浮点异常。这些检查不是 STM32 专有外设功能，而是嵌入式 C 浮点控制链路中的工程保护。

这类检查在不同层级并不完全一致。PID 层对 `deltaT` 有范围、`NaN` 和 `Inf` 回退；
AHRS 层对陀螺仪和加速度计输入有 `NaN`/`Inf` 清零，但当前没有看到对 `dt` 的同类范围回退。
因此异常时间步长要同时从 AHRS 和 PID 两个入口检查。

第四，I2C 阻塞读。`MPU_Read_Reg()` 调用：

```text
HAL_I2C_Mem_Read(..., 100)
```

本地 HAL I2C 源码中，`HAL_I2C_Mem_Read()` 会在多个阶段等待 BUSY、ADDR、
RXNE、BTF 等标志，并用 `HAL_GetTick()` 检查超时。
因此 MPU6050 读取是 500Hz 分支中的潜在阻塞点。

第五，HAL 提供阻塞、IT 和 DMA 三类通信模式。当前主线使用阻塞式 UART 发送和阻塞式 I2C 读取。
HAL 源码中存在 `HAL_UART_Transmit_IT()`、`HAL_UART_Transmit_DMA()`、
`HAL_I2C_Mem_Read_IT()` 和 `HAL_I2C_Mem_Read_DMA()` 等非阻塞入口。
本章只记录当前项目采用的阻塞路径，不把 IT/DMA 写成已接入主链路。

本章涉及的项目证据文件包括：

- `Core/Src/main.c`

- `Drivers/SRC/Src/computeMotorCommands.c`

```text

```

以及：

```text
SysTick_Handler()
```

以及：

```text

MargAHRSupdate()
```

```text
MPU6050_Read_And_Process()
```

此外，还要把帧标志链路单独记录：

```text
SysTick_Handler()

-> dt500Hz记录两次消费间隔
```

### 8.1 10Hz低频任务入口

`Core/Src/main.c` 中声明：

```text
last_print_tick = 0
print_interval = 100
```

主循环中实际判断使用的是字面量 100：

```text
HAL_GetTick() - last_print_tick >= 100
```

这表示当前 10Hz 周期由代码中的 100ms 判断决定，而不是由 `print_interval` 常量参与。教材不能写成 `print_interval` 当前驱动了低频任务周期。

进入 100ms 任务块后，源码先更新：

```text
last_print_tick = HAL_GetTick()
```

然后才执行 `if (systemReady)` 状态切换和后续 `printf()`。
因此本章描述低频周期时，应以“低频任务开始时刻”作为时间戳刷新点。
如果打印本身阻塞很久，下一次低频判断可能比“打印结束后再等 100ms”更早触发。

#### 8.1.1 10Hz块内部顺序证据边界

第4.4节已经区分“目标周期是 100ms”和“实测间隔约为 100ms【待验证】”。本节不再重复周期证明，而是只看 10Hz 块内部顺序：`Core/Src/main.c` 和 `Debug/Three-axis_cloud_platformV2.list` 能证明 CPU 语句先后，不能直接证明外部日志或控制输出已经完成。

| 顺序 | 源码动作 | 当前证据能证明什么 | 不能证明什么 |
|---|---|---|---|
| 1 | `HAL_GetTick() - last_print_tick >= 100` | 主循环采用 100ms 差值判断进入低频块；周期边界见第4.4节 | 不能证明真实低频间隔严格等于 100ms |
| 2 | `last_print_tick = HAL_GetTick()` | 时间戳在低频块开头刷新，`.list` 中保留第二次 `HAL_GetTick()` 调用 | 不能证明后续 `printf()` 已完成，也不能消除阻塞造成的周期漂移 |
| 3 | `if (systemReady)` 内提交 `return_state*` / `rollEnabled` | `systemReady` 门控的是 Pitch/Roll 状态切换逻辑 | 不能证明整个 10Hz 块都被 `systemReady` 包住 |
| 4 | `printf("%.6f,%d\r\n", ...)` | 源码存在低频打印入口 | 不能证明主机已经收到完整日志，也不能证明打印不会推迟下一次 500Hz 消费 |

因此，调试低频任务时应分别记录“进入低频块”“提交状态变量”“开始打印”“打印完成”四个时刻。只看到串口上出现一行日志，不能倒推出本轮状态提交一定已经按预期影响了后续 500Hz 控制帧。

### 8.2 systemReady门控

初始化结束后，代码执行：

```text
systemReady = true
```

它首先影响 SysTick 中的 500Hz 帧标志生成。`Core/Src/stm32f1xx_it.c` 中只有满足：

```text
systemReady == true && mpu6050Calibrating == false
```

才会推进 `frameCounter` 并产生 `frame_500Hz`。也就是说，`systemReady` 不是只影响低频任务，它还决定初始化完成前是否允许实时控制帧开始调度。

10Hz 低频任务内部也会判断：

```text
if (systemReady)
```

只有该标志为 true 时，才执行回中状态切换逻辑。

但 `printf("%.6f,%d\r\n", ...)` 位于这个 `if (systemReady)` 之后。
所以严格说，`systemReady` 门控的是 10Hz 状态切换，不是整个 10Hz 任务块中的所有语句。

注意，`systemReady` 当前不是传感器健康状态，也不是 AHRS 收敛状态。
它更接近“初始化流程走到 USER CODE BEGIN 2 末尾”的系统就绪标志。

`mpu6050Calibrating` 也在 `SysTick_Handler()` 中参与 500Hz 帧生成判断。
但当前 `main.c` 中温度校准调用被注释，常规开机主线没有实际进入
`mpu6050Calibration()` 的证据。这个变量应写成“已实现但当前主线未触发的校准暂停门控”。

### 8.3 startup_delay_counter门控

500Hz 实时控制分支每帧检查：

```text
startup_delay_counter < 1000
```

计数达到 1000 时，项目清空 PID 积分和 D 项历史，并打开 Pitch/Yaw。

这一段门控的位置很关键：它发生在 `MargAHRSupdate()` 之后、`computeMotorCommands()` 之前。也就是说，AHRS 在门控等待阶段仍然运行，而控制输出由轴使能状态限制。

同一代码块里还有一条启动完成提示：

```text
printf(">>> AHRS 收敛完成，俯仰轴控制已使能。\r\n")
```

这条打印不在 10Hz 低频任务中，而在 500Hz 分支中，并且位于 `computeMotorCommands()` 之前。
因此它是一次性启动事件，不是周期性低频打印；但它仍会进入这一帧的耗时预算。
调试 `executionTime500Hz` 的启动尖峰时，应把这条提示打印也纳入排查范围。

### 8.4 eepromConfig轴使能门控

`computeMotorCommands()` 内部分轴判断：

```text
if (eepromConfig.rollEnabled == true)
if (eepromConfig.pitchEnabled == true)
if (eepromConfig.yawEnabled == true)
```

某个轴未使能时，该轴分支不会执行 PID、限幅、电角合成和 `PWM_Motor_SetAngle()`。

这也是为什么第31章强调三轴不是开机同时闭环。第32章进一步说明：轴使能是运行门控，不只是配置项。

但从安全边界看，还要再拆一层。`PWM_Motor_Init()` 已经启动 TIM2、TIM3、TIM4 的 PWM 通道；
`tim.c` 中这些通道的初始 `Pulse` 为 500。轴未使能只表示本轮控制层不会改写该轴目标，
并不能单独证明相关 CCR 已经是 0，也不能证明 PWM 通道已停止。

因此调试“某轴未使能但接口仍有波形”时，不应只盯着 `eepromConfig.*Enabled`。
还要同时观察：

- 该轴最近一次是否调用过 `PWM_Motor_SetAngle()`。
- 对应 `TIMx->CCRn` 当前值是多少。
- PWM 通道是否已经由 `HAL_TIM_PWM_Start()` 启动。
- 是否存在显式清零 CCR 或停止 PWM 的代码路径。

### 8.5 Pitch/Roll回中门控

Pitch 分支中，`return_state` 决定电角计算方向和后续参数设置：

```text
return_state == true
return_state == false
```

Roll 分支中，`return_state_roll` 决定是回中输出路径，还是后续 PID 控制路径。

两个状态的切换判断放在 10Hz 低频任务中，而计数数据来自 500Hz 实时控制分支。这个跨频率设计要重点调试，否则读者可能只看 10Hz 代码，却找不到计数为何变化。

#### 8.5.1 跨频率状态提交的构建证据边界

Pitch/Roll 回中门控要拆成“计数更新”和“状态提交”两个最小单元。当前源码中，`return_state_count` 和 `return_state_count_roll` 在 `computeMotorCommands()` 的 500Hz 控制路径中更新；`return_state = false`、`eepromConfig.rollEnabled = true` 和 `return_state_roll = false` 则在 `main.c` 的 100ms 低频块中提交。

| 观察对象 | 源码证据 | 构建产物能辅助证明什么 | 不能证明什么 |
|---|---|---|---|
| 500Hz 先消费 | `main.c` 中 `if (frame_500Hz)` 位于 10Hz 判断之前，并在其中调用 `computeMotorCommands(dt500Hz)` | `.map` 可证明 `computeMotorCommands` 进入当前 Debug 产物，`.list` 可辅助定位已有产物中的 500Hz 分支和调用点 | 不能证明每个理论 2ms 帧都被消费，也不能证明 `.list` 一定与当前源码完全同步 |
| 计数在控制路径更新 | `computeMotorCommands()` 内根据 Pitch/Roll 角度窗口更新 `return_state_count*` | 源码能证明计数依赖控制帧内的姿态误差判断 | `.map` 不足以还原每一帧计数值，`.list` 也不能证明运行时连续满足阈值 |
| 10Hz 提交状态 | `main.c` 的 100ms 块写入 `return_state*` 和 `rollEnabled` | 源码能证明状态提交发生在低频块，而不是在计数达标那一行自动发生 | 不能证明提交后本轮已经重新进入 `computeMotorCommands()`，因为本轮 500Hz 分支在源码顺序上已经执行完 |
| 下一帧可见 | 后续再次消费 `frame_500Hz` 时，`computeMotorCommands()` 才会重新读取 `eepromConfig.*Enabled` 与 `return_state*` | 可以作为断点和日志设计依据：记录“计数达标、10Hz提交、下一次500Hz读取、首次输出变化” | 不能证明外部电机立即响应，也不能证明 CCR 或相电流已经变化【待验证】 |

因此，本章可以说“10Hz 低频任务提交了状态变量”，但不能写成“10Hz 判断立即改变已经执行过的 500Hz 输出”。若要把状态提交和执行层输出关联起来，至少需要同一轮日志或断点同时记录 `HAL_GetTick()`、500Hz 帧消费序号、`return_state*`、`rollEnabled`、`PWM_Motor_SetAngle()` 调用和对应 TIM CCR 值。

### 8.6 PID异常防护

`updatePID()` 对时间步长先做保护：

```text
deltaT > 0.01f
deltaT < 0.0001f
isnan(deltaT)
isinf(deltaT)
```

满足任一条件时，`deltaT` 回退为 0.002f。

随后，D 项、滤波后的 D 项和最终输出都各自检查异常。D 项还会被限制在：

```text
-300.0f 到 300.0f
```

这说明 PID 防护既处理非法浮点值，也处理过大的瞬态微分。

### 8.7 姿态解算异常防护

`MargAHRSupdate()` 的入口先清理陀螺仪和加速度计输入异常。四元数更新后，再检查四元数模长：

```text
norm_q < 1e-10f
isnan(norm_q)
```

如果异常，就重置为：

```text
q0 = 1.0f
q1 = q2 = q3 = 0.0f
```

最后，Roll、Pitch、Yaw 姿态角如果出现 `NaN`，会被置为 0。

边界也要说清楚：当前输出姿态角只显式检查 `NaN`，没有看到对姿态角结果的 `Inf` 检查；输入和 PID 层才同时检查 `NaN` 与 `Inf`。

另一个边界是 `dt`。`MargAHRSupdate()` 当前直接使用：

```text
halfT = dt * 0.5f
```

没有看到像 `updatePID()` 那样把异常或超范围 `dt` 回退为 0.002f 的代码。
因此 `dt500Hz` 异常时，AHRS 和 PID 的防护行为不同：

```text
AHRS: dt直接参与四元数积分，后续靠四元数/姿态NaN防护兜底
PID : deltaT先经过范围和NaN/Inf检查，不合格则回退到0.002f
```

教材不能把 PID 的 `deltaT` 回退泛化成整个控制链的统一时间步长保护。

### 8.8 MPU读取失败的数据替代

`MPU6050_Read_And_Process()` 调用 `MPU_Read_Reg()` 读取 14 字节 MPU6050 数据。

如果读取失败，函数执行：

```text
memset(rawAccel, 0, sizeof(rawAccel))
memset(rawGyro, 0, sizeof(rawGyro))
return
```

这会阻止旧的 raw 数据继续被使用，但并不会自动关闭三轴控制。
`main.c` 后续仍会从 `rawAccel[]`、`rawGyro[]` 缩放到 `sensors.*500Hz[]`，
并继续进入 `MargAHRSupdate()` 和 `computeMotorCommands()`。

还要注意，`sensors.*500Hz[]` 的计算包含零偏扣除和比例缩放。
所以 `rawAccel[]`、`rawGyro[]` 清零后，控制链看到的物理量不一定全为 0。

因此这段代码应写成“读取失败后的数据替代”，不能写成“读取失败后的安全停机”。

### 8.9 Error_Handler与运行期异常的边界

`Error_Handler()` 的实现是：

```text
__disable_irq()
while (1)
```

它会关闭中断并停在无限循环中。当前工程里，时钟、I2C、USART、TIM 等 HAL 初始化或配置失败路径会调用它。
这些路径属于初始化或外设配置错误处理。

但 MPU6050 初始化失败路径不同。`main.c` 中 `MPU6050_Init()` 返回失败后只打印：

```text
>>> MPU6050 初始化失败。
```

之后代码仍会继续初始化姿态控制状态，并最终执行 `systemReady = true`。
此外，`MPU6050_Read_And_Process()` 的运行期读取失败也不是进入 `Error_Handler()`，
而是清零 `rawAccel[]`、`rawGyro[]` 后返回。

因此，第32章不能把 `Error_Handler()` 写成“所有异常都会停机”的安全机制。
更准确的写法是：部分 HAL 初始化/配置失败会停在 `Error_Handler()`；
MPU 初始化失败和运行期读取失败当前更接近提示或数据替代路径，是否继续输出要继续看门控和 PWM 状态。

### 8.10 UART阻塞发送路径

USART3 当前参数是：

```text
BaudRate = 115200
WordLength = 8B
StopBits = 1
Parity = NONE
```

`_write()` 中按字符循环：

```text
for each char:
    HAL_UART_Transmit(&huart3, &char, 1, 0xFFFF)
```

本地 HAL UART 源码把 `HAL_UART_Transmit()` 标为 blocking mode，
并在函数内部等待 `UART_FLAG_TXE` 和 `UART_FLAG_TC`。

这意味着 10Hz 打印不是零成本任务。若打印内容变长，
或串口状态异常导致等待变久，主循环到达下一次 500Hz 消费的时间会被推迟。

当前 `usart.c::_write()` 是强定义；`syscalls.c::_write()` 带 weak 属性。
因此分析 `printf()` 时，应优先检查最终链接进镜像的是哪个 `_write()`。
从源码意图看，`usart.c` 中的强定义用于兼容 GCC/AC6，并直接绑定 USART3。

`HAL_UART_Transmit()` 的 timeout 参数为 `0xFFFF`。
在正常串口状态下，单字符发送主要受波特率、TXE/TC 等标志等待和函数开销影响；
在 UART 标志卡住或外设状态异常时，timeout 才会成为最坏等待边界。
由于 `_write()` 逐字符调用阻塞发送，最坏等待不是“一次 printf 一个 timeout”，
而是可能按字符数重复触发 timeout【待验证】。

#### 8.10.1 printf调用链的符号级边界

还要把源码级重定向和最终链接路径分开。`Core/Src/usart.c` 同时定义了 `fputc()` 和
`_write()`，但当前 Debug 构建产物中，`printf()` 的底层写出路径更直接的证据是
newlib 的 `_write_r()` 再调用最终链接的 `_write()`。

当前 `.map` / `.list` 能支持下面这条链：

```text
main.c::printf(...)
-> newlib nano printf / _printf_float
-> _write_r              0x0800ffd8
-> _write                0x0800257c  来自 ./Core/Src/usart.o
-> HAL_UART_Transmit     0x0800a3ee
```

这里有三个容易混淆的边界。

第一，`Core/Src/syscalls.c` 里的 `_write()` 是 weak 定义，并且在当前 `.map`
中对应输入段位于 `0x00000000`；它只能证明工程里存在默认 syscalls 模板，
不能作为最终串口输出路径的证据。最终 Flash 地址区的 `_write()` 来自
`Core/Src/usart.o`，所以分析 10Hz 打印时应以 `usart.c::_write()` 为准。

第二，`fputc()` 的存在不等于当前 `printf()` 必然先走 `fputc()`。
当前构建产物里可见 `_write_r` 调用 `_write`，而 `usart.c::_write()` 又逐字符调用
`HAL_UART_Transmit(&huart3, ..., 1, 0xFFFF)`。因此，教材应把 `fputc()` 写成源码中存在的
兼容重定向入口，把 `_write_r -> _write` 写成当前 Debug 产物能直接证明的 newlib 输出路径。

第三，`.map` 中出现 `HAL_UART_Transmit_IT()` 和 `HAL_UART_Transmit_DMA()` 只能证明这些
HAL 函数也进入了构建产物或库解析范围，不能证明 10Hz 打印已经改为中断或 DMA 发送。
只要 `_write()` 仍然调用阻塞式 `HAL_UART_Transmit()`，低频打印的实时影响就要按阻塞发送路径分析。

#### 8.10.2 printf输出完成度证据边界

调试时还要把“执行了 `printf()`”继续拆细。当前仓库能证明的是一条逐层下沉的输出路径，
而不是“串口助手一定收到了完整一行数据”：

| 层级 | 当前证据 | 能证明什么 | 不能证明什么 |
|---|---|---|---|
| 格式化入口 | `main.c` 中 10Hz 分支调用 `printf("%.6f,%d\r\n", ...)`，Debug `.map` 中可见 `printf`、`_printf_float` 和 newlib nano printf 相关符号 | 当前输出路径包含浮点格式化入口，格式化本身会占用 CPU 时间 | 不能证明格式化耗时已经被实测，也不能证明输出字符串长度恒定 |
| 重定向选择 | `Core/Src/usart.c` 提供强 `_write()`；`Core/Src/syscalls.c` 的 `_write()` 为 weak；`.map` 中最终地址区保留 `./Core/Src/usart.o` 的 `_write`，`syscalls.o` 的 `_write` 位于 `0x00000000` 输入段 | 当前 Debug 产物最终使用 USART3 版本 `_write()`，而不是 weak 空实现 | 不能证明板上已经下载该 ELF，也不能证明主机端串口配置正确 |
| 逐字符发送 | `usart.c::_write()` 对 `len` 中每个字符调用一次 `HAL_UART_Transmit(&huart3, ..., 1, 0xFFFF)`；`.list` 保留该调用 | 一行 `printf` 会被拆成多次 1 字节阻塞发送 | 不能把 timeout 当成整行一次性边界，异常时等待可能随字符数放大【待验证】 |
| HAL阻塞等待 | `HAL_UART_Transmit()` 内等待 `UART_FLAG_TXE`，最后等待 `UART_FLAG_TC`；`UART_WaitOnFlagUntilTimeout()` 用 `HAL_GetTick() - Tickstart` 判断超时 | HAL 层确实是轮询标志的 blocking mode，可能占用主循环时间 | 不能证明每个字符实际等待了多少微秒；需要 DWT、GPIO 或串口时间戳测量 |
| 外部接收结果 | 只能由串口助手日志、逻辑分析仪或主机侧时间戳证明 | 可以验证实际波特率、字符丢失、行间隔和阻塞尖峰 | 仓库源码和 `.map/.list` 不能证明 PC 已经收到完整行【待验证】 |

这张表的工程意义是：第32章讨论的不是“能不能打印”，而是“打印会不会影响实时控制”。
源码和构建产物能证明阻塞发送路径存在；真正的性能结论仍要看 `dt500Hz`、`executionTime500Hz`、
10Hz 行间隔和外部串口记录是否同时出现异常。

### 8.11 轮询低频任务的漂移边界

当前低频任务执行后设置：

```text
last_print_tick = HAL_GetTick()
```

这意味着它按“实际执行时刻”重新计时。
如果某一次打印或 500Hz 分支造成低频任务晚执行 20ms，
下一次低频任务通常会从这个晚到的时刻再等待 100ms。

因此，当前实现更接近“最小间隔 100ms 的轮询任务”，
而不是“严格相位锁定的 10Hz 周期任务”。

### 8.12 frame_500Hz布尔标志边界

`frame_500Hz` 的类型是 `bool`。它没有记录累计帧数。

如果主循环阻塞 6ms，期间 SysTick 理论上可能触发多个 500Hz 到期事件。
但只要 `frame_500Hz` 已经是 true，后续再次赋 true 不会增加可消费次数。

恢复后主循环只会进入一次：

```text
if (frame_500Hz)
```

因此，`dt500Hz` 可能显示一次较大的时间间隔，而不是主循环连续补跑多个 2ms 帧。
这不是一定错误，而是当前简化调度器的设计边界。

若未来要精确统计错过帧数，可考虑把布尔标志扩展为计数器。
但那会引出是否补跑、最多补跑几次、状态变量如何处理等新的实时调度问题，
不属于当前源码已实现行为。

### 8.13 构建产物证据边界

`Debug/Three-axis_cloud_platformV2.map` 能证明一批与本章相关的符号进入当前 Debug 构建产物：

- `.text.HAL_GetTick`
- `.text._write`
- `.text.HAL_UART_Transmit`
- `.text.computeMotorCommands`
- `.text.zeroPIDintegralError`
- `.text.zeroPIDstates`
- `.text.Error_Handler`
- `.text.printf` / `iprintf`
- `.bss.last_print_tick`
- `.bss.executionTime500Hz`
- `.bss.systemReady`
- `.bss.frame_500Hz`
- `.bss.mpu6050Calibrating`
- `.bss.pidCmd`
- `.bss.rollDiag`

这说明低频轮询、串口输出、控制计算、PID 状态清零、错误处理和关键门控变量至少进入了该 Debug 产物。
同时，`.map` 中还可看到来自 `Core/Src/syscalls.o` 的 `.text._write` 输入段地址为 `0x00000000`，
而最终进入镜像的是 `Core/Src/usart.o` 中位于 Flash 地址空间的 `.text._write`。
因此引用 `_write` 时要说明自己依据的是最终链接后的符号位置，而不是所有同名输入段。

`Debug/Three-axis_cloud_platformV2.list` 能进一步证明该 Debug 产物中存在这些指令级路径：

- `main()` 中存在 `HAL_GetTick() - last_print_tick >= 100` 的 10Hz 差值判断。
- 低频任务进入后会执行 `last_print_tick = HAL_GetTick()`。
- `_write()` 通过 `HAL_UART_Transmit(&huart3, ..., 1, 0xFFFF)` 逐字符阻塞发送。
- SysTick 路径会把 `frame_500Hz` 置为 true。
- AHRS、控制分支和 `updatePID()` 中存在 `isnan()`、`isinf()` 或 `deltaT` 回退类异常防护。

但这份 `.list` 不能直接当作当前 `Core/Src/main.c` 的完整证明。
当前源码在启动 1000 帧处写入的是 `pitchEnabled=true`、`rollEnabled=false`、`yawEnabled=true`；
而 `.list` 中对应位置显示的是 `pitchEnabled=true`、`rollEnabled=true`、`yawEnabled=false`。
当前源码的 10Hz `if (systemReady)` 内还包含 `return_state_count` 和 `return_state_count_roll` 的状态切换逻辑；
而 `.list` 对应片段未完整呈现当前源码中的这些路径。

所以，本章使用 `.map` / `.list` 时必须加上“构建新鲜度”条件：
它们能证明当前仓库里的 Debug 构建产物曾经包含哪些符号和汇编路径，
但在 `.list` 与 `Core/Src/main.c` 不一致时，不能证明当前源码已经被重新构建进最终固件。
若要把结论提升为“当前固件行为”，需要重新构建，或至少比对 ELF、`.map`、`.list` 与源码的生成时间和差异。

`.su` 和 `.cyclo` 还能补充静态栈与圈复杂度边界，但它们同样受构建新鲜度约束：

| 关注点 | 函数 | 静态栈用量 | 圈复杂度 | 构建产物证据 |
|---|---|---:|---:|---|
| 500Hz帧置位 | `SysTick_Handler()` | 16 字节 | 5 | `Debug/Core/Src/stm32f1xx_it.su` / `.cyclo` |
| 主循环与低频轮询 | `main()` | 56 字节 | 7 | `Debug/Core/Src/main.su` / `.cyclo` |
| 最终链接的串口重定向 | `_write()` | 32 字节 | 2 | `Debug/Core/Src/usart.su` / `.cyclo` |
| 阻塞式串口发送 | `HAL_UART_Transmit()` | 48 字节 | 10 | `Debug/Drivers/STM32F1xx_HAL_Driver/Src/stm32f1xx_hal_uart.su` / `.cyclo` |
| 非阻塞发送备选 | `HAL_UART_Transmit_IT()` | 24 字节 | 4 | `Debug/Drivers/STM32F1xx_HAL_Driver/Src/stm32f1xx_hal_uart.su` / `.cyclo` |
| DMA发送备选 | `HAL_UART_Transmit_DMA()` | 56 字节 | 5 | `Debug/Drivers/STM32F1xx_HAL_Driver/Src/stm32f1xx_hal_uart.su` / `.cyclo` |
| 控制分支门控 | `computeMotorCommands()` | 120 字节 | 48 | `Debug/Drivers/SRC/Src/computeMotorCommands.su` / `.cyclo` |
| PID异常防护 | `updatePID()` | 64 字节 | 34 | `Debug/Drivers/SRC/Src/pid.su` / `.cyclo` |
| AHRS数值防护 | `MargAHRSupdate()` | 80 字节 | 33 | `Debug/Drivers/SRC/Src/MargAHRS.su` / `.cyclo` |

这张表把本章的两个性能风险拆开了。第一类是低频打印路径：`printf()` 最终落到最终链接的
`Core/Src/usart.c::_write()`，再逐字符调用 `HAL_UART_Transmit()`。静态栈和圈复杂度不高，
但阻塞等待时间取决于 UART 标志和 timeout，不能由 `.su/.cyclo` 证明。第二类是控制与异常防护路径：
`computeMotorCommands()`、`updatePID()` 和 `MargAHRSupdate()` 的复杂度更高，
它们会影响 500Hz 分支内部耗时，但真实耗时仍必须用 `executionTime500Hz`、DWT 计时或 GPIO 翻转观测【待验证】。
同时，`HAL_UART_Transmit_IT()` 与 `HAL_UART_Transmit_DMA()` 只是 HAL 中存在的备选函数；
当前 `_write()` 主线没有调用它们，所以不能把这张表写成“项目已经使用中断或 DMA 串口发送”。

因此，本节只把源码、`.map/.list/.su/.cyclo` 作为仓库内和构建层证据：它们能说明符号、变量、调用关系、静态栈和圈复杂度边界。
它们不能证明真实 10Hz 周期、串口发送耗时、外部硬件表现、主机侧现象、传感器方向、电机安全状态、异常复现结果或闭环控制效果；当 `.list` 与源码不一致时，还必须先处理构建新鲜度问题。
上述结论需要日志、调试器记录、波形、电机输出观测或仓库外实测证据，缺少证据时保持【待验证】。

## 9. 调试方法

本节按统一调试结构组织：现象 -> 可能原因 -> 定位方法 -> 验证步骤 -> 解决方案 -> 经验总结。对第32章而言，核心是把 10Hz 低频任务、500Hz 实时分支、启动/回中门控、异常值防护、阻塞路径和构建产物新鲜度分开验证。

### 9.1 现象与可能原因

常见现象包括：10Hz 输出间隔不稳、轴使能没有按预期打开、回中状态释放比理想时间晚、异常输入被清零但电机状态未改变、`mpu_ok` 看似存在却没有阻止控制输出、`dt500Hz` 与 `executionTime500Hz` 表现不一致，或 `.list` 与当前源码不一致。可能原因包括串口逐字符阻塞、I2C 读取超时、500Hz 布尔标志只记录一次待处理、10Hz 状态提交晚于 500Hz 控制帧、异常防护只处理数值不等于停机，以及 Debug 构建产物不是最新源码生成。

### 9.2 定位方法：低频任务与启动门控

第一步，确认 10Hz 低频任务频率。

- 观察 `last_print_tick` 是否约每 100ms 更新一次。
- 确认 `last_print_tick` 在低频任务块开头更新，而不是在 `printf()` 结束后更新。
- 观察串口输出是否约 10Hz，而不是 500Hz。
- 区分 `systemReady` 门控的状态切换逻辑和未被该 `if` 包住的低频打印。
- 如果输出卡顿，优先检查 `HAL_UART_Transmit()` 阻塞时间。
- 记录连续输出间隔，区分目标 100ms 和实测 100ms【待验证】。
- 若间隔持续大于 100ms，继续检查 500Hz 分支耗时、I2C 读取和串口发送。

第二步，确认启动门控。

- 观察 `systemReady` 置为 true 后，`frame_500Hz` 是否开始周期性置位。
- 若 `mpu6050Calibrating == true`，确认 500Hz 帧标志是否被暂停。
- 搜索 `mpu6050Calibration()`，确认当前主线是否实际调用温度校准函数。
- 开机后观察 `startup_delay_counter` 从 0 增加到 1000。
- 计数到 1000 时观察 `zeroPIDintegralError()` 和 `zeroPIDstates()` 是否执行。
- 同时观察 `pitchEnabled`、`rollEnabled`、`yawEnabled` 的变化。
- 记录 `startup_delay_counter == 1000` 当帧是否出现启动提示 `printf()`。
- 若 `executionTime500Hz` 在启动附近出现尖峰，先区分是否由这条一次性打印造成【待验证】。

### 9.3 验证步骤：回中切换、异常防护与门控缺口

第三步，确认 Pitch 回中切换。

- 观察 `sensors.margAttitude500Hz[PITCH]` 是否进入小于 0.3 的范围。
- 观察 `return_state_count` 是否连续增加。
- 观察 10Hz 低频任务中 `return_state` 是否变为 false，并打开 `rollEnabled`。
- 继续记录下一次 500Hz 分支是否读到新的 `rollEnabled`，以及首次后续 `PWM_Motor_SetAngle()` 或 CCR 变化证据。

第四步，确认 Roll 回中切换。

- 观察 `pointingCmd[ROLL] - sensors.margAttitude500Hz[ROLL]` 的差值。
- 观察 `return_state_count_roll` 是否连续增加。
- 观察 `return_state_roll` 是否在低频任务中被置为 false。
- 继续记录状态提交后的下一次 `computeMotorCommands()`，不要把 10Hz 变量翻转直接等同为同一时刻的 PWM 输出变化。

第五步，确认异常值防护。

- 人为在调试器中把 `dt500Hz` 改成 0、极大值或 `NaN`，观察 `updatePID()` 是否回退到 0.002f。
- 同时观察 `MargAHRSupdate()` 是否在进入四元数积分前对同一个 `dt` 做范围回退。
- 观察 `pidCmd[]` 是否在异常后被置为 0 或被限幅。
- 观察 `sensors.margAttitude500Hz[]` 出现 `NaN` 时是否被清零。
- 观察 MPU6050 读取失败时 `rawAccel[]`、`rawGyro[]` 是否被清零。
- 同时观察 `sensors.accel500Hz[]`、`sensors.gyro500Hz[]` 是否受零偏扣除影响。
- 继续观察轴使能和 PWM 输出是否仍保持原状态，不要把数据清零直接写成停机。

第六步，确认门控缺口。

- 搜索 `mpu_ok` 的赋值和读取。
- 如果没有参与运行门控，就不要假设 MPU 初始化失败会阻止控制输出。

### 9.4 验证步骤：错误处理、阻塞路径与时间边界

第七步，确认 `Error_Handler()` 边界。

- 列出会调用 `Error_Handler()` 的 HAL 初始化和配置路径。
- 对比 MPU6050 初始化失败路径，确认它是打印失败提示后继续运行，而不是进入 `Error_Handler()`。
- 对比 MPU6050 运行期读取失败路径，确认它是 raw 数据替代，而不是关闭中断后停机。
- 记录三类路径的差异：初始化停住、初始化失败提示、运行期数据替代。

第八步，确认阻塞路径。

- 记录 `MPU6050_Read_And_Process()` 前后的时间戳，估算 I2C 读取耗时。
- 记录一次 `printf()` 的字符数，按 `字符数 * 10 / 115200` 估算理论线速时间。
- 单独记录启动完成提示的字符数，因为它位于 500Hz 分支内部。
- 若分析最坏等待，按逐字符 `HAL_UART_Transmit(..., 1, 0xFFFF)` 估算 timeout 放大边界。
- 同时观察 `dt500Hz`，确认阻塞是否推迟下一次 500Hz 帧消费【待验证】。

第九步，区分 `dt500Hz` 和 `executionTime500Hz`。

- 若 `executionTime500Hz` 大，优先检查 500Hz 分支内部耗时。
- 若 `dt500Hz` 大但 `executionTime500Hz` 正常，优先检查主循环中其他阻塞路径。
- 若两者都大，说明既可能有分支内部超时，也可能有分支外调度延迟。
- 注意 `frame_500Hz` 是布尔标志，不能证明阻塞期间没有丢掉理论帧。

第十步，验证 tick 回绕边界。

- 把 `last_print_tick` 临时设到接近 `0xFFFFFFFF` 的位置，观察差值比较是否仍按预期触发。
- 记录当前 `last_print_tick`、`HAL_GetTick()` 和比较结果。
- 若有人把时间戳改成有符号类型，再重新验证这条结论是否仍成立。
- 这是一个无符号时间门控边界测试，不等于实际长时间运行已经穿过 49.7 天。

### 9.5 解决方案：调试记录

- 记录 10Hz 低频任务触发条件、串口输出、状态切换和 500Hz 分支之间的时间关系。
- 对门控过程，应记录 `systemReady`、`mpu6050Calibrating`、启动计数和轴使能字段变化。
- 对校准门控，应记录 `mpu6050Calibration()` 是否被当前主线调用，以及 `mpu6050Calibrating` 置 true/false 的范围。
- 异常防护记录应包含异常输入、回退值、清零变量、限幅结果和后续控制输出。
- 时间步长异常记录应分开写 AHRS 的 `dt` 使用方式和 PID 的 `deltaT` 回退方式。
- 阻塞分析记录应包含 I2C 读取耗时、串口字符数、理论发送时间和实测主循环延迟。
- 时间分析应同时记录 `dt500Hz`、`executionTime500Hz`、`frame_500Hz` 消费时刻和 10Hz 打印时刻。
- `mpu_ok` 未参与门控时，应明确写成门控缺口，不能假设初始化失败会自动阻止输出。
- 启动阶段的 500Hz 内部打印应单独标注，避免和 10Hz 周期打印混为一类。
- `Error_Handler()` 记录应分清初始化/配置失败、MPU 初始化失败和 MPU 运行期读取失败。
- `HAL_GetTick()` 轮询边界应同时记录无符号类型、比较阈值和回绕安全性。
- 使用 `.map` / `.list` 时，应记录构建产物生成时间、源码提交号和关键源码片段是否一致；若 `.list` 与 `Core/Src/main.c` 不一致，只能把它写成 Debug 产物证据，不能写成当前源码行为证据。

### 9.6 经验总结：调试边界

当前仓库能证明低频任务、门控变量和异常防护代码路径；已有 Debug 构建产物还能证明部分符号和汇编路径进入过最终镜像。但当 `.list` 与当前源码不一致时，应先确认构建新鲜度。实际门控时序、异常复现结果、串口输出频率和电机安全状态需要运行日志或仓库外实测证据；缺少证据时保持【待验证】。

## 10. 常见问题

问题一：10Hz 低频任务是不是另一个中断。

不是。当前 10Hz 低频任务是主循环里的 `HAL_GetTick()` 时间差判断，不是独立中断服务函数。

问题二：`print_interval` 是否控制了当前打印周期。

当前没有看到它参与判断。代码中定义了 `print_interval = 100`，但主循环判断直接使用字面量 100。

问题三：AHRS 收敛等待期间，传感器和姿态解算是否停止。

没有停止。500Hz 实时控制分支仍然读取 MPU6050 并调用 `MargAHRSupdate()`。门控限制的是轴使能和后续控制输出。

问题四：Pitch 和 Roll 的回中计数为什么不是都在 10Hz 里完成。

因为计数条件依赖 500Hz 姿态和目标误差，计数更新在控制分支中进行；10Hz 低频任务只负责低频判断是否达到阈值并切换状态。

问题五：出现 `NaN` 后系统是否一定安全。

不能这样说。
当前代码确实有多层异常清零和回退，但异常清零可能掩盖根因。
调试时应该继续追踪异常来自传感器、时间步长、姿态计算还是 PID 参数。
在没有仓库外实测证据前，异常回退不能被描述成完整的硬件安全机制。

问题六：`mpu_ok` 是否是当前控制链路门控。

当前证据不足。它被声明为 false，但没有看到成功时置 true 或参与控制分支判断。

问题七：为什么 `deltaT` 异常时回退到 0.002f。

因为 500Hz 理想周期是 2ms，也就是 0.002s。回退值让 PID 在异常时间步长下仍接近设计频率运行。

问题八：运行门控是否能保证电机输出一定关闭。

不能只凭门控变量下结论。
门控能限制控制分支是否继续计算和调用输出函数，
但真实输出还取决于此前 CCR 状态、PWM 是否仍在运行以及是否有显式清零。
需要结合第30章输出映射和第33章点测记录验证。

问题九：MPU6050 读取失败后 raw 数据清零，是否表示系统已经安全。

不能这样说。
清零 raw 数据只是数据替代策略，主循环仍会继续缩放传感器量、更新 AHRS 并进入控制链。
是否安全还要看轴使能、PID 输出、PWM 当前状态和是否有专门失效门控。

问题十：为什么 10Hz 打印也可能影响 500Hz 控制。

因为 10Hz 打印和 500Hz 消费都在同一个主循环中执行。
`_write()` 当前逐字符调用阻塞式 `HAL_UART_Transmit()`。
打印期间如果新的 `frame_500Hz` 已经置位，主循环要等打印结束后才能消费。

问题十一：为什么当前 10Hz 任务不是严格相位锁定调度。

因为代码在 10Hz 任务块开头写入 `last_print_tick = HAL_GetTick()`。
如果本次进入任务块被推迟，下一轮 100ms 从推迟后的进入时刻重新计算。
它保证的是到期轮询，不是硬件定时器式精确提交。
如果阻塞发生在这句赋值之后，例如 `printf()` 自身阻塞超过 100ms，
下一轮主循环可能很快再次满足时间差条件。

问题十二：为什么 `executionTime500Hz` 正常，`dt500Hz` 仍可能异常。

因为前者只记录 500Hz 分支内部耗时，后者记录两次消费之间的实际间隔。
如果主循环在两次 500Hz 消费之间执行了阻塞式打印，`executionTime500Hz` 可以正常，
但下一次 `dt500Hz` 会变大。

问题十三：`frame_500Hz` 为 true 是否能证明只错过了一个帧。

不能。它是布尔标志，不是计数器。
阻塞期间多次到期都可能合并成同一个 true。
要证明错过帧数，需要额外计数或外部时间记录。

问题十四：启动完成提示是否属于 10Hz 低频任务。

不是。`>>> AHRS 收敛完成...` 这条提示位于 500Hz 分支内，只在启动计数到 1000 时执行一次。
它不是周期性低频打印，但可能造成启动附近某一帧 `executionTime500Hz` 变大【待验证】。

问题十五：进入 `Error_Handler()` 是否等于所有异常都会安全停机。

不能这样理解。
`Error_Handler()` 当前用于部分 HAL 初始化或配置失败路径；MPU 初始化失败只是打印提示，
运行期 MPU 读取失败则清零 raw 数据后返回。它们不是同一类安全策略。

问题十六：`systemReady` 是否门控整个 10Hz 低频任务。

不是。当前 `systemReady` 包住的是 10Hz 中的 Pitch/Roll 状态切换逻辑。
后面的 `printf("%.6f,%d\r\n", ...)` 在这个 `if` 之后。
由于 `systemReady` 在进入主循环前已被置 true，这个差异通常不影响当前运行，
但教材分析门控边界时不能把它写成“整个 10Hz 任务都被 systemReady 包住”。

问题十七：`mpu6050Calibrating` 当前是否一定会在开机时暂停 500Hz 控制帧。

不能这样写。`SysTick_Handler()` 确实检查 `mpu6050Calibrating == false`，
温度校准函数也会设置这个标志。
但当前 `main.c` 中 `mpu6050Calibration()` 调用处于注释状态，
所以主运行路径没有看到开机实际进入温度校准的证据。

问题十八：PID 已经保护异常 `deltaT`，是否说明 AHRS 的 `dt` 也被同样保护。

不能。`updatePID()` 会把异常或超范围 `deltaT` 回退为 0.002f；
但 `MargAHRSupdate()` 当前直接使用 `halfT = dt * 0.5f`，
没有看到同样的范围回退。异常 `dt500Hz` 需要分别检查 AHRS 和 PID 两个入口。

问题十九：轴未使能是否等于 PWM 输出已经关闭。

不能这样写。轴未使能时，`computeMotorCommands()` 不进入该轴分支，
所以不会执行该轴的 PID、限幅、电角合成和 `PWM_Motor_SetAngle()`。
但这只是停止新的控制层写入，不是寄存器级关断证据。
当前 TIM2、TIM3、TIM4 的 PWM 通道初始化 `Pulse` 为 500，
`PWM_Motor_Init()` 会启动相关通道；而全 CCR 清零主要出现在硬件诊断函数中。
所以判断安全输出时必须同时看轴使能、CCR 当前值、PWM 通道状态和是否有显式清零路径。

## 11. 实践任务

开始任务前，先回到本章第8节定位源码证据：10Hz 入口看 8.1，
`systemReady` 和启动门控看 8.2 与 8.3，轴使能和回中门控看 8.4 与 8.5，
异常防护看 8.6 至 8.9，UART 阻塞和低频漂移看 8.10 与 8.11，帧合并看 8.12。
构建产物边界看 8.13。第9节提供调试观察顺序。

任务一至任务四属于运行门控主线；任务五至任务十八属于异常防护、阻塞和边界证据。

任务一：定位 10Hz 低频任务入口。

在 `Core/Src/main.c` 中找到 `HAL_GetTick()`、`last_print_tick` 和字面量 `100` 的判断位置。
说明 10Hz 低频任务为什么不是 500Hz 实时控制分支。
若有串口接收条件，再记录连续 20 次输出的时间间隔，判断是否接近 100ms。
验收依据是低频任务表包含 100ms 轮询入口、串口观察条件和时间间隔记录。
若当前没有串口接收或仓库外实测条件，只记录验证方案和预期观察，结论保持【待验证】。

任务二：验证运行状态门控细节。

从上电开始记录 `startup_delay_counter`、`pitchEnabled`、`rollEnabled`、`yawEnabled` 的变化表。

## 12. 思考题

1. 为什么串口打印适合放在 10Hz 低频任务，而不适合放在 500Hz 实时控制分支中每帧执行。
2. 为什么启动门控要在 `MargAHRSupdate()` 之后、`computeMotorCommands()` 之前执行。
3. 如果 `return_state_count` 一直不能达到 650，Roll 轴会处于什么状态。
4. 如果 `HAL_UART_Transmit()` 阻塞时间很长，会同时影响哪些任务。
5. 为什么 PID 的 `deltaT` 异常要回退到 0.002f，而不是直接返回 0。
6. 姿态角 `NaN` 被置 0 后，系统可能出现哪些误判。
7. 如果要增强 MPU 初始化失败后的安全性，应该把门控加在初始化阶段、500Hz 实时控制分支，还是 `computeMotorCommands()` 内部，为什么。
8. 为什么门控变量存在，不代表所有输出路径都会自动进入安全状态。
9. 如果缺少串口日志、断点记录或仓库外实测，哪些门控时序和安全效果仍应保持【待验证】？
10. 为什么 `last_print_tick = HAL_GetTick()` 会带来低频任务周期漂移。
11. 如果要把阻塞式串口输出改成 DMA 输出，需要额外处理哪些缓冲区和完成回调问题。
12. 为什么布尔型 `frame_500Hz` 适合简单调度，却不适合统计错过帧数。
13. 如何用 `dt500Hz` 和 `executionTime500Hz` 区分“控制分支慢”和“主循环消费晚”。
14. 为什么启动完成提示虽然只执行一次，也可能影响启动附近的 500Hz 时间记录。
15. 为什么 `Error_Handler()`、`mpu_ok` 和 raw 数据清零不能混写成同一种安全停机机制。
16. 为什么 `HAL_GetTick() - last_print_tick >= 100` 比 `HAL_GetTick() >= last_print_tick + 100` 更适合当前低频任务。
17. 为什么 `systemReady` 门控状态切换，不等于它门控整个 10Hz 任务块。
18. 为什么 `mpu6050Calibrating` 已经参与 SysTick 判断，也不能说明当前主线一定执行温度校准。
19. 为什么 PID 的 `deltaT` 回退不能替代 AHRS 的 `dt` 边界检查。
20. 为什么轴使能关闭、`power_percent = 0` 和 `TIMx->CCRn = 0` 不能在当前项目中混为同一个安全动作。
21. 为什么旧 `.list` 中出现了某条汇编路径，也不能在源码已经变化后直接证明当前固件行为。

## 13. 本章总结

本章把第31章留下的系统稳定性问题展开为三条主线。

当前项目中：

- 10Hz 低频任务由主循环中的 `HAL_GetTick() - last_print_tick >= 100` 触发，不是独立中断。
- 10Hz 低频任务负责 Pitch/Roll 回中状态切换和串口状态输出，但两者的 `systemReady` 门控范围不同。
- `last_print_tick` 在 10Hz 任务块开头更新，低频周期漂移要结合时间戳更新位置和阻塞位置分析。
- 开机后 500Hz 实时控制分支先让 AHRS 运行约 2 秒，再清空 PID 积分和 D 项状态，并开启 Pitch/Yaw。
- Roll 不是开机立即开启，而是在 Pitch 回中计数达到阈值后由 10Hz 低频任务打开。
- `return_state*` 与 `rollEnabled` 的变化属于状态提交；执行层输出变化还要等后续 500Hz 分支读取新门控，并通过 `PWM_Motor_SetAngle()` 或 CCR 证据确认。
- `computeMotorCommands()` 内部通过 `eepromConfig.*Enabled` 对三轴控制分支进行门控。
- `MargAHRSupdate()`、`updatePID()` 和分轴控制分支共同承担 `NaN与Inf防护`。
- AHRS 当前没有看到对 `dt` 的范围回退，PID 层才把异常 `deltaT` 回退为 0.002f。
- MPU6050 读取失败会清零 `rawAccel[]` 与 `rawGyro[]`，但这属于数据替代，不等于安全停机。
- `mpu6050Calibrating` 能暂停 500Hz 帧生成，但当前 `main.c` 中温度校准调用处于注释状态。
- `main.c` 的温度校准提示与 `mpu6050Calibration.c` 的 `HAL_Delay(10000)` 字面值不一致，不能把“2分钟暂停”当成当前源码已证实的时长。
- 10Hz 打印走 `_write()` 逐字符阻塞发送，仍可能推迟主循环消费下一次 500Hz 标志。
- `printf()` 到主机显示之间还要经过格式化、newlib `_write` 重定向、逐字符 HAL 阻塞发送、UART TXE/TC 等待和外部接收，不能把源码调用写成主机已收到完整日志。
- 启动完成提示 `printf()` 位于 500Hz 分支内部，可能影响启动附近的单帧耗时【待验证】。
- `frame_500Hz` 是布尔标志，阻塞期间多个理论 500Hz 触发可能合并为一次消费。
- `dt500Hz` 与 `executionTime500Hz` 诊断对象不同，必须联合观察。
- `mpu_ok` 当前没有看到参与主运行门控的证据，不能把它写成已生效的安全开关。
- `Error_Handler()` 是部分初始化/配置失败路径，不是 MPU 运行期读取失败的通用停机机制。
- `HAL_GetTick()` 的无符号减法比较能跨 tick 回绕工作，但前提是时间戳和间隔仍处在正确类型和尺度内。
- 轴使能关闭只证明对应控制分支停止更新，不证明 PWM 通道停止或 CCR 已经清零。
- 当前 TIM2/TIM3/TIM4 PWM 通道初始 `Pulse=500`，执行层安全状态必须结合 CCR 当前值和显式关断路径验证。
- `.map/.list/.su/.cyclo` 的构建产物结论统一回到第8.13节判断：它们能证明某次 Debug 产物中的符号、汇编路径和静态资源边界；当 `.list` 与当前 `main.c` 不一致时，必须先重新构建或确认产物时间戳，不能把旧产物直接写成当前源码证据，也不能替代串口阻塞、10Hz 周期和 500Hz 实时裕量实测。

本章待验证分类：

| 类别 | 已由本章证明 | 仍保持【待验证】 |
|---|---|---|
| 低频任务时序 | `HAL_GetTick() - last_print_tick >= 100`、10Hz块入口和 `last_print_tick` 更新顺序已经明确。 | 真实 10Hz 间隔、轮询漂移幅度和阻塞后恢复节奏。 |
