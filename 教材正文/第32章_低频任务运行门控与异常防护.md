# 第32章 低频任务、运行门控与异常防护

## 3. 问题背景

500Hz 实时控制循环承担的是实时链路：

```text
读取传感器
-> 更新姿态
-> 计算控制量
-> 输出电机指令
```

但一个能运行的嵌入式控制系统，只靠高频链路还不够。它还需要回答这些问题：

- 开机后是否立刻允许电机参与控制。
- 某个轴是否已经完成回中，能否切换到后续控制状态。
- 调试信息多久输出一次，避免串口打印干扰实时任务。
- 姿态、时间步长、PID 输出一旦出现 `NaN` 或 `Inf`，应该在哪里处理。

本项目把这些问题分散在多个文件中：

- `Core/Src/main.c`
- `Drivers/SRC/Src/computeMotorCommands.c`
- `Drivers/SRC/Src/pid.c`
- `Drivers/SRC/Src/MargAHRS.c`

本章的任务是把它们整理成一条稳定性主线。

## 4. 核心概念

### 4.1 10Hz低频任务

`10Hz低频任务` 是主循环中每 100ms 执行一次的非实时任务入口。

项目中使用：

```text
HAL_GetTick() - last_print_tick >= 100
```

判断 100ms 是否到达。由于 `HAL_GetTick()` 依赖 SysTick 的 1ms 系统节拍，所以低频任务不是独立硬件中断，而是主循环中的轮询任务。

当前低频任务做两件事：

- 根据 `return_state_count` 和 `return_state_count_roll` 切换回中状态。
- 通过 `printf()` 输出 Roll 姿态和 Roll 回中计数。

### 4.2 运行状态门控

`运行状态门控` 指用状态变量控制某段逻辑是否允许运行。

当前项目至少有四层门控：

- `systemReady`：控制 `SysTick_Handler()` 是否推进 500Hz 帧标志，也控制 10Hz 状态切换逻辑是否执行。
- `startup_delay_counter`：控制 AHRS 收敛等待和三轴使能时机。
- `eepromConfig.pitchEnabled`、`rollEnabled`、`yawEnabled`：控制 `computeMotorCommands()` 内部各轴分支是否运行。
- `return_state`、`return_state_roll`：控制 Pitch/Roll 处于回中路径还是后续控制路径。

这些门控并不等价。`systemReady` 面向系统初始化，`startup_delay_counter` 面向开机收敛，`eepromConfig.*Enabled` 面向轴级控制，`return_state*` 面向轴内阶段切换。

### 4.3 NaN与Inf防护

`NaN` 表示不是一个有效数值，`Inf` 表示无穷大。它们可能来自非法数学运算、异常传感器输入、过小时间步长导致的除法放大，或未被约束的中间量。

控制系统中，异常浮点值的危险不在于变量本身难看，而在于它会继续传播：

```text
异常姿态
-> 异常误差
-> 异常PID
-> 异常电角
-> 异常PWM输出
```

因此项目在姿态解算、PID 和分轴控制中多次检查 `isnan()` 和 `isinf()`。

## 5. 工作原理

### 5.1 低频任务为什么放在主循环

串口打印和状态切换不需要 500Hz 的频率。如果每个 2ms 控制帧都打印一次，会明显增加串口阻塞风险，也会让调试输出难以阅读。

项目选择用 100ms 轮询，是一种简单的分频方式：

```text
SysTick提供1ms时间基准
-> HAL_GetTick()返回毫秒计数
-> 主循环判断是否经过100ms
-> 执行低频状态逻辑和打印
```

这种方式的优点是实现简单，不需要新增调度器；代价是低频任务的实际执行时刻仍受主循环是否被阻塞影响。如果 500Hz 实时控制分支或串口发送耗时过长，10Hz 低频任务也会延后。

### 5.2 启动门控的工作过程

项目初始化结束后先关闭三轴：

```text
pitchEnabled = false
rollEnabled = false
yawEnabled = false
```

进入 500Hz 实时控制分支后，系统先持续读取传感器并运行 `MargAHRSupdate()`。当：

```text
startup_delay_counter == 1000
```

时，代码执行：

```text
zeroPIDintegralError()
zeroPIDstates()
pitchEnabled = true
rollEnabled = false
yawEnabled = true
```

在 500Hz 下，1000 帧约等于 2 秒。这说明项目给 AHRS 一个收敛窗口，再打开 Pitch 和 Yaw。Roll 仍保持关闭，等待后续回中状态切换。

### 5.3 低频状态切换的工作过程

Pitch 回中计数在 500Hz 的 Pitch 分支中更新：

```text
fabs(sensors.margAttitude500Hz[PITCH]) < 0.3
```

满足时 `return_state_count` 增加，否则清零。

10Hz 低频任务再判断：

```text
return_state_count >= 650
pitchEnabled == true
```

如果满足，项目把 `return_state` 置为 false，并打开 Roll：

```text
rollEnabled = true
```

Roll 回中计数在 500Hz 的 Roll 分支中更新：

```text
fabsf(pointingCmd[ROLL] - sensors.margAttitude500Hz[ROLL]) < 0.2f
```

10Hz 低频任务再判断：

```text
return_state_count_roll >= 650
rollEnabled == true
```

如果满足，项目把 `return_state_roll` 置为 false。

因此，回中状态不是单一频率完成的：计数更新发生在 500Hz 实时控制分支，状态翻转发生在 10Hz 低频任务。

### 5.4 异常值防护的分层

项目中的 `NaN与Inf防护` 不是一个集中函数，而是分层防护。

第一层在姿态解算入口。`MargAHRSupdate()` 会把异常的 `gx`、`gy`、`gz`、`ax`、`ay`、`az` 置为 0。

第二层在四元数归一化。若 `norm_q < 1e-10f` 或 `norm_q` 为 `NaN`，项目把四元数重置为单位姿态。

第三层在姿态输出。Roll、Pitch、Yaw 计算结果若为 `NaN`，被置为 0。

第四层在 PID。`updatePID()` 会把异常或超范围的 `deltaT` 回退为 0.002s，把异常 D 项、滤波 D 项和输出置为 0。

第五层在分轴控制。Roll、Pitch、Yaw 分支分别对角度或 PID 输出做异常检查，再进入限幅和电角合成。

### 5.5 当前门控边界

项目中声明了：

```text
mpu_ok = false
```

并在 MPU6050 初始化成功或失败时打印信息。但当前证据没有看到 `mpu_ok` 被置为 true，也没有看到它参与 `computeMotorCommands()` 或 500Hz 实时控制分支门控。

因此，不能写成“MPU6050 初始化失败会自动阻止控制链路运行”。更稳妥的结论是：当前代码有初始化结果提示，但没有看到基于 `mpu_ok` 的运行门控证据。这是后续工程健壮性可以继续检查的点。

## 6. STM32实现机制

从 STM32 实现角度看，本章机制依赖三个基础资源。

第一，SysTick 和 HAL 毫秒计数。第09章已经证明 `HAL_GetTick()` 来自 HAL 毫秒时基；本章只使用它解释 10Hz 低频任务的 100ms 轮询入口，不重新展开 500Hz 实时控制循环的微秒时间测量。

第二，USART3 阻塞发送。`Core/Src/usart.c` 中 `fputc()` 和 `_write()` 最终都调用：

```text
HAL_UART_Transmit(&huart3, ..., 0xFFFF)
```

这意味着 `printf()` 会通过 USART3 发送，并且当前实现是阻塞式发送。把打印放在 10Hz 低频任务中，可以降低它对 500Hz 实时控制路径的干扰。

第三，C 标准数学异常检查。项目在多个 `.c` 文件中使用 `isnan()` 和 `isinf()` 判断浮点异常。这些检查不是 STM32 专有外设功能，而是嵌入式 C 浮点控制链路中的工程保护。

## 8. 代码分析

### 8.1 10Hz低频任务入口

`Core/Src/main.c` 中声明：

```text
last_print_tick = 0
print_interval = 100
```

主循环中实际判断使用的是字面量 100：

```text
HAL_GetTick() - last_print_tick >= 100
```

这表示当前 10Hz 周期由代码中的 100ms 判断决定，而不是由 `print_interval` 常量参与。教材不能写成 `print_interval` 当前驱动了低频任务周期。

### 8.2 systemReady门控

初始化结束后，代码执行：

```text
systemReady = true
```

它首先影响 SysTick 中的 500Hz 帧标志生成。`Core/Src/stm32f1xx_it.c` 中只有满足：

```text
systemReady == true && mpu6050Calibrating == false
```

才会推进 `frameCounter` 并产生 `frame_500Hz`。也就是说，`systemReady` 不是只影响低频任务，它还决定初始化完成前是否允许实时控制帧开始调度。

10Hz 低频任务内部也会判断：

```text
if (systemReady)
```

只有该标志为 true 时，才执行回中状态切换逻辑。

注意，`systemReady` 当前不是传感器健康状态，也不是 AHRS 收敛状态。它更接近“初始化流程走到 USER CODE BEGIN 2 末尾”的系统就绪标志。

### 8.3 startup_delay_counter门控

500Hz 实时控制分支每帧检查：

```text
startup_delay_counter < 1000
```

计数达到 1000 时，项目清空 PID 积分和 D 项历史，并打开 Pitch/Yaw。

这一段门控的位置很关键：它发生在 `MargAHRSupdate()` 之后、`computeMotorCommands()` 之前。也就是说，AHRS 在门控等待阶段仍然运行，而控制输出由轴使能状态限制。

### 8.4 eepromConfig轴使能门控

`computeMotorCommands()` 内部分轴判断：

```text
if (eepromConfig.rollEnabled == true)
if (eepromConfig.pitchEnabled == true)
if (eepromConfig.yawEnabled == true)
```

某个轴未使能时，该轴分支不会执行 PID、限幅、电角合成和 `PWM_Motor_SetAngle()`。

这也是为什么第31章强调三轴不是开机同时闭环。第32章进一步说明：轴使能是运行门控，不只是配置项。

### 8.5 Pitch/Roll回中门控

Pitch 分支中，`return_state` 决定电角计算方向和后续参数设置：

```text
return_state == true
return_state == false
```

Roll 分支中，`return_state_roll` 决定是回中输出路径，还是后续 PID 控制路径。

两个状态的切换判断放在 10Hz 低频任务中，而计数数据来自 500Hz 实时控制分支。这个跨频率设计要重点调试，否则读者可能只看 10Hz 代码，却找不到计数为何变化。

### 8.6 PID异常防护

`updatePID()` 对时间步长先做保护：

```text
deltaT > 0.01f
deltaT < 0.0001f
isnan(deltaT)
isinf(deltaT)
```

满足任一条件时，`deltaT` 回退为 0.002f。

随后，D 项、滤波后的 D 项和最终输出都各自检查异常。D 项还会被限制在：

```text
-300.0f 到 300.0f
```

这说明 PID 防护既处理非法浮点值，也处理过大的瞬态微分。

### 8.7 姿态解算异常防护

`MargAHRSupdate()` 的入口先清理陀螺仪和加速度计输入异常。四元数更新后，再检查四元数模长：

```text
norm_q < 1e-10f
isnan(norm_q)
```

如果异常，就重置为：

```text
q0 = 1.0f
q1 = q2 = q3 = 0.0f
```

最后，Roll、Pitch、Yaw 姿态角如果出现 `NaN`，会被置为 0。

边界也要说清楚：当前输出姿态角只显式检查 `NaN`，没有看到对姿态角结果的 `Inf` 检查；输入和 PID 层才同时检查 `NaN` 与 `Inf`。

## 9. 调试方法

第一步，确认 10Hz 低频任务频率。

- 观察 `last_print_tick` 是否约每 100ms 更新一次。
- 观察串口输出是否约 10Hz，而不是 500Hz。
- 如果输出卡顿，优先检查 `HAL_UART_Transmit()` 阻塞时间。

第二步，确认启动门控。

- 观察 `systemReady` 置为 true 后，`frame_500Hz` 是否开始周期性置位。
- 若 `mpu6050Calibrating == true`，确认 500Hz 帧标志是否被暂停。
- 开机后观察 `startup_delay_counter` 从 0 增加到 1000。
- 计数到 1000 时观察 `zeroPIDintegralError()` 和 `zeroPIDstates()` 是否执行。
- 同时观察 `pitchEnabled`、`rollEnabled`、`yawEnabled` 的变化。

第三步，确认 Pitch 回中切换。

- 观察 `sensors.margAttitude500Hz[PITCH]` 是否进入小于 0.3 的范围。
- 观察 `return_state_count` 是否连续增加。
- 观察 10Hz 低频任务中 `return_state` 是否变为 false，并打开 `rollEnabled`。

第四步，确认 Roll 回中切换。

- 观察 `pointingCmd[ROLL] - sensors.margAttitude500Hz[ROLL]` 的差值。
- 观察 `return_state_count_roll` 是否连续增加。
- 观察 `return_state_roll` 是否在低频任务中被置为 false。

第五步，确认异常值防护。

- 人为在调试器中把 `dt500Hz` 改成 0、极大值或 `NaN`，观察 `updatePID()` 是否回退到 0.002f。
- 观察 `pidCmd[]` 是否在异常后被置为 0 或被限幅。
- 观察 `sensors.margAttitude500Hz[]` 出现 `NaN` 时是否被清零。

第六步，确认门控缺口。

- 搜索 `mpu_ok` 的赋值和读取。
- 如果没有参与运行门控，就不要假设 MPU 初始化失败会阻止控制输出。

调试记录建议：

- 记录 10Hz 低频任务触发条件、串口输出、状态切换和 500Hz 分支之间的时间关系。
- 对门控过程，应记录 `systemReady`、`mpu6050Calibrating`、启动计数和轴使能字段变化。
- 异常防护记录应包含异常输入、回退值、清零变量、限幅结果和后续控制输出。
- `mpu_ok` 未参与门控时，应明确写成门控缺口，不能假设初始化失败会自动阻止输出。

调试边界：

当前仓库能证明低频任务、门控变量和异常防护代码路径。实际门控时序、异常复现结果、串口输出频率和电机安全状态需要运行日志或仓库外实测证据；缺少证据时保持【待验证】。

## 10. 常见问题

问题一：10Hz 低频任务是不是另一个中断。

不是。当前 10Hz 低频任务是主循环里的 `HAL_GetTick()` 时间差判断，不是独立中断服务函数。

问题二：`print_interval` 是否控制了当前打印周期。

当前没有看到它参与判断。代码中定义了 `print_interval = 100`，但主循环判断直接使用字面量 100。

问题三：AHRS 收敛等待期间，传感器和姿态解算是否停止。

没有停止。500Hz 实时控制分支仍然读取 MPU6050 并调用 `MargAHRSupdate()`。门控限制的是轴使能和后续控制输出。

问题四：Pitch 和 Roll 的回中计数为什么不是都在 10Hz 里完成。

因为计数条件依赖 500Hz 姿态和目标误差，计数更新在控制分支中进行；10Hz 低频任务只负责低频判断是否达到阈值并切换状态。

问题五：出现 `NaN` 后系统是否一定安全。

不能这样说。
当前代码确实有多层异常清零和回退，但异常清零可能掩盖根因。
调试时应该继续追踪异常来自传感器、时间步长、姿态计算还是 PID 参数。
在没有仓库外实测证据前，异常回退不能被描述成完整的硬件安全机制。

问题六：`mpu_ok` 是否是当前控制链路门控。

当前证据不足。它被声明为 false，但没有看到成功时置 true 或参与控制分支判断。

问题七：为什么 `deltaT` 异常时回退到 0.002f。

因为 500Hz 理想周期是 2ms，也就是 0.002s。回退值让 PID 在异常时间步长下仍接近设计频率运行。

问题八：运行门控是否能保证电机输出一定关闭。

不能只凭门控变量下结论。
门控能限制控制分支是否继续计算和调用输出函数，
但真实输出还取决于此前 CCR 状态、PWM 是否仍在运行以及是否有显式清零。
需要结合第30章输出映射和第33章点测记录验证。

## 12. 思考题

1. 为什么串口打印适合放在 10Hz 低频任务，而不适合放在 500Hz 实时控制分支中每帧执行。
2. 为什么启动门控要在 `MargAHRSupdate()` 之后、`computeMotorCommands()` 之前执行。
3. 如果 `return_state_count` 一直不能达到 650，Roll 轴会处于什么状态。
4. 如果 `HAL_UART_Transmit()` 阻塞时间很长，会同时影响哪些任务。
5. 为什么 PID 的 `deltaT` 异常要回退到 0.002f，而不是直接返回 0。
6. 姿态角 `NaN` 被置 0 后，系统可能出现哪些误判。
7. 如果要增强 MPU 初始化失败后的安全性，应该把门控加在初始化阶段、500Hz 实时控制分支，还是 `computeMotorCommands()` 内部，为什么。
8. 为什么门控变量存在，不代表所有输出路径都会自动进入安全状态。
9. 如果缺少串口日志、断点记录或仓库外实测，哪些门控时序和安全效果仍应保持【待验证】？

## 13. 本章总结

本章把第31章留下的系统稳定性问题展开为三条主线。

当前项目中：

- 10Hz 低频任务由主循环中的 `HAL_GetTick() - last_print_tick >= 100` 触发，不是独立中断。
- 10Hz 低频任务负责 Pitch/Roll 回中状态切换和串口状态输出。
- 开机后 500Hz 实时控制分支先让 AHRS 运行约 2 秒，再清空 PID 积分和 D 项状态，并开启 Pitch/Yaw。
- Roll 不是开机立即开启，而是在 Pitch 回中计数达到阈值后由 10Hz 低频任务打开。
- `computeMotorCommands()` 内部通过 `eepromConfig.*Enabled` 对三轴控制分支进行门控。
- `MargAHRSupdate()`、`updatePID()` 和分轴控制分支共同承担 `NaN与Inf防护`。
- `mpu_ok` 当前没有看到参与主运行门控的证据，不能把它写成已生效的安全开关。

本章边界：

- 本章证明软件门控和异常回退逻辑存在，不证明所有输出路径都会自动进入硬件安全状态。
- 门控效果需要结合 PWM 当前状态、CCR 是否清零、轴使能变量和仓库外点测继续验证。

下一章将进入 `下载调试与电机硬件诊断`，把前面建立的 PWM 输出、三轴映射和运行门控知识用于实际下载、观察和电机输出验证。
