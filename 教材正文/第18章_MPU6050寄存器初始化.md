# 第18章 MPU6050寄存器初始化

## 1. 本章目标

本章开始进入 MPU6050 驱动主线，但只处理初始化入口和寄存器配置，不进入连续读取、安装方向、标定、物理量缩放和姿态解算。

本章目标是：

- 理解 `MPU6050寄存器配置` 为什么依赖 `I2C主机通信` 和 `16位数据联合体解析`。
- 找到项目中 MPU6050 初始化的真实入口：`Core/Src/main.c` 中调用 `MPU6050_Init(ACCEL_FS_4G, GYRO_FS_500DPS, DLPF_CFG_260HZ)`。
- 理解 `Drivers/CustomDrivers/Inc/mpu6050.h` 中寄存器地址、配置值和灵敏度宏的作用。
- 理解 `Drivers/CustomDrivers/Src/mpu6050.c` 中 `MPU6050_Init()` 的初始化顺序。
- 能够根据串口日志和返回值判断初始化失败的大致位置。

本章知识链为：

`知识点总表` 中的 `MPU6050寄存器配置`
-> `知识依赖图` 中依赖 `I2C主机通信`、`16位数据联合体解析`

## 2. 前置知识

本章正式前置知识包括：

- `I2C主机通信`
- `16位数据联合体解析`

`I2C主机通信` 已在第14章讲过。第18章不重新讲 I2C 电气层和 HAL 初始化，而是直接使用 `HAL_I2C_Mem_Read()`、`HAL_I2C_Mem_Write()` 作为寄存器读写工具。

`16位数据联合体解析` 已在第17章讲过。虽然本章主要配置寄存器，还不读取 14 字节连续数据，但 MPU6050 初始化之后紧接着会进入原始数据读取，所以本章需要让读者保持对原始数据承接对象的上下文。

本章不要求读者已经掌握姿态解算、滤波或 PID。那些内容在后续章节展开。

## 3. 问题背景

MPU6050 上电后不能直接提供项目可用的数据。它需要先确认设备存在，再配置工作状态。对本项目来说，初始化至少要解决五个问题：

- I2C 地址是否正确，设备是否响应。
- 芯片是否已经从休眠状态唤醒。
- 采样率如何设置。
- 数字低通滤波器 DLPF 使用哪个配置。
- 加速度计和陀螺仪量程如何选择。

如果这一步失败，后面的连续读取、标定、滤波和姿态解算都没有可靠输入。因此第18章是传感器主线的“上电入口章”。

项目证据集中在三个文件：

- `Drivers/CustomDrivers/Inc/mpu6050.h`
- `Drivers/CustomDrivers/Src/mpu6050.c`
- `Core/Src/main.c`

## 4. 核心概念

### 4.1 设备地址和WHO_AM_I

`Drivers/CustomDrivers/Inc/mpu6050.h` 中定义：

```c
#define MPU6050_I2C_ADDR        (0x68<<1)
#define MPU_WHO_AM_I_VAL        0x68
#define REG_WHO_AM_I            0x75
```

MPU6050 的 AD0 接地时 7 位地址为 `0x68`。STM32 HAL 的内存读写接口通常使用左移后的地址形式，所以项目写为 `(0x68<<1)`。

如果 AD0 接 VCC，MPU6050 的 7 位地址会变为 `0x69`。在继续使用 STM32 HAL
`HAL_I2C_Mem_Read()` / `HAL_I2C_Mem_Write()` 这类接口时，传入地址也应保持 HAL
所需的左移形式，即 `(0x69<<1)`，总线地址字节为 `0xD2`。项目头文件中注释掉的备选写法
`0x69` 没有左移，不能直接照抄为 HAL 地址；这属于配置注释陷阱，不是本轮要修改的固件行为。

`REG_WHO_AM_I` 是设备 ID 寄存器。`MPU6050_Init()` 先读取它，并期望得到 `0x68`。这一检查不是功能配置，而是确认“总线、地址、供电、芯片响应”基本成立。

### 4.2 寄存器地址和配置值

`mpu6050.h` 把初始化涉及的寄存器地址集中定义出来：

- `REG_PWR_MGMT_1`：电源管理和唤醒。
- `REG_SMPLRT_DIV`：采样率分频。
- `REG_CONFIG`：DLPF 配置。
- `REG_GYRO_CONFIG`：陀螺仪量程。
- `REG_ACCEL_CONFIG`：加速度计量程。

同时，头文件定义了具体写入值，例如 `PWR1_WAKEUP_Z_GYRO`、`SMPLRT_DIV_100HZ`、`DLPF_CFG_260HZ`、`GYRO_FS_500DPS`、`ACCEL_FS_4G`。

这些宏把“魔法数字”变成可读名称。读者阅读 `MPU6050_Init()` 时，不需要先记住 `0x6B`、`0x19`、`0x1A`、`0x1B`、`0x1C` 分别代表什么。

### 4.3 配置值和寄存器位域

寄存器配置值不是普通整数，而是位域组合。以当前项目实参为例：

- `PWR1_WAKEUP_Z_GYRO = 0x03`，二进制为 `0000 0011`。它保持 `DEVICE_RESET=0`、
  `SLEEP=0`、`CYCLE=0`、`TEMP_DIS=0`，并把 `CLKSEL[2:0]` 设为 `3`，即选择 Z 轴陀螺仪
  PLL 作为时钟参考。
- `DLPF_CFG_260HZ = 0x00` 写入 `REG_CONFIG`。按手册，`REG_CONFIG` 的高两位保留，
  `EXT_SYNC_SET[2:0]` 位于中间三位，`DLPF_CFG[2:0]` 位于低三位；项目宏 `0x00` 表示
  关闭外部同步采样并选择 `DLPF_CFG=0`。
  这里的 `260HZ` 是项目宏名。按 MPU6050 寄存器表，`DLPF_CFG=0` 对应加速度计带宽 260Hz、
  陀螺仪带宽 256Hz、陀螺仪输出率 8kHz，不能把宏名机械理解为所有传感器通道都正好 260Hz。
- `GYRO_FS_500DPS = 0x08` 写入 `REG_GYRO_CONFIG`。它把 `FS_SEL[1:0]` 设为 `1`，
  对应正负 500 度每秒，同时保持 `XG_ST/YG_ST/ZG_ST` 自检位为 0。
- `ACCEL_FS_4G = 0x08` 写入 `REG_ACCEL_CONFIG`。它把 `AFS_SEL[1:0]` 设为 `1`，
  对应正负 4g，同时保持 `XA_ST/YA_ST/ZA_ST` 自检位为 0。

也就是说，当前写入值 `0x08` 的关键不是“十六进制 8”，而是二进制 `0000 1000`：
自检位仍为 0，量程选择位为 `01`。
如果未来调用者传入带自检位的值，`MPU6050_Init()` 后面的 `& 0x18` 仍只会更新灵敏度缓存，
但硬件寄存器已经可能进入自检相关配置；这类情况不能只靠灵敏度缓存判断。

这一层复查有一个细节：`mpu6050.h` 中部分位域注释与手册位域不完全一致，例如
`REG_CONFIG` 注释把 `DLPF_CFG` 写在高位区，`GYRO_CONFIG`/`ACCEL_CONFIG` 又弱化了自检位。
由于实际宏值 `0x00`、`0x08`、`0x10`、`0x18` 落在正确的功能位上，当前初始化写入意图仍可解释；
但教材分析位域时应以手册和实际二进制值为准，不把源码注释当作最终依据。

### 4.4 底层寄存器读写函数

`Drivers/CustomDrivers/Src/mpu6050.c` 中有两个静态函数：

```c
static HAL_StatusTypeDef MPU_Write_Reg(uint8_t reg_addr, uint8_t data);
static HAL_StatusTypeDef MPU_Read_Reg(uint8_t reg_addr, uint8_t *data, uint16_t len);
```

它们内部分别调用 `HAL_I2C_Mem_Write()` 和 `HAL_I2C_Mem_Read()`，并使用 `hi2c2`、`MPU6050_I2C_ADDR`、`I2C_MEMADD_SIZE_8BIT`。

这里的“Mem”不是 MCU 内存，而是 I2C 设备内部寄存器地址空间。`reg_addr` 表示 MPU6050 内部寄存器地址，`data` 是要写入或读取的数据。

### 4.5 写入确认、读回确认和实测确认

寄存器初始化容易被误解为“函数返回成功就等于硬件配置完全正确”。更严谨的拆分是三层：

- 写入确认：`HAL_I2C_Mem_Write()` 返回 `HAL_OK`，表示本次 I2C 传输流程未报错。
- 读回确认：再次读取对应寄存器，确认寄存器内容确实等于期望值。
- 实测确认：用日志、逻辑分析仪、示波器或闭环响应确认传感器输出和系统行为符合预期。

当前项目源码在初始化阶段检查了 `WHO_AM_I`，并检查每次寄存器写入的 HAL 返回值。
但源码没有在写入 `PWR_MGMT_1`、`SMPLRT_DIV`、`CONFIG`、`GYRO_CONFIG`、`ACCEL_CONFIG` 后立即读回校验。

因此本章可以证明“项目尝试写入这些配置，并要求 HAL 返回成功”，但不能把它等同于“仓库已有证据证明寄存器实值、采样率和动态噪声全部符合预期”。

### 4.6 已配置寄存器与未接入机制

寄存器初始化还要看“没有写什么”。当前 `MPU6050_Init()` 只写了唤醒、采样率、DLPF、陀螺仪量程和加速度计量程。仓库内没有看到以下初始化动作：

- 没有写 `REG_PWR_MGMT_2` 去单独关闭某些加速度计或陀螺仪轴。
- 没有写 `INT_ENABLE` 打开 Data Ready 中断。
- 没有读取 `INT_STATUS` 来确认某一帧数据已经准备好。
- 没有写 `FIFO_EN`、`USER_CTRL`，也没有读取 FIFO 计数或 FIFO 数据寄存器。
- 没有把 MPU6050 的中断引脚接入 EXTI 回调作为采样触发源。

这并不自动表示固件错误。它说明当前项目选择的是“主循环按 500Hz 节拍轮询 14 字节输出寄存器”的数据入口，而不是“MPU6050 数据就绪中断驱动读取”或“FIFO 缓冲批量读取”。这个边界应放在第18章讲清楚，因为初始化函数决定了后续第19章能否使用 Data Ready、FIFO 或中断证据。

这种选择有工程权衡。轮询式读取路径简单、调试点少，也避免在中断中执行阻塞式 I2C；代价是样本新鲜度和读取相位不能仅凭源码证明。若未来启用 Data Ready 中断，需要补充 `INT_ENABLE` 写入、GPIO/EXTI 配置、中断回调、I2C 读取调度和过载保护。若未来启用 FIFO，需要补充 FIFO 使能、FIFO 计数读取、溢出处理和主循环排空策略。

## 5. 工作原理

MPU6050 初始化可以理解为一个有限步骤链：

1. 延时等待上电稳定。
2. 读取 `WHO_AM_I`，确认芯片响应且 ID 正确。
3. 写 `PWR_MGMT_1`，唤醒设备并选择时钟源。
4. 写 `SMPLRT_DIV`，设置采样率分频。
5. 写 `CONFIG`，设置 DLPF。
6. 写 `GYRO_CONFIG`，设置陀螺仪量程。
7. 写 `ACCEL_CONFIG`，设置加速度计量程。
8. 根据传入量程更新内部灵敏度缓存。
9. 调用静态误差计算入口。
10. 返回初始化成功或失败。

这个顺序的关键是：先确认设备存在，再写配置；先唤醒设备，再配置采样和量程；先建立原始数据输出条件，再允许后续章节读取和处理数据。

本项目的 `MPU6050_Init()` 遇到任意关键寄存器写入失败都会返回 `false`。`Core/Src/main.c` 根据返回值打印初始化成功或失败，并用 LED 指示成功状态。
但这个返回值当前没有形成主循环门控。`Core/Src/main.c` 中虽然定义了 `bool mpu_ok = false`，但当前片段没有在初始化成功分支把它置为 `true`，也没有在 500Hz 分支用它阻止 `MPU6050_Read_And_Process()`。初始化 `if/else` 结束后，代码仍会继续设置 PID 参数、把 `systemReady = true`，并进入 `while(1)`。因此“打印初始化失败”只能证明失败被报告，不能自动证明后续传感器读取、AHRS 和控制计算已经被隔离。

还要注意，项目里的 `return true` 不是在寄存器写入后立即发生。源码在更新灵敏度缓存后会先调用
`MPU6050_ComputeStaticError()`，该函数结束后 `MPU6050_Init()` 才返回成功。因此项目意义上的
“初始化成功”包含一次静态误差采样入口，而不只是 WHO_AM_I 与寄存器写入。

采样率计算还要拆成两层。

第一层是 MPU6050 手册中的寄存器公式：

```text
Sample Rate = Gyroscope Output Rate / (1 + SMPLRT_DIV)
```

手册说明，当 `DLPF_CFG` 为 `0` 或 `7` 时，陀螺仪输出率为 8 kHz；当 DLPF 启用时，陀螺仪输出率为 1 kHz。

第二层是本项目源码当前写入的组合：

```text
DLPF_CFG_260HZ = 0x00
SMPLRT_DIV_100HZ = 9
```

按寄存器公式推导，当前组合对应的陀螺仪采样率基准是：

```text
8000 / (1 + 9) = 800 Hz
```

但这仍然不是“控制环真实读取频率”。控制环何时读取、是否丢样、是否重复使用旧数据，要结合第19章连续读取和第26章 500Hz 实时控制循环继续验证。
因此本章结论应写成：寄存器公式可推出当前配置的陀螺仪寄存器更新基准，不可直接推出整个项目数据流的实测频率。

从调度角度看，当前配置可推导出陀螺仪寄存器更新基准约 800Hz，而项目主循环读取节拍是 500Hz。若未来把 Data Ready 中断直接接入读取路径，就可能面对“传感器更新事件快于控制消费节拍”的问题：要么把读取/滤波/控制链路提升到相同节拍，要么做分频、丢弃、排队或 FIFO 缓冲。当前源码没有这些策略，所以本章不能把 Data Ready/FIFO 作为已经存在的同步能力。

## 6. STM32实现机制

从 STM32 角度看，本章的实现依赖第14章已经建立的 I2C2 主机通信能力。

`mpu6050.c` 通过：

```c
extern I2C_HandleTypeDef hi2c2;
```

引用 CubeMX/HAL 初始化出的 I2C2 句柄。`MPU_Write_Reg()` 使用：

```c
HAL_I2C_Mem_Write(&hi2c2, MPU6050_I2C_ADDR, reg_addr,
                  I2C_MEMADD_SIZE_8BIT, &data, 1, 100);
```

含义是：通过 I2C2，访问设备地址 `MPU6050_I2C_ADDR`，向 8 位寄存器地址 `reg_addr` 写入 1 字节数据，超时时间为 100 ms。

`MPU_Read_Reg()` 使用相同地址和寄存器地址宽度，只是把读出的数据放入调用者提供的缓冲区。

第18章只需要理解“STM32 通过 HAL I2C Mem 接口访问 MPU6050 寄存器”。至于 I2C 时序、引脚复用、时钟和错误状态细节，已经在第14章铺垫。

## 7. 项目中的应用

`Core/Src/main.c` 中初始化顺序显示，项目先初始化一阶低通滤波器、PID 状态，再调用：

```c
orientIMU();
MPU6050_Init(ACCEL_FS_4G, GYRO_FS_500DPS, DLPF_CFG_260HZ);
```

这里有两个边界需要分清：

- `orientIMU()` 设置 IMU 安装方向矩阵，属于第20章。
- `MPU6050_Init()` 配置 MPU6050 寄存器，属于本章。

`Core/Src/main.c` 的实参说明项目当前选择：

- 加速度计量程：`ACCEL_FS_4G`
- 陀螺仪量程：`GYRO_FS_500DPS`
- DLPF：`DLPF_CFG_260HZ`

同时，`Drivers/SRC/Src/config.c` 中也存在 `eepromConfig.dlpfSetting = DLPF_CFG_184HZ`。

但当前 `Core/Src/main.c` 调用 `MPU6050_Init()` 时没有把 `eepromConfig.dlpfSetting` 传进去，而是直接传入 `DLPF_CFG_260HZ`。

因此本章只能认定“当前初始化调用使用 260Hz 配置”，不能把 `config.c` 中的 DLPF 默认值当作硬件初始化实参。

这个设计有一个清晰权衡：把 DLPF、加速度计量程和陀螺仪量程硬编码在 `main.c` 调用点，初始化行为更直观，也更容易审查。
代价是 `eepromConfig.dlpfSetting` 暂时不会影响真实硬件配置，读者容易误以为配置对象已经接管 DLPF。

若后续希望运行时配置真正控制 MPU6050，应把 `eepromConfig.dlpfSetting` 接入 `MPU6050_Init()` 的第三个实参，或者建立单独的传感器配置应用函数。
这种改动属于固件行为修改，不在本轮教材复查范围内。

当前项目的另一个应用边界是“初始化不建立硬件采样触发链”。`Core/Src/main.c` 的 500Hz 分支在软件节拍到来后调用 `MPU6050_Read_And_Process()`；源码中没有看到 MPU6050 Data Ready 引脚触发 EXTI，也没有看到 FIFO 数据流被启用。因此第18章只负责证明传感器被配置为可读状态，第19章再证明项目如何按软件节拍读取输出寄存器。

如果未来希望降低样本相位不确定性，可以考虑两条路线：一是启用 Data Ready 中断，用传感器事件触发或标记读取；二是启用 FIFO，把传感器输出先放入器件内部缓冲区，再由 MCU 批量读取。前者会增加中断调度和 I2C 时序设计压力，后者会增加 FIFO 溢出、时间戳和排空策略问题。当前项目选择轮询读取，是更简单、证据链更短的实现方式，但样本新鲜度必须保持【待验证】。

还有一条运行门控边界：当前 `MPU6050_Init()` 的返回值没有阻断后续 500Hz 读取路径。若初始化失败后 I2C 总线仍处于异常状态，`MPU6050_Read_And_Process()` 仍可能在 500Hz 分支中进入 `MPU_Read_Reg()`；该底层读取超时参数为 100 ms，远大于 500Hz 控制周期的 2 ms 预算。因此调试初始化失败时，不能只看启动阶段日志，还要确认主循环是否仍在重复触发失败读取。

## 8. 代码分析

### 8.1 MPU_Write_Reg和MPU_Read_Reg

`MPU_Write_Reg()` 和 `MPU_Read_Reg()` 都是 `static` 函数，只在 `mpu6050.c` 内部使用。这是合理的边界：外部模块不应该随意写 MPU6050 寄存器，而应该通过更高层的初始化和读取函数访问。

`MPU_Write_Reg()` 每次只写 1 字节配置值。第18章涉及的配置寄存器都是 8 位寄存器，因此这种封装足够。

`MPU_Read_Reg()` 支持读取多个字节。本章只用它读取 `WHO_AM_I` 的 1 字节；第19章会使用同类接口读取连续 14 字节传感器数据。

### 8.2 设备ID检查

`MPU6050_Init()` 开始时延时 200 ms，然后最多重试 3 次读取 `REG_WHO_AM_I`。成功条件是：

```c
status == HAL_OK && check_id == 0x68
```

如果失败，函数会打印错误信息，包括 HAL 状态、期望 ID 和实际读数，并返回 `false`。

这段逻辑对调试很有价值。若状态不是 `HAL_OK`，优先怀疑 I2C 通信层、供电、线序或上拉。若状态正常但 ID 不对，则需要检查地址、芯片型号或总线上是否是其他设备。

但错误日志中的状态解释需要谨慎阅读。源码打印：

```c
Last Status: %d (1=NoAck, 2=Error, 3=Timeout, 4=Busy)
```

而本仓库 HAL 头文件 `stm32f1xx_hal_def.h` 中 `HAL_StatusTypeDef` 的枚举是：
`HAL_OK=0`、`HAL_ERROR=1`、`HAL_BUSY=2`、`HAL_TIMEOUT=3`。也就是说，日志中的
`1=NoAck`、`4=Busy` 与 HAL 状态枚举不匹配。NACK 更应该结合 I2C 错误码或
`HAL_I2C_GetError()` 一类接口判断，而不能只靠 `HAL_StatusTypeDef` 的数值命名。

### 8.2.1 失败返回路径定位表

`MPU6050_Init()` 的返回值虽然只有 `true/false`，但源码里的失败位置并不等价。把它拆成返回路径，调试时才能知道串口日志、断点和寄存器读回应放在哪里：

| 失败位置 | 源码条件 | 当前可见诊断 | 仍缺少的证据 |
|---|---|---|---|
| `WHO_AM_I` 三次重试失败 | `status != HAL_OK || check_id != 0x68` | `mpu6050.c` 会打印 `CRITICAL ERROR`、最后一次 HAL 状态、期望 ID 和实际读数，然后返回 `false`。 | HAL 状态不是 I2C 错误细分类；NACK、总线忙、上拉、AD0 电平和供电仍需 `hi2c2.ErrorCode`、逻辑分析仪或硬件实测【待验证】。 |
| `PWR_MGMT_1` 写入失败 | `MPU_Write_Reg(REG_PWR_MGMT_1, PWR1_WAKEUP_Z_GYRO) != HAL_OK` | 函数直接返回 `false`，`main.c` 只能打印统一的初始化失败信息。 | 当前源码没有打印失败寄存器，也没有读回 `REG_PWR_MGMT_1`；不能判断设备是否已被唤醒。 |
| `SMPLRT_DIV/CONFIG/GYRO_CONFIG/ACCEL_CONFIG` 任一写入失败 | 对应 `MPU_Write_Reg(...) != HAL_OK` | 函数直接返回 `false`，但不区分是哪一个配置寄存器失败。 | 需要断点、临时日志或写后读回表，才能定位采样率、DLPF、陀螺仪量程或加速度计量程中的具体失败点。 |
| 静态误差采样阶段读数失败 | `MPU6050_Read_And_Process()` 内部读 14 字节失败后清零并 `return` | `MPU6050_ComputeStaticError()` 没有错误返回值，`MPU6050_Init()` 仍可能走到 `return true`。 | 不能把初始化成功等同于静态误差采样全程有效；需要采样计数、失败计数或原始数据日志【待验证】。 |
| `main.c` 收到 `false` | `if (MPU6050_Init(...))` 的 `else` 分支 | 串口打印初始化失败；成功分支才点亮 LED。 | 当前 `mpu_ok` 未接入门控，`systemReady` 后续仍会置 `true`；不能由失败日志推出 500Hz 读取和控制计算已经停止。 |

这张表的核心不是要求立刻改代码，而是让读者分清“已定位失败阶段”和“只知道函数返回失败”。若要把初始化失败诊断提升到工程可维护水平，最小补充通常是给每次 `MPU_Write_Reg()` 增加失败寄存器名，或者在成功写入后读回五个配置寄存器；没有这些证据时，本章只能把失败范围定位到当前源码能区分的层级。

### 8.3 唤醒和采样率

设备 ID 成功后，代码写入：

```c
MPU_Write_Reg(REG_PWR_MGMT_1, PWR1_WAKEUP_Z_GYRO);
```

`PWR1_WAKEUP_Z_GYRO` 对应注释中的 `0000 0011`，含义是唤醒设备，并选择 Z 轴陀螺仪作为时钟源。

随后写入：

```c
MPU_Write_Reg(REG_SMPLRT_DIV, SMPLRT_DIV_100HZ);
```

`SMPLRT_DIV_100HZ` 在头文件中定义为 9。头文件注释说明，当 DLPF 启用时，陀螺仪输出率为 1 kHz，采样率约为 `1000 / (1 + 9) = 100Hz`。

但是当前 `Core/Src/main.c` 实际传入的是 `DLPF_CFG_260HZ`，也就是 `0x00`。同一头文件又提示 `DLPF_CFG` 为 0 或 7 时内部频率不是 1 kHz。

因此，本教材不能直接把当前硬件采样率断言为 100Hz。
更严谨的说法是：项目写入了 `SMPLRT_DIV_100HZ` 这个宏值，但在 `DLPF_CFG_260HZ` 配置下，
按 MPU6050 寄存器公式可推出陀螺仪寄存器更新基准为 `8000 / (1 + 9) = 800Hz`。
仍需【待验证】的是控制循环实际读取频率、是否重复读取旧样本、I2C 读取是否丢样，以及加速度/陀螺仪数据在项目数据流中的有效更新节奏。

还需要注意，`mpu6050.h` 中 `SMPLRT_DIV_100HZ`、`SMPLRT_DIV_200HZ`、`SMPLRT_DIV_500HZ` 出现了两组相同定义。当前值一致，不影响编译含义，但属于可读性上的重复定义，后续整理头文件时可以清理。

### 8.4 当前寄存器写入清单

把 `main.c` 实参和 `MPU6050_Init()` 内部写入放在一张表里，可以避免把宏名、寄存器地址和实际值混在一起。

| 顺序 | 寄存器 | 地址 | 写入来源 | 当前值 | 教学含义 |
| --- | --- | --- | --- | --- | --- |
| 1 | `REG_PWR_MGMT_1` | `0x6B` | `PWR1_WAKEUP_Z_GYRO` | `0x03` | 清除睡眠位并选择时钟源 |
| 2 | `REG_SMPLRT_DIV` | `0x19` | `SMPLRT_DIV_100HZ` | `9` | 对陀螺仪输出率分频 |
| 3 | `REG_CONFIG` | `0x1A` | `dlpf` 实参 | `0x00` | 当前为 `DLPF_CFG_260HZ` |
| 4 | `REG_GYRO_CONFIG` | `0x1B` | `gyro_fs` 实参 | `0x08` | 当前为 `GYRO_FS_500DPS` |
| 5 | `REG_ACCEL_CONFIG` | `0x1C` | `accel_fs` 实参 | `0x08` | 当前为 `ACCEL_FS_4G` |

这张表只能说明源码意图和写入路径。若要把它升级为硬件证据，需要在初始化后读回这些寄存器并记录结果。

还应同时记录“未写入清单”。当前 `mpu6050.h` 定义了 `REG_PWR_MGMT_2`，但 `MPU6050_Init()` 没有写它；头文件中也没有定义 `INT_ENABLE`、`INT_STATUS`、`FIFO_EN` 或 `USER_CTRL` 相关宏。用 `rg` 搜索当前仓库，也没有发现 MPU6050 Data Ready 或 FIFO 初始化路径。因此本章不能说初始化已经建立了数据就绪同步、轴待机管理或 FIFO 缓冲机制。

这类未写寄存器属于证据边界，而不是代码缺陷结论。因为第19章的当前主线正是从 `REG_ACCEL_XOUT_H` 开始执行 14 字节轮询读取，软件设计并不依赖这些机制。

### 8.5 DLPF和量程

`MPU6050_Init()` 的第三个参数 `dlpf` 被直接写入 `REG_CONFIG`：

```c
MPU_Write_Reg(REG_CONFIG, dlpf);
```

当前 `main.c` 传入 `DLPF_CFG_260HZ`。这意味着实际初始化使用的是头文件中 `0x00` 对应的 DLPF 配置。

这里还要拆出一个更细的边界：`dlpf` 不是“只会影响 DLPF_CFG 低三位的抽象参数”，而是被当作 `REG_CONFIG`
的整字节写入值。当前调用传入 `0x00`，所以高位和 `EXT_SYNC_SET` 都保持为 0；但如果未来把
`eepromConfig.dlpfSetting` 或外部调参值直接接入这里，必须先确认：

```text
(dlpf & ~0x07) == 0
```

也就是除了 `DLPF_CFG[2:0]` 外没有其它位被置 1。否则一次看似“只改低通带宽”的调用，
可能同时改变 `REG_CONFIG` 中的外部同步选择位或保留位。当前代码没有对 `dlpf` 做 `& 0x07`
掩码，也没有枚举范围检查；安全边界来自调用点只传入头文件中定义好的 `DLPF_CFG_*` 宏。

随后，函数写入陀螺仪和加速度计量程：

```c
MPU_Write_Reg(REG_GYRO_CONFIG, gyro_fs);
MPU_Write_Reg(REG_ACCEL_CONFIG, accel_fs);
```

当前项目实参分别是 `GYRO_FS_500DPS` 和 `ACCEL_FS_4G`。

`GYRO_FS_500DPS` 和 `ACCEL_FS_4G` 的共同特点是只设置量程选择位，不打开自检位。
源码后续用 `& 0x18` 提取量程位，正好对应 `GYRO_CONFIG` 的 `FS_SEL` 和 `ACCEL_CONFIG` 的 `AFS_SEL` 两个字段。

这也是为什么传入值必须保持在头文件宏定义的范围内。若调用者传入包含自检位或保留位的值，寄存器仍可能被写入，但后续灵敏度缓存只按 `0x18` 判断量程。

### 8.6 灵敏度缓存

完成寄存器写入后，函数根据 `accel_fs & 0x18` 和 `gyro_fs & 0x18` 更新内部静态变量：

```c
static float g_accel_sens = ACCEL_SENS_2G;
static float g_gyro_sens = GYRO_SENS_250;
```

这一步不是写 MPU6050 寄存器，而是让后续物理量换算知道当前量程对应的 LSB 系数。例如 `ACCEL_FS_4G` 对应 `ACCEL_SENS_4G`，`GYRO_FS_500DPS` 对应 `GYRO_SENS_500`。

物理量缩放本身留到第21章，本章只说明为什么初始化函数要缓存灵敏度。

### 8.7 静态误差计算入口

`MPU6050_Init()` 末尾调用 `MPU6050_ComputeStaticError()`。这说明项目在初始化流程末端进入静态误差相关处理。

`MPU6050_ComputeStaticError()` 内部循环 5000 次，每次调用 `MPU6050_Read_And_Process()`、
`MPU6050_ComputeTemError()`，然后执行 `DWT_Delay_us(1000)`。仅从延时语句计算，理想下限就是：

```text
5000 * 1000 us = 5 000 000 us = 5 s
```

实际启动耗时还要再加上 5000 次 I2C 连续读取、温度补偿计算、矩阵处理和循环开销。因此本项目
MPU6050 初始化路径不是“几次寄存器写入后立即进入控制环”，而是包含一个至少约 5 秒量级的启动采样阶段。
真实启动时长需要运行日志或测量工具确认，标记为【待验证】。

这里还有一个状态传播边界：`MPU6050_Read_And_Process()` 读取失败时会把 `rawAccel`、`rawGyro`
清零并 `return`，但 `MPU6050_ComputeStaticError()` 没有返回错误状态，`MPU6050_Init()` 也不会因为
静态误差采样期间的单次读取失败自动返回 `false`。因此初始化成功只能证明前面的 ID 与寄存器写入流程走通，
不能证明 5000 次静态采样全部有效。

静态误差估计的算法含义属于第21章 `传感器标定、零偏与物理量缩放` 的范围。本章只记录调用位置、
启动耗时下限和成功判定边界，不展开零偏数学细节。

这说明寄存器读写封装和初始化函数没有停留在“源文件里存在但未链接”的状态，而是被放入当前调试构建的
`.text` 段。`.list` 文件进一步把 `Core/Src/main.c` 与反汇编对应起来：`main()` 中的

在 `MPU6050_Init()` 内部，可以看到 `MPU_Read_Reg(REG_WHO_AM_I, ...)`，
以及对 `REG_PWR_MGMT_1`、`REG_SMPLRT_DIV`、`REG_CONFIG`、`REG_GYRO_CONFIG`、

因此，构建产物能补强三类结论：

- 当前固件构建确实包含 `MPU6050_Init()`、`MPU_Write_Reg()` 和 `MPU_Read_Reg()`。

- 初始化函数的 WHO_AM_I 读取、五个寄存器写入和灵敏度缓存分支在当前编译产物中都有对应指令路径。

当前 `Debug/Drivers/CustomDrivers/Src/mpu6050.su` 和
`Debug/Drivers/CustomDrivers/Src/mpu6050.cyclo` 还能给出本章函数的静态栈用量和圈复杂度边界：

- `MPU_Write_Reg()`：静态栈 32 字节，圈复杂度 1。
- `MPU_Read_Reg()`：静态栈 32 字节，圈复杂度 1。
- `MPU6050_Init()`：静态栈 32 字节，圈复杂度 19。
- `MPU6050_ComputeStaticError()`：静态栈 72 字节，圈复杂度 3。

HAL I2C 层也有对应构建产物：`Debug/Drivers/STM32F1xx_HAL_Driver/Src/stm32f1xx_hal_i2c.su`
记录 `HAL_I2C_Mem_Write()` 静态栈 40 字节、`HAL_I2C_Mem_Read()` 静态栈 56 字节；同目录
`.cyclo` 文件记录二者圈复杂度分别为 13 和 21。这说明本章的阻塞式寄存器读写最终会落到 HAL
阻塞内存读写函数，而这些 HAL 函数自身还有比项目封装更复杂的状态判断和等待路径。

但 `.map`、`.list`、`.su` 和 `.cyclo` 不能越界解释为硬件实测证据。它们不能证明：

- MPU6050 实物已经响应、`WHO_AM_I` 真实读回 `0x68`。
- 五个配置寄存器写入后真实保持了期望值，因为当前代码没有写后读回校验。
- `SMPLRT_DIV` 与 `DLPF_CFG` 对应的样本是否被 500Hz 主循环按预期消费。
- `MPU6050_ComputeStaticError()` 期间 5000 次读取全部有效。
- 初始化失败后主循环是否被门控停止。
- 运行时真实栈水位、I2C 超时概率、总线等待时长或一次完整启动的实际耗时。

这些仍需要串口日志、调试器变量、寄存器读回、逻辑分析仪、示波器或实际闭环响应记录；缺失时应保持
`【待验证】`。本节的意义只是把“源码路径存在”推进到“当前构建中确实有该路径和静态分析产物”，
不要把构建证据误写成运行时正确性证明。

### 8.9 权威资料与项目证据分层

本章还要把“手册结论”和“项目结论”分开。TDK/InvenSense 的 `MPU-6000 and MPU-6050 Register Map
and Descriptions, Revision 4.2` 可以证明寄存器编号、位域、复位默认值和采样率公式，例如
`WHO_AM_I` 位于 `0x75` 且默认值为 `0x68`，`SMPLRT_DIV` 位于 `0x19`，`CONFIG` 位于 `0x1A`，
`GYRO_CONFIG` 位于 `0x1B`，`ACCEL_CONFIG` 位于 `0x1C`，以及
`Sample Rate = Gyroscope Output Rate / (1 + SMPLRT_DIV)`。TDK Product Center 的 MPU-6050
页面还能作为器件级背景，确认它是 InvenSense 品牌的 6 轴 IMU，并支持 I2C 数字输出。

这些权威资料只能定义器件“应如何解释寄存器”，不能证明本项目“已经这样写入寄存器”。项目源码层的证据来自
`Drivers/CustomDrivers/Inc/mpu6050.h` 和 `Drivers/CustomDrivers/Src/mpu6050.c`：头文件给出宏名、
地址和值，`MPU6050_Init()` 先读 `REG_WHO_AM_I`，再依次写
`REG_PWR_MGMT_1`、`REG_SMPLRT_DIV`、`REG_CONFIG`、`REG_GYRO_CONFIG` 和
`REG_ACCEL_CONFIG`。`Core/Src/main.c` 的证据进一步把第三个实参固定为
`DLPF_CFG_260HZ`，而不是 `config.c` 中的 `eepromConfig.dlpfSetting`。

构建产物层只能说明这些源码路径进入了当前 Debug 固件：`.map` 证明函数和符号被链接，`.list` 证明
`main()` 跳转到 `MPU6050_Init()` 并保留寄存器读写调用，`.su` 与 `.cyclo` 证明当前编译配置下的
静态栈用量和圈复杂度。它们不能替代实物读回，也不能证明 AD0 电平、供电、上拉、电气时序或 MPU6050
内部寄存器最终保持了这些值。

因此本章最稳妥的结论应写成四层：

- 权威资料证明：这些寄存器地址、位域和采样率公式是 MPU6050 的定义。
- 源码证明：本项目尝试按这些宏和值执行初始化读写。
- 构建产物证明：这些初始化路径进入当前 Debug 构建。
- 运行与硬件证明：真实 `WHO_AM_I` 读数、写后读回值、总线波形、样本新鲜度和闭环响应仍需实测，缺失时保持【待验证】。

## 9. 调试方法

调试 MPU6050 初始化时，建议按以下顺序检查。

第一步，确认 `MPU6050_Init()` 是否被调用。可在 `Core/Src/main.c` 中调用处设置断点，或观察串口是否输出 `Starting MPU6050 Init`。

第二步，确认 `WHO_AM_I` 是否读取成功。正常情况下，日志应显示设备 ID 为 `0x68`。如果三次重试都失败，优先检查：

- `MPU6050_I2C_ADDR` 是否与 AD0 引脚状态一致。
- I2C2 的 SCL/SDA 引脚是否和硬件连接一致。
- MPU6050 是否供电正常。
- I2C 线是否有上拉。
- 第14章中的 I2C2 初始化是否正常执行。

第三步，区分 HAL 返回状态和 I2C 具体错误原因。`HAL_StatusTypeDef` 只能告诉读者结果属于
`OK`、`ERROR`、`BUSY` 还是 `TIMEOUT`；如果要进一步区分 NACK、仲裁丢失、总线错误等 I2C 原因，
应结合 `HAL_I2C_GetError(&hi2c2)` 或调试器里的 `hi2c2.ErrorCode`。当前项目错误日志里的
`1=NoAck, 2=Error, 3=Timeout, 4=Busy` 不应作为 HAL 枚举表使用。

第四步，逐项确认寄存器写入是否返回 `HAL_OK`。当前代码对每次写寄存器都检查 `status != HAL_OK`
并返回 `false`，可在每个 `MPU_Write_Reg()` 调用后设置断点。

第五步，必要时增加寄存器读回验证。写入返回 `HAL_OK` 只能证明 HAL 传输没有报告失败；
若要证明 `PWR_MGMT_1`、`SMPLRT_DIV`、`CONFIG`、`GYRO_CONFIG`、`ACCEL_CONFIG` 的真实值，
需要在初始化后读回对应寄存器，或者用逻辑分析仪观察 I2C 写入与读回帧。

第六步，确认 `main.c` 传入的量程和 DLPF 是否符合预期。特别注意当前硬件初始化实参是
`DLPF_CFG_260HZ`，不是 `config.c` 中的 `DLPF_CFG_184HZ`。

第七步，不要只根据 `SMPLRT_DIV_100HZ` 的名字判断当前真实采样率。当前 `DLPF_CFG_260HZ`
与头文件中的采样率公式存在条件差异，真实采样基准应作为【待验证】项记录。

第八步，观察静态误差采样阶段。若 `WHO_AM_I` 和寄存器写入已经成功，但启动长时间停留在初始化附近，
需要记住 `MPU6050_ComputeStaticError()` 有 5000 次循环和每次 1000 us 延时，启动耗时不能只按
200 ms 上电等待估算。

第九步，区分项目内证据和仓库外实测证据。当前仓库可以证明 I2C2 配置、设备地址、寄存器读写和串口失败提示；
AD0 实际电平、MPU6050 供电、外部上拉和线序需要硬件资料或仓库外实测，统一标记为【待验证】。

第十步，初始化成功后再进入第19章的数据读取验证。不要在初始化尚未成功时分析姿态角或 PID 输出。

第十一步，确认是否需要数据就绪或 FIFO 证据。若调试目标只是证明当前初始化路径能让 500Hz 主循环读到输出寄存器，当前寄存器写入清单已经足够作为软件入口证据；若调试目标是证明“每次读取都是新样本”或“没有漏掉 MPU6050 内部更新”，则必须额外增加 `INT_STATUS` Data Ready、外部中断、FIFO 计数、样本时间戳或逻辑分析仪证据。当前仓库没有这些证据。

第十二步，检查初始化失败后的门控链路。当前 `mpu_ok` 没有参与 500Hz 分支判断，`systemReady` 也会在初始化 `if/else` 之后被置为 `true`。若调试目标是“初始化失败后系统不再读取传感器或不再进入控制计算”，需要增加明确门控或记录现有行为，否则不能从 `MPU6050_Init()` 返回 `false` 自动推出这个结论。

调试记录建议：

- 记录 `MPU6050_Init()` 调用点、实参、`WHO_AM_I` 读回值、每次寄存器写入地址和值。
- 对初始化失败，应分开记录 HAL 返回状态、设备 ID、失败寄存器和串口日志。
- 对 I2C 失败，应补充 `hi2c2.ErrorCode` 或 `HAL_I2C_GetError(&hi2c2)`，不要只复制串口状态解释。
- 对初始化失败后的运行行为，应记录 `mpu_ok`、`systemReady`、`frame_500Hz` 和 `MPU6050_Read_And_Process()` 是否仍被执行。
- 采样率记录应同时包含 `SMPLRT_DIV`、`DLPF_CFG`、头文件宏名和手册/实测所需证据。
- 对样本同步能力，应分开记录“初始化寄存器已写入”“Data Ready 是否启用”“FIFO 是否启用”“主循环是否按 500Hz 轮询读取”；当前项目只有第一项和第四项有仓库源码证据。
- AD0 电平、供电、上拉、线序和真实采样基准属于仓库外证据，缺失时标记为【待验证】。

性能记录建议：

- `HAL_Delay(200)` 是固定启动等待，不属于稳态控制循环开销。
- `WHO_AM_I` 最多重试 3 次，每次失败后 `HAL_Delay(50)`，失败路径会明显拉长启动时间。
- 每次 `MPU_Write_Reg()` 和 `MPU_Read_Reg()` 都设置 100 ms 超时，异常总线状态可能让启动阶段等待更久。
- 若初始化失败后主循环仍反复调用 `MPU6050_Read_And_Process()`，单次 I2C 读取 100 ms 超时会完全超过 500Hz 的 2 ms 周期预算。
- `MPU6050_ComputeStaticError()` 至少包含 `5000 * 1000us = 5s` 的理想延时下限，
  还没有计入 5000 次 I2C 读取和计算开销。
- 当前初始化使用阻塞式 HAL I2C Mem 接口；它适合一次性启动配置，但不应把这种阻塞模型直接搬到高频控制环。
- 当前 DLPF 配置偏高带宽，噪声抑制较弱但传感器延迟较低；低带宽配置会降低噪声，同时增加相位延迟。

## 10. 常见问题

问题一：为什么 HAL 地址要写 `(0x68<<1)`。

STM32 HAL 的 I2C 设备地址参数通常使用左移后的 8 位地址格式。
本项目把 7 位地址 `0x68` 左移一位后传入 HAL。
读者调试时不要再额外左移一次。
如果后续使用示波器或逻辑分析工具核对总线，应区分“7 位设备地址”和“总线上带读写位的地址帧”。

问题二：`MPU_WHO_AM_I_VAL` 已经定义为 `0x68`，为什么代码里还直接比较 `0x68`。

这是源码中的一致性问题。`mpu6050.h` 定义了 `MPU_WHO_AM_I_VAL`，但 `MPU6050_Init()` 实现中直接使用 `0x68`。当前两者数值一致，不影响功能；后续整理时可以改成使用宏名，提高可读性。

问题三：为什么 `config.c` 里是 `DLPF_CFG_184HZ`，实际初始化却是 `DLPF_CFG_260HZ`。

因为 `Core/Src/main.c` 直接调用 `MPU6050_Init(ACCEL_FS_4G, GYRO_FS_500DPS, DLPF_CFG_260HZ)`，
没有使用 `eepromConfig.dlpfSetting`。
所以硬件实际写入值以 `main.c` 调用为准。

问题四：写入 `SMPLRT_DIV_100HZ` 是否一定表示当前 MPU6050 按 100Hz 输出。

不能这样下结论。`SMPLRT_DIV_100HZ` 宏值确实为 9，
但头文件中的 100Hz 公式以 DLPF 启用时的 1 kHz 内部频率为前提。
当前项目传入 `DLPF_CFG_260HZ=0x00`，而头文件提示 0/7 属于另一种内部频率条件。
按 MPU6050 寄存器公式，陀螺仪寄存器更新基准可推导为 800Hz。
仍需验证的是项目 500Hz 控制循环实际读取到了哪些样本、是否重复读取旧样本、是否存在 I2C 丢样或时序抖动。

问题五：初始化成功是否代表姿态角已经正确。

不代表。
初始化成功只说明 MPU6050 设备响应，并且基础寄存器写入流程走通。
姿态角还依赖连续读取、坐标转换、标定、滤波和姿态解算。
当前仓库没有仓库外实测证据证明初始化后的角度方向、噪声、漂移或动态响应已经正确。

问题六：为什么本章不讲 `MPU6050_Read_And_Process()`。

因为它属于第19章 `MPU6050连续数据读取`。本章只讲初始化，为后续读取建立前提。

问题七：寄存器初始化能否证明采样率、量程和滤波参数都符合预期。

只能证明当前源码试图写入这些寄存器。
采样率、量程和滤波参数是否真正符合预期，还需要读回寄存器或进行仓库外实测验证。
特别是 `SMPLRT_DIV_100HZ` 与 `DLPF_CFG_260HZ` 的组合存在解释边界，
教材不能把宏名直接当成硬件输出频率结论。

问题八：失败日志中的 `1=NoAck, 2=Error, 3=Timeout, 4=Busy` 是否等于 HAL 状态枚举。

不等于。
本仓库 HAL 头文件里 `HAL_StatusTypeDef` 是 `HAL_OK=0`、`HAL_ERROR=1`、`HAL_BUSY=2`、
`HAL_TIMEOUT=3`。
因此串口日志里的文字解释只能作为项目作者的调试提示，不能当作 HAL 官方枚举。
若要确认是否 NACK，应结合 I2C 错误码、调试器变量或总线波形。

问题九：为什么初始化成功前可能等待数秒。

因为 `MPU6050_Init()` 在寄存器写入和灵敏度缓存更新后还会调用
`MPU6050_ComputeStaticError()`。
该函数循环 5000 次，每次末尾有 `DWT_Delay_us(1000)`，
仅延时语句就给出约 5 秒的理想下限。
实际启动耗时还会包含 I2C 读取和计算开销，真实数值需要【待验证】。

问题十：为什么当前初始化没有启用 Data Ready 中断或 FIFO。

从源码证据看，当前项目采用 500Hz 软件节拍轮询输出寄存器的读取路线。Data Ready 中断和 FIFO 都会让数据入口从“主循环定期读取”变成“传感器事件或缓冲区驱动读取”，需要额外的寄存器写入、GPIO/EXTI 或 FIFO 排空策略。当前仓库没有这些代码证据，所以教材只能把它们写成可选扩展路线，不能写成已实现能力。

问题十一：不启用 Data Ready 是否意味着每次 500Hz 读取都可能是旧数据。

只能说“源码不能证明每次都是新数据”。当前第18章可推出寄存器更新基准，第19章可证明主循环每 500Hz 尝试读取一次输出寄存器；但没有 Data Ready、FIFO、样本序号或读回时间戳时，不能仅凭一次 14 字节读取成功判断样本相位和新鲜度。

问题十二：`MPU6050_Init()` 返回失败后，主循环是否会自动停止读取 MPU6050。

当前源码不能这样下结论。`Core/Src/main.c` 定义了 `mpu_ok`，但没有把它接入初始化成功分支或 500Hz 分支门控；`systemReady` 也在初始化 `if/else` 之后被置为 `true`。所以初始化失败会打印失败日志，但当前证据不能证明后续 `MPU6050_Read_And_Process()`、AHRS 更新和控制计算已经被停止。

## 13. 本章总结

本章完成了 MPU6050 传感器主线的初始化入口分析。

项目在 `Drivers/CustomDrivers/Inc/mpu6050.h` 中定义寄存器地址、配置值和灵敏度宏。
`Drivers/CustomDrivers/Src/mpu6050.c` 通过 `MPU_Write_Reg()`、`MPU_Read_Reg()` 封装 HAL I2C 寄存器访问。
`Core/Src/main.c` 调用 `MPU6050_Init(ACCEL_FS_4G, GYRO_FS_500DPS, DLPF_CFG_260HZ)` 启动硬件初始化。

本章已经证明：

- 设备识别依赖 `REG_WHO_AM_I` 和期望值 `0x68`。
- 唤醒依赖 `REG_PWR_MGMT_1` 和 `PWR1_WAKEUP_Z_GYRO`。
- 采样率、DLPF、陀螺仪量程、加速度计量程分别写入对应寄存器。
- 当前实际 DLPF 初始化参数来自 `Core/Src/main.c` 的实参，而不是 `config.c` 中的默认配置字段。
- `MPU6050_Init()` 直接把 `dlpf` 写入 `REG_CONFIG` 整字节；当前宏值安全，但未来接入可调配置时需要校验高位和外部同步位。
- 当前写入了 `SMPLRT_DIV_100HZ` 的宏值，但该宏名的 100Hz 前提是 DLPF 启用后的 1kHz 陀螺仪输出率。
- 按 MPU6050 手册公式，当前 `DLPF_CFG_260HZ` 与分频值 9 可推出 800Hz 陀螺仪寄存器更新基准；需要实测验证的是控制环实际读取频率和数据是否被重复或丢失。
- 控制环真实读取频率、丢样情况和动态响应仍需要结合运行日志或外部测量【待验证】。
- `MPU6050_Init()` 的成功路径包含 `MPU6050_ComputeStaticError()`，仅其中 5000 次微秒延时
  就给出约 5 秒启动耗时下限。
- 当前失败日志对 HAL 状态的文字解释与 `HAL_StatusTypeDef` 枚举不完全一致，NACK 等具体原因应结合
  I2C 错误码或总线波形判断。
- 项目头文件中部分 MPU6050 位域注释不能替代手册，教材分析应以手册位域和实际宏二进制值为准。
- 权威资料、项目源码、构建产物和实物运行证据应分层使用：手册定义寄存器语义，源码证明项目尝试写入，构建产物证明路径进入固件，硬件读回和总线记录才证明真实生效。
- `.map` 和 `.list` 能证明初始化函数、底层读写封装和寄存器写入路径进入当前构建，并能定位
  `MPU6050_Init` 的链接地址与 `main()` 调用跳转。
- `.su` 和 `.cyclo` 能补充当前构建下 `MPU6050_Init()`、寄存器读写封装、静态误差采样函数以及
  HAL 阻塞 I2C 读写函数的静态栈用量和圈复杂度，但不能替代运行时栈水位、总线耗时或启动耗时实测。
- 当前 `MPU6050_Init()` 的失败返回值没有形成 500Hz 主循环门控；`mpu_ok` 未参与读取判断，`systemReady` 会在初始化 `if/else` 之后置为 `true`。
- 当前初始化没有写 Data Ready 中断、FIFO 或轴待机相关寄存器；因此项目后续读取路径应解释为 500Hz 主循环轮询输出寄存器，而不是传感器中断或 FIFO 驱动。

本章保留以下边界：

- 不展开 `MPU6050_Read_And_Process()` 的连续读取流程。
- 不展开 `MPU6050_ComputeStaticError()` 的标定和零偏处理。
- 不把初始化成功等同于姿态解算正确。
- 不把初始化失败日志等同于主循环已经停止读取或停止控制。
- 不把 `.map` / `.list` 中的构建路径证据等同于 MPU6050 实物寄存器已经按预期写入。
- 不把 `.su` / `.cyclo` 中的静态栈和圈复杂度证据等同于运行时栈水位、I2C 超时概率或启动耗时实测。
- 不把当前初始化写成已经具备样本新鲜度证明；Data Ready、FIFO、样本序号或外部测量仍属于【待验证】。
