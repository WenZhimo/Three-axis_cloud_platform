## 2. 前置知识

本章正式前置知识是 `STM32F103RCTx芯片平台`。读者还应具备基本 C 语言阅读能力，能够识别数组、结构体、联合体和枚举的声明形式。

本章内部的学习顺序也很重要：

1. 先看 16 位原始数据如何用联合体承接。
2. 再看全局配置结构如何汇总系统参数。
3. 最后看电机轴枚举如何作为底层驱动 API 的入口参数。

这样安排的原因是，`全局配置结构eepromConfig_t` 在 `知识点/知识依赖图.md` 中依赖 `16位数据联合体解析`；两者同属第17章，所以本章必须在正文内部先完成联合体解释，再进入配置对象。

## 3. 问题背景

三轴云台项目不是单个外设例程。它同时包含传感器、姿态估计、滤波、PID、电机输出和调试输出。随着模块增加，如果每个模块都直接使用零散的全局变量，代码会很快失去可读性。

项目需要回答三个基础问题：

- MPU6050 输出的是连续字节，如何把两个字节恢复成一个有符号 16 位原始量。
- 控制系统有大量参数，如何统一保存默认值、调参值和运行使能状态。
- 控制算法关心 Roll、Pitch、Yaw，电机驱动关心具体 PWM 通道，二者之间如何传递轴信息。

第17章解决的是这些数据入口问题。它不负责讲 MPU6050 如何初始化，也不负责讲 PID 如何计算，更不负责讲三相正弦输出如何写 CCR。那些内容分别留给第18章、第23章以后和第29章以后。

## 4. 核心概念

### 4.1 16位数据联合体解析

MPU6050 的加速度、角速度和温度原始数据都是 16 位量，但 I2C 总线一次读取到的是字节流。项目在 `Drivers/CustomDrivers/Inc/mpu6050.h` 中定义：

```c
typedef union {
    int16_t value;
    uint8_t bytes[2];
} Int16_Converter_t;
```

这个联合体把同一段内存同时解释为两个 `uint8_t` 字节和一个 `int16_t` 数值。

后续读取函数可以先把字节放进 `bytes[0]`、`bytes[1]`，再通过 `value` 取得 16 位结果。

这个写法有两个前提需要显式讲清。

第一，MPU6050 寄存器输出顺序是高字节在前、低字节在后；项目在第19章对应代码中把高字节写入 `bytes[1]`，低字节写入 `bytes[0]`。

第二，项目运行在 STM32F103 Cortex-M3 环境，当前 CMSIS 代码路径按 little endian 处理。
`Drivers/CMSIS/Include/cmsis_gcc.h` 中也能看到 `__ARMEB__` 未定义时进入 little endian 分支。

因此，本章不能把联合体解析写成所有平台都无条件可移植的技巧。若要写成跨平台解析，显式移位组合通常更容易审查。

`Drivers/BGC/bgc32.h` 中也存在同类联合体：`int16andUint8_t`、`int32andUint8_t`、`uint16andUint8_t`。

这说明项目不只在 MPU6050 头文件里偶然使用这种模式，而是把“字节数组和整数视图共用内存”作为嵌入式数据解析手段。

### 4.2 全局配置结构eepromConfig_t

`eepromConfig_t` 定义在 `Drivers/CustomDrivers/Inc/mpu6050.h`。它不是单一外设配置，而是全项目运行参数集合，包含：

- 传感器温度补偿参数：`accelTCBiasSlope`、`accelTCBiasIntercept`、`gyroTCBiasSlope`、`gyroTCBiasIntercept`。
- 姿态估计和滤波参数：`KpAcc`、`KiAcc`、`KpMag`、`KiMag`、低通滤波时间常数。
- PID 参数数组：`PID[3]`。
- 电机功率和使能状态：`rollPower`、`pitchPower`、`yawPower`、`rollEnabled`、`pitchEnabled`、`yawEnabled`。
- 电机极对与输出限制相关参数：`rollMotorPoles`、`pitchMotorPoles`、`yawMotorPoles`、`rateLimit`。

它的全局实例在 `Core/Src/main.c` 中定义：

```c
eepromConfig_t eepromConfig;
```

默认值由 `Drivers/SRC/Src/config.c` 中的 `init_eepromConfig(bool eepromReset)` 写入。

阅读 `eepromConfig` 时不能只看结构体定义。它至少有五个阶段：

1. 类型定义：`Drivers/CustomDrivers/Inc/mpu6050.h` 定义字段集合。
2. 实例定义：`Core/Src/main.c` 定义全局对象 `eepromConfig`。
3. 默认初始化：`Drivers/SRC/Src/config.c` 写入默认值。
4. 启动覆盖：`Core/Src/main.c` 在 `init_eepromConfig(true)` 后覆盖 PID 和轴使能字段。
5. 运行读写：控制循环、PID、滤波、温度补偿和使能逻辑继续读取或修改部分字段。

所以 `eepromConfig_t` 不是“默认值清单”，而是项目状态建模的中心对象。
还要注意 `init_eepromConfig(true)` 的 `true` 在当前实现中表示进入默认写入分支。
`config.c` 没有实现 `eepromReset == false` 时的 EEPROM 读取分支，`writeEEPROM()` 也处于注释状态。
因此即使 `main.c` 注释写着“恢复之前的系统参数”，当前源码证据也只能证明它写入默认配置，不能证明从非易失存储恢复历史配置。

这一点可以写成状态转移，而不是只看函数名：

```text
C0 = .bss 清零后的 eepromConfig
F(true)  : C0 -> Cdefault
F(false) : Cx -> Cx
Coverride: Cdefault -> Cmain
```

其中 `F(false)` 在当前源码中没有 `else` 分支，因此更准确地说是“不改变当前对象”，不是“从 EEPROM 读取”。如果它在冷启动默认写入之前被调用，`eepromConfig` 会继续保持 `.bss` 清零后的 0 值；如果它在运行时被调用，已有运行状态和调参值会原样保留。教材讨论配置恢复时必须把这两种情况分开。

这里还要建立一张“EEPROM 证据等级表”，避免把不同层级的证据混在一起：

| 证据层级 | 当前项目证据 | 能证明什么 | 不能证明什么 |
|---|---|---|---|
| 函数名和注释 | `main.c` 注释写“恢复之前的系统参数”，函数名含 `eepromConfig` | 只能提示原作者可能想把该对象设计成可持久化配置入口 | 不能替代函数体证据，不能证明已经读取非易失存储 |
| 调用实参 | `.list` 显示 `main()` 先执行 `movs r0, #1`，再 `bl 8004bfc <init_eepromConfig>` | 当前 Debug 构建实际调用的是 `init_eepromConfig(true)` | 不能证明 `false` 分支在当前启动链路中出现 |
| 函数体分支 | `config.c` 只有 `if (eepromReset)` 默认写入分支，没有 `else` 读取分支 | `true` 路径会逐字段写入默认参数；`false` 路径当前等价于直接返回 | 不能证明存在 EEPROM 读取、校验、版本迁移或 CRC 校验 |
| 写入接口 | `config.c` 末尾只有被注释掉的 `writeEEPROM()`；业务源码未找到 `readEEPROM()`、`writeEEPROM()` 实现或调用 | 当前仓库没有形成配置持久化读写闭环 | 不能因为字段名叫 `eepromConfig` 就写成已实现持久化 |
| HAL/FLASH 库代码 | 工程包含 `stm32f1xx_hal_flash.c`，`Core/Inc/stm32f1xx_hal_conf.h` 启用 FLASH HAL | 库层具备 Flash 编程 API 的来源 | 不能证明项目业务代码已经用这些 API 保存或恢复配置 |
| 构建产物 | `.map` 给 `init_eepromConfig` 分配最终地址 `0x08004bfc`，大小 `0x348`；`.su/.cyclo` 给出 16 字节栈估计和圈复杂度 2 | 默认配置函数进入当前最终镜像，且存在一个 `if` 级别的静态分支复杂度 | 不能证明 EEPROM 持久化正确，也不能证明断电后参数会保留 |

这张表的教学意义是：配置持久化至少需要“存储介质、写入格式、读取入口、有效性校验、默认回退、版本兼容”几层证据。当前第17章只能证明默认配置对象和默认写入路径，不能把它提升为已经完成的 EEPROM 配置系统；相关行为仍保留为【待验证】。

### 4.3 枚举定义电机轴

`Drivers/CustomDrivers/Inc/drv_pwmMotors.h` 中定义了电机轴枚举：

```c
typedef enum {
    MOTOR_ROLL = 0,
    MOTOR_PITCH,
    MOTOR_YAW,
} MotorAxis_t;
```

底层接口 `PWM_Motor_SetAngle(MotorAxis_t axis, float angle_rad, float power_percent)` 使用这个枚举作为第一个参数。

这样，控制层调用时传入的是“哪个轴”，底层驱动再根据 `axis` 选择实际 PWM 通道组合。

### 4.4 数据对象生命周期

本章还需要补一个隐藏前置知识：全局对象不是从定义开始就等于“最终运行参数”。

`Core/Src/main.c` 中的 `eepromConfig_t eepromConfig` 没有显式初始化值。
在当前 STM32 工程中，链接脚本把未初始化全局对象放入 `.bss`，
启动文件 `startup_stm32f103rctx.s` 在调用 `main()` 前执行 `.bss` 清零。
因此，`eepromConfig` 的最早状态是清零后的 RAM 对象，
随后才由 `init_eepromConfig(true)` 写入默认值，再由 `main.c` 覆盖部分字段。

这说明第17章不能只讲结构体字段，还要讲对象生命周期：

- 存储位置：对象在 RAM 中占用空间。
- 初始状态：启动代码清零 `.bss`。
- 默认写入：`config.c` 写入默认配置。
- 启动覆盖：`main.c` 覆盖 PID、使能和部分状态。
- 运行改写：PID、控制逻辑、校准逻辑继续修改部分字段。
- 读者判断：只有沿着这条链路，才能知道某字段在当前主线中的真实值。

这些概念仍然服务于正式知识点 `全局配置结构eepromConfig_t`，不是新增知识点。

同一个 `main.c` 中还定义了 `sensors_t sensors`，但它的生命周期语义不同。`sensors` 的字段在 500Hz 主循环中由 `rawAccel`、`rawGyro` 换算结果、AHRS 输出和控制回路持续刷新；它描述的是“当前观测到的运行状态”，不是“启动时写入的一套配置”。把这两类对象分开看，读者才能理解为什么 `eepromConfig` 要沿着 `.bss -> 默认值 -> 覆盖值 -> 运行改写` 追踪，而 `sensors` 要沿着 `采样 -> 物理量换算 -> 姿态估计 -> 控制读取` 追踪。

## 5. 工作原理

本章三个对象分别位于项目数据流的不同层级。

`Int16_Converter_t` 解决输入数据的最小单位问题。传感器给出字节，控制算法需要整数和浮点物理量，中间必须有一步把字节组合成 16 位原始值。

`eepromConfig_t` 解决全局参数归属问题。姿态估计、滤波、PID、输出限幅和使能状态都需要参数。如果这些参数分散在多个文件里，调试时很难判断当前系统使用的到底是哪一套配置。把它们集中到 `eepromConfig` 后，初始化、运行修改和后续调参都有统一入口。

`MotorAxis_t` 解决抽象轴和硬件输出之间的接口问题。控制算法使用 Roll、Pitch、Yaw 描述姿态轴；底层驱动需要写 TIM2、TIM3、TIM4 的不同 CCR。枚举使上层不必直接知道所有通道细节。

三者连接起来后，项目形成一条清晰的基础数据路径：

传感器字节流 -> `Int16_Converter_t` 原始量 -> 后续传感器处理 -> `eepromConfig` 参数约束 -> 控制输出 -> `MotorAxis_t` 选择电机轴 -> PWM 电机驱动。

这条路径还可以按“数据对象角色”拆开：

- 输入承接对象：`rawAccel`、`rawGyro`、`rawMPU6050Temperature`。
- 配置参数对象：标定系数、滤波时间常数、PID 参数、功率和限幅。
- 实时观测状态对象：`sensors.accel500Hz`、`sensors.gyro500Hz`、`sensors.margAttitude500Hz`。
- 运行控制状态对象：轴使能、PID 历史项、姿态收敛后打开控制的状态。
- 接口标识对象：`MotorAxis_t` 把控制层轴名传递给底层驱动。

教材要让读者看到这些角色差异。否则结构体、联合体和枚举会被误读成同一种“变量”。

还可以再加一条审查规则：每个数据对象至少要回答六个问题。

1. 谁定义类型。
2. 谁分配存储。
3. 谁初始化。
4. 谁覆盖默认值。
5. 谁在运行中写入。
6. 谁在控制主线中读取。

如果只能回答前两个问题，只能说明“对象存在”。
如果能回答六个问题，才能说明它在当前项目中形成完整数据链。

第七个问题也很重要：谁通过指针别名间接写入。
例如 `computeMotorCommands.c` 调用 `updatePID(..., &eepromConfig.PID[ROLL_PID])`。
`pid.c` 中的 `updatePID()` 通过 `PIDparameters->iTerm`、`PIDparameters->lastDcalcValue`、`PIDparameters->lastDterm` 和 `PIDparameters->lastLastDterm` 回写同一块结构体内存。
因此，搜索 `eepromConfig.PID` 只能找到显式写入，不能穷尽通过 `PIDdata_t *PIDparameters` 发生的运行时改写。

从内存成本看，数据对象也不是免费的。

`Int16_Converter_t` 的有效载荷是 2 字节，所以 `rawAccel[3]`、`rawGyro[3]` 和 `rawMPU6050Temperature` 合计为：

`3 * 2 + 3 * 2 + 2 = 14` 字节

这刚好对应 MPU6050 一次连续读取的 14 字节数据入口。

`PIDdata_t` 至少包含 9 个 `float` 和 2 个 `uint8_t` 字段，
所以单个 PID 对象的字段下限为：

`9 * 4 + 2 = 38` 字节

`eepromConfig_t` 中的 `PID[3]` 仅 PID 字段下限就至少为：

`38 * 3 = 114` 字节

整个 `eepromConfig_t` 按字段计数至少包含 73 个 `float` 和 12 个 `uint8_t` 字段，
字段下限为：

`73 * 4 + 12 = 304` 字节

这还没有计算编译器为了 4 字节对齐插入的 padding。
当前 Debug `.map` 已把 `eepromConfig` 放进 `.bss.eepromConfig`，大小为 `0x144` 字节，也就是 324 字节；这比字段下限 304 字节多出 20 字节，说明对象实际尺寸必须按编译器布局结果来读，不能只做字段求和。
Arm AAPCS32 对复合类型的定义也支持这一点：聚合类型的对齐取最对齐成员，大小取足以容纳全部成员且满足对齐要求的最小倍数。
因此，最终 `sizeof(eepromConfig_t)` 应以目标编译器为准【待验证】。这段推导的目的不是精确替代编译器结果，而是让读者知道“集中配置对象”也会消耗 RAM，而且字段数量并不直接等于最终对象大小。

## 6. STM32实现机制

在 STM32 工程中，这些对象本身不是外设寄存器，但它们服务于外设数据流。

对于 I2C 读取，STM32 从 MPU6050 寄存器得到的是 `uint8_t` 缓冲区。由于 MPU6050 的原始加速度和角速度寄存器按高字节、低字节排列，项目需要在软件中完成字节组合。`Int16_Converter_t` 提供了一个轻量的数据承接容器。

对于配置对象，STM32F103RCTx 没有操作系统级配置管理。项目用一个全局结构体实例 `eepromConfig` 作为运行期参数中心。它在 `main()` 初始化阶段通过 `init_eepromConfig(true)` 写入默认值，后续模块直接读取或修改其中字段。

全局对象带来方便，也带来边界风险。任何包含 `mpu6050.h` 的模块都可能访问 `eepromConfig`，因此调试时必须追踪“谁写入”和“谁读取”。

从启动机制看，`eepromConfig` 的值经历了三层初始化。

第一层是启动文件清零 `.bss`。`startup_stm32f103rctx.s` 中的 `LoopFillZerobss` 会把 `_sbss` 到 `_ebss` 区间写 0。

第二层是项目默认配置。`init_eepromConfig(true)` 在 `config.c` 中逐字段写入默认参数。

第三层是项目当前调参覆盖。`main.c` 在初始化后重新写入三个轴的 PID 参数和轴使能状态。

因此调试 `eepromConfig.PID[ROLL_PID].P` 时，不能只看 `config.c` 中的 10.0f。
当前启动代码后续把它覆盖为 0.03f，运行时调试应以覆盖后的值为准。

当前项目中的写入证据包括：

- `config.c` 写入默认参数。
- `main.c` 覆盖 PID 参数和轴使能。
- `pid.c` 清零或设置 PID 积分和 D 项历史状态。
- `updatePID()` 通过 `PIDdata_t *PIDparameters` 指针回写 PID 积分和 D 项历史；当前调用点传入的是 `&eepromConfig.PID[...]`。
- `main.c` 的 500Hz/10Hz 逻辑改写轴使能状态。
- `mpu6050.c` 的 `orientIMU()` 会把 `imuOrientation` 写为 10。
- `mpu6050Calibration.c` 中温度补偿标定函数会写入斜率和截距，但主线调用证据需单独核对。

读取证据包括：

- `computeMotorCommands.c` 读取轴使能、PID 参数和 `rateLimit`。
- `firstOrderFilter.c` 读取低通滤波时间常数。
- `mpu6050.c` 读取温度补偿斜率、截距和 `imuOrientation`。
- `MargAHRS.c` 读取 `KpAcc` 参与姿态误差反馈。

这些读写都围绕普通全局对象展开。
当前 `eepromConfig` 不是 `const`，也不是 `volatile`，字段写入没有统一封装函数。
这说明它更像“共享运行状态和配置容器”，而不是严格只读配置表。
本章不把它评价为错误，但要让读者看见这种工程取舍。

对于电机轴枚举，STM32 的定时器输出最终仍然落实到寄存器写入。例如 `PWM_Motor_SetAngle()` 内部会根据 `MotorAxis_t` 选择不同的 `TIMx->CCRn`。枚举不是硬件能力，而是项目在硬件寄存器之上建立的一层软件接口。

## 7. 项目中的应用

本章涉及的项目证据文件包括：

- `Drivers/CustomDrivers/Inc/mpu6050.h`
- `Drivers/BGC/bgc32.h`
- `Drivers/SRC/Src/config.c`
- `Core/Src/main.c`
- `Drivers/CustomDrivers/Src/mpu6050.c`
- `Drivers/CustomDrivers/Inc/drv_pwmMotors.h`
- `Drivers/CustomDrivers/Src/drv_pwmMotors.c`
- `Drivers/SRC/Src/computeMotorCommands.c`

在项目中，`Int16_Converter_t` 的实例包括 `rawAccel[3]`、`rawGyro[3]` 和 `rawMPU6050Temperature`。

它们在 `Drivers/CustomDrivers/Src/mpu6050.c` 中分配实际存储空间。

`Drivers/CustomDrivers/Inc/mpu6050.h` 通过 `extern` 把这些对象暴露给其他编译单元。

`Drivers/CustomDrivers/Src/mpu6050.c` 的读取函数进一步说明它们的生命周期：

1. I2C 读取失败时，`rawAccel` 和 `rawGyro` 被清零。
2. I2C 读取成功后，14 字节缓冲区按高低字节写入联合体。
3. 字节组合后的 `value` 被转成临时直轴数据。
4. `orientationMatrix` 对数据进行旋转。
5. 旋转结果再写回 `rawAccel[axis].value` 和 `rawGyro[axis].value`。

因此，`rawAccel` 和 `rawGyro` 在当前项目中并不总是“刚从寄存器读出的未处理原始值”。
读取后它们会被安装方向矩阵处理后的结果覆盖。这一点应留到第19章继续展开。

`eepromConfig` 在 `Core/Src/main.c` 中定义实际对象。

`Drivers/CustomDrivers/Inc/mpu6050.h` 中声明 `extern eepromConfig_t eepromConfig`。

`Drivers/SRC/Src/config.c` 负责默认值初始化。

`Core/Src/main.c` 中还会在初始化后覆盖部分 PID 和使能字段。

这里存在一个注释与实现的边界。
`main.c` 注释把 `init_eepromConfig(true)` 描述为恢复之前的系统参数，但 `config.c` 的函数体只有 `if (eepromReset)` 默认写入分支。
当前仓库没有看到 `eepromReset == false` 的恢复路径，也没有看到实际 EEPROM 读取函数被调用。
所以教材应以函数体为准，把它写成“强制默认配置写入”，而不是“历史配置恢复”。

`Drivers/SRC/Src/computeMotorCommands.c`、`Drivers/SRC/Src/pid.c`、`Drivers/SRC/Src/firstOrderFilter.c` 等后续模块会读取它。

更准确地说，`eepromConfig` 的当前使用可以分为四类：

- 默认配置源：`config.c`。
- 启动阶段覆盖源：`main.c`。
- 运行状态写入者：`pid.c`、`computeMotorCommands.c`、`main.c`、`mpu6050.c`。
- 指针别名写入者：`updatePID()` 通过调用方传入的 `&eepromConfig.PID[...]` 修改 PID 历史字段。
- 控制参数读取者：`computeMotorCommands.c`、`pid.c`、`MargAHRS.c`。
- 传感器与滤波参数读取者：`mpu6050.c`、`firstOrderFilter.c`。

这组分类能避免一个常见误读：`eepromConfig_t` 不是纯参数对象。
例如 `P/I/D/windupGuard` 更像可调参数，
`iTerm/lastDcalcValue/lastDterm/lastLastDterm` 是 PID 运行历史状态，
`rollEnabled/pitchEnabled/yawEnabled` 是控制门控状态，
`imuOrientation` 又会在 `orientIMU()` 中被覆盖。

把这些字段放在同一个结构体里便于集中观察，
但也让“配置”和“运行状态”的边界变模糊。
教材要把这种混装讲清楚，而不是把结构体名字直接理解为 EEPROM 持久化配置。

`MotorAxis_t` 在 `Drivers/CustomDrivers/Inc/drv_pwmMotors.h` 中定义。

`Drivers/CustomDrivers/Src/drv_pwmMotors.c` 中的 `PWM_Motor_SetAngle()` 根据 `MOTOR_PITCH`、`MOTOR_ROLL`、`MOTOR_YAW` 写入不同定时器通道。

`Drivers/SRC/Src/computeMotorCommands.c` 是控制层调用该接口的主要证据。

## 8. 代码分析

### 8.1 Int16_Converter_t的定义和实例

`Drivers/CustomDrivers/Inc/mpu6050.h` 中定义 `Int16_Converter_t`，随后声明：

```c
extern Int16_Converter_t rawAccel[3];
extern Int16_Converter_t rawGyro[3];
extern Int16_Converter_t rawMPU6050Temperature;
```

`Drivers/CustomDrivers/Src/mpu6050.c` 中定义实际变量：

```c
Int16_Converter_t rawAccel[3];
Int16_Converter_t rawGyro[3];
Int16_Converter_t rawMPU6050Temperature;
```

这体现了 C 工程中的常见组织方式：头文件声明类型和外部变量，源文件分配实际存储。后续第19章讲 MPU6050 连续读取时，再分析这些变量如何装入寄存器读出的 14 字节数据。

这里还要区分“声明”和“定义”：

- `extern Int16_Converter_t rawAccel[3]` 是声明，告诉其他文件对象存在。
- `Int16_Converter_t rawAccel[3]` 是定义，真正分配 RAM。
- 如果头文件里直接定义全局对象，会导致多重定义风险。

这不是语法枝节，而是嵌入式工程能否稳定链接的基本边界。

### 8.1.1 联合体解析的边界

第19章源码会把 MPU6050 高字节写入 `bytes[1]`，低字节写入 `bytes[0]`。

在当前 little endian STM32 编译目标上，这样可以使 `value` 得到期望的 16 位结果。

但这种写法把字节序知识藏在了联合体访问里。等价的显式写法是：

```c
int16_t value = (int16_t)((high << 8) | low);
```

显式移位更啰嗦，但对跨平台审查更清楚。联合体更短，但需要读者知道目标端字节序和项目赋值顺序。

还要区分“对象名”和“观察时刻”。`rawAccel`、`rawGyro` 的名字包含 raw，但 `Drivers/CustomDrivers/Src/mpu6050.c` 中的 `MPU6050_Read_And_Process()` 会把它们作为多阶段承接对象使用：

| 观察时刻 | 源码证据 | 此时能说明什么 | 不能说明什么 |
|---|---|---|---|
| `MPU_Read_Reg(...)` 返回前 | 数据只在局部 `I2C2_Buffer_Rx[14]` 中 | 可以检查 I2C 读取是否成功 | 不能用 `rawAccel[].value` 判断本次寄存器数据 |
| 高低字节写入后 | `bytes[1]` 接高字节，`bytes[0]` 接低字节 | 可以检查联合体字节序和 16 位直轴计数值 | 还不能说明安装方向已经处理 |
| `straightAccelData[]/straightGyroData[]` 赋值后 | 临时数组从 `raw*.value` 取值 | 可以观察矩阵旋转前的直轴数据 | 函数返回后这组局部值不可直接观察 |
| `matrixMultiply(...)` 后 | `rotatedAccelData[]/rotatedGyroData[]` 承接 `orientationMatrix` 结果 | 可以观察项目坐标处理后的临时结果 | 不能证明方向矩阵和实物安装一致【待验证】 |
| `raw*.value` 写回后 | 旋转结果覆盖 `rawAccel[axis].value` 和 `rawGyro[axis].value` | 函数结束后的全局对象是项目坐标入口值 | 不能再把它直接称为未处理寄存器原始值 |

`rawMPU6050Temperature` 不参与这条方向矩阵写回链。它在当前函数中只由温度高低字节组合为 `value`，后续由 `MPU6050_ComputeTemError()` 读取参与温度补偿计算。调试时如果只看全局对象最终值，就会漏掉“寄存器直轴值”和“项目坐标入口值”之间的阶段差异。

### 8.2 eepromConfig_t的字段分组

`eepromConfig_t` 的字段很多，但可以按用途分组阅读：

- 标定类：加速度计、陀螺仪、磁力计偏置相关字段。
- 姿态估计类：`KpAcc`、`KiAcc`、`KpMag`、`KiMag`。
- 控制类：`PID[3]`、`rateLimit`、轴使能字段。
- 电机类：功率、极数、角度限制和自动跟随相关字段。
- 滤波类：500Hz 与 50Hz 低通时间常数。

`Drivers/SRC/Src/config.c` 的 `init_eepromConfig(true)` 会逐项写入默认值。

例如它默认将 `rollEnabled`、`pitchEnabled`、`yawEnabled` 设为 `false`，并设置三个轴的 PID 初值、功率和限幅参数。

按字段性质看，`PID[3]` 又要继续拆开：

- 可调参数：`B`、`P`、`I`、`D`、`windupGuard`、`dErrorCalc`、`type`。
- 运行历史：`iTerm`、`lastDcalcValue`、`lastDterm`、`lastLastDterm`。

这也是为什么 `pid.c` 会修改 `eepromConfig.PID[index].iTerm` 和 D 项历史。
如果读者把整个 `PID[3]` 都当成“固定配置”，就会误判 PID 运行状态的来源。

进一步按当前源码消费关系拆开，`PIDdata_t` 至少包含五类字段：

| 字段 | 当前角色 | 主要证据 | 教学边界 |
|---|---|---|---|
| `P/I/D` | 当前三轴主线调参字段 | `config.c` 写默认值；`main.c` 启动后覆盖三轴；`updatePID()` 用它们合成输出。 | 可以写成当前 PID 计算参数；不能只看 `config.c` 默认值判断运行值。 |
| `B` | 非角度 PID 分支的命令前馈系数 | `config.c` 写默认值；`updatePID()` 的 `type != ANGULAR` 分支读取 `B`。 | 当前默认 `type` 为 `ANGULAR`，三轴主线是否走非角度分支没有证据，不能把 `B` 写成当前三轴有效参数。 |
| `type`、`dErrorCalc` | 算法模式选择字段 | `config.c` 默认写 `ANGULAR` 和 `D_ERROR`；`updatePID()` 用它们选择角度误差包裹和 D 项来源。 | 可以写成当前算法分支选择字段；具体控制效果仍需第27、28章分析。 |
| `iTerm`、`lastDcalcValue`、`lastDterm`、`lastLastDterm` | 运行历史状态字段 | `initPID()`、`zeroPIDintegralError()`、`zeroPIDstates()` 和 `updatePID()` 都会写入；`computeMotorCommands.c` 还在 Roll 轴接管时显式清零。 | 必须视为运行状态，不能随 EEPROM 配置一起理解为只读参数。 |
| `windupGuard` | 已写入但当前消费证据弱的限幅字段 | `config.c` 和 `main.c` 都写入它；当前 `pid.c` 的积分限幅使用硬编码 `10.0f`，未读取 `PIDparameters->windupGuard`。 | 只能写成保留或未接入的限幅参数入口，不能写成当前积分限幅实际来源。 |

这张表说明一个更细的工程边界：同一个结构体数组里，既有当前运行会读取的参数，也有运行中会被覆盖的历史状态，还有字段名看起来有意义但当前主线未消费的保留字段。第17章只负责建立这种读写角色地图；PID 公式、D 项滤波和积分抗饱和的数学细节留到后续 PID 章节继续展开。

需要注意，虽然类型名里带有 EEPROM，但当前证据只显示默认配置写入和全局使用；本轮未发现可证明实际 EEPROM 读写流程的主线证据，因此本章只把它称为“全局配置结构”，不把它扩展为已完成的非易失存储系统。

`init_eepromConfig(bool eepromReset)` 的参数名也不能反向证明存在完整 EEPROM 流程。
当前唯一调用点传入 `true`，函数体在 `true` 分支逐项写默认值，`writeEEPROM()` 被注释。
如果未来要证明“恢复配置”，需要看到读取非易失存储、校验版本或校验和、再写入 `eepromConfig` 的完整路径。

按当前函数体，`eepromReset == false` 的数学语义是恒等映射：

```text
init_eepromConfig(false): eepromConfig_next = eepromConfig_current
```

这条式子有两个工程含义。第一，它不会把默认值写入尚未初始化的字段。第二，它不会清除 PID 历史、轴使能或运行中被改写的字段。若未来有人把调用改成 `init_eepromConfig(false)`，必须同时审查调用发生的时间点：是在 `.bss` 清零后、默认写入前，还是在系统已经运行一段时间之后。

还需要进一步区分“字段存在”和“当前主线消费”。

例如 `dlpfSetting` 在 `config.c` 中有默认值。

但 `main.c` 当前调用 `MPU6050_Init(ACCEL_FS_4G, GYRO_FS_500DPS, DLPF_CFG_260HZ)`，不是直接传入 `eepromConfig.dlpfSetting`。

再比如滤波时间常数在 `firstOrderFilter.c` 中被读取，但其中部分滤波器初始化代码处于注释状态。

所以教材不能把每个字段都写成当前控制主线已生效参数。字段应按“已消费”“默认存在但未进入当前主线”“运行中被改写”“保留扩展”分类。

### 8.3 eepromConfig的实例化和初始化

`Core/Src/main.c` 中的 `eepromConfig_t eepromConfig` 是实际全局对象。

`main()` 初始化阶段调用 `init_eepromConfig(true)`，随后又对部分字段进行项目当前调参覆盖，例如三个轴的 PID 参数和轴使能状态。

这说明项目配置并非只来自 `config.c` 的默认值。读者调试时需要同时检查：

- `Drivers/SRC/Src/config.c` 中的默认配置。
- `Core/Src/main.c` 初始化后对 `eepromConfig` 的二次修改。
- 后续运行过程中控制逻辑是否改写使能或状态字段。

当前源码可以归纳出一条生命周期链：

`main.c` 全局定义
-> 启动文件清零 `.bss`
-> `init_eepromConfig(true)` 默认写入
-> `initFirstOrderFilter()` 使用滤波参数
-> `initPID()` 清零 PID 历史状态
-> `main.c` 覆盖 PID 调参值
-> 500Hz 控制循环读取配置
-> 低频逻辑改写使能字段。

这条顺序解释了一个常见调试误区：只看 `config.c` 默认值，并不一定能知道运行时值。
比如 Roll/Pitch/Yaw PID 的 `P/I/D/windupGuard` 在 `main.c` 中被重新写入，调试时应以覆盖后的值为准。

这里还要补一个时间点：`.bss` 清零后的 `eepromConfig` 在 `init_eepromConfig(true)` 之前虽然已经占用 RAM，但字段值只是 0。
若某个函数在默认配置写入前读取它，会得到清零值，而不是 `config.c` 的默认参数。
当前 `main.c` 的顺序是先调用 `init_eepromConfig(true)`，再初始化滤波器和 PID，这是正确阅读生命周期的关键证据。

### 8.3.1 eepromConfig的读写边界

按当前源码，写入 `eepromConfig` 的主要位置包括：

- `config.c`：默认值。
- `main.c`：启动阶段 PID 和使能覆盖，运行阶段使能切换。
- `pid.c`：PID 积分项和 D 项历史状态清零或设置。
- `updatePID()`：通过 `PIDdata_t *PIDparameters` 写入 `iTerm`、`lastDcalcValue`、`lastDterm` 和 `lastLastDterm`；当前三轴调用均传入 `&eepromConfig.PID[...]`。
- `mpu6050.c`：`orientIMU()` 将 `imuOrientation` 写为 10。
- `mpu6050Calibration.c`：温度补偿标定函数会写入斜率和截距，但主线中校准调用被注释。

读取 `eepromConfig` 的主要位置包括：

- `computeMotorCommands.c`：轴使能、PID 参数、`rateLimit`。
- `firstOrderFilter.c`：滤波时间常数。
- `mpu6050.c`：温度补偿参数和 IMU 安装方向。
- `pid.c`：PID 历史状态所在数组。

这些证据说明 `eepromConfig` 是共享状态对象，不是只读配置常量。
如果未来要提高可维护性，应考虑把只读参数、可调参数和运行状态分离。

读写边界还可以用“字段消费等级”记录：

- 已初始化且当前主线读取：例如 `PID[]`、`rateLimit`、`KpAcc`、滤波时间常数。
- 已初始化但当前主线证据较弱：例如部分 autopan、限位和磁力计相关字段。
- 运行中会被改写：例如 PID 历史项、轴使能、`imuOrientation`。
- 名称暗示持久化但未验证：例如 `eepromConfig_t` 整体和被注释的 `writeEEPROM()`。

这种分级比简单列字段更有工程价值。

把这组分级落到字段组上，可以得到更细的读写审查表：

| 字段组 | 已有源码证据 | 教学边界 |
|---|---|---|
| `PID[ROLL/PITCH/YAW]` 的 `P/I/D/windupGuard` 与历史项 | `config.c` 写默认值；`main.c` 在启动阶段覆盖三轴 PID；`computeMotorCommands.c` 把 `&eepromConfig.PID[axis]` 传给 `updatePID()`；`pid.c` 通过 `PIDdata_t *` 回写历史项。 | 可以证明它既是参数入口又是 PID 状态容器；不能把 `config.c` 默认值当成最终运行值，也不能仅搜索 `eepromConfig.PID` 就穷尽指针别名写入。 |
| `rollEnabled/pitchEnabled/yawEnabled` | `config.c` 默认写为 `false`；`main.c` 启动后先关闭三轴，再在姿态收敛和计数条件下改写；`computeMotorCommands.c` 用它们决定三轴控制分支是否执行。 | 可以证明轴使能是运行门控字段；不能证明真实姿态已经稳定，也不能证明电机实际安全启停【待验证】。 |
| `rateLimit` | `config.c` 写入 `45.0f * D2R`；`computeMotorCommands.c` 在 Roll 斜坡限制和 Pitch/Yaw 输出变化限制中读取它。 | 可以证明它是控制输出变化率约束参数；不能证明该限速值对当前机械负载最优，调参效果仍需实测【待验证】。 |
| 姿态与滤波参数 `KpAcc`、低通时间常数 | `config.c` 写 `KpAcc` 和各类 LowPassTau；`MargAHRS.c` 读取 `KpAcc`；`firstOrderFilter.c` 读取 500Hz 加速度与部分 50Hz 指令滤波时间常数。 | 可以证明部分字段已经进入主线计算；像 `KiAcc/KpMag/KiMag`、部分被注释滤波支路或磁力计相关字段，当前消费证据弱，不能写成完整启用。 |
| 温补与安装方向字段 | `config.c` 写温度补偿斜率/截距和 `imuOrientation` 默认值；`mpu6050.c` 读取温补系数计算 bias，`orientIMU()` 又把 `imuOrientation` 写为 10 后用于方向选择。 | 可以证明这些字段参与传感器数据处理或方向选择；不能证明温补参数已由标定流程得到，也不能证明安装方向与实物一致【待验证】。 |
| `dlpfSetting`、autopan、限位、指令输入等字段 | `config.c` 有默认写入；本轮源码检索未看到当前 `main.c` 或控制主线直接消费 `dlpfSetting`，且 `main.c` 调用 `MPU6050_Init(..., DLPF_CFG_260HZ)` 直接传入常量。 | 这些字段应归入“已初始化但当前主线消费证据弱”；不能因为位于 `eepromConfig_t` 中就写成已经参与运行闭环。 |

弱消费字段还要继续拆成“默认存在”“主线读取”“主线实参覆盖”三个层次：

| 字段 | 默认写入证据 | 当前主线证据 | 应降低的结论强度 |
|---|---|---|---|
| `KpAcc` | `config.c` 写 `0.98f` | `MargAHRS.c` 用 `eepromConfig.KpAcc` 修正 `gx/gy/gz` | 可以写成当前 AHRS 主线消费字段。 |
| `KiAcc` | `config.c` 写 `0.02f` | 当前 `MargAHRS.c` 检索未看到 `eepromConfig.KiAcc` 参与积分反馈 | 只能写成默认配置字段，不能写成当前积分校正已启用。 |
| `KpMag/KiMag` | `config.c` 写默认值 | `main.c` 调用 `MargAHRSupdate(..., 0, 0, 0, false, dt500Hz)`，当前 `MargAHRS.c` 未读取 `eepromConfig.KpMag/KiMag` | 只能说明保留了磁力计参数入口，不能写成磁力计闭环已经参与姿态估计。 |
| `dlpfSetting` | `config.c` 写 `DLPF_CFG_184HZ` | `main.c` 初始化 MPU6050 时直接传 `DLPF_CFG_260HZ` | 当前主线 DLPF 由调用实参决定，不能把 `dlpfSetting` 默认值写成实际硬件配置。 |

这张细表的目的不是否定字段设计，而是让教材表述从“结构体里有这个字段”降级到“当前主线是否真的消费这个字段”。对出版级教材来说，默认值、实际调用实参和运行时效果必须分开写；其中运行时效果仍需要调试记录或硬件实测【待验证】。

同样的规则也适用于 autopan、限位和指令输入字段：

| 字段组 | 定义和默认值证据 | 当前主线消费证据 | 教学边界 |
|---|---|---|---|
| `rollAutoPanEnabled/pitchAutoPanEnabled/yawAutoPanEnabled` | `mpu6050.h` 定义字段，`config.c` 默认写 `false` | `computeMotorCommands.c` 中存在 `autoPan()` 函数，但当前检索未看到这些 enable 字段参与调用条件 | 只能证明保留了自动平移配置入口，不能证明 autopan 功能已由配置对象启用。 |
| `rollRateCmdInput/pitchRateCmdInput/yawRateCmdInput` | `mpu6050.h` 定义字段，`config.c` 默认写 `true` | 当前控制主线使用全局 `pointingCmd[]`，未看到这些字段参与选择目标来源 | 只能写成命令输入模式的配置字段，不能写成当前目标命令切换逻辑已经生效。 |
| `gimbal*Limit` 六个限位字段 | `mpu6050.h` 定义 Roll/Pitch/Yaw 左右或上下限位，`config.c` 写角度默认值 | 当前源码检索未看到这些字段在 `computeMotorCommands.c` 中用于夹紧 `pointingCmd[]` 或电机输出 | 只能说明结构体中预留了云台角度限位参数，不能证明当前闭环已经执行软件限位【待验证】。 |

这里的关键区分是：函数存在、字段存在、默认值存在，都还不是“功能启用”。要把某个功能写成当前主线行为，至少还要看到读取字段、参与分支判断或参与数值计算的证据。

字段分级可以按四步判定，而不是凭字段名猜测：

```text
第1级：头文件定义字段
第2级：config.c 或 main.c 写入字段
第3级：主线函数读取字段并参与分支或数值计算
第4级：.list/.map 证明该读取路径进入当前最终构建
```

以 `autoPan()` 为例，`computeMotorCommands.c` 中确实定义了自动平移函数，`.su/.cyclo` 也给出 `autoPan` 的静态栈估计和圈复杂度条目；但 `.map` 把 `.text.autoPan` 放在 `Discarded input sections`，地址为 `0x00000000`、大小为 `0xec`，`.list` 中也未发现跳转到 `autoPan` 的调用指令。与此同时，`rollAutoPanEnabled/pitchAutoPanEnabled/yawAutoPanEnabled` 只在 `config.c` 中被默认写为 `false`，没有进入 `computeMotorCommands()` 的分支条件。因此它们最多到达“定义和默认写入”层级，不能升级为“当前主线功能”。

`rollRateCmdInput/pitchRateCmdInput/yawRateCmdInput` 与 `gimbal*Limit` 也是同类问题。当前 `.list` 能看到 `pointingCmd[ROLL/PITCH/YAW]` 被控制主线读取，也能看到 `eepromConfig.rateLimit` 参与输出变化限制；但未看到 rate input 字段选择目标来源，也未看到限位字段夹紧 `pointingCmd[]` 或电机输出。教材如果要讲这些字段，应写成“结构体保留的配置入口”，而不是“已生效的输入模式切换或软件限位”。

低通滤波时间常数也需要按初始化支路继续拆开：

| 字段组 | 默认写入证据 | 当前初始化证据 | 教学边界 |
|---|---|---|---|
| `accelX/Y/Z500HzLowPassTau` | `config.c` 均写 `0.1f` | `firstOrderFilter.c` 用三者初始化 `ACCEL_X/Y/Z_500HZ_LOWPASS`，Z 轴初始输入和输出为 `-9.8065f` | 可以证明 500Hz 加速度三轴滤波槽当前被初始化；不能证明滤波后的值已经在主线替代未滤波值【待验证】。 |
| `pitchRatePointingCmd50HzLowPassTau` | `config.c` 写 `0.0f` | `firstOrderFilter.c` 当前启用 `PITCH_RATE_POINTING_50HZ_LOWPASS` 初始化 | 可以证明 Pitch rate 指令滤波槽被初始化；是否被后续计算调用还需继续看调用点。 |
| `pitchAttPointingCmd50HzLowPassTau` | `config.c` 写 `0.25f` | `firstOrderFilter.c` 当前启用 `PITCH_ATT_POINTING_50HZ_LOWPASS` 初始化 | 可以证明 Pitch attitude 指令滤波槽被初始化；不能自动推广到 Roll/Yaw。 |
| Roll/Yaw rate 与 Roll/Yaw attitude 时间常数 | `config.c` 有默认写入 | `firstOrderFilter.c` 中对应初始化代码处在注释块内 | 只能写成字段和滤波槽保留存在，不能写成当前初始化全部启用。 |

`firstOrderFilter.h` 定义 `NUMBER_OF_FIRST_ORDER_FILTERS` 为 9，`.map` 中 `firstOrderFilters` 位于 `.bss.firstOrderFilters` 且大小为 `0xb4`。这能证明 9 个滤波槽的 RAM 已分配；但是否每个槽都被初始化、是否每个槽都被运行时调用，仍必须回到 `initFirstOrderFilter()` 和调用点逐项核对。

滤波对象还要再拆成“槽初始化”和“滤波函数运行调用”两个最小单元。`Core/Src/main.c` 在启动阶段调用 `initFirstOrderFilter()`，`.list` 中也能看到 `main()` 通过 `bl 8004f44 <initFirstOrderFilter>` 进入该初始化函数；`.map` 则给 `initFirstOrderFilter` 分配了最终 Flash 地址 `0x08004f44`，大小为 `0x30c`。这只能证明滤波系数和历史值初始化路径进入当前构建。

相反，当前源码检索只发现 `firstOrderFilter()` 的函数定义和头文件声明，没有发现其它模块调用它；`.list` 中也未发现跳转到 `firstOrderFilter` 的调用指令。更重要的是，`.map` 把 `.text.firstOrderFilter` 放在 `Discarded input sections` 区域，地址显示为 `0x00000000`、大小为 `0x6c`，而不是像 `initFirstOrderFilter()` 一样进入最终 `.text` 并获得 Flash 地址。`Debug/Drivers/SRC/Src/firstOrderFilter.su` 和 `.cyclo` 中出现 `firstOrderFilter` 条目，只能说明编译阶段生成过静态栈估计和圈复杂度记录，不能反推出链接后的最终镜像保留了该函数，也不能证明运行时滤波输出已经参与传感器或指令链路。

所以第17章在讲 `LowPassTau` 时必须保持这个边界：`firstOrderFilters[]` 已分配、部分槽已初始化，是当前证据能证明的事实；`firstOrderFilter()` 的运行时调用、滤波后数据替换原始数据、滤波参数对控制效果的影响，当前都仍属于【待验证】。

### 8.3.2 构建产物证据边界

第17章讨论的是数据结构和配置对象，不能只停在头文件定义和源码引用。
当前 Debug 构建还可以用 `Debug/Three-axis_cloud_platformV2.map` 和 `Debug/Three-axis_cloud_platformV2.list` 给出一层更硬的构建产物证据。

`.map` 显示 `eepromConfig` 位于 `.bss.eepromConfig`，地址为 `0x200003e0`，大小为 `0x144` 字节；`sensors` 位于 `.bss.sensors`，地址为 `0x20000528`，大小为 `0x3c` 字节。
同一个 `.map` 还显示 `rawAccel`、`rawGyro`、`rawMPU6050Temperature`、`firstOrderFilters` 分别进入 `.bss`，说明这些全局对象在当前最终镜像中实际分配了 RAM，而不是只存在于头文件声明中。

函数层面，`.map` 显示 `computeMotorCommands`、`init_eepromConfig`、`updatePID`、`zeroPIDintegralError` 和 `zeroPIDstates` 都具有最终 Flash 地址。
这证明配置初始化、PID 状态清理、PID 更新和电机命令计算相关函数没有被链接器丢弃，已经进入当前 Debug ELF。

`.list` 提供更细的调用路径证据。
在 `main()` 对应反汇编中可以看到 `init_eepromConfig(true)` 先装载参数 `1`，再分支调用 `init_eepromConfig`；后续还能看到对 `zeroPIDintegralError()`、`zeroPIDstates()` 和 `computeMotorCommands(dt500Hz)` 的分支调用。
同一段 `.list` 也显示 `main()` 对 `eepromConfig.pitchEnabled`、`rollEnabled`、`yawEnabled` 以及 PID 字段进行启动阶段写入。
在 `computeMotorCommands()` 对应反汇编中，则能看到三轴控制分支调用 `updatePID()`，并且源码行旁边保留了 `&eepromConfig.PID[ROLL_PID]`、`&eepromConfig.PID[PITCH_PID]` 等实参线索。

`.su` / `.cyclo` 还能补充另一层边界：它们不描述全局对象的 RAM 大小，而是描述当前 Debug 编译中相关函数的静态栈估计和圈复杂度。对本章来说，这些条目适合用来区分“数据对象存在”和“操作这些对象的函数路径有多复杂”。

| 函数 | 静态栈估计 | 圈复杂度 | 证据文件 | 和本章数据对象的关系 |
|---|---:|---:|---|---|
| `main()` | 56 字节 | 7 | `Debug/Core/Src/main.su` / `Debug/Core/Src/main.cyclo` | 所在编译单元定义 `eepromConfig`；函数体执行启动覆盖、更新 `sensors`，调用配置、滤波和控制入口。 |
| `init_eepromConfig()` | 16 字节 | 2 | `Debug/Drivers/SRC/Src/config.su` / `Debug/Drivers/SRC/Src/config.cyclo` | 默认写入 `eepromConfig` 字段；当前仍不能证明 EEPROM 恢复。 |
| `initPID()` | 16 字节 | 2 | `Debug/Drivers/SRC/Src/pid.su` / `Debug/Drivers/SRC/Src/pid.cyclo` | 初始化 PID 状态字段，帮助区分参数字段和运行状态字段。 |
| `updatePID()` | 64 字节 | 34 | `Debug/Drivers/SRC/Src/pid.su` / `Debug/Drivers/SRC/Src/pid.cyclo` | 通过 `PIDdata_t *` 回写 `eepromConfig.PID[]` 中的积分和 D 项历史。 |
| `zeroPIDintegralError()` | 16 字节 | 2 | `Debug/Drivers/SRC/Src/pid.su` / `Debug/Drivers/SRC/Src/pid.cyclo` | 清理 `eepromConfig.PID[]` 中的积分状态。 |
| `zeroPIDstates()` | 16 字节 | 2 | `Debug/Drivers/SRC/Src/pid.su` / `Debug/Drivers/SRC/Src/pid.cyclo` | 清理 `eepromConfig.PID[]` 中的 D 项历史状态。 |
| `computeMotorCommands()` | 120 字节 | 48 | `Debug/Drivers/SRC/Src/computeMotorCommands.su` / `Debug/Drivers/SRC/Src/computeMotorCommands.cyclo` | 读取 `sensors`、`eepromConfig.PID[]`、轴使能和 `rateLimit`，并输出电机命令。 |
| `MPU6050_Read_And_Process()` | 64 字节 | 3 | `Debug/Drivers/CustomDrivers/Src/mpu6050.su` / `Debug/Drivers/CustomDrivers/Src/mpu6050.cyclo` | 写入 `rawAccel`、`rawGyro` 等原始和方向处理后的传感器数据。 |
| `orientIMU()` | 4 字节 | 12 | `Debug/Drivers/CustomDrivers/Src/mpu6050.su` / `Debug/Drivers/CustomDrivers/Src/mpu6050.cyclo` | 写入 `eepromConfig.imuOrientation`，体现配置对象也承载运行期状态。 |
| `initFirstOrderFilter()` | 24 字节 | 1 | `Debug/Drivers/SRC/Src/firstOrderFilter.su` / `Debug/Drivers/SRC/Src/firstOrderFilter.cyclo` | 读取 `eepromConfig` 中的滤波时间常数，初始化 `firstOrderFilters`。 |

这张表不能替代 `.map` 的对象地址证据。比如 `eepromConfig` 的 `0x144` 字节来自 `.map` 的 `.bss.eepromConfig`，不是来自任何 `.su` 条目；`init_eepromConfig()` 的 16 字节只表示该函数自身栈帧估计，不表示 `eepromConfig_t` 的结构体大小。它也不能证明最大系统栈深、真实执行耗时、字段写入顺序的运行时完整性或 EEPROM 持久化；这些仍需要调用链分析、运行水位、DWT / GPIO 计时、日志或硬件实测【待验证】。

这里还要单独检查构建产物的新鲜度边界。
当前 `Core/Src/main.c` 中启动覆盖值已经是 `eepromConfig.PID[YAW_PID].P = 0.03f`、`eepromConfig.PID[PITCH_PID].P = 0.01f` 和 `eepromConfig.PID[PITCH_PID].I = 0.0f`，但现有 `Debug/Three-axis_cloud_platformV2.list` 仍保留 `YAW_PID.P = 2.0f`、`PITCH_PID.P = 0.5f` 和 `PITCH_PID.I = 0.05f`。
这组差异说明 `.list` 是某次 Debug 构建留下的快照，能够证明“该构建产物”里的调用和写入，但不能自动证明它已经由当前工作区源码重新生成。
因此本章引用 `.list` 时应写成“该 Debug 构建中可以看到”，不能写成“当前源码重新编译后一定如此”，更不能据此判断板上正在运行的固件参数。

为了避免把不同证据混在一起，本章还需要给构建产物建立一个优先级判定表：

| 证据类型 | 能直接证明什么 | 不能证明什么 | 本章写法 |
|---|---|---|---|
| 当前源码 | 当前工作区里类型定义、全局定义、函数体、字段赋值和调用意图。 | 不能单独证明这些源码已经重新编译并链接进当前 ELF。 | 写成“当前源码中定义/调用/赋值”。 |
| `.map` 最终地址 | `.text` 位于 `0x080...`、`.bss` 位于 `0x200...` 时，可以证明函数或全局对象进入该 Debug ELF 的最终 Flash / RAM 布局。 | 不能证明函数一定被运行时执行，也不能证明字段值正确。 | 写成“该 Debug 构建中已保留/已分配”。 |
| `.map` 的 `Discarded input sections` 或 `0x00000000` 条目 | 可以证明目标文件里曾有对应输入段，或链接器记录了被丢弃的候选段。 | 不能证明函数进入最终可执行代码段，也不能证明主线会调用它。 | 写成“源码/编译阶段有痕迹，但最终保留证据弱”。 |
| `.list` 反汇编和源码注释 | 可以证明该 Debug 构建的机器指令、分支调用、立即数写入和部分实参形态。 | 当 `.list` 注释中的源码值与当前源码不一致时，不能把旧注释值当成当前源码事实。 | 写成“该 Debug 构建中可以看到”。 |
| `.su` / `.cyclo` | 可以证明编译阶段生成了函数静态栈估计和圈复杂度条目。 | 不能替代 `.map` 的最终保留证据，不能证明真实最大栈深、真实耗时或运行时调用。 | 写成“编译静态分析条目显示”。 |
| 构建产物时间戳 | `Three-axis_cloud_platformV2.elf/.map/.list` 的时间戳可以辅助判断它们是不是同一轮 Debug 构建快照。 | 不能证明板上烧录的就是这个镜像，也不能证明工作区源码已经重新构建。 | 写成“该时间点的 Debug 构建快照”。 |

因此同一个结论要按“源码事实”和“构建事实”分开书写。比如当前 `Core/Src/main.c` 已经把 `YAW_PID.P` 改为 `0.03f`，这是当前源码事实；而 `.list` 里仍能看到 `YAW_PID.P = 2.0f`，只能说明旧 Debug 构建快照中曾经写入过这个值。反过来，`.map` 中 `computeMotorCommands` 位于 `0x080043b4`，这是该 Debug ELF 保留函数的强证据；但它仍不能证明电机实际输出正确，真实运行行为必须继续保留为【待验证】。

这些构建产物能证明三类事情：

- 哪些全局对象实际占用 `.bss` RAM。
- 哪些函数进入最终代码段，没有只停留在源码文件里。
- 哪些调用路径在当前构建中形成了从 `main()` 到配置初始化、PID 状态清理、控制计算和 `updatePID()` 的可执行路径。

但它们不能证明四类事情：

- 不能证明 PID 参数运行时一定稳定或控制效果正确；这仍需要调试器、串口记录或硬件实测【待验证】。
- 不能证明 `eepromConfig` 已经完成 EEPROM 持久化；当前 `.map` / `.list` 没有给出读取非易失存储并恢复字段的主线证据。
- 不能证明 `sensors` 中的物理量一定等于真实姿态；`.list` 只能证明写入和读取路径存在，传感器标定、安装方向和滤波效果仍需后续章节与实测验证。
- 不能证明构建产物一定和当前源码完全同步；若 `.list` 中的源码行数值与 `Core/Src/main.c` 不一致，应优先把它视为构建快照新鲜度问题，而不是把旧产物里的数值当成当前源码事实。

所以本章使用 `.map` / `.list` 的结论应写成“该 Debug 构建中对象、函数和调用路径存在”，而不是写成“运行时行为已经正确”或“当前源码已经重新生成同一产物”。
这条边界尤其适用于 `eepromConfig_t`：类型名暗示 EEPROM，但构建产物目前只能支持“全局配置对象进入镜像并被主线调用”，不能支持“历史配置已经从 EEPROM 恢复”。

把这条规则落到教材写作上，可以形成一张最小证据闭环表：

| 想写出的结论 | 最少需要的证据链 | 当前第17章可证明的程度 |
|---|---|---|
| `eepromConfig` 是实际 RAM 对象 | `mpu6050.h` 类型和 `extern` 声明 -> `main.c` 全局定义 -> `.map` 中 `.bss.eepromConfig` 地址和大小 | 可以证明对象进入当前 Debug 构建，并在 `.bss` 中占用 `0x144` 字节。 |
| `eepromConfig` 会被默认写入 | `main()` 调用 `init_eepromConfig(true)` -> `config.c` 的 `if (eepromReset)` 分支逐字段赋值 -> `.list` 中存在对 `init_eepromConfig` 的分支调用 | 可以证明默认写入路径进入该 Debug 构建；不能证明 `false` 分支会恢复 EEPROM。 |
| PID 历史项会在运行中改写 | `computeMotorCommands.c` 传入 `&eepromConfig.PID[axis]` -> `pid.c` 通过 `PIDdata_t *PIDparameters` 写 `iTerm` 和 D 项历史 -> `.list` 中存在三轴 `updatePID()` 调用 | 可以证明 PID 数组既是参数入口也是状态容器；不能只靠搜索 `eepromConfig.PID` 判断全部写入点。 |
| `sensors` 是实时观测状态 | `bgc32.h` 定义 `sensors_t` -> `main.c` 定义并写入 500Hz 加速度/角速度 -> `MargAHRS.c` 写姿态角 -> `computeMotorCommands.c` 读取姿态 | 可以证明它是运行数据容器；不能证明传感器数值、安装方向或姿态估计已经硬件验证。 |
| 配置具有 EEPROM 持久化 | 需要看到非易失存储读取、版本/校验检查、失败回退、写入路径和主线调用 | 当前证据不足，只能保留 `【待验证】`。 |

这张表的教学价值在于：每个数据结构结论都要同时回答“类型在哪里、存储在哪里、构建产物是否包含、谁写入、谁读取、还有什么不能证明”。缺少任意一层时，教材应降低结论强度，而不是用变量名、注释或函数名替代证据。

### 8.4 MotorAxis_t和PWM_Motor_SetAngle

`Drivers/CustomDrivers/Inc/drv_pwmMotors.h` 中 `PWM_Motor_SetAngle()` 的声明把 `MotorAxis_t` 作为第一个参数。

`Drivers/CustomDrivers/Src/drv_pwmMotors.c` 中的实现使用 `switch (axis)` 分派到不同硬件通道。

从当前源码看，头文件注释将 `MOTOR_ROLL`、`MOTOR_PITCH`、`MOTOR_YAW` 分别标注为对应 Mot0、Mot1、Mot2。

但 `drv_pwmMotors.c` 内部实际分派中，`MOTOR_PITCH` 写 MOT0，`MOTOR_ROLL` 写 MOT1，`MOTOR_YAW` 写 MOT2。

这里存在“头文件注释”和实现映射不完全一致的风险。后续讲第29章、第30章时必须继续核对硬件映射证据。本章先记录为项目证据差异，不在没有仓库外硬件资料的情况下自行修正源码。

`Drivers/SRC/Src/computeMotorCommands.c` 中可见控制层调用：

- Roll 相关分支中出现 `PWM_Motor_SetAngle(MOTOR_PITCH, ...)`。
- Pitch 相关分支中出现 `PWM_Motor_SetAngle(MOTOR_ROLL, ...)`。
- Yaw 相关分支中出现 `PWM_Motor_SetAngle(MOTOR_YAW, ...)`。

这组关系不能只靠枚举名字理解，至少要拆成四层：

| 层级 | 当前证据 | 本章能写出的结论 | 仍不能证明 |
|---|---|---|---|
| 枚举数值层 | `drv_pwmMotors.h` 定义 `MOTOR_ROLL = 0`、`MOTOR_PITCH = 1`、`MOTOR_YAW = 2`。 | `MotorAxis_t` 是传给底层驱动的接口标识。 | 不能证明 0/1/2 天然等于物理 Roll/Pitch/Yaw 电机。 |
| 头文件注释层 | 注释写 `MOTOR_ROLL` 对应 Mot0、`MOTOR_PITCH` 对应 Mot1、`MOTOR_YAW` 对应 Mot2。 | 注释给出了作者意图或早期命名线索。 | 当注释和实现不一致时，不能单独作为最终通道依据。 |
| 驱动实现层 | `PWM_Motor_SetAngle()` 的 `switch (axis)` 中，`MOTOR_PITCH` 写 `TIM3->CCR2/CCR3/CCR4`，`MOTOR_ROLL` 写 `TIM3->CCR1`、`TIM2->CCR4/CCR3`，`MOTOR_YAW` 写 `TIM4->CCR4`、`TIM2->CCR2`、`TIM4->CCR3`。 | 可以证明该函数内部的枚举到 CCR 写入路径。 | 不能证明机械安装方向、相序、电机实际响应方向已经正确。 |
| 控制调用层 | `computeMotorCommands.c` 的 Roll 分支传 `MOTOR_PITCH`，Pitch 分支传 `MOTOR_ROLL`，Yaw 分支传 `MOTOR_YAW`；`.list` 中对应调用前分别装载 `r0 = 1`、`r0 = 0`、`r0 = 2`。 | 可以证明该 Debug 构建中控制轴到执行器枚举存在交叉映射。 | 不能解释这种交叉映射的硬件安装原因，也不能证明闭环效果正确。 |

`.map` 给 `PWM_Motor_SetAngle` 分配最终 Flash 地址 `0x08002748`，大小 `0x280`，说明该函数进入该 Debug ELF；`.su/.cyclo` 给出 80 字节静态栈估计和圈复杂度 13，说明函数内部包含限幅、角度归一化、三相 CCR 计算和 `switch` 分派等多分支路径。但这些构建产物仍不能替代示波器相序检查、电机单轴点动、方向闭环验证或中断关闭时间实测【待验证】。

因此调试电机轴映射时，证据优先级应是：先看 `computeMotorCommands.c` 当前控制分支传入哪个 `MotorAxis_t`，再看 `drv_pwmMotors.c` 中该枚举实际写哪些 `TIMx->CCRn`，最后用硬件诊断确认对应物理电机和方向。头文件注释只能作为索引线索，不能越过实现和实测证据。

这再次提示：控制轴、执行器枚举和底层 CCR 通道之间存在项目特定映射。完整 CCR 通道关系应留到第30章电机映射章节，本章只要求读者知道枚举是接口边界；硬件安装原因仍缺少外部资料或仓库外实测证据，不能在本章推断。

### 8.5 运行状态结构的旁证

虽然本章正式知识点不包含 `sensors_t` 和 `rollDiag_t`，但它们能帮助读者理解“数据结构不是变量清单”。

`Drivers/BGC/bgc32.h` 中的 `sensors_t` 保存 500Hz 加速度、陀螺仪和姿态数据。
它不是默认配置集合，而是实时观测状态集合。`Core/Src/main.c` 在 500Hz 分支中写入 `sensors.accel500Hz[]` 和 `sensors.gyro500Hz[]`，`Drivers/SRC/Src/MargAHRS.c` 把姿态角写入 `sensors.margAttitude500Hz[]`，`Drivers/SRC/Src/computeMotorCommands.c` 再读取 `sensors.margAttitude500Hz[]` 参与控制计算。
因此调试 `sensors` 时要追踪同一控制周期内的采样、换算、姿态更新和控制读取顺序；调试 `eepromConfig` 时要追踪 `.bss` 清零、默认配置、启动覆盖和运行改写顺序。

`Drivers/SRC/Inc/computeMotorCommands.h` 中的 `rollDiag_t` 保存 Roll 控制链上的目标、当前角、误差、电角度、PID 原始值、限幅值和调试符号。

这两个对象说明：项目中有些结构体是配置参数，有些是实时状态，有些是诊断快照。
第17章只正式展开 `eepromConfig_t`，但读者应从现在开始建立这种分类意识。

### 8.5.1 `sensors_t` 的运行状态边界

`Drivers/BGC/bgc32.h` 中的 `sensors_t` 当前包含五组数组：`accel500Hz[3]`、`evvgcCFAttitude500Hz[3]`、`margAttitude500Hz[3]`、`gyro500Hz[3]` 和 `mag10Hz[3]`。它不是启动默认值容器，而是控制循环里的实时状态快照。`Core/Src/main.c` 在 500Hz 分支里写入 `sensors.accel500Hz[]` 与 `sensors.gyro500Hz[]`，随后调用 `MargAHRSupdate(...)`；`Debug/Three-axis_cloud_platformV2.list` 显示 `MargAHRSupdate` 最终把姿态角写回 `sensors.margAttitude500Hz[]`，而 `computeMotorCommands()` 再从同一组字段读出姿态参与控制。

`Debug/Three-axis_cloud_platformV2.map` 则给出 `sensors` 的 `.bss` 放置位置 `0x20000528`、大小 `0x3c`，说明它在当前 Debug 构建中确实以 RAM 状态对象存在。它和 `eepromConfig` 的区别很关键：后者更像跨函数共享的配置容器，前者更像单帧或单周期的状态载体。把两者混在一起，读者就会误把“运行中更新的姿态值”当成“启动时写入的默认配置”。

因此，`sensors` 的教学重点不是“它能保存什么默认参数”，而是“它如何在一次控制周期内完成原始采样 -> 物理量换算 -> 姿态更新 -> 控制读取”。它能证明状态流向，但不能证明姿态值一定真实正确、传感器安装方向一定无误、滤波与解算一定足够稳定；这些仍要靠调试日志、姿态对比和硬件实测【待验证】。

把五组字段继续拆开看，可以得到更精确的消费边界：

| 字段 | 当前证据 | 教学边界 |
|---|---|---|
| `accel500Hz[3]` | `Core/Src/main.c` 在 500Hz 分支由 `rawAccel`、温补偏置和比例因子写入；`MargAHRSupdate(...)` 读取它作为加速度输入。 | 可以证明主线把原始加速度转换为姿态解算输入；不能证明比例因子、安装方向或偏置已经硬件校准正确【待验证】。 |
| `gyro500Hz[3]` | `Core/Src/main.c` 在 500Hz 分支由 `rawGyro`、运行偏置、温补偏置和比例因子写入；随后传给 `MargAHRSupdate(...)`。 | 可以证明主线把陀螺仪转换值送入 AHRS；不能证明零偏长期稳定或温补参数有效【待验证】。 |
| `margAttitude500Hz[3]` | `Drivers/SRC/Src/MargAHRS.c` 写 Roll/Pitch/Yaw，并对 NaN 做回退；`computeMotorCommands.c` 读取它参与三轴控制。 | 可以证明它是当前控制主线消费的姿态状态；不能证明 AHRS 输出和真实姿态一致【待验证】。 |
| `evvgcCFAttitude500Hz[3]` | `Drivers/SRC/Src/evvgcCF.c` 有写入代码，但当前源码检索未看到 `main.c` 主线调用该互补滤波入口；`.map` 中 `evvgcCF.o` 普通代码段停留在 `0x00000000` 附近的输入段记录。 | 只能说明结构体为互补滤波保留了状态槽，当前主线消费证据弱，不能写成正在参与闭环控制。 |
| `mag10Hz[3]` | `bgc32.h` 定义字段；当前主线检索未看到对 `sensors.mag10Hz[]` 的有效读写。 | 只能归入保留或未启用状态字段，不能据此推断磁力计参与姿态解算。 |

`sensors_t` 还要额外关注“状态新鲜度”。`Core/Src/main.c` 的 500Hz 分支顺序是：先执行 `MPU6050_Read_And_Process()`，再写 `sensors.accel500Hz[]` 和 `sensors.gyro500Hz[]`，随后调用 `MargAHRSupdate(...)` 写 `sensors.margAttitude500Hz[]`，最后调用 `computeMotorCommands(dt500Hz)` 读取姿态并计算电机输出。这个顺序说明，当前主线控制读到的 `margAttitude500Hz[]` 是同一轮 500Hz 分支中 AHRS 刚更新过的状态；但 `evvgcCFAttitude500Hz[]` 和 `mag10Hz[]` 没有同等级的新鲜度证据。

因此，调试 `sensors` 不能只看数组是否非零，而要先问三个问题：本轮循环是否写入过该字段，后续函数是否读取该字段，`.list/.map` 是否证明这条写入-读取路径进入当前构建。`evvgcCF.c` 中的 `initOrientation()` 和 `getOrientation()` 虽然有源码和 `.su/.cyclo` 条目，但 `.map` 中 `.text.initOrientation`、`.text.getOrientation` 和 `.bss.accAngleSmooth` 都位于 `0x00000000` 的输入/丢弃区域，`.list` 也未发现主线调用；所以它们不能证明 `evvgcCFAttitude500Hz[]` 当前参与闭环。

## 9. 调试方法

调试第17章相关对象时，优先做静态核对，再做运行观察。

静态核对：

- 在 `Drivers/CustomDrivers/Inc/mpu6050.h` 中确认 `Int16_Converter_t`、`eepromConfig_t` 和 `extern eepromConfig` 是否存在。
- 在 `Drivers/CustomDrivers/Src/mpu6050.c` 中确认 `rawAccel`、`rawGyro` 和 `rawMPU6050Temperature` 是否有唯一实际定义。
- 在 `Core/Src/main.c` 中确认 `eepromConfig_t eepromConfig` 是否有唯一实际定义。
- 在 `Core/Src/main.c` 中确认 `sensors_t sensors` 是否有唯一实际定义，并把它归类为实时数据容器，而不是配置恢复入口。
- 在 `STM32F103RCTx_FLASH.ld` 中确认 `.bss` 位于 RAM，且 `_sbss/_ebss` 提供给启动文件。
- 在 `Core/Startup/startup_stm32f103rctx.s` 中确认 `.bss` 在 `main()` 前被清零。
- 在 `Drivers/SRC/Src/config.c` 中确认 `init_eepromConfig(true)` 写入的默认值是否符合当前调试目的。
- 在 `Drivers/SRC/Src/config.c` 中确认 `eepromReset == false` 是否有恢复分支；当前没有该分支时，不要把函数名解释为 EEPROM 恢复。
- 在 `Core/Src/main.c` 中确认默认值之后是否存在二次覆盖。
- 在 `Drivers/SRC/Src/pid.c` 中确认 PID 历史状态是否被初始化或清零。
- 在 `Drivers/SRC/Src/pid.c` 中确认 `updatePID()` 是否通过 `PIDparameters` 指针回写历史字段，并回到调用点确认指针是否来自 `eepromConfig.PID[]`。
- 在 `Drivers/SRC/Src/firstOrderFilter.c` 中确认滤波时间常数是否被读取。
- 在 `Drivers/CustomDrivers/Inc/drv_pwmMotors.h` 和 `Drivers/CustomDrivers/Src/drv_pwmMotors.c` 中核对枚举与 `switch` 分支。
- 在 `Debug/Three-axis_cloud_platformV2.map` 中确认 `eepromConfig`、`sensors`、`rawAccel`、`rawGyro`、`rawMPU6050Temperature` 和 `firstOrderFilters` 是否进入 `.bss`，并记录地址与大小。
- 在 `Debug/Three-axis_cloud_platformV2.map` 中确认 `init_eepromConfig`、`computeMotorCommands`、`updatePID`、`zeroPIDintegralError` 和 `zeroPIDstates` 是否具有最终代码地址。
- 在 `Debug/Three-axis_cloud_platformV2.list` 中确认 `main()` 是否实际调用 `init_eepromConfig(true)`、PID 状态清理函数和 `computeMotorCommands()`，再核对 `computeMotorCommands()` 是否把 `&eepromConfig.PID[axis]` 传给 `updatePID()`。
- 记录 `.map` / `.list` 的证据边界：它们能证明对象和调用路径进入当前构建，不能证明控制效果、传感器真实姿态或 EEPROM 持久化已经正确。

运行观察：

- 在调试器中观察 `eepromConfig.rollEnabled`、`eepromConfig.pitchEnabled`、`eepromConfig.yawEnabled`。
  确认初始化后这些字段是否被 `Core/Src/main.c` 改写。
- 观察 `eepromConfig.PID[ROLL_PID]`、`eepromConfig.PID[PITCH_PID]`、`eepromConfig.PID[YAW_PID]`，确认实际运行参数是否等于你以为的默认值。
- 观察 `eepromConfig.rateLimit`，确认它在 Roll、Pitch、Yaw 输出限速中的使用位置。
- 观察 `rawAccel[axis].bytes[1]`、`rawAccel[axis].bytes[0]` 和 `rawAccel[axis].value` 的关系。
- 对 `PWM_Motor_SetAngle()` 设置断点，检查控制层传入的 `axis` 与期望轴是否一致。
- 对 `rawAccel`、`rawGyro` 设置观察表达式，等待第19章引入连续读取后再验证字节组合是否符合 MPU6050 数据顺序。

调试记录建议：

- 记录每个结构体或全局对象的声明文件、实际定义文件、初始化函数和主要读写函数。
- 记录全局对象是否位于 `.bss`、是否在 `main()` 前被清零、是否在项目初始化函数中被重写。
- 对 `eepromConfig` 字段，应分开记录 `config.c` 默认值、`main.c` 运行时改写值和调试器实测值。
- 对 `PID[3]`，应区分可调参数字段和运行历史字段。
- 对 `PID[3]` 的运行历史，应同时记录显式写入和 `PIDdata_t *` 指针别名写入；只搜索 `eepromConfig.PID` 不足以覆盖全部运行时改写。
- 对联合体解析，应记录寄存器高低字节、写入的 `bytes[]` 下标、目标平台字节序和最终 `value`。
- 对电机轴枚举，应记录枚举值、调用处 `axis`、`switch` 分支和后续 CCR 通道映射入口。
- 未发现完整 EEPROM 读写主线时，只能写“配置对象存在”，持久化保存能力保持【待验证】。

性能与风险记录建议：

- `eepromConfig` 是全局共享对象，访问成本低，但模块耦合高。
- `eepromConfig_t` 至少包含 73 个 `float` 和 12 个 `uint8_t` 字段，RAM 下限约 304 字节，最终大小需用 `sizeof` 在目标编译器验证。
- `rawAccel` 和 `rawGyro` 是全局原始数据入口，读写顺序必须和 500Hz 控制循环保持一致。
- `rawAccel[3]`、`rawGyro[3]` 和 `rawMPU6050Temperature` 合计承接 14 字节 MPU6050 连续读数据，但函数结束后会被方向矩阵结果覆盖。
- `PWM_Motor_SetAngle()` 内部会短暂关闭中断更新三相 CCR，本章只记录接口边界，实时影响留到电机输出章节继续分析。
- 若未来把配置写入 Flash 或 EEPROM，需要增加一致性、写入寿命和掉电保护设计；当前保持【待验证】。

#### 9.2 最小运行核对表

如果要把第17章的对象关系落到调试器里，最少只需核对四条链路：

1. `eepromConfig`：先确认 `.bss` 清零，再确认 `init_eepromConfig(true)` 默认写入，最后确认 `main.c` 是否覆盖 PID、使能和其他关键字段。
2. `PID[3]`：分开看 `config.c` 默认值、`main.c` 启动覆盖值、`computeMotorCommands.c` 的参数传入，以及 `pid.c` 通过 `PIDdata_t *` 回写的历史项。
3. `sensors`：先确认 `main.c` 写入 `accel500Hz/gyro500Hz`，再确认 `MargAHRS.c` 写入 `margAttitude500Hz`，最后确认 `computeMotorCommands.c` 读取的是同一对象。
4. `rawAccel/rawGyro/rawMPU6050Temperature`：确认它们先承接 I2C 原始字节，再被 `Int16_Converter_t` 解释，最后是否被后续方向矩阵结果或温度补偿覆盖。

这张表的作用不是替代源码阅读，而是让调试者快速判断当前看到的是默认值、启动覆盖值、运行改写值还是旧的 Debug 快照。如果只看其中一条链路，就很容易把 `eepromConfig` 误判成 EEPROM 恢复结果，把 `sensors` 误判成配置对象，或者漏掉 `PIDdata_t *` 的别名写回。

## 10. 常见问题

问题一：为什么不直接用移位运算组合两个字节。

移位运算当然可以。
本项目使用联合体是为了让原始字节和 16 位值共享一个变量入口。
它简洁，但也要求读者注意字节顺序。
仓库内证据显示第19章会按 `bytes[1]` 写高字节、`bytes[0]` 写低字节。
因此本章只能说明结构设计，不能脱离第19章的赋值顺序单独判断解析是否正确。

问题二：`eepromConfig_t` 是否表示项目已经完成 EEPROM 保存。

不能这样下结论。当前证据能证明的是：项目定义了名为 `eepromConfig_t` 的全局配置结构，
并通过 `init_eepromConfig(true)` 写默认值。
本章未发现完整 EEPROM 读写主线证据，因此非易失保存能力标记为【待验证】。

问题三：为什么 `eepromConfig_t` 定义在 `mpu6050.h` 里。

从模块边界看，这不是最理想的位置，因为配置对象已经超出 MPU6050 单一驱动范围。
但当前源码确实这样组织，并且许多模块通过包含 `mpu6050.h` 取得 `eepromConfig`。
教材应先尊重现状，再在后续重构讨论中评价边界。

问题四：`MOTOR_ROLL` 是否一定驱动 Roll 物理电机。

不能只看名字判断。
当前 `drv_pwmMotors.c` 和 `computeMotorCommands.c` 显示控制轴到执行器枚举之间存在项目特定映射，
甚至存在 Roll 分支调用 `MOTOR_PITCH` 的情况。
没有更多硬件安装证据前，应以源码调用路径和第30章的 CCR 写入路径为准，
不能把枚举名称直接解释成物理电机安装原因。

问题五：`init_eepromConfig(true)` 写入的默认值是否都会进入运行主线。

不能这样外推。
默认值写入配置结构，只说明这些参数具备集中管理入口。
是否进入运行主线，还要看后续函数是否读取对应字段。
例如第18章会看到 `dlpfSetting` 默认值存在，但当前初始化调用直接传入 `DLPF_CFG_260HZ`。
因此本章讲“配置对象”，不是承诺每个字段都已经被当前主线消费。

问题六：为什么只看 `config.c` 会误判 PID 参数。

因为 `main.c` 在调用 `init_eepromConfig(true)` 之后又覆盖了三个轴的 PID 参数和部分运行状态。

例如 `config.c` 给 Roll/Yaw 的 `P` 默认值是 10.0f，但 `main.c` 后续把 Roll 的 `P` 写为 0.03f，把 Yaw 的 `P` 写为 0.03f。
所以调试时必须沿着生命周期看最终值，而不是停在默认值。

问题七：`rawAccel` 和 `rawGyro` 是否一定是纯原始寄存器值。

不完全是。
`MPU6050_Read_And_Process()` 先把 I2C 字节写入联合体，再通过 `orientationMatrix` 做安装方向旋转，最后把旋转后的结果写回 `rawAccel` 和 `rawGyro`。

因此它们在函数结束后更接近“经过项目坐标处理的传感器入口值”。
寄存器级原始字节顺序应在第19章继续分析。

问题八：为什么全局配置对象既方便又危险。

方便在于多个模块能通过同一个对象共享参数，调试器也能集中观察。
危险在于写入点分散，任何包含头文件的模块都可能修改字段。

所以出版级教材不能只列字段，还要列字段生命周期、读写者和证据边界。

问题九：为什么 `eepromConfig` 在 `main()` 之前不是随机值。

因为它是未显式初始化的全局对象，链接脚本把这类对象放入 `.bss`，
启动文件在调用 `main()` 前把 `.bss` 清零。

但清零值也不等于项目默认配置。
项目默认配置要等 `init_eepromConfig(true)` 执行后才成立。
所以调试时应区分“启动清零值”“默认配置值”“启动覆盖值”和“运行改写值”。

问题十：为什么 `PID[3]` 不能全部当成配置参数。

因为 `PIDdata_t` 同时包含 `P/I/D` 这类调参字段，也包含 `iTerm`、`lastDcalcValue`、`lastDterm`、`lastLastDterm` 这类运行历史字段。

`pid.c` 会在初始化、清零和更新过程中改写这些历史字段。
因此 `eepromConfig.PID[axis]` 既是参数入口，也是 PID 状态容器。

问题十一：为什么 `init_eepromConfig(true)` 不能写成恢复 EEPROM 配置。

因为当前函数体只有 `if (eepromReset)` 默认写入分支，当前唯一调用点传入 `true`。
仓库没有看到 `eepromReset == false` 的恢复逻辑，也没有看到实际 EEPROM 读取函数参与主线。

所以 `init_eepromConfig(true)` 在当前项目中应解释为写入默认配置。
函数名和注释可以提示设计意图，但教材结论必须以源码执行路径为准。

问题十二：为什么搜索 `eepromConfig.PID` 还不够。

因为 `computeMotorCommands.c` 把 `&eepromConfig.PID[ROLL_PID]`、`&eepromConfig.PID[PITCH_PID]` 和 `&eepromConfig.PID[YAW_PID]` 传给 `updatePID()`。
进入 `pid.c` 后，代码通过 `PIDparameters->iTerm` 和 D 项历史字段回写同一块内存。

这属于指针别名写入。
审查运行状态时，既要搜显式的 `eepromConfig.PID[...]`，也要沿着指针实参追踪被调函数内部的 `PIDparameters->...` 写入。

## 13. 本章总结

本章完成了驱动篇的第一个基础台阶：不是讲某个外设如何工作，而是建立项目数据对象的阅读地图。

`Int16_Converter_t` 是传感器原始字节进入项目的入口。`eepromConfig_t` 是系统参数、默认值和运行状态的集中入口。`MotorAxis_t` 是控制算法调用底层电机驱动的接口入口。

本章已经证明：

- `Int16_Converter_t` 定义在 `Drivers/CustomDrivers/Inc/mpu6050.h`，实例在 `Drivers/CustomDrivers/Src/mpu6050.c`。
- `rawAccel` 和 `rawGyro` 在读取函数中会经历字节组合、矩阵旋转和写回，不只是静态变量名。
- `eepromConfig_t` 定义在 `Drivers/CustomDrivers/Inc/mpu6050.h`，实例在 `Core/Src/main.c`，默认初始化在 `Drivers/SRC/Src/config.c`。
- `eepromConfig` 在进入 `main()` 前先由启动文件清零 `.bss`，随后才被默认值和启动覆盖值改写。
- `eepromConfig` 的最终运行值还会受到 `main.c` 覆盖、`pid.c` 状态清零和运行时使能逻辑影响。
- `sensors_t sensors` 是实时观测状态容器，当前证据链是 `main.c` 写 500Hz 物理量、`MargAHRS.c` 写姿态角、`computeMotorCommands.c` 读取姿态参与控制；它不能被当成配置对象或 EEPROM 恢复入口。
- `updatePID()` 通过 `PIDdata_t *PIDparameters` 指针回写 PID 历史字段，当前三轴调用点传入的是 `&eepromConfig.PID[...]`。
- `.map` 证明 `eepromConfig`、`sensors`、`rawAccel`、`rawGyro`、`rawMPU6050Temperature` 和 `firstOrderFilters` 已进入当前 Debug 构建的 `.bss`，并证明 `init_eepromConfig`、`computeMotorCommands`、`updatePID`、`zeroPIDintegralError` 和 `zeroPIDstates` 具有最终代码地址。
- `.list` 证明 `main()` 对配置初始化、启动覆盖、PID 状态清理和 `computeMotorCommands()` 存在实际分支调用，`computeMotorCommands()` 到 `updatePID(..., &eepromConfig.PID[axis])` 的路径也进入当前构建。
- `.su/.cyclo` 证明配置、PID 状态、传感器读取、滤波初始化和控制命令相关函数具有静态栈估计与圈复杂度条目，但不能把函数栈估计当成结构体大小或系统最大栈深。
- `init_eepromConfig(true)` 当前证明的是默认配置写入，不证明 EEPROM 历史配置恢复。
- `init_eepromConfig(false)` 当前没有恢复逻辑，等价于保持 `eepromConfig` 当前值不变；调用时机不同，保留的可能是 `.bss` 零值，也可能是运行中状态。
- `eepromConfig_t` 至少包含 73 个 `float` 和 12 个 `uint8_t` 字段，字段有效载荷下限约 304 字节，最终大小需目标编译器验证。
- `PID[3]` 同时包含 PID 调参字段和运行历史字段，因此不能被整体当成只读配置。
- `MotorAxis_t` 定义在 `Drivers/CustomDrivers/Inc/drv_pwmMotors.h`，用于 `PWM_Motor_SetAngle()` 的轴选择。

本章边界：

- 不把 `eepromConfig_t` 扩展为已验证 EEPROM 持久化系统，该能力仍为【待验证】。
- 不把 `main.c` 中“恢复之前的系统参数”注释直接当成执行事实；当前执行路径没有 EEPROM 恢复证据。
- 不把 `.map` / `.list` / `.su` / `.cyclo` 的构建证据扩展为运行正确性证据；控制效果、传感器真实姿态、系统最大栈深、真实耗时和 EEPROM 持久化仍需调试器、日志或硬件实测【待验证】。
- 不提前解释最终电机物理映射，只记录当前源码中的枚举、调用和实现证据。
- 不把联合体解析推广为所有平台通用写法；其正确性依赖当前字节赋值顺序和目标平台字节序。
