## 3. 问题背景

三轴云台项目的调试常见问题可以分成两类。

第一类是下载和连接问题。例如工程能编译，但调试器连不上目标板；或者能下载，但不知道当前程序是否真的停在 `main()`。

第二类是电机输出问题。例如算法在运行，但电机不动、某一相没有输出、某个轴方向异常。此时如果直接观察完整控制链路，会同时涉及 IMU、AHRS、PID、电角换算和 PWM 映射，排查范围太大。

本章的思路是分层排查：

```text
先确认下载调试链路
-> 再确认 SWD 引脚配置
-> 再确认 PWM 通道和 GPIO 复用
-> 最后用硬件诊断函数绕过算法检查底层输出
```

这样可以避免一开始就把所有问题都归因于 PID 参数。

## 4. 核心概念

### 4.1 ST-LINK与SWD调试配置

`ST-LINK与SWD调试配置` 描述的是 STM32CubeIDE 如何启动调试会话、连接目标芯片、下载 ELF 文件、加载符号并停在指定入口。

当前工程的 `Three-axis_cloud_platformV2 Debug.launch` 中可以看到：

- 使用 ST-LINK 调试设备。
- `swd_mode=true`。
- 调试器启动模式为 remote。
- GDB 命令为 `target remote`。
- 程序文件为 `Debug/Three-axis_cloud_platformV2.elf`。
- 下载前执行构建、下载程序并加载符号。
- `verify_flash_download=true`。
- `stopAt=main`。
- `reset_strategy=connect_under_reset`。

这些配置共同说明：当前工程的调试入口不是串口下载，而是 ST-LINK 通过 SWD 连接目标芯片。

### 4.2 SWD引脚保留

SWD 至少需要 SWDIO 和 SWCLK 两根调试信号。当前 `.ioc` 中：

```text
PA13 -> SYS_JTMS-SWDIO
PA14 -> SYS_JTCK-SWCLK
```

同时，`Core/Src/stm32f1xx_hal_msp.c` 中 `HAL_MspInit()` 调用：

```text
__HAL_AFIO_REMAP_SWJ_NOJTAG()
```

这表示关闭 JTAG，保留 SWD。第08章已经解释过 AFIO 与调试口重映射，本章把它接到实际下载调试配置。

### 4.3 电机硬件诊断

`电机硬件诊断` 的目标是绕过上层控制算法，直接验证底层 PWM 通道、GPIO 复用和驱动输出是否有响应。

当前项目提供：

```text
PWM_Motor_Hardware_Diagnosis()
```

该函数每 1000ms 切换一次 `step`，先关闭所有三轴相关 CCR，然后按 step 给某一相写固定 `duty=1000`。

关键边界是：当前函数中 Pitch 和 Yaw 多处写入被注释，实际未完整执行三轴自动点测。当前未注释的写入主要覆盖 Roll 所在的三相通道。

还要注意，`PWM_Motor_Hardware_Diagnosis()` 不是“一次调用完成三相扫描”的函数。
它内部使用静态 `step` 和 `last_tick`，只有满足：

```text
HAL_GetTick() - last_tick >= 1000
```

时才会更新一次 CCR。若只在调试器里手动调用一次，可能只切换一个相位；
如果当前 tick 间隔不足 1000ms，甚至可能不改任何 CCR。
因此实际点测需要周期性反复调用，并记录每次调用时的 tick、step 和 CCR。

### 4.4 诊断函数调用边界

当前搜索只发现 `PWM_Motor_Hardware_Diagnosis()` 的声明和定义，没有发现 `Core/Src/main.c` 或当前主控制路径调用它。

因此，本章不能写成“项目运行时自动执行电机硬件诊断”。更准确的说法是：项目保留了一个底层点测函数，但当前主程序没有接入该函数；若要执行，需要调试器手动调用，或在专门调试构建中临时接入。

### 4.5 证据层级

下载调试和硬件诊断要区分五类证据：

- 仓库配置证据：`.launch`、`.ioc`、`tim.c` 和驱动源码能证明工程“打算如何配置”。
- 构建产物证据：`Debug/Three-axis_cloud_platformV2.elf` 能证明某次构建产生了可下载映像。
- 下载校验证据：`verify_flash_download=true` 只能说明调试会话配置了下载后校验。
- 运行时软件证据：断点、变量、寄存器窗口能证明当前固件“是否走到某段代码”。
- 仓库外硬件证据：示波器、逻辑分析仪、电源电流、驱动板状态和电机响应仍然是【待验证】。

如果只看到 `.ioc` 配置了 PWM，不能推出引脚上一定有正确波形；如果只看到 `CCR` 被写入，也不能推出电机相序、方向、力矩和温升安全。

同样，`.launch` 指向某个 ELF 文件，不等于板上此刻一定运行最新源码。
从源码到 CPU 运行之间至少隔着构建、产物选择、下载、Flash 校验、复位入口和断点命中几层证据。
如果其中任一环节使用了旧产物，调试器仍可能正常连接，但观察到的是旧固件行为。

同样，只看到某个宏名也不能推出当前引脚用途。`Core/Inc/gpio.h` 中定义了
`LED2_GPIO=GPIOA`、`LED2_PIN=GPIO_PIN_3`，但当前搜索没有看到 `LED2`
被主线代码使用；而 `.ioc` 和 `Core/Src/tim.c` 把 PA3 配为 `TIM2_CH4`。
本章调试 PA3 时，应以实际初始化和复用配置为证据，而不是以未使用的宏名为证据。

### 4.6 轴命名与物理接口边界

本项目同时存在“逻辑轴”“枚举名”“物理接口”和“定时器通道”四层命名。调试时不能只看中文注释或枚举顺序。

`drv_pwmMotors.h` 中枚举注释写的是：

```text
MOTOR_ROLL  -> 对应 Mot0
MOTOR_PITCH -> 对应 Mot1
MOTOR_YAW   -> 对应 Mot2
```

但 `PWM_Motor_SetAngle()` 的 `switch` 注释写的是：

```text
MOTOR_PITCH -> P轴 -> 物理接口 MOT0 -> TIM3 CCR2/3/4
MOTOR_ROLL  -> R轴 -> 物理接口 MOT1 -> TIM3 CCR1, TIM2 CCR4/3
MOTOR_YAW   -> Y轴 -> 物理接口 MOT2 -> TIM4 CCR4, TIM2 CCR2, TIM4 CCR3
```

二者注释存在不一致。对调试结论而言，真正能作为硬证据的是函数实际写入的 `TIMx->CCRn` 和 GPIO 复用映射。机械轴命名是否与实物一致，需要结合接线、安装方向和实测响应继续验证。

### 4.7 初始CCR与安全关闭边界

电机硬件诊断还要区分“轴控制分支未运行”和“PWM 输出已经被清零”。

`Core/Src/tim.c` 中 TIM2、TIM3、TIM4 的 PWM 通道初始化都使用：

```text
Period = 3599
Pulse  = 500
OCMode = TIM_OCMODE_PWM1
```

这意味着初始化配置层面存在一个默认比较值。`PWM_Motor_Init()` 随后启动 9 个 PWM 通道，
但没有在函数末尾把这些通道的 CCR 统一清零。
因此，三轴 `eepromConfig.*Enabled == false` 只能说明 `computeMotorCommands()` 的轴分支不写新输出，
不能单独证明所有 PWM 通道当前 CCR 为 0，也不能证明引脚或驱动已经处于安全静默状态【待验证】。

同理，`PWM_Motor_Hardware_Diagnosis()` 的“关闭全部 CCR”动作也不是每次函数入口都会执行。
它位于：

```text
if (current_tick - last_tick >= 1000)
```

这个到期分支内部。若函数未被调用、未持续调用，或调用时尚未达到 1000ms，
诊断函数会直接返回，不会执行清零。调试记录必须把“调用了函数”和“进入了到期清零分支”
拆开写。

## 5. 工作原理

### 5.1 下载调试链路

当前调试链路可以概括为：

```text
源码 .c/.h
-> 编译和链接
-> 生成 Debug/Three-axis_cloud_platformV2.elf
-> STM32CubeIDE Debug.launch 选择该 ELF
-> 启动 ST-LINK GDB Server
-> arm-none-eabi-gdb 远程连接
-> 下载 ELF 对应的映像到 Flash
-> 校验 Flash 下载
-> 加载符号
-> 停在 main
-> 断点或变量证明当前运行路径
```

这个链路解决的是“固件是否真的进入目标芯片、调试器是否能控制 CPU、变量和断点是否能被观察”的问题。

如果下载链路不稳定，后面的电机诊断结论都不可靠。因为你无法确定目标板上运行的是不是当前工程生成的固件。
如果构建产物已经陈旧，即使下载和校验成功，也只能证明旧 ELF 被写入并校验，
不能证明刚修改的源码已经进入板上 Flash。

这里需要区分四个对象：

```text
当前源码
-> Debug ELF 构建产物
-> 板上 Flash 内容
-> CPU 当前正在执行的指令
```

`.launch`、构建日志、ELF 时间戳、下载日志、停在 `main()`、断点命中和变量值分别覆盖不同对象。
只有这些证据能串起来，才能把结论写成“当前源码对应的固件正在目标板运行”。

### 5.2 SWD与AFIO的关系

STM32F1 系列中，调试相关引脚和普通 GPIO/复用功能之间存在复用关系。当前项目保留 PA13/PA14 作为 Serial Wire，并通过 `NOJTAG` 关闭 JTAG。

这带来两个结果：

- 调试器仍然可以通过 SWD 连接目标芯片。
- JTAG 占用的部分调试资源被释放，避免和工程中其他外设复用安排冲突。

本章只确认 SWD 下载调试链路。串口输出属于第11章和第32章，USB 支线属于第15章和第16章。

### 5.3 PWM硬件映射复查

电机硬件诊断依赖第30章已经确认的三轴执行器映射。本章不重复展开完整 `MOTOR_PITCH/MOTOR_ROLL/MOTOR_YAW -> TIM CCR` 表，只沿当前诊断函数实际未注释的 Roll 点测路径继续分析。

结合 `PWM_Motor_Hardware_Diagnosis()` 的实际写入和 `Core/Src/tim.c` 的 GPIO 复用，当前重点观察的是：

```text
step 0 -> TIM3_CH1 / PA6
step 1 -> TIM2_CH4 / PA3
step 2 -> TIM2_CH3 / PA2
```

### 5.4 PWM_Motor_Hardware_Diagnosis的流程

诊断函数流程为：

```text
检查 PWM 是否初始化
-> 每 1000ms 进入一次诊断切换
-> duty = 1000, off = 0
-> 关闭全部三轴相关 CCR
-> 根据 step 只打开一个相位
-> step 在 0、1、2 之间循环
```

当前未注释的实际写入为：

```text
step 0 -> TIM3->CCR1 = duty
step 1 -> TIM2->CCR4 = duty
step 2 -> TIM2->CCR3 = duty
```

这对应 Roll 物理输出路径。Pitch 的 `TIM3->CCR2/3/4` 和 Yaw 的 `TIM4->CCR4`、`TIM2->CCR2`、`TIM4->CCR3` 在该函数中有注释痕迹，但当前没有实际写入。

从调用语义看，这个流程还隐含一个“反复调用”前提。
函数开头读取 `HAL_GetTick()`，只有当当前 tick 与静态 `last_tick` 的差值达到 1000ms，
才会进入 CCR 更新块。进入后先写 `last_tick = current_tick`，再根据当前 `step` 写一相，
最后 `step++`，超过 2 后回到 0。

因此，诊断输出的时间线更准确地写成：

```text
周期性调用函数
-> 未到1000ms：直接返回，不改变CCR
-> 到达1000ms：关闭全部CCR，打开当前step对应相位
-> step递增，等待下一次到期调用
```

这解释了一个常见现象：调试器里“执行过一次诊断函数”，不等于示波器上会看到三相轮流点测。
必须确认函数被持续调用，且每次到期后 CCR 是否真的变化。

这里还要区分“调试器墙钟时间”和“MCU 内部 tick”。当前项目的 `SysTick_Handler()`
会先调用 `HAL_IncTick()`，而 HAL 默认实现中 `HAL_IncTick()` 递增 `uwTick`，
`HAL_GetTick()` 返回的也是 `uwTick`。因此诊断函数里的：

```text
current_tick - last_tick >= 1000
```

比较的是 MCU 运行过程中累计的 HAL tick，不是电脑端调试器窗口里等待了多少秒。
如果 CPU 停在断点或单步暂停，正常的 SysTick 中断处理不会按运行状态持续推进；
此时调试者在 IDE 里等 1 秒，并不等于 `HAL_GetTick()` 增加了 1000。
验证 1000ms 门控时，应记录暂停前后的 `HAL_GetTick()` 或 `uwTick`，
并让目标板真正连续运行，而不是只依赖人工等待。

这里还要加一个初始化边界。TIM2/TIM3/TIM4 的 PWM 初始化 `Pulse` 当前都是 500。
因此，如果诊断函数没有进入 1000ms 到期分支，或者正常控制分支也没有写新的 CCR，
调试器里看到 `CCR=500` 可能只是 CubeMX 初始化配置留下的比较值。
它既不是诊断函数的 `duty=1000`，也不是“输出已经关闭”的证据。

### 5.5 诊断函数的价值和限制

它的价值是把排查范围缩小到最底层：

```text
定时器通道
-> GPIO复用
-> 相线输出
-> 驱动级响应
```

它的限制也很明确：

- 当前没有主程序或主控制路径调用证据。
- 当前不是完整三轴自动诊断。
- 它依赖周期性调用，单次手动调用不等于完整三步点测。
- 它直接写 CCR，不能证明 PID、姿态解算或电角换算正确。
- 它使用固定 `duty=1000`，需要结合 `ARR=3599` 理解占空比大约为 27.8%。
- 它的清零动作只在 1000ms 到期分支内发生，未到期调用不会改变现有 CCR。
- `PWM_Motor_Init()` 当前没有逐项检查 `HAL_TIM_PWM_Start()` 返回值，
  因此 `isInitialized=1` 不是每个 PWM 通道都成功输出的充分证据。

因此，诊断通过只能说明底层某些输出通道可响应；诊断失败也需要继续区分是 PWM 未初始化、函数未执行、通道映射错误、外部连接问题还是驱动级问题。

### 5.6 诊断输出与闭环输出的互斥

`PWM_Motor_Hardware_Diagnosis()` 和 `computeMotorCommands()` 都会写同一批 `TIMx->CCRn`。区别在于：

- 诊断函数按 `HAL_GetTick()` 每 1000ms 切换一次相位。
- 主控制函数在 500Hz 分支中调用，理想周期约 2ms。
- 当前 `main.c` 在 500Hz 分支末尾调用 `computeMotorCommands(dt500Hz)`。

还要注意一层更细的覆盖关系：诊断函数当前实际点测的是 `MOTOR_ROLL`
物理输出组，也就是 `TIM3->CCR1`、`TIM2->CCR4` 和 `TIM2->CCR3`。
但当前控制代码中，Pitch 控制分支调用的是
`PWM_Motor_SetAngle(MOTOR_ROLL, ...)`，因此直接覆盖这组三个诊断
CCR 的不是 Roll 控制分支，而是 Pitch 控制分支。Roll 控制分支当前调用
`PWM_Motor_SetAngle(MOTOR_PITCH, ...)`，对应的是 `TIM3->CCR2/3/4`。

因此，如果把诊断函数临时接入主循环，同时仍然让闭环控制运行，
闭环输出很可能在下一帧覆盖诊断写入。调试时应把诊断看成“独占模式”，至少要满足：

- 明确停用或绕过 `computeMotorCommands()` 对同一电机通道的写入。
- 若点测当前实际执行的 `MOTOR_ROLL` 物理输出组，应重点关闭
  `eepromConfig.pitchEnabled`，因为 Pitch 分支会写同一组 CCR。
- 明确关闭被测物理输出组对应的正常控制使能，或在专门诊断构建中只运行点测函数。
- 记录诊断函数在控制链路中的调用位置，避免同一循环内先写诊断、后被闭环覆盖。

这个互斥关系属于软件层面的证据，不需要上电机就能从调用关系推出。但是否真的无覆盖、引脚是否输出、驱动是否响应，仍需寄存器观察和外部测量验证。

### 5.7 波形验证的三层指标

电机点测不能只看“电机动不动”。更稳妥的验证顺序是：

```text
寄存器指标 -> 引脚波形指标 -> 负载响应指标
```

寄存器指标包括 `CCR1/CCR3/CCR4` 是否按 step 变化，`CCMR1/CCMR2` 中对应 `OCxPE` 是否置位，`CNT` 是否计数，`CR1.CEN` 是否置位，`CCER.CCxE` 是否打开。引脚波形指标包括频率、占空比、更新边界和引脚对应关系。负载响应指标包括驱动级输出、电流、电机温升和机械响应。

其中前两类可以直接支撑“PWM 输出路径是否成立”的判断；负载响应涉及硬件状态，必须标为【待验证】，不能由源码直接推出。

还要注意，`CCR` 变化和引脚占空比生效不是同一层证据。当前 HAL 的 `HAL_TIM_PWM_ConfigChannel()` 会设置输出比较预装载位 `OCxPE`，所以寄存器窗口看到 `TIMx->CCRn` 变为 0 或 1000，只能证明软件写入和比较值入口已经变化；它是否已经影响当前 PWM 周期的外部波形，还要结合 `OCxPE`、更新事件边界、`CNT` 位置和示波器结果判断。

## 6. STM32实现机制

本章涉及十一个 STM32 实现机制。

第一，SWD 调试接口。PA13/PA14 被配置为 Serial Wire，对应 SWDIO 和 SWCLK。调试器通过这些引脚控制内核停机、下载、断点和变量观察。

第二，调试启动配置。`.launch` 文件不是固件源码，但它决定了 IDE 调试会话如何启动。
例如 `stopAt=main` 让调试开始时停在 `main()`，
`verify_flash_download=true` 让下载后执行校验。
但 Flash 校验的对象是已选择的构建产物，不是工作区里所有源码文件。
如果最新源码没有成功重新构建，校验通过也不能证明板上运行的是最新编辑内容。

第三，调试符号。`.cproject` 中 Debug 配置使用 ELF 产物和 `g3` 调试级别，
`.launch` 中的 `loadList` 同时配置了构建、下载和加载符号。
符号信息能帮助变量窗口、断点和源码级单步定位，但它要求符号文件与板上固件匹配。
如果符号来自新 ELF，而 Flash 中仍是旧固件，变量观察和断点位置可能产生误导。

第四，PWM CCR 直接写入。`PWM_Motor_Hardware_Diagnosis()` 不经过 `PWM_Motor_SetAngle()` 的正弦查表，而是直接写 `TIMx->CCRn`。这使它适合做硬件点测，但也意味着它绕过了正常三相正弦输出逻辑。

第五，PWM 通道启动。`HAL_TIM_PWM_Start()` 在本仓库 HAL 源码中会调用
`TIM_CCxChannelCmd(..., TIM_CCx_ENABLE)` 打开捕获比较通道，并通过
`__HAL_TIM_ENABLE()` 使能定时器。也就是说，点测前不仅要看 `CCR`，
还要看通道使能和计数器是否真的运行。

第六，CCR 预装载与更新事件。`Core/Src/tim.c` 中 TIM2、TIM3、TIM4 的
`AutoReloadPreload` 为 Disable，这只说明 ARR 预装载关闭；但 HAL 的
`HAL_TIM_PWM_ConfigChannel()` 会给 PWM 通道设置 `OCxPE`。因此诊断函数直接写
`TIMx->CCRn` 时，调试者应把它理解为“写入比较值入口”，不要直接写成
“同一个 CPU 写周期内外部波形已经改变”。外部波形何时采用新比较值，还要看
更新事件和示波器观测。

第七，GPIO 复用输出。`HAL_TIM_MspPostInit()` 把 PA1/PA2/PA3、PA6/PA7、
PB0/PB1、PB8/PB9 配为 `GPIO_MODE_AF_PP`。这里的 `GPIO_SPEED_FREQ_LOW`
是 GPIO 输出驱动速度配置，不等同于 PWM 频率；PWM 频率仍由定时器时钟、
`PSC` 和 `ARR` 决定。

第八，HAL tick 与调试暂停。`stm32f1xx_it.c` 的 `SysTick_Handler()` 调用
`HAL_IncTick()`，HAL 默认实现把 `uwTick` 按 `uwTickFreq` 递增，
而 `HAL_GetTick()` 返回 `uwTick`。所以 `PWM_Motor_Hardware_Diagnosis()`
的 1000ms 门控依赖目标 MCU 运行时的 HAL tick 推进，不依赖主机端等待时间。
停在断点时，调试器能观察变量，但不能把“人在 IDE 里等了一秒”直接写成
`HAL_GetTick()` 已经增加 1000。

第九，调试冻结配置边界。HAL 头文件中存在 `__HAL_DBGMCU_FREEZE_TIM2()`、
`__HAL_DBGMCU_FREEZE_TIM3()`、`__HAL_DBGMCU_FREEZE_TIM4()` 这类宏，
用于控制部分定时器在调试停机时的行为。但当前应用源码搜索没有看到
TIM2/TIM3/TIM4 的 freeze/unfreeze 调用。因此，断点暂停时 PWM 定时器是否继续计数、
引脚是否保持输出、`CNT` 是否变化，仍应通过 `DBGMCU->CR`、`TIMx->CNT`
和示波器记录确认，不能从“正在调试”四个字直接推出。

第十，初始 PWM 比较值。TIM2、TIM3、TIM4 的各 PWM 通道配置了 `Pulse=500`，
`ARR=3599`，所以初始化配置对应的比较值约为：

```text
500 / (3599 + 1) = 13.9%
```

这不是硬件点测的 `duty=1000`，也不是清零状态。若初始化后尚未进入诊断到期分支，
或闭环控制没有覆盖某个通道，调试器里看到的 `CCR=500` 需要先按初始化配置解释。

第十一，HAL 返回值与寄存器证据。`HAL_TIM_PWM_Start()` 本身可能返回
`HAL_ERROR`，例如通道状态不是 READY 时会提前返回。当前 `PWM_Motor_Init()`
没有保存或判断 9 次 `HAL_TIM_PWM_Start()` 的返回值，最后直接设置
`isInitialized = 1` 并返回 `HAL_OK`。所以调试时应把 `isInitialized`
看成软件流程标志，而不是全部通道成功输出的最终证据。

## 7. 项目中的应用

本章涉及的项目证据文件包括：

- `Three-axis_cloud_platformV2 Debug.launch`
- `Three-axis_cloud_platformV2.ioc`
- `Core/Src/stm32f1xx_hal_msp.c`
- `Core/Src/tim.c`
- `Core/Src/main.c`
- `Drivers/CustomDrivers/Src/drv_pwmMotors.c`
- `Drivers/CustomDrivers/Inc/drv_pwmMotors.h`

项目中的下载调试链路是：

```text
当前源码
-> Debug 构建
-> Debug/Three-axis_cloud_platformV2.elf
-> Debug.launch
-> ST-LINK
-> SWD
-> verify flash download
-> stop at main
```

这条链路中，`.launch` 的 `loadList` 显示 `fPerformBuild=true`、
`fDownload=true`、`fLoadSymbols=true`。这说明调试配置希望先构建、再下载、再加载符号。
但教材仍要把“配置希望这样做”和“本次调试实际完成”区分开：
实际完成情况应由构建日志、ELF 更新时间、下载日志、Flash 校验结果和断点命中共同证明。

如果显式执行当前诊断函数，项目中的电机诊断链路是：

```text
PWM_Motor_Init()
-> 启动 TIM2/TIM3/TIM4 PWM
-> PWM_Motor_Hardware_Diagnosis()
-> 关闭所有相关 CCR
-> 对 Roll 三相依次写 duty
```

当前主程序中只看到 `PWM_Motor_Init()` 被调用，没有看到 `PWM_Motor_Hardware_Diagnosis()` 被调用。

`main.c` 调用 `PWM_Motor_Init()` 时没有检查返回值；而 `PWM_Motor_Init()`
内部也没有逐项处理 `HAL_TIM_PWM_Start()` 的返回结果。
因此项目证据只能说明“程序尝试启动这些 PWM 通道”，不能单凭该调用证明
9 个通道都已经正常输出。寄存器窗口和示波器仍是必要验证。

还要注意，TIM2/TIM3/TIM4 在 `MX_TIMx_Init()` 阶段已经把 PWM `Pulse`
配置为 500。`PWM_Motor_Init()` 启动通道后，只清零了 TIM2/TIM3/TIM4 的
`CNT`，没有统一清零所有 CCR。因此，轴使能关闭、诊断函数尚未到期、
闭环分支尚未写入时，CCR 仍可能保留初始化比较值。
这说明“未进入控制分支”和“输出寄存器已经关闭”不是同一个结论。

还需要注意一条调用关系证据：`computeMotorCommands.c` 的 Roll 控制分支
当前调用 `PWM_Motor_SetAngle(MOTOR_PITCH, ...)`，Pitch 控制分支调用
`PWM_Motor_SetAngle(MOTOR_ROLL, ...)`。这说明控制轴名称和 PWM 驱动枚举名称
之间存在工程内映射关系，不能在诊断表里直接按字面名称等同。

`main.c` 中启动延时结束时会设置 `eepromConfig.pitchEnabled = true`、
`eepromConfig.rollEnabled = false`、`eepromConfig.yawEnabled = true`。
随后 10Hz 分支在 `return_state_count >= 650` 且 Pitch 已使能时，才会把
`eepromConfig.rollEnabled` 置为 true。也就是说，在常规控制路径中，
当前诊断函数实际写入的 `MOTOR_ROLL` 物理输出组，很早就可能被 Pitch
控制分支覆盖；Roll 控制分支是否启用，则还取决于后续状态条件。

教材在本章只采用“实际写入的 CCR 和引脚”作为诊断依据。至于这种轴名交叉是机械安装、历史命名还是临时调试造成的，需要项目作者或硬件实测继续确认。

## 8. 代码分析

### 8.1 Debug.launch关键字段

`Three-axis_cloud_platformV2 Debug.launch` 中的关键字段包括：

- `swd_mode=true`：使用 SWD 模式。
- `startServer=true`：启动调试服务器。
- `remoteCommand=target remote`：GDB 远程连接。
- `PROGRAM_NAME=Debug/Three-axis_cloud_platformV2.elf`：调试和下载目标文件。
- `verify_flash_download=true`：下载后校验 Flash。
- `stopAt=main`：启动调试后停在 `main()`。
- `reset_strategy=connect_under_reset`：连接策略为复位下连接。
- `enable_swv=false`：当前 SWV 未启用。
- `enable_live_expr=true`：启用 Live Expressions。

这些字段说明当前工程具备在线调试变量和断点的配置基础。
其中 `loadList` 还包含 `fPerformBuild=true`、`fDownload=true` 和
`fLoadSymbols=true`，说明调试入口会尝试把构建、下载和符号加载串在一起。

这里的边界很重要：

- `PROGRAM_NAME` 证明调试配置选择了哪个 ELF。
- `fPerformBuild=true` 证明调试配置要求调试前构建，不证明本次构建一定成功。
- `fDownload=true` 证明调试配置要求下载，不证明目标板当前正在运行该映像。
- `verify_flash_download=true` 证明配置了下载后校验，不证明诊断函数已经执行。
- `fLoadSymbols=true` 证明会加载符号，不证明符号与板上固件一定匹配。
- `stopAt=main` 证明配置了入口断点，不证明本次调试会话已经真实停住。

因此，下载调试结论应写成证据链，而不是单点结论。
更可靠的写法是：构建成功、ELF 时间戳更新、下载日志显示校验通过、
调试器停在 `main()`，并且 `PWM_Motor_Init()` 断点命中。
这些条件同时满足时，才能较有把握地说当前固件进入了预期路径。

#### 8.1.1 从调试配置到板上运行的证据边界

调试排查时，建议把“下载成功”继续拆成五层记录。每一层证明的对象不同，不能互相替代：

| 层级 | 证据对象 | 当前仓库能直接证明什么 | 仍需现场确认什么 |
|---|---|---|---|
| 调试配置 | `.launch` 中 `PROGRAM_NAME`、`fPerformBuild`、`fDownload`、`fLoadSymbols` | IDE 调试入口指向 `Debug/Three-axis_cloud_platformV2.elf`，并配置构建、下载、加载符号 | 本次调试会话是否真的执行了这些动作【待验证】 |
| 构建产物 | `Debug/Three-axis_cloud_platformV2.elf`、`.map`、`.list` | 仓库中存在 Debug 产物及符号/反汇编证据 | 产物是否来自最新源码和本次成功构建【待验证】 |
| Flash 校验 | `verify_flash_download=true` | 调试配置要求下载后校验 Flash | 校验日志是否通过，以及校验对象是否就是最新 ELF【待验证】 |
| CPU 入口 | `stopAt=main`、`DEBUGGER_STOP_AT_MAIN=true` | 调试配置要求停在 `main()` | 本次会话是否实际停在 `main()`，PC/SP 是否合理【待验证】 |
| 预期路径 | `PWM_Motor_Init()`、`computeMotorCommands()` 等断点 | 源码和 `.list` 能给出这些路径的位置 | 断点是否命中、变量值是否与预期一致、板上是否正在跑该路径【待验证】 |

因此，`.launch` 是调试意图证据，`.elf/.map/.list` 是构建产物证据，下载日志和断点命中才是现场运行证据。缺少后两者时，教材只能说“工程配置支持该调试链路”，不能写成“目标板已经运行最新固件”。

`.cproject` 也提供了构建配置证据。当前 Debug 配置的产物扩展名是 `elf`，
C/C++ 和汇编调试级别使用 `g3`。这有利于源码级断点和变量观察，
但调试信息不是运行时证据；变量窗口仍依赖 CPU 停机、实时访问能力和符号匹配。

### 8.2 SWD引脚配置

`.ioc` 中配置：

```text
PA13.Mode=Serial_Wire
PA13.Signal=SYS_JTMS-SWDIO
PA14.Mode=Serial_Wire
PA14.Signal=SYS_JTCK-SWCLK
```

`HAL_MspInit()` 中配置：

```text
__HAL_AFIO_REMAP_SWJ_NOJTAG()
```

这两部分共同构成“保留 SWD，关闭 JTAG”的项目证据。

### 8.3 PWM初始化前提

`Core/Src/main.c` 中调用：

```text
PWM_Motor_Init()
```

`PWM_Motor_Init()` 内部启动 9 个 PWM 通道，并将 TIM2、TIM3、TIM4 的 CNT 清零。

但这里有一个调试边界：9 次 `HAL_TIM_PWM_Start()` 的返回值没有被保存或判断。
函数末尾直接执行：

```text
isInitialized = 1
return HAL_OK
```

所以 `PWM_Motor_Init()` 返回 `HAL_OK` 更接近“初始化流程执行到函数末尾”，
不能单独证明每一个 PWM 通道都成功打开。

如果没有先执行初始化，诊断函数开头会因为：

```text
if (!isInitialized) return
```

直接退出。
如果 `isInitialized=1`，仍要继续检查 `CR1.CEN`、`CCER.CCxE`、`CNT`
和外部波形，才能证明定时器与通道确实在运行。

#### 8.3.1 PWM启动返回值丢失边界

`PWM_Motor_Init()` 当前把 9 次 `HAL_TIM_PWM_Start()` 当作顺序动作执行，但没有把每次调用的返回值保存到局部变量，也没有在失败时提前返回错误。这样会形成一个常见调试陷阱：函数最终 `return HAL_OK`，并不等于 9 个通道的 `HAL_TIM_PWM_Start()` 都返回了 `HAL_OK`。

| 证据层 | 当前可见证据 | 可以证明 | 不能证明 |
|---|---|---|---|
| 源码调用 | `drv_pwmMotors.c` 中连续调用 9 次 `HAL_TIM_PWM_Start()` | 初始化流程尝试启动 TIM3、TIM2、TIM4 的相关 PWM 通道 | 每次 HAL 调用的返回值是否成功 |
| 构建产物 | `.list` 中可见 9 个 `bl <HAL_TIM_PWM_Start>` 调用点 | 当前 Debug 产物保留了这些调用路径 | 调用运行时返回 `HAL_OK` 还是错误 |
| 软件标志 | 函数末尾写 `isInitialized = 1` 并 `return HAL_OK` | 代码执行到了函数末尾 | 通道使能位、定时器计数和外部引脚波形都已正常 |
| 寄存器观察 | `CR1.CEN`、`CCER.CCxE`、`CNT`、`CCRn` | 可以验证定时器和通道是否处于运行/使能状态 | 电机驱动板、电源、电流和机械负载是否正常【待验证】 |

因此，排查“诊断函数不输出”时，不应只看 `PWM_Motor_Init()` 的返回值。更稳妥的记录方式是：先确认 `PWM_Motor_Init()` 断点命中，再逐个检查 TIM2/TIM3/TIM4 的 `CR1.CEN`、对应 `CCER.CCxE`、`CNT` 是否递增和 `CCRn` 是否符合预期，最后用示波器确认引脚波形。

### 8.4 初始Pulse与CCR残留边界

`Core/Src/tim.c` 中 TIM2、TIM3、TIM4 的 PWM 通道配置使用同一类参数：

```text
sConfigOC.OCMode = TIM_OCMODE_PWM1
sConfigOC.Pulse = 500
Period = 3599
```

因此初始化配置层面的默认占空比约为：

```text
500 / 3600 = 13.9%
```

`PWM_Motor_Init()` 启动通道并清零 TIM2/TIM3/TIM4 的 `CNT`，
但没有在初始化末尾执行：

```text
TIMx->CCRn = 0
```

所以调试时如果看到 `CCR=500`，不能直接判断诊断函数已经写入，也不能判断闭环输出正在工作。
它可能只是 `MX_TIMx_Init()` 配置通道时写入的初始 Pulse。

### 8.5 诊断函数关闭所有CCR

进入诊断 step 后，函数先执行：

```text
TIM3->CCR2 = off; TIM3->CCR3 = off; TIM3->CCR4 = off
TIM3->CCR1 = off; TIM2->CCR4 = off; TIM2->CCR3 = off
TIM4->CCR4 = off; TIM2->CCR2 = off; TIM4->CCR3 = off
```

这一步的作用是先让所有已知三轴输出通道归零，再打开当前要测试的一相。这样可以减少多相同时输出造成的判断混乱。

但这一步只有进入 1000ms 到期分支才会执行。若函数因为 `isInitialized == 0`
提前返回，或者因为 `current_tick - last_tick < 1000` 没到期直接返回，
则不会清零任何 CCR。

### 8.6 诊断函数当前实际覆盖范围

当前实际未注释的写入只有：

```text
TIM3->CCR1 = duty
TIM2->CCR4 = duty
TIM2->CCR3 = duty
```

这与 `MOTOR_ROLL` 的三相映射一致。函数注释写的是“所有电机的 A/B/C 相”，但源码中 Pitch 和 Yaw 写入被注释。因此教材必须以实际写入为准，不能沿用注释写成完整三轴诊断。

### 8.7 诊断函数未接入主循环

当前搜索结果显示：

- `PWM_Motor_Hardware_Diagnosis()` 在头文件中声明。
- `PWM_Motor_Hardware_Diagnosis()` 在 `drv_pwmMotors.c` 中定义。
- 未发现 `main.c` 或当前主控制路径调用该函数。

这说明它当前是一个保留调试工具，而不是开机自动运行的测试流程。

函数内部还有两个静态变量：

```text
static uint8_t step = 0
static uint32_t last_tick = 0
```

每次调用先读取：

```text
current_tick = HAL_GetTick()
```

只有 `current_tick - last_tick >= 1000` 时才会更新 CCR。
这意味着它必须被反复调用，才能形成 1 秒一步的点测序列。
单次调用最多只更新一个 step；如果 tick 间隔不足 1000ms，则不会更新任何 CCR。

### 8.8 duty和ARR的关系

TIM2、TIM3、TIM4 的 `Period=3599`，也就是计数周期长度约为 3600。诊断函数写：

```text
duty = 1000
```

因此固定输出占空比约为：

```text
1000 / 3600 = 27.8%
```

这个值用于点测，不代表正常闭环控制功率，也不代表 `PWM_Motor_SetAngle()` 的三相正弦输出幅值。

### 8.9 PWM启动后的寄存器观察点

`PWM_Motor_Init()` 调用 `HAL_TIM_PWM_Start()` 启动 9 个通道，但当前没有检查这些调用的返回值。调试时不能只看 `isInitialized=1`，还应观察寄存器状态。

推荐最小观察点包括：

```text
TIM2/TIM3/TIM4->CR1.CEN
TIM2/TIM3/TIM4->CNT
TIMx->CCER 中对应 CCxE 位
TIMx->CCMR1/CCMR2 中对应 OCxPE 位
TIMx->CCRn
```

`CR1.CEN` 和 `CNT` 用于确认定时器是否运行，`CCER.CCxE` 用于确认通道是否打开，`OCxPE` 用于判断 CCR 写入是否经过输出比较预装载路径，`CCRn` 用于确认诊断或闭环是否正在写比较值。只有 `CCRn` 变化而 `CCxE` 未打开时，引脚不一定输出；`CCRn` 已变化且 `OCxPE` 置位时，也要结合更新事件和波形确认新占空比从哪个 PWM 周期开始生效。

还应区分“初始 Pulse”和“诊断 duty”。`Core/Src/tim.c` 中 TIM2/TIM3/TIM4
各 PWM 通道初始 `Pulse=500`，而诊断函数到期后会先把全部相关 `CCR`
写为 0，再把当前 step 对应通道写为 1000。
如果观察到 `CCR=500`，更像是 CubeMX 初始化配置留下的初始比较值；
如果观察到 `CCR=1000` 按 step 出现，才更接近诊断函数已经到期执行的证据。

### 8.10 轴名交叉的代码证据

当前控制链路中，逻辑轴和 PWM 驱动枚举并非简单同名对应：

```text
Roll 控制分支  -> PWM_Motor_SetAngle(MOTOR_PITCH, ...)
Pitch 控制分支 -> PWM_Motor_SetAngle(MOTOR_ROLL, ...)
Yaw 控制分支   -> PWM_Motor_SetAngle(MOTOR_YAW, ...)
```

这不一定表示代码错误，也可能来自机械安装方向、接口命名历史或调试阶段映射调整。但教材不能把它自动解释为“轴名完全一致”。复查时应把每个控制分支最终写入的 `TIMx->CCRn` 展开，再与实物轴响应对照。

### 8.11 诊断写入被覆盖的代码证据

`PWM_Motor_Hardware_Diagnosis()` 写入 `TIM3->CCR1`、`TIM2->CCR4` 和
`TIM2->CCR3`。正常闭环中，`PWM_Motor_SetAngle(MOTOR_ROLL, ...)`
也会写这些寄存器。当前调用关系要拆开看：

```text
诊断函数实际写入      -> MOTOR_ROLL 物理输出组 -> TIM3 CCR1, TIM2 CCR4/CCR3
Pitch 控制分支正常输出 -> MOTOR_ROLL 物理输出组 -> TIM3 CCR1, TIM2 CCR4/CCR3
Roll 控制分支正常输出  -> MOTOR_PITCH 物理输出组 -> TIM3 CCR2/CCR3/CCR4
Yaw 控制分支正常输出   -> MOTOR_YAW  物理输出组 -> TIM4 CCR4, TIM2 CCR2, TIM4 CCR3
```

所以，若只讨论当前诊断函数未注释的三相点测，最直接的软件覆盖者是
Pitch 控制分支。`main.c` 在启动延时结束后会先打开 Pitch 和 Yaw，
而 Roll 使能会在后续状态条件满足后再打开；因此诊断隔离不能只看
`rollEnabled`，更要检查 `pitchEnabled` 是否仍在 500Hz 分支中工作。

如果诊断函数与闭环函数同时运行，最后一次写入者决定本周期寄存器值。由于闭环理想周期约 2ms，而诊断切换周期为 1000ms，诊断输出可能只短暂存在，甚至在同一循环内被覆盖。排查时应记录：

- 诊断函数是否被调用。
- 调用点位于 `computeMotorCommands()` 之前还是之后。
- 被测物理输出组对应的正常控制分支是否关闭。
- 点测 `MOTOR_ROLL` 物理输出组时，`pitchEnabled` 是否仍为 true。
- 观察到的 `CCR` 值是否能稳定保持到下一个诊断 step。

### 8.12 示波器验收口径

在 TIM2/TIM3/TIM4 均使用 `PSC=0`、`ARR=3599` 且 APB1 定时器时钟为 72MHz 的前提下，理论 PWM 频率为：

```text
f_pwm = 72MHz / ((0 + 1) * (3599 + 1)) = 20kHz
```

诊断 duty 为 1000 时，理论占空比为：

```text
D = 1000 / 3600 = 27.8%
```

示波器或逻辑分析仪应分别验证：

- 被测引脚是否是 PA6、PA3、PA2。
- 频率是否接近 20kHz。
- 高电平占空比是否接近 27.8%。
- 在 `CCR` 改变后的下一个或后续 PWM 周期，占空比是否按预期更新。
- step 是否约每 1000ms 在三相之间切换。

这些是引脚层面的验证口径。电机是否转动、方向是否正确、驱动是否过热、电流是否安全，仍属于负载层面【待验证】。

#### 8.12.1 官方资料与调试现场证据边界

第33章的下载调试结论还要加入官方资料层。ST 的 ST-LINK GDB server / STM32CubeIDE 调试资料可以支撑“ST-LINK、GDB server、下载、加载符号、Flash 校验、停在入口”等调试会话概念；STM32F10x 参考手册可以支撑 AFIO/SWJ 配置、TIM 输出比较、CCR、CCER、CNT、预装载和更新事件等寄存器语义；STM32 定时器应用笔记可以支撑 PWM 频率、占空比和定时器通道诊断的通用分析口径。

但官方资料层只能证明工具和外设语义，不证明本项目这次调试会话已经真实完成。项目配置层只能证明 `.launch` 指向 `Debug/Three-axis_cloud_platformV2.elf`、启用 SWD、配置 Flash 校验和 `stopAt=main`；源码层只能证明 `HAL_MspInit()` 保留 SWD、`PWM_Motor_Init()` 尝试启动 9 个 PWM 通道、诊断函数存在但当前无主线调用证据；构建产物层只能证明相关符号和调用路径进入当前 Debug 产物或只停留在输入段。真正的现场层必须由下载日志、PC/SP、断点命中、`DBGMCU->CR`、TIM 寄存器、示波器波形、电源电流和电机响应记录支撑。

因此，本章不能把“ST-LINK 官方支持 SWD 调试”写成“当前桌面已经连接目标板”，也不能把“TIM PWM 官方语义成立”写成“PA6/PA3/PA2 已经输出正确波形”。官方资料是判读证据的语法书，项目源码和构建产物是当前工程的静态证据，板上日志与测量才是运行事实；缺少现场记录时保持【待验证】。

### 8.13 构建产物证据边界

`Debug/Three-axis_cloud_platformV2.map` 和 `Debug/Three-axis_cloud_platformV2.list` 可以用来补强“当前 Debug 构建实际包含了哪些符号、入口调用和 HAL/PWM 路径”，但它们不能替代下载日志、断点命中、寄存器观察或示波器记录。

当前 `.map` 的最终地址区能看到 `MX_TIM2_Init()`、`MX_TIM3_Init()`、`MX_TIM4_Init()`、`PWM_Motor_Init()`、`PWM_Motor_SetAngle()`、`HAL_TIM_PWM_Start()`、`HAL_TIM_PWM_ConfigChannel()`，以及 `htim2`、`htim3`、`htim4` 等符号。这说明这些函数或对象进入了当前 Debug 产物。`.list` 中还能看到 `main()` 依次调用 `MX_TIM3_Init()`、`MX_TIM2_Init()`、`MX_TIM4_Init()` 和 `PWM_Motor_Init()`；进入 `PWM_Motor_Init()` 后，又能看到 9 次 `HAL_TIM_PWM_Start()` 调用、`TIM2->CNT` / `TIM3->CNT` / `TIM4->CNT` 清零，以及 `isInitialized = 1`。

需要特别分清的是，`.map` 中 `PWM_Motor_Hardware_Diagnosis()` 当前出现在 `0x00000000` 输入段区域，而不是最终 Flash 地址区。因此它不能被写成“当前最终固件主线已经包含并可达的诊断入口”，更不能据此推断板上正在执行该函数。当前 `.list` 也只能证明 `main()` 调用了 `PWM_Motor_Init()`，没有证明 `main()` 或 500Hz 主控制路径调用了 `PWM_Motor_Hardware_Diagnosis()`。

#### 8.13.1 .map行的判读边界

当前 Debug 构建命令给出了这个判读边界的直接原因。`Debug/Drivers/CustomDrivers/Src/subdir.mk`
中编译 `drv_pwmMotors.c` 时使用了 `-ffunction-sections -fdata-sections`，这会让函数和数据分别进入
独立输入 section；`Debug/makefile` 的链接命令又使用 `-Wl,--gc-sections` 和
`-Wl,-Map="Three-axis_cloud_platformV2.map"`。按照 GCC 与 GNU ld 的官方语义，前者为每个函数或数据项生成独立
section，后者会按入口符号、命令行未定义符号以及重定位引用递归标记需要保留的输入 section，未被引用的输入 section
可以被链接垃圾回收。因此，`.map` 文件顶部的 `Discarded input sections` 不是普通最终地址区，
而是“曾经由目标文件提供、但没有映射进最终输出 section”的记录区。

读 `.map` 时不能只搜索函数名，还要看它出现在哪一类地址区。当前 `drv_pwmMotors.o`
里可以形成一个很好的对照：

```text
.text.PWM_Motor_Hardware_Diagnosis  0x00000000  0xe0  ./Drivers/CustomDrivers/Src/drv_pwmMotors.o
.text.PWM_Motor_Init                0x080026ac  0x9c  ./Drivers/CustomDrivers/Src/drv_pwmMotors.o
.text.PWM_Motor_SetAngle            0x08002748  0x280 ./Drivers/CustomDrivers/Src/drv_pwmMotors.o
```

`0x080026ac` 和 `0x08002748` 位于 STM32F103 的 Flash 地址空间，
并且后面还有可被调试器识别的全局符号行：

```text
0x080026ac                PWM_Motor_Init
0x08002748                PWM_Motor_SetAngle
```

这类证据可以支撑“函数进入当前 Debug 最终镜像”。再结合 `.list` 中
`main()` 对 `PWM_Motor_Init()` 的 `bl` 指令，才能继续支撑“主线会调用该函数”。

相反，`PWM_Motor_Hardware_Diagnosis()` 当前只看到 `0x00000000` 输入段记录，
没有看到 `0x0800....` 形式的最终 Flash 地址，也没有看到从 `main()` 或
500Hz 控制路径跳入它的 `bl` 指令。因此更准确的结论是：

```text
源码保留了诊断函数
-> 编译中形成过该函数输入段
-> 当前 Debug 最终镜像未证明保留该函数入口
-> 当前主线未证明会调用该函数
```

如果调试者把 `0x00000000` 输入段误读成最终地址，就会产生两个错误结论：
一是误以为可以在当前固件里直接断到该函数；二是误以为诊断逻辑可能正在后台运行。
这两个结论都需要重新用最终 Flash 地址、`.list` 调用路径和板上断点命中证明。

#### 8.13.2 诊断函数可达性证据边界

`PWM_Motor_Hardware_Diagnosis()` 的证据链要比普通源码搜索再细一层。调试结论至少应拆成五级：

| 层级 | 当前证据 | 能证明什么 | 不能证明什么 |
|---|---|---|---|
| 源码声明/定义 | `drv_pwmMotors.h` 声明函数，`drv_pwmMotors.c` 定义函数体 | 仓库中确实保留了诊断工具源码 | 不能证明该函数进入最终固件 |
| 对象文件输入段 | `.map` 中可见 `.text.PWM_Motor_Hardware_Diagnosis 0x00000000`，`.su/.cyclo` 给出 24 字节栈和圈复杂度 8 | 编译阶段曾生成该函数的输入段和静态分析记录 | `0x00000000` 输入段不能写成最终 Flash 可达地址 |
| 最终镜像可达 | 需要 `.map` 最终地址区出现该函数的非零 Flash 地址 | 才能说明链接后未被丢弃，调试器具备符号入口 | 当前 Debug 产物没有给出这一级证据 |

这张表的重点不是否定诊断函数的价值，而是避免把“存在源码”和“正在点测硬件”混在一起。
如果未来要把它接入专门调试构建，教材应重新记录新的 `.map` 最终地址、`.list` 调用路径、
断点命中记录和 CCR/波形证据；在当前仓库状态下，只能把它写成保留诊断工具。

`.su` 和 `.cyclo` 能给出本章相关函数的静态资源边界，但证据层级还要再分清：

| 关注点 | 函数 | 静态栈用量 | 圈复杂度 | 构建产物证据 |

但不能证明它进入最终镜像的可达代码区；最终可达性仍应以 `.map` 中是否有 Flash 地址和 `.list` 中是否有主线调用为准。

## 9. 调试方法

本节按统一调试结构组织：现象 -> 可能原因 -> 定位方法 -> 验证步骤 -> 解决方案 -> 经验总结。对第33章而言，核心是把下载会话、SWD 连接、PWM 初始化、诊断函数是否执行、诊断独占条件、Roll 三相点测和外部波形验证分开记录，避免把静态配置或源码存在误写成板上硬件已经通过。

### 9.1 现象与可能原因

常见现象包括无法下载、调试器连接不稳定、CPU 没有停在 `main()`、PWM 初始化后无波形、`PWM_Motor_Hardware_Diagnosis()` 没有按预期扫描、Roll 三相点测被闭环输出覆盖，或示波器波形与理论频率/占空比不一致。可能原因包括 ELF 不是最新构建、SWDIO/SWCLK/复位/电源/地线异常、PA13/PA14 未保留为 SWD、9 路 PWM 启动返回值未逐项记录、诊断函数没有主线调用点、1000ms tick 门控未到期、CPU 停在断点导致 `uwTick` 不增长，或诊断输出没有与 500Hz 闭环输出隔离。

### 9.2 定位方法：下载配置、SWD 和 PWM 初始化

第一步，确认下载配置。

- 打开 `Three-axis_cloud_platformV2 Debug.launch`。
- 确认 `PROGRAM_NAME` 指向 `Debug/Three-axis_cloud_platformV2.elf`。
- 确认 `swd_mode=true`、`verify_flash_download=true`、`stopAt=main`。
- 确认 `loadList` 中包含构建、下载和加载符号配置。
- 查看本次构建日志是否成功，记录 ELF 更新时间和大小。
- 查看下载日志和 Flash 校验结果，避免把旧 ELF 当成最新源码。
- 启动调试后确认 CPU 实际停在 `main()`，再继续判断后续初始化。

第二步，确认 SWD 引脚保留。

- 在 `.ioc` 中确认 PA13/PA14 是 Serial Wire。
- 在 `HAL_MspInit()` 中确认使用 `__HAL_AFIO_REMAP_SWJ_NOJTAG()`。
- 如果调试器连接不稳定，优先排查 SWDIO、SWCLK、复位、电源和地线连接。

第三步，确认 PWM 初始化已经执行。

- 在 `PWM_Motor_Init()` 设置断点。
- 确认函数返回前 `isInitialized` 被置为 1。
- 确认 TIM2/TIM3/TIM4 的相关 PWM 通道已经启动。
- 不要只用 `PWM_Motor_Init()` 返回值判断成功，因为 9 次 `HAL_TIM_PWM_Start()` 返回值当前没有逐项检查。
- 继续观察 `CR1.CEN`、`CNT`、`CCER.CCxE` 和对应 `CCRn`。
- 若初始化后看到 `CCR=500`，先按 `MX_TIMx_Init()` 的初始 `Pulse=500` 解释，
  不要直接写成诊断函数已经执行或输出已经关闭。

### 9.3 验证步骤：诊断入口和独占条件

第四步，确认诊断函数是否被执行。

- 搜索 `PWM_Motor_Hardware_Diagnosis()` 调用点。
- 如果没有调用点，就不能期待系统自动执行硬件点测。
- 若使用调试器手动调用或临时接入，应先确保正常控制输出被隔离，避免诊断输出和闭环输出互相覆盖。
- 若手动调用，只调用一次不能证明三相扫描；应记录 `HAL_GetTick()`、`last_tick`、`step` 和 CCR 是否到期变化。
- 若在断点处暂停后手动等待，不要把电脑端等待时间当成 `HAL_GetTick()` 增量；应让目标板继续运行，并记录运行前后的 `uwTick` 或 `HAL_GetTick()`。
- 若临时接入，应让函数在循环中持续执行，否则 1000ms 门控不会形成稳定点测序列。

第五步，确认诊断独占条件。

- 临时接入诊断前，确认同一批 `TIMx->CCRn` 没有被闭环继续写入。
- 若仍执行 `computeMotorCommands()`，记录诊断函数调用点和闭环函数调用点的先后顺序。
- 当前实际点测 `MOTOR_ROLL` 物理输出组时，优先检查并关闭
  `eepromConfig.pitchEnabled`，因为 Pitch 分支会写 `TIM3->CCR1`、
  `TIM2->CCR4` 和 `TIM2->CCR3`。
- 推荐在专门调试构建中关闭被测物理输出组对应的控制使能，
  避免 500Hz 输出覆盖 1000ms 点测。

### 9.4 验证步骤：Roll 三相点测和 PWM 波形

第六步，观察 Roll 三相输出。

- `step=0` 时观察 TIM3_CH1 / PA6。
- `step=1` 时观察 TIM2_CH4 / PA3。
- `step=2` 时观察 TIM2_CH3 / PA2。
- 每个 step 间隔约 1000ms。
- 如果 step 长时间不变，先检查函数是否持续调用，以及 `HAL_GetTick() - last_tick` 是否达到 1000。
- 如果手动等待 1 秒后 step 仍不变，先确认这 1 秒内 CPU 是否处于运行态，而不是停在断点；同时记录 `uwTick` 是否增加约 1000。
- 如果看到 `CCR=500` 而不是 0/1000 切换，先区分这是 TIM 初始化 Pulse、闭环输出覆盖，还是诊断函数到期写入。

第七步，测量 PWM 波形。

- 示波器触发先选 PA6，再切换到 PA3 和 PA2。
- 理论频率按 20kHz 预期观察，真实波形保持【待验证】。
- 理论占空比按 27.8% 预期观察，驱动级实际输出保持【待验证】。
- 若 `CCR` 正常但引脚无波形，优先检查 `CCER.CCxE`、`OCxPE`、更新事件、GPIO 复用和探头接地。

### 9.5 解决方案：分层排查

第八步，诊断失败时分层排查。

- 若 CCR 没有变化，检查函数是否执行和 `isInitialized` 是否为 1。
- 若 `isInitialized=1` 但 `CR1.CEN` 或 `CCER.CCxE` 异常，回到 PWM 启动返回值和 HAL 通道状态排查。
- 若函数执行但 CCR 不变，检查 1000ms tick 门控是否到期。
- 若函数未到期直接返回，不要期待它已经清零全部 CCR。
- 若 CCR 变化但引脚无输出，检查通道使能、输出比较预装载位、更新事件、GPIO 复用和定时器通道配置。
- 若引脚有输出但电机无响应，继续检查外部供电、连接和驱动级问题。
- 若 Roll 正常但 Pitch/Yaw 无诊断输出，先确认源码中 Pitch/Yaw 写入当前是注释状态。
- 若诊断输出不稳定，检查是否被 500Hz 闭环输出覆盖；
  当前 Roll 物理点测尤其要检查 Pitch 分支是否仍在写同一组 CCR。

### 9.6 解决方案：调试记录

- 记录 `.launch` 下载配置、SWD 保留证据、断点命中位置和固件版本。
- 对电机诊断，应记录函数是否被调用、`isInitialized`、TIM/CCR 变化和对应引脚。
- 周期诊断记录应包含 `HAL_GetTick()`、`last_tick`、`step`、是否到期和本次写入的 CCR。
- 若用 CCR 判断点测是否生效，应同步记录对应 `OCxPE` 位、`CNT` 位置和下一周期波形变化，避免把软件写入证据直接等同为引脚已经更新。
- 若涉及断点或单步，应额外记录 CPU 是运行态还是暂停态，以及暂停前后的 `uwTick`。
- 若观察 PWM 波形随断点变化，应记录 `DBGMCU->CR` 中 TIM2/TIM3/TIM4 调试冻结位、`TIMx->CNT` 和示波器波形。
- 诊断输出与闭环输出应分开记录，临时接入诊断前要记录控制输出隔离条件。
- 轴名、枚举名、物理接口名和引脚名应分别记录，不要混在同一列。
- SWD 连线、电源、驱动级、外部 PWM 波形和电机响应都属于仓库外实测证据。
- `.map` / `.list` / ELF 记录应区分最终地址符号、`0x00000000` 输入段、源码调用点和板上运行证据。
- 构建产物记录应包含源码修改点、构建时间、ELF 路径、ELF 时间戳、下载校验结果、
  `main()` 停止证据和关键断点命中证据。

### 9.7 经验总结

第33章的调试结论应按“配置 -> 下载 -> 运行 -> 寄存器 -> 引脚 -> 负载”的顺序收敛。`.launch`、`.ioc`、源码和构建产物能证明调试入口、SWD 保留、PWM 初始化路径和诊断函数证据层级；断点、PC/SP、`uwTick`、TIM 寄存器和 CCR 变化能证明某次运行状态；示波器、电源电流、驱动板状态和电机响应才能证明真实硬件点测结果。缺少现场记录时，不把诊断函数存在、CCR 写入或理论 20kHz 写成硬件已通过。

## 10. 常见问题

问题一：当前项目是否使用 SWD 调试。

当前仓库配置为 SWD 调试。
`.launch` 中 `swd_mode=true`，`.ioc` 中 PA13/PA14 配置为 Serial Wire。
但这只能证明工程调试配置，不能证明当前桌面环境已经连接到真实目标板。

问题二：调试启动后是否会停在 `main()`。

按当前配置会停在 `main()`，因为 `.launch` 中 `stopAt=main`，同时 `DEBUGGER_STOP_AT_MAIN=true`。
但这是配置层面的结论。一次具体调试会话是否真的停住，还要看调试器状态、
调用栈、PC 位置和断点命中记录。

问题三：`PWM_Motor_Hardware_Diagnosis()` 是否会自动运行。

当前没有证据。只发现声明和定义，未发现主程序调用。

问题四：诊断函数是否完整测试三轴。

当前不是。Pitch 和 Yaw 的相位写入在函数中被注释，实际未注释的点测覆盖 Roll 三相。

问题五：诊断函数通过后是否说明 PID 正确。

不能。
诊断函数绕过了姿态解算、PID、电角合成和部分主线正弦输出路径。
即使点测有响应，也只能证明被测试通道在该诊断条件下可观察到输出。
它不能证明闭环 PID、姿态方向或三轴映射已经正确。

问题六：为什么诊断前要先关闭全部 CCR。

为了让每次只观察一个相位输出，减少多个通道同时输出带来的误判。

问题七：为什么第33章不重新展开三相 PWM 原理。

三相 PWM 原理和映射已经在第30章讲过。本章只把它用于下载后点测和调试流程。

问题八：`PWM_Motor_Hardware_Diagnosis()` 没有自动调用，为什么还要讲。

因为它是仓库内存在的底层排查工具，能帮助读者理解如何绕开上层控制链路做点测。
但当前主线没有自动调用证据。
如果要实际启用，应单独设计进入条件、退出条件和记录方式，
不能把它当成当前固件默认会执行的诊断流程。

问题九：为什么手动调用了诊断函数，示波器上仍可能看不到稳定点测输出。

一种软件层面的原因是闭环输出覆盖。`computeMotorCommands()` 在 500Hz 分支中写 `CCR`，而诊断函数每 1000ms 才切换一次。若二者同时运行，最后写入者决定寄存器当前值。

另一种原因是调用方式不对。诊断函数依赖 `HAL_GetTick()` 与静态 `last_tick`，
只有到期才更新 CCR。单次手动调用不等于连续点测；如果未到 1000ms，
本次调用可能直接返回而不改变任何寄存器。

问题十：看到 `MOTOR_ROLL` 就能直接判断它一定对应物理 Roll 轴吗。

不能。当前头文件注释、驱动实现注释和控制调用关系存在命名交叉。排查时应以 `TIMx->CCRn`、GPIO 引脚和实测机械响应为准，机械轴归属在没有外部证据时保持【待验证】。

问题十一：`GPIO_SPEED_FREQ_LOW` 会把 PWM 频率降到低频吗。

不会按这个字段直接决定 PWM 周期。PWM 频率由定时器时钟、`PSC` 和 `ARR` 决定。GPIO speed 影响的是输出驱动特性，真实边沿质量和负载表现需要示波器验证。

问题十二：`PWM_Motor_Init()` 返回 `HAL_OK` 是否证明 9 个 PWM 通道都正常。

不能单独这样判断。
当前函数没有逐项检查 `HAL_TIM_PWM_Start()` 的返回值，最后直接设置 `isInitialized=1`。
因此还要看 `CR1.CEN`、`CNT`、`CCER.CCxE`、`CCRn` 和实际引脚波形【待验证】。

问题十三：PA3 既有 `LED2` 宏又是 TIM2_CH4，调试时听谁的。

听实际初始化和复用配置。当前 `gpio.h` 里有 `LED2` 指向 PA3 的宏定义，
但未看到主线使用；`.ioc` 与 `tim.c` 把 PA3 配为 `TIM2_CH4`。
本章点测 PA3 时应按 TIM2_CH4 处理。

问题十四：`.launch` 指向 ELF 是否证明板上正在运行最新源码。

不能单独证明。它只能说明调试配置选择了哪个 ELF。
还需要构建成功、ELF 时间戳更新、下载校验通过、停在 `main()`、
关键断点命中和变量状态匹配，才能把“当前源码”与“板上运行固件”连起来。

问题十五：`verify_flash_download=true` 能证明什么。

它证明调试配置要求下载后校验 Flash 内容与下载映像一致。
它不能证明源码已经重新构建，不能证明电机诊断函数被调用，
也不能证明引脚波形、电机响应或闭环控制正确。

问题十六：为什么看到 `CCR=500` 不能直接判断诊断函数已经执行。

因为 TIM2、TIM3、TIM4 的 PWM 初始化配置本身就设置了 `Pulse=500`。
诊断函数的点测 duty 是 1000，并且只有进入 1000ms 到期分支才会先清零再写 duty。
所以 `CCR=500` 更可能首先指向初始化比较值；要结合 `step`、`last_tick`、
函数是否进入到期分支和闭环是否覆盖来判断。

问题十七：轴使能为 false 是否等于 PWM 输出一定为 0。

不能。轴使能为 false 只说明 `computeMotorCommands()` 对应轴分支不会写新的控制输出。
TIM 通道是否仍在运行、CCR 是否保留初始化值或旧值、外部引脚是否有波形，
还要看定时器寄存器和示波器记录【待验证】。

问题十八：调试器里等了 1 秒，为什么诊断函数仍可能不到期。

因为诊断函数判断的是 `HAL_GetTick() - last_tick >= 1000`。
当前 `HAL_GetTick()` 返回 HAL 的 `uwTick`，而 `uwTick` 由 SysTick 中断里的
`HAL_IncTick()` 推进。CPU 停在断点或单步暂停时，正常运行中的 tick 推进不等同于
主机端墙钟等待。调试时应让目标板连续运行，并记录暂停前后 `uwTick` 或
`HAL_GetTick()` 的实际差值。

问题十九：断点暂停时 PWM 定时器一定会停止吗。

不能直接下结论。HAL 提供了 TIM2/TIM3/TIM4 的 DBGMCU freeze 宏，
但当前应用源码没有看到这些宏被调用。断点暂停时 `TIMx->CNT` 是否继续变化、
PWM 引脚是否保持输出，要看 `DBGMCU->CR`、定时器寄存器和示波器记录【待验证】。

在目标板上分别记录断点暂停前后 `DBGMCU->CR`、TIM2/TIM3/TIM4 的 `CNT`
和外部 PWM 波形是否继续变化。
验收依据是结论分清“当前源码是否配置调试冻结”和“硬件实际暂停表现”两层证据；缺少板上寄存器或波形记录时保持【待验证】。

## 12. 思考题

1. 为什么调试配置中 `stopAt=main` 对初学者定位初始化问题很重要。
2. 为什么保留 SWD 的同时关闭 JTAG，可以减少引脚资源冲突。
3. 为什么电机硬件诊断应该绕过 PID 和姿态解算。
4. 如果 `PWM_Motor_Hardware_Diagnosis()` 没有调用点，调试者应该如何判断它是否真的执行过。
5. 为什么当前诊断函数不能被写成完整三轴自动点测。
6. 如果 Roll 三相点测正常，但闭环控制时 Roll 异常，排查重点应转向哪些模块。
7. 如果要把 Pitch/Yaw 也纳入诊断，应优先复查哪些映射关系。
8. 为什么“点测成功”只能证明某条链路可响应，不能证明闭环控制已经正确。
9. 为什么诊断函数和闭环控制函数同时写 `CCR` 时，必须先确认最后写入者。
10. 为什么轴名注释、枚举名和物理电机接口不能直接互相替代。
11. 如果示波器频率正确但电机无响应，下一步应如何区分驱动级和负载级问题。
12. 为什么 `PWM_Motor_Hardware_Diagnosis()` 必须周期性调用，而不是手动调用一次就结束。
13. 为什么 `isInitialized=1` 只能作为软件流程标志，不能替代通道使能和波形证据。
14. 当一个引脚同时有旧宏名和当前外设复用证据时，应该如何确定调试口径。
15. 为什么 `verify_flash_download=true` 不能替代构建日志、ELF 时间戳和断点命中证据。
16. 如果符号文件与板上固件不匹配，变量窗口和源码级断点可能产生哪些误导。
17. 为什么初始化 `Pulse=500` 会影响诊断前的 CCR 解释。
18. 为什么轴使能关闭不能直接等价为 PWM 输出关闭。
19. 为什么诊断函数“会清零全部 CCR”的结论必须附带“进入 1000ms 到期分支”这个条件。
20. 为什么调试器暂停时等待 1 秒，不一定等于 `HAL_GetTick()` 增加了 1000。
21. 为什么断点暂停时 PWM 是否继续输出，需要看 DBGMCU 冻结配置、定时器寄存器和外部波形。
22. 为什么 `.map` 中只出现在 `0x00000000` 输入段的诊断函数，不能直接证明它已进入当前最终固件并正在运行。

## 13. 本章总结

本章把下载调试链路和电机硬件诊断连接起来。

当前项目中：

- `Three-axis_cloud_platformV2 Debug.launch` 配置了 ST-LINK、SWD、远程 GDB、Flash 下载校验、停在 `main()` 和目标 ELF。
- `.cproject` 中 Debug 配置生成 ELF，并使用 `g3` 调试级别支持源码级调试。
- `.launch` 的构建、下载、加载符号配置不等于本次调试已完成这些动作。
- `verify_flash_download=true` 校验的是下载映像与 Flash，不证明源码已更新或诊断函数已执行。
- `.ioc` 中 PA13/PA14 保留为 Serial Wire，`HAL_MspInit()` 中关闭 JTAG 保留 SWD。
- `PWM_Motor_Hardware_Diagnosis()` 是一个底层点测函数，用于绕过算法直接写 CCR。
- 当前诊断函数没有主程序或主控制路径调用证据。
- 当前诊断函数实际未注释的写入主要覆盖 Roll 三相，不是完整三轴自动诊断。
- 诊断函数依赖 `HAL_GetTick()`、`last_tick` 和 `step`，必须周期性调用才会形成 1000ms 一步的点测序列。
- `HAL_GetTick()` 返回的是 MCU 运行时推进的 `uwTick`，调试器暂停时的主机等待时间不能直接当作诊断门控时间。
- 当前应用源码没有看到 TIM2/TIM3/TIM4 调试冻结宏调用，断点暂停下 PWM 定时器是否继续运行需要寄存器和波形证据。
- `PWM_Motor_Init()` 当前未逐项检查 `HAL_TIM_PWM_Start()` 返回值，`isInitialized=1` 不是所有通道成功输出的充分证据。
- TIM2/TIM3/TIM4 的 PWM 初始化 `Pulse=500`，所以 `CCR=500` 可能只是初始化比较值，不是诊断 duty 或安全关闭证据。
- `HAL_TIM_PWM_ConfigChannel()` 会设置对应 `OCxPE`，所以调试诊断时应把 CCR 写入、预装载路径、更新事件和外部波形分开记录。
- `PWM_Motor_Hardware_Diagnosis()` 只有进入 1000ms 到期分支时才清零全部 CCR；未到期调用不会改变现有 CCR。
- 诊断函数和闭环控制函数可能写同一批 CCR，临时接入诊断时必须确认独占条件。
- 当前代码中 Pitch 控制分支调用 `PWM_Motor_SetAngle(MOTOR_ROLL, ...)`，
  会覆盖诊断函数实际写入的 Roll 物理输出组；只看 `rollEnabled`
  不能判断当前点测是否独占。
- 头文件注释、驱动实现注释和控制调用关系中存在轴名交叉，调试结论应回到 CCR 和引脚证据。
- `HAL_TIM_PWM_Start()` 打开通道并使能定时器，诊断时应同时观察 `CR1.CEN`、`CNT`、`CCER.CCxE` 和 `CCRn`。
- ST-LINK、STM32F10x 参考手册和 STM32 定时器应用笔记能支撑调试工具、AFIO/SWJ 和 TIM/PWM 的通用语义，但不能替代本项目 `.launch`、源码、`.map/.list`、断点、寄存器和示波器证据。
- `.map/.list/.su/.cyclo` 的构建产物结论统一回到第8.13节判断：它们能证明 Debug 产物中的最终地址符号、主线调用、HAL PWM 路径和源码级静态资源边界；`PWM_Motor_Hardware_Diagnosis()` 当前只见于 `0x00000000` 输入段，诊断证据链只支持“源码存在”和“对象输入段”两层，不能写成当前最终固件主线可达，也不能替代板上断点和波形证据。
- PA3 当前应按 TIM2_CH4 复用点测；`LED2` 宏名未见主线使用，不能替代实际复用证据。
- 理论 PWM 频率为 20kHz，诊断 duty 的理论占空比约 27.8%；真实波形仍需示波器验证。
- 诊断函数通过只能证明底层部分 PWM 输出链路可响应，不能证明姿态解算、PID 或电角换算正确。

本章边界：

- 本章证明工程下载调试配置和底层点测工具存在，不证明当前桌面环境已经连接目标板。
- 本章证明调试配置指向 ELF 和 Flash 校验流程，不证明板上此刻运行最新源码。
- 本章证明部分 PWM 初始化和 HAL PWM 符号进入 Debug 构建，不证明 `PWM_Motor_Hardware_Diagnosis()` 已经进入主线调用路径。
- 本章使用 `.su/.cyclo` 只能补充源码级静态资源边界，不能把被链接器丢弃或主线未调用的诊断函数写成已执行功能。
- 点测成功只能证明被测链路可响应，不能替代闭环控制、姿态方向和调参验证。
- 轴使能关闭、诊断函数未调用、诊断函数未到期和 CCR 清零是不同证据层级。
- CPU 暂停、HAL tick 到期、定时器继续计数和外部 PWM 继续输出是不同证据层级。
- 物理轴方向、电机相序、驱动电流、温升和稳定性都属于仓库外实测证据。
