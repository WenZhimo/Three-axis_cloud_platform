## 3. 问题背景

姿态解算不只有一种实现。第23章讲的四元数方案适合三维姿态连续更新，但实现复杂度更高。

为什么还要讲一个当前没有进入主线的备用方案？原因是，备用算法能帮助读者看清姿态估计的另一种工程取舍。

互补滤波方案更直观：陀螺仪短时间响应快，但会漂移；加速度计能提供重力方向，但受线性加速度干扰。把两者按不同时间尺度组合，就能得到一个简单姿态估计。

项目中保留 `evvgcCF.c` 的价值在于：

- 它给出一个更容易理解的姿态解算备选思路。
- 它可以作为调试时对照 `MargAHRS.c` 的参考。
- 它为后续姿态算法替换或降级提供结构入口。

但当前项目运行证据也很明确：

- `Debug/objects.list` 中有 `evvgcCF.o` 条目，说明该文件参与构建。
- `Core/Inc/main.h` 包含了 `evvgcCF.h`，说明接口对工程可见。
- `Core/Src/main.c` 当前 500Hz 主循环调用 `MargAHRSupdate()`。
- 当前没有发现 `Core/Src/main.c` 调用 `getOrientation()`。
- `computeMotorCommands.c` 读取的是 `sensors.margAttitude500Hz`，不是 `sensors.evvgcCFAttitude500Hz`。

所以本章的关键词是“备用方案”，不是“当前主线”。
对象文件存在和头文件被包含都只能证明“能被编译或被引用”，不能证明函数在运行时执行。
当前是否进入控制闭环，必须看 `main.c` 调用点和控制侧姿态数组读取。

备用方案还有一个容易被忽略的风险：它不是只缺一个调用点。
它内部对加速度 Z 轴符号、陀螺仪单位和 Roll/Yaw 更新状态都有自己的假设。
未来若要恢复它，必须先校准这些输入约定，而不是把当前主线的 `sensors.accel500Hz` 和 `sensors.gyro500Hz` 直接接进去。

## 4. 核心概念

### 4.1 互补滤波

`互补滤波` 是把两个传感器的优势组合起来：

- 陀螺仪积分：短时间响应快，适合跟踪快速变化，但长期会漂移。
- 加速度角：长期能提供重力方向参考，但短时间容易被震动和线性加速度影响。

在 `evvgcCF.c` 中，核心形式是：

```text
orient[PITCH] = orient[PITCH] + gyroRate[PITCH] * dt
              + 0.0002f * (smoothAcc[PITCH] - orient[PITCH])
```

前半部分是陀螺仪积分，后半部分是加速度角对当前姿态的缓慢拉回。

如果把 `0.0002f` 记为 `k`，并先忽略陀螺项，公式可以改写为：

```text
orient_new = (1 - k) * orient_old + k * smoothAcc
```

也就是说，姿态误差每帧按 `1 - k` 的比例衰减。当前 `k = 0.0002`，如果未来在
500Hz 循环中调用该函数，误差衰减因子就是 `0.9998`。它不是快速“贴合”加速度角，
而是非常缓慢地把陀螺积分结果拉回长期参考。

这里的“每帧”很关键。`0.0002f` 没有乘以 `dt`，因此它不是按秒定义的连续时间增益，
而是一次函数调用对应一次拉回。若调用频率改变，等效时间常数也会改变。以同一个 `k` 为例：

```text
tau = -Ts / ln(1 - k)
```

500Hz 时 `Ts = 0.002s`，`tau ~= 10s`；100Hz 时 `Ts = 0.01s`，`tau ~= 50s`。
所以恢复备用方案时，不能只搬运 `0.0002f`，还必须同时记录调用频率。

经典互补滤波也常写成：

```text
angle = alpha * (angle + gyro * dt) + (1 - alpha) * accel_angle
```

本项目备用代码的形式更接近：

```text
angle = angle + gyro * dt + k * (smoothAcc - angle)
```

二者思想一致，都是短期用陀螺仪、长期用加速度角；但代码形式并不完全相同。
教材分析时应按项目源码展开，而不是把标准公式直接套到项目上。

#### 4.1.1 理论来源与项目实现边界

Mahony、Hamel 和 Pflimlin 在 IEEE Transactions on Automatic Control 2008 年论文
`Nonlinear Complementary Filters on the Special Orthogonal Group` 中，把互补滤波放在
SO(3) 姿态观测器框架下讨论，研究对象包括低成本 IMU 的噪声和陀螺零偏问题。
这能说明“用陀螺仪短期积分、用重力/方向观测长期校正姿态”不是单纯经验拼接，
而是可以被写成姿态观测器问题。

但这条理论依据不能直接证明本项目备用文件已经具备同等性质。第24章必须把证据分层：

| 证据层级 | 能证明什么 | 不能证明什么 |
|---|---|---|
| 权威理论层 | 互补滤波可以作为低成本 IMU 姿态估计的一类严肃方法，SO(3) 框架下还能讨论收敛性和陀螺零偏估计 | 不能证明 `evvgcCF.c` 的简化公式、三轴单位和符号约定已经满足论文中的观测器条件 |
| 项目源码层 | `getOrientation()` 体现了陀螺积分加加速度角缓慢拉回的互补滤波思想 | 不能证明 Roll/Yaw 已经具备完整长期参考，也不能证明输入单位与当前 `sensors.gyro500Hz` 一致 |
| 构建产物层 | `.map/.list/.su/.cyclo` 可以审查对象文件、段裁剪、静态栈用量和圈复杂度 | 不能证明被裁剪函数会在目标板运行，也不能替代 500Hz 实测耗时 |
| 运行验证层 | 恢复调用后，调试器、DWT 计时和运动测试才能证明实际输入、姿态输出方向和实时预算【待验证】 | 不能由论文标题、对象文件存在或头文件包含反推当前闭环已经使用互补滤波 |

因此，本章引用互补滤波理论时只用来解释设计思想和审查维度；
项目结论仍以 `evvgcCF.c`、`main.c`、`computeMotorCommands.c` 和 Debug 构建产物为准。

### 4.2 加速度角

加速度角是由加速度计测得的重力方向计算出的姿态角。项目中 Pitch 加速度角计算为：

```c
accAngle[PITCH] = atan2f(accData[XAXIS], -accData[ZAXIS]);
```

这个角不是控制输出，而是互补滤波中的长期参考。

### 4.3 平滑加速度角

`smoothAcc[PITCH]` 使用如下形式更新：

```text
smoothAcc[PITCH] = (smoothAcc[PITCH] * 99 + accAngle[PITCH]) / 100
```

这是一种简单低通形式。新角度只占 `1/100`，历史值占 `99/100`，因此它变化很慢，适合作为长期方向参考。

把它写成标准一阶 IIR 形式：

```text
smoothAcc[n] = 0.99 * smoothAcc[n-1] + 0.01 * accAngle[n]
```

其中新输入权重 `alpha = 0.01`。若假设未来以 500Hz 调用 `getOrientation()`，
采样周期 `Ts = 0.002s`，该低通的离散极点是 `0.99`，等效时间常数近似为：

```text
tau = -Ts / ln(0.99)
    = -0.002 / ln(0.99)
    ~= 0.199s
```

这个推导只说明代码参数对应的滤波尺度；当前 `getOrientation()` 没进入主线，
所以不能把它写成当前姿态输出已经具有约 `0.2s` 加速度角平滑时间常数。

### 4.4 姿态输出容器

`Drivers/BGC/bgc32.h` 中的 `sensors_t` 同时定义了：

```c
float evvgcCFAttitude500Hz[3];
float margAttitude500Hz[3];
```

前者对应 `evvgcCF.c` 这套互补滤波备用方案，后者对应当前主线 `MargAHRS.c` 输出。当前控制链路读取 `margAttitude500Hz`。

#### 4.4.1 字段存在与数据流消费边界

这里要再拆开一个容易混淆的证据边界：结构体字段存在，不等于该字段已经形成当前闭环数据流。

`Debug/Three-axis_cloud_platformV2.map` 中可以看到 `.bss.sensors`，说明 `Core/Src/main.c` 定义的全局 `sensors` 对象在当前 Debug 镜像中拥有 RAM 地址。可是 `.map` 记录的是整个全局对象的段分配，不能细分证明 `sensors_t` 内部每一个字段都被写入、被读取或被控制算法消费。因此：

| 观察对象 | 能证明什么 | 不能证明什么 |
|---|---|---|
| `bgc32.h` 中的 `evvgcCFAttitude500Hz[3]` 字段 | 数据结构预留了互补滤波姿态输出容器 | 不能证明当前 500Hz 主线写入了该字段 |
| `.map` 中的 `.bss.sensors` | 全局 `sensors` 对象进入最终 RAM 布局 | 不能证明 `evvgcCFAttitude500Hz` 这个子字段有运行期数据流 |
| `MargAHRS.c` 写 `sensors.margAttitude500Hz[...]` | 当前 AHRS 生产者写入主线姿态字段 | 不能证明备用字段同步更新 |
| `computeMotorCommands.c` 读 `sensors.margAttitude500Hz[...]` | 当前控制消费者读取主线姿态字段 | 不能证明控制闭环消费 `evvgcCFAttitude500Hz` |

换句话说，`sensors_t` 同时包含两个姿态数组，只能说明项目的数据结构为两条姿态路径都留了位置。当前数据流证据是：

```text
MPU6050_Read_And_Process()
-> sensors.accel500Hz / sensors.gyro500Hz
-> MargAHRSupdate()
-> sensors.margAttitude500Hz
-> computeMotorCommands()
```

备用互补滤波字段要进入闭环，至少还需要看到两类证据：第一，`initOrientation()` 或 `getOrientation(..., sensors.evvgcCFAttitude500Hz, ...)` 这样的写入路径拥有最终镜像地址并在 `.list` 中可达；第二，控制侧明确读取 `sensors.evvgcCFAttitude500Hz`，或者有受控的同步代码把备用输出转交给当前控制实际读取的字段。当前构建中 `initOrientation()` 和 `getOrientation()` 位于 `.map` 的丢弃段，所以不能把 `evvgcCFAttitude500Hz` 解释成已经参与当前闭环的姿态量。

### 4.5 输入约定边界

备用算法的公式不仅依赖变量名，还依赖输入单位和坐标符号。

当前 `getOrientation()` 对 Pitch 加速度角使用：

```c
accAngle[PITCH] = atan2f(accData[XAXIS], -accData[ZAXIS]);
```

它等价于假设传入的 `accData[ZAXIS]` 与公式中的重力方向约定匹配。可是 `initOrientation()` 内部自己构造 `sensors.accel500Hz[ZAXIS]` 时写成：

```c
sensors.accel500Hz[ZAXIS] =
    -((float)rawAccel[ZAXIS].value - accelTCBias[ZAXIS]) * ACCEL_SCALE_FACTOR;
```

而当前 `Core/Src/main.c` 的 500Hz 主线在第21章已经记录为不再对 Z 轴额外取反。二者不一定矛盾，因为它们属于不同路径；但如果未来直接复用当前主线传感器数组调用 `getOrientation()`，就必须重新验证 Pitch 正负号和零位。

## 5. 工作原理

从算法角度看，`evvgcCF.c` 可以分成初始化和运行更新两部分。

初始化阶段由 `initOrientation()` 表示。它循环读取 150 次 MPU6050 数据，计算 Roll/Pitch 的加速度角平均值，然后写入：

```c
sensors.evvgcCFAttitude500Hz[PITCH]
sensors.evvgcCFAttitude500Hz[ROLL]
sensors.evvgcCFAttitude500Hz[YAW]
```

但当前 `Drivers/SRC/Inc/evvgcCF.h` 中 `initOrientation()` 的声明被注释，`Core/Src/main.c` 也没有调用它。因此这段初始化逻辑只能作为备用方案分析，不能写成当前开机主线。

这还带来一个接口层面的恢复顺序：`getOrientation()` 在头文件中公开声明，`initOrientation()` 只在
`.c` 文件里有定义而没有对外声明。当前 `Debug/Drivers/SRC/Src/subdir.mk` 的编译命令使用
`-std=gnu11` 和 `-Wall`，如果未来在其他文件里直接调用一个没有可见原型的函数，至少会触发编译诊断。
因此恢复备用方案不应从“直接在 `main.c` 写一行调用”开始，而应先恢复头文件接口、再确定调用时序和输入输出数组。

运行阶段由 `getOrientation()` 表示。它的输入包括：

- `smoothAcc`：加速度角平滑状态。
- `orient`：输出姿态角数组。
- `accData`：加速度输入。
- `gyroData`：陀螺仪输入。
- `dt`：积分时间。

运行机制是：

1. 用加速度计算 Pitch 加速度角。
2. 对 Pitch 加速度角做低通平滑。
3. 把 Pitch 轴陀螺仪数据换算成弧度角速度。
4. 用陀螺仪积分更新姿态。
5. 用加速度角和当前姿态的差值做小比例修正。

这就是互补滤波的核心：短期跟随陀螺仪，长期靠加速度修正。

这里还缺一个当前源码没有替我们完成的动作：输入适配。
`getOrientation()` 接受裸指针 `accData` 和 `gyroData`。
它不会自己判断这些数组来自 raw LSB、deg/s、rad/s，还是已经按当前主线符号处理过的 `sensors`。
恢复备用方案前必须在调用点明确单位和符号。

## 6. STM32实现机制

本章同样没有新增 STM32 外设。互补滤波发生在 CPU 软件层，主要使用：

- `atan2f()` 计算加速度角。
- `cosf()`、`sinf()` 处理 Roll/Yaw 与 Pitch 的耦合关系。
- 普通浮点乘加完成低通和平衡修正。
- `HAL_Delay(2)` 用于 `initOrientation()` 中的采样间隔。

STM32F103RCTx 没有硬件 FPU，因此这些浮点运算仍由软件浮点完成。和第23章的四元数方案相比，互补滤波代码更直观，但不能因此假定它一定更适合当前项目。当前项目已经选择 `MargAHRSupdate()` 作为 500Hz 主线。

还要注意一个工程边界：`evvgcCF.c` 参与构建不等于参与运行。`Debug/objects.list` 能确认 `evvgcCF.o` 进入构建产物列表，但调用关系必须以 `Core/Src/main.c` 和全工程调用搜索为准。

如果未来恢复 `initOrientation()`，还会带来一个启动时序成本。代码中循环 150 次，每次
`HAL_Delay(2)`，仅延时下界就是：

```text
150 * 2ms = 300ms
```

再加上每次 `MPU6050_Read_And_Process()`、`MPU6050_ComputeTemError()` 和三角函数计算，
实际耗时会更长。当前主线没有调用它，因此不能把这 300ms 写成当前开机耗时；它只属于
备用方案恢复前必须评估的启动代价。

编译产物还能提供另一个静态证据：当前 `Debug/Drivers/SRC/Src/evvgcCF.su` 记录
`initOrientation` 的静态栈用量为 40 字节，`getOrientation` 为 56 字节；
`Debug/Drivers/SRC/Src/evvgcCF.cyclo` 记录二者圈复杂度分别为 2 和 1。这个结果只能说明函数本体
看起来不复杂、局部栈帧不大，不能替代运行耗时测量。三角函数的 `libm` 调用、I2C 读取和
`HAL_Delay(2)` 的时间成本都不在“函数本体栈用量小”这个结论里。

若未来把 `getOrientation()` 放进 500Hz 主循环，它每帧至少包含 `atan2f()`、`cosf()`、
`sinf()` 和多次软件浮点乘加。STM32F103 没有硬件 FPU，是否满足实时预算需要用
`executionTime500Hz` 或 DWT 记录验证，不能仅凭函数看起来短就断言性能足够。

## 7. 项目中的应用

## 8. 代码分析

### 8.1 accAngleSmooth状态

`evvgcCF.c` 定义：

```c
float accAngleSmooth[3];
```

它用于保存平滑后的加速度角。这个状态和第22章的 `firstOrderFilters[]` 不同；它不是通用一阶滤波器结构，而是互补滤波文件内部的数组状态。

### 8.2 initOrientation初始化

`initOrientation()` 内部循环 150 次，每次读取 MPU6050 并计算加速度角。最后把平均值写入 `sensors.evvgcCFAttitude500Hz`。

当前边界有三点：

- `evvgcCF.h` 中 `initOrientation()` 声明被注释。
- `Core/Src/main.c` 当前没有调用 `initOrientation()`。
- 代码注释写“仅初始化 PITCH”，但实际也给 Roll 平均值赋值，并把 Yaw 置为 0。

因此这段代码可以解释备用方案的初始化思路，但不能作为当前开机姿态初始化流程。
更严格地说，它目前不是一个对外完整暴露的初始化 API：函数定义存在，头文件原型被注释，
调用点缺失。恢复时应同时审查“能否编译调用”和“调用后算法是否可信”两层问题。

### 8.3 Pitch加速度角

`getOrientation()` 中当前启用的加速度角计算是：

```c
accAngle[PITCH] = atan2f(accData[XAXIS], -accData[ZAXIS]);
```

Roll 加速度角计算在该函数内被注释：

```c
// accAngle[ROLL] = atan2f(...)
```

这意味着 Pitch 有新的加速度角参考，Roll 在运行阶段没有更新 `smoothAcc[ROLL]` 的加速度角来源。

### 8.4 Pitch互补滤波公式

Pitch 平滑加速度角为：

```c
smoothAcc[PITCH] = ((smoothAcc[PITCH] * 99.0f) + accAngle[PITCH]) / 100.0f;
```

Pitch 姿态更新为：

```c
orient[PITCH] = (orient[PITCH] + gyroRate[PITCH] * dt)
              + 0.0002f * (smoothAcc[PITCH] - orient[PITCH]);
```

可以拆成两部分：

- `orient[PITCH] + gyroRate[PITCH] * dt`：陀螺仪短期积分。
- `0.0002f * (smoothAcc[PITCH] - orient[PITCH])`：加速度角长期修正。

`0.0002f` 是非常小的修正系数，表示每帧只轻微拉向加速度角。

如果设 `k = 0.0002`，并假设 `smoothAcc[PITCH]` 不变、`gyroRate[PITCH] = 0`，
则误差满足：

```text
error[n] = smoothAcc - orient[n]
error[n+1] = (1 - k) * error[n]
```

若未来按 500Hz 调用，`Ts = 0.002s`，等效误差衰减时间常数为：

```text
tau_c = -Ts / ln(1 - k)
      = -0.002 / ln(0.9998)
      ~= 10s
```

这说明 Pitch 姿态会很慢地向平滑加速度角靠拢。这样的好处是线性加速度和震动不容易立刻
污染姿态；代价是陀螺零偏导致的慢漂移也会被很慢地修正。该结论仍是“若接入 500Hz 主线”
的参数分析，不是当前运行表现。

还要注意代码形式和常见教材公式有一个细微差异。常见写法：

```text
angle_new = alpha * (angle_old + gyro * dt) + (1 - alpha) * acc
```

会同时缩放上一帧角度和本帧陀螺积分项。而当前备用代码等价于：

```text
orient_new = (1 - k) * orient_old + gyroRate * dt + k * smoothAcc
```

陀螺积分项 `gyroRate * dt` 没有被 `(1 - k)` 缩放。由于 `k = 0.0002` 很小，二者数值差异通常很小；
但教材不能把它们写成完全相同的公式。恢复算法或重写参数时，应按当前源码形式重新推导。

还要注意，`gyroRate[PITCH]` 的单位假设非常关键：

```c
gyroRate[PITCH] = gyroData[PITCH] * 0.0174532925f;
```

这个常数是 `pi / 180`，表示从 `deg/s` 转为 `rad/s`。如果 `gyroData[PITCH]` 是
`57.2958 deg/s`，转换结果约为 `1 rad/s`，这是合理的。可是第21章已经说明当前主线
`sensors.gyro500Hz` 是乘过 `GYRO_SCALE_FACTOR` 的 `rad/s`。如果未来直接把
`sensors.gyro500Hz[PITCH] = 1 rad/s` 传入这里，代码会得到：

```text
gyroRate[PITCH] = 1 * 0.0174532925 = 0.01745 rad/s
```

角速度会被缩小约 `57.3` 倍。恢复备用方案前必须先决定 `gyroData` 的输入单位，
再决定是否保留这次角度到弧度转换。

还有一个输入有效性边界。`getOrientation()` 直接使用：

```c
accAngle[PITCH] = atan2f(accData[XAXIS], -accData[ZAXIS]);
orient[PITCH] = orient[PITCH] + gyroRate[PITCH] * dt + ...
```

函数内部没有检查 `accData`、`gyroData`、`orient`、`smoothAcc` 或 `dt` 是否为 `NaN/Inf`，也没有像第22章 `MargAHRSupdate()` 那样用加速度模长阈值判断重力观测是否可用。因此，若未来恢复备用方案，调用点必须先保证输入数组已经完成读取有效性检查、单位转换、符号约定和时间步长保护；否则 `atan2f()` 与 `gyroRate * dt` 仍会照常计算，但结果不能被解释为可信姿态。

### 8.4.1 加速度Z轴符号边界

`initOrientation()` 内部构造备用姿态初始化输入时，Z 轴加速度被额外取反：

```c
sensors.accel500Hz[ZAXIS] =
    -((float)rawAccel[ZAXIS].value - accelTCBias[ZAXIS]) * ACCEL_SCALE_FACTOR;
```

但当前主线 `Core/Src/main.c` 中的第21章缩放路径是：

```c
sensors.accel500Hz[ZAXIS] =
    ((float)rawAccel[ZAXIS].value - accelTCBias[ZAXIS]) * ACCEL_SCALE_FACTOR;
```

`getOrientation()` 又用 `-accData[ZAXIS]` 参与 Pitch 加速度角计算：

```c
accAngle[PITCH] = atan2f(accData[XAXIS], -accData[ZAXIS]);
```

因此，备用方案恢复前必须把三件事放在同一张表里验证：

| 项目 | 当前主线 | 备用初始化 | 备用运行 |
| --- | --- | --- | --- |
| Z 轴缩放 | 不额外取反 | 额外取反 | 由调用者传入 |
| Pitch 公式 | `MargAHRSupdate()` 内部四元数输出 | 初始化平均加速度角 | `atan2f(X, -Z)` |
| 当前是否运行 | 是 | 否 | 否 |

这不是说备用方案一定错误，而是说明它和当前主线使用的坐标符号约定没有自动对齐。没有静止姿态、手动俯仰动作和第20章方向矩阵回查之前，不能断言备用 Pitch 输出方向正确。

### 8.5 Roll/Yaw代码与注释不一致

`getOrientation()` 的注释说 ROLL/YAW 代码全部注释，但实际末尾仍有 Roll/Yaw 更新语句：

```c
gyroRate[ROLL] = ...
orient[ROLL] = ...
gyroRate[YAW] = ...
orient[YAW] = ...
```

同时，`smoothAcc[ROLL]` 的更新在 `getOrientation()` 中被注释掉。也就是说，Roll/Yaw 代码不是完全删除状态，而是处在不完整或半保留状态。

这里必须标记为【待验证】：如果未来要恢复 `evvgcCF.c`，需要重新确认 Roll/Yaw 的加速度角来源、符号约定和运行稳定性。

更具体地说，`orient[ROLL]` 的修正项仍然使用：

```c
0.0002f * (smoothAcc[ROLL] - orient[ROLL])
```

但 `getOrientation()` 当前没有更新 `smoothAcc[ROLL]`。如果没有先调用 `initOrientation()`，
全局 `accAngleSmooth[ROLL]` 只会保持 C 静态存储期的初始零值；如果调用过 `initOrientation()`，
它也只是开机平均得到的初值，运行阶段不会随新的 Roll 加速度角更新。因此 Roll 分支不是
“运行中的 Roll 互补滤波”，更像“陀螺积分加一个朝固定 Roll 参考缓慢拉回的半保留逻辑”。
Yaw 分支则完全没有加速度或磁力计长期参考，只是根据耦合后的角速度继续积分，长期漂移无法被当前代码约束。

### 8.5.1 Roll/Yaw单位边界

Roll/Yaw 还有一个比“注释不一致”更具体的单位边界。Pitch 轴写成：

```c
gyroRate[PITCH] = gyroData[PITCH] * 0.0174532925f;
```

这说明 Pitch 路径假设输入 `gyroData[PITCH]` 是 `deg/s`，然后转为 `rad/s`。

但 Roll/Yaw 保留代码是：

```c
gyroRate[ROLL] = gyroData[ROLL] * cosf(fabsf(orient[PITCH]))
               + gyroData[YAW] * sinf(orient[PITCH]);
gyroRate[YAW]  = gyroData[YAW] * cosf(fabsf(orient[PITCH]))
               - gyroData[ROLL] * sinf(orient[PITCH]);
```

这里没有乘以 `0.0174532925f`。如果 `gyroData[ROLL/YAW]` 也是 `deg/s`，则 Roll/Yaw
积分会把角度每秒直接当作弧度每秒使用；如果传入的是 `rad/s`，Pitch 又会被重复缩小。
因此该备用函数目前没有一个对三轴同时自洽的单位入口。恢复它之前，必须先统一接口约定：
要么调用点传入度每秒并补齐 Roll/Yaw 转换，要么传入弧度每秒并移除 Pitch 的二次转换。
具体采取哪种方案属于源码修改和算法恢复任务，本章只记录审查边界。

### 8.6 当前主线未调用getOrientation

当前全工程搜索可以看到：

- `evvgcCF.c` 定义 `getOrientation()`。
- `evvgcCF.h` 声明 `getOrientation()`。
- `Core/Src/main.c` 没有调用 `getOrientation()`。
- `Core/Src/main.c` 调用 `MargAHRSupdate()`。

因此不能用 `getOrientation()` 的输出解释当前电机控制表现。当前电机控制读取的是 `sensors.margAttitude500Hz`。

### 8.7 构建产物证据边界

`Debug/objects.list` 中存在：

```text
evvgcCF.o
```

这只能证明源文件被编译为对象文件，并被纳入链接输入列表。`Debug/Three-axis_cloud_platformV2.map` 中的 `LOAD ./Drivers/SRC/Src/evvgcCF.o` 进一步说明链接器读取了这个对象文件。

但当前构建同时具备两类裁剪条件：

- `Debug/Drivers/SRC/Src/subdir.mk` 的编译命令包含 `-ffunction-sections -fdata-sections`，函数和数据会被放入可独立处理的段。
- `Debug/makefile` 的链接命令包含 `-Wl,--gc-sections`，未被引用的输入段可以在链接阶段被丢弃。

因此第24章这里需要把证据分成四层：

| 证据层级 | 当前证据 | 能证明什么 | 不能证明什么 |
|---|---|---|---|
| 编译对象存在 | `Debug/objects.list` 中有 `evvgcCF.o` | `evvgcCF.c` 被编译成对象文件 | 不能证明函数保留在最终镜像 |
| 链接输入加载 | `.map` 中有 `LOAD ./Drivers/SRC/Src/evvgcCF.o` | 链接器读取了该对象文件 | 不能证明对象内所有段都有运行地址 |
| 段裁剪结果 | `.map` 的 `Discarded input sections` 下有 `.bss.accAngleSmooth`、`.text.initOrientation`、`.text.getOrientation`，地址均为 `0x00000000` | 当前 Debug 构建中这些状态和函数段被裁剪，没有最终 Flash/RAM 运行地址 | 不能证明它们会在运行时执行 |
| 运行调用状态 | `Core/Src/main.c` 调用 `MargAHRSupdate()`，没有调用 `getOrientation()` | 当前 500Hz 姿态主线不是 `evvgcCF.c` | 不能证明备用方案恢复后的效果 |

所以更准确的结论是：`evvgcCF.c` 当前被编译为对象文件，也作为链接输入被读取；但 `accAngleSmooth`、`initOrientation()` 和 `getOrientation()` 在当前 Debug 构建中位于 `.map` 的丢弃段，没有最终镜像地址。不能把它们写成“函数本体已进入固件镜像”，更不能写成“当前运行主线已经调用互补滤波”。

当前 `Debug/Three-axis_cloud_platformV2.list` 检索不到 `initOrientation()` 或 `getOrientation()` 的反汇编入口，也没有看到 `main()` 或 500Hz 分支调用它们。这个结果只能作为与 `.map` 裁剪结果一致的旁证：它说明当前 `.list` 不能提供备用互补滤波的可达调用路径证据。最终判断仍应以 `.map` 的丢弃段、源码调用搜索和控制侧姿态数组读取共同约束，不能只凭 `.list` 中“没看到”就单独下结论。

`Debug/Drivers/SRC/Src/evvgcCF.su` 和 `Debug/Drivers/SRC/Src/evvgcCF.cyclo` 还要放在更早的一层理解：它们是编译阶段为 `evvgcCF.c` 生成的函数级静态报告，能说明 `initOrientation()`、`getOrientation()` 在该编译单元中的栈帧估算和圈复杂度；但在 `--gc-sections` 之后，即使这些报告仍然存在，也不能反推对应 `.text.*` 段已经拥有最终 Flash 地址。也就是说，`.su/.cyclo` 能提示“恢复备用方案后哪些函数值得测栈和分支”，不能证明当前固件已经保留或执行这些函数。

如果未来恢复 `getOrientation()` 或 `initOrientation()`，必须重新构建并检查 `.map`：相关 `.text.*` 段应从 `Discarded input sections` 移到带有真实 Flash 地址的输出段，相关 `.bss.*` 状态也应拥有实际 RAM 地址。同时还要在 `.list` 或反汇编中确认调用点，而不是只看 `objects.list`。

### 8.8 备用方案恢复的最小证据闭环

如果未来要把 `evvgcCF.c` 从“备用方案”恢复成“当前姿态来源”，不能只在 `main.c` 里补一行 `getOrientation()` 调用。至少要同时闭合四类证据：

| 闭环项 | 当前状态 | 恢复前必须补齐的证据 |
|---|---|---|
| 接口闭环 | `getOrientation()` 有头文件声明，`initOrientation()` 声明被注释 | 先恢复初始化接口声明，并确认 `-std=gnu11 -Wall` 下跨文件调用没有原型诊断 |
| 数据流闭环 | 控制链路读取 `sensors.margAttitude500Hz`，备用输出在 `sensors.evvgcCFAttitude500Hz` | 明确 `computeMotorCommands()` 是否切换输入数组，或者在 AHRS 后把备用输出同步到控制实际读取的字段 |
| 构建闭环 | `.text.getOrientation`、`.text.initOrientation`、`.bss.accAngleSmooth` 当前位于丢弃段 | 重新构建后确认这些段拥有最终 Flash/RAM 地址，并在 `.list` 中看到可达调用路径 |
| 行为闭环 | 单位、符号、初始化、`dt`、输入有效性和 Roll/Yaw 长期参考仍有待验证项 | 用静止姿态、单轴手动运动、DWT 计时和串口日志证明输出方向、幅值、实时预算和控制响应 |

这个闭环也解释了为什么本章不直接给出“切换到互补滤波”的代码建议。Mahony 等人的互补滤波论文可以证明互补滤波是严肃的姿态估计思想，Oppenheim 与 Schafer 的离散时间信号处理教材也能支撑一阶 IIR 平滑的理论背景；但本仓库里的 `evvgcCF.c` 是一个被裁剪的备用实现，是否能替代当前 `MargAHRSupdate()`，必须由项目级证据重新证明。

### 8.8.1 备用姿态源切换判定表

把备用方案写成“当前姿态源”之前，可以用下面这张最小判定表逐项核对。它的重点不是重复说明互补滤波公式，而是防止把“候选代码存在”误判成“运行数据流已经切换”。

| 判定项 | 当前证据 | 当前结论 |
|---|---|---|
| 源码候选 | `Drivers/SRC/Src/evvgcCF.c` 定义 `initOrientation()`、`getOrientation()`，`Drivers/SRC/Inc/evvgcCF.h` 声明 `getOrientation()` | 备用姿态算法源码存在，接口部分可见 |
| 构建输入 | `Debug/objects.list` 有 `evvgcCF.o`，`.map` 有 `LOAD ./Drivers/SRC/Src/evvgcCF.o` | 对象文件参与构建并被链接器读取 |
| 最终保留 | `.map` 中 `.bss.accAngleSmooth`、`.text.initOrientation`、`.text.getOrientation` 位于丢弃段，地址为 `0x00000000` | 当前最终镜像没有为这些备用状态和函数分配运行地址 |
| 调用路径 | `Core/Src/main.c` 调用 `MargAHRSupdate()`，当前未发现 `getOrientation()` 可达调用路径；`.list` 中有 `MargAHRSupdate` 入口和调用 | 当前 500Hz 姿态更新路径仍是 `MargAHRSupdate()` |
| 输出生产者 | `MargAHRS.c` 写入 `sensors.margAttitude500Hz[...]`；`evvgcCF.c` 中写 `sensors.evvgcCFAttitude500Hz[...]` 的初始化段未进入当前主线 | 当前可证明的姿态生产者是主线 AHRS，不是备用互补滤波 |
| 控制消费者 | `computeMotorCommands.c` 读取 `sensors.margAttitude500Hz[...]` | 当前控制闭环消费的是主线姿态数组 |

因此，第24章对 `evvgcCF.c` 的结论只能写成：备用互补滤波源码和构建输入存在，但当前构建、调用路径、输出生产者和控制消费者都不能证明它已经成为姿态源。若未来要切换，必须让上述表格中“最终保留、调用路径、输出生产者、控制消费者”四项同时改变，并保留新的 `.map/.list` 与运行验证记录。

### 8.8.2 控制消费者证据边界

姿态算法是否进入闭环，不能只看“谁能产生姿态”，还必须看“控制器实际消费谁”。当前工程里这条链路可以分成三层：

| 层级 | 当前证据 | 能证明什么 | 不能证明什么 |
|---|---|---|---|
| 姿态生产者 | `Drivers/SRC/Src/MargAHRS.c` 写入 `sensors.margAttitude500Hz[ROLL/PITCH/YAW]` | 当前主线 AHRS 会把四元数结果转换到主线姿态数组 | 不能证明备用 `evvgcCFAttitude500Hz` 同步得到相同姿态 |
| 控制消费者源码 | `Drivers/SRC/Src/computeMotorCommands.c` 在 Roll、Pitch、Yaw 分支读取 `sensors.margAttitude500Hz[...]` | 控制器的机械角反馈来自主线姿态数组 | 不能仅凭 `sensors_t` 中存在备用字段，就说控制器会自动读取备用姿态 |
| 构建产物旁证 | `.map` 中 `computeMotorCommands` 有最终 Flash 地址，`.list` 中能看到 `computeMotorCommands(dt500Hz)` 调用以及对 `sensors.margAttitude500Hz[...]` 的源码-反汇编对应行 | 当前 Debug 镜像保留了控制消费者函数，且 `.list` 与源码读字段一致 | 不能替代运行时断点、串口日志或运动测试；也不能证明未来切换后的闭环稳定性 |

这层边界的工程意义很直接：备用互补滤波要成为“当前姿态源”，必须同时切换姿态生产者和控制消费者。
任一端仍停留在当前主线数组，闭环输入都不能算完成切换；完成源码修改后还要用新的 `.map/.list`
和实际运动测试验证方向、幅值和稳定性【待验证】。

## 9. 调试方法

调试备用互补滤波时，按统一调试结构组织：现象 -> 可能原因 -> 定位方法 -> 验证步骤 -> 解决方案 -> 经验总结。

### 9.1 现象与可能原因

常见现象包括：`evvgcCF.c` 出现在构建输入里却没有运行地址、`getOrientation()` 有源码但主线没有调用、备用姿态数组存在却没有被控制器消费、Pitch 公式看起来可用但 Roll/Yaw 状态不完整。可能原因是把“对象文件参与构建”“函数段被最终保留”“函数被主线调用”“输出被控制器读取”混成同一层证据。

### 9.2 定位方法：构建、调用与控制输入

第一步，确认构建状态。

- 查看 `Debug/objects.list` 是否包含 `evvgcCF.o` 条目。
- 如果包含，只能说明 `evvgcCF.c` 被编译为对象文件。
- 查看 `Debug/Three-axis_cloud_platformV2.map` 是否有 `LOAD ./Drivers/SRC/Src/evvgcCF.o`。
- 如果有，只能说明该对象文件被链接器读入。
- 查看 `.map` 的 `Discarded input sections`，确认 `.bss.accAngleSmooth`、`.text.initOrientation`、`.text.getOrientation` 是否仍在 `0x00000000`。
- 如果这些段仍在丢弃区，说明当前最终镜像没有为它们分配 Flash/RAM 运行地址。
- 查看 `Core/Inc/main.h` 是否包含 `evvgcCF.h`。
- 头文件包含只能说明声明可见，仍不能证明函数被调用。

第二步，确认调用状态。

- 搜索 `getOrientation(`。
- 如果只有定义和声明，没有 `Core/Src/main.c` 调用，就说明它没有进入当前主线。

第三步，确认当前控制输入。

- 在 `computeMotorCommands.c` 中观察姿态读取位置。
- 当前 Roll/Pitch/Yaw 控制读取的是 `sensors.margAttitude500Hz`。
- 如果要切换到互补滤波输出，必须明确改为 `sensors.evvgcCFAttitude500Hz`，并重新验证所有轴符号。

### 9.3 验证步骤：公式、单位与时间尺度

第四步，单独审查互补滤波公式。

- 在 `getOrientation()` 内观察 `accAngle[PITCH]`、`smoothAcc[PITCH]`、`gyroRate[PITCH]` 和 `orient[PITCH]`。
- 只在明确调用该函数后，才把这些变量作为运行数据解释。

第五步，手算时间尺度。

- 根据 `smoothAcc = 0.99 * smoothAcc + 0.01 * accAngle` 计算加速度角低通时间常数。
- 根据 `k = 0.0002` 计算姿态误差拉回时间常数。
- 如果调用频率不是 500Hz，应使用实际 `dt` 或实际调用频率重新计算。
- 记录 `0.0002f` 没有乘以 `dt`，它的等效时间常数随调用频率变化。

### 9.4 解决方案：待验证项与调试记录

第六步，记录待验证项。

- 记录 `initOrientation()` 是否应重新声明并调用。
- 记录 `evvgcCF.h` 中 `initOrientation()` 原型是否恢复；若没有恢复，不要把它写成已经公开可调用的接口。
- 记录 Roll/Yaw 代码与注释不一致的问题。
- 记录 `smoothAcc[ROLL]` 是否在运行阶段更新；若没有更新，不能把 Roll 分支写成完整互补滤波。
- 记录 Yaw 是否存在磁力计或其他长期参考；若没有参考，长期漂移保持【待验证】。
- 记录 `gyroRate[PITCH] = gyroData[PITCH] * 0.0174532925f` 是否与当前 `gyro500Hz` 单位一致。
- 记录 Roll/Yaw 保留代码是否缺少与 Pitch 一致的角度到弧度转换。
- 记录备用 `initOrientation()` 与当前主线在 Z 轴取反上的差异。
- 记录 `dt`、`accData`、`gyroData` 是否已在调用点完成有效性检查；`getOrientation()` 内部没有 `NaN/Inf` 清理和加速度模长门控。

调试记录建议：

- 记录 `evvgcCF.o` 是否出现在 `objects.list`、`.map` 是否 `LOAD` 该对象、`getOrientation()` 是否仍在丢弃段、调用搜索结果和当前控制输入使用的姿态数组。
- 记录 `Debug/Drivers/SRC/Src/evvgcCF.su` 和 `.cyclo` 中的静态栈用量、圈复杂度，并标明这不是运行耗时证据。
- 对备用算法，只记录源码公式、注释差异、单位假设和未进入主线的证据。
- 若未来恢复互补滤波，应新增调用点、初始化入口、输入单位说明和三轴符号验证记录。
- 对符号验证，应至少包含静止姿态、手动 Pitch 正向动作、第20章方向矩阵和备用公式四类证据。
- 在没有主线调用和外部运动测试前，不应把互补滤波输出写成当前项目姿态结果。

### 9.5 经验总结：调试边界

当前仓库能证明互补滤波源文件被编译为对象文件、该对象作为链接输入被读取、备用函数段在当前 Debug 构建中被裁剪，以及当前主线没有调用 `getOrientation()`。姿态输出方向、动态响应、噪声表现和替代 AHRS 的效果需要恢复调用后的运动测试证据；缺少证据时保持【待验证】。

## 10. 常见问题

问题一：`evvgcCF.c` 被编译了，为什么还说不是当前主线。

因为对象文件参与构建不等于函数段保留在最终镜像，更不等于运行调用。
当前 `.map` 把 `getOrientation()` 放在 `Discarded input sections`，`Core/Src/main.c` 调用 `MargAHRSupdate()`，没有调用 `getOrientation()`。
控制模块读取 `sensors.margAttitude500Hz`，不是 `sensors.evvgcCFAttitude500Hz`。

问题二：互补滤波是不是比四元数姿态解算更简单。

从代码直观性看更简单。
但当前 `evvgcCF.c` 的 Roll/Yaw 状态不完整，且没有进入主循环。
不能仅凭简单就认为它更适合当前项目。
是否可替换主线算法，还要验证单位、符号、状态初始化和控制响应。

问题三：`0.0002f` 是什么意思。

它是加速度角对姿态的修正系数。值越小，越相信陀螺仪积分；值越大，越快拉向加速度角，但也更容易受加速度干扰影响。

问题四：`gyroRate[PITCH] = gyroData[PITCH] * 0.0174532925f` 是否正确。

当前需要标记为【待验证】。
第21章确认当前 `sensors.gyro500Hz` 已经乘以 `GYRO_SCALE_FACTOR` 转为弧度每秒。
如果未来把 `sensors.gyro500Hz` 传给 `getOrientation()`，
这里再次乘以角度转弧度系数可能导致单位错误。
若传入的是度每秒，则这个转换才合理。

问题四补充：Roll/Yaw 的陀螺仪单位是否和 Pitch 一致。

当前不能确认一致。Pitch 显式乘以 `0.0174532925f`，Roll/Yaw 保留代码没有同样转换。
这说明备用函数恢复前不仅要检查 Pitch 单位，还要统一三轴输入单位。

问题五：为什么第24章不把 `evvgcCF.c` 接回主线。

教材目标是解释项目，而不是擅自改项目。当前证据显示主线使用 `MargAHRSupdate()`。切换姿态算法属于功能修改，需要单独设计、验证和调参。

问题六：互补滤波和第22章一阶低通滤波是什么关系。

二者都用“慢变化参考”抑制噪声或漂移，但对象不同。第22章的一阶低通滤波是通用信号平滑模块；本章的互补滤波是姿态估计策略，用陀螺仪积分和加速度角互相补偿。

问题七：能否用本章的 `0.0002f` 直接指导当前主线调参。

不能直接使用。
`0.0002f` 属于 `evvgcCF.c` 的备用互补滤波路径，
当前主线姿态更新使用 `MargAHRSupdate()` 和 `KpAcc`。
调当前主线时应回到第22章和第23章的四元数姿态解算链路，
不能把备用算法常数当成当前运行参数。

问题八：恢复备用方案时，能否直接传入当前 `sensors.accel500Hz`。

不能直接默认可用。当前主线 Z 轴缩放没有额外取反，而 `evvgcCF.c` 的 `initOrientation()` 内部对 Z 轴取了负号，`getOrientation()` 又使用 `atan2f(X, -Z)`。
这些符号约定需要统一验证。
如果直接接入当前主线数组，Pitch 的零位和正负方向可能与控制侧期望不一致【待验证】。

问题九：`0.0002f` 是否是固定物理时间常数。

不是。它是每次调用时的离散修正系数，源码没有把它乘以 `dt`。同样的 `0.0002f` 在 500Hz
调用时约对应 `10s` 拉回时间常数，在 100Hz 调用时约对应 `50s`。因此讨论这个参数时必须同时说明调用频率。

问题十：Roll/Yaw 末尾有代码，为什么还说不是完整三轴互补滤波。

因为 Roll 的运行阶段没有更新 `smoothAcc[ROLL]`，修正项只能拉向零值或开机初值；Yaw 没有加速度或磁力计长期参考。
这两轴不能按 Pitch 那样解释为“陀螺积分 + 运行中的加速度长期校正”。

问题十一：如果要恢复备用方案，能否只在 `main.c` 中直接调用 `initOrientation()`。

不建议这样做。当前 `initOrientation()` 在 `evvgcCF.c` 中有函数定义，但 `evvgcCF.h` 中对应声明被注释。
在当前 `-std=gnu11 -Wall` 的构建设置下，跨文件调用没有可见原型的函数至少会触发编译诊断。
工程上也会让接口边界不清楚。恢复顺序应是：先恢复头文件声明，再确定它位于 `MPU6050_Init()`、
`MargAHRSupdate()` 或备用 `getOrientation()` 调用链中的具体位置，最后再验证输入单位、轴符号和输出数组。

问题十二：`.su` 显示 `getOrientation()` 只有 56 字节静态栈用量，是否说明它一定能放进 500Hz 主循环。

不能。`.su` 是编译器生成的静态栈使用报告，不能表示三角函数、I2C 读写或调度等待的运行耗时。
它能作为内存压力线索，但实时性仍要用 `executionTime500Hz`、DWT 或示波器 GPIO 翻转来测量。

3. 如果把 `0.0002f` 改大，姿态响应和抗干扰能力会怎样变化。
4. 为什么“参与构建”不能证明“参与当前主线运行”。

6. 互补滤波方案和四元数方案分别适合什么调试场景。
7. 如果要把备用互补滤波接回主线，为什么必须同时更新数据流、单位验证、控制输入和调参记录？
8. 为什么备用算法的加速度符号约定不能只看变量名判断？
9. 为什么 `0.0002f` 不能脱离调用频率单独讨论？
10. 为什么 Roll/Yaw 有代码不等于它们已经具备完整长期参考？

## 13. 本章总结

本章完成了 `互补滤波` 备用方案的定位。

当前项目中：

- `evvgcCF.c` 实现了互补滤波思路。
- `getOrientation()` 的 Pitch 主线由加速度角低通、陀螺仪积分和小比例加速度修正组成。
- `smoothAcc` 的 `0.99/0.01` 平滑若按 500Hz 调用，等效时间常数约为 `0.199s`。
- `0.0002f` 姿态拉回项若按 500Hz 调用，等效误差衰减时间常数约为 `10s`。
- `0.0002f` 是每次调用的离散修正系数，不是独立于调用频率的连续时间常数。
- 当前备用代码等价于 `orient_new = (1-k)*orient_old + gyroRate*dt + k*smoothAcc`，
  与常见 `alpha*(angle+gyro*dt)+(1-alpha)*acc` 写法不完全相同。
- `.map/.list/.su/.cyclo` 的构建产物结论统一回到第8.7节判断：它们能证明 `evvgcCF.o` 作为构建输入存在，也能证明备用状态和函数当前位于丢弃段或缺少可达调用路径，但不能证明备用互补滤波已经进入最终固件主线或目标板运行。
- `Core/Inc/main.h` 包含 `evvgcCF.h`，但头文件包含不等于运行调用。
- `Core/Src/main.c` 当前没有调用 `getOrientation()`。
- 当前控制链路使用 `sensors.margAttitude500Hz`，不是 `sensors.evvgcCFAttitude500Hz`。
- `initOrientation()` 在 `.c` 中有定义，但 `evvgcCF.h` 中声明被注释；恢复备用方案前应先恢复接口声明。
- `evvgcCF.c` 中 Roll/Yaw 注释与实际代码不完全一致，属于【待验证】边界。
- Roll 运行阶段没有更新 `smoothAcc[ROLL]`，Yaw 没有长期航向参考，因此不能把保留代码写成完整三轴互补滤波。
- `gyroRate[PITCH] = gyroData[PITCH] * 0.0174532925f` 假设输入为 `deg/s`，若直接传入当前
  主线 `rad/s` 角速度会形成重复转换风险。
- `getOrientation()` 内部没有对 `dt`、`accData`、`gyroData` 做 `NaN/Inf` 清理，也没有加速度模长门控；恢复前必须把输入有效性检查放在调用点。
- Roll/Yaw 保留代码没有和 Pitch 一样显式乘以 `0.0174532925f`，三轴陀螺单位入口需要恢复前重新统一。
- `initOrientation()` 的 Z 轴加速度缩放带负号，当前主线 Z 轴缩放不再额外取反；恢复备用方案前必须验证 Pitch 符号约定。
- 当前 Debug 构建报告显示 `initOrientation()`/`getOrientation()` 函数本体静态栈用量不大、圈复杂度低，
  但这不能替代 500Hz 运行耗时测量。
- 若未来恢复该方案，必须先验证初始化入口、轴符号、陀螺仪单位和输出数组切换。
- 备用方案恢复至少要闭合接口、数据流、构建可达性和行为验证四类证据，不能只凭一行调用或对象文件存在判定为可替代主线。

本章保留以下边界：

- 不把 `evvgcCF.c` 写成当前主线姿态来源。
- 不把备用方案的输入单位和符号约定默认等同于当前主线。
- 不把 `getOrientation()` 写成已经自带完整输入有效性防护的姿态入口。
- 不在教材中擅自完成姿态算法切换，只说明恢复前需要验证哪些证据。
