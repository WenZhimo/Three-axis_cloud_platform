## 1. 本章目标

- 理解 `SysTick_Handler()`、`frameCounter`、`frame_500Hz` 和 `main()` 主循环之间的触发关系。

- 分清 500Hz 实时控制路径、10Hz 低频任务和 TIM6 初始化边界。

## 2. 前置知识

本章正式前置知识包括：

- `SysTick系统节拍`
- `DWT周期计数器`

第09章已经说明，项目用 1ms SysTick 节拍产生 `frame_500Hz` 标志。第10章已经说明，项目用 `micros()` 得到微秒级时间戳。第19章、第23章和第25章已经分别讲过 MPU6050 读取、姿态输出和 `sensors_t` 状态容器。

本章不展开 MPU6050 寄存器细节，不展开 `MargAHRSupdate()` 的四元数计算，也不展开 `computeMotorCommands()` 内部 PID 算法。它只回答一个调度问题：500Hz 这一帧里，谁先执行，谁后执行，时间间隔如何传递。

## 3. 问题背景

三轴云台需要不断读取 IMU、更新姿态、计算控制量并输出电机指令。如果这些动作没有稳定节拍，控制系统就会出现三个问题：

- 采样间隔不稳定，姿态积分的 `dt` 不可靠。
- 控制输出更新不稳定，电机补偿时快时慢。
- 调试时无法判断问题来自传感器、算法、控制器，还是任务调度。

但实时控制也不能简单全部塞进中断里。当前项目通过一种折中方式组织 500Hz 实时控制循环：

```text
SysTick 中断只负责置位 frame_500Hz
-> main() 主循环发现标志
-> 清除标志并计算 dt500Hz
-> 读取 IMU
-> 缩放并写入 sensors_t
-> 更新姿态
-> 执行启动门控
-> 调用 computeMotorCommands(dt500Hz)
```

这种结构的核心思想是：中断要短，主循环做重任务，实际 `dt` 用微秒时间戳测量，而不是假定每一帧一定精确等于 2ms。

## 4. 核心概念

### 4.1 1ms系统节拍

`SysTick` 在当前项目中配置为 1ms 一次。每次进入 `SysTick_Handler()`，代码先调用 `HAL_IncTick()`，使 HAL 的毫秒时间继续前进。

这一步来自第09章。第26章关心的是：项目如何从 1ms 节拍派生出 500Hz 实时控制入口。

### 4.2 frameCounter

`frameCounter` 是项目自己的帧计数器。`Core/Inc/main.h` 中定义：

```c
#define FRAME_COUNT   1000
#define COUNT_500HZ   2
```

`SysTick_Handler()` 在系统就绪且 MPU6050 不处于校准状态时递增 `frameCounter`。

当 `frameCounter > FRAME_COUNT` 时，它回到 1。这里的 `FRAME_COUNT=1000` 可以理解为一个 1秒范围内的 1ms 计数循环。

### 4.3 frame_500Hz

`frame_500Hz` 是 500Hz 实时控制循环触发标志。当前代码在 `SysTick_Handler()` 中执行：

```c
if ((frameCounter % COUNT_500HZ) == 0)
{
    frame_500Hz = true;
}
```

由于 `COUNT_500HZ=2`，而 SysTick 是 1ms 节拍，所以每两个 SysTick 周期置位一次，对应约 2ms 一帧，也就是 500Hz。

### 4.4 dt500Hz

`dt500Hz` 是当前 500Hz 帧与上一帧之间的实际时间间隔，单位为秒。`main()` 主循环进入 500Hz 实时控制分支后执行：

```c
currentTime = micros();
deltaTime500Hz = currentTime - previous500HzTime;
previous500HzTime = currentTime;
dt500Hz = (float)deltaTime500Hz / 1000000.0f;
```

它不是固定写死的 `0.002f`。这让姿态解算和控制函数能看到实际帧间隔。

`micros()` 的内部也值得拆开看。它读取 `HAL_GetTick()` 的毫秒数和 `SysTick->VAL`
的当前倒计数值，并用：

```text
elapsed_us_in_this_ms = (SysTick->LOAD + 1 - SysTick->VAL) * 1000 / (SysTick->LOAD + 1)
```

把当前 1ms 周期内已经过去的计数换算成微秒。因此，`micros()` 不是直接读取 DWT 周期计数器；
它是基于 HAL 毫秒 tick 和 SysTick 倒计数寄存器拼出的微秒时间戳。

### 4.4.1 micros的双采样窗口

`micros()` 不是一次原子硬件捕获，而是一次很短的“双采样”：

```c
uint32_t m0 = HAL_GetTick();
uint32_t u0 = SysTick->VAL;
uint32_t m1 = HAL_GetTick();
uint32_t u1 = SysTick->VAL;
```

这段代码的意图是处理读取过程中刚好跨过 1ms 边界的情况。如果 `m1 != m0`，函数放弃第一组 `m0/u0`，改用第二组 `m1/u1`；如果两次毫秒 tick 相同，则使用第一组 `m0/u0`。这个设计比只读一次 `HAL_GetTick()` 和 `SysTick->VAL` 更稳健，因为它能识别一部分“读到旧毫秒数、却已经接近或跨过新 SysTick 周期”的边界。

但这仍然不是完整的临界区保护。函数没有在读取 `HAL_GetTick()` 与 `SysTick->VAL` 时关闭中断，也没有读取 `SysTick->CTRL` 的 `COUNTFLAG` 来确认是否刚发生计数回卷。结合 CMSIS 中 `SysTick_Type` 对 `LOAD/VAL/CTRL` 的定义，以及 HAL 中 `HAL_GetTick()` 直接返回 `uwTick` 的实现，本章只能把 `micros()` 写成“基于 HAL tick 与 SysTick 当前值的近似微秒时间戳”。它足够支撑当前源码中的 `deltaTime500Hz` 计算路径，但不能单独证明所有中断延迟、关中断窗口或长时间阻塞场景下的时间戳绝对单调。若调试中看到偶发异常大的 `deltaTime500Hz`，应同时记录 `HAL_GetTick()`、`SysTick->VAL`、`frame_500Hz` 消费时刻和是否有长中断/阻塞路径【待验证】。

首帧还有一个边界。初始化末尾先执行：

```c
previous500HzTime = micros();
```

之后主循环等待第一次 `frame_500Hz` 被消费。第一帧 `dt500Hz` 因此表示“初始化末尾记录点”
到“第一次进入 500Hz 分支”的时间差，而不是硬件捕获到的严格 2ms 周期。正常情况下它应接近
首次 500Hz 触发间隔，但是否精确等于 2000us 仍应以调试记录为准。

还有一个长期运行边界。`micros()` 返回 `uint32_t`，单位是微秒。其取值范围约为：

```text
2^32 us = 4294967296 us
        ~= 4294.97 s
        ~= 71.58 min
```

也就是说，微秒时间戳约 71.6 分钟会回绕一次。当前代码用无符号差值
`deltaTime500Hz = currentTime - previous500HzTime` 计算短时间间隔；只要两次消费之间
没有接近整个 32 位回绕周期，这种差值仍可用于 2ms 级周期测量。调试日志如果直接比较
绝对 `micros()` 值，则必须考虑回绕，不能把时间戳突然变小误判为系统倒退。

### 4.5 executionTime500Hz

`executionTime500Hz` 记录 500Hz 实时控制分支本身耗费了多少微秒。它在分支末尾通过：

```c
executionTime500Hz = micros() - currentTime;
```

得到。这个值不是控制输入，而是性能观察量。调试时可以用它判断 500Hz 实时控制循环是否接近或超过 2ms 预算。

### 4.6 触发周期与消费周期

本章需要区分两个时间概念。

第一个是触发周期。由于 `COUNT_500HZ=2`，SysTick 为 1ms，
所以理想触发周期为：

```text
T_trigger = COUNT_500HZ * 1ms = 2ms
```

第二个是消费周期。`dt500Hz` 不是从 `COUNT_500HZ` 直接推导出来的，
而是由主循环两次消费 `frame_500Hz` 标志之间的 `micros()` 差值计算得到。

如果主循环没有被阻塞，二者应接近。如果主循环被 I2C、UART 打印或控制计算拖慢，
`frame_500Hz` 仍按 SysTick 节奏置位，但主循环消费会延后，`dt500Hz` 会变大。

### 4.7 布尔标志与错过帧

当前 `frame_500Hz` 是布尔标志，不是计数器。

如果主循环 5ms 才处理完一次 500Hz 分支，期间 SysTick 可能已经多次尝试置位
`frame_500Hz`。但布尔值只能保持为 `true`，不会记录“已经积压了 2 帧或 3 帧”。

因此，当前代码能告诉主循环“至少有一帧需要处理”，但不能告诉主循环“错过了几帧”。

如果未来需要统计错过帧，通常要引入待处理帧计数、信号量、溢出计数器或时间戳队列。

### 4.8 共享标志的可见性

`frame_500Hz` 还有另一层工程含义：它是中断上下文和主循环上下文之间的共享标志。

中断里写：

```c
frame_500Hz = true;
```

主循环里读并清：

```c
if (frame_500Hz)
{
    frame_500Hz = false;
}
```

这类变量通常要让编译器知道“它可能在当前执行流之外被改变”。在 CMSIS 中，`__IO`
被定义为 `volatile`，HAL 的毫秒 tick 变量 `uwTick` 也是 `__IO uint32_t`。
这说明 HAL 自己处理中断更新时间基时，会显式表达这种外部可变性。

当前项目里的 `frame_500Hz`、`systemReady` 和 `frameCounter` 声明为普通变量，
不是 `volatile`。本章不修改源码，但教材应把它记录为工程审查点：当前调试通过
不能反向证明编译器优化、优化等级变化或后续重构下仍然没有共享变量可见性风险。

### 4.9 2ms预算与剩余时间

500Hz 对应单帧理论预算约为 2000 微秒。调试时可以用：

```text
slack_us = 2000us - executionTime500Hz
```

粗略估算 500Hz 分支内部剩余时间。

但这个公式有边界。`executionTime500Hz` 从主循环进入 500Hz 分支后开始统计，
不等于完整系统调度延迟。它不直接包含主循环进入分支前已经发生的等待，
也不完整表达 10Hz 任务对下一次消费时刻的影响。

## 5. 工作原理

当前 500Hz 实时控制循环可以分成五层。

第一层，初始化建立时间基础。

`main()` 先调用 `HAL_Init()`、`SystemClock_Config()` 和各外设初始化函数。随后在用户初始化段中再次调用：

```c
HAL_SYSTICK_Config(SystemCoreClock / 1000);
HAL_NVIC_SetPriority(SysTick_IRQn, 0, 0);
```

这一步让 SysTick 保持 1ms 节拍。项目还启用 DWT 计数器，用于第10章讲过的 `DWT_Delay_us()` 等微秒等待路径；本章的 `dt500Hz` 则由 `micros()` 的 HAL/SysTick 时间戳路径计算。

第二层，系统就绪后中断开始产生帧标志。

`systemReady` 在初始化末尾置为 `true`。之后 `SysTick_Handler()` 只有在：

```c
(systemReady == true) && (mpu6050Calibrating == false)
```

成立时才推进 `frameCounter` 并可能置位 `frame_500Hz`。这意味着开机初始化阶段和 MPU6050 校准阶段不会正常推进 500Hz 实时控制帧。

这里还要分清两种校准位置。当前 `main.c` 中 `mpu6050Calibration()` 调用是注释状态，而且位置在
`systemReady = true` 与 `previous500HzTime = micros()` 之前。如果未来只是在这个启动初始化位置恢复校准，
500Hz 帧本来还没有开始消费，校准结束后再记录 `previous500HzTime`，首帧 `dt500Hz` 不会自然包含整段校准耗时。

但如果未来把温度校准放到运行中触发，`mpu6050Calibration()` 会把 `mpu6050Calibrating` 置为
`true`，函数结束时再置回 `false`。在这段期间 SysTick 不推进 `frameCounter`，也不置位新的
`frame_500Hz`；同时 `previous500HzTime` 不会在校准期间自动刷新。恢复后的第一次 500Hz 消费可能会看到
一个包含暂停时间的 `deltaTime500Hz`，除非恢复路径显式重置 `previous500HzTime` 或丢弃恢复首帧。

第三层，主循环消费标志。

`main()` 的 `while (1)` 中检查：

```c
if (frame_500Hz)
{
    frame_500Hz = false;
    ...
}
```

注意，当前设计不是在中断中读 MPU6050 或直接算控制，而是在主循环中执行这些耗时任务。`stm32f1xx_it.c` 中保留的注释也说明，在中断中读 I2C 会耗时，并且与主循环重复读取可能造成总线冲突。

第四层，主循环按顺序执行控制链路。

当前 500Hz 实时控制分支中的顺序是：

```text
micros() 计算 dt500Hz
-> MPU6050_Read_And_Process()
-> 写入 sensors.accel500Hz / sensors.gyro500Hz
-> MargAHRSupdate(..., dt500Hz)
-> startup_delay_counter 启动门控
-> computeMotorCommands(dt500Hz)
-> executionTime500Hz 性能记录
```

这条顺序非常重要。`computeMotorCommands()` 读取的是已经更新后的 `sensors.margAttitude500Hz`。如果姿态更新没有在控制计算前执行，控制输入就会落后一帧或使用旧值。

第五层，10Hz 低频任务与 500Hz 实时控制循环并存。

500Hz 实时控制分支之后，主循环还用：

```c
if (HAL_GetTick() - last_print_tick >= 100)
```

执行约 10Hz 的低频状态处理和串口打印。这个 10Hz 低频任务和 500Hz 实时控制循环共用同一个 `while (1)`，但它不属于 500Hz 实时控制路径。分析实时控制时，不能把 10Hz 打印当成控制算法的一部分。

还要注意，`Core/Src/main.c` 中定义了：

```c
const uint32_t print_interval = 100;
```

但当前 10Hz 条件使用的是字面量 `100`，不是 `print_interval` 变量。因此如果未来只改
`print_interval` 常量，不能证明低频任务周期已经改变；必须同步检查实际条件表达式。

从调度角度看，当前设计的关键不是“SysTick 有没有每 2ms 置位一次”，
而是“主循环能否在下一次置位前完成本帧工作并回到轮询点”。

如果 500Hz 分支加上低频任务造成主循环短暂阻塞，后续表现通常是：

```text
SysTick 继续置位 frame_500Hz
-> 布尔标志保持 true
-> main() 稍后只消费一次
-> deltaTime500Hz 变大
-> AHRS 和控制函数收到更大的 dt500Hz
```

对姿态解算而言，四元数积分增量近似与 `omega * dt500Hz` 成正比。
同样的角速度下，`dt500Hz` 变大意味着单帧姿态积分步长变大，数值误差和瞬态跳变风险上升。

对控制链路而言，`computeMotorCommands(dt500Hz)` 接收的是原始 `dt500Hz`。
其中部分分支会构造 `safeDt`，`updatePID()` 内部也会把过大、过小、NaN 或 Inf 的
`deltaT` 回退为 `0.002f`。但 `MargAHRSupdate()` 当前收到的仍是进入分支时计算出的
原始 `dt500Hz`，所以不能把 PID 内部保护误读为整条 500Hz 链路都已经完成时间保护。

这个方案的设计权衡也很清楚。

“中断短、主循环重”的优点是避免在中断中执行 I2C 读取、浮点姿态解算和电机控制，
降低中断占用时间，也减少 I2C 总线在中断和主循环之间被重复访问的风险。
代价是主循环如果被阻塞，布尔标志会合并多个触发事件，实时性退化不如计数器或 RTOS 任务直观。

“定时器中断直接执行完整闭环”的优点是触发抖动更小，调度边界更硬。
代价是中断执行时间变长，I2C 阻塞、浮点计算和串口调试都可能拉高中断延迟，
并影响其它中断响应。当前项目源码注释已经把 I2C 放在中断里读的风险点出来，
所以本章按现有实现讲解主循环消费方案。

## 6. STM32实现机制

从 STM32 机制看，本章涉及几个层次。

第一是内核 SysTick。SysTick 是 Cortex-M3 内核提供的系统定时器。项目用它产生 1ms 中断，并通过 `HAL_IncTick()` 维持 HAL 时间基准。

第二是主循环轮询标志。中断侧只设置 `frame_500Hz`，主循环侧清除并执行任务。这种“中断置位、主循环消费”的结构能降低中断处理时间，避免在中断里做 I2C 读取和复杂计算。

第三是微秒时间测量。`micros()` 使用 `HAL_GetTick()` 和 `SysTick->VAL` 组合出微秒时间戳。500Hz 实时控制循环进入后用两次时间戳差得到 `deltaTime500Hz`，再换算成秒得到 `dt500Hz`。

这里要注意一个容易混淆的边界。项目确实启用了 DWT 计数器，并在其它微秒延时路径中使用。
但当前 `dt500Hz` 的直接时间戳来源是 `main.c::micros()` 中的 `HAL_GetTick()` 和
`SysTick->VAL`，不是直接读取 `DWT->CYCCNT` 得到的周期计数。
因此分析 `dt500Hz` 精度时，要按 SysTick 时间基准和 `SystemCoreClock` 配置来审查。

这个结论要以函数实现为准，而不是以零散注释为准。当前仓库中仍能看到
`sysTickCycleCounter` 全局变量，以及 `stm32f1xx_it.c` 里一段已经被块注释包住的旧逻辑：

```c
/* ...
sysTickCycleCounter = DWT->CYCCNT;  // 必须读DWT，micros()要用
sysTickUptime++;
*/
```

这段注释只能说明项目历史上曾考虑或保留过 DWT 参与计时的路径，不能证明当前
`micros()` 正在读取 `DWT->CYCCNT`。当前有效代码中，`SysTick_Handler()` 确实会调用
`micros()` 计算 `deltaTime1000Hz/executionTime1000Hz`，而 `micros()` 本体仍然读取
`HAL_GetTick()`、`SysTick->VAL` 和 `SysTick->LOAD`。因此阅读实时路径时要分清
“变量存在”“注释存在”“有效语句存在”和“函数实现实际读取”的四个证据层级。

还要注意 SysTick 配置来源。`HAL_Init()` 会初始化 HAL 时间基准；`HAL_RCC_ClockConfig()`
也可能在时钟配置后重新初始化 tick。当前用户代码随后又显式调用：

```c
HAL_SYSTICK_Config(SystemCoreClock / 1000);
HAL_NVIC_SetPriority(SysTick_IRQn, 0, 0);
```

因此教材分析当前运行时的 1ms 节拍，应以最终生效的 SysTick 配置和 `SystemCoreClock`
为准，而不是只看 `HAL_Init()` 初始阶段的 tick 配置。

第四是共享状态的编译器可见性。HAL 的 `uwTick` 用 `__IO` 声明，而 CMSIS 中
`__IO` 对应 `volatile`。这说明被 SysTick 中断更新、又被普通函数读取的时间基准，
需要显式告诉编译器不要按普通局部执行流假设优化。当前项目的 `frame_500Hz`
也具有 ISR 写、主循环读写的特征，但源码中它不是 `volatile`。本章只能记录这个审查点，
不能声称当前实现已经具备共享标志声明上的最佳实践。

第五是阻塞式外设调用对主循环实时性的影响。`MPU6050_Read_And_Process()` 内部调用
`MPU_Read_Reg()`，后者使用 `HAL_I2C_Mem_Read(..., 100)`。这个 `100` 是 HAL I2C
阻塞读的超时参数。正常总线下实际耗时可能远低于该值，但在总线异常、设备无响应或状态忙时，
阻塞路径会威胁 2ms 控制预算；实际发生概率和持续时间需要硬件日志或示波器证据，保持【待验证】。

第六是调试输出的阻塞风险。`Core/Src/usart.c` 中 USART3 波特率为 115200，
`_write()` 对每个字节调用一次 `HAL_UART_Transmit(&huart3, ..., 1, 0xFFFF)`。
HAL 的阻塞发送会等待 TXE 和 TC 标志。按 8N1 串口格式估算，1 字节约 10 bit：

```text
T_byte ~= 10 / 115200 s ~= 86.8us
```

如果一次打印 20 字节，仅线缆串行化时间理论上约为：

```text
20 * 86.8us ~= 1.74ms
```

这还没有包含 HAL 循环、格式化和等待标志的额外开销。当前打印频率约 10Hz，
不是每个 500Hz 帧都打印，但它与 500Hz 分支共用主循环，所以仍应纳入实时性调试。

项目还初始化了 TIM6，并在初始化末尾执行：

```c
__HAL_TIM_SET_COUNTER(&htim6, 0);
```

但当前 `frame_500Hz` 的置位证据在 `Core/Src/stm32f1xx_it.c` 的 `SysTick_Handler()` 中。

`Core/Src/tim.c` 中能看到 TIM6 初始化，当前没有看到它产生 500Hz 实时控制循环触发标志或进入主控制循环的证据。

因此本章不能把 TIM6 写成当前 500Hz 触发源。

## 8. 代码分析

### 8.1 main.h中的控制周期定义

`Core/Inc/main.h` 中和本章直接相关的是：

```c
uint32_t micros(void);
extern uint16_t frameCounter;
extern bool frame_500Hz;
extern float dt500Hz;

#define FRAME_COUNT   1000
#define COUNT_500HZ   2
```

`COUNT_500HZ` 是从 1ms 节拍得到 500Hz 实时控制循环触发标志的关键。`dt500Hz` 则是后续姿态解算和控制计算共用的时间步长。

### 8.2 SysTick_Handler只设置标志

`Core/Src/stm32f1xx_it.c` 中，`SysTick_Handler()` 的主线是：

```text
HAL_IncTick()
-> 检查 systemReady 和 mpu6050Calibrating
-> frameCounter++
-> frameCounter 超过 FRAME_COUNT 后回到 1
-> 每 COUNT_500HZ 帧置位 frame_500Hz
```

这里没有执行 `MPU6050_Read_And_Process()`。源码中保留的相关调用位于注释块内，这与第19章“中断不直接读 I2C”的结论一致。

还要注意 `HAL_IncTick()` 与项目 500Hz 帧门控不是同一个层级。`HAL_IncTick()` 位于
`systemReady && !mpu6050Calibrating` 判断之前；因此即使 `mpu6050Calibrating == true`
导致 `frameCounter` 和 `frame_500Hz` 停止推进，HAL 的 `uwTick` 仍会继续递增。

这个顺序解释了两个现象：

```text
HAL_IncTick() 继续推进 -> HAL_GetTick() 继续变大 -> HAL_Delay() 可以结束
项目帧门控暂停   -> frameCounter 不推进      -> frame_500Hz 不产生新帧
```

`Drivers/STM32F1xx_HAL_Driver/Src/stm32f1xx_hal.c` 中，`HAL_Delay()` 通过
`HAL_GetTick() - tickstart` 等待超时；`HAL_GetTick()` 返回 `uwTick`，而 `HAL_IncTick()`
递增 `uwTick`。所以校准函数里的 `HAL_Delay(10000)` 依赖的是 HAL 毫秒时间基准，不是
`frame_500Hz`。不能把“500Hz 帧暂停”误解成“整个 SysTick/HAL 时间停止”。

### 8.3 main循环消费frame_500Hz

`Core/Src/main.c` 中真正的 500Hz 工作发生在：

```c
if (frame_500Hz)
{
    frame_500Hz = false;
    currentTime = micros();
    deltaTime500Hz = currentTime - previous500HzTime;
    previous500HzTime = currentTime;
    dt500Hz = (float)deltaTime500Hz / 1000000.0f;
    ...
}
```

先清除标志，再记录当前时间。这样下一次 SysTick 置位时，主循环可以再次进入分支。

首帧 `deltaTime500Hz` 的前一个时间戳来自初始化末尾的 `previous500HzTime = micros()`。
因此验证周期稳定性时，最好把第一帧和后续稳定帧分开记录。第一帧偏大或偏小不一定代表
长期 500Hz 失稳；连续多帧偏离 2000us 才更能说明调度、I2C、打印或控制计算存在持续影响。

这里还要记录 32 位微秒时间戳回绕边界。`currentTime`、`previous500HzTime`
和 `deltaTime500Hz` 都是 `uint32_t`。短间隔无符号相减适合本项目的 2ms 级测量，
但如果调试脚本把 `currentTime` 当作永远递增的绝对时间，就会在约 71.6 分钟后遇到回绕。

还要记录门控暂停后的恢复首帧。如果 `frame_500Hz` 曾因 `mpu6050Calibrating == true`
停止产生，而 `previous500HzTime` 没有同步重置，那么恢复后的第一次：

```text
deltaTime500Hz = currentTime - previous500HzTime
```

可能不是 2ms，而是“暂停前最后一次消费”到“恢复后第一次消费”的总间隔。这个值会直接变成
`dt500Hz`，并传给 `MargAHRSupdate()` 和 `computeMotorCommands()`。因此运行中校准恢复路径
必须单独记录，不能把它和普通连续帧混在一起计算周期稳定性。
教材中的周期判断应优先看差值和连续帧统计，而不是孤立的绝对时间戳大小。

如果运行中校准使用当前字面量 `HAL_Delay(10000)`，则 HAL 时间会推进约 10000ms；
但项目 500Hz 帧在 `mpu6050Calibrating == true` 期间不会补发。恢复后第一帧可近似理解为：

```text
deltaTime500Hz_after_resume ≈ t_resume_first_consume - t_last_consume_before_pause
missed_frame_count 不可由 frame_500Hz 还原
```

这里的重点不是把 10 秒写成已实测暂停时间，而是建立调试判断：HAL 延时结束、项目帧恢复、
恢复首帧 `dt500Hz` 变大，是三件不同的事。

### 8.4 一帧内的数据和控制顺序

进入 500Hz 实时控制分支后，顺序是固定的：

1. 调用 `MPU6050_Read_And_Process()` 读取并处理原始 IMU 数据。
2. 写入 `sensors.accel500Hz` 和 `sensors.gyro500Hz`。
3. 调用 `MargAHRSupdate(..., dt500Hz)` 更新 `sensors.margAttitude500Hz`。
4. 根据 `startup_delay_counter` 处理开机收敛门控。
5. 调用 `computeMotorCommands(dt500Hz)`。
6. 写入 `executionTime500Hz`。

本章只分析顺序，不展开第27章的 PID 内部计算。

### 8.5 启动门控的位置

`startup_delay_counter` 在 500Hz 实时控制分支内递增。计数达到 1000 时，项目清空 PID 积分和微分状态，并打开部分控制轴：

```text
1000帧 / 500Hz = 2秒
```

这说明开机前 2 秒主要用于等待 AHRS 姿态收敛。门控发生在姿态更新之后、控制计算之前，因此第27章分析 PID 时要注意：控制函数每帧仍被调用，但轴使能状态会影响实际输出路径。

### 8.6 TIM6边界

`Core/Src/main.c` 调用了 `MX_TIM6_Init()`，并在系统准备完成前清零 TIM6 计数器。`Core/Src/tim.c` 中也能看到 TIM6 的 `Prescaler` 和 `Period` 配置。

但当前没有发现 `HAL_TIM_Base_Start_IT(&htim6)`、TIM6 中断处理函数置位 `frame_500Hz`，或 TIM6 回调驱动主循环的证据。当前 500Hz 实时控制循环触发标志仍应追踪到 `SysTick_Handler()`。

这一点要和“TIM6 具备作为基本定时器的硬件能力”分开。ST `RM0008`
说明 TIM6/TIM7 属于基本定时器，可作为时间基准并产生更新事件；HAL TIM 驱动也提供
`HAL_TIM_Base_Start_IT()` 和弱定义的 `HAL_TIM_PeriodElapsedCallback()`。但是教材判断当前项目触发源时，
不能从“芯片和 HAL 支持这件事”直接跳到“本项目正在这样用”。

若要证明 TIM6 实际驱动 500Hz，证据链至少应同时包含：

1. `MX_TIM6_Init()` 配置了 TIM6 的时钟、PSC 和 ARR。
2. 主程序调用 `HAL_TIM_Base_Start_IT(&htim6)` 或等价启动逻辑。
3. 工程中存在有效的 `TIM6_IRQHandler()`，并在其中调用 `HAL_TIM_IRQHandler(&htim6)`。
4. `HAL_TIM_PeriodElapsedCallback()` 或用户中断路径写入 `frame_500Hz`。
5. `.list` 能把上述调用链和写标志路径对应到当前构建产物。

当前仓库只满足其中一部分：`Three-axis_cloud_platformV2.ioc` 和 `Core/Src/tim.c`
能证明 TIM6 被配置为内部时钟来源，`Core/Startup/startup_stm32f103rctx.s` 中也有
`TIM6_IRQHandler` 向量名；但该向量仍是弱符号并指向 `Default_Handler`，
全工程没有发现 `HAL_TIM_Base_Start_IT(&htim6)` 调用，也没有发现 TIM6 路径写入
`frame_500Hz`。因此 `.map` 中出现 `HAL_TIM_Base_Start_IT`、`HAL_TIM_PeriodElapsedCallback`
或 `TIM6_IRQHandler`，只能说明 HAL 驱动和启动向量中存在这些符号，不能证明 TIM6
已经成为当前 500Hz 触发源。

### 8.7 10Hz低频任务边界

500Hz 实时控制分支之外，`main()` 还有 10Hz 低频任务。它处理状态计数、回中状态和串口打印。低频任务会消耗主循环时间，因此调试实时性时需要观察它是否影响 `executionTime500Hz` 和下一次 500Hz 实时控制分支进入时间。

但从数据流上看，10Hz 打印不是控制闭环的核心计算路径。

低频任务周期还存在一个可维护性边界：源码同时存在 `print_interval = 100` 和条件中的
字面量 `100`。当前实际生效的是条件表达式里的 `100`。教材不能把 `print_interval`
写成当前低频任务周期的唯一配置入口。

### 8.8 布尔标志合并帧的代码含义

中断侧写入是：

```c
frame_500Hz = true;
```

主循环侧消费是：

```c
if (frame_500Hz)
{
    frame_500Hz = false;
    ...
}
```

这里没有 `pendingFrames++`、没有队列，也没有记录“置位时标志已经为 true”的分支。
因此如果主循环在一次 500Hz 分支、I2C 读取或低频打印中停留过久，后续只能通过
`deltaTime500Hz` 看出两次消费之间时间变长，不能从 `frame_500Hz` 本身还原错过帧数。

### 8.9 I2C读取失败路径

`Drivers/CustomDrivers/Src/mpu6050.c` 中，`MPU_Read_Reg()` 调用：

```c
HAL_I2C_Mem_Read(&hi2c2, MPU6050_I2C_ADDR, reg_addr,
                 I2C_MEMADD_SIZE_8BIT, data, len, 100);
```

`MPU6050_Read_And_Process()` 在 500Hz 分支内调用。若读失败，代码清空
`rawAccel` 和 `rawGyro` 后返回：

```c
memset(rawAccel, 0, sizeof(rawAccel));
memset(rawGyro, 0, sizeof(rawGyro));
return;
```

这带来两个教材级结论。

第一，I2C 是 500Hz 分支的关键耗时来源之一。正常耗时要实测，异常路径上还要考虑 HAL 超时。

第二，读失败后的零值会继续进入后续 `sensors_t` 缩放和姿态链路。当前仓库没有连续错误日志，
所以不能断言它已经导致姿态异常，只能把该路径作为需要验证的故障注入点。

### 8.10 UART打印与2ms预算

`Core/Src/usart.c` 中 `_write()` 逐字节发送：

```c
for (DataIdx = 0; DataIdx < len; DataIdx++)
{
    HAL_UART_Transmit(&huart3, (uint8_t *)&ptr[DataIdx], 1, 0xFFFF);
}
```

HAL UART 阻塞发送内部会等待发送数据寄存器空和传输完成标志。
这说明 `printf()` 不是“把字符串丢出去就立刻返回”的异步操作。

以 115200 baud、8N1 格式估算，20 字节串口输出的线缆时间约 1.74ms。
如果打印内容更长，或者 HAL 等待标志、格式化输出、串口异常叠加，主循环消费下一次
`frame_500Hz` 的时刻就可能被推迟。是否已经影响当前硬件运行，需要通过日志或外部测量【待验证】。

### 8.11 dt500Hz保护边界

`main()` 计算出的 `dt500Hz` 会同时进入：

```c
MargAHRSupdate(..., dt500Hz);
computeMotorCommands(dt500Hz);
```

`Drivers/SRC/Src/pid.c::updatePID()` 内部对异常 `deltaT` 有回退：

```c
if (deltaT > 0.01f || deltaT < 0.0001f || isnan(deltaT) || isinf(deltaT))
{
    deltaT = 0.002f;
}
```

`computeMotorCommands.c` 中部分路径也会构造 `safeDt`。但这些保护发生在控制链路内部，
不能反推出 AHRS 入口也使用了同样的 `safeDt`。当前第26章应把结论限制为：

```text
PID 内部存在 deltaT 回退证据
AHRS 入口仍接收 main() 计算出的原始 dt500Hz
```

### 8.12 共享标志声明边界

当前源码中，`Core/Src/main.c` 定义：

```c
bool systemReady = false;
uint16_t frameCounter = 0;
bool frame_500Hz = false;
```

`Core/Inc/main.h` 中对应声明也是普通 `extern`：

```c
extern bool systemReady;
extern uint16_t frameCounter;
extern bool frame_500Hz;
```

但这些变量跨越了中断上下文和主循环上下文：

```text
SysTick_Handler() 读 systemReady，写 frameCounter / frame_500Hz
main() 写 systemReady，读写 frame_500Hz
```

对比 HAL 的时间基准，`stm32f1xx_hal.h` 声明 `uwTick` 为 `extern __IO uint32_t uwTick`，
`stm32f1xx_hal.c` 定义 `__IO uint32_t uwTick`，CMSIS 中 `__IO` 是 `volatile`。
因此，教材应明确：当前项目的调度逻辑依赖共享全局变量，但这些变量没有采用
HAL 时间基准那样的 `volatile` 声明。这个结论来自源码声明形式，而不是运行现象。

`volatile` 也不是完整的并发保护。它能约束编译器不要缓存或消除访问，但不能让
“读标志再清标志”自动变成不可打断的事务，也不能解决布尔标志合并帧问题。
如果未来要提升调度可靠性，需要同时考虑 `volatile`、短临界区、计数器化待处理帧、
错过帧统计和调度延迟日志。

### 8.13 构建产物证据边界

源码能说明“作者写了什么”，构建产物能进一步说明“这些符号和调用路径是否进入了当前 Debug 镜像”。
本章可用的构建产物主要是 `Debug/Three-axis_cloud_platformV2.map` 和
`Debug/Three-axis_cloud_platformV2.list`。

`.map` 中可以看到下列函数已经进入最终镜像：

```text
micros
SysTick_Handler
HAL_IncTick
MPU6050_Read_And_Process
MargAHRSupdate
computeMotorCommands
```

同一份 `.map` 还给出了 500Hz 路径关键全局对象的 `.bss` 分配证据：

```text

.bss.previous500HzTime    0x20000578
.bss.executionTime500Hz   0x2000057c
.bss.dt500Hz              0x20000580
.bss.systemReady          0x20000584
.bss.frameCounter         0x20000586
.bss.frame_500Hz          0x20000588
```

#### 8.13.1 时间变量读数语义边界

第26章里最容易跳步的地方，是把几个时间变量都理解成“500Hz 是否稳定”的同一种证据。
当前源码和 `.map` 能把它们拆成四类：

| 变量 | `.map` 证据 | 源码写入位置 | 能回答的问题 | 不能回答的问题 |
|---|---|---|---|---|
| `currentTime` | `.bss.currentTime 0x20000564 0x4` | 进入 500Hz 分支后立即写入 `micros()` | 本轮分支执行耗时的起点 | SysTick 何时置位 `frame_500Hz` |
| `previous500HzTime` | `.bss.previous500HzTime 0x20000578 0x4` | 初始化末尾写一次，之后每次消费帧时更新 | 上一次主循环消费 500Hz 标志的时间戳 | 上一次 SysTick 触发的精确时刻 |
| `deltaTime500Hz` | `.bss.deltaTime500Hz 0x20000574 0x4` | `currentTime - previous500HzTime` | 两次主循环消费之间的微秒间隔 | 布尔标志期间合并了多少次触发 |
| `dt500Hz` | `.bss.dt500Hz 0x20000580 0x4` | `deltaTime500Hz / 1000000.0f` | 传给 AHRS 和控制函数的秒级步长 | 该步长是否已经被上下限保护 |
| `executionTime500Hz` | `.bss.executionTime500Hz 0x2000057c 0x4` | 分支末尾 `micros() - currentTime` | 本轮 500Hz 分支内部墙钟耗时 | 下一轮消费是否会被 10Hz 任务推迟 |

注意 `previous500HzTime = currentTime` 发生在 IMU 读取、姿态更新和控制计算之前。
所以 `deltaTime500Hz` 的语义是“本次消费入口到上次消费入口的间隔”，不是“上一次分支结束到本次分支开始的空闲时间”。同理，`executionTime500Hz` 是“本次分支入口到本次分支末尾”的耗时，不包含本次分支结束后到下一次消费前的 10Hz 打印或其它主循环工作。

因此调试日志里至少要同时记录两组量：一组是 `deltaTime500Hz/dt500Hz`，用于判断消费间隔和算法步长；另一组是 `executionTime500Hz`，用于判断本轮分支内部是否吃掉 2ms 预算。只看其中任意一个，都不能单独证明“500Hz 调度完整稳定”。若还要证明触发到消费的响应延迟，需要额外的 GPIO 翻转、断点计数或逻辑分析仪记录【待验证】。

`.list` 的价值在于把源码行和反汇编放在一起看。当前列表文件可证明：

- `SysTick_Handler()` 中先调用 `HAL_IncTick()`。
- 产物中存在读取 `systemReady`、检查 `mpu6050Calibrating`、递增 `frameCounter`、
  并按 `COUNT_500HZ` 置位 `frame_500Hz` 的指令路径。

### 8.14 executionTime500Hz测量窗口边界

`executionTime500Hz` 很容易被误读成“500Hz 调度完全满足 2ms”。源码能证明的实际含义更窄：

```c
if (frame_500Hz)
{
    frame_500Hz = false;
    currentTime = micros();
    ...
    computeMotorCommands(dt500Hz);
    executionTime500Hz = micros() - currentTime;
}
```

因此，`executionTime500Hz` 的起点不是 SysTick 置位 `frame_500Hz` 的时刻，而是主循环已经进入
`if (frame_500Hz)` 后调用 `micros()` 的时刻。它覆盖本轮 500Hz 分支内部的 MPU6050 读取、
物理量缩放、AHRS 更新、启动门控、控制计算，以及分支内部被中断打断的墙钟时间；但它不覆盖
SysTick 置位后到主循环真正进入分支之前的等待时间，也不覆盖 500Hz 分支结束后同一个
`while (1)` 中执行的 10Hz 打印对下一帧消费时刻的影响。

所以实时性调试至少要拆成三个窗口：

| 窗口 | 可用变量或观察点 | 能证明什么 | 不能证明什么 |
|---|---|---|---|
| 触发到消费 | `SysTick_Handler()` 置位、`if (frame_500Hz)` 入口断点或 GPIO | 主循环响应 500Hz 标志的延迟 | 不能由 `executionTime500Hz` 单独反推 |
| 分支执行 | `currentTime` 到 `executionTime500Hz` 写入 | 本次 500Hz 分支内部耗时 | 不能证明下一次消费不会被 10Hz 打印或其它主循环代码推迟 |
| 消费间隔 | `deltaTime500Hz` / `dt500Hz` 连续日志 | 两次主循环消费之间的实际间隔 | 不能还原布尔标志合并掉的具体错过帧数 |

一个典型误判是：看到 `executionTime500Hz < 2000us`，就认为系统没有丢帧。这个结论证据不足。
如果某次 500Hz 分支耗时 1200us，但分支之后恰好执行了一次阻塞式串口打印，那么下一次
主循环消费 `frame_500Hz` 仍可能晚于理论 2ms。相反，如果 `deltaTime500Hz` 连续接近 2000us，
但缺少触发到消费的 GPIO 或断点记录，也只能说明两次消费间隔稳定，不能单独证明 SysTick
置位瞬间到主循环响应之间没有抖动。

因此，本章把 `executionTime500Hz` 写成“分支执行耗时观察量”，而不是完整实时性验收指标。
要证明 500Hz 周期真正稳定，至少还需要连续 `deltaTime500Hz` 日志、`executionTime500Hz`
日志、低频任务耗时记录，以及必要时用 GPIO 翻转或逻辑分析仪观察 SysTick 置位、主循环入口和
分支出口三个时刻；缺少这些运行证据时，周期稳定性和丢帧数量继续保持【待验证】。

### 本节证据边界

本节只根据当前仓库说明文件、函数、宏、变量和调用关系。注释掉的历史代码、备用变量和
Debug 中间文件只能作为线索，不能单独证明当前运行路径。`.map` 与 `.list` 可以证明部分符号、
对象分配和构建快照中的指令路径，但不能替代运行时测量。运行时频率、外部硬件表现、主机侧现象、
传感器方向、电机响应或真实控制效果仍需调试记录、日志或仓库外实测证据；缺少证据时保持【待验证】。

## 9. 调试方法

第一步，确认 SysTick 是否产生 500Hz 实时控制循环触发标志。

- 在 `SysTick_Handler()` 中观察 `frameCounter`。
- 确认 `COUNT_500HZ` 为 2。
- 观察 `frame_500Hz` 是否大约每 2ms 置位一次。

第二步，确认主循环是否消费标志。

- 在 `Core/Src/main.c` 的 `if (frame_500Hz)` 内设置断点。
- 确认进入分支后 `frame_500Hz` 被清为 `false`。
- 确认 `previous500HzTime` 每帧更新。

第三步，观察时间间隔。

- 观察 `deltaTime500Hz` 是否接近 2000 微秒。
- 观察 `dt500Hz` 是否接近 `0.002f`。
- 将第一帧 `dt500Hz` 和后续稳定帧分开记录，因为第一帧依赖初始化末尾的
  `previous500HzTime = micros()`。
- 若做长时间日志，记录 `micros()` 的 32 位回绕边界，避免把约 71.6 分钟后的时间戳变小误判为异常倒退。
- 如果 `dt500Hz` 偶尔明显偏大，继续观察是否有 10Hz 打印、I2C 读取或控制计算耗时。

第四步，观察执行耗时。

- 观察 `executionTime500Hz`。
- 如果它接近或超过 2000 微秒，说明 500Hz 实时控制循环已经接近超时或可能错过下一帧。
- 当前 `frame_500Hz` 是布尔标志，不是待处理帧计数。如果主循环处理不过来，标志只能保持为真，不能记录错过了几帧。

建议把实时性调试拆成四类事件，而不是只看一个变量：

- 触发生成：SysTick 是否按预期每 2ms 尝试置位。
- 标志消费：主循环何时进入 `if (frame_500Hz)`。
- 帧完成：`executionTime500Hz` 是否低于 2000 微秒。
- 截止期满足：下一次触发前主循环是否已经返回轮询点。

如果允许做临时调试改动，可以增加一个“置位时标志已为 true”的计数器：

```text
if frame_500Hz already true when SysTick wants to set it:
    overrun_count++
```

这类计数器比单纯观察 `dt500Hz` 更直接，因为它能暴露布尔标志合并帧的发生次数。
本轮只更新教材，不修改项目源码。

第五步，检查共享标志声明边界。

- 查看 `frame_500Hz`、`systemReady`、`frameCounter` 是否为普通全局变量。
- 对照 HAL 的 `uwTick`，确认 HAL 时间基准使用 `__IO`/`volatile` 表达中断更新时间。
- 如果改变编译优化等级后出现偶发不进 500Hz 分支，应把共享标志可见性列入排查清单。

这一步不能用一次正常运行来判定“没有问题”。它检查的是源码声明和编译器优化边界。
当前教材只记录风险，不要求在本轮修改固件。

第六步，确认启动门控。

- 观察 `startup_delay_counter` 是否从 0 递增到 1000。
- 到达 1000 时确认 `zeroPIDintegralError()` 和 `zeroPIDstates()` 被调用。
- 确认轴使能状态变化符合当前调试意图。

第七步，核对构建产物。

- 在 `.map` 中确认 `SysTick_Handler`、`micros`、`MPU6050_Read_And_Process`、
  `MargAHRSupdate` 和 `computeMotorCommands` 已进入镜像。
- 在 `.map` 中确认 `frame_500Hz`、`frameCounter`、`dt500Hz`、`deltaTime500Hz`
  和 `executionTime500Hz` 有 `.bss` 分配记录。
- 在 `.list` 中确认 SysTick 置位路径和 `main()` 消费路径是否与当前源码一致。
- 如果 `.list` 中的立即数或源码注释与当前源码不同，先把它当作旧构建快照，重新构建后再作为最新固件证据。

第八步，排除 TIM6 误判。

- 在 `Core/Src/tim.c` 中确认 TIM6 只是被初始化。
- 搜索是否存在 TIM6 中断或回调置位 `frame_500Hz`。
- 如果没有相关证据，就继续按 SysTick 主线调试。

第九步，确认低频任务周期入口。

- 查看 `print_interval` 常量。
- 查看 10Hz 条件是否真正使用该常量。
- 若条件仍写作 `HAL_GetTick() - last_print_tick >= 100`，记录实际生效入口是字面量 `100`。

调试记录建议：

- 记录 SysTick 置位、主循环消费、`previous500HzTime` 更新和 `dt500Hz` 计算四段证据。
- 对耗时风险，应记录 `executionTime500Hz`、I2C 读取、姿态更新、PID 和电机输出各自观察点。
- 启动门控记录应包含计数值、PID 状态清零函数、轴使能变化和对应帧号。
- 对打印风险，应记录打印字符串长度、打印周期、串口波特率和打印前后 `micros()` 差值。
- 对 I2C 风险，应记录正常读取耗时、读失败次数、失败后原始值是否被清零和后续姿态输入。
- 对共享标志风险，应记录变量声明、编译优化等级、是否使用 `volatile` 和是否存在临界区。
- 若没有连续日志或外部计时证据，500Hz 周期稳定性和丢帧数量保持【待验证】。

## 10. 常见问题

问题一：500Hz 循环是不是直接在 SysTick 中断里执行。

不是。当前 SysTick 中断只置位 `frame_500Hz`，真正的 MPU6050 读取、姿态更新和控制计算发生在 `main()` 主循环中。

问题二：为什么不在中断里直接读 MPU6050。

I2C 读取耗时较长。当前源码注释已经说明，在中断中读 I2C 并和主循环重复读取，可能造成总线冲突和读数异常。因此当前实现把中断职责限制为短操作。

问题三：项目初始化了 TIM6，为什么不说 500Hz 来自 TIM6。

因为当前证据显示 `frame_500Hz` 在 `SysTick_Handler()` 中置位。TIM6 的初始化和清零存在，但没有看到它驱动 `frame_500Hz` 或主控制循环。

问题四：`dt500Hz` 为什么不用固定 0.002。

实际系统会有中断延迟、主循环耗时、I2C 读取耗时和低频任务干扰。使用 `micros()` 实测帧间隔能让姿态和控制函数看到更接近实际的时间步长。

问题五：如果 `executionTime500Hz` 超过 2000 微秒会怎样。

这说明单帧执行时间已经超过 500Hz 周期预算。当前 `frame_500Hz` 是布尔标志，不能累计错过帧数；如果长期超时，控制周期会变慢，`dt500Hz` 会变大，控制稳定性需要重新评估。

问题六：为什么 `systemReady` 会影响 500Hz 实时控制循环触发标志。

`SysTick_Handler()` 只有在系统初始化完成后才推进项目帧计数。这样可以避免外设和配置还没准备好时就进入传感器读取和控制流程。

问题六补充：`mpu6050Calibrating` 结束后，下一帧 `dt500Hz` 是否一定回到 0.002 秒。

不能这样保证。若校准发生在启动阶段、且 `previous500HzTime = micros()` 在校准结束后才执行，
首帧 `dt500Hz` 仍按启动后的记录点计算。若校准发生在运行阶段，SysTick 会在校准期间停止推进
项目帧标志，但 `previous500HzTime` 不会自动随之刷新，恢复后的第一帧可能包含整段暂停时间。
当前仓库没有看到运行中恢复校准后的 `previous500HzTime` 重置逻辑。

问题七：10Hz 打印是否会影响 500Hz 实时控制循环。

它不属于 500Hz 实时控制路径，但它和 500Hz 实时控制分支同在主循环中。如果串口输出阻塞时间过长，可能间接影响下一次主循环消费 `frame_500Hz` 的及时性。

问题七补充：修改 `print_interval` 是否一定能改变 10Hz 任务周期。

当前不能这样认为。源码虽然定义了 `const uint32_t print_interval = 100`，
但实际判断条件写的是字面量 `100`。要改变低频任务周期，必须以实际条件表达式为准。

问题八：`dt500Hz` 实测后，是否就能保证 500Hz 周期不会丢帧。

不能。
`dt500Hz` 能记录两次主循环消费之间的实际时间差，
但当前 `frame_500Hz` 是布尔标志，不记录错过了几个节拍。
如果主循环长期阻塞，后续只会看到更大的时间间隔，而不是完整的积压帧队列。
所以调试时仍要同时观察执行时间、打印阻塞和 I2C 读取耗时。

问题九：`executionTime500Hz` 小于 2000 微秒，是否就一定没有实时性问题。

不一定。
它统计的是进入 500Hz 分支后到分支末尾的耗时，
不能完整覆盖进入分支前主循环已经被其它任务占用的时间。
如果 10Hz 打印刚好发生在两次 500Hz 消费之间，下一次消费仍可能延后。

问题十：为什么 I2C 读取失败也要放进实时性章节。

因为它既影响数据正确性，也影响时间预算。
当前 I2C 读是 500Hz 分支内的阻塞式 HAL 调用，失败后还会把 `rawAccel` 和 `rawGyro` 清零。
这意味着故障注入时要同时观察“读失败耗时”和“零值进入姿态链路”两个结果。

问题十一：`frame_500Hz` 没有写成 `volatile` 是否一定会导致故障。

不能这样断言。是否表现为故障与编译优化、访问模式和链接结果有关，当前仓库也没有运行日志证明它已经造成问题。
但它确实是 ISR 与主循环共享标志，源码声明又不是 `volatile`，所以教材应把它列为审查点。
HAL 的 `uwTick` 用 `__IO` 声明，说明中断更新时间基通常会显式表达外部可变性。

问题十二：把 `frame_500Hz` 改成 `volatile` 是否就解决了所有实时性问题。

不能。`volatile` 主要解决编译器可见性问题，不会把布尔标志变成计数器，
也不会让读标志、清标志和执行任务成为不可打断的完整事务。
若要解决错过帧统计和过载诊断，还需要待处理帧计数、溢出计数、短临界区或更明确的调度机制。

问题十三：`micros()` 是否是 DWT 直接计时。

当前不是。项目启用了 DWT，但 `main.c::micros()` 的实现使用 `HAL_GetTick()` 和 `SysTick->VAL`。
DWT 可用于其它微秒延时或周期计数路径，但不能把当前 `dt500Hz` 直接写成 DWT 周期计数差。

问题十四：`micros()` 时间戳会不会回绕。

会。它返回 `uint32_t` 微秒值，约 71.6 分钟回绕一次。当前 500Hz 周期用短间隔无符号差值计算，
适合 2ms 级测量；但长时间日志或人工比较绝对时间戳时必须考虑回绕。

## 11. 实践任务

开始任务前，先回到本章第8节定位 500Hz 标志生成、主循环消费、一帧内顺序、启动门控和 TIM6 边界；第9节提供实时循环调试顺序。

任务一：画出 500Hz 触发链路。

从 `HAL_IncTick()`、`frameCounter`、`frame_500Hz` 画到 `main()` 的 `if (frame_500Hz)`。

任务二：画出一帧内执行顺序。

从 `micros()` 开始，依次画出 `MPU6050_Read_And_Process()`、`sensors_t` 写入、`MargAHRSupdate()`。
再继续画出启动门控、`computeMotorCommands()` 和 `executionTime500Hz`。
验收依据是一帧顺序表包含姿态写入位置、控制读取位置和新旧数据结论。

任务三：验证 `dt500Hz`。

在调试器中观察 `deltaTime500Hz` 和 `dt500Hz`，记录它们是否接近 2000 微秒和 0.002 秒。
验收依据是周期记录表包含首帧、稳定帧、实际观察值、干扰来源、长期实时性边界和结论。

任务四：验证 TIM6 边界。

搜索 `frame_500Hz` 的写入位置，并说明为什么当前不能把 TIM6 当作 500Hz 触发源。
验收依据是触发源表包含 `SysTick_Handler()` 写入、TIM6 搜索结果和触发源结论。

任务五：观察超时风险。

观察 `executionTime500Hz`。如果它明显接近 2000 微秒，列出可能的耗时来源：I2C 读取、姿态计算、控制计算、10Hz 打印或其他阻塞任务。
验收依据是耗时记录表包含观察值、潜在来源、风险结论和单项定位边界。

任务六：建立帧周期日志。

连续记录 `deltaTime500Hz`、`executionTime500Hz` 和 `slack_us`。
至少区分正常帧、打印发生前后、I2C 读失败或人为断开传感器后的帧。
验收依据是日志能说明触发周期、消费周期和分支耗时三者是否一致。

任务七：估算 UART 打印预算。

统计一次 10Hz `printf()` 实际输出字节数，按 `10 / 115200` 秒估算每字节线缆时间。
再和 `executionTime500Hz`、下一帧 `deltaTime500Hz` 进行对比。
验收依据是表格包含字节数、理论线缆时间、实测耗时和是否影响下一帧消费的结论。

任务八：确认低频任务周期配置入口。

对照 `print_interval` 常量和 `HAL_GetTick() - last_print_tick >= 100` 条件表达式。
说明当前真正决定 10Hz 任务周期的是哪个表达式。
验收依据是表格包含常量定义、实际条件、是否一致和未来修改建议。

任务九：验证 I2C 异常路径。

在安全条件下制造或模拟一次 MPU6050 读失败，观察 `MPU6050_Read_And_Process()` 是否清零原始数据。
同时记录该帧耗时、后续 `sensors_t` 输入和姿态解算入口值。
验收依据是记录同时覆盖数据影响和时间影响；没有硬件证据时结论保持【待验证】。

任务十：验证启动门控的时间尺度。

观察 `startup_delay_counter` 从 0 到 1000 的过程，说明为什么 500Hz 下 1000 帧约等于 2 秒。
验收依据是门控时间表包含帧数、目标周期、估算时间和实际影响因素。

任务十一：审查 ISR 共享标志声明。

列出 `frame_500Hz`、`systemReady`、`frameCounter` 和 HAL `uwTick` 的声明形式。
说明哪些变量由中断写入、哪些由主循环读写、哪些使用了 `volatile` 或 `__IO`。
验收依据是表格能区分“布尔标志合并帧风险”和“共享变量编译器可见性风险”。

任务十二：验证 `micros()` 的时间基准来源。

阅读 `main.c::micros()`，列出它读取的 `HAL_GetTick()`、`SysTick->VAL` 和 `SysTick->LOAD`。
说明它为什么不是直接读取 `DWT->CYCCNT`，并计算 `uint32_t` 微秒时间戳的回绕周期。
同时搜索 `sysTickCycleCounter` 和 `DWT->CYCCNT`，把“有效代码”“注释块”“变量声明”
分开记录，避免把历史注释误写成当前 `dt500Hz` 时间来源。
验收依据是记录表包含时间来源、1ms 内微秒换算公式、回绕时间、DWT 注释边界和长时间日志注意事项。

任务十三：验证校准门控恢复首帧。

阅读 `mpu6050Calibration()` 中 `mpu6050Calibrating = true/false` 的位置，并对照
`SysTick_Handler()` 中的 `(systemReady == true) && (mpu6050Calibrating == false)` 门控。
分别分析“启动阶段恢复校准”和“运行阶段触发校准”两种情况。
验收依据是时序表包含 `systemReady`、`mpu6050Calibrating`、`previous500HzTime`、
`frame_500Hz`、`HAL_IncTick()`、`HAL_GetTick()`、`HAL_Delay()` 和恢复后第一帧
`deltaTime500Hz` 的预期边界，并能说明“HAL 时间继续推进”与“500Hz 帧暂停”不是同一个现象。

## 12. 思考题

1. 为什么当前项目选择“中断置位、主循环执行”而不是“中断中直接执行完整控制链路”。
2. `COUNT_500HZ` 从 2 改为 1，会对控制周期和 CPU 负载产生什么影响。
3. 如果 `dt500Hz` 偶尔变成 0.006 秒，姿态积分和 PID 控制可能分别受到什么影响。
4. 为什么 `frame_500Hz` 用布尔标志时无法记录错过了几帧。
5. 10Hz 串口输出为什么应尽量保持短小。
6. 如果未来要改用 TIM6 触发 500Hz，应至少补充哪些源码证据才能更新教材结论。
7. 如果主循环出现短暂阻塞，为什么“下一帧能跑起来”不等于“前面没有丢帧”。
8. 若没有连续日志或断点计数记录，500Hz 周期稳定性为什么仍应保持【待验证】？
9. 为什么 `executionTime500Hz` 不能单独证明整条调度链路没有延迟。
10. 如果 `HAL_I2C_Mem_Read()` 在异常路径上接近超时，2ms 控制预算会怎样被破坏。
11. 为什么 PID 内部 `deltaT` 回退不能替代 AHRS 入口的 `dt500Hz` 保护。
12. 如果要把 `frame_500Hz` 从布尔标志改成计数器，需要重新审查哪些并发问题。
13. 为什么 `volatile` 能帮助共享标志可见性，却不能记录错过帧数量。
14. 为什么 HAL 的 `uwTick` 声明方式可以作为审查项目共享时间变量的参考。
15. 为什么 `micros()` 的实现来源会影响对 `dt500Hz` 精度和回绕的判断？
16. 为什么启动阶段校准和运行阶段校准对恢复首帧 `dt500Hz` 的影响不同？

## 13. 本章总结

本章把前面已经讲过的 SysTick、微秒计时、MPU6050 读取、姿态解算和状态容器组织成 500Hz 实时控制循环。

当前项目中：

- `Core/Inc/main.h` 定义 `FRAME_COUNT=1000` 和 `COUNT_500HZ=2`。
- `SysTick_Handler()` 在系统就绪且 MPU6050 不校准时递增 `frameCounter`。
- `HAL_IncTick()` 位于项目帧门控之前；校准期间 HAL `uwTick` 继续推进，但 `frameCounter`
  和 `frame_500Hz` 不推进。
- 当前注释掉的 `mpu6050Calibration()` 位于 `systemReady = true` 与 `previous500HzTime = micros()` 之前；
  若未来改为运行中触发校准，恢复首帧 `dt500Hz` 需要单独验证。
- 每两个 1ms SysTick 周期置位一次 `frame_500Hz`。
- `main()` 主循环消费 `frame_500Hz`，并用 `micros()` 计算 `dt500Hz`。
- 当前 `micros()` 使用 `HAL_GetTick()` 与 `SysTick->VAL`，不是直接读取 DWT 周期计数器。
- `sysTickCycleCounter` 和注释块中的 `DWT->CYCCNT` 只能证明存在历史或备用线索，不能推翻
  `main.c::micros()` 的当前实现证据。
- `micros()` 返回 `uint32_t` 微秒时间戳，约 71.6 分钟回绕一次；周期分析应看短间隔差值和连续帧日志。
- 首帧 `dt500Hz` 的前一时间戳来自初始化末尾的 `previous500HzTime = micros()`，
  应与后续稳定帧分开观察。
- 500Hz 实时控制分支内依次执行 IMU 读取、传感器物理量写入、`MargAHRSupdate()`、启动门控和 `computeMotorCommands(dt500Hz)`。
- `executionTime500Hz` 用于观察单帧执行耗时。
- `executionTime500Hz` 的测量窗口从主循环进入 500Hz 分支后的 `currentTime = micros()` 开始，不覆盖 SysTick 置位到主循环响应的等待时间，也不覆盖分支结束后低频任务对下一帧消费的影响。
- TIM6 当前有初始化和计数器清零证据，但没有作为 500Hz 触发源的证据。
- `.map` 中出现 TIM/HAL 通用符号或弱 `TIM6_IRQHandler` 向量，不能替代
  `HAL_TIM_Base_Start_IT(&htim6)`、有效 IRQHandler、回调写标志和 `.list` 调用链证据。
- 10Hz 低频任务和 500Hz 实时控制循环同在主循环中，但不属于同一条控制计算链路。
- 当前 10Hz 条件使用字面量 `100`，不是已定义的 `print_interval` 常量。
- 触发周期由 SysTick 和 `COUNT_500HZ` 决定，实际消费周期由主循环能否及时返回决定。
- 当前布尔标志不能累计错过帧数，只能表达“至少有一帧待处理”。
- 当前 `frame_500Hz`、`systemReady` 和 `frameCounter` 没有按 HAL `uwTick` 的方式声明为
  `volatile`/`__IO`，属于共享标志声明边界。
- I2C 阻塞读、UART 阻塞打印、AHRS 和控制计算都应纳入 2ms 预算审查。
- `.su/.cyclo` 能提供 500Hz 路径关键函数的静态栈用量和圈复杂度线索，
  但不能替代真实执行时间、单帧栈峰值或分支覆盖证据。
- PID 内部存在 `deltaT` 回退证据，但 AHRS 入口仍接收原始 `dt500Hz`。
- `.map` 和 `.list` 可以证明 500Hz 关键符号、全局对象和构建快照调用路径进入固件，
  但不能证明真实周期稳定性、硬件通信成功或电机实际输出正确。

目标回扣：

- 本章已经把 SysTick 标志、主循环消费、微秒计时和 `dt500Hz` 串成当前 500Hz 执行顺序。
- 本章已经分清 500Hz 实时控制路径、10Hz 低频任务和 TIM6 初始化边界。
- 本章已经为第27章 PID 控制提供时间入口和执行顺序前置。

本章保留以下工程边界：

- 当前 `frame_500Hz` 是布尔标志，如果主循环长期超过 2ms，无法累计错过帧数。
- 当前共享标志不是 `volatile`，是否会在特定优化配置下产生问题需要专项验证。
- 当前 `dt500Hz` 没有在进入姿态函数前做上下限保护，后续异常防护章节需要继续审查。
- 当前 `micros()` 使用双采样降低 1ms 边界读数不一致风险，但不是完整临界区或硬件捕获，异常长中断、关中断窗口和阻塞路径下的绝对时间戳单调性仍需运行日志验证。
- 运行中校准暂停后的恢复首帧可能出现异常大的 `deltaTime500Hz`，当前仓库没有看到自动重置时间戳证据。
- 当前 `.su/.cyclo` 不能写成运行时栈水位、函数耗时或 2ms 预算达标证明。
- 当前仓库缺少连续运行日志，因此 500Hz 周期稳定性、I2C 异常概率和 UART 打印实际阻塞时间仍为【待验证】。

下一章将进入 `PID控制核心`，在本章建立的 500Hz 时间步长和执行顺序基础上，分析 `computeMotorCommands(dt500Hz)` 如何把姿态角、目标角和 PID 参数组织成三轴控制输出。
