# 第26章 500Hz实时控制循环

## 1. 本章目标

第25章已经说明 `sensors_t` 是 500Hz 数据流中的公共状态容器。本章继续向外看一层：这些数据在什么时间点被更新，姿态解算和控制计算为什么能按固定节拍运行。

本章目标是：

- 理解 `SysTick_Handler()`、`frameCounter`、`frame_500Hz` 和 `main()` 主循环之间的触发关系。
- 明确当前项目的 500Hz 实时控制循环不是在中断中直接执行，而是在主循环中消费标志位后执行。
- 理解 `micros()`、`deltaTime500Hz` 和 `dt500Hz` 如何把实际时间间隔传给姿态解算和控制函数。
- 分清 500Hz 实时控制路径、10Hz 低频任务和 TIM6 初始化边界。
- 为第27章 PID 控制建立时间入口和执行顺序前置。

## 2. 前置知识

本章正式前置知识包括：

- `SysTick系统节拍`
- `DWT周期计数器`

第09章已经说明，项目用 1ms SysTick 节拍产生 `frame_500Hz` 标志。第10章已经说明，项目用 `micros()` 得到微秒级时间戳。第19章、第23章和第25章已经分别讲过 MPU6050 读取、姿态输出和 `sensors_t` 状态容器。

本章不展开 MPU6050 寄存器细节，不展开 `MargAHRSupdate()` 的四元数计算，也不展开 `computeMotorCommands()` 内部 PID 算法。它只回答一个调度问题：500Hz 这一帧里，谁先执行，谁后执行，时间间隔如何传递。

## 3. 问题背景

三轴云台需要不断读取 IMU、更新姿态、计算控制量并输出电机指令。如果这些动作没有稳定节拍，控制系统就会出现三个问题：

- 采样间隔不稳定，姿态积分的 `dt` 不可靠。
- 控制输出更新不稳定，电机补偿时快时慢。
- 调试时无法判断问题来自传感器、算法、控制器，还是任务调度。

但实时控制也不能简单全部塞进中断里。当前项目通过一种折中方式组织 500Hz 实时控制循环：

```text
SysTick 中断只负责置位 frame_500Hz
-> main() 主循环发现标志
-> 清除标志并计算 dt500Hz
-> 读取 IMU
-> 缩放并写入 sensors_t
-> 更新姿态
-> 执行启动门控
-> 调用 computeMotorCommands(dt500Hz)
```

这种结构的核心思想是：中断要短，主循环做重任务，实际 `dt` 用微秒时间戳测量，而不是假定每一帧一定精确等于 2ms。

## 4. 核心概念

### 4.1 1ms系统节拍

`SysTick` 在当前项目中配置为 1ms 一次。每次进入 `SysTick_Handler()`，代码先调用 `HAL_IncTick()`，使 HAL 的毫秒时间继续前进。

这一步来自第09章。第26章关心的是：项目如何从 1ms 节拍派生出 500Hz 实时控制入口。

### 4.2 frameCounter

`frameCounter` 是项目自己的帧计数器。`Core/Inc/main.h` 中定义：

```c
#define FRAME_COUNT   1000
#define COUNT_500HZ   2
```

`SysTick_Handler()` 在系统就绪且 MPU6050 不处于校准状态时递增 `frameCounter`。

当 `frameCounter > FRAME_COUNT` 时，它回到 1。这里的 `FRAME_COUNT=1000` 可以理解为一个 1秒范围内的 1ms 计数循环。

### 4.3 frame_500Hz

`frame_500Hz` 是 500Hz 实时控制循环触发标志。当前代码在 `SysTick_Handler()` 中执行：

```c
if ((frameCounter % COUNT_500HZ) == 0)
{
    frame_500Hz = true;
}
```

由于 `COUNT_500HZ=2`，而 SysTick 是 1ms 节拍，所以每两个 SysTick 周期置位一次，对应约 2ms 一帧，也就是 500Hz。

### 4.4 dt500Hz

`dt500Hz` 是当前 500Hz 帧与上一帧之间的实际时间间隔，单位为秒。`main()` 主循环进入 500Hz 实时控制分支后执行：

```c
currentTime = micros();
deltaTime500Hz = currentTime - previous500HzTime;
previous500HzTime = currentTime;
dt500Hz = (float)deltaTime500Hz / 1000000.0f;
```

它不是固定写死的 `0.002f`。这让姿态解算和控制函数能看到实际帧间隔。

### 4.5 executionTime500Hz

`executionTime500Hz` 记录 500Hz 实时控制分支本身耗费了多少微秒。它在分支末尾通过：

```c
executionTime500Hz = micros() - currentTime;
```

得到。这个值不是控制输入，而是性能观察量。调试时可以用它判断 500Hz 实时控制循环是否接近或超过 2ms 预算。

## 5. 工作原理

当前 500Hz 实时控制循环可以分成五层。

第一层，初始化建立时间基础。

`main()` 先调用 `HAL_Init()`、`SystemClock_Config()` 和各外设初始化函数。随后在用户初始化段中再次调用：

```c
HAL_SYSTICK_Config(SystemCoreClock / 1000);
HAL_NVIC_SetPriority(SysTick_IRQn, 0, 0);
```

这一步让 SysTick 保持 1ms 节拍。项目还启用 DWT 计数器，用于第10章讲过的 `DWT_Delay_us()` 等微秒等待路径；本章的 `dt500Hz` 则由 `micros()` 的 HAL/SysTick 时间戳路径计算。

第二层，系统就绪后中断开始产生帧标志。

`systemReady` 在初始化末尾置为 `true`。之后 `SysTick_Handler()` 只有在：

```c
(systemReady == true) && (mpu6050Calibrating == false)
```

成立时才推进 `frameCounter` 并可能置位 `frame_500Hz`。这意味着开机初始化阶段和 MPU6050 校准阶段不会正常推进 500Hz 实时控制帧。

第三层，主循环消费标志。

`main()` 的 `while (1)` 中检查：

```c
if (frame_500Hz)
{
    frame_500Hz = false;
    ...
}
```

注意，当前设计不是在中断中读 MPU6050 或直接算控制，而是在主循环中执行这些耗时任务。`stm32f1xx_it.c` 中保留的注释也说明，在中断中读 I2C 会耗时，并且与主循环重复读取可能造成总线冲突。

第四层，主循环按顺序执行控制链路。

当前 500Hz 实时控制分支中的顺序是：

```text
micros() 计算 dt500Hz
-> MPU6050_Read_And_Process()
-> 写入 sensors.accel500Hz / sensors.gyro500Hz
-> MargAHRSupdate(..., dt500Hz)
-> startup_delay_counter 启动门控
-> computeMotorCommands(dt500Hz)
-> executionTime500Hz 性能记录
```

这条顺序非常重要。`computeMotorCommands()` 读取的是已经更新后的 `sensors.margAttitude500Hz`。如果姿态更新没有在控制计算前执行，控制输入就会落后一帧或使用旧值。

第五层，10Hz 低频任务与 500Hz 实时控制循环并存。

500Hz 实时控制分支之后，主循环还用：

```c
if (HAL_GetTick() - last_print_tick >= 100)
```

执行约 10Hz 的低频状态处理和串口打印。这个 10Hz 低频任务和 500Hz 实时控制循环共用同一个 `while (1)`，但它不属于 500Hz 实时控制路径。分析实时控制时，不能把 10Hz 打印当成控制算法的一部分。

## 6. STM32实现机制

从 STM32 机制看，本章涉及三个层次。

第一是内核 SysTick。SysTick 是 Cortex-M3 内核提供的系统定时器。项目用它产生 1ms 中断，并通过 `HAL_IncTick()` 维持 HAL 时间基准。

第二是主循环轮询标志。中断侧只设置 `frame_500Hz`，主循环侧清除并执行任务。这种“中断置位、主循环消费”的结构能降低中断处理时间，避免在中断里做 I2C 读取和复杂计算。

第三是微秒时间测量。`micros()` 使用 `HAL_GetTick()` 和 `SysTick->VAL` 组合出微秒时间戳。500Hz 实时控制循环进入后用两次时间戳差得到 `deltaTime500Hz`，再换算成秒得到 `dt500Hz`。

项目还初始化了 TIM6，并在初始化末尾执行：

```c
__HAL_TIM_SET_COUNTER(&htim6, 0);
```

但当前 `frame_500Hz` 的置位证据在 `Core/Src/stm32f1xx_it.c` 的 `SysTick_Handler()` 中。`Core/Src/tim.c` 中能看到 TIM6 初始化，当前没有看到它产生 500Hz 实时控制循环触发标志或进入主控制循环的证据。因此本章不能把 TIM6 写成当前 500Hz 触发源。

## 8. 代码分析

### 8.1 main.h中的控制周期定义

`Core/Inc/main.h` 中和本章直接相关的是：

```c
uint32_t micros(void);
extern uint16_t frameCounter;
extern bool frame_500Hz;
extern float dt500Hz;

#define FRAME_COUNT   1000
#define COUNT_500HZ   2
```

`COUNT_500HZ` 是从 1ms 节拍得到 500Hz 实时控制循环触发标志的关键。`dt500Hz` 则是后续姿态解算和控制计算共用的时间步长。

### 8.2 SysTick_Handler只设置标志

`Core/Src/stm32f1xx_it.c` 中，`SysTick_Handler()` 的主线是：

```text
HAL_IncTick()
-> 检查 systemReady 和 mpu6050Calibrating
-> frameCounter++
-> frameCounter 超过 FRAME_COUNT 后回到 1
-> 每 COUNT_500HZ 帧置位 frame_500Hz
```

这里没有执行 `MPU6050_Read_And_Process()`。源码中保留的相关调用位于注释块内，这与第19章“中断不直接读 I2C”的结论一致。

### 8.3 main循环消费frame_500Hz

`Core/Src/main.c` 中真正的 500Hz 工作发生在：

```c
if (frame_500Hz)
{
    frame_500Hz = false;
    currentTime = micros();
    deltaTime500Hz = currentTime - previous500HzTime;
    previous500HzTime = currentTime;
    dt500Hz = (float)deltaTime500Hz / 1000000.0f;
    ...
}
```

先清除标志，再记录当前时间。这样下一次 SysTick 置位时，主循环可以再次进入分支。

### 8.4 一帧内的数据和控制顺序

进入 500Hz 实时控制分支后，顺序是固定的：

1. 调用 `MPU6050_Read_And_Process()` 读取并处理原始 IMU 数据。
2. 写入 `sensors.accel500Hz` 和 `sensors.gyro500Hz`。
3. 调用 `MargAHRSupdate(..., dt500Hz)` 更新 `sensors.margAttitude500Hz`。
4. 根据 `startup_delay_counter` 处理开机收敛门控。
5. 调用 `computeMotorCommands(dt500Hz)`。
6. 写入 `executionTime500Hz`。

本章只分析顺序，不展开第27章的 PID 内部计算。

### 8.5 启动门控的位置

`startup_delay_counter` 在 500Hz 实时控制分支内递增。计数达到 1000 时，项目清空 PID 积分和微分状态，并打开部分控制轴：

```text
1000帧 / 500Hz = 2秒
```

这说明开机前 2 秒主要用于等待 AHRS 姿态收敛。门控发生在姿态更新之后、控制计算之前，因此第27章分析 PID 时要注意：控制函数每帧仍被调用，但轴使能状态会影响实际输出路径。

### 8.6 TIM6边界

`Core/Src/main.c` 调用了 `MX_TIM6_Init()`，并在系统准备完成前清零 TIM6 计数器。`Core/Src/tim.c` 中也能看到 TIM6 的 `Prescaler` 和 `Period` 配置。

但当前没有发现 `HAL_TIM_Base_Start_IT(&htim6)`、TIM6 中断处理函数置位 `frame_500Hz`，或 TIM6 回调驱动主循环的证据。当前 500Hz 实时控制循环触发标志仍应追踪到 `SysTick_Handler()`。

### 8.7 10Hz低频任务边界

500Hz 实时控制分支之外，`main()` 还有 10Hz 低频任务。它处理状态计数、回中状态和串口打印。低频任务会消耗主循环时间，因此调试实时性时需要观察它是否影响 `executionTime500Hz` 和下一次 500Hz 实时控制分支进入时间。

但从数据流上看，10Hz 打印不是控制闭环的核心计算路径。

### 本节证据边界

本节只根据当前仓库说明文件、函数、宏、变量和调用关系。运行时频率、外部硬件表现、主机侧现象、传感器方向、电机响应或真实控制效果仍需调试记录、日志或仓库外实测证据；缺少证据时保持【待验证】。

## 9. 调试方法

第一步，确认 SysTick 是否产生 500Hz 实时控制循环触发标志。

- 在 `SysTick_Handler()` 中观察 `frameCounter`。
- 确认 `COUNT_500HZ` 为 2。
- 观察 `frame_500Hz` 是否大约每 2ms 置位一次。

第二步，确认主循环是否消费标志。

- 在 `Core/Src/main.c` 的 `if (frame_500Hz)` 内设置断点。
- 确认进入分支后 `frame_500Hz` 被清为 `false`。
- 确认 `previous500HzTime` 每帧更新。

第三步，观察时间间隔。

- 观察 `deltaTime500Hz` 是否接近 2000 微秒。
- 观察 `dt500Hz` 是否接近 `0.002f`。
- 如果 `dt500Hz` 偶尔明显偏大，继续观察是否有 10Hz 打印、I2C 读取或控制计算耗时。

第四步，观察执行耗时。

- 观察 `executionTime500Hz`。
- 如果它接近或超过 2000 微秒，说明 500Hz 实时控制循环已经接近超时或可能错过下一帧。
- 当前 `frame_500Hz` 是布尔标志，不是待处理帧计数。如果主循环处理不过来，标志只能保持为真，不能记录错过了几帧。

第五步，确认启动门控。

- 观察 `startup_delay_counter` 是否从 0 递增到 1000。
- 到达 1000 时确认 `zeroPIDintegralError()` 和 `zeroPIDstates()` 被调用。
- 确认轴使能状态变化符合当前调试意图。

第六步，排除 TIM6 误判。

- 在 `Core/Src/tim.c` 中确认 TIM6 只是被初始化。
- 搜索是否存在 TIM6 中断或回调置位 `frame_500Hz`。
- 如果没有相关证据，就继续按 SysTick 主线调试。

调试记录建议：

- 记录 SysTick 置位、主循环消费、`previous500HzTime` 更新和 `dt500Hz` 计算四段证据。
- 对耗时风险，应记录 `executionTime500Hz`、I2C 读取、姿态更新、PID 和电机输出各自观察点。
- 启动门控记录应包含计数值、PID 状态清零函数、轴使能变化和对应帧号。
- 若没有连续日志或外部计时证据，500Hz 周期稳定性和丢帧数量保持【待验证】。

## 10. 常见问题

问题一：500Hz 循环是不是直接在 SysTick 中断里执行。

不是。当前 SysTick 中断只置位 `frame_500Hz`，真正的 MPU6050 读取、姿态更新和控制计算发生在 `main()` 主循环中。

问题二：为什么不在中断里直接读 MPU6050。

I2C 读取耗时较长。当前源码注释已经说明，在中断中读 I2C 并和主循环重复读取，可能造成总线冲突和读数异常。因此当前实现把中断职责限制为短操作。

问题三：项目初始化了 TIM6，为什么不说 500Hz 来自 TIM6。

因为当前证据显示 `frame_500Hz` 在 `SysTick_Handler()` 中置位。TIM6 的初始化和清零存在，但没有看到它驱动 `frame_500Hz` 或主控制循环。

问题四：`dt500Hz` 为什么不用固定 0.002。

实际系统会有中断延迟、主循环耗时、I2C 读取耗时和低频任务干扰。使用 `micros()` 实测帧间隔能让姿态和控制函数看到更接近实际的时间步长。

问题五：如果 `executionTime500Hz` 超过 2000 微秒会怎样。

这说明单帧执行时间已经超过 500Hz 周期预算。当前 `frame_500Hz` 是布尔标志，不能累计错过帧数；如果长期超时，控制周期会变慢，`dt500Hz` 会变大，控制稳定性需要重新评估。

问题六：为什么 `systemReady` 会影响 500Hz 实时控制循环触发标志。

`SysTick_Handler()` 只有在系统初始化完成后才推进项目帧计数。这样可以避免外设和配置还没准备好时就进入传感器读取和控制流程。

问题七：10Hz 打印是否会影响 500Hz 实时控制循环。

它不属于 500Hz 实时控制路径，但它和 500Hz 实时控制分支同在主循环中。如果串口输出阻塞时间过长，可能间接影响下一次主循环消费 `frame_500Hz` 的及时性。

问题八：`dt500Hz` 实测后，是否就能保证 500Hz 周期不会丢帧。

不能。
`dt500Hz` 能记录两次主循环消费之间的实际时间差，
但当前 `frame_500Hz` 是布尔标志，不记录错过了几个节拍。
如果主循环长期阻塞，后续只会看到更大的时间间隔，而不是完整的积压帧队列。
所以调试时仍要同时观察执行时间、打印阻塞和 I2C 读取耗时。

## 13. 本章总结

本章把前面已经讲过的 SysTick、微秒计时、MPU6050 读取、姿态解算和状态容器组织成 500Hz 实时控制循环。

当前项目中：

- `Core/Inc/main.h` 定义 `FRAME_COUNT=1000` 和 `COUNT_500HZ=2`。
- `SysTick_Handler()` 在系统就绪且 MPU6050 不校准时递增 `frameCounter`。
- 每两个 1ms SysTick 周期置位一次 `frame_500Hz`。
- `main()` 主循环消费 `frame_500Hz`，并用 `micros()` 计算 `dt500Hz`。
- 500Hz 实时控制分支内依次执行 IMU 读取、传感器物理量写入、`MargAHRSupdate()`、启动门控和 `computeMotorCommands(dt500Hz)`。
- `executionTime500Hz` 用于观察单帧执行耗时。
- TIM6 当前有初始化和计数器清零证据，但没有作为 500Hz 触发源的证据。
- 10Hz 低频任务和 500Hz 实时控制循环同在主循环中，但不属于同一条控制计算链路。

目标回扣：

- 本章已经把 SysTick 标志、主循环消费、微秒计时和 `dt500Hz` 串成当前 500Hz 执行顺序。
- 本章已经分清 500Hz 实时控制路径、10Hz 低频任务和 TIM6 初始化边界。
- 本章已经为第27章 PID 控制提供时间入口和执行顺序前置。

本章保留两个工程边界：

- 当前 `frame_500Hz` 是布尔标志，如果主循环长期超过 2ms，无法累计错过帧数。
- 当前 `dt500Hz` 没有在进入姿态和控制函数前做上下限保护，后续异常防护章节需要继续审查。

下一章将进入 `PID控制核心`，在本章建立的 500Hz 时间步长和执行顺序基础上，分析 `computeMotorCommands(dt500Hz)` 如何把姿态角、目标角和 PID 参数组织成三轴控制输出。
