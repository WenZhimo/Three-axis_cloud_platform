# 第06章 系统时钟树

## 1. 本章目标

- 看懂本项目系统时钟从 HSE、PLL 到 SYSCLK、HCLK、PCLK1、PCLK2 和 USB 时钟的配置链路。
- 理解为什么 SysTick、TIM、I2C、UART 和 USB 都必须建立在正确的系统时钟前提上。
- 能从 `SystemClock_Config()`、`.ioc` 和 HAL RCC 源码中交叉验证项目实际频率。
- 为后续 MSP、GPIO 复用、中断节拍、定时器、通信和 USB 章节建立共同时钟基础。

## 2. 前置知识

- STM32F103RCTx芯片平台
- STM32 HAL驱动库

本章只使用第01章确认的平台前提和第04章确认的 HAL 构建前提。第05章已经建立 CMSIS 与寄存器访问边界，本章不会展开 DWT、SysTick 中断、定时器 PWM 或通信外设细节。

## 3. 问题背景

三轴云台项目里，时间不是一个抽象数字。姿态采样、控制循环、电机 PWM、串口输出和 USB 通信都依赖同一套时钟基础。

如果系统时钟理解错误，后续会出现一串连锁问题：

- `SystemCoreClock / 1000` 得不到预期的 1ms 节拍基础。
- 定时器的计数频率和 PWM 频率判断会偏离项目配置。
- I2C、UART 和 USB 的外设时钟来源无法解释。
- DWT 周期数、微秒延时和执行时间统计会失去频率前提。

因此第06章要解决的问题不是“项目是不是 72MHz”这么简单，而是要说明：这个 72MHz 从哪里来，经过哪些分频，如何传递给总线和外设，以及项目文件如何证明这些结论。

## 4. 核心概念

- 系统时钟树：从振荡源、PLL、系统时钟选择、总线分频到外设时钟的层级关系。
- HSE：外部高速时钟源。本项目配置为开启 HSE，并以 HSE 作为 PLL 来源。
- HSI：内部高速时钟源。本项目 `SystemClock_Config()` 中保持 HSI 开启，但系统时钟来源选择的是 PLLCLK。
- PLL：锁相环，用于把输入时钟倍频到目标系统频率。
- SYSCLK：系统时钟来源，本项目选择 PLLCLK。
- HCLK：AHB 总线时钟，也是 `SystemCoreClock` 表示的核心时钟频率。
- PCLK1：APB1 总线时钟，本项目配置为 HCLK 二分频。
- PCLK2：APB2 总线时钟，本项目配置为 HCLK 不分频。
- APB定时器时钟：当 APB 分频不为 1 时，对应定时器时钟可能不同于该 APB 总线时钟。本项目 `.ioc` 明确给出 APB1 与 APB2 定时器时钟均为 72MHz。
- USB时钟：USB 外设需要独立的外设时钟选择，本项目选择 PLLCLK 经过 1.5 分频得到 48MHz。
- `SystemCoreClock`：CMSIS/HAL 使用的核心时钟变量，本项目后续用它配置 SysTick 周期。
- Flash等待周期：CPU 访问 Flash 时需要的等待状态，本项目 `HAL_RCC_ClockConfig()` 使用 `FLASH_LATENCY_2`。
- 时钟源选择：决定某一级时钟从哪里来，例如 SYSCLK 选择 PLLCLK。
- 时钟门控使能：决定某个外设是否获得时钟输入，例如 `__HAL_RCC_TIM2_CLK_ENABLE()`。
- 配置频率：由源码、`.ioc` 和 HAL 计算得到的频率。
- 实测频率：由调试器、MCO 输出、示波器、逻辑分析仪或 USB 主机观察得到的频率。

这些概念都服务于正式知识点 `系统时钟树`，不新增结构外知识点。

## 5. 工作原理

本项目的时钟树可以从输入、倍频、选择、分频和外设分支五步理解。

第一步是输入时钟。

`Core/Inc/stm32f1xx_hal_conf.h` 中 `HSE_VALUE` 为 `8000000U`。

`SystemClock_Config()` 开启 HSE，并设置 HSE 预分频为 1。因此进入 PLL 的 HSE 基础值按项目配置为 8MHz。

当前仓库没有独立硬件设计文件确认实际外部时钟器件，仓库外实测频率仍应标记为【待验证】；教材只确认工程配置中的 HSE 值。

第二步是 PLL 倍频。

`SystemClock_Config()` 设置 PLL 来源为 HSE，倍频为 `RCC_PLL_MUL9`。按工程配置计算：

- PLL 输入：8MHz
- PLL 倍频：9
- PLLCLK：72MHz

`.ioc` 中 `RCC.PLLCLKFreq_Value=72000000` 与这一计算一致。

这一段也可以写成公式：

```text
f_PLLCLK = f_HSE / HSEPrediv * PLLMUL
         = 8 MHz / 1 * 9
         = 72 MHz
```

第三步是系统时钟选择。

`SystemClock_Config()` 设置 `SYSCLKSource = RCC_SYSCLKSOURCE_PLLCLK`。

`.ioc` 中 `RCC.SYSCLKSource=RCC_SYSCLKSOURCE_PLLCLK`。因此项目系统时钟来源是 PLLCLK，而不是 HSE 直连或 HSI。

第四步是总线分频。

项目配置为：

- AHB：`RCC_SYSCLK_DIV1`，所以 HCLK 为 72MHz。
- APB1：`RCC_HCLK_DIV2`，所以 PCLK1 为 36MHz。
- APB2：`RCC_HCLK_DIV1`，所以 PCLK2 为 72MHz。

对应公式是：

```text
f_SYSCLK = f_PLLCLK = 72 MHz
f_HCLK   = f_SYSCLK / 1 = 72 MHz
f_PCLK1  = f_HCLK / 2   = 36 MHz
f_PCLK2  = f_HCLK / 1   = 72 MHz
```

`.ioc` 中 `RCC.AHBFreq_Value=72000000`、`RCC.APB1Freq_Value=36000000`、`RCC.APB2Freq_Value=72000000` 与代码一致。

对后续定时器分析还有一个关键点：APB1 预分频不为 1 时，对应定时器时钟会高于 APB1 总线时钟。

本项目 APB1 为 36MHz，但 `.ioc` 记录 APB1 定时器时钟为 72MHz。

这个差值必须在后续定时器章节使用，而不是在第06章展开到 PSC、ARR 和 CCR。

第五步是外设分支。

本章只建立时钟前提，不展开外设配置。当前可确认的关键分支是：

- APB1 定时器时钟：`.ioc` 记录为 72MHz。
- APB2 定时器时钟：`.ioc` 记录为 72MHz。
- USB 时钟：`SystemClock_Config()` 选择 `RCC_USBCLKSOURCE_PLL_DIV1_5`，`.ioc` 记录为 48MHz。

这样，项目形成了一条清晰链路：8MHz HSE 进入 PLL，倍频到 72MHz，作为 SYSCLK，再分发到 AHB、APB1、APB2 和 USB 分支。

### 5.1 证据层级

本章的“72MHz、36MHz、48MHz”不是同一种证据。为了避免把工程配置误写成实物测量，教材按四层理解：

```text
源码配置证据
-> HAL 计算证据
-> 运行时寄存器/变量观察
-> 目标板物理测量或主机侧现象
```

- 源码配置证据：`SystemClock_Config()` 和 `.ioc` 说明工程希望如何配置时钟。
- HAL 计算证据：`HAL_RCC_GetSysClockFreq()`、`HAL_RCC_GetPCLK1Freq()` 和 `HAL_RCC_GetPCLK2Freq()` 说明 HAL 如何从寄存器与常量计算频率。
- 运行观察证据：调试器读取 `RCC->CFGR`、`RCC->CR`、`SystemCoreClock`、`SysTick->LOAD` 和 `FLASH->ACR`。
- 物理证据：MCO 输出测量、外部晶振测量、USB 主机枚举和外部波形记录。

当前仓库能充分证明前两层，能给出运行观察的检查对象，但没有现场调试截图、MCO 波形或主机侧日志。因此实测频率、USB 枚举成功和外设实际波形仍保持【待验证】。

### 5.2 时钟源选择与时钟门控

第06章讲的是时钟来源和分频，不等于每个外设已经获得了工作时钟。

例如：

- `SystemClock_Config()` 决定 TIM2 所在的 APB1 定时器时钟前提。
- `Core/Src/tim.c` 中 `__HAL_RCC_TIM2_CLK_ENABLE()` 才打开 TIM2 外设时钟门。
- `TIM_CLOCKSOURCE_INTERNAL` 说明定时器使用内部时钟，而不是外部触发时钟。

所以后续分析定时器时要分两步：

```text
时钟树给出频率前提
-> MSP/初始化代码使能外设并选择工作模式
```

只有前者，不能证明外设已经运行；只有后者，也不能单独推导频率。

## 6. STM32实现机制

STM32F1 工程里，系统时钟配置主要由 RCC、FLASH 等机制共同完成。

在本项目中，`SystemClock_Config()` 使用三个结构体：

- `RCC_OscInitTypeDef`：配置 HSE、HSI 和 PLL。
- `RCC_ClkInitTypeDef`：配置 SYSCLK 来源、AHB 分频、APB1 分频和 APB2 分频。
- `RCC_PeriphCLKInitTypeDef`：配置 USB 外设时钟来源。

调用顺序也有工程意义：

1. `HAL_RCC_OscConfig()` 先配置并启动振荡源和 PLL。
2. `HAL_RCC_ClockConfig()` 再切换 SYSCLK、设置总线分频，并使用 `FLASH_LATENCY_2`。
3. `HAL_RCCEx_PeriphCLKConfig()` 最后配置 USB 外设时钟分支。

`Drivers/STM32F1xx_HAL_Driver/Src/stm32f1xx_hal_rcc.c` 中可以看到 `HAL_RCC_ClockConfig()` 会更新 `SystemCoreClock`。

其值来自 `HAL_RCC_GetSysClockFreq()` 再结合 AHB 分频。

也就是说，本项目后面调用 `HAL_SYSTICK_Config(SystemCoreClock / 1000)` 时，依赖的是 `SystemClock_Config()` 已经完成时钟配置并更新后的核心时钟值。

`Core/Src/system_stm32f1xx.c` 提供 `SystemCoreClock` 初值和 `SystemCoreClockUpdate()` 函数。

当前项目主流程没有直接调用 `SystemCoreClockUpdate()`。

第06章把它作为 CMSIS 系统时钟维护机制的证据，而不是项目主流程中的实际调用。

还需要注意 HAL 的默认时间基准流程：

- `HAL_Init()` 会调用 `HAL_InitTick(TICK_INT_PRIORITY)`，此时默认复位时钟仍以 HSI 为基础。
- `HAL_RCC_ClockConfig()` 完成时钟切换后，会再次调用 `HAL_InitTick(uwTickPrio)` 适配新的 `SystemCoreClock`。
- 项目用户代码随后又显式调用 `HAL_SYSTICK_Config(SystemCoreClock / 1000)`。

这说明 SysTick 配置在启动过程中不是“一次写完永不改变”。调试 SysTick 时要看最后一次配置后的 `SysTick->LOAD`、`SysTick->CTRL` 和优先级设置。

如果 `SystemCoreClock` 为 72MHz，传给 `HAL_SYSTICK_Config()` 的参数为 72000。
CMSIS 的 `SysTick_Config()` 会把 `SysTick->LOAD` 写为 `ticks - 1`。
因此，预期重装载寄存器为 71999。

这个数值是配置推导结果，不等于已经证明 1ms 中断真实按 1kHz 发生。真实节拍还要观察 `HAL_IncTick()`、`HAL_GetTick()`、`SysTick->CTRL` 和调试记录。

### 6.1 工程选型含义

本项目配置为 HSE + PLL + 72MHz SYSCLK。从工程角度看，这种选择有几个直接后果：

- 72MHz 给 500Hz 控制循环、DWT 计时、PWM 输出和串口打印留下更高 CPU 预算。
- APB1 采用 HCLK/2，使 PCLK1 为 36MHz，符合 HAL RCC 源码注释中 STM32F10xxx 的 PCLK1 上限。
- APB2 保持 72MHz，给 GPIO 复用、高级定时器和 USART1 等 APB2 外设保留较高时钟前提。
- USB 选择 PLL/1.5，正好从 72MHz 得到 48MHz USB 时钟。

为什么不是 HSI 直接作为 SYSCLK？
当前项目没有写明设计说明，但从代码和 HAL 注释可判断：HSI 受电压和温度影响。
同时，HSI 无法直接给出当前 USB 48MHz 分支。
项目配置选择 HSE 进入 PLL，更适合作为全系统和 USB 的共同频率基础。
目标板实际晶振准确性仍需【待验证】。

为什么不是更低 SYSCLK？降低主频可以降低功耗和时钟压力，但会减少 500Hz 控制循环、DWT 微秒计时、软件滤波和调试输出的时间预算。本项目当前配置明显优先保证控制与外设时序余量，而不是最低功耗。

## 7. 项目中的应用

本章对应项目运行流程中的系统配置层。

直接相关文件：

- `Core/Src/main.c`
- `Three-axis_cloud_platformV2.ioc`
- `Core/Src/system_stm32f1xx.c`
- `Core/Inc/stm32f1xx_hal_conf.h`
- `Drivers/STM32F1xx_HAL_Driver/Src/stm32f1xx_hal_rcc.c`
- `Drivers/STM32F1xx_HAL_Driver/Src/stm32f1xx_hal_rcc_ex.c`
- `Drivers/STM32F1xx_HAL_Driver/Src/stm32f1xx_hal.c`
- `Drivers/STM32F1xx_HAL_Driver/Src/stm32f1xx_hal_cortex.c`
- `Core/Src/tim.c`
- `Core/Src/i2c.c`
- `Core/Src/usart.c`
- `USB_DEVICE/Target/usbd_conf.c`

它们之间的关系是：

- `main.c` 中 `SystemClock_Config()` 是项目时钟配置入口。
- `.ioc` 保存 CubeMX 生成的 RCC 频率结果，可用于核对代码中的时钟选择和分频。
- `stm32f1xx_hal_conf.h` 提供 `HSE_VALUE` 与 `HSI_VALUE` 等频率常量。
- `system_stm32f1xx.c` 提供 `SystemCoreClock` 变量和更新函数。
- `stm32f1xx_hal_rcc.c` 提供 `HAL_RCC_OscConfig()`、`HAL_RCC_ClockConfig()` 和系统频率计算逻辑。
- `stm32f1xx_hal_rcc_ex.c` 提供 USB 外设时钟分支配置和 USB 频率计算逻辑。
- `stm32f1xx_hal.c` 和 `stm32f1xx_hal_cortex.c` 说明 HAL tick 与 SysTick 配置如何依赖 `SystemCoreClock`。
- `tim.c` 说明 TIM2、TIM3、TIM4、TIM6 和 TIM8 后续将使用本章建立的定时器时钟前提。
- `i2c.c` 说明 I2C2 配置会进一步依赖 APB1/PCLK1。
- `usart.c` 说明 USART3 波特率寄存器计算会依赖 APB1/PCLK1。
- `usbd_conf.c` 说明 USB 外设时钟门会被使能，但不证明主机枚举成功。

在 `main()` 的执行顺序中，项目先调用 `HAL_Init()`，再调用 `SystemClock_Config()`。

随后代码初始化 GPIO、TIM、I2C、UART、USB 等模块。

最后在用户初始化区域调用 `HAL_SYSTICK_Config(SystemCoreClock / 1000)`，并启用 DWT 周期计数器。

这说明系统时钟位于所有外设和时间基准之前。后续章节分析任何周期、波特率、总线速度或 PWM 频率，都必须先回到本章确认的时钟树。

从后续章节看，本章提供的频率前提会被这些模块消费：

- SysTick：`HAL_SYSTICK_Config(SystemCoreClock / 1000)`。
- DWT 延时：`us * (SystemCoreClock / 1000000)`。
- TIM2/TIM3/TIM4 PWM：`PSC=0`、`ARR=3599` 的频率推导需要 72MHz 定时器时钟。
- TIM6：`PSC=35` 的计数频率推导依赖 APB1 定时器时钟。
- TIM8：`PSC=3`、`ARR=999` 的更新频率推导依赖 APB2 定时器时钟。
- I2C2：HAL I2C 初始化调用 `HAL_RCC_GetPCLK1Freq()` 计算 `CR2`、`CCR` 和 `TRISE`。
- USART3：HAL UART 初始化调用 `HAL_RCC_GetPCLK1Freq()` 计算 `BRR`。
- USB：`RCC_USBCLKSOURCE_PLL_DIV1_5` 给 USB 分支提供 48MHz 配置前提。

## 8. 代码分析

### 1. `SystemClock_Config()` 的入口位置

`main.c` 在 `HAL_Init()` 之后调用 `SystemClock_Config()`。这意味着 HAL 基础初始化先建立最低限度运行环境，项目最终时钟树随后由 `SystemClock_Config()` 切换到目标配置。

`SystemClock_Config()` 不接收参数，也不返回频率值。它通过写 RCC 配置结构体并调用 HAL RCC 函数改变芯片时钟状态；如果配置失败，就进入 `Error_Handler()`。

### 2. 振荡源与 PLL 配置

`RCC_OscInitStruct.OscillatorType = RCC_OSCILLATORTYPE_HSE`，说明本次主要配置 HSE。

`HSEState = RCC_HSE_ON`、`HSEPredivValue = RCC_HSE_PREDIV_DIV1`、`PLLSource = RCC_PLLSOURCE_HSE`、`PLLMUL = RCC_PLL_MUL9` 共同说明一条配置链路。

按当前仓库配置，项目使用 HSE 不分频后送入 PLL，并通过 9 倍频得到 72MHz PLLCLK。

`HSIState = RCC_HSI_ON` 表明 HSI 保持开启，但系统时钟来源仍由后面的 `SYSCLKSource` 决定。本章不能把 HSI 写成项目 SYSCLK 来源。

### 3. SYSCLK、AHB 和 APB 分频

`RCC_ClkInitStruct.SYSCLKSource = RCC_SYSCLKSOURCE_PLLCLK`，说明系统时钟选择 PLLCLK。

`AHBCLKDivider = RCC_SYSCLK_DIV1`，所以 HCLK 等于 SYSCLK。项目 `.ioc` 记录 HCLK 为 72MHz。

`APB1CLKDivider = RCC_HCLK_DIV2`，所以 PCLK1 为 36MHz。`APB2CLKDivider = RCC_HCLK_DIV1`，所以 PCLK2 为 72MHz。

`.ioc` 还记录 `RCC.APB1TimFreq_Value=72000000` 和 `RCC.APB2TimFreq_Value=72000000`。这为后续定时器章节建立前提，但本章不展开 PSC、ARR、CCR 和 PWM 模式。

### 4. Flash 延迟

`HAL_RCC_ClockConfig()` 的第二个参数是 `FLASH_LATENCY_2`。这表示项目在切换到当前高速时钟配置时，同时设置 Flash 等待周期。

`Drivers/STM32F1xx_HAL_Driver/Src/stm32f1xx_hal_rcc.c` 的注释中给出 SYSCLK 与 Flash latency 的对应关系。

当 SYSCLK 位于 48MHz 到 72MHz 区间时，对应 2WS。当前项目 SYSCLK 配置为 72MHz，因此 `FLASH_LATENCY_2` 与项目内 HAL 源码依据一致。

更严格的器件级依据仍应回到 STM32F103 数据手册或参考手册复核；本章先用当前仓库中的 HAL 源码和项目配置建立工程内证据链。

如果 Flash latency 设置偏低，CPU 从 Flash 取指或读常量时可能不稳定。仓库源码只能证明项目配置了 `FLASH_LATENCY_2`，不能证明目标板在所有电压、温度和批次条件下都留有裕量。

### 5. USB 时钟分支

`PeriphClkInit.PeriphClockSelection = RCC_PERIPHCLK_USB`，说明本项目单独配置 USB 外设时钟。

`UsbClockSelection = RCC_USBCLKSOURCE_PLL_DIV1_5`，配合 72MHz PLLCLK 得到 48MHz USB 时钟。`.ioc` 中 `RCC.USBFreq_Value=48000000` 与此一致。

`stm32f1xx_hal_rcc_ex.c` 中 USB 分支会把该选择写入 RCC 配置，并在频率计算函数中按 `PLLCLK * 2 / 3` 得到 48MHz。

USB 48MHz 是枚举和通信的必要前提之一，但不是充分条件。USB 还需要引脚、上拉、电源、描述符、中断和主机侧枚举共同成立；这些留到 USB 章节验证。

这里仍然只讲时钟来源，不展开 USB Device 初始化、描述符或 CDC 通信。

### 6. `SystemCoreClock` 的更新路径

`system_stm32f1xx.c` 中 `SystemCoreClock` 初始值为 8MHz。项目进入 `SystemClock_Config()` 后，`HAL_RCC_ClockConfig()` 会在 HAL RCC 源码中更新该变量。

这解释了为什么 `main.c` 后续可以用 `SystemCoreClock / 1000` 配置 1ms SysTick 基础：它不是直接使用初始 8MHz，而是在时钟配置后使用更新后的核心时钟值。

`HAL_RCC_GetSysClockFreq()` 和 `SystemCoreClockUpdate()` 的注释都提醒：
计算频率基于 `HSE_VALUE`、`HSI_VALUE` 和当前选择的时钟源。
它们不是频率计。
若外部晶振与 `HSE_VALUE` 不一致，软件计算值会随配置自洽，但真实频率会偏离。

### 7. APB总线与外设计算入口

HAL RCC 提供 `HAL_RCC_GetPCLK1Freq()` 和 `HAL_RCC_GetPCLK2Freq()`。它们从 `SystemCoreClock` 出发，再根据 `RCC->CFGR` 中 APB 分频位计算 PCLK。

在本项目中：

- I2C2 位于 APB1，HAL I2C 初始化读取 `HAL_RCC_GetPCLK1Freq()`。
- USART3 位于 APB1，HAL UART 初始化读取 `HAL_RCC_GetPCLK1Freq()`。
- TIM2、TIM3、TIM4 和 TIM6 属于 APB1 定时器，`.ioc` 给出的定时器时钟为 72MHz。
- TIM8 属于 APB2 定时器，`.ioc` 给出的定时器时钟为 72MHz。

因此，后续不能把所有外设都简单说成“用 72MHz”。不同外设使用 SYSCLK、HCLK、PCLK 或定时器时钟，入口不同。

## 9. 调试方法

本章阶段的调试目标是确认项目时钟树配置与预期一致，而不是调外设功能。

可观察对象：

- `main.c` 中 `SystemClock_Config()` 是否在外设初始化前调用。
- `SystemClock_Config()` 中 HSE、PLL 来源、PLL 倍频和 SYSCLK 来源是否与 `.ioc` 一致。
- `.ioc` 中 `RCC.SYSCLKFreq_VALUE` 是否为 72MHz。
- `.ioc` 中 `RCC.APB1Freq_Value` 是否为 36MHz，`RCC.APB2Freq_Value` 是否为 72MHz。
- `.ioc` 中 `RCC.APB1TimFreq_Value` 和 `RCC.APB2TimFreq_Value` 是否为 72MHz。
- `.ioc` 中 `RCC.USBFreq_Value` 是否为 48MHz。
- `stm32f1xx_hal_conf.h` 中 `HSE_VALUE` 是否为 8MHz。
- `stm32f1xx_hal_rcc.c` 的 Flash latency 注释表是否支持 72MHz SYSCLK 使用 `FLASH_LATENCY_2`。
- `main.c` 中 `HAL_SYSTICK_Config(SystemCoreClock / 1000)` 是否发生在 `SystemClock_Config()` 之后。
- `stm32f1xx_hal.c` 中 `HAL_InitTick()` 是否也会按 `SystemCoreClock` 配置 HAL tick。
- `stm32f1xx_hal_cortex.c` 中 `HAL_SYSTICK_Config()` 是否调用 `SysTick_Config()`。
- `core_cm3.h` 中 `SysTick_Config()` 是否写 `SysTick->LOAD = ticks - 1`。
- `stm32f1xx_hal_rcc_ex.c` 中 USB 频率是否按 PLL 分支计算。
- `tim.c` 中 `__HAL_RCC_TIMx_CLK_ENABLE()` 是否只是外设时钟门控，不是频率来源。
- `i2c.c`、`usart.c` 中外设初始化是否通过 HAL 间接读取 PCLK。

常见异常定位：

- 系统节拍不准：先检查 `SystemClock_Config()` 是否被调用，再检查 `SystemCoreClock` 更新路径。
- `SysTick->LOAD` 不是 71999：检查最后一次 `HAL_SYSTICK_Config()` 的参数和 `SystemCoreClock` 当前值。
- 定时器频率推导不一致：先核对 `.ioc` 的 APB 定时器时钟，再进入定时器章节分析 PSC 与 ARR。
- USB 时钟异常：先核对 PLLCLK 是否为 72MHz，再核对 USB 分频是否为 1.5。
- 代码和 `.ioc` 频率不一致：以当前源码和当前 `.ioc` 交叉核对，不能只看其中一个文件。
- 修改 HSE 常量后频率仍异常：检查 `HSE_VALUE`、仓库外实测频率【待验证】和 PLL 配置是否同时一致。
- `HAL_RCC_ClockConfig()` 返回错误：检查 HSE ready、PLL ready、SYSCLK switch、Flash latency 和超时路径。
- I2C 或 UART 配置异常：先确认 PCLK1 计算值，再进入对应通信章节分析寄存器。
- USB 主机不识别：48MHz USB 时钟只是前置条件，还要在 USB 章节检查引脚、描述符、中断和主机日志。

调试记录建议：

- 记录源码配置值、`.ioc` 频率值、HAL 更新变量和推导结果。
- 记录 `RCC->CR` 中 HSE/PLL ready 位，`RCC->CFGR` 中 SW/SWS、HPRE、PPRE1、PPRE2 和 USBPRE 位。
- 记录 `SystemCoreClock`、`HAL_RCC_GetHCLKFreq()`、`HAL_RCC_GetPCLK1Freq()` 和 `HAL_RCC_GetPCLK2Freq()` 的观察值。
- 记录 `FLASH->ACR` latency 位和 `SysTick->LOAD`。
- 若通过 MCO 或示波器测量时钟，记录测量点、分频设置、仪器截图和误差范围。
- 对时钟异常，先分清工程配置证据、运行时变量观察和仓库外实测频率。
- 对 USB、定时器或 SysTick 的后续问题，只在本章记录共同频率前提，细节转入对应章节。

## 10. 常见问题

### 1. 为什么项目不直接从 `main()` 的外设初始化开始分析？

因为外设初始化使用的总线时钟已经由 `SystemClock_Config()` 决定。先跳到 TIM、I2C、UART 或 USB，会失去频率前提。

定时器周期、串口波特率、I2C 时钟、USB 48MHz 和 SysTick 都依赖时钟树。第06章先建立系统频率和总线频率，后续章节才能判断参数是否有意义，而不是只背初始化函数名。

### 2. HSI 开启是否表示系统时钟来自 HSI？

不表示。项目代码中 `HSIState = RCC_HSI_ON` 只说明 HSI 保持开启；系统时钟来源由 `SYSCLKSource = RCC_SYSCLKSOURCE_PLLCLK` 决定。

判断系统时钟来源要看选择字段，而不是只看某个振荡器是否开启。当前仓库内证据指向 HSE 进入 PLL，PLL 输出作为 SYSCLK；HSI 开启只能作为备用或内部机制相关状态记录。

### 3. 为什么 HSE 8MHz 倍频 9 后是 72MHz？

工程配置中 `HSE_VALUE` 为 8MHz，HSE 预分频为 1，PLL 倍频为 9，所以 PLLCLK 为 72MHz。`.ioc` 的 `RCC.PLLCLKFreq_Value=72000000` 与这一结果一致。

这个结论属于仓库内配置与源码交叉验证。实际板卡外部晶振是否真为 8MHz，还需要硬件资料或外部测量支撑；缺少这些证据时，实物频率仍保持【待验证】。

### 4. 为什么 APB1 是 36MHz，但 APB1 定时器时钟是 72MHz？

这是 STM32 时钟树中 APB 分频与定时器时钟关系造成的结果。本项目 `.ioc` 已明确记录 APB1 总线为 36MHz、APB1 定时器时钟为 72MHz。定时器如何使用这个频率，留到后续定时器章节展开。

这也是为什么不能只看 PCLK1 就推导 PWM 频率。定时器章节需要使用“定时器时钟”而不是普通总线时钟作为输入，再结合 PSC、ARR、CCR 计算输出行为。

### 5. `SystemCoreClockUpdate()` 是否在项目主流程中被调用？

当前主流程中没有直接调用它。
项目通过 `HAL_RCC_ClockConfig()` 更新 `SystemCoreClock`。
`SystemCoreClockUpdate()` 在本章中作为 CMSIS 系统时钟维护机制出现，不作为项目主流程调用点。

因此调试时要区分“机制存在”和“当前调用路径”。
如果只搜索到函数定义，不能写成主流程调用。
如果观察 `SystemCoreClock` 数值，也要记录它是在 `HAL_RCC_ClockConfig()` 后被更新，
而不是由未调用函数自动完成。

### 6. 为什么第06章不展开具体外设的配置参数？

因为本章只建立共同时钟前提。外设初始化、引脚复用、中断、PWM、通信和 USB 中间件都有各自章节，提前展开会破坏教学顺序。

本章给后续章节提供频率依据：哪些频率来自 SYSCLK，哪些来自 APB，哪些需要专用 48MHz。具体外设是否工作、波形是否正确、主机是否识别，仍必须在对应章节结合源码和实测证据验证。

### 7. `HAL_Init()` 已经配置 SysTick，为什么后面还要看 `SystemClock_Config()`？

`HAL_Init()` 会在默认复位时钟基础上配置 HAL tick。随后 `SystemClock_Config()` 切换到项目目标时钟，`HAL_RCC_ClockConfig()` 又会调用 `HAL_InitTick()` 适配新频率。

本项目用户代码后续还显式调用 `HAL_SYSTICK_Config(SystemCoreClock / 1000)`。因此调试时应看最后一次配置后的 `SysTick->LOAD`，不能只看 `HAL_Init()` 阶段。

### 8. `SystemCoreClock` 是不是等于示波器测到的频率？

不一定。`SystemCoreClock` 是软件变量，依据寄存器配置和 `HSE_VALUE`、`HSI_VALUE` 计算。

如果外部晶振真实频率、焊接状态或时钟源状态与工程假设不一致，软件变量可能仍然显示 72MHz，但实物频率并不正确。实测需要 MCO、示波器、逻辑分析仪或调试记录。

### 9. 外设时钟使能是否等于外设频率已经正确？

不等于。`__HAL_RCC_TIM2_CLK_ENABLE()` 只是打开 TIM2 时钟门。

TIM2 的频率前提来自系统时钟树和 APB1 定时器时钟；TIM2 是否按预期计数，还要看定时器初始化、计数器启动、中断或 PWM 输出状态。

### 10. 为什么 USB 48MHz 不能直接证明 USB 可用？

USB 48MHz 只是 USB 外设工作的必要频率条件。项目还需要 USB 外设时钟门、GPIO、USB Device 中间件、描述符、端点、中断和主机侧枚举共同成立。

本章只能证明 `RCC_USBCLKSOURCE_PLL_DIV1_5` 配置和 `.ioc` 48MHz 结果。USB 是否枚举成功保持到 USB 章节验证。

## 13. 本章总结

本章建立了三轴云台项目的系统时钟树。

已经确认的结论是：

- 工程配置中的 HSE 值为 8MHz。
- 项目使用 HSE 作为 PLL 来源，PLL 倍频为 9，得到 72MHz PLLCLK。
- SYSCLK 来源为 PLLCLK。
- AHB/HCLK 为 72MHz。
- APB1/PCLK1 为 36MHz，APB2/PCLK2 为 72MHz。
- `.ioc` 记录 APB1 与 APB2 定时器时钟均为 72MHz。
- USB 时钟由 PLLCLK 经过 1.5 分频得到 48MHz。
- `HAL_RCC_ClockConfig()` 会更新 `SystemCoreClock`，后续 SysTick 配置依赖该变量。
- `HAL_Init()`、`HAL_RCC_ClockConfig()` 和用户区 `HAL_SYSTICK_Config()` 都可能参与 SysTick 配置。
- 按 72MHz HCLK 推导，最终 1ms SysTick 配置对应 `SysTick->LOAD=71999`。
- I2C2 和 USART3 的初始化分别依赖 APB1/PCLK1 计算寄存器。
- TIM2/TIM3/TIM4/TIM6/TIM8 的周期推导必须使用 `.ioc` 给出的定时器时钟前提。

本章边界：

- 本章基于 `.ioc` 和 `SystemClock_Config()` 建立仓库内时钟配置结论。
- HSE 实物频率、目标板时钟稳定性和外设实际波形仍属于仓库外实测证据。
- `SystemCoreClock`、PCLK 和 USB 频率函数是软件计算值，不是物理测量值。
- 外设时钟门控语句只能证明外设获得时钟输入条件，不证明外设功能已经正确运行。

下一章可以进入 MSP 初始化与 GPIO 复用，因为系统时钟已经建立，后续才能解释外设时钟使能、引脚复用和外设初始化之间的关系。
