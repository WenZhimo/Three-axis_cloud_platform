# 第03章 启动流程与内存布局

## 1. 本章目标

- 建立复位、启动文件、链接脚本和 `main()` 入口之间的结构关系。
- 看懂链接脚本如何描述 FLASH、RAM、栈、堆和关键段位置。
- 看懂 `Reset_Handler` 如何调用 `SystemInit()`、初始化 `.data`、清零 `.bss`、调用 C 运行时初始化并进入 `main()`。
- 为后续 Newlib 适配、堆栈检查、中断系统和运行时初始化建立前置基础。

## 2. 前置知识

- STM32F103RCTx芯片平台
- STM32CubeIDE工程配置
- 链接脚本与内存布局

这些前置主要对应第01章、第02章和第03章。

本章内部顺序为：先讲 `链接脚本与内存布局`，再讲 `Cortex-M3内核启动`。因为启动文件使用的 `_estack`、`_sidata`、`_sdata`、`_edata`、`_sbss` 和 `_ebss` 都来自链接脚本。

## 3. 问题背景

前两章已经确认项目目标平台和 STM32CubeIDE 工程配置。现在要回答一个更靠近运行时的问题：芯片复位后，程序是怎样从一片 FLASH 镜像走到 `main()` 的？

在普通 C 程序里，读者常常从 `main()` 开始看代码。但在 STM32 项目中，`main()` 不是 CPU 执行的第一条项目代码。复位后，CPU 先依赖向量表取得初始栈顶和复位入口，再进入启动文件。启动文件完成最小运行环境准备后，才会跳转到 `main()`。

本项目中，这条链路由四个文件共同证明：

- `STM32F103RCTX_FLASH.ld` 定义入口、内存区域和段布局。
- `Core/Startup/startup_stm32f103rctx.s` 定义向量表和 `Reset_Handler`。
- `Core/Src/system_stm32f1xx.c` 提供 `SystemInit()`。
- `Core/Src/main.c` 提供应用入口 `main()`。

本章只讲启动链路和内存布局，不展开系统时钟树、外设初始化和业务循环。

## 4. 核心概念

- 链接脚本与内存布局：描述程序各段放入 FLASH/RAM 的规则，并向启动文件提供边界符号。
- Cortex-M3内核启动：复位后由向量表、初始栈顶、复位入口和启动代码共同完成的上电入口过程。
- 向量表：存放初始栈顶、复位入口和异常/中断入口地址的表。
- `Reset_Handler`：复位后进入的启动函数。
- `_estack`：链接脚本计算出的 RAM 最高地址，用作初始栈顶。
- `.data`：需要从 FLASH 拷贝到 RAM 的已初始化数据段；它同时具有 FLASH 中的加载地址和 RAM 中的运行地址。
- `.bss`：启动时需要清零的未初始化数据段。
- 加载地址与运行地址：加载地址是镜像烧录后初值所在的位置，运行地址是 CPU 读写变量时使用的位置。本项目 `.data` 的加载地址来自 FLASH，运行地址位于 RAM。
- C 运行时初始化：进入 `main()` 前执行库级初始化的步骤，本项目启动文件调用 `__libc_init_array`。

这些概念都来自第03章的两个正式知识点，不新增教材结构外知识点。

## 5. 工作原理

STM32 程序从复位到 `main()` 可以拆成两条互相咬合的线。

第一条线是链接线：

1. 链接脚本声明入口为 `Reset_Handler`。
2. 链接脚本定义 RAM 起始地址、RAM 长度、FLASH 起始地址和 FLASH 长度。
3. 链接脚本把 `.isr_vector`、`.text`、`.rodata` 放入 FLASH。
4. 链接脚本把 `.data` 的运行位置放入 RAM，同时记录其加载位置来自 FLASH。
5. 链接脚本把 `.bss` 放入 RAM，并导出起止符号。
6. 链接脚本预留堆和栈检查区域。

第二条线是启动线：

1. CPU 通过向量表取得 `_estack` 和 `Reset_Handler`。
2. `Reset_Handler` 先调用 `SystemInit()`。
3. `Reset_Handler` 使用 `_sidata`、`_sdata` 和 `_edata` 把 `.data` 初值从 FLASH 拷贝到 RAM。
4. `Reset_Handler` 使用 `_sbss` 和 `_ebss` 将 `.bss` 清零。
5. `Reset_Handler` 调用 `__libc_init_array`。
6. `Reset_Handler` 调用 `main()`。

这就是为什么链接脚本和启动文件必须一起读：链接脚本定义“数据在哪里”，启动文件执行“启动时怎么搬、怎么清、往哪里跳”。

这两条线之间还有一个容易被忽略的边界：`.data` 不是“全部在 FLASH”或“全部在 RAM”。它的初值必须随固件镜像存放在 FLASH 中，否则掉电后初值会丢失；但全局变量运行时需要被 CPU 修改，所以运行位置必须在 RAM 中。链接脚本用 `_sidata = LOADADDR(.data)` 记录 FLASH 初值位置，用 `.data >RAM AT> FLASH` 表示运行地址在 RAM、加载地址在 FLASH。启动文件再用 `_sidata`、`_sdata`、`_edata` 把这段初值搬到 RAM。这个动作完成后，C 代码才可以像普通变量一样读写这些全局初值。

`.bss` 的逻辑相反。未初始化全局变量不需要在 FLASH 镜像里保存一份全 0 数据，否则会浪费 FLASH 空间。链接脚本只需要给 `.bss` 分配 RAM 地址范围，启动文件在进入 `main()` 前把 `_sbss` 到 `_ebss` 清零。这样既满足 C 语言“静态存储期对象默认初始化为 0”的语义，又避免把大量 0 写进固件镜像。

## 6. STM32实现机制

本项目的 STM32 实现机制可以从三个层面确认。

第一层是内存区域：

- `STM32F103RCTX_FLASH.ld` 中 RAM 起始地址为 `0x20000000`，长度为 `48K`。
- `STM32F103RCTX_FLASH.ld` 中 FLASH 起始地址为 `0x8000000`，长度为 `256K`。
- `_estack = ORIGIN(RAM) + LENGTH(RAM)`，也就是把 RAM 末端作为初始栈顶。
- `_Min_Heap_Size = 0x200`，`_Min_Stack_Size = 0x400`，用于保留最小堆栈检查空间。

把 RAM 边界代入可得：`_estack = 0x20000000 + 48 * 1024 = 0x2000C000`。这不是一个“栈变量”，而是向量表第一项写入的初始栈指针值。CPU 复位后读取该值作为 MSP 的初始值，然后才读取第二项 `Reset_Handler` 作为复位入口。

第二层是段布局：

- `.isr_vector` 被保留在 FLASH，用于放置启动向量表。
- `.text` 和 `.rodata` 放在 FLASH，用于代码和只读数据。
- `.data` 运行在 RAM，但初始值来自 FLASH。
- `.bss` 运行在 RAM，启动时由启动文件清零。
- `._user_heap_stack` 用于检查 RAM 中是否还能容纳预留堆和栈。

这里的 `._user_heap_stack` 更像链接阶段的空间占位和越界检查，不等于运行时已经把所有栈空间“写满”或“固定占用”。实际栈从 `_estack` 附近向低地址增长，实际堆通常从 `_end` 之后向高地址增长；两者是否会碰撞，需要结合 map 文件、运行路径、调试器栈深度观察和动态分配行为继续验证。

第三层是启动代码：

- `startup_stm32f103rctx.s` 声明 `.cpu cortex-m3`。
- `g_pfnVectors` 的前两项分别是 `_estack` 和 `Reset_Handler`。
- `Reset_Handler` 调用 `SystemInit()`。
- `Reset_Handler` 执行 `.data` 拷贝和 `.bss` 清零。
- `Reset_Handler` 调用 `__libc_init_array` 后进入 `main()`。
- `Default_Handler` 作为弱定义的默认处理入口，未被项目强定义替换的处理函数会落到默认入口。

当前 `Debug/Three-axis_cloud_platformV2.list` 还能从构建结果侧佐证段地址：`.isr_vector` 位于 `0x08000000`，`.data` 的运行地址位于 `0x20000000`、加载地址位于 `0x08015b70`，`.bss` 从 `0x20000370` 开始。构建产物属于当前 Debug 构建的结果证据，能用于交叉验证，但不应反过来替代链接脚本和源码作为配置来源。

注意：启动文件中的设备向量表包含芯片级入口，并不表示每个入口都已经成为本项目业务主线。正式功能分析必须等待对应章节。

## 7. 项目中的应用

本章对应项目中的运行入口层。

直接相关文件：

- `STM32F103RCTX_FLASH.ld`
- `Core/Startup/startup_stm32f103rctx.s`
- `Core/Src/system_stm32f1xx.c`
- `Core/Src/main.c`

它们之间的关系是：

- `.cproject` 在第02章已经确认链接脚本指向 `STM32F103RCTX_FLASH.ld`。
- 链接脚本定义入口和内存布局，并提供启动文件使用的符号。
- 启动文件中的向量表使用 `_estack` 和 `Reset_Handler`。
- `Reset_Handler` 调用 `SystemInit()`，然后准备 C 运行环境。
- `SystemInit()` 在当前文件中主要保留向量表重定位和外部存储相关条件代码；具体系统时钟配置不在本函数中完成。
- `main.c` 中存在 `int main(void)`，启动链路最终进入这里。

从项目主线看，本章位于所有外设初始化和控制逻辑之前。没有本章的启动和内存基础，后续 `HAL_Init()`、系统时钟配置、外设初始化和控制循环都没有运行前提。

## 8. 代码分析

### 1. 链接脚本入口与内存区域

`STM32F103RCTX_FLASH.ld` 使用 `ENTRY(Reset_Handler)` 指定入口符号。它不是直接跳到 `main()`，而是先进入启动文件中的 `Reset_Handler`。

`MEMORY` 区域定义了两块主要存储：

- RAM：`ORIGIN = 0x20000000`，`LENGTH = 48K`
- FLASH：`ORIGIN = 0x8000000`，`LENGTH = 256K`

这与第01章平台边界一致，也与第02章链接脚本配置一致。

从地址计算看，48K RAM 的末地址为 `0x2000C000`，这正是 `_estack` 的计算结果。256K FLASH 覆盖范围从 `0x08000000` 到 `0x08040000` 之前，向量表放在 FLASH 起点处，使复位后取向量的地址和镜像入口保持一致。

### 2. 栈、堆和段边界符号

链接脚本定义 `_estack`、`_Min_Heap_Size` 和 `_Min_Stack_Size`。其中 `_estack` 被启动向量表使用，堆和栈的预留值则为后续 Newlib 适配和内存使用分析建立前提。

链接脚本还定义：

- `_sidata`：`.data` 初值在 FLASH 中的加载地址。
- `_sdata`、`_edata`：`.data` 在 RAM 中的起止地址。
- `_sbss`、`_ebss`：`.bss` 在 RAM 中的起止地址。

这些符号不是普通变量，而是链接阶段生成给启动代码使用的地址标记。

更细地看，`.data` 的拷贝长度由 `_edata - _sdata` 决定，拷贝源起点是 `_sidata`。如果 `_sdata = 0x20000000`、`_edata = 0x20000370`，则当前构建中 `.data` 运行区大小为 `0x370` 字节；`Debug/Three-axis_cloud_platformV2.list` 中也显示 `.data` 大小为 `00000370`。这类数值可以作为本地构建证据，但它会随源码、库和编译选项变化。

`.bss` 的清零长度由 `_ebss - _sbss` 决定。当前构建产物显示 `.bss` 从 `0x20000370` 开始，大小为 `0x1e58` 字节。它包含 HAL 句柄、控制状态、姿态解算变量、USB 缓冲等运行态对象；这些对象进入 `main()` 前应处于 0 初值状态，后续初始化函数再逐步写入真实配置。

### 3. 启动文件中的复位入口

`startup_stm32f103rctx.s` 中的 `Reset_Handler` 执行顺序很明确：

- 调用 `SystemInit()`。
- 拷贝 `.data`。
- 清零 `.bss`。
- 调用 `__libc_init_array`。
- 调用 `main()`。

这解释了为什么全局初始化变量在进入 `main()` 前已经有值，未初始化全局变量在进入 `main()` 前已经为 0。

汇编循环可以进一步拆开：

- `.data` 拷贝阶段，`r0` 装入 `_sdata`，`r1` 装入 `_edata`，`r2` 装入 `_sidata`，`r3` 作为偏移量从 0 开始递增。
- 每轮 `ldr r4, [r2, r3]` 从 FLASH 初值区读取 4 字节，`str r4, [r0, r3]` 写入 RAM 运行区，随后 `adds r3, r3, #4` 前进到下一字。
- 循环条件 `adds r4, r0, r3` 和 `cmp r4, r1` 判断当前 RAM 写入位置是否还小于 `_edata`。小于则继续拷贝。
- `.bss` 清零阶段，`r2` 指向 `_sbss`，`r4` 指向 `_ebss`，`r3` 固定为 0。每轮 `str r3, [r2]` 写入一个 0，再让 `r2` 增加 4 字节，直到到达 `_ebss`。

这段代码默认段边界按 4 字节对齐。链接脚本在 `.data` 和 `.bss` 周围多次使用 `. = ALIGN(4)`，正是为了让启动文件可以按字循环处理，而不需要逐字节处理尾部。

### 4. `SystemInit()` 的项目边界

`Core/Src/system_stm32f1xx.c` 的注释说明 `SystemInit()` 在复位后、进入主程序前由启动文件调用。

当前项目中的 `SystemInit()` 函数没有直接配置最终项目时钟树，只保留了条件编译下的外部存储初始化入口和向量表重定位入口。系统时钟的项目配置在 `main.c` 的后续函数中出现，留到第06章系统时钟树分析。

这也解释了一个常见误读：`system_stm32f1xx.c` 文件头部注释会从通用模板角度描述系统时钟和 `SystemCoreClock`，但当前项目的 `SystemInit()` 实际代码只在条件满足时执行 `SystemInit_ExtMemCtl()`，并在定义 `USER_VECT_TAB_ADDRESS` 时写 `SCB->VTOR`。是否启用了这些条件，要看当前宏定义和构建配置，不能只凭模板注释下结论。

### 5. `main()` 入口

`Core/Src/main.c` 中存在 `int main(void)`。启动文件调用 `main()` 后，项目才进入 HAL 初始化、系统配置、外设初始化和主循环。第03章只确认入口关系，不展开后续业务执行。

当前 `main()` 中的后续顺序可以作为边界提示：`HAL_Init()`、`SystemClock_Config()`、多个 `MX_*_Init()` 和 `while (1)` 都发生在启动文件调用 `main()` 之后。因此本章证明的是“C 入口之前的最低运行环境”，不是 HAL 时基、中断优先级、USB、I2C、PWM 或控制循环已经正确运行。

### 本节证据边界

本节只根据当前仓库说明文件、函数、宏、变量和调用关系。运行时频率、外部硬件表现、主机侧现象、传感器方向、电机响应或真实控制效果仍需调试记录、日志或仓库外实测证据；缺少证据时保持【待验证】。

## 9. 调试方法

本章阶段的调试目标是确认程序能从复位入口走到 `main()`，以及链接脚本与启动文件符号一致。

可观察对象：

- `.cproject` 中链接脚本是否仍指向 `STM32F103RCTX_FLASH.ld`。
- `STM32F103RCTX_FLASH.ld` 中是否存在 `ENTRY(Reset_Handler)`。
- 链接脚本中是否定义 `_estack`、`_sidata`、`_sdata`、`_edata`、`_sbss` 和 `_ebss`。
- 启动文件中是否引用这些符号。
- 启动文件中是否存在 `g_pfnVectors`、`Reset_Handler` 和 `Default_Handler`。
- `Core/Src/system_stm32f1xx.c` 中是否存在 `SystemInit()`。
- `Core/Src/main.c` 中是否存在 `int main(void)`。
- `Debug/makefile` 是否使用 `-T".../STM32F103RCTX_FLASH.ld"` 链接当前镜像。
- `Debug/objects.list` 是否包含 `./Core/Startup/startup_stm32f103rctx.o`。
- `Debug/Three-axis_cloud_platformV2.list` 或 `.map` 中 `.isr_vector`、`.data` 和 `.bss` 的地址是否与链接脚本规则一致。

常见异常定位：

- 链接失败并提示找不到启动符号：检查链接脚本入口和启动文件中的 `Reset_Handler`。
- 运行后全局变量初值异常：检查 `.data` 段符号和启动文件拷贝逻辑。
- 未初始化全局变量不是 0：检查 `.bss` 段符号和启动文件清零逻辑。
- 进入异常默认入口后停住：检查是否有处理函数未被项目实现，或启动链路是否跳到了错误入口。
- 栈或堆空间不足：先查看链接脚本中的 RAM 长度、最小堆栈设置和 `._user_heap_stack` 预留。
- `.map` 中 RAM 使用量看似正常但运行仍异常：继续检查真实调用深度、中断嵌套、库函数栈消耗、动态分配路径和异常现场，不能只用链接成功证明运行安全。

调试记录建议：

- 记录链接脚本符号、启动文件引用位置、启动动作和对应结论。
- 对异常入口停住的情况，记录当前 PC、向量表入口、实际处理函数和是否落入 `Default_Handler`。
- 对栈堆问题，只记录链接脚本与调试器观察证据，不把单次停机直接写成内存规划错误。
- 对 `.data` 和 `.bss` 问题，记录 `_sidata/_sdata/_edata/_sbss/_ebss` 的 map 地址，并在调试器中观察进入 `main()` 前后的 RAM 内容变化；没有现场观察时，结论保持【待验证】。

调试边界：

当前仓库能证明启动文件、链接脚本和 `main()` 入口之间的静态关系。复位后真实执行路径、栈损坏、异常原因和调试器停机位置需要运行现场证据；缺少证据时保持【待验证】。

## 10. 常见问题

### 1. 为什么不从 `main()` 开始讲？

因为在 STM32 上，复位后 CPU 先读取向量表，装载初始栈指针，再进入复位处理函数。启动文件完成数据搬运、零初始化和系统初始化后，才调用 `main()`。

`main()` 是应用入口，但不是复位后的第一入口。先讲启动链路，可以让读者理解后续为什么链接脚本、栈顶、`.data` 和 `.bss` 会影响程序是否能正常进入应用代码。

### 2. 为什么链接脚本会影响启动？

启动文件要使用 `_estack`、`.data` 和 `.bss` 的边界符号。这些符号由链接脚本生成。如果链接脚本错误，启动文件即使逻辑正确，也可能搬错数据或清错区域。

当前仓库能证明的是启动文件引用这些符号，链接脚本定义这些内存区域。至于某次异常停机是否一定由链接脚本导致，还需要调试器栈、异常入口和内存观察记录支撑，不能只凭现象下结论。

### 3. `SystemInit()` 是不是本项目最终时钟配置？

不是。当前 `SystemInit()` 是启动阶段被调用的系统初始化入口，当前实现主要保留向量表重定位和条件初始化入口。项目具体系统时钟配置留到第06章。

这一点很关键：启动阶段存在 `SystemInit()`，不代表最终时钟已经配置完成。本项目的时钟树证据来自 `main.c` 中后续的 `SystemClock_Config()`、HAL RCC 调用和 `.ioc` 频率记录。

### 4. 向量表里存在很多入口，是否说明项目都用了？

不说明。向量表是芯片启动和异常入口结构的一部分。是否成为项目功能主线，必须看对应源码、配置和后续章节证据。

例如向量表提供了外设中断入口，只能说明芯片和启动文件具备这个入口。真正要判断项目是否使用某个中断，还要看 `.ioc`、NVIC 配置、MSP 初始化、实际中断函数和业务调用。这个边界会在第09章继续展开。

### 5. 为什么要关心堆和栈？

因为链接脚本中明确预留了最小堆和最小栈空间。后续 Newlib 适配、内存分配和运行稳定性都依赖这部分边界。

不过，第03章只能说明仓库中的内存规划和启动边界，不能单独证明运行时堆栈永远足够。若要判断栈溢出、堆分配失败或异常复位，需要结合调试器现场、map 文件和运行路径继续验证。

### 6. `.data` 为什么有两个地址？

因为它同时服务两个需求：固件镜像需要在掉电后保留初值，所以初值放在 FLASH；程序运行时需要修改变量，所以运行地址放在 RAM。链接脚本中的 `AT> FLASH` 描述加载位置，`>RAM` 描述运行位置，`LOADADDR(.data)` 把加载地址导出给启动文件。

如果只放在 FLASH，普通可写全局变量无法像 RAM 变量一样被修改；如果只放在 RAM，复位后初值没有非易失来源。启动阶段的拷贝正是为了解决这个矛盾。

## 13. 本章总结

本章建立了三轴云台项目从复位到 `main()` 的启动路径。

已经确认的结论是：

- 链接脚本定义 `Reset_Handler` 入口、FLASH/RAM 区域、堆栈预留和段布局。
- 启动文件中的向量表使用 `_estack` 和 `Reset_Handler`。
- `Reset_Handler` 调用 `SystemInit()`，再完成 `.data` 拷贝、`.bss` 清零、C 运行时初始化和 `main()` 调用。
- `.data` 的加载地址在 FLASH、运行地址在 RAM，`.bss` 只需要 RAM 区间并在启动阶段清零。
- 当前 Debug 构建产物可交叉验证 `.isr_vector`、`.data` 和 `.bss` 的实际地址，但不能替代源码和链接脚本作为配置来源。
- 当前 `SystemInit()` 不展开项目最终时钟树，系统时钟配置留到后续章节。
- `main.c` 中的 `int main(void)` 是启动链路进入应用层的入口。

本章边界：

- 本章证明固件如何从复位入口进入 `main()`，不证明向量表中每个中断都参与当前业务。
- 时钟树、外设初始化和运行调度只记录入口关系，具体行为留到后续章节验证。

下一章可以进入 HAL 工程裁剪与构建产物，因为启动路径已经说明了工程镜像如何被放置、如何启动，以及为什么构建结果必须与链接脚本和启动文件一致。
