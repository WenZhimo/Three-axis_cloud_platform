## 3. 问题背景

前面章节已经分别解释了很多模块：

这些模块单独理解还不够。真实项目运行时，关键问题是：它们按什么顺序执行，哪些状态决定某个轴是否参与控制，输出如何从姿态角一路传到电机。

本章的任务就是把这条链路变成读者可以调试、可以定位问题、可以扩展的系统地图。

## 4. 核心概念

### 4.1 三轴云台控制架构

`三轴云台控制架构` 是项目从传感器输入到电机输出的整体组织方式。它不是单个函数，而是多个模块共同形成的运行结构。

本项目中，系统架构至少包含五层：

- 调度层：`frame_500Hz` 触发的 500Hz 实时控制循环和 10Hz 低频任务。
- 采集层：`MPU6050_Read_And_Process()` 和原始传感器数据。
- 姿态层：`MargAHRSupdate()` 和 `sensors.margAttitude500Hz[]`。
- 控制层：`computeMotorCommands()`、`updatePID()` 和 `pidCmd[]`。
- 执行层：`PWM_Motor_SetAngle()` 和 TIM CCR。

### 4.2 自动跟随/回中算法

当前项目中，`自动跟随/回中算法` 有两类证据。

第一类是待接入的自动跟随函数：

```text
autoPan()
YAP_DEADBAND
MOTORPOS2SETPNT
AUTOPANSMOOTH
```

但当前搜索只发现 `autoPan()` 定义，没有发现当前主控制路径调用它。`rollAutoPanEnabled`、`pitchAutoPanEnabled`、`yawAutoPanEnabled` 在配置结构中存在，默认设置为 false，当前也未发现参与控制路径。

第二类是当前运行证据更明确的回中状态机：

```text
return_state
return_state_roll
return_state_count
return_state_count_roll
```

这些变量分别在 `computeMotorCommands.c` 和 `Core/Src/main.c` 中配合使用，用于控制 Pitch/Roll 的回中阶段和后续控制阶段切换。

因此本章讲 `自动跟随/回中算法` 时，以当前运行证据更强的回中状态机为主，把 `autoPan()` 标为待接入边界，而不是当前主控制路径的一部分。

### 4.3 轴使能

轴使能由 `eepromConfig.pitchEnabled`、`eepromConfig.rollEnabled`、`eepromConfig.yawEnabled` 控制。

`Core/Src/main.c` 初始化后先把三轴都关闭：

```text
pitchEnabled = false
rollEnabled = false
yawEnabled = false
```

AHRS 收敛计数达到 1000 后，项目开启 Pitch 和 Yaw，保持 Roll 关闭：

```text
pitchEnabled = true
rollEnabled = false
yawEnabled = true
```

随后 10Hz 低频任务根据 `return_state_count` 打开 Roll，并根据 `return_state_count_roll` 解除 Roll 的回中状态。

### 4.4 三轴非对称性

本项目三轴不是完全相同的控制模板。

- Roll 有 `return_state_roll` 分支、`rollTargetSlew`、电角误差积分暂停和 `rollDiag`。
- Pitch 有 `return_state` 分支、`return_state_count`，并在不同状态下改变 `current_elec` 符号。
- Yaw 当前没有看到目标斜坡调用，使用 `safe_target` 处理角度环绕后进入 PID。
- 三轴都依赖 `updatePID()` 的 `ANGULAR` 类型处理角度误差，但进入 PID 前的误差构造方式不同。

因此教材不能写成“三轴完全相同，只是参数不同”。更准确的表达是：三轴共享控制框架，但具体分支和状态机不同。

### 4.5 多速率状态机

`多速率状态机` 指状态变量的采样、累积和提交不在同一个频率中完成。

当前项目里，Pitch/Roll 回中计数在 500Hz 实时控制分支中累积，
但状态释放在 10Hz 低频任务中判断：

```text
500Hz: return_state_count++
10Hz : if return_state_count >= 650 -> return_state = false
```

这种结构的优点是实时路径只做轻量条件判断，低频路径处理状态切换和打印。
代价是状态释放不是“计数刚到阈值立刻发生”，而是要等下一次 10Hz 检查。

从状态机教学角度，可以把本项目的回中逻辑拆成四个最小单元：

- 状态：例如 `return_state == true` 表示 Pitch 仍在回中阶段。
- 扩展状态变量：例如 `return_state_count` 记录条件连续满足的帧数。
- 守卫条件：例如 `return_state_count >= 650 && pitchEnabled == true`。
- 动作：例如把 `return_state` 置为 false，并打开 `rollEnabled`。

NASA F Prime 的状态机文档也把状态、转移、守卫和动作分开描述。这个拆法能帮助读者
避免把“计数变量变化”误解成“状态已经切换”。在本项目中，计数是在 500Hz 分支累计，
而真正切换状态的动作发生在 10Hz 分支。

### 4.6 架构级耦合

系统架构中有两类耦合。

第一类是数据流耦合。传感器缩放、姿态解算、PID 和电角输出按顺序依赖，
前一级的符号、单位或 NaN 处理会影响后一级。

第二类是状态耦合。`computeMotorCommands()` 开头把 `holdIntegrators` 置为 false，
Roll 分支在电角误差过大时可能把它置为 true。
由于 Pitch 和 Yaw 分支随后把这个全局标志传入 `updatePID()`，
Roll 的积分保持状态可能影响同一帧后续轴的积分更新。

这不是本章要修改的源码行为，而是读者调试系统架构时必须知道的隐含前置知识。

还有一类容易误判的耦合是“预留变量耦合”。`computeMotorCommands.c` 中存在
`pitchTargetSlew`、`yawTargetSlew`、`yawCtrlAngle`、`yawAxisWasEnabled`、
`pitchGateOpen` 和 `rollSettledTime` 等静态变量，也存在 `PITCH_TARGET_SLEW_RAD_S`、
`YAW_TARGET_SLEW_RAD_S`、`YAW_CTRL_LPF_TAU_S`、`YAW_ERR_DEADBAND_RAD`、
`ROLL_SETTLE_ERR_RAD`、`ROLL_SETTLE_CMD_RAD`、`ROLL_SETTLE_TIME_S` 等宏。

这些名字看起来像完整的 Pitch/Yaw 目标斜坡、Yaw 低通或 Roll 收敛门控状态机。
但当前搜索证据显示，它们没有参与当前主控制路径，或者只作为定义存在。
所以系统架构分析必须区分“已运行状态变量”和“预留/未接入状态变量”。
否则读者会以为当前项目已经实现了更多状态机层级。

### 4.7 诊断字段有效性

系统架构调试不能只看“变量存在”。还要确认该变量在当前分支是否刷新。

`rollDiag` 是 Roll 后续 PID 分支的诊断结构，字段包括 `pidRaw`、`pidClamped`、
`pidApplied`、`stepLimit`、`errElecRad` 和 `holdI`。但源码中这些字段只在
`eepromConfig.rollEnabled == true` 且 `return_state_roll == false` 的后续 PID
分支里被赋值。

因此，以下情况不能直接把 `rollDiag` 当成当前 Roll 闭环诊断证据：

- Roll 轴尚未使能。
- Roll 仍处于 `return_state_roll == true` 的回中阶段。
- 本帧没有进入 Roll 后续 PID 分支。

在这些情况下，`rollDiag` 可能保持初值或上一轮旧值。调试记录必须同时写下
`rollEnabled`、`return_state_roll` 和当前分支路径，才能解释 `rollDiag` 是否有效。

### 4.8 分支顺序与跨轴传播条件

`computeMotorCommands()` 的分支顺序是 Roll、Pitch、Yaw。

这个顺序决定了 `holdIntegrators` 的传播方向：Roll 后续 PID 分支可能把
`holdIntegrators` 置为 true，然后 Pitch/Yaw 在同一轮函数调用中读取这个值。
反过来，Pitch/Yaw 不会影响本轮已经执行完的 Roll 分支。

所以跨轴积分保持不是无条件存在，而是同时依赖：

- Roll 已经使能。
- Roll 不在回中分支，而是进入后续 PID 分支。
- `roll_error_electrical` 超过 `ROLL_I_ENABLE_ERR_RAD`。
- Pitch/Yaw 在同一轮调用中仍然使能并执行到 `updatePID()`。

这把“全局变量影响三轴”拆成了可验证的条件链。

### 4.9 PID角度误差归一化入口

角度控制和普通线性量控制的区别在于误差要考虑 `-π` 到 `π` 的环绕。
例如 `179°` 到 `-179°` 的最短角度误差应接近 `2°`，而不是 `-358°`。

当前项目有两个层级会处理角度环绕。

第一层在 `computeMotorCommands.c` 内部，局部调用 `wrapToPif()`。
Roll 后续 PID 分支会先计算：

```text
roll_error_mech = wrapToPif(rollTargetSlew - roll_angle)
target_electrical_angle = current_electrical_angle
                        + roll_error_mech * mechanical2electricalDegrees[ROLL]
```

Yaw 分支也会先计算：

```text
error_mech = wrapToPif(target_angle - yaw_angle)
safe_target = yaw_angle + error_mech
```

第二层在 `pid.c::updatePID()` 内部。该函数先做：

```text
error = command - state
```

如果 `PIDparameters->type == ANGULAR`，再调用 `standardRadianFormat(error)`。
`config.c` 中 Roll、Pitch、Yaw 三个 PID 的 `type` 都初始化为 `ANGULAR`。
`main.c` 后续覆盖的是部分 P/I/D 和历史状态，没有看到把 `type` 改成其它类型的证据。

这带来一个容易误读的点：Pitch 分支虽然计算了：

```text
error_mech = wrapToPif(target_angle - pitch_angle)
```

但当前代码没有把这个 `error_mech` 直接送入 `updatePID()`。
Pitch 传入 PID 的仍是 `target_angle` 和 `pitch_angle`。
因此 Pitch 的 PID 角度环绕主要依赖 `updatePID()` 中的 `ANGULAR` 类型处理。

所以调试三轴误差时，应分别记录：

- PID 入口的 `command` 和 `state`。
- 分支内局部计算的 `error_mech` 或 `roll_error_mech`。
- `PIDparameters->type` 是否仍为 `ANGULAR`。
- 输出阶段的 `stator_electrical_angle` 是否再次被 `wrapToPif()` 包裹。

还要注意 `standardRadianFormat()` 只执行一次 `±2π` 修正，不是 `while` 循环式归一化。
在当前姿态和目标角都保持在常规弧度范围内时通常足够；若未来引入多圈目标、
累计角或未归一化外部命令，则需要重新验证误差归一化边界【待验证】。

### 4.10 架构门控层级

系统架构里有多种“看起来像开关”的变量，但它们控制的层级不同。

可以把当前项目拆成六层门控：

- 调度门控：`systemReady == true && mpu6050Calibrating == false` 时，`SysTick_Handler()` 才推进 `frameCounter` 并产生 `frame_500Hz`。
- 传感器初始化提示：`MPU6050_Init()` 的返回值只用于打印成功或失败信息，当前未看到 `mpu_ok` 被置为 true，也未看到它参与 500Hz 或电机输出门控。
- 启动延时门控：`startup_delay_counter < 1000` 用于延后 Pitch/Yaw 轴使能，并在达到阈值时清 PID 状态。
- 轴级门控：`eepromConfig.pitchEnabled`、`rollEnabled`、`yawEnabled` 决定 `computeMotorCommands()` 内部是否进入对应轴分支。
- 轴内阶段门控：`return_state` 和 `return_state_roll` 决定 Pitch/Roll 处于回中阶段还是后续控制阶段。
- 执行层初始化门控：`PWM_Motor_Init()` 启动 PWM 通道并设置 `isInitialized`，`PWM_Motor_SetAngle()` 在 `isInitialized == 0` 时直接返回。

这些门控不能混写。例如，`systemReady` 允许实时帧开始产生，不等于 MPU6050 初始化一定成功；
`PWM_Motor_Init()` 启动了硬件 PWM 通道，不等于三轴闭环控制已经使能；
`startup_delay_counter == 1000` 开启 Pitch/Yaw，也不等于 Roll 已经进入后续 PID。

### 4.10.1 硬件启动与闭环接管

第31章里最容易混淆的是“PWM 通道已经启动”和“某轴已经进入闭环接管”。
这两件事不是一个层级。按当前源码可以再拆成一条更细的阶段链：

| 阶段 | 当前证据 | 能说明什么 | 不能说明什么 |
|---|---|---|---|
| PWM 硬件启动 | `PWM_Motor_Init()` 调用 9 次 `HAL_TIM_PWM_Start()`，随后把 `isInitialized` 置 1 | 底层 PWM 通道被允许工作 | 不能说明闭环控制已经开始 |
| 控制链可执行 | `computeMotorCommands(dt500Hz)` 每个 500Hz 帧都会被主循环调用 | 控制函数本身进入运行路径 | 不能说明某个轴分支一定真正写 CCR |
| 轴级使能 | `eepromConfig.pitchEnabled`、`rollEnabled`、`yawEnabled` 决定分支是否进入 | 某个轴是否参与控制计算 | 不能说明该轴已经摆脱回中阶段 |
| 阶段接管 | `startup_delay_counter == 1000` 后只先开启 Pitch/Yaw，Roll 仍由后续计数控制 | 轴的接管顺序是分批的 | 不能说明三轴同时进入后续 PID |
| 输出入口允许 | `PWM_Motor_SetAngle()` 只要 `isInitialized == 1` 就允许写 CCR | 软件允许进入执行器层 | 不能说明本轮一定写了有效的新 CCR |

因此，调试时应把“PWM 已启动”“控制层已调用”“轴级分支已进入”“CCR 已写入”“电机真实响应”分开记录。
如果只看到 `isInitialized == 1` 或 `PWM_Motor_Init()` 成功，就推断闭环已经接管，会把硬件初始化证据误写成控制完成证据【待验证】。

还有两个变量需要特别警惕：

```text
attitude_deg_prev[]
attitude_delta_initialized
```

它们的注释看起来像“姿态收敛判断”相关变量，但当前搜索证据只发现定义，
未看到它们参与启动延时判断。因此当前开机等待更准确地说是固定帧数延时，
不是基于姿态变化量的真实收敛检测。姿态是否已经稳定仍需调试数据证明【待验证】。

## 5. 工作原理

### 5.1 从SysTick到500Hz实时控制帧

第26章已经确认当前调度方式是：`SysTick_Handler()` 置位 `frame_500Hz`，`main()` 主循环清除该标志并执行 500Hz 实时控制循环。

第31章需要补一个系统级边界：`SysTick_Handler()` 不是无条件产生 500Hz 控制帧。当前代码只有在：

```text
systemReady == true && mpu6050Calibrating == false
```

成立时，才递增 `frameCounter` 并在 500Hz 分频点设置 `frame_500Hz = true`。

这说明 `systemReady` 是实时帧生成的前置条件之一。`main()` 中的 `if (frame_500Hz)` 分支消费标志时，不再二次检查 `systemReady`；因此调试时应把“帧是否产生”和“帧产生后执行了什么”分开观察。

#### 500Hz帧完成度证据边界

Arm Cortex-M3 的 SysTick 本质上提供的是一个递减计数器和到零后的异常请求语义。
在本项目里，`SysTick_Handler()` 又把这个硬件节拍进一步降级为一个软件布尔标志：

```text
SysTick到期
-> SysTick_Handler()
-> systemReady/mpu6050Calibrating门控
-> frameCounter分频
-> frame_500Hz = true
-> main()下一轮轮询到标志
-> 清除frame_500Hz并执行控制链
```

因此，“500Hz帧”至少要拆成以下几个完成度层级：

| 层级 | 仓库内证据 | 能证明什么 | 不能证明什么 |
|---|---|---|---|
| 硬件节拍到期 | Cortex-M3 SysTick 官方语义；`HAL_IncTick()` 位于 `SysTick_Handler()` 开头 | 内核具备周期性异常入口，HAL tick 在该入口递增 | 不能证明项目控制链已经执行 |
| 软件帧产生 | `systemReady == true && mpu6050Calibrating == false` 后递增 `frameCounter`，满足 `COUNT_500HZ` 时写 `frame_500Hz = true` | 当前源码存在 500Hz 帧置位条件 | 不能证明主循环已经及时消费该标志 |
| 主循环消费 | `main()` 中 `if (frame_500Hz)` 先清零标志，再计算 `dt500Hz` | 当前源码把一次已观察到的标志转成一次控制链执行 | 布尔标志不能累计多个未消费帧；阻塞后可能只看到一次消费 |
| 控制链完成 | 同一分支内依次调用 `MPU6050_Read_And_Process()`、`MargAHRSupdate()`、`computeMotorCommands(dt500Hz)` 并写 `executionTime500Hz` | 当前源码存在采集、姿态、控制和耗时统计顺序 | 不能证明本帧传感器数据有效、控制输出稳定或电机响应安全 |
| 运行时实时性 | `dt500Hz`、`executionTime500Hz`、GPIO翻转或DWT记录 | 可以验证帧间隔、分支内部耗时和是否存在延迟尖峰 | 没有实测记录时，不能把理论 500Hz 写成真实闭环频率【待验证】 |

这个拆分也解释了为什么本章不能只看 `executionTime500Hz`。
它记录的是 500Hz 分支内部从 `currentTime = micros()` 到分支末尾的耗时；
而 `dt500Hz` 记录的是两次主循环消费 `frame_500Hz` 之间的间隔。
如果主循环被 I2C、串口打印或其他路径阻塞，`executionTime500Hz` 可能只显示某次分支内部耗时，
但 `dt500Hz` 会暴露上一帧到这一帧之间是否已经被拉长。两者必须联合观察。

### 5.2 采集层

500Hz 实时控制分支先调用：

```text
MPU6050_Read_And_Process()
```

这个函数读取 MPU6050 原始数据，并更新 `rawAccel[]`、`rawGyro[]` 等原始量。随后 `main.c` 把原始量转换为控制链使用的物理量：

```text
sensors.accel500Hz[]
sensors.gyro500Hz[]
```

这一步把第19章、第21章的传感器读取和缩放结果接入当前架构。

但“函数被调用”不等于“本帧传感器数据有效”。`MPU6050_Read_And_Process()` 中，
若 `MPU_Read_Reg(REG_ACCEL_XOUT_H, ...)` 返回失败，代码会把 `rawAccel[]` 和
`rawGyro[]` 清零并直接返回。随后 `main.c` 仍会继续缩放这些原始量，并继续调用
`MargAHRSupdate()`。

因此，采集层至少要拆成三个最小教学单元：

- 调用发生：500Hz 分支确实调用了 `MPU6050_Read_And_Process()`。
- 通信成功：I2C 读 14 字节返回 `HAL_OK`。
- 数据可信：原始值、方向矩阵、零偏和物理缩放都需要单独验证。

当前 `mpu_ok` 变量没有参与这条运行门控，不能写成“MPU6050 初始化失败会自动阻止控制链路继续运行”。更稳妥的表述是：初始化成功或失败会打印信息，但运行时数据有效性仍要通过原始值、I2C 状态和姿态输出共同验证【待验证】。

### 5.3 姿态层

传感器物理量准备好后，`main.c` 调用：

```text
MargAHRSupdate(...)
```

当前磁力计输入参数传入 `0, 0, 0`，并且最后一个布尔参数为 `false`。这说明当前姿态更新路径以陀螺仪和加速度计为主。

这里要把“调用 `MargAHRSupdate()`”再拆成几层证据：

| 证据层 | 当前仓库证据 | 可以说明 | 不能说明 |
| --- | --- | --- | --- |
| 接口名称 | 函数名和头文件参数包含 `mx/my/mz` 与 `magDataUpdate` | 接口保留了 MARG 形式的磁力计入口 | 当前主链路已经使用磁力计航向修正 |
| 主循环实参 | `main.c` 传入 `0, 0, 0, false` | 当前 500Hz 主线没有提供有效磁力计样本 | Yaw 航向已经被磁力计约束 |
| 函数体修正项 | `MargAHRS.c` 中当前可见 `exAcc/eyAcc/ezAcc` 修正 `gx/gy/gz`，而 `exMag/exMagInt` 等只作为全局对象存在 | 当前姿态更新主要依赖陀螺积分和加速度重力反馈 | 已实现完整九轴 MARG 融合或 EKF |
| 构建产物 | `.list` 中保留 `MargAHRSupdate(..., 0, 0, 0, false, dt500Hz)` 调用旁注，`.map` 中 `exMag*` 停留在 `0x00000000` 输入段记录 | 当前 Debug 快照包含该姿态入口和磁力计参数未启用的调用形态 | 运行时姿态、航向漂移和磁干扰表现已经验证【待验证】 |

`MargAHRSupdate()` 更新姿态后，控制层读取的是：

```text
sensors.margAttitude500Hz[ROLL]
sensors.margAttitude500Hz[PITCH]
sensors.margAttitude500Hz[YAW]
```

### 5.4 启动延时和控制使能

AHRS 刚启动时，姿态估计可能尚未稳定。因此 `main.c` 使用：

```text
startup_delay_counter < 1000
```

在 500Hz 理想消费频率下等待约 2 秒。计数达到 1000 后，代码清空 PID 积分和 D 项状态：

```text
zeroPIDintegralError()
zeroPIDstates()
```

然后开启 Pitch 和 Yaw，Roll 暂时保持关闭。

这个设计说明：项目不是传感器一读到数据就立即驱动所有电机，而是先等待固定数量的 500Hz 控制帧，再打开部分控制轴。

这里必须避免一个常见跳步：`startup_delay_counter` 计的是被主循环消费的 500Hz 帧数，
不是独立硬件计时器直接给出的绝对墙钟时间。若主循环阻塞导致多个 `frame_500Hz`
触发合并为一次布尔标志消费，那么 1000 次消费对应的真实时间可能偏离理想 2 秒【待验证】。

同样，虽然代码注释把 `attitude_deg_prev[]` 和 `attitude_delta_initialized`
描述为姿态收敛相关变量，但当前没有看到它们参与 `startup_delay_counter` 的判断。
也就是说，本项目当前启动控制接管依据是固定帧数延时，而不是
“姿态角变化量已经小于某阈值”的闭环收敛检测。

### 5.5 computeMotorCommands的角色

`computeMotorCommands(dt500Hz)` 是控制层和执行层之间的枢纽。

它的输入包括：

- `dt500Hz`
- `sensors.margAttitude500Hz[]`
- `pointingCmd[]`
- `eepromConfig.PID[]`
- `eepromConfig.*Enabled`

它的输出包括：

- `pidCmd[]`
- `outputRate[]`
- `rollDiag`
- 对 `PWM_Motor_SetAngle()` 的调用

它还会修改部分状态，例如 `holdIntegrators`、`return_state_count`、`return_state_count_roll`、`mechanical2electricalDegrees[]` 的部分轴值。

### 5.6 回中状态与轴切换

Pitch 回中相关状态在 Pitch 分支中更新：

```text
if return_state == true:
    根据 sensors.margAttitude500Hz[PITCH] 是否接近 0 更新 return_state_count
```

10Hz 低频任务中，如果 `return_state_count >= 650` 且 Pitch 已使能，则：

```text
return_state = false
rollEnabled = true
```

这表示 Pitch 进入后续状态后，Roll 才被打开。

Roll 回中相关状态在 Roll 分支中更新：

```text
if return_state_roll == true:
    根据 pointingCmd[ROLL] 与 sensors.margAttitude500Hz[ROLL] 的差值更新 return_state_count_roll
```

10Hz 低频任务中，如果 `return_state_count_roll >= 650` 且 Roll 已使能，则：

```text
return_state_roll = false
```

这表示 Roll 从回中路径切到后续 PID 路径。

这里还要补一个同轮主循环顺序边界。当前 `main.c` 的 `while (1)` 中，
`if (frame_500Hz)` 分支排在 `if (HAL_GetTick() - last_print_tick >= 100)` 分支之前。
因此，如果某一轮主循环先执行 500Hz 分支，并在 `computeMotorCommands()` 内把
`return_state_count` 或 `return_state_count_roll` 更新到阈值，
紧随其后的 10Hz 分支才会提交 `return_state = false`、`rollEnabled = true`
或 `return_state_roll = false`。已经执行完的这一次 `computeMotorCommands()`
使用的是状态切换前的门控；状态提交只能影响后续 500Hz 控制帧，不能倒改刚才那一帧的分支路径。

反过来，如果某一轮主循环没有消费新的 `frame_500Hz`，10Hz 分支看到的就是上一轮控制帧留下的计数值。
所以调试时要把“计数更新时刻”“10Hz 提交时刻”和“下一帧控制分支可见时刻”分开记录。
这个结论来自当前主循环源码顺序；真实延迟还受 `HAL_GetTick()` 触发相位、主循环阻塞和串口打印耗时影响。

### 5.7 回中时间阈值推导

启动延时由 500Hz 分支中的计数器决定：

```text
T_startup = 1000 / 500Hz = 2s
```

Pitch 和 Roll 的回中释放阈值都是 650 个 500Hz 计数。
理想情况下，如果条件连续满足，最短保持时间为：

```text
T_return_min = 650 / 500Hz = 1.3s
```

但状态释放判断在 10Hz 低频任务中执行，低频任务周期约为：

```text
T_low = 100ms
```

因此实际释放时刻不是精确的 1.3s，而是在计数达到阈值后的下一次 10Hz 检查。
若主循环没有被阻塞，理想释放延迟约在 `[0, 100ms)` 内。
若 I2C 阻塞、串口打印或其他任务推迟主循环，真实延迟还会增加【待验证】。

还要注意，这些计数器记录的是“条件连续满足的控制帧次数”。

Pitch 只有在 `return_state == true` 且：

```text
fabs(sensors.margAttitude500Hz[PITCH]) < 0.3
```

时增加 `return_state_count`，否则清零。

Roll 只有在 `return_state_roll == true` 且：

```text
fabs(pointingCmd[ROLL] - sensors.margAttitude500Hz[ROLL]) < 0.2
```

时增加 `return_state_count_roll`，否则清零。

所以 `650` 不是累计总帧数，而是连续满足守卫条件的帧数。只要中间一次条件不满足，计数就会回到 0。

角度阈值也可以换算成读者更熟悉的角度：

```text
0.3rad ≈ 17.2°
0.2rad ≈ 11.5°
```

所以这两个阈值并不是“已经严格稳定”的证明，只是当前代码采用的进入下一状态条件。

### 5.8 三轴输出路径汇总

在当前代码中，三轴输出路径是：

- Roll 控制分支调用 `PWM_Motor_SetAngle(MOTOR_PITCH, ...)`。
- Pitch 控制分支调用 `PWM_Motor_SetAngle(MOTOR_ROLL, ...)`。
- Yaw 控制分支调用 `PWM_Motor_SetAngle(MOTOR_YAW, ...)`。

这个映射延续第30章发现的项目特定“控制轴 -> 执行器枚举 -> CCR通道”关系。调试时不能只按轴名称直觉判断，必须看实际传入的 `MotorAxis_t`，再沿 `PWM_Motor_SetAngle()` 的 `switch` 分支追到具体 CCR。

从系统架构角度，还应再加一层外部证据。完整链路不是三层，而是四层：

```text
控制分支名
-> 传入 PWM_Motor_SetAngle() 的 MotorAxis_t
-> drv_pwmMotors.c 实际写入的 TIMx->CCRn / GPIO
-> 实物电机对应的机械轴响应
```

前三层可以由当前仓库源码和寄存器观察支撑；第四层必须依赖接线、安装方向、
示波器、电流和机械响应等仓库外实测。也就是说，即使代码层确认 Pitch 分支
写到了 `MOTOR_ROLL` 对应的 CCR 组，也不能直接推出真实硬件上的 Pitch 机械轴
一定响应正确。机械轴归属、转向和相序仍然应保持【待验证】。

还要把“PWM硬件已启动”和“轴闭环已接管”分开。

`PWM_Motor_Init()` 在初始化阶段已经启动 TIM2/TIM3/TIM4 的相关 PWM 通道，
并设置 `isInitialized = 1`。但 `main()` 随后仍把三轴 `eepromConfig.*Enabled`
先置为 false。也就是说，执行层硬件通道准备好，不等于控制层三轴分支已经输出闭环命令。

若某个轴关闭，`computeMotorCommands()` 仍会在每个 500Hz 帧被调用，但该轴分支不会执行。
因此调试电机无响应时，应同时检查：

- PWM 初始化是否成功走到 `isInitialized = 1`。
- 轴级 `eepromConfig.*Enabled` 是否为 true。
- 轴内 `return_state*` 是否仍处于回中路径。
- 对应分支是否真的调用了 `PWM_Motor_SetAngle()`。
- CCR 变化是否能代表外部驱动和电机真实动作【待验证】。

### 5.9 为什么当前采用主循环PID架构

从仓库证据看，当前项目采用的是：

```text
SysTick置位标志
主循环消费标志
单次串行执行采集、姿态、PID、输出
```

这种结构与 PID 控制匹配度高：实现简单、状态少、调试断点清楚，
也便于在 STM32F103RCTx 的 2ms 控制预算内观察每一层耗时。

如果改成把 I2C 读取、AHRS 和控制计算都放进 SysTick 中断，
会增加中断占用时间。项目中 `stm32f1xx_it.c` 的注释也明确提到，
I2C 读取耗时较多，放在中断里还可能带来总线冲突和读数异常风险。

如果改成 LQR 或 MPC，教材必须先补充状态空间模型、状态向量、
系统矩阵、代价函数、约束条件和运行时求解器。
当前仓库没有这些模型和优化器证据；Debug 构建还使用
`-mcpu=cortex-m3` 与 `-mfloat-abi=soft`。因此本章只能把 PID 架构写成
当前项目的已实现方案，不能把 LQR/MPC 写成已经具备的替代实现。

### 5.10 Roll诊断字段的刷新时机

`rollDiag` 的刷新发生在 Roll 后续 PID 分支末尾。按源码条件展开：

```text
rollEnabled == true
-> return_state_roll == false
-> 计算 roll_error_mech / roll_error_electrical
-> updatePID(...)
-> 限幅与速率限制
-> PWM_Motor_SetAngle(MOTOR_PITCH, ...)
-> 写入 rollDiag.*
```

如果 Roll 还在回中阶段，代码会走另一条路径：

```text
rollEnabled == true
-> return_state_roll == true
-> 更新 return_state_count_roll
-> 更新 rollTargetSlew
-> PWM_Motor_SetAngle(MOTOR_PITCH, ...)
```

这条回中路径没有刷新 `rollDiag.pidRaw`、`rollDiag.pidApplied` 或 `rollDiag.holdI`。
因此，看到 `rollDiag` 数值不变化时，不能立刻判断 Roll 控制没有运行；
要先确认当前是否还处于回中路径。

### 5.11 holdIntegrators传播矩阵

`holdIntegrators` 每次进入 `computeMotorCommands()` 都先被置为 false。
随后只有 Roll 后续 PID 分支可能把它置为 true。可以把本帧传播条件整理为：

```text
Roll关闭                 -> holdIntegrators保持false -> Pitch/Yaw正常按false传入
Roll回中阶段             -> holdIntegrators保持false -> Pitch/Yaw正常按false传入
Roll后续PID且误差未超阈值 -> holdIntegrators保持false -> Pitch/Yaw正常按false传入
Roll后续PID且误差超阈值   -> holdIntegrators变true  -> Pitch/Yaw本帧按true传入
```

这说明 `holdIntegrators` 是“有条件的前向传播”。它不会从 Pitch/Yaw 反向影响本帧 Roll，
也不会在下一帧自动保持为 true，因为下一次函数入口会重新置 false。

### 5.12 三轴角度误差进入PID的差异

把三轴按 PID 入口拆开，可以看到更细的非对称性。

Roll 后续 PID 分支不是直接把机械目标角和机械当前角送入 PID。
它先用机械误差构造电角目标：

```text
roll_error_mech
-> current_electrical_angle
-> target_electrical_angle
-> updatePID(target_electrical_angle, current_electrical_angle, ...)
```

Pitch 分支的路径更直接：

```text
target_angle = pointingCmd[PITCH]
pitch_angle = PITCH_SENSOR_SIGN * sensors.margAttitude500Hz[PITCH]
-> updatePID(target_angle, pitch_angle, ...)
```

虽然代码中存在 `error_mech = wrapToPif(target_angle - pitch_angle)`，
但当前它没有参与后续 PID 输入和电角输出计算。这意味着 Pitch 调试时，
不能只观察 `error_mech`，还要确认 `updatePID()` 内部根据 `ANGULAR` 类型得到的误差。

Yaw 分支处在两者之间。它先把目标角转成靠近当前角的 `safe_target`，
再调用 PID：

```text
error_mech = wrapToPif(target_angle - yaw_angle)
safe_target = yaw_angle + error_mech
-> updatePID(safe_target, yaw_angle, ...)
```

这三条路径都能形成角度控制，但调试含义不同：

- Roll 的 PID 输入已经转换到电角层。
- Pitch 的 PID 输入仍是机械角层。
- Yaw 的 PID 输入是经过最短路径修正后的机械角层。

因此，系统架构图不能只写“姿态角进入 PID”。更准确的说法是：
不同轴在进入 PID 前经过不同的角度构造、符号处理和坐标层转换。

## 6. STM32实现机制

从 STM32 实现角度看，本章系统架构依赖六类机制。

第一，SysTick 提供调度节拍。第09章和第26章已经确认它通过 `frame_500Hz`
触发主循环中的 500Hz 实时控制循环。当前 `SysTick_Handler()` 还先检查
`systemReady == true && mpu6050Calibrating == false`，
所以系统就绪和校准状态是帧标志生成的前置条件。

第二，I2C 和 GPIO/PWM 外设承载输入输出。I2C2 读取 MPU6050，TIM2/TIM3/TIM4 输出三相 PWM。

第三，DWT/SysTick 时间基准提供 `micros()`，用于计算 `dt500Hz` 和统计 `executionTime500Hz`。本章只使用这个时间入口，不重新展开第10章和第26章的时间测量细节。

第四，主循环承担组合逻辑。采集、姿态更新、门控、PID 和输出都在主循环的 500Hz 实时控制分支中串接，而不是拆散到多个中断中执行。

第五，中断和主循环之间用布尔标志通信。`SysTick_Handler()` 只设置 `frame_500Hz`，
不直接执行 MPU6050 读取和控制计算。这样可以缩短中断占用时间，
但布尔标志也意味着如果主循环长期阻塞，多个 500Hz 触发可能被合并为一次消费。

这里还隐含一个时间边界：`startup_delay_counter`、`return_state_count` 和
`return_state_count_roll` 都随着被消费的控制帧或轴分支条件变化，而不是硬件自动按真实时间累加。
因此文档中所有“2s”和“1.3s”都应理解为理想 500Hz 条件下的换算值。

第六，当前 Debug 构建目标是 Cortex-M3，并使用软浮点 ABI。
这不禁止使用浮点 PID、AHRS 和三角函数，但要求它们共同落入 2ms 控制预算。
如果未来引入矩阵规模更大的 LQR/MPC/EKF 主链路，需要重新测量
`dt500Hz`、`executionTime500Hz`、栈使用和最坏阻塞时间。

本章不展开低频任务的完整职责。10Hz 低频任务中的状态切换和串口打印属于第32章的重点，本章只说明它对回中状态机有影响。

## 7. 项目中的应用

本章涉及的项目证据文件包括：

- `Core/Inc/main.h`
- `Core/Src/stm32f1xx_it.c`
- `Core/Src/main.c`
- `Drivers/CustomDrivers/Src/mpu6050.c`
- `Drivers/SRC/Src/MargAHRS.c`
- `Drivers/SRC/Src/computeMotorCommands.c`
- `Drivers/SRC/Inc/computeMotorCommands.h`
- `Drivers/CustomDrivers/Src/drv_pwmMotors.c`
- `Drivers/BGC/bgc32.h`

项目中的核心链路是：

```text
systemReady && !mpu6050Calibrating
-> frame_500Hz
-> MPU6050_Read_And_Process()
-> sensors.accel500Hz[] / sensors.gyro500Hz[]
-> MargAHRSupdate()
-> sensors.margAttitude500Hz[]
-> startup_delay_counter 与 eepromConfig.*Enabled
-> computeMotorCommands(dt500Hz)
-> updatePID()
-> PWM_Motor_SetAngle()
-> TIM CCR
```

这条链路把前面章节的所有主线汇合到一起。

需要注意的是，这条链路确认的是当前仓库中的软件数据流和 CCR 写入路径。

它不能单独确认外部驱动级、电机绕组接线、供电能力、实际转向和闭环稳定性已经正确。

它也不能单独确认传感器本帧有效。若 MPU6050 运行时读取失败，当前读取函数会把原始加速度和陀螺仪清零；而 `mpu_ok` 当前没有参与主运行门控。因此系统图应把“调用读取函数”和“传感器数据可信”画成两个不同节点。

这些结论仍需要第33章和第34章中的硬件诊断、仓库外实测记录和安全调参证据。

缺少实测记录时，仓库外电机响应保持【待验证】。

还要区分两种时间证据：

- `executionTime500Hz` 记录 500Hz 分支内部耗时。
- `dt500Hz` 反映两次消费 500Hz 帧之间的实际时间间隔。

如果 10Hz 打印、I2C 阻塞或其他主循环任务推迟下一次消费，
`executionTime500Hz` 可能仍然看起来正常，但 `dt500Hz` 会暴露帧间隔拉长。
因此系统级调试不能只看单个耗时变量。

`autoPan()` 和 `rollAutoPanEnabled` 等字段当前不是这条主链路的一部分。本章把它们作为自动跟随/回中主题下的待接入边界，但不把它们写成当前运行中的 Yaw 自动跟随功能。

## 8. 代码分析

### 8.1 main初始化阶段

`main()` 的初始化顺序可以分成三段。

第一段是 HAL 和外设：

```text
HAL_Init()
SystemClock_Config()
MX_GPIO_Init()
MX_TIM3_Init()
MX_TIM2_Init()
MX_TIM4_Init()
MX_I2C2_Init()
MX_TIM8_Init()
MX_USART3_UART_Init()
MX_USB_DEVICE_Init()
MX_TIM6_Init()
```

第二段是系统支撑：

```text
HAL_SYSTICK_Config()
DWT->CTRL 开启
init_eepromConfig(true)
PWM_Motor_Init()
initFirstOrderFilter()
initPID()
orientIMU()
MPU6050_Init()
```

第三段是控制状态初始化：

```text
pitchEnabled = false
rollEnabled = false
yawEnabled = false
PID参数调试覆盖
systemReady = true
previous500HzTime = micros()
```

这个顺序说明，控制循环开始前，PWM、滤波器、PID 和 IMU 都已经初始化，但三轴控制仍处于关闭状态。

这里有三个容易误读的初始化边界。

第一，`PWM_Motor_Init()` 先于三轴使能执行。它启动 PWM 通道并使 `PWM_Motor_SetAngle()` 不再因 `isInitialized == 0` 直接返回，但这只是执行层准备好，不代表 Pitch/Roll/Yaw 闭环分支已经打开。

第二，`MPU6050_Init()` 的返回值在 `main()` 中只决定打印成功或失败分支。当前没有看到 `mpu_ok = true` 这样的赋值，也没有看到 `mpu_ok` 被后续 500Hz 分支读取。因此不能把 `mpu_ok` 解释成当前已生效的传感器安全门控。

第三，`systemReady = true` 发生在 PID 参数覆盖之后。该标志会被 `SysTick_Handler()` 用作 500Hz 帧生成前置条件，也会被 10Hz 低频任务用作状态切换前置条件。它不是传感器健康标志，也不是 AHRS 收敛完成标志。

### 8.2 500Hz实时控制分支

`if (frame_500Hz)` 内部完成：

1. 清除 `frame_500Hz`。
2. 计算 `dt500Hz`。
3. 读取 MPU6050。
4. 缩放加速度和陀螺仪。
5. 调用 `MargAHRSupdate()`。
6. 执行启动延时和轴使能。
7. 调用 `computeMotorCommands(dt500Hz)`。
8. 记录 `executionTime500Hz`。

这就是项目 500Hz 实时控制路径。

但这个分支入口来自前一级中断门控：

```text
systemReady == true && mpu6050Calibrating == false
-> frameCounter++
-> frame_500Hz = true
```

也就是说，`systemReady` 在中断侧决定 500Hz 帧是否产生；一旦 `frame_500Hz` 已经被置位，主循环的 500Hz 分支本身不再重复检查 `systemReady`。

`MPU6050_Read_And_Process()` 的失败路径也属于本节必须记录的证据。
若 I2C 读失败，它会清零 `rawAccel[]` 和 `rawGyro[]` 并返回；
主循环随后仍会完成物理量缩放、姿态更新和控制函数调用。
因此排查“姿态数据异常”时，要把 I2C 返回状态、原始数组是否被清零、
`sensors.*` 物理量和 `MargAHRSupdate()` 输出放在同一张表里观察。

### 8.3 computeMotorCommands分轴结构

`computeMotorCommands()` 进入后先执行：

```text
holdIntegrators = false
```

然后按轴使能分别判断 Roll、Pitch、Yaw。

Roll 受 `eepromConfig.rollEnabled` 和 `return_state_roll` 双重影响。

Pitch 受 `eepromConfig.pitchEnabled` 和 `return_state` 影响。

Yaw 主要受 `eepromConfig.yawEnabled` 影响。

因此，三轴不是固定顺序里的三个无条件 PID，而是由使能和状态决定是否进入控制分支。

### 8.4 Pitch/Roll回中状态机

Pitch 回中状态的计数入口在 Pitch 分支：

```text
fabs(sensors.margAttitude500Hz[PITCH]) < 0.3
```

满足时增加 `return_state_count`，否则清零。

Roll 回中状态的计数入口在 Roll 分支：

```text
fabsf(pointingCmd[ROLL] - sensors.margAttitude500Hz[ROLL]) < 0.2f
```

满足时增加 `return_state_count_roll`，否则清零。

10Hz 低频任务再根据这些计数改变状态。由于计数更新在 500Hz 实时控制分支，状态切换判断在 10Hz 低频任务，实际切换时刻受两者共同影响。

按当前 `main.c` 顺序，同一轮主循环内的可见性可以写成：

```text
500Hz分支执行computeMotorCommands()
-> return_state_count*可能在函数内部更新
-> 10Hz分支检查阈值并提交return_state*或rollEnabled
-> 下一次500Hz分支才按新状态选择控制路径
```

这能解释一个常见调试现象：断点看到 `return_state_count >= 650`，
并不表示刚结束的那一次 `computeMotorCommands()` 已经使用释放后的状态。
如果 10Hz 分支是在本轮 500Hz 分支之后才提交动作，新的 `return_state*`
要到后续控制帧才对分支选择生效。

结合第5节的推导，`650` 个连续有效计数对应理想 1.3s。
若条件中途不满足，计数会清零，状态机需要重新累计。

### 8.5 holdIntegrators跨轴边界

`computeMotorCommands()` 入口先执行：

```text
holdIntegrators = false
```

Roll 后续 PID 分支中，如果电角误差超过 `ROLL_I_ENABLE_ERR_RAD`，
代码会把 `holdIntegrators` 置为 true。

同一轮函数调用中，Pitch 和 Yaw 分支随后把 `holdIntegrators` 传入 `updatePID()`。
因此这个变量不是纯 Roll 局部状态，而是可能跨轴传播的全局积分保持信号。

这类耦合调试时很容易被忽略。若发现 Pitch/Yaw 积分项不按预期增长，
除了检查各自 PID 参数，也要观察本帧 Roll 分支是否把 `holdIntegrators` 改为 true。

需要进一步限定的是：Roll 回中分支不会设置 `holdIntegrators`。
只有 `return_state_roll == false` 的后续 PID 分支中，代码才会根据
`fabsf(roll_error_electrical) > ROLL_I_ENABLE_ERR_RAD` 得到 `rollHoldIntegrators`，
并在其为 true 时把全局 `holdIntegrators` 置为 true。

因此 Pitch/Yaw 的积分是否被暂停，应按下面顺序判断：

```text
rollEnabled?
return_state_roll?
roll_error_electrical是否超过ROLL_I_ENABLE_ERR_RAD?
Pitch/Yaw本帧是否实际执行updatePID()?
```

缺少任一条件时，不能把 Pitch/Yaw 积分行为归因于 Roll 的全局保持信号。

### 8.6 输出限幅和时间尺度差异

Roll 后续 PID 分支使用：

```text
rollStepLimit = eepromConfig.rateLimit * safeDt
```

这表示 Roll 的输出步长限制与实际控制周期有关。

Pitch 和 Yaw 分支当前直接比较：

```text
outputRate[axis] > eepromConfig.rateLimit
```

这表示它们使用的是每帧差值与 `rateLimit` 的直接比较。
这一区别已经在第28章讨论过，本章只把它放回系统架构中提醒读者：
三轴共享 `computeMotorCommands()`，但输出平滑的时间尺度并不完全一致。

### 8.7 PID角度误差归一化边界

`pid.c::updatePID()` 的角度误差处理路径是：

```text
error = command - state
if PIDparameters->type == ANGULAR:
    error = standardRadianFormat(error)
```

`config.c` 初始化时把 Roll、Pitch、Yaw 三个 PID 的 `type` 都设为 `ANGULAR`。
这说明三轴 PID 都具备角度误差包裹入口。

但各轴进入该入口前的数据不同。

Roll 后续 PID 分支传入的是电角：

```text
updatePID(target_electrical_angle, current_electrical_angle, ...)
```

Pitch 分支传入的是机械角：

```text
updatePID(target_angle, pitch_angle, ...)
```

Yaw 分支传入的是用最短路径误差修正后的机械角：

```text
updatePID(safe_target, yaw_angle, ...)
```

所以看到某个轴 PID 输出异常时，应先确认异常发生在哪个层级：

```text
姿态机械角
-> 分支内符号和wrap处理
-> PID入口command/state
-> updatePID内部ANGULAR误差
-> 电角合成
-> PWM输出
```

这一层级划分尤其适合排查 `±π` 附近跳变、目标角突变和符号方向错误。
仓库源码能证明这些变量和函数关系，但不能证明实物过零点响应已经正确【待验证】。

### 8.8 autoPan边界

`autoPan()` 的输入是 `motorPos` 和 `setpoint`，内部根据死区和 `stepSmooth` 调整目标值。它确实体现了自动跟随思想。

但当前主源码搜索结果只发现 `autoPan()` 定义，没有发现调用点。`eepromConfig.rollAutoPanEnabled`、`pitchAutoPanEnabled`、`yawAutoPanEnabled` 也只见配置字段和默认值，未见参与当前控制路径。

所以本章不能写成“Yaw 当前自动跟随由 autoPan 驱动”。更准确的写法是：项目存在自动跟随函数和配置字段，但它们属于待接入边界；当前运行主线以回中状态机和轴使能切换为主要证据。

同类边界还包括一些看起来像状态机骨架的静态变量和宏：

```text
pitchTargetSlew
yawTargetSlew
yawCtrlAngle
yawAxisWasEnabled
pitchGateOpen
rollSettledTime
PITCH_TARGET_SLEW_RAD_S
YAW_TARGET_SLEW_RAD_S
YAW_CTRL_LPF_TAU_S
YAW_ERR_DEADBAND_RAD
ROLL_SETTLE_ERR_RAD / ROLL_SETTLE_CMD_RAD / ROLL_SETTLE_TIME_S
```

当前主源码搜索结果显示，Pitch/Yaw 目标斜坡宏和这些静态变量没有形成运行链路。
例如当前实际调用 `moveTowardsAnglef()` 的路径仍集中在 Roll 轴。
因此本章不能把它们写成当前已经运行的 Pitch/Yaw 目标斜坡、Yaw 低通控制或 Roll
收敛门控算法。它们更适合作为后续重构或待接入设计的证据边界。

### 8.9 输出到执行器

`computeMotorCommands()` 最终通过 `PWM_Motor_SetAngle()` 输出。它不直接写 TIM CCR，而是先完成机械角、电角、PID 和限幅逻辑，再把电气角度和功率百分比交给底层电机驱动。

底层 `drv_pwmMotors.c` 再根据 `MotorAxis_t` 写 TIM2/TIM3/TIM4 的 CCR。这个分层使得控制层不用直接关心每个 GPIO 引脚，但也要求 `MotorAxis_t` 映射保持清楚。

因此，系统级调试不能只停在“某个控制分支调用了输出函数”。至少要同时记录：

```text
控制分支：Roll / Pitch / Yaw
传入枚举：MOTOR_PITCH / MOTOR_ROLL / MOTOR_YAW
寄存器组：TIMx->CCRn
外部证据：GPIO波形、电机相线、真实轴响应
```

前三项属于仓库内和 MCU 寄存器层证据；外部证据缺失时，本章只能证明软件链路连通，
不能证明物理轴响应、转向或闭环效果正确。

#### 8.9.1 诊断旁路与主闭环分离

`PWM_Motor_Hardware_Diagnosis()` 也是一个能直接写 CCR 的底层函数，但它和
`computeMotorCommands()` / `PWM_Motor_SetAngle()` 的角色不同。当前仓库源码中它确实存在；
`Debug/Three-axis_cloud_platformV2.map` 中也能看到 `.text.PWM_Motor_Hardware_Diagnosis`
停留在 `0x00000000` 输入段记录。与此相对，`PWM_Motor_SetAngle` 在 `.map` 中有
`0x08002748` 最终 Flash 地址，`computeMotorCommands` 有 `0x080043b4` 最终 Flash 地址，
并且 `Debug/Three-axis_cloud_platformV2.list` 能看到 500Hz 主路径调用
`computeMotorCommands(dt500Hz)` 以及 `computeMotorCommands()` 内多处 `PWM_Motor_SetAngle()` 调用。

所以系统架构图不能把 `PWM_Motor_Hardware_Diagnosis()` 画成 500Hz 主闭环的一部分。
更准确的画法是：它属于诊断旁路，能够在调试时绕开主控制链路直接驱动同一批 CCR；
但当前构建产物只能证明“函数定义存在、主路径未见调用入口”，不能证明它已经参与运行，
也不能证明它是否会与主闭环发生争用。这部分仍需后续调试记录继续验证【待验证】。

#### 8.9.2 控制分支到执行器的四层证据边界

当我们把三轴控制从 `computeMotorCommands()` 往下追踪时，证据不能只停在“函数被调用了”这一层。更稳妥的拆法是把它分成四层：

| 层级 | 需要记录的对象 | 当前仓库能证明什么 | 不能证明什么 |
|---|---|---|---|
| 控制分支 | `Roll / Pitch / Yaw` 的 `if` 分支 | 500Hz 控制循环确实进入了某个轴分支 | 不能证明该轴已经进入后续 PID 或输出阶段 |
| 执行器枚举 | 传入 `PWM_Motor_SetAngle()` 的 `MotorAxis_t` | 源码能直接看到 `MOTOR_PITCH`、`MOTOR_ROLL`、`MOTOR_YAW` 的实参 | 不能把枚举名直接等同于真实机械轴 |
| CCR 写入 | `drv_pwmMotors.c` 里的 `TIM2->CCRn` / `TIM3->CCRn` / `TIM4->CCRn` | 能证明最终写到了哪组定时器比较寄存器 | 不能证明外部波形、驱动相序或转向已经正确 |
| 机械响应 | 电机转向、力矩、抖动、温升 | 当前仓库没有直接证据 | 需要示波器、接线和实机测量，仍为【待验证】 |

这个拆分的意义是：调试时先确认“控制分支名”和“传入枚举”是否符合预期，再确认“CCR 写入目标”是否正确，最后才讨论实物响应。否则很容易把控制层的命名误读成硬件事实。

### 8.10 rollDiag字段刷新边界

`computeMotorCommands.h` 中 `rollDiag_t` 定义了多个 Roll 诊断字段：

```text
targetMechRad, targetSlewMechRad, currentRawMechRad, currentCtrlMechRad
errMechRad, errElecRad, targetElecRad, currentElecRad
pidRaw, pidClamped, pidApplied, dPidRaw, stepLimit, holdI
```

但在 `computeMotorCommands.c` 中，`rollDiag.*` 的赋值集中出现在 Roll 后续 PID 分支末尾。
Roll 回中分支只更新 `return_state_count_roll`、`rollTargetSlew` 并调用
`PWM_Motor_SetAngle(MOTOR_PITCH, ...)`，没有同步刷新这些诊断字段。

因此 `rollDiag` 的正确读法是：

```text
rollEnabled == true && return_state_roll == false -> rollDiag可作为本帧Roll PID诊断
其它情况 -> rollDiag可能是初值或旧值，不能单独作为本帧证据
```

这类“诊断变量有效期”是系统级调试的一部分。变量存在、能在调试器中看到，
并不等于它每个控制阶段都代表当前状态。

### 8.11 门控变量对照表

复查系统架构时，可以把关键变量整理成对照表：

| 变量/状态 | 当前证据 | 架构结论 |
| --- | --- | --- |
| `systemReady` | `main()` 置 true；`SysTick_Handler()` 和 10Hz 任务读取 | 初始化完成后的调度/低频状态门控，不是传感器健康门控 |
| `mpu6050Calibrating` | `SysTick_Handler()` 读取 | 校准期间阻止 500Hz 帧生成 |
| `mpu_ok` | `main.c` 定义为 false，当前未发现置 true 或参与读取 | 不能写成已生效 MPU 运行门控 |
| `startup_delay_counter` | 500Hz 分支中累加到 1000 后开启 Pitch/Yaw | 固定消费帧延时，不是姿态变化量收敛检测 |
| `attitude_deg_prev[]` / `attitude_delta_initialized` | 当前只发现定义 | 不能写成当前实际收敛判断依据 |
| `eepromConfig.*Enabled` | `computeMotorCommands()` 分轴读取 | 轴级控制分支门控 |
| `return_state` / `return_state_roll` | Pitch/Roll 分支和 10Hz 任务共同使用 | 轴内回中/后续控制阶段门控 |
| `isInitialized` | `PWM_Motor_Init()` 设置，`PWM_Motor_SetAngle()` 读取 | 执行层函数入口门控，不代表闭环已使能 |

这张表的教学价值在于：它把“系统能跑起来”“传感器可信”“轴被允许控制”“当前阶段允许 PID”“底层函数能写 CCR”拆成不同问题。调试时必须逐层确认，不能看到某一个标志为 true 就推断整条闭环已经有效。

### 8.12 构建产物证据边界

复查系统架构时，除了读源码，还可以把当前 Debug 构建生成的 `.map` 和 `.list` 当作第二层证据。
这层证据回答的是“哪些符号进入了最终镜像”和“编译后的主路径是否仍保留这些调用”，不是回答“真实云台是否已经稳定工作”。

当前 `Debug/Three-axis_cloud_platformV2.map` 可以证明下列函数或对象进入了 Debug 镜像：

- 控制调度和执行函数：`.text.SysTick_Handler`、`.text.MPU6050_Read_And_Process`、`.text.MargAHRSupdate`、`.text.computeMotorCommands`、`.text.updatePID`、`.text.zeroPIDintegralError`、`.text.zeroPIDstates`、`.text.PWM_Motor_SetAngle`。
- 控制路径中的全局对象或门控变量：`.bss.sensors`、`.bss.systemReady`、`.bss.frame_500Hz`、`.bss.startup_delay_counter`、`.bss.isInitialized`、`.bss.mpu6050Calibrating`、`.data.holdIntegrators`、`.bss.rollDiag`。
- `autoPan` 在 `.map` 的输入段中可见为 `.text.autoPan 0x00000000`，说明源码函数存在并参与对象文件，但当前未看到带最终 Flash 地址的主控制路径调用证据，因此仍应按“定义存在、主路径未接入”描述。

当前 `Debug/Three-axis_cloud_platformV2.list` 可以进一步证明编译后的调用链仍保留了这些关键步骤：

- `SysTick_Handler()` 在满足 `systemReady == true && mpu6050Calibrating == false` 后设置 `frame_500Hz = true`。
- `main()` 消费 `frame_500Hz`，随后调用 `MPU6050_Read_And_Process()`、`MargAHRSupdate(...)`，并在 `startup_delay_counter` 达到 1000 时调用 `zeroPIDintegralError()` 和 `zeroPIDstates()`。
- `main()` 在 500Hz 分支中调用 `computeMotorCommands(dt500Hz)`。
- `computeMotorCommands()` 入口写 `holdIntegrators = false`，随后在 Roll/Pitch/Yaw 分支中分别保留 `updatePID()` 和 `PWM_Motor_SetAngle()` 调用。
- Roll 后续 PID 分支中保留 `rollDiag.*` 字段刷新语句，因此 `rollDiag` 是“特定分支刷新”的诊断对象，而不是任何时刻都有效的全局状态快照。

但系统架构章尤其要注意 Debug 构建快照新鲜度。当前源码在
`startup_delay_counter == 1000` 后写入：

```text
pitchEnabled = true
rollEnabled  = false
yawEnabled   = true
```

而当前 `Debug/Three-axis_cloud_platformV2.list` 对应片段仍显示：

```text
pitchEnabled = true
rollEnabled  = true
yawEnabled   = false
```

同一份 `.list` 中，Yaw 分支调用 `PWM_Motor_SetAngle(MOTOR_YAW, ..., 40.0f)`，
而当前源码中 Yaw 分支调用参数是 `30.0f`。这说明当前 Debug `.list`
至少在“启动后轴使能组合”和“Yaw 输出功率参数”两处不等于最新源码文本。

因此，本章使用 `.list` 时应拆成三层：

| 证据层 | 可以证明 | 不能证明 |
|---|---|---|
| 当前源码 | 最新文本中的门控组合、功率参数和状态切换意图 | 这些文本一定已经重新构建并烧录 |
| 当前 `.list` | 某个 Debug 快照中保留了 500Hz 主链路、PID 调用和 PWM 输出路径 | 每个立即数、布尔写入值都与最新源码一致 |
| 重新构建后的 `.list` / 固件版本记录 | 可以把源码文本和实际镜像再次对齐 | 仍不能替代运行时日志、示波器或电机响应验证 |

所以，系统架构图中的“当前源码行为”应以 `Core/Src/main.c` 和
`Drivers/SRC/Src/computeMotorCommands.c` 为准；`.list` 当前只适合证明
架构路径曾进入 Debug 构建，不适合直接证明最新轴使能策略已经进入最终镜像。
若要宣称某次烧录固件采用了最新门控组合，需要重新构建后复查 `.list`，
并记录对应固件版本。缺少这一步时，硬件实际运行的轴使能组合保持【待验证】。

`.su` 和 `.cyclo` 还能把这条系统主链路的静态资源边界拆开看：

| 链路位置 | 函数 | 静态栈用量 | 圈复杂度 | 构建产物证据 |
|---|---|---:|---:|---|
| 500Hz置位 | `SysTick_Handler()` | 16 字节 | 5 | `Debug/Core/Src/stm32f1xx_it.su` / `.cyclo` |
| 主循环调度 | `main()` | 56 字节 | 7 | `Debug/Core/Src/main.su` / `.cyclo` |
| 传感器读取 | `MPU6050_Read_And_Process()` | 64 字节 | 3 | `Debug/Drivers/CustomDrivers/Src/mpu6050.su` / `.cyclo` |
| 姿态解算 | `MargAHRSupdate()` | 80 字节 | 33 | `Debug/Drivers/SRC/Src/MargAHRS.su` / `.cyclo` |
| PID更新 | `updatePID()` | 64 字节 | 34 | `Debug/Drivers/SRC/Src/pid.su` / `.cyclo` |

这张表的重点不是证明“系统实时性已经满足”，而是帮助读者识别架构中最值得继续量测的节点。
当前静态复杂度最高的两个业务函数是 `computeMotorCommands()` 和 `updatePID()`，它们分别承接三轴分支和角度型 PID 误差处理；
`MargAHRSupdate()` 也有较高圈复杂度，因为姿态解算内部包含归一化、误差反馈和数值保护。
如果后续要证明 500Hz 闭环有足够裕量，不能只引用这张表，还要用第10章的 DWT 计时方法或 GPIO 翻转法记录每个阶段的真实耗时【待验证】。

这些构建产物能够加强本章的架构结论：500Hz 标志、传感器读取、姿态解算、启动清零、三轴 PID 入口和 PWM 输出入口确实进入了当前固件构建。
但它们仍然不能证明运行时传感器数据可信、姿态已经收敛、PID 参数正确、三轴闭环稳定、电机真实响应正确、外部波形满足预期，或 500Hz 实时性在全部调试打印和硬件负载下满足要求。
这些结论仍需要日志、断点记录、波形、主机侧记录或安全实测；缺少证据时保持【待验证】。

### 本节证据边界

本节只根据当前仓库说明文件、函数、宏、变量和调用关系。运行时频率、外部硬件表现、主机侧现象、传感器方向、电机响应或真实控制效果仍需调试记录、日志或仓库外实测证据；缺少证据时保持【待验证】。

## 9. 调试方法

第一步，确认实时调度。

- 先观察 `systemReady` 是否已经为 true。
- 观察 `mpu6050Calibrating` 是否为 false。
- 观察 `frame_500Hz` 是否周期性置位并在主循环中清除。
- 观察 `dt500Hz` 是否接近 0.002 秒。
- 观察 `executionTime500Hz` 是否明显小于 2ms。
- 同时记录 `dt500Hz` 和 `executionTime500Hz`，区分帧间隔拉长和分支内部耗时过长。
- 若 10Hz 打印开启，注意 `_write()` 逐字节调用 `HAL_UART_Transmit()` 会占用主循环时间。

第二步，确认传感器到姿态链路。

- 搜索并确认 `mpu_ok` 当前是否参与运行门控。
- 观察 `MPU6050_Init()` 成功/失败打印不能替代运行时数据有效性判断。
- 观察 `rawAccel[]`、`rawGyro[]` 是否更新。
- 若 `rawAccel[]`、`rawGyro[]` 同时归零，回到 `MPU6050_Read_And_Process()` 检查 I2C 读返回路径。
- 观察 `sensors.accel500Hz[]`、`sensors.gyro500Hz[]` 是否随之变化。
- 观察 `sensors.margAttitude500Hz[]` 是否由 `MargAHRSupdate()` 更新。

第三步，确认启动门控。

- 开机初期观察 `startup_delay_counter`。
- 把 `startup_delay_counter` 理解为已消费 500Hz 帧计数，而不是独立墙钟计时。
- 搜索 `attitude_deg_prev[]` 和 `attitude_delta_initialized`，确认它们当前是否参与收敛判断。
- 计数到 1000 后观察 `pitchEnabled`、`rollEnabled`、`yawEnabled`。
- 确认 `zeroPIDintegralError()` 和 `zeroPIDstates()` 在使能前执行。

第四步，确认回中状态。

- 观察 `return_state`、`return_state_count`、`return_state_roll`、`return_state_count_roll`。
- 注意计数更新在 500Hz 实时控制分支，状态切换判断在 10Hz 低频任务。
- 把 `return_state_count=650` 换算为理想 1.3s，并记录实际释放时间是否额外延迟【待验证】。
- 若断点显示计数刚达到阈值，但本次控制仍走旧分支，检查断点顺序：500Hz 分支已经执行完，10Hz 分支随后才提交状态，下一次 500Hz 控制帧才会看见新状态。

第五步，确认分轴输出。

- 先确认 `PWM_Motor_Init()` 是否已设置执行层初始化状态。
- Roll 分支观察 `rollDiag`、`pidCmd[ROLL]` 和传入 `PWM_Motor_SetAngle()` 的 `MOTOR_PITCH`。
- Pitch 分支观察 `pidCmd[PITCH]` 和传入的 `MOTOR_ROLL`。
- Yaw 分支观察 `pidCmd[YAW]` 和传入的 `MOTOR_YAW`。
- 继续沿第30章映射表记录这些枚举最终写入的 TIMx->CCRn 和 GPIO 引脚。
- 最后单独记录真实机械轴响应、转向、相序和温升；没有实测时不要把软件轴名写成硬件结论。
- 这些变量和 CCR 变化只能确认软件输出路径；若要确认实际电机方向和闭环效果，还需要仓库外实测、温升记录和安全夹具条件，未测时保持【待验证】。

第六步，确认 PID 角度误差入口。

- 在 Roll 后续 PID 分支记录 `target_electrical_angle` 和 `current_electrical_angle`。
- 在 Pitch 分支记录 `target_angle`、`pitch_angle` 和局部 `error_mech`。
- 在 Yaw 分支记录 `target_angle`、`yaw_angle`、`error_mech` 和 `safe_target`。
- 在 `updatePID()` 内确认 `PIDparameters->type == ANGULAR` 时执行了误差包裹。
- 若在 `±π` 附近出现跳变，应区分分支内 `wrapToPif()` 和 PID 内 `standardRadianFormat()`。

第七步，确认跨轴积分保持。

- 在 `computeMotorCommands()` 入口观察 `holdIntegrators=false`。
- 在 Roll 后续 PID 分支观察 `rollHoldIntegrators`。
- 若 `holdIntegrators` 被 Roll 置为 true，继续观察 Pitch/Yaw 的 `updatePID()` 是否收到 true。
- 若 Roll 关闭或仍处于 `return_state_roll == true`，不要期待本帧 Roll 改写 `holdIntegrators`。
- 不要把 Pitch/Yaw 积分不增长直接归因于本轴参数，先排查全局积分保持标志。

第八步，确认 rollDiag 是否代表本帧。

- 观察 `eepromConfig.rollEnabled`。
- 观察 `return_state_roll`。
- 只有 Roll 已使能且 `return_state_roll == false` 时，才把 `rollDiag` 作为本帧 Roll PID 诊断。
- 若 Roll 仍在回中阶段，记录 `return_state_count_roll` 和 `rollTargetSlew`，不要只看 `rollDiag`。

第九步，确认 autoPan 边界。

- 搜索 `autoPan(`，确认当前是否有调用点。
- 若只有定义没有调用，就不要把自动跟随写成当前运行功能。
- 同时搜索 `pitchTargetSlew`、`yawTargetSlew`、`yawCtrlAngle`、`pitchGateOpen` 等变量。
- 对只见定义、未见运行路径读写的变量，标记为预留或未接入边界。

调试记录建议：

- 记录从 `frame_500Hz` 到传感器读取、姿态解算、PID、电角和 PWM 的完整链路。
- 对启动和回中状态，应记录计数器、状态变量、使能字段、清零函数、阈值换算和 10Hz/500Hz 分工。
- 对门控变量，应分开记录 `systemReady`、`mpu6050Calibrating`、`mpu_ok`、`startup_delay_counter`、轴使能、`return_state*` 和 `isInitialized`。
- 对传感器有效性，应记录初始化结果、I2C 读返回路径、原始数组是否清零和姿态输出。
- 分轴输出记录应包含控制分支、传入枚举、PID 输出、电角结果和最终 CCR 写入入口。
- 对角度误差，应记录分支内包裹值、PID 入口 command/state 和 `ANGULAR` 类型。
- 对跨轴状态，应记录 `holdIntegrators`、`rollHoldIntegrators`、Pitch/Yaw 积分项和本帧轴执行顺序。
- 对 Roll 诊断，应记录 `rollDiag` 是否在本帧刷新；不能只记录字段值。
- 对状态机变量，应区分已运行状态、扩展状态变量、守卫条件、动作和未接入变量。
- 对构建产物，应记录 `.map` 中函数/对象是否带最终地址、`.list` 中主路径是否保留对应调用，并把输入段 `0x00000000` 记录为边界而不是运行证据。
- 真实闭环效果、三轴方向、负载响应和温升缺少仓库外安全实测时保持【待验证】。

## 10. 常见问题

问题一：三轴是否在 AHRS 初始化后立刻全部开启。

不是。AHRS 收敛计数达到 1000 后，代码开启 Pitch 和 Yaw，Roll 仍为 false。Roll 后续由 10Hz 低频任务根据 `return_state_count` 打开。

问题二：`computeMotorCommands()` 是否每帧都对三轴执行 PID。

不是。它每个 500Hz 帧会被调用，但内部三轴分支受 `eepromConfig.*Enabled` 和回中状态影响。某个轴关闭时不会进入该轴控制分支。

问题三：`autoPan()` 是否正在驱动 Yaw 自动跟随。

当前没有运行证据。`autoPan()` 存在并实现了死区和平滑逻辑，但当前主控制路径未发现调用点。

问题四：为什么 Roll 控制分支传入 `MOTOR_PITCH`。

这是当前项目的控制轴到执行器枚举映射结果。第30章已经说明实际 CCR 写入要以 `PWM_Motor_SetAngle()` 的 `switch` 为准；第31章只记录控制层调用关系，不把它解释成已经证明的硬件安装原因。

问题五：10Hz 低频任务是否属于本章主线。

本章只说明 10Hz 低频任务影响回中状态切换。10Hz 状态任务、运行状态门控和异常防护的完整分析属于第32章。

问题六：为什么启动前要清 PID 状态。

AHRS 收敛阶段姿态可能不稳定，如果积分项或 D 项历史状态在此阶段累积，开控制瞬间可能出现突变。`zeroPIDintegralError()` 和 `zeroPIDstates()` 用于降低这个风险。

问题七：三轴控制架构串起来后，是否表示每个环节都已经闭环验证。

不能。
架构图和调用链只能证明数据从传感器、姿态、PID 到电机输出的主线关系。
闭环是否正确，还要分别验证方向、比例、限幅、门控、三相映射和实物响应。
教材应把本章作为整合视角，而不是把整合关系等同于仓库外调试完成。

问题八：为什么 `return_state_count >= 650` 不等于精确 1.3 秒立刻释放。

650 个计数在 500Hz 下对应理想 1.3s，但释放判断在 10Hz 低频任务中。
因此计数达到阈值后，还要等下一次 10Hz 检查。
如果主循环被阻塞，释放时间还会继续后移【待验证】。

问题九：为什么 Pitch/Yaw 积分异常时也要看 Roll 分支。

因为 `holdIntegrators` 是全局变量。Roll 后续 PID 分支可能把它置为 true，
而 Pitch/Yaw 随后把同一个标志传入 `updatePID()`。
所以某个轴的积分行为可能受到前面轴分支状态影响。

问题十：为什么本项目当前采用 PID 主循环架构，而不是 LQR 或 MPC。

当前仓库实现的是 PID、限幅、斜坡和电角合成，没有状态空间矩阵、
代价函数、约束求解器或预测时域证据。
在 Cortex-M3 软浮点构建和 2ms 控制预算下，PID 主循环架构更容易验证和调试。
LQR/MPC 不是概念上不能用，而是需要额外模型、算力评估和实测证据。

问题十一：`rollDiag` 有数值，是否一定代表本帧 Roll PID 正在运行。

不一定。`rollDiag` 的赋值只出现在 Roll 后续 PID 分支。
如果 Roll 关闭或仍处于 `return_state_roll == true` 的回中阶段，
这些字段可能是初值或旧值。必须同时检查 `rollEnabled` 和 `return_state_roll`。

问题十二：为什么 `holdIntegrators` 是全局变量，却不是每一帧都会影响 Pitch/Yaw。

因为它在 `computeMotorCommands()` 入口先被置为 false。
只有 Roll 后续 PID 分支执行，并且 `roll_error_electrical` 超过阈值时，
它才会被置为 true，并向本帧后续 Pitch/Yaw 分支传播。

问题十三：源码里有 `pitchTargetSlew`、`yawTargetSlew` 等变量，是否说明 Pitch/Yaw 目标斜坡已经接入。

不能这样判断。
当前搜索证据显示这些变量和相关宏主要是定义存在，未形成当前主控制路径中的读写链路。
教材应把它们列为预留或未接入边界，而不是写成已经运行的 Pitch/Yaw 斜坡或 Yaw 低通控制。

问题十四：Pitch 分支已经计算 `error_mech`，是否说明它直接用该变量做 PID 误差。

不能。
当前 Pitch 分支调用 `updatePID(target_angle, pitch_angle, ...)`，
没有把局部 `error_mech` 作为 PID 输入。
Pitch 的角度误差包裹主要依赖 `updatePID()` 内部的 `ANGULAR` 类型处理。

问题十五：`systemReady` 是否只影响 10Hz 低频任务。

不是。当前 `SysTick_Handler()` 也读取 `systemReady`，只有 `systemReady == true`
且 `mpu6050Calibrating == false` 时才推进 `frameCounter` 并产生 `frame_500Hz`。
但主循环消费 `frame_500Hz` 后，500Hz 分支内部不再二次检查 `systemReady`。

问题十六：`mpu_ok` 是否会在 MPU6050 初始化失败时阻止控制链路。

当前不能这样写。`mpu_ok` 被定义为 false，但当前搜索没有看到它在初始化成功后置 true，
也没有看到 500Hz 分支或 `computeMotorCommands()` 读取它。
初始化失败会打印信息；是否阻止后续运行，当前没有基于 `mpu_ok` 的代码证据。

问题十七：PWM 通道已经在初始化阶段启动，为什么还要看轴使能。

因为 PWM 初始化属于执行层准备，轴使能属于控制层门控。
`PWM_Motor_Init()` 让底层 `PWM_Motor_SetAngle()` 具备写 CCR 的前提，
但 `computeMotorCommands()` 是否进入某个轴分支仍由 `eepromConfig.*Enabled`
和 `return_state*` 决定。硬件通道已启动不等于闭环控制已经接管。

问题十八：`startup_delay_counter == 1000` 是否证明 AHRS 已经真实收敛。

不能。当前代码用的是固定 500Hz 消费帧计数，并未看到
`attitude_deg_prev[]` 或 `attitude_delta_initialized` 参与姿态变化量判断。
所以它只能证明等待计数达到阈值，不证明姿态稳定性已经由数据验证完成【待验证】。

问题十九：为什么计数已经达到 650，本次 `computeMotorCommands()` 仍可能走旧分支。

因为当前主循环先执行 500Hz 分支，`computeMotorCommands()` 在这里完成本帧分支选择和计数更新；
随后 10Hz 分支才检查阈值并提交 `return_state*` 或 `rollEnabled` 变化。
如果计数是在这一次 500Hz 分支内刚达到阈值，状态提交发生在函数返回之后，
新的门控只会影响后续 500Hz 控制帧。

## 12. 思考题

1. 为什么项目把 500Hz 实时控制循环放在主循环中由标志触发，而不是直接放在 SysTick 中断里执行。
2. 为什么 AHRS 收敛后只先开启 Pitch 和 Yaw，而 Roll 需要后续条件打开。
3. 如果 `return_state_count` 一直不能达到 650，系统会停留在哪种控制状态。
4. 为什么三轴共享 `computeMotorCommands()`，但不能写成三轴完全对称。
5. `autoPan()` 属于待接入边界时，教材和调试文档应如何描述它。
6. 为什么控制层调用 `PWM_Motor_SetAngle()` 而不是直接写 TIM CCR。
7. 如果要新增一个系统状态监控变量，应该放在 500Hz 实时路径还是 10Hz 低频路径，依据是什么。
8. 如果某个轴在架构图里连通，为什么仍要分别确认它的使能、门控和输出路径。
9. 为什么 `executionTime500Hz` 正常时，`dt500Hz` 仍可能暴露调度延迟。
10. `holdIntegrators` 作为全局变量，会给分轴调试带来什么隐含耦合。
11. 如果要把当前 PID 架构升级为 LQR/MPC，需要新增哪些模型和验证证据。
12. 为什么 `rollDiag` 字段有值，不等于它一定在本帧被刷新。
13. 为什么判断 Pitch/Yaw 积分暂停时，要同时记录 Roll 分支是否进入后续 PID 阶段。
14. 为什么 Pitch 分支计算了 `error_mech`，仍要检查 `updatePID()` 的 `ANGULAR` 类型。
15. 为什么 `systemReady` 允许 500Hz 帧生成，不等于传感器数据一定可信。
16. 为什么 `mpu_ok` 变量存在，不等于它已经成为运行时安全门控。
17. 为什么固定 1000 个控制帧延时不能替代真正的姿态收敛判定。
18. 为什么 PWM 通道已经启动，仍不能直接推断某个轴已经进入闭环控制。
19. 为什么 `return_state_count` 在本轮达到 650，不等于本轮 `computeMotorCommands()` 已经使用释放后的状态。
20. 为什么 `.map/.list` 能证明符号和调用路径进入当前构建，却不能证明传感器可信、闭环稳定或电机真实响应。

## 13. 本章总结

本章把前30章的模块合并为三轴云台项目的系统控制架构。

当前项目中：

- `SysTick_Handler()` 在 `systemReady == true && mpu6050Calibrating == false` 时设置 `frame_500Hz`，主循环消费该标志。
- SysTick硬件节拍、`frame_500Hz` 软件置位、主循环消费、控制链完成和外部闭环效果是五个证据层级，不能互相替代。
- 500Hz 实时控制分支读取 MPU6050、缩放传感器物理量、调用 `MargAHRSupdate()`、执行启动门控并调用 `computeMotorCommands()`。
- `mpu_ok` 当前没有看到参与 500Hz 或控制输出门控的证据，不能写成已生效传感器健康开关。
- `MPU6050_Read_And_Process()` 运行时读失败会清零原始传感器数组，调用链存在不等于本帧数据可信。
- `startup_delay_counter` 计的是被消费的 500Hz 控制帧，不是独立墙钟时间，也不是姿态变化量收敛检测。
- `startup_delay_counter` 到 1000 后，项目清空 PID 状态并开启 Pitch/Yaw，Roll 仍需后续回中条件打开。
- `PWM_Motor_Init()` 启动底层 PWM 通道，但三轴闭环是否执行仍由轴使能和轴内状态决定。
- `computeMotorCommands()` 是三轴控制枢纽，但内部 Roll、Pitch、Yaw 分支并不对称。
- 当前运行证据中的 `自动跟随/回中算法` 主要体现为 Pitch/Roll 回中状态和使能切换。
- Pitch/Roll 回中计数在 500Hz 分支累积，状态释放在 10Hz 低频任务提交。
- 若 10Hz 提交发生在同一轮 500Hz 分支之后，新的回中/使能状态只能影响后续 500Hz 控制帧。
- `holdIntegrators` 是可能跨轴传播的全局积分保持标志。
- `holdIntegrators` 的跨轴影响只在 Roll 后续 PID 分支满足误差阈值时发生。
- Roll、Pitch、Yaw 进入 PID 前的角度误差构造不同，不能只用统一模板解释。
- `rollDiag` 只在 Roll 后续 PID 分支刷新，Roll 关闭或回中阶段不能单独依赖它判断本帧状态。
- `autoPan()` 和 AutoPan 配置字段存在，但当前主控制路径未发现调用证据。
- `pitchTargetSlew`、`yawTargetSlew`、`yawCtrlAngle` 等变量当前没有形成主控制路径证据。
- 状态机分析应区分状态、扩展状态变量、守卫条件、动作和未接入变量。
- `.map/.list` 可以证明关键函数、全局对象和主路径调用进入当前 Debug 构建，但不能替代运行时闭环、外部波形或硬件响应证据。
- 控制层最终通过 `PWM_Motor_SetAngle()` 进入第30章讲过的三相 PWM 执行器链路。
- 系统级输出证据应拆成控制分支名、`MotorAxis_t`、TIMx->CCRn/GPIO 和真实机械响应四层；前三层可由仓库与寄存器证明，真实机械响应仍需外部实测。
- `.su/.cyclo` 可以给出系统主链路函数的静态栈用量和圈复杂度边界，但不能替代 DWT 计时、GPIO 翻转或真实负载下的实时性证明。

本章边界：

- 本章证明系统主线如何串联，不证明每个轴已经完成闭环实测。
- 架构连通仍需分别验证姿态方向、PID 参数、轴使能、门控状态和电机输出响应。
- 轴名、枚举名、物理接口名和实物机械轴响应不能互相替代。
- 调度门控、传感器初始化提示、启动延时、轴使能、回中状态和执行层初始化是不同层级，不能互相替代。
- `dt500Hz`、`executionTime500Hz` 和 10Hz 打印耗时需要联合观察，不能用单一指标证明实时性。
- 构建产物只能证明当前 Debug 镜像中的符号、地址和反汇编调用关系，不能证明运行时数据质量、控制稳定性或硬件安全性。
- 静态栈和圈复杂度只能提示资源审查优先级，不能直接证明 500Hz 主链路在所有硬件状态下都有实时裕量。

下一章将进入 `低频任务、运行门控与异常防护`，继续分析 10Hz 低频任务、运行状态门控和 NaN/Inf 等异常防护如何支撑系统稳定运行。
