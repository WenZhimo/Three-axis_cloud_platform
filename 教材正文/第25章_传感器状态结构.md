# 第25章 传感器状态结构

## 1. 本章目标

第21章到第24章已经分别讲过传感器物理量、当前主线姿态解算和备用姿态方案。现在需要把这些结果放回同一个数据容器中理解：`传感器数据结构sensors_t`。

本章目标是：

- 理解 `sensors_t` 为什么是项目 500Hz 数据流的公共状态容器。
- 分清 `accel500Hz`、`gyro500Hz`、`margAttitude500Hz`、`evvgcCFAttitude500Hz` 和 `mag10Hz` 的含义。
- 明确当前主线中每个字段的写入者和读取者。
- 区分当前控制使用的姿态数组和备用姿态数组。
- 区分“字段已经被更新或读取”和“控制轴已经使能并产生有效输出”。
- 为第26章 500Hz 实时控制循环和第27章 PID 控制建立数据入口。

本章知识链为：

`知识点总表` 中的 `传感器数据结构sensors_t`
-> `知识依赖图` 中 `传感器数据结构sensors_t` 依赖 `传感器物理量缩放` 和 `欧拉角计算`
-> `学习优先级` 中 P1 核心基础知识
-> `教学顺序` 第43项
-> `教材章节` 第25章。

## 2. 前置知识

本章正式前置知识包括：

- `传感器物理量缩放`
- `欧拉角计算`

第17章已经讲过 `全局配置结构eepromConfig_t`，它用于保存配置参数；本章的 `sensors_t` 用于保存运行时传感器和姿态状态。二者都是全局结构，但用途不同：

- `eepromConfig_t` 偏向“系统配置和控制参数”。
- `sensors_t` 偏向“每帧运行数据和姿态结果”。

本章不展开 500Hz 调度机制，也不展开 PID 算法。它只回答一个问题：项目把传感器数据和姿态数据放在哪里，谁写入它，谁读取它。

## 3. 问题背景

三轴云台项目不是一个函数从头算到尾。传感器读取、物理量缩放、姿态解算、控制计算和调试输出分散在不同文件中。如果每个模块都直接访问原始全局变量，数据流会很难追踪。

`sensors_t` 的作用是提供一个统一状态容器，让关键运行数据有固定位置：

```text
MPU6050 原始数据
-> sensors.accel500Hz / sensors.gyro500Hz
-> sensors.margAttitude500Hz
-> computeMotorCommands()
```

它解决的是“模块之间如何交接数据”的问题，而不是某个传感器或控制算法本身的问题。

读懂 `sensors_t` 后，读者就能顺着字段追踪项目主线：哪个模块负责写入，哪个模块负责读取，哪些字段当前只是保留。

## 4. 核心概念

### 4.1 sensors_t

`sensors_t` 定义在 `Drivers/BGC/bgc32.h`：

```c
typedef struct sensors_t
{
    float accel500Hz[3];
    float evvgcCFAttitude500Hz[3];
    float margAttitude500Hz[3];
    float gyro500Hz[3];
    float mag10Hz[3];
} sensors_t;
```

它是一个结构体类型。项目在 `Core/Src/main.c` 中定义全局实例：

```c
sensors_t sensors;
```

后续代码访问的 `sensors.accel500Hz`、`sensors.gyro500Hz` 等字段，都是这个全局实例中的成员。

### 4.2 字段不是同等主线

`sensors_t` 中每个字段都存在，但当前项目主线并不等同使用所有字段。

当前主线明确使用：

- `sensors.accel500Hz`
- `sensors.gyro500Hz`
- `sensors.margAttitude500Hz`

备用或未进入当前主线的字段包括：

- `sensors.evvgcCFAttitude500Hz`
- `sensors.mag10Hz`

判断字段是否进入主线，不能只看结构体定义，必须看写入点和读取点。

更工程化地说，每个字段都应该回答四个问题：

```text
谁写入？
谁读取？
单位是什么？
这个值在当前帧是否新鲜？
```

如果只能回答“结构体里有这个字段”，还不能说明它是当前有效运行数据。`sensors_t`
是公共状态表，不是事实保证表。字段存在只是提供存储位置；是否有效，要由数据流证明。

还要再拆一层：字段被读取，不等于控制输出已经生效。当前 `Core/Src/main.c` 每次 500Hz
分支都会调用 `computeMotorCommands(dt500Hz)`，但 `computeMotorCommands()` 内部还要看
`eepromConfig.rollEnabled/pitchEnabled/yawEnabled`。因此 `margAttitude500Hz` 被控制函数读取，
只能证明它进入控制函数的输入视野；某个轴是否真正执行控制逻辑，还要看该轴使能标志。

### 4.3 三轴数组下标

`sensors_t` 的字段大多是长度为 3 的数组。项目通过宏表示轴：

```c
XAXIS / YAXIS / ZAXIS
ROLL / PITCH / YAW
```

这些宏让同一个数组可以表达三维向量或三轴姿态角。阅读代码时要注意上下文：`accel500Hz[XAXIS]` 是加速度 X 轴，`margAttitude500Hz[ROLL]` 是横滚姿态角。

### 4.4 频率后缀

`500Hz` 表示这些数据主要服务 500Hz 主循环。它不是 C 语言类型的一部分，而是项目命名约定，用来提醒读者这个字段属于高速控制链路。

`mag10Hz` 的命名表示磁力计低频数据容器，但当前项目主线没有发现写入或使用 `sensors.mag10Hz` 的证据。

频率后缀还隐含一个调试要求：读者观察某个字段时，应同时记录它上一次写入发生在什么时刻。
例如 `margAttitude500Hz` 在 500Hz 分支中更新，而 `Core/Src/main.c` 的串口打印在 10Hz
低频分支中读取 Roll。串口看到的是最近一次 500Hz 姿态结果，而不是 10Hz 重新计算出的姿态。

### 4.5 原始变量、中间缓存和状态字段

本项目里有几组名字相近的数据，很容易混在一起：

| 数据层级 | 代表变量 | 数据类型 | 当前作用 |
| --- | --- | --- | --- |
| 驱动原始层 | `rawAccel/rawGyro` | `Int16_Converter_t[3]` | I2C 读取、字节拼接和方向矩阵后的整数计数 |
| 历史/中间缓存层 | `accelData500Hz/gyroData500Hz` | `int16_t[3]` | 有全局定义，但当前 SysTick 中相关写入代码被注释 |
| 主线状态层 | `sensors.accel500Hz/gyro500Hz` | `float[3]` | 当前 500Hz 分支缩放后的物理量输入 |
| 姿态状态层 | `sensors.margAttitude500Hz` | `float[3]` | 当前 AHRS 输出并被控制函数读取的姿态角 |

因此，`rawAccel[XAXIS].value` 不等于 `sensors.accel500Hz[XAXIS]`。
前者仍是整数计数层，后者已经经过偏置扣除和比例缩放。`accelData500Hz`
和 `gyroData500Hz` 虽然名字带 500Hz，但当前证据显示它们主要留在被注释的
SysTick 旧路径中，不能当作当前主线控制输入。

这一层区分对调试很重要：如果调试器里看到 `rawAccel` 为 0，应先检查 I2C
读取失败、方向矩阵和原始层处理；如果看到 `sensors.accel500Hz` 异常，应继续检查
偏置、比例因子和 500Hz 分支写入。两者属于同一链路的不同阶段，不能直接互相替代。

## 5. 工作原理

当前 500Hz 主线中，`sensors_t` 的数据流可以分成四段。

第一段，`Core/Src/main.c` 从 MPU6050 读取并缩放数据。第21章已经说明，主循环中写入：

```c
sensors.accel500Hz[XAXIS/YAXIS/ZAXIS]
sensors.gyro500Hz[ROLL/PITCH/YAW]
```

加速度写入时扣除温度偏置并乘以 `ACCEL_SCALE_FACTOR`。陀螺仪写入时扣除运行偏置和温度偏置，并乘以 `GYRO_SCALE_FACTOR`。

第二段，`Core/Src/main.c` 把这些字段作为参数传给 `MargAHRSupdate()`：

```text
sensors.gyro500Hz -> gx/gy/gz
sensors.accel500Hz -> ax/ay/az
```

第三段，`Drivers/SRC/Src/MargAHRS.c` 在姿态更新末尾写入：

```c
sensors.margAttitude500Hz[ROLL]
sensors.margAttitude500Hz[PITCH]
sensors.margAttitude500Hz[YAW]
```

第四段，`Drivers/SRC/Src/computeMotorCommands.c` 读取 `sensors.margAttitude500Hz`，作为 Roll/Pitch/Yaw 控制输入。

但当前主线还有一个启动门控。开机后 `Core/Src/main.c` 先把三个控制轴使能全部设为 `false`。
500Hz 分支仍会读取传感器、更新 `sensors`、运行 `MargAHRSupdate()`，并调用
`computeMotorCommands(dt500Hz)`。当 `startup_delay_counter` 增加到 1000 时，代码清零 PID
积分和 D 项状态，再把 Pitch 和 Yaw 使能打开，Roll 仍保持关闭。按 500Hz 估算，
1000 帧约为：

```text
1000 / 500 = 2s
```

所以，第25章的数据流结论应分成两层：

- 数据层：`sensors.margAttitude500Hz` 在启动延时期间也会被 AHRS 更新。
- 控制层：各轴控制是否执行，还受 `eepromConfig.*Enabled` 门控影响。

因此，当前主线不是所有字段同时工作，而是：

```text
accel500Hz + gyro500Hz
-> MargAHRSupdate()
-> margAttitude500Hz
-> computeMotorCommands()
```

按生产者和消费者整理如下：

| 字段 | 当前写入者 | 当前读取者 | 单位/含义 | 主线状态 |
| --- | --- | --- | --- | --- |
| `accel500Hz` | `Core/Src/main.c` 500Hz 分支 | `MargAHRSupdate()` | `m/s^2` 量级加速度 | 当前主线 |
| `gyro500Hz` | `Core/Src/main.c` 500Hz 分支 | `MargAHRSupdate()` | `rad/s` 角速度 | 当前主线 |
| `margAttitude500Hz` | `MargAHRSupdate()` | `computeMotorCommands()`、10Hz 打印 | `rad` 姿态角 | 当前主线 |
| `evvgcCFAttitude500Hz` | `evvgcCF.c` 保留路径 | 当前控制无读取证据 | `rad` 姿态角 | 备用方案 |
| `mag10Hz` | 无主线写入证据 | 无主线读取证据 | 磁力计容器 | 保留字段 |

这张表的价值在于防止“按名字猜数据流”。例如 `evvgcCFAttitude500Hz` 名字也带
`500Hz`，但当前没有主线调用 `getOrientation()`；`mag10Hz` 名字暗示低频磁力计，
但当前没有写入或读入 `MargAHRSupdate()` 的证据。

严格说，表中的“写入者”还要区分两种情况：

- 当前主线写入者：在现有 500Hz 控制路径中确实被调用。
- 潜在或保留写入者：源码中存在写入语句，但当前没有调用证据。

`Core/Src/main.c` 和 `MargAHRSupdate()` 属于当前主线写入者。`evvgcCF.c`
中的互补滤波写入更接近保留写入者，因为当前没有发现 `initOrientation()`
或 `getOrientation()` 被主线调用。做代码审查时，不能只用 `rg sensors.`
的搜索结果直接推导“当前运行一定会写入”。

同样，不能只用 `rg sensors.margAttitude500Hz` 的读取结果推导“该轴电机已经输出”。
`computeMotorCommands.c` 内部对 Roll、Pitch、Yaw 分别有 `eepromConfig.*Enabled`
判断。字段读写表解决的是数据所有权问题；控制使能表解决的是执行门控问题。

## 6. STM32实现机制

`sensors_t` 本身不是 STM32 外设，而是 C 语言结构体。它在 STM32 工程中的意义体现在内存和模块共享上。

项目在 `Core/Src/main.c` 中定义：

```c
sensors_t sensors;
```

这意味着 `sensors` 是一个全局变量。`Drivers/BGC/bgc32.h` 中的：

```c
extern sensors_t sensors;
```

让其他 `.c` 文件可以通过头文件声明访问同一个全局实例。

从内存角度看，`sensors_t` 包含 5 个 `float[3]` 数组，共 15 个 `float`。

在当前工具链中，一个 `float` 通常占 4 字节，因此结构体数据本体约 60 字节，不含编译器可能引入的对齐细节。

对于 STM32F103RCTx 来说，这个状态容器本身不是内存压力来源。

真正需要注意的是并发和时序：`sensors` 被 500Hz 主循环写入和读取。

当前项目主要在主循环中顺序执行写入、姿态更新和控制计算。

本章没有发现中断直接改写 `sensors` 字段的证据。

第26章会继续检查 500Hz 调度边界。

结合 `Core/Src/stm32f1xx_it.c` 可以看到，当前 SysTick 中断主要设置 `frame_500Hz = true`。
中断里曾经保留过读取 MPU6050 并保存原始数据的注释代码，但当前实际路径没有在中断中
写入 `sensors`。这让本章的数据一致性相对简单：一次 500Hz 分支中，代码先写
`accel500Hz/gyro500Hz`，再更新 `margAttitude500Hz`，最后调用 `computeMotorCommands()`。

如果未来把 I2C 读取、姿态解算或磁力计写入搬到中断、DMA 回调或另一个任务中，
`sensors_t` 就不再只是普通全局状态表，还会涉及并发访问。那时需要重新考虑：

- 是否需要双缓冲，避免控制函数读到半更新状态。
- 是否需要短临界区，保护多字段一致性。
- 是否需要 `volatile` 或更明确的数据发布机制。
- 是否需要帧号或时间戳，判断字段是否属于同一采样周期。

当前仓库没有实现这些机制，所以教材只能把它们列为扩展设计边界，不能反向声称当前已经具备。

## 7. 项目中的应用

本章涉及的项目证据文件包括：

- `Drivers/BGC/bgc32.h`
- `Core/Src/main.c`
- `Drivers/SRC/Src/MargAHRS.c`
- `Drivers/SRC/Src/evvgcCF.c`

`bgc32.h` 定义 `sensors_t` 类型和 `extern sensors_t sensors` 声明。

`Core/Src/main.c` 定义全局变量 `sensors`，并在 500Hz 主循环中写入 `accel500Hz` 和 `gyro500Hz`。

`computeMotorCommands.c` 读取 `margAttitude500Hz`，用于后续控制输入。

## 8. 代码分析

### 8.1 accel500Hz

`accel500Hz[3]` 保存缩放后的三轴加速度。当前主线写入位置在 `Core/Src/main.c`：

```c
sensors.accel500Hz[XAXIS]
sensors.accel500Hz[YAXIS]
sensors.accel500Hz[ZAXIS]
```

它们随后传入 `MargAHRSupdate()`。当前主线没有发现 `computeMotorCommands()` 直接读取 `accel500Hz` 的证据。

需要注意备用路径的差异。`evvgcCF.c` 的 `initOrientation()` 中也写入了 `sensors.accel500Hz`，
但 Z 轴使用了负号：

```c
sensors.accel500Hz[ZAXIS] = -((float)rawAccel[ZAXIS].value - accelTCBias[ZAXIS]) * ACCEL_SCALE_FACTOR;
```

而当前 500Hz 主线中 Z 轴没有再次取反。由于 `initOrientation()` 当前没有进入主线，
这不会影响当前运行结论；但如果未来恢复备用互补滤波初始化，它可能临时写入一套不同符号约定的
`accel500Hz`。这就是公共状态容器的风险：同一个字段被不同路径写入时，必须先统一单位和符号。

### 8.2 gyro500Hz

`gyro500Hz[3]` 保存缩放后的三轴角速度。当前主线写入位置同样在 `Core/Src/main.c`：

```c
sensors.gyro500Hz[ROLL]
sensors.gyro500Hz[PITCH]
sensors.gyro500Hz[YAW]
```

它们作为 `gx/gy/gz` 传入 `MargAHRSupdate()`。第24章提到，如果未来把这些值传给 `getOrientation()`，需要重新确认单位，避免重复角度转弧度。

### 8.3 margAttitude500Hz

`margAttitude500Hz[3]` 是当前主线姿态输出。写入者是 `MargAHRSupdate()`：

```c
sensors.margAttitude500Hz[ROLL]
sensors.margAttitude500Hz[PITCH]
sensors.margAttitude500Hz[YAW]
```

读取者包括：

- `Core/Src/main.c` 的 10Hz 打印。
- `Drivers/SRC/Src/computeMotorCommands.c` 的 Roll/Pitch/Yaw 控制输入。

因此它是当前控制链路中最重要的姿态数组。

### 8.4 evvgcCFAttitude500Hz

`evvgcCFAttitude500Hz[3]` 是互补滤波备用方案的姿态数组。写入者在 `evvgcCF.c`：

```c
sensors.evvgcCFAttitude500Hz[PITCH]
sensors.evvgcCFAttitude500Hz[ROLL]
sensors.evvgcCFAttitude500Hz[YAW]
```

当前没有发现 `computeMotorCommands.c` 读取它，也没有发现 `Core/Src/main.c` 调用 `getOrientation()`。所以它不能代表当前控制使用的姿态。

### 8.5 mag10Hz

`mag10Hz[3]` 是磁力计数据容器。当前项目中，`Core/Src/main.c` 调用 `MargAHRSupdate()` 时传入的磁力计参数为 `0, 0, 0, false`，全工程也没有发现 `sensors.mag10Hz` 的写入或读取证据。

因此 `mag10Hz` 只能写成保留字段。不能把它写成当前航向修正输入。

### 8.6 结构体注释与当前事实

`bgc32.h` 中 `margAttitude500Hz` 旁边的注释写作“卡尔曼姿态角”。但第23章已经确认当前写入者是 `MargAHRSupdate()`，不是卡尔曼滤波实现。

这里应按代码调用关系解释为“当前主线 AHRS 姿态角”。注释与实际算法命名不一致，后续若整理源码注释可再修正；教材当前只记录边界，不改源码。

### 8.7 数据新鲜度与帧内顺序

当前 500Hz 分支中的顺序是：

```text
MPU6050_Read_And_Process()
-> 写 accel500Hz / gyro500Hz
-> MargAHRSupdate()
-> 写 margAttitude500Hz
-> computeMotorCommands(dt500Hz)
```

因此 `computeMotorCommands()` 读取到的姿态角应来自同一轮 500Hz 分支中刚刚完成的
`MargAHRSupdate()`。如果某一帧 MPU6050 读取失败，第19章已经指出部分原始数据会被清零；
这些清零值随后仍可能进入 `sensors.accel500Hz/gyro500Hz` 的缩放链路。也就是说，
`sensors_t` 只保存当前值，不保存“本帧是否可信”的质量标志。

这带来一个教学边界：看到 `sensors.accel500Hz` 有数值，不等于本帧传感器读取成功；
看到 `margAttitude500Hz` 有姿态角，也不等于该姿态已经通过外部基准验证。若要建立更强的
工程诊断能力，未来可以考虑为 `sensors_t` 或相邻状态结构增加时间戳、帧号或有效性标志。

### 8.8 不要把同名 500Hz 变量混为一谈

`Drivers/CustomDrivers/Src/mpu6050.c` 中定义了 `accelData500Hz[3]`
和 `gyroData500Hz[3]`，`mpu6050.h` 中也有对应 `extern` 声明。
但当前 `Core/Src/stm32f1xx_it.c` 中把它们写入的代码位于注释块内，实际中断路径只设置
`frame_500Hz = true`。所以当前教材不能把它们写成有效的 500Hz 数据交接点。

更准确的数据层次应写成：

```text
rawAccel/rawGyro
-> main.c 中偏置扣除和比例缩放
-> sensors.accel500Hz / sensors.gyro500Hz
-> MargAHRSupdate()
-> sensors.margAttitude500Hz
-> computeMotorCommands()
```

如果未来恢复 SysTick 内读取或重新启用 `accelData500Hz/gyroData500Hz`，
需要重新审查它们与 `sensors_t` 的所有权关系。否则可能出现两套 500Hz
缓存并存：一套是中断里保存的整数快照，另一套是主循环里缩放后的浮点状态。
这会让调试表面上看起来“都有 500Hz”，但实际数据阶段、单位和新鲜度完全不同。

### 8.9 姿态字段与控制使能门控

`Core/Src/main.c` 在初始化后显式关闭三个控制轴：

```c
eepromConfig.pitchEnabled = false;
eepromConfig.rollEnabled = false;
eepromConfig.yawEnabled = false;
```

随后 500Hz 分支仍然按顺序执行传感器读取、`sensors` 写入、`MargAHRSupdate()` 和
`computeMotorCommands(dt500Hz)`。启动延时计数到 1000 帧时，代码执行：

```c
zeroPIDintegralError();
zeroPIDstates();
eepromConfig.pitchEnabled = true;
eepromConfig.rollEnabled = false;
eepromConfig.yawEnabled = true;
```

这说明 `sensors.margAttitude500Hz` 的更新和控制轴使能不是同一个概念。
启动前约 2 秒内，姿态字段可以持续刷新，但各轴控制分支是否执行要看使能标志。
启动计数到达 1000 的同一轮 500Hz 分支中，PID 状态先被清零，Pitch/Yaw 使能被打开，
随后才调用 `computeMotorCommands(dt500Hz)`。

因此调试时要分清三类现象：

| 现象 | 能证明什么 | 不能证明什么 |
| --- | --- | --- |
| `margAttitude500Hz` 在变化 | AHRS 正在写姿态字段 | 电机控制已经使能 |
| `computeMotorCommands()` 被调用 | 控制函数进入本帧流程 | 每个轴分支都执行 |
| `pitchEnabled/yawEnabled` 为真 | 对应轴控制分支允许执行 | 姿态方向和 PID 参数一定正确 |

这也是本章和第26章、第27章之间的边界：本章解释状态字段在哪里；
第26章解释它们何时更新；第27章再解释控制算法如何使用这些状态。

### 本节证据边界

本节只根据当前仓库说明文件、函数、宏、变量和调用关系。运行时频率、外部硬件表现、主机侧现象、传感器方向、电机响应或真实控制效果仍需调试记录、日志或仓库外实测证据；缺少证据时保持【待验证】。

## 9. 调试方法

第一步，观察写入顺序。

- 在 `Core/Src/main.c` 中 `sensors.accel500Hz` 和 `sensors.gyro500Hz` 写入处设置断点。
- 确认这些值在 `MargAHRSupdate()` 调用前已经更新。

第二步，观察姿态输出。

- 在 `MargAHRSupdate()` 末尾观察 `sensors.margAttitude500Hz`。
- 确认它在 `computeMotorCommands()` 调用前已经更新。

第三步，观察控制读取。

- 在 `computeMotorCommands.c` 中观察 `sensors.margAttitude500Hz[ROLL/PITCH/YAW]`。
- 确认当前控制链路读取的是主线姿态数组。
- 同时记录 `eepromConfig.rollEnabled/pitchEnabled/yawEnabled`。
- 不要把控制函数被调用直接等同于三个轴都已经输出。

第四步，确认保留字段。

- 搜索 `sensors.evvgcCFAttitude500Hz` 和 `sensors.mag10Hz`。
- 如果只发现定义或保留写入，没有当前控制读取，就不要把它们作为主线数据解释。

第五步，检查字段单位。

- `accel500Hz` 来自 `ACCEL_SCALE_FACTOR`。
- `gyro500Hz` 来自 `GYRO_SCALE_FACTOR`。
- `margAttitude500Hz` 来自 `atan2f()` 和 `asinf()`，单位是弧度。

调试时不要把这些数组简单都称为“传感器值”；它们处在不同处理阶段，单位也不同。

第六步，区分原始层和状态层。

- 观察 `rawAccel/rawGyro` 时，记录它们仍是方向矩阵处理后的整数计数。
- 观察 `sensors.accel500Hz/gyro500Hz` 时，记录偏置和比例因子已经参与计算。
- 搜索 `accelData500Hz/gyroData500Hz` 时，确认当前写入是否仍处于注释代码中。

如果一个问题只出现在 `sensors` 层，不要马上回到 I2C 字节解析；也要检查缩放、
偏置和主循环写入顺序。如果一个问题在 `rawAccel/rawGyro` 层已经出现，
再继续看 `sensors` 只会看到被放大的后果。

第七步，检查帧内新鲜度。

- 在 500Hz 分支开始处记录 `currentTime` 和 `deltaTime500Hz`。
- 在写入 `accel500Hz/gyro500Hz` 后记录一组输入。
- 在 `MargAHRSupdate()` 后记录 `margAttitude500Hz`。
- 在 `computeMotorCommands()` 入口记录控制侧实际读取的姿态。

如果四处记录不能对应同一帧，就不要把它们放进同一条因果链。串口 10Hz 打印也要标明它读取的是最近一次 500Hz 更新后的值。

第八步，检查启动门控。

- 复位后记录 `startup_delay_counter`。
- 在计数小于 1000 时，确认 `sensors.margAttitude500Hz` 是否仍在更新。
- 在计数等于 1000 的那一帧，记录 `zeroPIDintegralError()`、`zeroPIDstates()` 和三轴使能标志。
- 进入 `computeMotorCommands()` 后，确认哪些轴分支因使能标志执行。

调试记录建议：

- 记录每个 `sensors` 字段的写入位置、读取位置、单位和所属处理阶段。
- 对主线字段，应按“原始读取、物理量缩放、姿态输出、控制读取”顺序记录数据流。
- 对备用字段和磁力计字段，应记录定义、搜索结果和当前是否进入控制读取路径。
- 对跨模块共享字段，应记录是否存在多个写入者，尤其是备用路径是否采用不同单位或符号。
- 对控制侧读取，应同时记录轴使能标志，避免把“字段进入函数”误写成“轴已闭环输出”。
- 没有调用证据时，只能写“字段保留或未进入主线”，不能写成当前运行数据来源。

调试边界：

当前仓库能证明 `sensors` 字段的数据流位置和读写关系。字段数值是否物理合理、方向是否正确、噪声是否可接受，需要串口日志、断点记录或仓库外运动测试；缺少证据时保持【待验证】。

## 10. 常见问题

问题一：`sensors_t` 是不是只保存传感器原始数据。

不是。它保存的是运行状态，包括缩放后的加速度、缩放后的角速度、当前主线姿态角和备用姿态角。原始 MPU6050 整数值仍在 `rawAccel/rawGyro` 等变量中。

问题二：当前 PID 使用哪个姿态数组。

当前控制链路读取 `sensors.margAttitude500Hz`。没有发现 `computeMotorCommands.c` 读取 `sensors.evvgcCFAttitude500Hz` 的证据。

问题三：`mag10Hz` 是否参与当前 Yaw 修正。

当前没有证据。`MargAHRSupdate()` 调用时磁力计参数为 `0, 0, 0, false`，`sensors.mag10Hz` 没有发现写入或读取主线。

问题四：为什么 `sensors_t` 和 `eepromConfig_t` 都是全局结构。

它们解决的问题不同。`eepromConfig_t` 保存配置和控制参数，`sensors_t` 保存运行数据和姿态结果。一个偏“参数”，一个偏“状态”。

问题五：为什么不把所有数据都做成局部变量传来传去。

在嵌入式工程中，多个模块共享周期性状态很常见。全局结构能让数据位置稳定，便于调试器观察。但代价是必须严格管理写入顺序和字段含义，否则容易出现旧数据或误用保留字段。

问题六：`margAttitude500Hz` 注释写“卡尔曼姿态角”，教材为什么不这么讲。

因为当前源码证据显示它由 `MargAHRSupdate()` 写入，当前没有卡尔曼滤波实现写入该字段。教材以调用关系和数据流为准，源码注释不一致处作为边界记录。

问题七：字段名带 `500Hz` 或 `10Hz` 是否一定表示当前更新频率。

不能只看字段名判断。
`sensors.accel500Hz`、`sensors.gyro500Hz` 和 `sensors.margAttitude500Hz`
在当前 500Hz 主线中有写入证据。
但 `mag10Hz` 当前没有主线写入或读取证据。
因此频率名称应结合实际写入路径验证，避免把保留字段误当成当前有效数据。

问题八：`sensors_t` 是否能说明本帧数据有效。

不能。当前结构体只有数值字段，没有时间戳、帧号、读取成功标志或质量评分。
如果 I2C 读取失败、原始数据被清零、姿态输出触发 NaN 保护，`sensors_t` 仍然只保存结果值。
判断本帧是否可靠，需要结合读取返回路径、异常防护、调试日志和第32章的运行门控分析。

问题九：为什么多个模块都能写 `sensors` 会有风险。

全局结构方便共享，但也降低了字段所有权边界。
当前主线主要由 `Core/Src/main.c` 和 `MargAHRS.c` 写入关键字段。
备用 `evvgcCF.c` 也保留了写 `accel500Hz` 和 `evvgcCFAttitude500Hz` 的代码。
如果未来恢复这些路径，必须先确认它们不会用不同单位、符号或时序覆盖当前主线数据。

问题十：`accelData500Hz/gyroData500Hz` 是不是当前传感器状态入口。

当前不能这样写。它们在 `mpu6050.c` 中有全局定义，在 `mpu6050.h` 中有声明，
但本章复查到的写入点位于 `stm32f1xx_it.c` 的注释块内。当前 500Hz 主线入口是
`Core/Src/main.c` 写入的 `sensors.accel500Hz/gyro500Hz`，不是这两个整数数组。

问题十一：`margAttitude500Hz` 被 `computeMotorCommands()` 读取，是否说明电机已经闭环输出。

不能直接这样说。`computeMotorCommands()` 内部还受 `eepromConfig.rollEnabled`、
`eepromConfig.pitchEnabled` 和 `eepromConfig.yawEnabled` 控制。启动前约 1000 个 500Hz
帧内，姿态字段会更新，控制函数也会被调用，但轴分支是否执行要看使能标志。
因此要把“姿态数据进入控制函数”和“某轴控制输出生效”分开记录。
