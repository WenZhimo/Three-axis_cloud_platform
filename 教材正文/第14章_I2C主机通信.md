# 第14章 I2C主机通信

## 1. 本章目标

- 理解项目为什么需要 I2C 主机通信来访问 MPU6050。
- 看懂 `Core/Src/i2c.c` 中 I2C2 的 100kHz、7位地址和 PB10/PB11 配置。
- 能追踪 `main.c` 中 I2C2 初始化、MPU6050 初始化和 500Hz 读取之间的顺序关系。
- 能理解 `mpu6050.c` 中 `HAL_I2C_Mem_Write()` 与 `HAL_I2C_Mem_Read()` 的项目作用。
- 明确本章只建立 I2C 通信通道，不提前展开 MPU6050 寄存器配置、连续数据解析和姿态算法。

## 2. 前置知识

- 系统时钟树
- HAL MSP初始化机制
- GPIO输出与复用

第06章已经说明外设通信依赖系统时钟，第07章已经说明 MSP 初始化和 GPIO 复用，第13章已经完成定时器边界说明。本章在这些基础上进入传感器通信路径：I2C2 如何成为 STM32 与 MPU6050 之间的寄存器访问通道。

本章不展开 MPU6050 的完整寄存器含义，不展开 14 字节连续数据解析，不展开安装方向矩阵、物理量缩放和姿态解算。这些内容安排在后续传感器和算法章节。

## 3. 问题背景

三轴云台需要实时知道姿态变化。姿态数据来自 MPU6050，但 STM32 不能直接读取一个“姿态角”变量。它必须先通过通信总线访问传感器寄存器，再把原始字节送入后续的数据处理链路。

当前项目选择 I2C2 作为 MPU6050 的通信通道。证据链可以分成三段：

1. `.ioc` 将 PB10 配置为 I2C2_SCL，将 PB11 配置为 I2C2_SDA。
2. `Core/Src/i2c.c` 初始化 `hi2c2`，配置 100kHz、7位地址模式和 I2C2 MSP。
3. `Drivers/CustomDrivers/Src/mpu6050.c` 通过 `HAL_I2C_Mem_Write()` 和 `HAL_I2C_Mem_Read()` 访问 MPU6050 寄存器。

如果第14章不先讲清这条通信路径，后续 MPU6050 寄存器配置、连续读取和物理量缩放都会缺少入口。

## 4. 核心概念

- I2C主机通信：由 STM32 作为主机发起时钟和读写事务，外设作为从设备响应。
- SCL：I2C 时钟线，本项目使用 PB10。
- SDA：I2C 数据线，本项目使用 PB11。
- 开漏输出：I2C 常用的引脚输出方式，本项目 PB10/PB11 配置为 `GPIO_MODE_AF_OD`。
- 7位地址：I2C 设备的逻辑地址长度。本项目 `hi2c2.Init.AddressingMode` 为 `I2C_ADDRESSINGMODE_7BIT`。
- HAL 设备地址参数：STM32 HAL 的 I2C 读写函数使用 `DevAddress` 参数；本项目宏 `MPU6050_I2C_ADDR` 定义为 `(0x68 << 1)`。
- 寄存器地址：传感器内部的功能位置，例如 `REG_WHO_AM_I`，本章只把它作为通信目标，不展开寄存器语义。
- Mem Read/Write：HAL 提供的“向某个设备的某个寄存器读写数据”的接口，本项目用它封装 MPU6050 低层访问。

这些概念服务于正式知识点 `I2C主机通信`，不新增结构外知识点。

## 5. 工作原理

I2C 通信的基本模型是“主机指定设备地址，再指定寄存器地址，然后读或写数据”。

在本项目中，STM32 是主机，MPU6050 是从设备。`mpu6050.h` 中定义 `MPU6050_I2C_ADDR` 为 `(0x68 << 1)`。这表示项目按 MPU6050 逻辑地址 0x68 构造 HAL 所需的设备地址参数。`mpu6050.c` 中的底层函数再把这个地址传给 HAL：

- `MPU_Write_Reg(reg_addr, data)` 调用 `HAL_I2C_Mem_Write()`，向指定寄存器写 1 字节。
- `MPU_Read_Reg(reg_addr, data, len)` 调用 `HAL_I2C_Mem_Read()`，从指定寄存器连续读 `len` 字节。

从通信层看，初始化阶段和读取阶段使用同一条 I2C2 通道：

- 初始化阶段先读取 `REG_WHO_AM_I`，确认返回值是否为 0x68。
- 初始化阶段还会写入若干配置寄存器。具体配置含义留到 MPU6050 寄存器章节。
- 运行阶段从 `REG_ACCEL_XOUT_H` 开始读取 `MPU_DATA_LENGTH` 个字节。具体字节解析留到连续读取章节。

I2C2 本身的工作前提来自 `i2c.c`：100kHz 时钟、7位地址模式、PB10/PB11 复用开漏、I2C2 外设时钟使能。缺少任一环节，HAL 读写函数都可能失败。

## 6. STM32实现机制

`Core/Src/i2c.c` 中的 `MX_I2C2_Init()` 是 I2C2 初始化入口。它设置：

- `hi2c2.Instance = I2C2`
- `ClockSpeed = 100000`
- `DutyCycle = I2C_DUTYCYCLE_2`
- `OwnAddress1 = 0`
- `AddressingMode = I2C_ADDRESSINGMODE_7BIT`
- `DualAddressMode = I2C_DUALADDRESS_DISABLE`
- `GeneralCallMode = I2C_GENERALCALL_DISABLE`
- `NoStretchMode = I2C_NOSTRETCH_DISABLE`

这些配置说明当前项目把 I2C2 作为标准模式主机通道使用。`OwnAddress1` 对当前主机访问 MPU6050 不是核心参数；真正用于访问传感器的是 `HAL_I2C_Mem_Read/Write()` 中传入的设备地址宏。

`HAL_I2C_MspInit()` 在 I2C2 分支中完成底层资源配置：

- 启用 GPIOB 时钟。
- 将 PB10/PB11 配置为 I2C2_SCL/I2C2_SDA。
- 设置 GPIO 模式为 `GPIO_MODE_AF_OD`。
- 设置 GPIO 速度为 High。
- 启用 I2C2 外设时钟。

`.ioc` 是这段代码的配置来源。它记录 PB10 为 I2C2_SCL、PB11 为 I2C2_SDA，也记录项目启用了 I2C2。

## 7. 项目中的应用

在 `main()` 的初始化顺序中，I2C2 位于 MPU6050 初始化之前：

1. `SystemClock_Config()` 建立系统时钟。
2. `MX_GPIO_Init()` 建立基础 GPIO。
3. 定时器初始化后，`MX_I2C2_Init()` 初始化 I2C2。
4. 用户初始化区中调用 `orientIMU()` 设置安装方向矩阵。
5. 随后调用 `MPU6050_Init(...)`，它通过 I2C2 读取设备 ID 并写配置寄存器。
6. 进入主循环后，500Hz 任务中调用 `MPU6050_Read_And_Process()`，它通过 I2C2 读取传感器数据。

从项目主线看，本章位于“传感器数据进入系统”的入口。I2C2 只负责把字节读出来；这些字节如何表示加速度、角速度、温度，如何做坐标转换和物理量缩放，属于后续章节。

## 8. 代码分析

### 1. `MX_I2C2_Init()`

入口是 `main()` 的外设初始化流程。输入是 CubeMX 生成的配置参数，输出是 `hi2c2` 句柄和 I2C2 外设被初始化。

风险点是初始化顺序。若 `MPU6050_Init()` 早于 `MX_I2C2_Init()`，`hi2c2` 尚未完成配置，HAL 读写就没有可靠前提。当前 `main.c` 中 `MX_I2C2_Init()` 先执行，满足这一前提。

### 2. `HAL_I2C_MspInit()`

该函数在 `HAL_I2C_Init(&hi2c2)` 内部被调用。它不设置通信协议参数，而是设置底层硬件资源。

对 I2C2 来说，它完成 PB10/PB11 的复用开漏配置，并启用 I2C2 时钟。这里承接第07章的 MSP/GPIO 知识：外设初始化不是只配置 `hi2c2.Init`，还必须让引脚和时钟准备好。

### 3. `MPU_Write_Reg()`

`MPU_Write_Reg()` 是 `mpu6050.c` 中的静态底层写函数。输入是寄存器地址和 1 字节数据。它调用：

- `&hi2c2`
- `MPU6050_I2C_ADDR`
- `reg_addr`
- `I2C_MEMADD_SIZE_8BIT`
- 数据指针
- 长度 1
- 超时 100

这说明项目把 MPU6050 的寄存器写操作统一封装成一个低层函数。后续初始化寄存器时，只需要给它寄存器地址和数据。

### 4. `MPU_Read_Reg()`

`MPU_Read_Reg()` 是底层读函数。输入是寄存器地址、数据缓冲区和长度。它调用 `HAL_I2C_Mem_Read()`，从 MPU6050 指定寄存器连续读取数据。

初始化时，它读取 `REG_WHO_AM_I` 的 1 字节；运行时，它从 `REG_ACCEL_XOUT_H` 开始读取 `MPU_DATA_LENGTH` 个字节。读取内容的意义留到后续传感器章节，本章只确认 I2C 读路径。

### 5. `MPU6050_Init()`

`MPU6050_Init()` 中有一个 I2C 通信健康检查：最多重试 3 次读取设备 ID。若 HAL 状态不是 `HAL_OK` 或读到的 ID 不是 0x68，函数打印错误并返回 `false`。

这段逻辑为 I2C 调试提供了项目内观察点：串口输出中的状态码、读回值和失败提示可以帮助判断通信是否已经建立。

### 6. `MPU6050_Read_And_Process()`

该函数调用 `MPU_Read_Reg(REG_ACCEL_XOUT_H, I2C2_Buffer_Rx, MPU_DATA_LENGTH)`。如果读取失败，它清空 `rawAccel` 和 `rawGyro` 后返回。

这说明 I2C 读取失败会直接影响后续姿态输入。第14章只讲“读取失败如何在通信入口体现”，不展开数据解析和矩阵旋转。

## 9. 调试方法

I2C 调试应先确认工程配置，再确认项目运行输出。

配置检查：

- `.ioc` 中 PB10 是否为 I2C2_SCL，PB11 是否为 I2C2_SDA。
- `i2c.c` 中 `ClockSpeed` 是否为 100000。
- `i2c.c` 中 `AddressingMode` 是否为 7位地址。
- `HAL_I2C_MspInit()` 是否配置 PB10/PB11 为 `GPIO_MODE_AF_OD`。
- `HAL_I2C_MspInit()` 是否启用 I2C2 外设时钟。

运行检查：

- `main.c` 中 `MX_I2C2_Init()` 是否早于 `MPU6050_Init()`。
- `mpu6050.h` 中 `MPU6050_I2C_ADDR` 是否为 `(0x68 << 1)`。
- `MPU6050_Init()` 是否能读回 `WHO_AM_I` 值 0x68。
- 失败时串口是否打印状态码、读回值和检查提示。
- 500Hz 任务中 `MPU6050_Read_And_Process()` 是否会进入读取失败保护。

当前工作树没有外部连线、上拉器件和实际示波观察证据，因此物理连接和外部波形只能标记为【待验证】。本章只确认工程内部 I2C2 配置和软件调用路径。

## 10. 常见问题

### 1. 为什么 I2C2 引脚要配置为开漏？

触发条件：读者看到 `GPIO_MODE_AF_OD`。

可能原因：I2C 总线通过开漏方式让多个设备共享 SCL/SDA 线。项目中 PB10/PB11 采用复用开漏模式，符合 I2C 通信需求。外部上拉是否存在需要硬件证据，当前标记为【待验证】。

### 2. 为什么宏是 `(0x68 << 1)`，而不是直接 0x68？

触发条件：读者看到 `MPU6050_I2C_ADDR`。

可能原因：项目头文件中用 0x68 作为 MPU6050 逻辑地址来源，再左移 1 位传给 STM32 HAL 的 I2C 设备地址参数。教材按当前代码解释，不另行推断其他地址接法。

### 3. `HAL_I2C_Mem_Read()` 和普通读有什么区别？

触发条件：读者第一次看到 Mem Read。

可能原因：本项目访问的是“设备内部寄存器”。因此不仅要给设备地址，还要给寄存器地址和寄存器地址宽度。本项目使用 `I2C_MEMADD_SIZE_8BIT`。

### 4. I2C 初始化成功是否等于 MPU6050 一定在线？

触发条件：`MX_I2C2_Init()` 没报错，但设备 ID 读取失败。

可能原因：I2C2 初始化只说明 MCU 外设配置完成，不证明外部传感器响应。项目用 `WHO_AM_I` 读取和返回值判断传感器是否可访问。

### 5. 为什么第14章不讲 MPU6050 所有寄存器？

触发条件：本章已经出现 `REG_WHO_AM_I` 和 `REG_ACCEL_XOUT_H`。

可能原因：这些寄存器在本章只作为 I2C 读写目标。寄存器配置、连续数据读取和字节解析分别属于后续章节，提前展开会破坏教学顺序。

## 11. 实践任务

1. 在 `.ioc` 中找到 PB10/PB11 的 I2C2 配置。验收依据是能说明 SCL/SDA 分别落在哪个引脚。
2. 在 `i2c.c` 中找出 `MX_I2C2_Init()` 的 100kHz、7位地址和 NoStretch 配置。验收依据是能说明当前 I2C2 的基本通信参数。
3. 在 `HAL_I2C_MspInit()` 中追踪 GPIOB、PB10/PB11 和 I2C2 时钟使能。验收依据是能解释为什么协议参数之外还需要 MSP。
4. 在 `mpu6050.c` 中找出 `MPU_Write_Reg()` 和 `MPU_Read_Reg()`。验收依据是能说明它们分别调用哪个 HAL I2C 函数。
5. 在 `main.c` 中追踪 `MX_I2C2_Init()`、`MPU6050_Init()` 和 `MPU6050_Read_And_Process()` 的先后关系。验收依据是能画出 I2C 通信进入传感器数据流的顺序。

## 12. 思考题

1. 为什么 I2C 外设初始化和 MPU6050 在线检测是两个不同问题？
2. 如果 `WHO_AM_I` 读回值不是 0x68，应该优先检查哪些项目内证据？
3. 为什么本项目用两个静态函数封装 MPU6050 寄存器读写，而不是在所有位置直接调用 HAL？
4. 如果 `MPU6050_Read_And_Process()` 读取失败并清空原始数据，后续姿态链路可能受到什么影响？
5. 为什么第14章只讲通信通道，而把寄存器配置和数据解析留到后续章节？

## 13. 本章总结

本章建立了三轴云台项目中 I2C2 主机通信的证据链。

已经确认的结论是：

- `.ioc` 将 PB10/PB11 配置为 I2C2_SCL/I2C2_SDA。
- `i2c.c` 将 I2C2 配置为 100kHz、7位地址模式。
- `HAL_I2C_MspInit()` 启用 GPIOB 和 I2C2 时钟，并将 PB10/PB11 配置为复用开漏。
- `mpu6050.h` 定义 `MPU6050_I2C_ADDR` 为 `(0x68 << 1)`。
- `mpu6050.c` 用 `HAL_I2C_Mem_Write()` 和 `HAL_I2C_Mem_Read()` 封装 MPU6050 寄存器访问。
- `main.c` 先初始化 I2C2，再初始化 MPU6050，并在 500Hz 任务中读取传感器数据。

下一章可以进入 USB FS 设备与中间件。与此同时，本章也为后续 MPU6050 寄存器初始化和连续读取章节建立了通信前置。
