# 第14章 I2C主机通信

> 导航：上一章：[第13章_TIM8与TIM6项目配置](第13章_TIM8与TIM6项目配置.md) ｜ 下一章：[第15章_USB FS设备与中间件](第15章_USB FS设备与中间件.md)

## 1. 本章目标

- 理解项目为什么需要 I2C 主机通信来访问 MPU6050。
- 看懂 `Core/Src/i2c.c` 中 I2C2 的 100kHz、7位地址和 PB10/PB11 配置。
- 能追踪 `main.c` 中 I2C2 初始化、MPU6050 初始化和 500Hz 读取之间的顺序关系。
- 能理解 `mpu6050.c` 中 `HAL_I2C_Mem_Write()` 与 `HAL_I2C_Mem_Read()` 的项目作用。
- 明确本章只建立 I2C 通信通道，不提前展开 MPU6050 寄存器配置、连续数据解析和姿态算法。

## 2. 前置知识

- 系统时钟树
- HAL MSP初始化机制
- GPIO输出与复用

第06章已经说明外设通信依赖系统时钟，第07章已经说明 MSP 初始化和 GPIO 复用，第13章已经完成定时器边界说明。本章在这些基础上进入传感器通信路径：I2C2 如何成为 STM32 与 MPU6050 之间的寄存器访问通道。

本章不展开 MPU6050 的完整寄存器含义，不展开 14 字节连续数据解析，不展开安装方向矩阵、物理量缩放和姿态解算。这些内容安排在后续传感器和算法章节。

## 3. 问题背景

三轴云台需要实时知道姿态变化。姿态数据来自 MPU6050，但 STM32 不能直接读取一个“姿态角”变量。它必须先通过通信总线访问传感器寄存器，再把原始字节送入后续的数据处理链路。

当前项目选择 I2C2 作为 MPU6050 的通信通道。证据链可以分成三段：

1. `.ioc` 将 PB10 配置为 I2C2_SCL，将 PB11 配置为 I2C2_SDA。
2. `Core/Src/i2c.c` 初始化 `hi2c2`，配置 100kHz、7位地址模式和 I2C2 MSP。
3. `Drivers/CustomDrivers/Src/mpu6050.c` 通过 `HAL_I2C_Mem_Write()` 和 `HAL_I2C_Mem_Read()` 访问 MPU6050 寄存器。

如果第14章不先讲清这条通信路径，后续 MPU6050 寄存器配置、连续读取和物理量缩放都会缺少入口。

## 4. 核心概念

- I2C主机通信：由 STM32 作为主机发起时钟和读写事务，外设作为从设备响应。
- SCL：I2C 时钟线，本项目使用 PB10。
- SDA：I2C 数据线，本项目使用 PB11。
- 总线空闲：I2C 空闲时 SCL 和 SDA 应处于高电平；当前仓库没有外部波形证据，仍需【待验证】。
- 开漏输出：I2C 常用的引脚输出方式，本项目 PB10/PB11 配置为 `GPIO_MODE_AF_OD`。
- 上拉电阻：开漏输出只能主动拉低，释放后的高电平依赖上拉路径；项目源码只证明 MCU 引脚模式，
  不证明板上外部上拉、电源和连线正确。
- START：主机开始一次 I2C 事务的总线条件，STM32 HAL 内部通过设置 `I2C_CR1_START` 产生。
- STOP：主机结束一次 I2C 事务的总线条件，STM32 HAL 内部通过设置 `I2C_CR1_STOP` 产生。
- 重复起始：不释放总线的再次 START，常用于“先写寄存器地址、再读寄存器数据”的组合事务。
- 地址阶段：主机发送从设备地址和读写方向，等待从设备应答。本项目使用 7位地址模式。
- 读写位：7位地址之后还有 1 位方向位；HAL 会在发送地址时生成写方向或读方向字节。
- ACK/NACK：接收方对每个字节的应答或非应答。HAL 中的 `HAL_I2C_ERROR_AF` 常和应答失败相关。
- 7位地址：I2C 设备的逻辑地址长度。本项目 `hi2c2.Init.AddressingMode` 为 `I2C_ADDRESSINGMODE_7BIT`。
- HAL 设备地址参数：STM32 HAL 的 I2C 读写函数使用 `DevAddress` 参数；本项目宏 `MPU6050_I2C_ADDR` 定义为 `(0x68 << 1)`。
- 地址左移：MPU6050 的逻辑 7 位地址是 0x68，传给本 HAL 接口前左移为 0xD0；
  读方向字节由 HAL 再置位最低位得到。
- 时钟拉伸控制位：STM32F1 的 `I2C_CR1_NOSTRETCH` 注释为从机模式下的 Clock Stretching Disable。本项目作为主机访问 MPU6050，`NoStretchMode` 字段不能单独证明外部从设备是否真的拉伸 SCL。
- 寄存器地址：传感器内部的功能位置，例如 `REG_WHO_AM_I`，本章只把它作为通信目标，不展开寄存器语义。
- Mem Read/Write：HAL 提供的“向某个设备的某个寄存器读写数据”的接口，本项目用它封装 MPU6050 低层访问。
- 阻塞式访问：函数在当前调用上下文中等待标志位、数据收发或超时结束。本项目当前使用阻塞式 Mem Read/Write。

这些概念服务于正式知识点 `I2C主机通信`，不新增结构外知识点。

## 5. 工作原理

I2C 通信的基本模型是“主机指定设备地址，再指定寄存器地址，然后读或写数据”。

在本项目中，STM32 是主机，MPU6050 是从设备。

从电气层看，I2C 不是两根普通推挽线。
主机和从设备通常通过开漏方式把线拉低，通过上拉路径让线回到高电平。
因此 START、STOP、ACK 这些协议事件最终都依赖 SCL/SDA 电平变化。
项目源码只能证明 PB10/PB11 被配置为复用开漏，不能证明外部上拉、电源和线缆状态。

最小的寄存器写事务可以拆成：

`START -> 设备地址+写方向 -> ACK -> 寄存器地址 -> ACK -> 数据字节 -> ACK -> STOP`

最小的寄存器读事务通常可以拆成：

`START -> 设备地址+写方向 -> ACK -> 寄存器地址 -> ACK -> 重复起始 -> 设备地址+读方向 -> ACK -> 数据字节 -> NACK/STOP`

这就是为什么 `HAL_I2C_Mem_Read()` 不只是“读几个字节”，而是先完成一次寄存器地址定位，再切换到读方向取回数据。

`mpu6050.h` 中定义 `MPU6050_I2C_ADDR` 为 `(0x68 << 1)`。

这表示项目按 MPU6050 逻辑地址 0x68 构造 HAL 所需的设备地址参数。`mpu6050.c` 中的底层函数再把这个地址传给 HAL：

- `MPU_Write_Reg(reg_addr, data)` 调用 `HAL_I2C_Mem_Write()`，向指定寄存器写 1 字节。
- `MPU_Read_Reg(reg_addr, data, len)` 调用 `HAL_I2C_Mem_Read()`，从指定寄存器连续读 `len` 字节。

这里容易出现一个教学误区：0x68 没有变成另一个器件地址。
STM32F1 HAL 的函数注释要求把数据手册中的 7 位地址左移后传入接口；
随后 HAL 用 `I2C_7BIT_ADD_WRITE()` 清最低位形成写方向地址，
用 `I2C_7BIT_ADD_READ()` 置最低位形成读方向地址。
所以逻辑地址仍是 0x68，线上的地址字节通常是 0xD0 写和 0xD1 读。
不同逻辑分析仪可能显示 7 位地址 0x68，也可能显示 8 位地址字节 0xD0/0xD1。

从通信层看，初始化阶段和读取阶段使用同一条 I2C2 通道：

- 初始化阶段先读取 `REG_WHO_AM_I`，确认返回值是否为 0x68。
- 初始化阶段还会写入若干配置寄存器。具体配置含义留到 MPU6050 寄存器章节。
- 运行阶段从 `REG_ACCEL_XOUT_H` 开始读取 `MPU_DATA_LENGTH` 个字节。具体字节解析留到连续读取章节。

I2C2 本身的工作前提来自 `i2c.c`：100kHz 时钟、7位地址模式、PB10/PB11 复用开漏、I2C2 外设时钟使能。缺少任一环节，HAL 读写函数都可能失败。

按 100kHz 标准模式估算，1 个 I2C 字节通常对应 8 个数据位和 1 个 ACK/NACK 位，也就是约 9 个 SCL 周期。

一次 14 字节寄存器连续读至少包含：

- 设备地址写阶段：9 个 SCL 周期。
- 8 位寄存器地址阶段：9 个 SCL 周期。
- 设备地址读阶段：9 个 SCL 周期。
- 14 个数据字节及应答阶段：`14 * 9 = 126` 个 SCL 周期。

因此理想下限约为：

`(9 + 9 + 9 + 126) / 100000 = 0.00153s`

也就是约 1.53ms。这个估算不包含 HAL 软件等待、总线上升沿时间、器件内部响应、可能的时钟拉伸和调试输出开销，所以只能作为通信负载下限，不是实测耗时。

与 500Hz 控制帧的 2ms 周期相比，1.53ms 的理想下限已经约占 76.5%。
如果未来改成 400kHz Fast-mode，纯总线位传输下限可以粗略降到约 0.38ms。
但这需要传感器规格、总线电容、上升沿、上拉电阻和实测波形共同支持；
当前项目证据仍是 100kHz 标准模式。

还要注意 `mpu6050.c` 对读写函数传入的超时值是 100ms。
这不是一次正常读取会花 100ms，而是异常等待的上限参数。
如果总线持续忙、没有 ACK 或标志位不变化，阻塞等待可能远超 2ms 控制帧。
当前仓库没有运行时超时日志，因此只能标记为风险点和调试重点。

把这个风险量化后更直观：`100ms / 2ms = 50`，一次运行期 I2C 读超时的最坏等待上限，相当于 50 个 500Hz 控制帧。初始化阶段的 `WHO_AM_I` 检查最多重试 3 次，每次读寄存器的 HAL 超时参数是 100ms，失败后还会 `HAL_Delay(50)`；若每次都等到超时，单是这段重试等待就可能达到 `3 * (100ms + 50ms) = 450ms`，还未计入串口打印时间。这个计算只来自当前代码参数，不等于实测耗时。

## 6. STM32实现机制

`Core/Src/i2c.c` 中的 `MX_I2C2_Init()` 是 I2C2 初始化入口。它设置：

- `hi2c2.Instance = I2C2`
- `ClockSpeed = 100000`
- `DutyCycle = I2C_DUTYCYCLE_2`
- `OwnAddress1 = 0`
- `AddressingMode = I2C_ADDRESSINGMODE_7BIT`
- `DualAddressMode = I2C_DUALADDRESS_DISABLE`
- `GeneralCallMode = I2C_GENERALCALL_DISABLE`
- `NoStretchMode = I2C_NOSTRETCH_DISABLE`

这些配置说明当前项目把 I2C2 作为标准模式主机通道使用。`OwnAddress1` 对当前主机访问 MPU6050 不是核心参数；真正用于访问传感器的是 `HAL_I2C_Mem_Read/Write()` 中传入的设备地址宏。

`NoStretchMode = I2C_NOSTRETCH_DISABLE` 的字面含义容易误读。
这里是“禁用 NoStretch”，也就是没有设置 `I2C_CR1_NOSTRETCH`。
但 CMSIS 设备头文件对 `I2C_CR1_NOSTRETCH` 的注释是从机模式下的 Clock Stretching Disable。
本项目当前把 STM32 作为主机访问 MPU6050，因此不能把这个字段解释为“已经允许并验证了 MPU6050 时钟拉伸”。
是否真的发生从设备拉伸 SCL，需要波形或逻辑分析仪证据；当前保持【待验证】。

这组初始化参数还会被 HAL 转换为 I2C 外设寄存器值。
`HAL_I2C_Init()` 先通过 `HAL_RCC_GetPCLK1Freq()` 取得 PCLK1，
再设置 `CR2.FREQ`、`TRISE` 和 `CCR`。
项目 `.ioc` 记录 `RCC.APB1Freq_Value=36000000`，因此 I2C2 的 PCLK1 依据是 36MHz。

对当前 100kHz 标准模式，可以得到：

- `freqrange = 36MHz / 1MHz = 36`，写入 `CR2.FREQ`。
- `TRISE = freqrange + 1 = 37`，对应 HAL 宏 `I2C_RISE_TIME()` 的标准模式分支。
- `CCR ≈ 36MHz / (2 * 100kHz) = 180`，对应 HAL 宏 `I2C_SPEED_STANDARD()`。

这说明 `ClockSpeed = 100000` 不是只停留在结构体字段里，
而是经由 PCLK1、CCR 和 TRISE 落到 STM32F1 I2C 外设时序寄存器。
当前 `DutyCycle = I2C_DUTYCYCLE_2` 在结构体中存在，但 HAL 的 `I2C_SPEED()` 宏在 `ClockSpeed <= 100000` 时走标准模式分支。
标准模式 CCR 计算只使用 PCLK1、目标速度和系数 2。也就是说，`DutyCycle` 真正影响的是 Fast-mode 分支，不能把当前 100kHz 标准模式的时序解释成由 `DutyCycle` 决定。

`HAL_I2C_MspInit()` 在 I2C2 分支中完成底层资源配置：

- 启用 GPIOB 时钟。
- 将 PB10/PB11 配置为 I2C2_SCL/I2C2_SDA。
- 设置 GPIO 模式为 `GPIO_MODE_AF_OD`。
- 设置 GPIO 速度为 High。
- 启用 I2C2 外设时钟。

`.ioc` 是这段代码的配置来源。它记录 PB10 为 I2C2_SCL、PB11 为 I2C2_SDA，也记录项目启用了 I2C2。

HAL 源码把 `HAL_I2C_Mem_Write()` 和 `HAL_I2C_Mem_Read()` 归入 Blocking mode。
也就是说，当前调用线程会在函数内部等待 `BUSY`、`SB`、`ADDR`、`TXE`、`RXNE`、`BTF` 等状态变化。
等待结束条件是传输完成、出错或超时。

对寄存器写，HAL 内部会请求 memory write：产生 START，发送写方向设备地址，等待地址应答，再发送寄存器地址和数据，最后产生 STOP。

对寄存器读，HAL 内部会请求 memory read：先以写方向发送寄存器地址，再产生重复起始，切到读方向接收数据。这个过程对应本章第5节的“先定位寄存器，再读取数据”。

本项目当前没有发现 `HAL_I2C_Mem_Read_IT()`、`HAL_I2C_Mem_Read_DMA()`、`I2C2_EV_IRQHandler()` 或 I2C DMA 业务路径。

更准确地说，启动文件中存在弱定义的 `I2C2_EV_IRQHandler` 和 `I2C2_ER_IRQHandler`，
但 `Core/Src` 中没有用户实现，也没有 I2C2 NVIC 业务初始化证据。
因此第14章只能把 I2C2 写成阻塞式主机寄存器访问通道，
不能写成中断或 DMA 驱动的传感器采集链路。

## 7. 项目中的应用

在 `main()` 的初始化顺序中，I2C2 位于 MPU6050 初始化之前：

1. `SystemClock_Config()` 建立系统时钟。
2. `MX_GPIO_Init()` 建立基础 GPIO。
3. 定时器初始化后，`MX_I2C2_Init()` 初始化 I2C2。
4. 用户初始化区中调用 `orientIMU()` 设置安装方向矩阵。
5. 随后调用 `MPU6050_Init(...)`，它通过 I2C2 读取设备 ID 并写配置寄存器。
6. 进入主循环后，500Hz 实时控制循环中调用 `MPU6050_Read_And_Process()`，它通过 I2C2 读取传感器数据。

从项目主线看，本章位于“传感器数据进入系统”的入口。I2C2 只负责把字节读出来；这些字节如何表示加速度、角速度、温度，如何做坐标转换和物理量缩放，属于后续章节。

项目里还有一个很有教学价值的边界：`stm32f1xx_it.c` 的 SysTick 用户区曾注释掉在中断内读取 MPU6050 的做法。
代码注释说明读 I2C 会耗时，且在中断里读又在 main 中读会造成总线冲突和读数为 0。
当前实现采用“中断只置 `frame_500Hz` 标志，主循环再读 I2C”的方式。

这不是简单的代码风格选择，而是实时性权衡：I2C 阻塞读会占用当前执行上下文；把它放在中断服务函数里，会延长中断占用时间，也可能和主循环访问同一 `hi2c2` 句柄产生冲突。当前项目选择在主循环中执行阻塞读，降低中断路径复杂度。

为什么使用 I2C 而不是 SPI，本章只能依据当前仓库说明。
项目已经配置 I2C2、PB10/PB11 和 `HAL_I2C_Mem_Read/Write()`，未发现 SPI 外设配置或 MPU6050 SPI 访问路径。
若未来改 SPI，需要新增硬件引脚、片选管理、驱动封装和章节证据，不能把它当成当前项目方案。

从工程权衡看，I2C 的优势是两线制、寄存器访问模型清楚、接线资源少；
代价是带宽和时序裕量有限，且开漏上升沿受电阻和总线电容影响。
SPI 通常有更高带宽和更确定的边沿时序，但需要片选和更多引脚。
当前项目选择 I2C 的可证明理由是代码和 `.ioc` 已经完整围绕 I2C2 建立，
而不是因为仓库已经证明 I2C 在 500Hz 闭环中有充分裕量。

## 8. 代码分析

### 1. `MX_I2C2_Init()`

入口是 `main()` 的外设初始化流程。输入是 CubeMX 生成的配置参数，输出是 `hi2c2` 句柄和 I2C2 外设被初始化。

风险点是初始化顺序。若 `MPU6050_Init()` 早于 `MX_I2C2_Init()`，`hi2c2` 尚未完成配置，HAL 读写就没有可靠前提。当前 `main.c` 中 `MX_I2C2_Init()` 先执行，满足这一前提。

### 2. `HAL_I2C_MspInit()`

该函数在 `HAL_I2C_Init(&hi2c2)` 内部被调用。它不设置通信协议参数，而是设置底层硬件资源。

对 I2C2 来说，它完成 PB10/PB11 的复用开漏配置，并启用 I2C2 时钟。这里承接第07章的 MSP/GPIO 知识：外设初始化不是只配置 `hi2c2.Init`，还必须让引脚和时钟准备好。

### 3. `MPU_Write_Reg()`

`MPU_Write_Reg()` 是 `mpu6050.c` 中的静态底层写函数。输入是寄存器地址和 1 字节数据。它调用：

- `&hi2c2`
- `MPU6050_I2C_ADDR`
- `reg_addr`
- `I2C_MEMADD_SIZE_8BIT`
- 数据指针
- 长度 1
- 超时 100

这说明项目把 MPU6050 的寄存器写操作统一封装成一个低层函数。后续初始化寄存器时，只需要给它寄存器地址和数据。

### 4. `MPU_Read_Reg()`

`MPU_Read_Reg()` 是底层读函数。输入是寄存器地址、数据缓冲区和长度。它调用 `HAL_I2C_Mem_Read()`，从 MPU6050 指定寄存器连续读取数据。

初始化时，它读取 `REG_WHO_AM_I` 的 1 字节；运行时，它从 `REG_ACCEL_XOUT_H` 开始读取 `MPU_DATA_LENGTH` 个字节。读取内容的意义留到后续传感器章节，本章只确认 I2C 读路径。

从 HAL 机制看，`HAL_I2C_Mem_Read()` 的关键步骤是：

- 等待 `I2C_FLAG_BUSY` 释放。
- 设置状态为 `HAL_I2C_STATE_BUSY_RX` 和 `HAL_I2C_MODE_MEM`。
- 发送写方向地址和内部寄存器地址。
- 产生重复起始。
- 发送读方向地址。
- 按剩余字节数处理 ACK/NACK、`RXNE`、`BTF` 和 STOP。

这些步骤解释了为什么同一个“读寄存器”调用会同时涉及写方向地址和读方向地址。

### 5. `MPU6050_Init()`

`MPU6050_Init()` 中有一个 I2C 通信健康检查：最多重试 3 次读取设备 ID。若 HAL 状态不是 `HAL_OK` 或读到的 ID 不是 0x68，函数打印错误并返回 `false`。

这段逻辑为 I2C 调试提供了项目内观察点：串口输出中的状态码、读回值和失败提示可以帮助判断通信是否已经建立。

需要注意的是，当前打印中的状态码来自 `HAL_StatusTypeDef`，
不是 `HAL_I2C_ERROR_AF`、`HAL_I2C_ERROR_TIMEOUT` 这类 I2C 细分错误码。

若要进一步区分 NACK、总线忙、仲裁丢失或超时，
需要在调试时结合 `HAL_I2C_GetError(&hi2c2)` 或直接观察 `hi2c2.ErrorCode`；
当前仓库未把该细分错误码打印出来。

这里还要修正一个容易误导读者的调试注释。
`stm32f1xx_hal_def.h` 中 `HAL_StatusTypeDef` 的枚举是：

- `HAL_OK = 0`
- `HAL_ERROR = 1`
- `HAL_BUSY = 2`
- `HAL_TIMEOUT = 3`

因此 `mpu6050.c` 打印的“1=NoAck, 2=Error, 3=Timeout, 4=Busy”
不能当作严格枚举映射。
NACK 更接近 I2C 错误码中的 `HAL_I2C_ERROR_AF`，
需要通过 `HAL_I2C_GetError(&hi2c2)` 或 `hi2c2.ErrorCode` 进一步确认。

### 6. `MPU6050_Read_And_Process()`

该函数调用 `MPU_Read_Reg(REG_ACCEL_XOUT_H, I2C2_Buffer_Rx, MPU_DATA_LENGTH)`。如果读取失败，它清空 `rawAccel` 和 `rawGyro` 后返回。

这说明 I2C 读取失败会直接影响后续姿态输入。第14章只讲“读取失败如何在通信入口体现”，不展开数据解析和矩阵旋转。

### 7. 阻塞式 I2C 与 500Hz 循环

500Hz 控制帧的周期是 2ms。按本章第5节估算，14 字节 I2C 连续读在 100kHz 下仅总线位传输下限就约 1.53ms。这个数字说明 I2C 读取是 500Hz 帧中的重要时间预算项。

但本章不能据此断言项目必然超时或必然稳定运行，因为真实耗时还需要 DWT 计时、逻辑分析仪、示波器或运行日志证明。当前只能得出工程判断：100kHz 阻塞式 I2C 读在 500Hz 循环中必须被重点测量。

这里还有一层实时性边界：正常路径关注 1.53ms 级别的总线传输下限，异常路径关注 100ms 级别的阻塞上限。前者决定每帧是否有足够剩余预算运行姿态解算和控制输出，后者决定故障发生时主循环会被卡住多久。教材不能只写平均或理想时间，否则会漏掉传感器掉线、线缆松动、地址 NACK 时的系统行为。

### 8. 阻塞、IT 与 DMA 的边界

HAL 头文件同时提供 Blocking、Interrupt 和 DMA 三类 I2C API。当前项目只发现阻塞式 `HAL_I2C_Mem_Read()` 和 `HAL_I2C_Mem_Write()` 的业务调用。

阻塞式访问优点是调用关系简单、调试直观，适合先把 MPU6050 寄存器读写链路跑通。代价是函数执行期间 CPU 要等待总线状态，不能把这段时间用于其它任务。

IT/DMA 方案可以降低主循环等待时间，但会引入回调、缓冲区生命周期、错误恢复和总线互斥问题。当前项目没有这些证据，所以只能作为后续优化方向讨论，不能写成已经采用的实现。

### 9. 构建产物证据边界

`Debug/Three-axis_cloud_platformV2.map` 对 I2C 的证据也要分层看。当前 `HAL_I2C_Mem_Write()` 和 `HAL_I2C_Mem_Read()` 在内存映射区有最终地址，说明阻塞式 Mem 读写函数进入了最终 ELF。

同一个 `.map` 中，`HAL_I2C_Mem_Write_IT()`、`HAL_I2C_Mem_Read_IT()`、`HAL_I2C_Mem_Write_DMA()`、`HAL_I2C_Mem_Read_DMA()` 和 `HAL_I2C_IsDeviceReady()` 出现在地址为 `0x00000000` 的输入段附近，属于被丢弃的函数段证据，不能写成当前项目已经链接或调用了 IT/DMA/设备就绪检测路径。

更强的业务调用证据来自 `Debug/Three-axis_cloud_platformV2.list`。当前 `.list` 能看到 `MPU_Write_Reg()` 加载设备地址字节 `0xD0`、超时常量 `100`，并 `bl` 到 `HAL_I2C_Mem_Write()`；也能看到 `MPU_Read_Reg()` 以同样设备地址和超时参数 `bl` 到 `HAL_I2C_Mem_Read()`。

`Debug/Core/Src/i2c.su` 和 `Debug/Core/Src/i2c.cyclo` 还能给出 I2C2 初始化层的函数级静态资源边界：

- `MX_I2C2_Init` 的静态栈使用量为 8 字节，圈复杂度为 2。
- `HAL_I2C_MspInit` 的静态栈使用量为 40 字节，圈复杂度为 2。

`Debug/Drivers/CustomDrivers/Src/mpu6050.su` 和对应 `.cyclo` 文件能覆盖 MPU6050 寄存器访问封装层：

- `MPU_Write_Reg` 和 `MPU_Read_Reg` 的静态栈使用量均为 32 字节，圈复杂度均为 1。
- `MPU6050_Init` 的静态栈使用量为 32 字节，圈复杂度为 19。
- `MPU6050_Read_And_Process` 的静态栈使用量为 64 字节，圈复杂度为 3。

`Debug/Drivers/STM32F1xx_HAL_Driver/Src/stm32f1xx_hal_i2c.su` 和对应 `.cyclo` 文件能看到 HAL 阻塞式 Mem 读写及其内部内存请求函数：

- `HAL_I2C_Mem_Write` 的静态栈使用量为 40 字节，圈复杂度为 13。
- `HAL_I2C_Mem_Read` 的静态栈使用量为 56 字节，圈复杂度为 21，`.su` 中带有 `ignoring_inline_asm` 标记。
- `I2C_RequestMemoryWrite` 的静态栈使用量为 40 字节，圈复杂度为 9。
- `I2C_RequestMemoryRead` 的静态栈使用量为 40 字节，圈复杂度为 14。

这些 `.su/.cyclo` 条目只能证明当前编译选项下生成了函数级静态栈和圈复杂度记录。它们不能证明 SCL/SDA 波形正确、MPU6050 在线、100kHz 读取足够稳定，也不能把 `.su` 中出现但 `.map` 已丢弃的 IT/DMA 函数写成当前业务路径。

因此，本章判断 I2C 当前为阻塞式寄存器访问，不应只依据 HAL 头文件提供了哪些 API，而应同时引用：源码调用、`.list` 调用点、`.map` 最终地址或丢弃段，以及当前未发现 I2C2 EV/ER IRQ、DMA 和 IT 回调业务路径。

## 9. 调试方法

I2C 调试应先确认工程配置，再确认项目运行输出。

配置检查：

- `.ioc` 中 PB10 是否为 I2C2_SCL，PB11 是否为 I2C2_SDA。
- `i2c.c` 中 `ClockSpeed` 是否为 100000。
- `i2c.c` 中 `AddressingMode` 是否为 7位地址。
- `HAL_I2C_MspInit()` 是否配置 PB10/PB11 为 `GPIO_MODE_AF_OD`。
- `HAL_I2C_MspInit()` 是否启用 I2C2 外设时钟。

运行检查：

- `main.c` 中 `MX_I2C2_Init()` 是否早于 `MPU6050_Init()`。
- `mpu6050.h` 中 `MPU6050_I2C_ADDR` 是否为 `(0x68 << 1)`。
- `MPU6050_Init()` 是否能读回 `WHO_AM_I` 值 0x68。
- 失败时串口是否打印状态码、读回值和检查提示。
- 500Hz 实时控制循环中 `MPU6050_Read_And_Process()` 是否会进入读取失败保护。
- 若可单步或观察寄存器，检查 `I2C2->SR2.BUSY`、`I2C2->SR1.AF`、`I2C2->SR1.BERR`、`I2C2->SR1.ARLO` 和 `I2C2->SR1.RXNE/TXE`。
- 若临时增加调试输出，区分 `HAL_StatusTypeDef` 和 `HAL_I2C_GetError(&hi2c2)` 的细分错误码。
- 若使用逻辑分析仪，优先确认 7 位地址 0x68、地址字节 0xD0/0xD1、寄存器 0x75/0x3B 和 ACK/NACK。
- 若测量时间预算，优先用 DWT 包围 `MPU6050_Read_And_Process()`，
  再和逻辑分析仪上的 SCL 周期数互相校验。
- 若要验证异常路径，记录一次读取失败从进入 `MPU_Read_Reg()` 到返回的 DWT 耗时，并和 100ms 超时参数、2ms 控制帧周期对照。

当前仓库没有外部连线、上拉器件和实际示波观察证据，因此物理连接和外部波形只能标记为【待验证】。本章只确认工程内部 I2C2 配置和软件调用路径。

调试记录建议：

- 记录 I2C2 引脚、GPIO 开漏模式、通信参数、MSP 时钟使能和 HAL I2C 调用点。
- 分开记录 MCU 外设配置、MPU6050 寄存器读写封装和设备实际响应，避免把软件调用当作硬件成功。
- 上拉电阻、电源、接线、波形、ACK/NACK 现场结果和真实器件响应都属于仓库外实测证据。
- 若没有串口日志或调试截图，`WHO_AM_I` 读回值和读取失败保护触发情况应标记为【待验证】。
- 500Hz 循环中 I2C 读取耗时应优先用 DWT 或逻辑分析仪验证，不能只用公式估算代替实测。

总线卡死排查建议：

- 若 `I2C_FLAG_BUSY` 长时间保持置位，先区分是软件句柄忙，还是 SCL/SDA 物理线被拉低。
- 若 SDA 或 SCL 被外设持续拉低，检查传感器供电、复位、接线、上拉和地线；这些都需要现场证据【待验证】。
- 可尝试重新初始化 I2C 外设、复位传感器或断电重上电；是否有效需要实测记录。
- 手动产生若干 SCL 脉冲释放 SDA 是常见工程恢复思路，但当前项目没有实现，不能写成现有能力。

## 10. 常见问题

### 1. 为什么 I2C2 引脚要配置为开漏？

触发条件：读者看到 `GPIO_MODE_AF_OD`。

可能原因：I2C 总线通过开漏方式让多个设备共享 SCL/SDA 线。项目中 PB10/PB11 采用复用开漏模式，符合 I2C 通信需求。外部上拉是否存在需要硬件证据，当前标记为【待验证】。

开漏配置只能说明 MCU 引脚的电气模式，不等于整条总线一定已经可用。
所以本章必须把“模式正确”和“外部连线、上拉、设备响应正确”分开写。

### 2. 为什么宏是 `(0x68 << 1)`，而不是直接 0x68？

触发条件：读者看到 `MPU6050_I2C_ADDR`。

可能原因：项目头文件中用 0x68 作为 MPU6050 逻辑地址来源，再左移 1 位传给 STM32 HAL 的 I2C 设备地址参数。教材按当前代码解释，不另行推断其他地址接法。

这体现的是 HAL 地址参数格式，而不是 MPU6050 本身地址变了。
教材只记录当前工程如何传参，不擅自替换成别的写法。

### 3. `HAL_I2C_Mem_Read()` 和普通读有什么区别？

触发条件：读者第一次看到 Mem Read。

可能原因：本项目访问的是“设备内部寄存器”。因此不仅要给设备地址，还要给寄存器地址和寄存器地址宽度。本项目使用 `I2C_MEMADD_SIZE_8BIT`。

这也是为什么第14章要讲“寄存器读写”，而不是只讲“收发字节”。
I2C 在这里是寄存器访问通道，不只是裸数据搬运。

### 4. I2C 初始化成功是否等于 MPU6050 一定在线？

触发条件：`MX_I2C2_Init()` 没报错，但设备 ID 读取失败。

可能原因：I2C2 初始化只说明 MCU 外设配置完成，不证明外部传感器响应。项目用 `WHO_AM_I` 读取和返回值判断传感器是否可访问。

这也是为什么本章需要把“MCU 侧配置成功”和“器件侧响应成功”拆开。
前者是内部证据，后者需要设备 ID 或外部测量证据支持。

### 5. 为什么不建议在 SysTick 中断里直接读 MPU6050？

触发条件：读者看到 500Hz 标志来自中断，想把传感器读取也放进中断。

可能原因：当前项目的 I2C 读取是阻塞式访问，且 `stm32f1xx_it.c` 已经留下注释：中断里读 I2C 耗时，并且和 main 中读取同一总线可能造成冲突。当前实现让中断只置位 `frame_500Hz`，主循环再执行读取。

这体现了实时系统中的常见边界：中断适合做短小、确定的触发动作；慢速外设通信更适合放到可控的任务上下文中执行，除非项目已经设计好 IT/DMA、互斥和错误恢复。

### 6. 为什么本项目没有直接使用 I2C DMA 或中断模式？

触发条件：读者看到 HAL 头文件提供 `HAL_I2C_Mem_Read_IT()` 和 `HAL_I2C_Mem_Read_DMA()`。

可能原因：当前仓库没有发现对应业务调用、I2C2 中断入口或 I2C DMA 配置。阻塞式方案更直接，便于早期验证 MPU6050 寄存器通路；IT/DMA 方案需要额外处理回调、缓冲区和错误恢复。

因此教材只能把 IT/DMA 写成可优化方向，而不能写成项目当前实现。

### 7. 100kHz I2C 是否一定够 500Hz 读取？

触发条件：读者看到项目每 2ms 读取一次 MPU6050。

可能原因：14 字节连续读在 100kHz 下的理想总线下限约 1.53ms，已经占用 2ms 控制帧中的很大比例。真实耗时还会受 HAL 等待、器件响应、上升沿和其它代码影响。

所以正确结论不是“够”或“不够”，而是“必须实测”。没有 DWT 计时或逻辑分析仪记录时，保持【待验证】。

### 8. 为什么不直接把 I2C 改成 400kHz？

触发条件：读者看到 100kHz 下 14 字节读取的理想下限已经接近 2ms 控制帧。

可能原因：MPU-6050 产品规格支持最高 400kHz 的 I2C 通信，
理论上可以降低总线位传输时间。
但 Fast-mode 不是只改一个数字：还要确认总线电容、上拉电阻、上升沿、
模块布线、MPU6050 实际模块和 STM32F1 I2C 时序寄存器是否都满足要求。

当前项目证据是 `ClockSpeed = 100000`，没有 400kHz 波形或稳定性记录。
因此 400kHz 只能作为后续优化方向，不能写成已经验证的当前方案。

如果只从 HAL 寄存器计算看，400kHz 会让 `I2C_SPEED()` 进入 Fast-mode 分支。
在 36MHz PCLK1 且 `I2C_DUTYCYCLE_2` 下，CCR 近似为 `36MHz / (3 * 400kHz) = 30`；
TRISE 则进入 Fast-mode 上升时间分支，约为 `(36 * 300ns / 1000ns) + 1 = 11`。
这些只是配置推导，不等于总线波形已经满足 Fast-mode 的上升沿、电容和稳定性要求。

### 9. 为什么第14章不讲 MPU6050 所有寄存器？

触发条件：本章已经出现 `REG_WHO_AM_I` 和 `REG_ACCEL_XOUT_H`。

可能原因：这些寄存器在本章只作为 I2C 读写目标。寄存器配置、连续数据读取和字节解析分别属于后续章节，提前展开会破坏教学顺序。

第14章只建立通信通道，第18章再讲初始化寄存器，第19章再讲连续读取。
这样能让读者先明白“怎么读”，再明白“读到什么”。

3. 为什么本项目用两个静态函数封装 MPU6050 寄存器读写，而不是在所有位置直接调用 HAL？
4. 如果 `MPU6050_Read_And_Process()` 读取失败并清空原始数据，后续姿态链路可能受到什么影响？
5. 为什么第14章只讲通信通道，而把寄存器配置和数据解析留到后续章节？
6. I2C2 初始化成功、HAL 读写返回 `HAL_OK`、MPU6050 数据可信，这三者之间还缺哪些验证环节？
7. 为什么 `HAL_I2C_Mem_Read()` 读寄存器时要先发送写方向地址，再使用重复起始切换到读方向？
8. 100kHz 阻塞式 I2C 在 500Hz 控制帧中可能带来哪些时间预算风险？
9. 如果未来把 I2C 读取改成 DMA，需要新增哪些代码证据和调试证据？

11. `HAL_StatusTypeDef` 和 `HAL_I2C_GetError()` 分别回答什么问题？
12. 若逻辑分析仪显示地址 0xD0/0xD1，而代码写的是 0x68，这两种显示为什么不矛盾？

## 13. 本章总结

本章建立了三轴云台项目中 I2C2 主机通信的证据链。

已经确认的结论是：

- `.ioc` 将 PB10/PB11 配置为 I2C2_SCL/I2C2_SDA。
- `i2c.c` 将 I2C2 配置为 100kHz、7位地址模式。
- `HAL_I2C_MspInit()` 启用 GPIOB 和 I2C2 时钟，并将 PB10/PB11 配置为复用开漏。
- HAL 根据 36MHz PCLK1、100kHz 标准模式计算 `CR2.FREQ`、`TRISE` 和 `CCR`。
- 当前 100kHz 标准模式下，`DutyCycle` 字段不是 CCR 计算的决定因素；它主要影响 Fast-mode 分支。
- `mpu6050.h` 定义 `MPU6050_I2C_ADDR` 为 `(0x68 << 1)`。
- `mpu6050.c` 用 `HAL_I2C_Mem_Write()` 和 `HAL_I2C_Mem_Read()` 封装 MPU6050 寄存器访问。
- `Debug/Three-axis_cloud_platformV2.map` 与 `.list` 进一步确认阻塞式 Mem 读写进入最终 ELF，并存在从 `MPU_Write_Reg()`/`MPU_Read_Reg()` 到 HAL 阻塞 API 的实际分支调用。
- 当前 Debug `.su/.cyclo` 文件显示：`MX_I2C2_Init` 为 8 字节静态栈、圈复杂度 2；`MPU6050_Read_And_Process` 为 64 字节静态栈、圈复杂度 3；`HAL_I2C_Mem_Read` 为 56 字节静态栈、圈复杂度 21。
- `main.c` 先初始化 I2C2，再初始化 MPU6050，并在 500Hz 实时控制循环中读取传感器数据。

本章边界：

- 本章证明 MCU 侧 I2C2 初始化和 MPU6050 寄存器访问路径，不证明传感器数据已经可信。
- `HAL_OK`、`WHO_AM_I`、连续读取、方向矩阵、零偏缩放和姿态解算需要在后续章节逐层验证。
- I2C2 当前是阻塞式寄存器访问路径；`.map` 中 IT/DMA/`HAL_I2C_IsDeviceReady()` 相关函数处于 `0x00000000` 输入段，`.list` 未给出这些路径的业务调用证据。
- 14 字节读取的 1.53ms 是 100kHz 下的理论总线时间下限，不是实测运行时间。
- 100ms 是 HAL 阻塞读写的异常等待上限参数；它在 500Hz 主循环中相当于约 50 个控制帧，只能作为风险预算，不能替代 DWT 或总线波形实测。
- 外部上拉、电源、SCL/SDA 波形、ACK/NACK、从设备时钟拉伸、400kHz 可行性和总线恢复效果均需【待验证】。

下一章可以进入 USB FS 设备与中间件。与此同时，本章也为后续 MPU6050 寄存器初始化和连续读取章节建立了通信前置。
