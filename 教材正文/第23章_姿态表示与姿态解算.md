# 第23章 姿态表示与姿态解算

## 1. 本章目标

第22章已经说明 `MargAHRSupdate()` 会用加速度计重力方向修正陀螺仪输入。本章继续沿着当前 500Hz 主线向后走：修正后的角速度如何更新姿态状态，姿态状态如何变成 `ROLL/PITCH/YAW`，以及这些角度为什么还需要 `角度归一化` 才能安全进入后续控制链路。

本章目标是：

- 理解 `角度归一化` 解决的问题：角度具有周期性，直接相减会得到错误的大误差。
- 理解 `四元数姿态解算` 在项目中的主线位置：用 `q0/q1/q2/q3` 保存姿态，用陀螺仪角速度积分更新姿态。
- 理解 `欧拉角计算` 的用途：把四元数姿态转换为 `sensors.margAttitude500Hz[ROLL/PITCH/YAW]`，供后续 PID 和电机输出使用。
- 区分当前主线 `MargAHRSupdate()` 和备用方案 `evvgcCF.c`。
- 明确当前项目没有启用磁力计修正，也没有在姿态更新阶段调用 `wrapToPif()`。

本章知识链为：

`知识点总表` 中的 `角度归一化`、`四元数姿态解算`、`欧拉角计算`
-> `知识依赖图` 中 `四元数姿态解算` 依赖 `传感器物理量缩放`、`三维矩阵乘法`、`角度归一化` 和 `加速度计重力校正`，`欧拉角计算` 依赖 `四元数姿态解算` 和 `角度归一化`
-> `学习优先级` 中 P1/P2 知识
-> `教学顺序` 第39、40、41项
-> `教材章节` 第23章。

## 2. 前置知识

本章正式前置知识包括：

- `三维矩阵乘法`
- `传感器物理量缩放`
- `加速度计重力校正`

本章内部学习顺序是：

1. 先学习 `角度归一化`。
2. 再学习 `四元数姿态解算`。
3. 最后学习 `欧拉角计算`。

第27章会正式展开 `PID控制`，第29章会正式展开 `机械角与电角转换`。本章只讲这些后续章节为什么需要姿态角和角度归一化，不提前分析 PID 输出约束或三相电机输出。

## 3. 问题背景

云台控制需要知道当前机体姿态。第21章已经把 MPU6050 原始数据缩放为物理量，第22章已经说明加速度重力方向可以修正陀螺仪漂移。现在剩下的问题是：项目如何把每一帧角速度变成连续姿态。

如果直接保存 Roll、Pitch、Yaw 三个角，有两个麻烦。

第一，角度有周期性。例如 `179°` 和 `-179°` 在数值上相差 `358°`，但实际只差 `2°`。后续控制如果直接相减，电机会沿错误方向大幅动作。

第二，欧拉角在接近某些姿态时容易出现耦合和奇异性。项目当前主线用四元数保存内部姿态，再转换成欧拉角输出给控制模块。这样可以让内部积分更稳定，同时保留 Roll、Pitch、Yaw 这种直观的控制输入。

当前项目路径是：

```text
sensors.gyro500Hz / sensors.accel500Hz
-> MargAHRSupdate()
-> q0/q1/q2/q3
-> sensors.margAttitude500Hz[ROLL/PITCH/YAW]
-> computeMotorCommands()
```

本章只讲到 `sensors.margAttitude500Hz` 输出为止。

## 4. 核心概念

### 4.1 角度归一化

`角度归一化` 是把任意弧度角限制到一个固定周期区间内。项目中最常见的区间是 `[-PI, PI]`。

`Drivers/SRC/Src/computeMotorCommands.c` 中的 `wrapToPif()` 使用循环处理：

```c
while (angle > PI)
    angle -= TWO_PI;
while (angle < -PI)
    angle += TWO_PI;
```

它的意义不是改变真实方向，而是选择等价角度中最方便控制的那个表示。后续控制误差、目标角逼近和电角输出都依赖这一点。

### 4.2 四元数姿态解算

四元数用四个数表示三维姿态：

```text
q = [q0, q1, q2, q3]
```

可以把 `q0` 理解为标量部分，把 `q1/q2/q3` 理解为向量部分。项目中这四个全局变量定义在 `Drivers/SRC/Src/MargAHRS.c`。

四元数姿态解算的主线不是“直接得到角度”，而是：

1. 用第一帧加速度估计初始 Roll/Pitch。
2. 用当前角速度计算四元数增量。
3. 把增量加到当前四元数。
4. 对四元数归一化。
5. 由四元数计算欧拉角。

项目中几个关键量的意义如下：

- `q0/q1/q2/q3`：当前姿态四元数。
- `q0q0/q0q1/.../q3q3`：四元数乘积缓存，用于减少重复乘法。
- `gx/gy/gz`：进入姿态更新的角速度输入，已经经过第21章缩放和第22章重力校正。
- `dt`：本帧积分时间，来自 500Hz 主循环的实测时间差。
- `halfT`：`0.5 * dt`，对应四元数运动学中的 `0.5` 系数。

### 4.3 欧拉角计算

欧拉角是 Roll、Pitch、Yaw 三个角度。它直观，适合人理解，也适合后续控制模块按轴使用。

项目输出位置是：

```c
sensors.margAttitude500Hz[ROLL]
sensors.margAttitude500Hz[PITCH]
sensors.margAttitude500Hz[YAW]
```

这些值来自 `MargAHRSupdate()` 末尾的 `atan2f()` 和 `asinf()`。

### 4.4 当前主线与备用方案

`Drivers/SRC/Src/MargAHRS.c` 是当前 500Hz 主线姿态解算入口。

`Drivers/SRC/Src/evvgcCF.c` 中也有 `initOrientation()` 和 `getOrientation()`，但当前 `Core/Src/main.c` 没有调用它们。它们属于备用互补滤波方案，不能写成当前主线姿态输出来源。

## 5. 工作原理

本章可以从一帧 500Hz 循环看完整机制。

第一步，`Core/Src/main.c` 计算 `dt500Hz`，读取并缩放传感器数据。

第二步，`Core/Src/main.c` 调用：

```c
MargAHRSupdate(gx, gy, gz, ax, ay, az, 0, 0, 0, false, dt500Hz);
```

其中 `gx/gy/gz` 是第21章缩放后的陀螺仪角速度，单位已经是弧度每秒；`ax/ay/az` 是缩放后的加速度。

第三步，`MargAHRSupdate()` 第一次运行时调用 `MargAHRSinit()`。它用加速度计算初始 `initialRoll` 和 `initialPitch`，并假设初始 Yaw 为 0。

第四步，`MargAHRSupdate()` 使用第22章讲过的加速度重力校正修正 `gx/gy/gz`。

第五步，姿态积分开始。代码用 `halfT = dt * 0.5f`，计算四元数增量：

```text
q0i = (-q1*gx - q2*gy - q3*gz) * halfT
q1i = ( q0*gx + q2*gz - q3*gy) * halfT
q2i = ( q0*gy - q1*gz + q3*gx) * halfT
q3i = ( q0*gz + q1*gy - q2*gx) * halfT
```

这组式子来自四元数运动学关系：

```text
q_dot = 0.5 * q * omega
```

项目采用一阶欧拉积分：

```text
q_new = q_old + q_dot * dt
```

第六步，四元数归一化。理论上单位四元数长度应为 1，但离散积分和浮点误差会让长度逐渐偏离。项目计算：

```text
norm_q = q0*q0 + q1*q1 + q2*q2 + q3*q3
```

若 `norm_q` 异常，就重置为 `[1,0,0,0]`；否则用 `1 / sqrt(norm_q)` 缩放四个分量。

第七步，由四元数计算 Roll、Pitch、Yaw，写入 `sensors.margAttitude500Hz`。

## 6. STM32实现机制

本章没有新增 STM32 外设，核心发生在 Cortex-M3 CPU 的软件浮点运算中。

项目中的实现特点是：

- `sinRoll/cosRoll/sinPitch/cosPitch` 使用 `sinf()`、`cosf()`。
- 姿态更新和欧拉角计算使用 `sqrtf()`、`atan2f()`、`asinf()`。
- STM32F103RCTx 没有硬件 FPU，这些浮点运算由编译器和运行库以软件方式完成。
- 500Hz 主循环中每帧都会调用 `MargAHRSupdate()`，因此姿态解算耗时会影响后续控制周期。

当前项目已经在第10章建立了 DWT 微秒计时，在 `Core/Src/main.c` 中也有 `executionTime500Hz = micros() - currentTime`。因此调试本章性能时，可以观察 500Hz 循环总耗时，而不是凭感觉判断算法是否过重。

需要注意，`MargAHRS.c` 中定义了 `PITCH_OUTPUT_LIMIT_RAD` 和 `GIMBAL_LOCK_COS_THRESH`，但当前文件没有使用这两个宏进行 Pitch 输出限幅或万向节锁保护。因此本章不能把它们写成当前已经生效的姿态输出保护机制。

## 8. 代码分析

### 8.1 角度归一化入口

`computeMotorCommands.c` 中的 `wrapToPif()` 会把角度包裹到 `[-PI, PI]`。它在 Roll 目标逼近、Roll/Pitch/Yaw 误差计算和定子电角输出中多次出现。

`pid.c` 中的 `standardRadianFormat()` 也处理角度误差，但它只加减一次 `2 * PI`。这意味着它适合已经接近 `[-PI, PI]` 的误差；如果输入远超一个周期，`wrapToPif()` 的循环形式更稳妥。后续讲 PID 时还要继续检查这一点。

`MargAHRS.c` 也定义了一个静态 `wrapToPif()`，但当前文件内没有调用证据。因此不能把它写成当前姿态更新阶段的实际角度归一化入口。

### 8.2 MargAHRSinit初始姿态

`MargAHRSinit()` 使用加速度计算初始姿态：

```c
initialRoll  = atan2f(-ay, -az);
initialPitch = atan2f(ax, -az);
```

然后根据半角公式构造初始四元数：

```c
q0 = cosRoll * cosPitch;
q1 = sinRoll * cosPitch;
q2 = cosRoll * sinPitch;
q3 = -sinRoll * sinPitch;
```

这里没有使用磁力计，所以初始 Yaw 被固定为 0。当前 `Core/Src/main.c` 传入的磁力计参数也是 `0, 0, 0, false`，因此本章不展开磁力计航向修正。

`MargAHRSupdate()` 的函数签名中保留了 `mx/my/mz` 和 `magDataUpdate`，但当前函数体没有使用这些参数形成磁力计误差或航向反馈。它们只能说明接口保留了扩展位置，不能说明当前项目已经启用 MARG 的磁力计部分。

### 8.3 MargAHRSupdate输入防护

`MargAHRSupdate()` 开头会检查 `gx/gy/gz` 和 `ax/ay/az` 是否为 `NaN` 或 `Inf`。异常值会被置为 0。

这不是姿态解算数学本身，而是工程防护。第32章会系统讲 `NaN与Inf防护`；本章只说明它存在，避免读者把异常清零误解为滤波或标定。

### 8.4 四元数积分

第22章的加速度重力校正完成后，`gx/gy/gz` 已经被 `KpAcc` 修正。随后四元数增量由当前四元数和角速度共同决定：

```c
q0i = (-q1 * gx - q2 * gy - q3 * gz) * halfT;
q1i = ( q0 * gx + q2 * gz - q3 * gy) * halfT;
q2i = ( q0 * gy - q1 * gz + q3 * gx) * halfT;
q3i = ( q0 * gz + q1 * gy - q2 * gx) * halfT;
```

这里的 `halfT` 是 `dt * 0.5f`。如果 `dt500Hz` 异常，积分步长也会异常；当前 `MargAHRSupdate()` 内部没有对 `dt` 做上下限保护，时间基准可靠性需要由第26章的 500Hz 调度来保证。

### 8.5 四元数归一化

更新后，代码计算：

```c
float norm_q = q0*q0 + q1*q1 + q2*q2 + q3*q3;
```

如果 `norm_q < 1e-10f` 或者 `norm_q` 是 `NaN`，项目把四元数重置为单位姿态：

```c
q0 = 1.0f;
q1 = q2 = q3 = 0.0f;
```

否则把四个分量除以长度。这个处理可以抑制数值漂移，是四元数积分后必须保留的工程步骤。

### 8.6 欧拉角输出

归一化后，项目更新乘积缓存 `q0q0/q0q1/.../q3q3`，再计算：

```c
sensors.margAttitude500Hz[ROLL ] = atan2f(...);
sensors.margAttitude500Hz[PITCH] = -asinf(...);
sensors.margAttitude500Hz[YAW  ] = atan2f(...);
```

输出单位是弧度。后续控制模块直接读取这些弧度值，例如 `computeMotorCommands.c` 中的：

```c
float pitch_angle = PITCH_SENSOR_SIGN * sensors.margAttitude500Hz[PITCH];
float yaw_angle = sensors.margAttitude500Hz[YAW];
```

如果 `asinf()` 或 `atan2f()` 结果变成 `NaN`，项目会把对应姿态角置为 0。这里没有对 `asinf()` 的输入先做区间钳位，因此极端数值误差导致 Pitch 清零的可能性需要在第32章继续关注。

同样，虽然文件顶部定义了 Pitch 输出限制和万向节锁阈值相关宏，当前欧拉角输出路径没有调用这些宏。若后续要加入 Pitch 限幅或奇异姿态保护，应先回到源码和知识库补充证据，再写入教材主线。

### 8.7 evvgcCF不是当前主线

`evvgcCF.c` 中的 `initOrientation()` 和 `getOrientation()` 使用加速度角、陀螺仪积分和小比例校正形成互补滤波。它还写入 `sensors.evvgcCFAttitude500Hz`。

但是：

- `evvgcCF.h` 中 `initOrientation()` 声明被注释。
- 当前全工程没有发现 `getOrientation()` 的调用点。
- `Core/Src/main.c` 当前主线使用 `sensors.margAttitude500Hz`，不是 `sensors.evvgcCFAttitude500Hz`。

因此第24章会把它作为 `互补滤波` 备用方案讲解，本章只把它作为对照证据。

## 9. 调试方法

第一步，确认当前姿态主线。

- 在 `Core/Src/main.c` 的 `MargAHRSupdate()` 调用处设置断点。
- 观察 `sensors.gyro500Hz`、`sensors.accel500Hz` 和 `dt500Hz`。
- 单步进入 `Drivers/SRC/Src/MargAHRS.c`。

第二步，观察初始化边界。

- 复位后观察 `MargAHRSinitialized`。
- 第一次进入 `MargAHRSupdate()` 时应调用 `MargAHRSinit()`。
- 记录 `q0/q1/q2/q3` 的初始值，并结合静止姿态判断 Roll/Pitch 是否合理。

第三步，观察四元数积分。

- 在 `q0i/q1i/q2i/q3i` 计算处观察 `gx/gy/gz` 和 `halfT`。
- 确认 `dt500Hz` 接近 0.002 秒。
- 若 `dt500Hz` 过大或过小，应先回到 500Hz 调度链路排查。

第四步，观察四元数归一化。

- 观察 `norm_q`。
- 正常情况下它应接近 1。
- 如果触发重置为 `[1,0,0,0]`，说明姿态状态已经异常，需要检查输入数据和 `dt500Hz`。

第五步，观察欧拉角输出。

- 观察 `sensors.margAttitude500Hz[ROLL/PITCH/YAW]`。
- 静止状态下 Roll/Pitch 应与安装姿态和第20章坐标变换一致。
- 当前项目没有外部航向观测，Yaw 长时间漂移属于预期风险，不应误判为单纯代码错误。

第六步，确认备用方案未进入主线。

- 搜索 `getOrientation(` 和 `sensors.evvgcCFAttitude500Hz` 的调用关系。
- 如果当前没有调用点，就不要用 `evvgcCF.c` 的结果解释主线姿态输出。

调试记录建议：

- 记录 `MargAHRSupdate()` 调用点、输入三轴数据、`dt500Hz`、初始化状态和四元数初值。
- 对姿态更新，应记录积分前后四元数、归一化模长、异常重置条件和欧拉角输出。
- Yaw 结论应分开记录源码积分结果、外部航向观测缺失和长期漂移风险。
- Roll/Pitch 方向正确性需要结合第20章坐标变换和仓库外运动测试，缺失时标记为【待验证】。

## 10. 常见问题

问题一：为什么不直接用 Roll/Pitch/Yaw 做内部积分。

欧拉角直观，但角度周期和姿态奇异性会让积分和耦合处理更麻烦。
当前项目选择用四元数保存内部姿态，再输出欧拉角给控制链路。
这也是为什么本章先讲姿态表示，再讲第27章和第28章中的角度误差处理。

问题二：`wrapToPif()` 是否在 `MargAHRSupdate()` 中生效。

当前没有调用证据。`MargAHRS.c` 中定义了静态 `wrapToPif()`，但姿态更新和欧拉角输出没有调用它。角度归一化主要在后续控制链路中使用。

问题三：为什么 Pitch 使用 `-asinf()`。

这是项目当前姿态坐标约定的一部分，和第20章安装方向、控制轴符号一起决定最终正负号。这里必须按源码记录，不能凭通用公式改写。

问题四：Yaw 为什么可能漂移。

当前 `Core/Src/main.c` 传入磁力计为 `0, 0, 0, false`，`MargAHRSupdate()` 中也没有使用磁力计修正。Yaw 主要由陀螺仪积分得到，长期漂移是当前主线的已知风险。

问题五：`evvgcCF.c` 是否输出当前控制使用的姿态。

当前不是。当前控制读取 `sensors.margAttitude500Hz`，而 `evvgcCF.c` 写的是 `sensors.evvgcCFAttitude500Hz`，且当前没有调用证据。

问题六：四元数归一化和角度归一化是不是同一件事。

不是。四元数归一化是让四元数长度保持为 1；角度归一化是把弧度角包裹到 `[-PI, PI]`。二者都叫归一化，但对象和目的不同。

问题七：`PITCH_OUTPUT_LIMIT_RAD` 和 `GIMBAL_LOCK_COS_THRESH` 是否已经保护了姿态输出。

当前没有。它们在 `MargAHRS.c` 中有定义，但没有调用证据。当前 Pitch 输出只在结果为 `NaN` 时被置为 0，没有看到基于这两个宏的限幅或奇异姿态保护。

问题八：姿态角不是 `NaN`，是否就表示姿态估计可信。

不能这样判断。
非 `NaN` 只能说明数值格式没有进入明显异常。
姿态是否可信还取决于第20章的坐标方向、第21章的零偏与缩放、
第22章的加速度修正，以及陀螺仪漂移和外部运动条件。
当前仓库缺少仓库外实测证据，不能仅凭有限数值断言姿态正确。

## 13. 本章总结

本章完成了从传感器物理量到姿态角输出的主线解释。

当前项目中：

- `角度归一化` 主要服务后续控制误差、目标角逼近和电角输出。
- `MargAHRSupdate()` 是当前 500Hz 主线的姿态更新入口。
- `MargAHRSinit()` 用第一帧加速度估计初始 Roll/Pitch，并把初始 Yaw 置为 0。
- `MargAHRSupdate()` 用修正后的 `gx/gy/gz` 更新 `q0/q1/q2/q3`。
- 四元数更新后必须做长度归一化，否则离散积分误差会积累。
- `sensors.margAttitude500Hz[ROLL/PITCH/YAW]` 是当前主线输出给控制模块的姿态角。
- `mx/my/mz`、`magDataUpdate`、`PITCH_OUTPUT_LIMIT_RAD` 和 `GIMBAL_LOCK_COS_THRESH` 在当前姿态主线中没有形成已生效的修正或保护。
- `evvgcCF.c` 是备用互补滤波方案，当前没有进入主线。

目标回扣：

- 本章已经把角度归一化、四元数姿态解算和欧拉角计算放回当前 500Hz 主线。
- 本章已经说明 `sensors.margAttitude500Hz` 是当前控制模块读取的姿态角来源。
- 本章只闭合软件姿态输出链路，不证明 Roll/Pitch/Yaw 方向和幅值已经通过仓库外运动测试。

本章保留三个边界：

- 不把 `MargAHRS.c` 中未调用的 `wrapToPif()` 写成姿态更新阶段已生效逻辑。
- 不把 `evvgcCF.c` 写成当前主线姿态来源。
- 不提前展开 PID 控制和电机输出，只说明它们为什么依赖姿态角和角度归一化。

下一章将进入 `互补滤波备用方案`，专门解释 `evvgcCF.c` 为什么存在、它与当前 `四元数姿态解算` 的差异，以及如何判断备用姿态算法是否进入项目主线。
