## 1. 本章目标

第22章已经说明 `MargAHRSupdate()` 会用加速度计重力方向修正陀螺仪输入。本章继续沿着当前
500Hz 主线向后走：修正后的角速度如何更新姿态状态，姿态状态如何变成
`ROLL/PITCH/YAW`，以及这些角度为什么还需要 `角度归一化` 才能安全进入后续控制链路。

本章目标是：

- 理解 `角度归一化` 解决的问题：角度具有周期性，直接相减会得到错误的大误差。
- 理解 `四元数姿态解算` 在项目中的主线位置：用 `q0/q1/q2/q3` 保存姿态，用陀螺仪角速度积分更新姿态。
- 理解 `欧拉角计算` 的用途：把四元数姿态转换为 `sensors.margAttitude500Hz[ROLL/PITCH/YAW]`，供后续 PID 和电机输出使用。
- 区分当前主线 `MargAHRSupdate()` 和备用方案 `evvgcCF.c`。
- 明确当前项目没有启用磁力计修正，也没有在姿态更新阶段调用 `wrapToPif()`。
- 识别第一帧加速度有效性对 `MargAHRSinit()` 初始姿态的影响。
- 区分低加速度模长时“跳过重力校正”和“停止四元数更新”这两种不同结论。

## 3. 问题背景

云台控制需要知道当前机体姿态。第21章已经把 MPU6050 原始数据缩放为物理量，第22章已经说明加速度重力方向可以修正陀螺仪漂移。现在剩下的问题是：项目如何把每一帧角速度变成连续姿态。

如果直接保存 Roll、Pitch、Yaw 三个角，有两个麻烦。

第一，角度有周期性。例如 `179°` 和 `-179°` 在数值上相差 `358°`，但实际只差 `2°`。后续控制如果直接相减，电机会沿错误方向大幅动作。

第二，欧拉角在接近某些姿态时容易出现耦合和奇异性。项目当前主线用四元数保存内部姿态，再转换成欧拉角输出给控制模块。这样可以让内部积分更稳定，同时保留 Roll、Pitch、Yaw 这种直观的控制输入。

当前项目路径是：

```text
sensors.gyro500Hz / sensors.accel500Hz
-> MargAHRSupdate()
-> q0/q1/q2/q3
-> sensors.margAttitude500Hz[ROLL/PITCH/YAW]
-> computeMotorCommands()
```

本章只讲到 `sensors.margAttitude500Hz` 输出为止。

还有一个容易被忽略的启动边界：`MargAHRSupdate()` 第一次运行时会立刻用当前加速度输入初始化 Roll/Pitch。这个初始化发生在加速度模长阈值判断之前，因此第一帧输入是否可信，会直接影响初始四元数。

## 4. 核心概念

### 4.1 角度归一化

`角度归一化` 是把任意弧度角限制到一个固定周期区间内。项目中最常见的区间是 `[-PI, PI]`。

`Drivers/SRC/Src/computeMotorCommands.c` 中的 `wrapToPif()` 使用循环处理：

```c
while (angle > PI)
    angle -= TWO_PI;
while (angle < -PI)
    angle += TWO_PI;
```

它的意义不是改变真实方向，而是选择等价角度中最方便控制的那个表示。后续控制误差、目标角逼近和电角输出都依赖这一点。

一个最小例子可以说明它为什么必要。若目标角是 `-179°`，当前角是 `179°`，
直接相减会得到：

```text
-179° - 179° = -358° ~= -6.248 rad
```

但真实最短转动只需要跨过 `PI/-PI` 边界：

```text
2° ~= 0.0349 rad
```

`wrapToPif()` 的任务就是把“数值上很大、物理上等价”的误差变成最短周期误差。
如果没有这一步，后续 PID 可能把很小的跨界误差当成一次接近整圈的大误差处理。

### 4.2 四元数姿态解算

四元数用四个数表示三维姿态：

```text
q = [q0, q1, q2, q3]
```

可以把 `q0` 理解为标量部分，把 `q1/q2/q3` 理解为向量部分。项目中这四个全局变量定义在 `Drivers/SRC/Src/MargAHRS.c`。

单位四元数更完整的姿态含义是：

```text
q = [cos(theta / 2),
     ux * sin(theta / 2),
     uy * sin(theta / 2),
     uz * sin(theta / 2)]
```

其中 `[ux, uy, uz]` 是单位旋转轴，`theta` 是绕该轴旋转的角度。这里出现半角，是因为
四元数旋转使用 `q` 和它的共轭共同作用在向量上。对姿态表示而言，必须满足：

```text
q0^2 + q1^2 + q2^2 + q3^2 = 1
```

这也是为什么项目每次积分后都要做四元数归一化。还要注意，`q` 和 `-q` 表示同一个
空间姿态；调试时看到四元数整体变号，不应立刻判断姿态跳变，必须结合欧拉角或旋转矩阵验证。

项目选择四元数作为内部状态，是在几个方案之间取折中：

- 直接积分欧拉角最直观，但会遇到角度周期、轴间耦合和奇异姿态问题。
- 保存 3x3 方向余弦矩阵没有四元数半角概念，但需要维护 9 个元素和正交性。
- 单位四元数只需 4 个数，适合 500Hz 软件循环，但必须维护长度为 1。

四元数姿态解算的主线不是“直接得到角度”，而是：

1. 用第一帧加速度估计初始 Roll/Pitch。
2. 用当前角速度计算四元数增量。
3. 把增量加到当前四元数。
4. 对四元数归一化。
5. 由四元数计算欧拉角。

项目中几个关键量的意义如下：

- `q0/q1/q2/q3`：当前姿态四元数。
- `q0q0/q0q1/.../q3q3`：四元数乘积缓存，用于减少重复乘法。
- `gx/gy/gz`：进入姿态更新的角速度输入，已经经过第21章缩放和第22章重力校正。
- `dt`：本帧积分时间，来自 500Hz 主循环的实测时间差。
- `halfT`：`0.5 * dt`，对应四元数运动学中的 `0.5` 系数。

### 4.3 欧拉角计算

欧拉角是 Roll、Pitch、Yaw 三个角度。它直观，适合人理解，也适合后续控制模块按轴使用。

项目输出位置是：

```c
sensors.margAttitude500Hz[ROLL]
sensors.margAttitude500Hz[PITCH]
sensors.margAttitude500Hz[YAW]
```

这些值来自 `MargAHRSupdate()` 末尾的 `atan2f()` 和 `asinf()`。

### 4.4 当前主线与备用方案

`Drivers/SRC/Src/MargAHRS.c` 是当前 500Hz 主线姿态解算入口。

`Drivers/SRC/Src/evvgcCF.c` 中也有 `initOrientation()` 和 `getOrientation()`，但当前 `Core/Src/main.c` 没有调用它们。它们属于备用互补滤波方案，不能写成当前主线姿态输出来源。

### 4.5 初始化有效性

姿态解算不仅依赖每一帧积分，还依赖初始姿态。当前项目的初始姿态来自第一帧传入 `MargAHRSupdate()` 的 `ax/ay/az`。

这带来一个工程边界：如果第一帧加速度来自读取失败后的清零路径、`NaN/Inf` 清零路径，或者设备正在明显运动，那么初始 Roll/Pitch 可能不代表真实静止姿态。源码能证明初始化会执行，但不能证明第一帧一定可靠。

当前 `MargAHRSinitialized` 一旦置为 1，后续代码没有自动重新初始化逻辑。因此调试姿态问题时，不能只看后续 500Hz 循环是否在运行，还要回看第一帧初始化输入。

还要避免把 `startup_delay_counter` 误解为初始化保护。`Core/Src/main.c` 的顺序是先调用
`MargAHRSupdate()`，再进入启动延时计数和电机使能逻辑。也就是说，启动延时可以推迟控制输出真正接管电机，
但不会推迟第一帧 `MargAHRSinit()`。若第一帧加速度不可用，它会成为姿态递推的初始条件：

```text
q_init = f(a0)
q[k+1] = Integrate(q[k], gyro[k], accel_correction[k], dt[k])
```

所以坏的第一帧不是“单次坏样本”这么简单，而可能变成后续所有姿态估计的起点偏差。

## 5. 工作原理

本章可以从一帧 500Hz 循环看完整机制。

第一步，`Core/Src/main.c` 计算 `dt500Hz`，读取并缩放传感器数据。

第二步，`Core/Src/main.c` 调用：

```c
MargAHRSupdate(gx, gy, gz, ax, ay, az, 0, 0, 0, false, dt500Hz);
```

其中 `gx/gy/gz` 是第21章缩放后的陀螺仪角速度，单位已经是弧度每秒；`ax/ay/az` 是缩放后的加速度。

第三步，`MargAHRSupdate()` 第一次运行时调用 `MargAHRSinit()`。
它用加速度计算初始 `initialRoll` 和 `initialPitch`，并假设初始 Yaw 为 0。
当前源码在初始化前只做 `NaN/Inf` 清零，没有先用 `norm > 0.1f` 验证加速度模长。

第四步，`MargAHRSupdate()` 使用第22章讲过的加速度重力校正修正 `gx/gy/gz`。

第五步，姿态积分开始。代码用 `halfT = dt * 0.5f`，计算四元数增量：

```text
q0i = (-q1*gx - q2*gy - q3*gz) * halfT
q1i = ( q0*gx + q2*gz - q3*gy) * halfT
q2i = ( q0*gy - q1*gz + q3*gx) * halfT
q3i = ( q0*gz + q1*gy - q2*gx) * halfT
```

这组式子来自四元数运动学关系：

```text
q_dot = 0.5 * q * omega
```

这里的 `omega` 不是普通三维向量直接相乘，而是把角速度写成纯四元数：

```text
omega_q = [0, gx, gy, gz]
```

四元数乘法展开为：

```text
q * omega_q =
[-q1*gx - q2*gy - q3*gz,
  q0*gx + q2*gz - q3*gy,
  q0*gy - q1*gz + q3*gx,
  q0*gz + q1*gy - q2*gx]
```

再乘以 `0.5 * dt`，就得到代码中的 `q0i/q1i/q2i/q3i`。因此 `halfT` 不是经验系数，
而是四元数运动学微分方程中的 `0.5` 与积分步长 `dt` 的组合。

项目采用一阶欧拉积分：

```text
q_new = q_old + q_dot * dt
```

更精确的姿态增量可以写成指数形式：

```text
delta_q = [cos(|omega| * dt / 2),
           u * sin(|omega| * dt / 2)]
```

其中 `u = omega / |omega|`。当前项目没有使用指数映射，而是使用一阶欧拉积分再归一化。
这个选择计算量低，适合 STM32F103 的软件浮点 500Hz 循环；代价是当 `|omega| * dt`
较大时，离散积分误差会变大。因此 `dt500Hz` 稳定、陀螺仪量程合理和四元数归一化都很关键。

还可以用一个最小量级估算看出为什么归一化不是装饰步骤。假设当前姿态为单位四元数
`q = [1,0,0,0]`，只有一个很小的 `gz`，欧拉积分后：

```text
q_temp = [1, 0, 0, gz * dt / 2]
```

此时四元数模长平方变为：

```text
||q_temp||^2 = 1 + (gz * dt / 2)^2
```

只要角速度和时间步长不为 0，模长就会略大于 1。单帧偏差通常很小，但 500Hz 循环会反复积分；
如果没有归一化，这种数值偏差会持续累积，最终让“四元数表示旋转”这个前提被破坏。
这就是第8.5节归一化步骤的数学原因。

第六步，四元数归一化。理论上单位四元数长度应为 1，但离散积分和浮点误差会让长度逐渐偏离。项目计算：

```text
norm_q = q0*q0 + q1*q1 + q2*q2 + q3*q3
```

若 `norm_q` 异常，就重置为 `[1,0,0,0]`；否则用 `1 / sqrt(norm_q)` 缩放四个分量。

第七步，由四元数计算 Roll、Pitch、Yaw，写入 `sensors.margAttitude500Hz`。

## 6. STM32实现机制

本章没有新增 STM32 外设，核心发生在 Cortex-M3 CPU 的软件浮点运算中。

项目中的实现特点是：

- `sinRoll/cosRoll/sinPitch/cosPitch` 使用 `sinf()`、`cosf()`。
- 姿态更新和欧拉角计算使用 `sqrtf()`、`atan2f()`、`asinf()`。
- STM32F103RCTx 没有硬件 FPU，这些浮点运算由编译器和运行库以软件方式完成。
- 500Hz 主循环中每帧都会调用 `MargAHRSupdate()`，因此姿态解算耗时会影响后续控制周期。

当前项目已经在第10章建立了 DWT 微秒计时，在 `Core/Src/main.c` 中也有 `executionTime500Hz = micros() - currentTime`。因此调试本章性能时，可以观察 500Hz 循环总耗时，而不是凭感觉判断算法是否过重。

从工程选型看，当前主线更接近“陀螺积分 + 加速度比例校正 + 四元数归一化”的轻量 AHRS。
它没有维护 EKF 所需的状态协方差、观测矩阵和噪声矩阵，也没有使用磁力计形成航向观测。
这类实现的优点是计算量低、参数少、适合 Cortex-M3 软件浮点；缺点是无法像 EKF 那样显式
建模零偏和观测噪声，也无法在没有外部航向观测时长期约束 Yaw 漂移。

因此本章不能把当前实现描述为“完整 MARG 融合”或“EKF 姿态估计”。更准确的表述是：
接口名保留了 `MARG` 形态，当前调用方式和函数体实际运行的是 6 轴 IMU 姿态更新。
同理，`Core/Src/main.c` 注释中出现了 `Mahony/Madgwick` 字样，但当前 `MargAHRSupdate()`
没有 Madgwick 常见的梯度下降误差项，也没有 `beta` 增益参数。教材应按函数体实际路径描述：
加速度方向误差经 `KpAcc` 比例反馈修正陀螺仪，再做四元数积分和归一化。

需要注意，`MargAHRS.c` 中定义了 `PITCH_OUTPUT_LIMIT_RAD` 和 `GIMBAL_LOCK_COS_THRESH`，但当前文件没有使用这两个宏进行 Pitch 输出限幅或万向节锁保护。因此本章不能把它们写成当前已经生效的姿态输出保护机制。

## 7. 项目中的应用

本章涉及的项目证据文件包括：

- `Core/Src/main.c`
- `Drivers/SRC/Src/MargAHRS.c`
- `Drivers/SRC/Inc/MargAHRS.h`
- `Drivers/SRC/Src/computeMotorCommands.c`
- `Drivers/SRC/Src/pid.c`
- `Drivers/SRC/Src/evvgcCF.c`
- `Drivers/BGC/bgc32.h`

`Core/Src/main.c` 负责在 500Hz 主循环中调用 `MargAHRSupdate()`，并在其后调用 `computeMotorCommands()`。

`MargAHRS.c` 保存四元数状态，完成初始化、加速度重力校正、四元数更新、四元数归一化和欧拉角输出。

`bgc32.h` 定义 `sensors_t`，其中 `margAttitude500Hz[3]` 是当前主线姿态输出容器。
需要注意，`bgc32.h` 注释把它写成“卡尔曼姿态角”，但当前仓库没有看到该数组由 Kalman/EKF
状态方程或协方差更新写入。实际写入点在 `MargAHRS.c` 的欧拉角输出处。因此教材应按
“当前 AHRS 主线姿态角容器”讲解，而不能仅凭注释把它定义为卡尔曼滤波输出。

`computeMotorCommands.c` 和 `pid.c` 确认 `角度归一化` 会在后续控制链路中使用。本章只建立原因和入口，具体 PID 行为留到第27章和第28章。

`evvgcCF.c` 确认项目中存在另一套欧拉角/互补滤波思路，但当前 `Core/Src/main.c` 没有调用它。

### 7.1 构建产物证据边界

除源码本身外，`Debug/Three-axis_cloud_platformV2.map` 和 `Debug/Three-axis_cloud_platformV2.list`
还能给本章提供一层“最终构建确实包含这些路径”的证据。

`.map` 文件显示 `MargAHRSinit` 和 `MargAHRSupdate` 已分配到 Flash 地址，
`q0`、`q1`、`q2`、`q3`、`q0q0` 至 `q3q3`、`exAcc`、`eyAcc`、`ezAcc`、`halfT`
等姿态状态和误差中间量进入最终 RAM 布局。它还显示 `MargAHRS.o` 触发了
`asinf`、`atan2f`、`sqrtf`、`cosf` 等 `libm.a` 符号进入链接。这里不把 `sinf`
作为 `MargAHRS.o` 的 `.map` 结论，因为当前 `.map` 中 `sinf` 的归因行指向
`drv_pwmMotors.o`，不能仅凭该行证明它由姿态模块拉入。

`.list` 文件进一步显示，`main()` 的 500Hz 主循环中存在到 `MargAHRSupdate()` 的实际
`bl` 调用；在 `MargAHRSupdate()` 输出 `sensors.margAttitude500Hz[ROLL/PITCH/YAW]`
时，可以看到 `atan2f`、`asinf`、`atan2f` 的调用路径。这些证据能证明当前固件构建中
确实存在“加速度误差比例反馈、四元数积分/归一化、欧拉角输出”的执行入口和链接结果。

`Debug/Drivers/SRC/Src/MargAHRS.su` 与 `Debug/Drivers/SRC/Src/MargAHRS.cyclo`
还能补充一层静态分析边界：`MargAHRSinit()` 的静态栈用量为 48 字节、圈复杂度为 1；
`MargAHRSupdate()` 的静态栈用量为 80 字节、圈复杂度为 33；
`calculateAccConfidence()` 的静态栈用量为 24 字节、圈复杂度为 1。`Debug/Core/Src/main.su`
与 `Debug/Core/Src/main.cyclo` 同时显示 `main()` 的静态栈用量为 56 字节、圈复杂度为 7。
这些数据能说明当前构建对入口函数做过静态栈和分支复杂度分析，也能提示
`MargAHRSupdate()` 是一个分支较密集的姿态更新函数；但它们不能替代运行时栈水位、
总调用链栈深、真实耗时或分支覆盖测试。尤其是 `calculateAccConfidence()` 虽然出现在
`.su/.cyclo` 中，但当前主线还需要 `.list` 调用证据才能把它写成已生效的置信度门控。

但构建产物不能证明运行时姿态角一定正确，也不能证明传感器安装方向、采样周期裕量、
陀螺/加速度单位换算、磁力计航向约束或硬件噪声表现已经满足控制要求。这些仍属于
`【待验证】` 的实机问题。Mahony 论文和 Madgwick 报告只能作为算法分类参考，当前项目的
权威依据仍然是 `MargAHRS.c` 函数体和构建产物；现有源码支持的结论是“类似 Mahony 的
加速度误差比例反馈 + 四元数积分/归一化”，不能扩写成完整 Madgwick、完整 MARG 或 EKF。

### 7.2 姿态状态、输出使用与运行正确性分层

第23章最容易混淆的不是公式本身，而是“姿态状态存在”“姿态输出被使用”和“姿态估计正确”这三件事。

- 姿态状态层：`MargAHRS.c` 定义 `q0/q1/q2/q3`、乘积缓存 `q0q0` 至 `q3q3`、`halfT` 和 `MargAHRSinitialized`。`.map` 能证明这些对象进入当前 RAM 布局，`.list` 能证明 `MargAHRSupdate()` 中确实存在初始化、积分、归一化和缓存刷新指令。这一层只能证明状态对象和更新代码进入构建。
- 姿态输出层：`MargAHRSupdate()` 把 `atan2f/asinf/atan2f` 的结果写入 `sensors.margAttitude500Hz[ROLL/PITCH/YAW]`；`Core/Src/main.c` 的 500Hz 主循环在姿态更新后调用 `computeMotorCommands(dt500Hz)`，10Hz 打印路径还读取 `sensors.margAttitude500Hz[ROLL]`。这一层能证明姿态角是当前控制链路读取的输入来源。
- 运行正确性层：Roll/Pitch/Yaw 的符号方向、幅值比例、动态延迟、噪声水平、第一帧初始化可靠性、低模长退化行为和异常 `dt` 后果，都不能只靠 `.map/.list` 证明。它们需要第20章坐标方向回查、静态姿态记录、受控单轴转动、DWT 计时、串口日志或外部姿态基准。缺少这些证据时继续写成【待验证】。

因此，本章可以说“当前构建包含四元数姿态状态、500Hz 姿态更新入口和欧拉角输出写入路径”，也可以说“控制模块读取的是 `sensors.margAttitude500Hz`”。但不能把这些证据升级成“姿态方向已经实测正确”“欧拉角没有延迟问题”或“闭环控制稳定性已经验证”。

## 8. 代码分析

### 8.1 角度归一化入口

`computeMotorCommands.c` 中的 `wrapToPif()` 会把角度包裹到 `[-PI, PI]`。它在 Roll 目标逼近、Roll/Pitch/Yaw 误差计算和定子电角输出中多次出现。

`pid.c` 中的 `standardRadianFormat()` 也处理角度误差，但它只加减一次 `2 * PI`。
这意味着它适合已经接近 `[-PI, PI]` 的误差；如果输入远超一个周期，
`wrapToPif()` 的循环形式更稳妥。后续讲 PID 时还要继续检查这一点。

`MargAHRS.c` 也定义了一个静态 `wrapToPif()`，但当前文件内没有调用证据。因此不能把它写成当前姿态更新阶段的实际角度归一化入口。

这里还要区分两层角度归一化入口。

第一层在 `computeMotorCommands.c`。Roll 目标逼近、Roll 电角误差、Pitch 机械误差、Yaw
机械误差和定子电角输出都能看到 `wrapToPif()`。这一层直接服务控制链路，是本章说“姿态角
需要周期处理”的主要项目证据。

第二层在 `pid.c`。所有默认 PID 类型在 `config.c` 中被设为 `ANGULAR`，因此 `updatePID()`
内部会对 `command - state` 调用 `standardRadianFormat()`。但 `standardRadianFormat()`
只加减一次 `2 * PI`，并不是通用的多圈归一化函数。

这带来一个教学边界：第23章只证明后续链路确实存在角度归一化需求；至于每个轴是否总能把
PID 输入限制到单周期附近，要到第27章和第28章结合 Roll/Pitch/Yaw 各自调用方式继续分析。
例如 Pitch 分支当前先计算了 `error_mech = wrapToPif(target_angle - pitch_angle)`，
但传给 `updatePID()` 的仍是 `target_angle` 和 `pitch_angle`；Yaw 分支则构造了
`safe_target = yaw_angle + error_mech` 后再调用 PID。二者不是完全相同的误差路径。

### 8.2 MargAHRSinit初始姿态

`MargAHRSinit()` 使用加速度计算初始姿态：

```c
initialRoll  = atan2f(-ay, -az);
initialPitch = atan2f(ax, -az);
```

然后根据半角公式构造初始四元数：

```c
q0 = cosRoll * cosPitch;
q1 = sinRoll * cosPitch;
q2 = cosRoll * sinPitch;
q3 = -sinRoll * sinPitch;
```

这里没有使用磁力计，所以初始 Yaw 被固定为 0。当前 `Core/Src/main.c` 传入的磁力计参数也是 `0, 0, 0, false`，因此本章不展开磁力计航向修正。

`MargAHRSupdate()` 的函数签名中保留了 `mx/my/mz` 和 `magDataUpdate`，但当前函数体没有使用这些参数形成磁力计误差或航向反馈。它们只能说明接口保留了扩展位置，不能说明当前项目已经启用 MARG 的磁力计部分。

#### 8.2.1 首帧初始化的构建证据边界

当前 Debug 构建的 `.map` 文件显示，`MargAHRSinit` 被分配到 `0x0800373c`，
`MargAHRSupdate` 被分配到 `0x08003938`，`q0`、`halfT` 和 `MargAHRSinitialized`
也进入了最终 RAM 布局。`.list` 文件进一步显示，`MargAHRSupdate()` 会读取
`MargAHRSinitialized`，在未初始化时通过 `bl 800373c <MargAHRSinit>` 进入初始化路径，
随后再把 `MargAHRSinitialized` 置为 1。

这只能证明“首帧初始化路径确实进入当前二进制”，还可以证明 `MargAHRSinit()` 内部包含
两次 `atan2f()` 计算 `initialRoll/initialPitch`，以及后续 `cosf()` 半角三角函数调用。
但 `.map/.list` 不能证明第一次传入的 `ax/ay/az` 一定来自有效 I2C 样本，也不能证明上电时云台静止、
安装方向正确或加速度模长已经可信。因此首帧初始化的运行时可靠性仍应保持【待验证】，
需要用断点或日志记录第一帧 `ax/ay/az`、模长、`initialRoll/initialPitch` 和初始化后的
`q0/q1/q2/q3`。

### 8.3 MargAHRSupdate输入防护

`MargAHRSupdate()` 开头会检查 `gx/gy/gz` 和 `ax/ay/az` 是否为 `NaN` 或 `Inf`。异常值会被置为 0。

这不是姿态解算数学本身，而是工程防护。第32章会系统讲 `NaN与Inf防护`；本章只说明它存在，避免读者把异常清零误解为滤波或标定。

### 8.3.1 初始化与低模长边界

`MargAHRSupdate()` 的初始化发生在加速度模长判断之前：

```c
if (MargAHRSinitialized == 0)
{
    MargAHRSinit(ax, ay, az, mx, my, mz);
    MargAHRSinitialized = 1;
}
```

后面才计算加速度模长：

```c
norm = sqrt(ax*ax + ay*ay + az*az);

if (norm > 0.1f)
{
    /* 归一化、重力误差、KpAcc 修正 */
}
```

所以 `norm > 0.1f` 只保护加速度重力校正分支，不保护第一次 `MargAHRSinit()`。如果第一帧加速度输入已经被第21章所说的读取失败路径清零，或者被本函数开头的 `NaN/Inf` 防护清零，初始化仍会执行。

这里不能根据源码断言实际硬件启动时一定出错；只能说明当前程序没有“第一帧加速度模长有效才允许初始化”的门控。要证明初始化可靠，必须记录第一帧 `ax/ay/az`、模长、`initialRoll/initialPitch` 和后续姿态输出。

还要结合 `Core/Src/main.c` 的调用顺序审查。当前 500Hz 循环先执行 `MargAHRSupdate(...)`，
然后才执行 `startup_delay_counter < 1000` 的启动延时逻辑。因此“等待 AHRS 收敛后再打开控制”保护的是
电机控制使能和 PID 状态清理，不是 `MargAHRSinit()` 的第一帧输入门控。

还要区分另一条边界：如果某一帧 `norm <= 0.1f`，源码只是跳过加速度归一化和 `KpAcc` 修正。四元数积分、四元数归一化和欧拉角输出仍然继续执行。换句话说，低模长不是“整帧姿态更新停止”，而是“本帧没有加速度重力反馈”。

### 8.4 四元数积分

第22章的加速度重力校正完成后，`gx/gy/gz` 已经被 `KpAcc` 修正。随后四元数增量由当前四元数和角速度共同决定：

```c
q0i = (-q1 * gx - q2 * gy - q3 * gz) * halfT;
q1i = ( q0 * gx + q2 * gz - q3 * gy) * halfT;
q2i = ( q0 * gy - q1 * gz + q3 * gx) * halfT;
q3i = ( q0 * gz + q1 * gy - q2 * gx) * halfT;
```

这里的 `halfT` 是 `dt * 0.5f`。如果 `dt500Hz` 异常，积分步长也会异常；当前 `MargAHRSupdate()` 内部没有对 `dt` 做上下限保护，也没有像 `gx/gy/gz`、`ax/ay/az` 那样先检查 `isnan(dt)` 或 `isinf(dt)`。时间基准可靠性需要由第26章的 500Hz 调度来保证。

这个边界还可以和控制链路形成对照。`pid.c` 的 `updatePID()` 会把异常 `deltaT` 回退为 `0.002f`，
`computeMotorCommands.c` 的部分分支也会先构造 `safeDt`。但 `MargAHRSupdate()` 在四元数积分前
没有类似判断，`halfT = dt * 0.5f` 会直接使用传入值。也就是说，同一个 `dt500Hz` 在姿态解算入口
和后续 PID 入口的保护级别不同；调试姿态跳变时，不能只看 PID 是否已经保护了 `deltaT`。

#### 8.4.1 halfT时间步长的构建证据边界

当前 Debug 构建中，`.map` 文件显示 `MargAHRSupdate` 进入 Flash，`halfT` 进入 RAM：
`MargAHRSupdate` 位于 `0x08003938`，`.bss.halfT` 位于 `0x20000d78`。`.list` 文件进一步显示，
`Core/Src/main.c` 的 500Hz 主循环存在到 `MargAHRSupdate()` 的 `bl 8003938 <MargAHRSupdate>` 调用，
而 `MargAHRSupdate()` 内部保留了 `halfT = dt * 0.5f`，后续 `q0i/q1i/q2i/q3i`
都使用这个 `halfT` 参与四元数增量计算。

这组证据能证明当前二进制没有把 `halfT` 优化成固定常量，也能证明四元数积分路径确实依赖函数入口传入的
`dt`。结合 `Core/Src/main.c`，该 `dt` 来自 `dt500Hz = deltaTime500Hz / 1000000.0f`，
并在同一轮 500Hz 循环中传给 `MargAHRSupdate(..., dt500Hz)` 和 `computeMotorCommands(dt500Hz)`。
但这些静态证据不能证明 `deltaTime500Hz` 在实机上总是 2000 微秒附近，也不能证明中断延迟、断点暂停或调试打印
不会拉大单帧时间步长。因此“姿态积分使用传入时间步长”可以由源码和构建产物证明；
“传入时间步长在运行时始终稳定可靠”仍然保持【待验证】。

#### 8.4.2 safeDt与deltaT的保护差异边界

同一个 `dt500Hz` 在当前链路里有三种不同语义。第一层是 `MargAHRSupdate(..., dt500Hz)`：
姿态积分内部直接执行 `halfT = dt * 0.5f`，没有先对 `dt` 做范围检查或 `NaN/Inf` 回退。
第二层是 `computeMotorCommands(dt500Hz)`：Roll 分支会把入口 `dt` 转成 `safeDt`，
并在 `safeDt > 0.01f`、`safeDt < 0.0001f`、`isnan(safeDt)` 或 `isinf(safeDt)` 时回退到
`0.002f`；但 Pitch 和 Yaw 分支调用 `updatePID()` 时仍直接传入 `dt`。第三层是
`pid.c` 的 `updatePID()`：它接收 `deltaT` 后再次做范围与非有限值检查，异常时回退到
`0.002f`。

`.list` 能把这三层边界落到当前构建：`MargAHRSupdate()` 中保留了 `halfT = dt * 0.5f`，
`computeMotorCommands()` 中保留了 `safeDt = dt` 和 `safeDt = 0.002f`，各轴后续都有
到 `updatePID` 的 `bl 8005334 <updatePID>` 调用，`updatePID()` 内部也保留了
`deltaT = 0.002f` 的回退路径。`.map` 只能证明 `MargAHRSupdate`、`computeMotorCommands`、
`updatePID` 和 `.bss.halfT` 进入当前镜像；它不能证明某一帧实际走了哪条回退路径。

因此，调试时间步长异常时不能只写“PID 已经保护 `deltaT`”。更准确的记录方式是同时保存
`dt500Hz`、AHRS 内部 `halfT`、Roll 分支 `safeDt`、传入 Pitch/Yaw 的 `dt`、`updatePID()`
内部实际使用的 `deltaT`。缺少这些运行数据时，姿态积分、控制限速和 PID 计算是否共享同一个安全时间步长，
都应继续保持【待验证】。

#### 8.4.3 每帧姿态更新的最小核对链

把 `MargAHRSupdate()` 拆成运行核对链，可以避免把“函数被调用”误写成“姿态估计一定可信”：

| 核对点 | 源码/构建证据 | 能证明什么 | 仍需运行记录证明什么 |
|---|---|---|---|
| 输入进入 AHRS | `main.c` 调用 `MargAHRSupdate(sensors.gyro500Hz[], sensors.accel500Hz[], 0,0,0,false, dt500Hz)`，`.list` 有对应 `bl` 调用 | 当前构建每帧把第21章缩放结果送入 AHRS 入口 | 本帧 I2C 读取是否成功、样本是否新鲜、输入方向是否实测正确 |
| 加速度比例修正 | `norm > 0.1f` 分支内计算 `exAcc/eyAcc/ezAcc`，并执行 `gx += exAcc * KpAcc` 等语句 | 加速度重力误差可修正角速度输入 | 本帧是否进入该分支，以及线性加速度是否污染重力方向 |
| 时间步长进入积分 | `halfT = dt * 0.5f`，`q0i..q3i` 都乘 `halfT` | 四元数增量依赖传入 `dt500Hz` | `dt500Hz` 是否稳定、是否出现断点暂停或打印阻塞导致的大步长 |
| 四元数状态更新 | `q0 += q0i` 到 `q3 += q3i`，`.map` 中 `q0..q3` 进入 RAM | 当前姿态状态对象会被本帧增量更新 | 更新前后四元数是否仍对应真实姿态 |
| 归一化与异常处理 | `norm_q < 1e-10f || isnan(norm_q)` 时重置，否则调用 `sqrtf(norm_q)` 归一化 | 当前构建包含四元数长度维护和 NaN 防护路径 | `norm_q` 是否越界、是否出现 `Inf`，以及重置是否造成控制输入跳变 |
| 欧拉角输出 | `atan2f/asinf/atan2f` 写入 `sensors.margAttitude500Hz[]`，下游控制读取该数组 | 当前控制链路读取的是 AHRS 输出的欧拉角 | Roll/Pitch/Yaw 方向、幅值、延迟和连续性是否经过实测验证 |

这张表的边界是：它能把第22章输入来源、第23章四元数更新和第27章控制入口接成一条源码证据链；但每一帧的真实可信度仍取决于运行日志、断点记录、DWT 时间、姿态动作记录和闭环观察。

可以用一个最小算例检查公式方向。假设当前四元数是单位姿态：

```text
q = [1, 0, 0, 0]
```

并且只有很小的 Yaw 角速度 `gz`，`gx = gy = 0`。代入代码：

```text
q0i = 0
q1i = 0
q2i = 0
q3i = gz * dt / 2
```

积分后 `q3` 会变为一个很小的正数。再代入本章第8.6节的 Yaw 输出公式，在小角度近似下：

```text
Yaw ~= 2 * q3 ~= gz * dt
```

这和“角速度乘以时间得到小角度增量”的直觉一致。这个算例只能验证项目公式的局部方向；
实际三轴耦合、安装方向和符号仍需结合第20章以及仓库外运动测试。

### 8.5 四元数归一化

更新后，代码计算：

```c
float norm_q = q0*q0 + q1*q1 + q2*q2 + q3*q3;
```

如果 `norm_q < 1e-10f` 或者 `norm_q` 是 `NaN`，项目把四元数重置为单位姿态：

```c
q0 = 1.0f;
q1 = q2 = q3 = 0.0f;
```

否则把四个分量除以长度。这个处理可以抑制数值漂移，是四元数积分后必须保留的工程步骤。
但这个保护也有代价：重置为 `[1,0,0,0]` 相当于把内部姿态状态拉回零姿态，而不是恢复到
“上一帧可信姿态”。它可以阻断 `NaN` 继续传播，却可能在控制输入中表现为一次姿态突变。
因此如果调试时看到 `norm_q < 1e-10f` 或 `isnan(norm_q)` 触发，结论应写成“姿态状态已异常并被重置”，
不能写成“姿态估计已自动修复且仍然可信”。

还要注意当前判断没有显式检查 `isinf(norm_q)`。如果异常 `dt` 或异常四元数分量让 `norm_q` 变成正无穷，代码会进入 `else` 分支并执行 `normR = 1.0f / sqrtf(norm_q)`。在 IEEE 浮点语义下，这可能把 `normR` 变成 0；若某个四元数分量已经是无穷，再执行 `q *= normR` 可能继续产生非有限值。因此调试四元数异常时，不能只记录 `isnan(norm_q)`，还应记录 `isinf(norm_q)` 和归一化后的 `q0/q1/q2/q3`。

#### 8.5.1 归一化异常分支的构建证据边界

当前 Debug 构建的 `.map` 文件显示，`q0`、`q1`、`q2`、`q3` 以及
`q0q0` 到 `q3q3` 的乘积缓存都进入 RAM 布局，`sqrtf` 也从 `libm.a`
进入最终镜像。`.list` 文件进一步显示，`MargAHRSupdate()` 中先计算
`norm_q = q0*q0 + q1*q1 + q2*q2 + q3*q3`，随后存在
`if (norm_q < 1e-10f || isnan(norm_q))` 的重置分支；未进入重置分支时，
反汇编保留了到 `sqrtf` 的调用，并继续执行 `q0/q1/q2/q3 *= normR`。

因此，`.map/.list` 可以证明当前二进制包含“异常重置”和“按模长重缩放”两条路径，
也能证明四元数归一化不是只停留在源码注释层面。但它们不能证明运行时 `norm_q`
总能落在安全范围内，也不能证明 `isinf(norm_q)` 不会发生。教材中只能把这部分写成
“当前实现有 `norm_q < 1e-10f` 与 `NaN` 防护，且构建产物证明该路径入链”，
不能写成“四元数状态在所有异常输入下都会自动恢复可信”。实机验证仍需要记录归一化前后的
`q0/q1/q2/q3`、`norm_q`、`isnan(norm_q)` 和 `isinf(norm_q)`。

### 8.6 欧拉角输出

归一化后，项目更新乘积缓存 `q0q0/q0q1/.../q3q3`，再计算：

```c
sensors.margAttitude500Hz[ROLL ] = atan2f(...);
sensors.margAttitude500Hz[PITCH] = -asinf(...);
sensors.margAttitude500Hz[YAW  ] = atan2f(...);
```

输出单位是弧度。后续控制模块直接读取这些弧度值，例如 `computeMotorCommands.c` 中的：

```c
float pitch_angle = PITCH_SENSOR_SIGN * sensors.margAttitude500Hz[PITCH];
float yaw_angle = sensors.margAttitude500Hz[YAW];
```

#### 8.6.1 三轴输出函数差异的构建证据边界

当前源码中，Roll 和 Yaw 都用 `atan2f(y, x)` 形式输出，Pitch 则用
`-asinf(2.0f * (q1q3 - q0q2))` 输出。`.map` 文件能证明 `atan2f`
和 `asinf` 都由 `Drivers/SRC/Src/MargAHRS.o` 引入并进入最终镜像；
`.list` 文件进一步显示，`MargAHRSupdate()` 末尾先后存在 Roll 的
`bl 801367c <atan2f>`、Pitch 的 `bl 801363c <asinf>`、Yaw 的
`bl 801367c <atan2f>`，随后才进入三轴输出的 `isnan()` 清零检查。

这组证据说明当前构建里三轴欧拉角不是同一种数学函数路径：Roll/Yaw 的输出由两个中间量共同决定，
Pitch 的输出由一个 `pitch_arg` 决定，因此 Pitch 会额外受到 `asinf` 输入域的约束。
但 `.map/.list` 不能证明安装方向、符号约定和欧拉角序列在实机运动中都与教材推导完全一致，
也不能证明 NaN 清零不会表现为控制输入突变。因此本节只能把“Roll/Yaw 使用 `atan2f`，
Pitch 使用 `asinf`，并且 NaN 清零发生在函数调用之后”写成构建结论；姿态输出的连续性、
符号方向和控制影响仍需要运动记录或断点数据验证。

如果 `asinf()` 或 `atan2f()` 结果变成 `NaN`，项目会把对应姿态角置为 0。这里没有对 `asinf()` 的输入先做区间钳位，因此极端数值误差导致 Pitch 清零的可能性需要在第32章继续关注。

Pitch 这一项尤其值得单独看：

```text
pitch_arg = 2 * (q1q3 - q0q2)
Pitch = -asin(pitch_arg)
```

理论上，只要 `q` 是单位四元数，`pitch_arg` 应位于 `[-1, 1]`。但如果输入角速度异常、
`dt` 异常、四元数归一化前后出现数值问题，`pitch_arg` 有可能略微越界。
更稳妥的工程写法通常会先把 `pitch_arg` 钳位到 `[-1, 1]` 再调用 `asinf()`。
当前项目没有这一步，而是在结果为 `NaN` 后把 Pitch 置 0。这个处理能避免 NaN 继续传播，
但也可能把一次数值异常表现为姿态角突变，具体影响需要运行日志或断点记录验证。

同样，虽然文件顶部定义了 Pitch 输出限制和万向节锁阈值相关宏，当前欧拉角输出路径没有调用这些宏。若后续要加入 Pitch 限幅或奇异姿态保护，应先回到源码和知识库补充证据，再写入教材主线。

#### 8.6.2 Pitch反三角输入域的构建证据边界

`Debug/Three-axis_cloud_platformV2.list` 能把上面的源码路径落到当前 Debug 构建里：在
`MargAHRSupdate()` 末尾，反汇编先计算 `q1q3 - q0q2`，再把该值加到自身形成
`2 * (q1q3 - q0q2)`，随后调用 `asinf`，再通过异或符号位得到负号并写入
`sensors.margAttitude500Hz[PITCH]`。紧接着，`.list` 又显示代码读取 Pitch 输出并调用
`__aeabi_fcmpun` 执行 `isnan()` 判断，若比较结果表示无序，也就是出现 `NaN`，就把 Pitch
写回 `0.0f`。

这条构建证据可以证明三件事：

- 当前固件确实链接并调用了 `asinf`，而不是只在源码里存在公式。
- 当前固件的 NaN 处理发生在 `asinf` 输出之后。
- 当前固件没有在 `asinf` 调用之前插入 `pitch_arg` 钳位或 `PITCH_OUTPUT_LIMIT_RAD` 限幅逻辑。

但它不能证明 `pitch_arg` 在实际运行中永远位于 `[-1, 1]`。`.map/.list` 只能说明当前二进制包含了哪些指令和调用路径，不能替代对 `dt500Hz`、角速度输入、四元数归一化前后数值和第一帧初始化状态的运行记录。因此本章只能把“Pitch 输入域目前缺少预钳位，NaN 后清零能阻断传播”写成源码与构建产物结论，不能写成“Pitch 输出在所有姿态下都连续可信”。

调试这条边界时，记录表至少应包含 `pitch_arg`、`norm_q`、归一化后的 `q0/q1/q2/q3`、
`asinf` 输出是否为 `NaN`、Pitch 是否被清零、`dt500Hz`、`gx/gy/gz` 和
`MargAHRSinitialized`。如果缺少这些运行数据，Pitch 越界是否会在实机上发生、发生后对控制输入的影响，都继续保持【待验证】。

#### 8.6.3 NaN清零进入控制链路的构建证据边界

还要把 AHRS 输出端和控制入口连起来看。`MargAHRSupdate()` 在欧拉角计算后只检查
`isnan(sensors.margAttitude500Hz[ROLL/PITCH/YAW])`，若结果为 `NaN` 就把对应角写成
`0.0f`。随后 `Core/Src/main.c` 的 500Hz 主循环调用 `computeMotorCommands(dt500Hz)`。
在 `computeMotorCommands.c` 中，Roll、Pitch、Yaw 又分别读取
`sensors.margAttitude500Hz[ROLL/PITCH/YAW]` 作为控制输入，并在各自分支内再次检查
`isnan()` 或 `isinf()`。

这带来一个容易漏掉的边界：如果 `MargAHRSupdate()` 已经把某个 NaN 姿态角清成 `0.0f`，
那么下游 `computeMotorCommands()` 看到的是一个有限数值，而不是原始 NaN。以 Pitch 为例，
`computeMotorCommands.c` 里非有限 `pitch_angle` 的回退值是 `1.5f`，但 AHRS 上游清零后，
Pitch 分支不会因为“非有限输入”进入这个回退路径，而是会把 0 弧度当作当前姿态参与
`wrapToPif(target_angle - pitch_angle)` 和后续 `updatePID()`。

`.map` 能证明 `computeMotorCommands` 和 `updatePID` 进入当前镜像；`.list` 能证明
`main()` 中存在 `bl 80043b4 <computeMotorCommands>`，并能定位 Roll、Pitch、Yaw
三轴从 `sensors.margAttitude500Hz[]` 读入控制分支的路径。但这些证据仍然不能证明清零后的控制响应
是否安全，只能证明“清零值会沿当前构建链路继续进入控制计算”。因此调试姿态 NaN 时，不能只记录
AHRS 是否清零，还应同时记录 `computeMotorCommands()` 入口看到的三轴角度、目标角、`error_mech`
和 `pidCmd[]`。

#### 8.6.4 wrapToPif与PID入口的构建证据边界

欧拉角进入控制链路后，还会遇到另一层“角度包裹”边界。`computeMotorCommands.c` 中的
`wrapToPif()` 只负责把角度包裹到 `[-PI, PI]` 附近；它不是姿态估计器，也不会重新判断
AHRS 输出是否可信。当前三轴进入 PID 的方式并不完全相同：Roll 分支先把
`roll_angle` 包裹，再计算 `roll_error_mech` 和电角度误差；Pitch 分支计算
`error_mech = wrapToPif(target_angle - pitch_angle)`，但随后 `updatePID()` 仍接收
`target_angle` 与 `pitch_angle`；Yaw 分支则先计算 `error_mech`，再构造
`safe_target = yaw_angle + error_mech` 传入 `updatePID()`。

`.map` 能证明 `wrapToPif` 和 `updatePID` 都进入当前 Debug 镜像；`.list` 能证明
Roll、Pitch、Yaw 分支都存在到 `wrapToPif` 的 `bl 80042f4 <wrapToPif>` 调用，
也能证明各轴后续调用 `bl 8005334 <updatePID>`。这些构建产物可以证明“角度包裹和 PID
入口路径实际进入了当前固件”，但不能证明包裹后的误差一定符合云台机械最短路径，也不能证明三轴
不同入口写法在所有姿态突变下等价。因此教材只能把这部分写成“当前三轴控制入口的实现差异”，
不能把它外推成“三轴控制对异常姿态有完全一致的保护”。调试时应分别记录 Roll/Pitch/Yaw 的
`target_angle`、当前角、`error_mech`、传入 `updatePID()` 的 command/state 和最终 `pidCmd[]`。

### 8.7 evvgcCF不是当前主线

`evvgcCF.c` 中的 `initOrientation()` 和 `getOrientation()` 使用加速度角、陀螺仪积分和小比例校正形成互补滤波。它还写入 `sensors.evvgcCFAttitude500Hz`。

但是：

- `evvgcCF.h` 中 `initOrientation()` 声明被注释。
- 当前全工程没有发现 `getOrientation()` 的调用点。
- `Core/Src/main.c` 当前主线使用 `sensors.margAttitude500Hz`，不是 `sensors.evvgcCFAttitude500Hz`。

因此第24章会把它作为 `互补滤波` 备用方案讲解，本章只把它作为对照证据。

## 9. 调试方法

本节按统一调试结构组织：现象 -> 可能原因 -> 定位方法 -> 验证步骤 -> 解决方案 -> 经验总结。对第23章而言，核心是把 `MargAHRSupdate()` 主线调用、第一帧初始化、四元数积分、归一化异常、低加速度模长分支、欧拉角输出和备用互补滤波方案分开验证。

### 9.1 现象与可能原因

常见现象包括姿态角不更新、第一帧初始化姿态异常、`dt500Hz` 波动导致积分步长异常、四元数模长异常后姿态突变、`norm <= 0.1f` 时误以为姿态更新停止、Yaw 长期漂移，或把 `evvgcCF.c` 备用方案输出误当成当前主线。可能原因包括前序 `sensors.*500Hz` 输入无效、第一帧加速度不可信、`MargAHRSupdate()` 内部没有对 `dt` 做异常回退、四元数归一化只显式检查 `NaN`、当前没有磁力计航向修正，或备用互补滤波没有调用证据。

### 9.2 定位方法：当前主线和初始化边界

第一步，确认当前姿态主线。

- 在 `Core/Src/main.c` 的 `MargAHRSupdate()` 调用处设置断点。
- 观察 `sensors.gyro500Hz`、`sensors.accel500Hz` 和 `dt500Hz`。
- 单步进入 `Drivers/SRC/Src/MargAHRS.c`。

第二步，观察初始化边界。

- 复位后观察 `MargAHRSinitialized`。
- 第一次进入 `MargAHRSupdate()` 时应调用 `MargAHRSinit()`。
- 记录 `q0/q1/q2/q3` 的初始值，并结合静止姿态判断 Roll/Pitch 是否合理。

### 9.3 验证步骤：四元数积分和归一化

第三步，观察四元数积分。

- 在 `q0i/q1i/q2i/q3i` 计算处观察 `gx/gy/gz` 和 `halfT`。
- 确认 `dt500Hz` 接近 0.002 秒。
- 记录 `halfT` 是否等于传入 `dt500Hz * 0.5f`，并确认 `MargAHRSupdate()` 内部没有把异常 `dt` 回退为 0.002 秒，也没有对 `dt` 执行 `isnan/isinf` 清理。
- 对照 `updatePID()` 的 `deltaT` 保护，区分“姿态积分时间步长”和“PID 计算时间步长”的保护边界。
- 若 `dt500Hz` 过大或过小，应先回到 500Hz 调度链路排查。

第四步，观察四元数归一化。

- 观察 `norm_q`。
- 正常情况下它应接近 1。
- 如果触发重置为 `[1,0,0,0]`，说明姿态状态已经异常，需要检查输入数据和 `dt500Hz`。
- 如果 `norm_q` 不是 `NaN` 但已经是 `Inf`，当前源码不会进入重置分支，应继续观察归一化后的四元数分量是否仍为有限值。
- 重置发生后还应继续观察 `sensors.margAttitude500Hz` 和控制输出，确认是否出现姿态角突变。

### 9.4 验证步骤：低模长分支和欧拉角输出

第五步，观察低模长分支。

- 在 `norm = sqrt(ax*ax + ay*ay + az*az)` 后观察 `norm`。
- 如果 `norm <= 0.1f`，记录本帧是否跳过 `KpAcc` 重力修正。
- 继续观察 `q0i/q1i/q2i/q3i`，确认四元数积分仍会执行。
- 不要把“重力校正跳过”误写成“姿态更新停止”。

第六步，观察欧拉角输出。

- 观察 `sensors.margAttitude500Hz[ROLL/PITCH/YAW]`。
- 静止状态下 Roll/Pitch 应与安装姿态和第20章坐标变换一致。
- 当前项目没有外部航向观测，Yaw 长时间漂移属于预期风险，不应误判为单纯代码错误。

### 9.5 解决方案：备用方案和小角度手算边界

第七步，确认备用方案未进入主线。

- 搜索 `getOrientation(` 和 `sensors.evvgcCFAttitude500Hz` 的调用关系。
- 如果当前没有调用点，就不要用 `evvgcCF.c` 的结果解释主线姿态输出。

第八步，做单轴小角度手算。

- 设 `q = [1,0,0,0]`、`gx = 0`、`gy = 0`、`gz` 为小量。
- 手算 `q3i = gz * dt / 2`。
- 再用 Yaw 输出公式验证 `Yaw ~= gz * dt`。
- 若调试器观察值方向相反，应优先检查轴定义、安装方向、传感器符号和控制侧符号。

### 9.6 解决方案：调试记录

- 记录 `MargAHRSupdate()` 调用点、输入三轴数据、`dt500Hz`、初始化状态和四元数初值。
- 初始化记录应包含第一帧 `ax/ay/az`、加速度模长、`initialRoll/initialPitch` 和 `MargAHRSinitialized` 置位时刻。
- 对姿态更新，应记录积分前后四元数、归一化模长、`isnan/isinf(norm_q)`、异常重置条件和欧拉角输出。
- Yaw 结论应分开记录源码积分结果、外部航向观测缺失和长期漂移风险。
- Roll/Pitch 方向正确性需要结合第20章坐标变换和仓库外运动测试，缺失时标记为【待验证】。

### 9.7 经验总结

第23章的调试结论应按“输入有效性 -> 初始化可信度 -> 积分步长 -> 四元数有限性 -> 欧拉角输出 -> 控制入口”逐层收敛。`.map/.list` 能证明 `MargAHRSupdate()`、`q0/q1/q2/q3` 和 `sensors.margAttitude500Hz[]` 写入路径进入构建；断点能证明某次运行的积分、归一化和欧拉角输出；Roll/Pitch/Yaw 的方向、幅值、延迟和长期稳定性仍需要第20章方向回查、受控单轴运动、DWT 计时和外部姿态基准。缺少这些记录时，不把“姿态主线存在”写成“姿态估计已经实测正确”。

## 10. 常见问题

问题一：为什么不直接用 Roll/Pitch/Yaw 做内部积分。

欧拉角直观，但角度周期和姿态奇异性会让积分和耦合处理更麻烦。
当前项目选择用四元数保存内部姿态，再输出欧拉角给控制链路。
这也是为什么本章先讲姿态表示，再讲第27章和第28章中的角度误差处理。

问题二：`wrapToPif()` 是否在 `MargAHRSupdate()` 中生效。

当前没有调用证据。`MargAHRS.c` 中定义了静态 `wrapToPif()`，但姿态更新和欧拉角输出没有调用它。角度归一化主要在后续控制链路中使用。

问题二补充：`standardRadianFormat()` 是否等价于 `wrapToPif()`。

不完全等价。`wrapToPif()` 使用循环，可以处理多圈输入；`standardRadianFormat()` 只加减一次
`2 * PI`，更适合已经接近一个周期范围内的误差。当前教材不能把二者写成完全相同的通用函数。

问题三：为什么 Pitch 使用 `-asinf()`。

这是项目当前姿态坐标约定的一部分，和第20章安装方向、控制轴符号一起决定最终正负号。这里必须按源码记录，不能凭通用公式改写。

问题四：Yaw 为什么可能漂移。

当前 `Core/Src/main.c` 传入磁力计为 `0, 0, 0, false`，`MargAHRSupdate()` 中也没有使用磁力计修正。Yaw 主要由陀螺仪积分得到，长期漂移是当前主线的已知风险。

问题四补充：源码注释写了 `Mahony/Madgwick`，能否把当前实现称为 Madgwick 滤波。

当前不能。仓库中没有看到 Madgwick 常见的 `beta` 参数或梯度下降修正项；当前函数体实际采用
加速度方向误差的比例反馈加四元数积分。名称和注释只能作为线索，不能替代源码证据。

问题五：`evvgcCF.c` 是否输出当前控制使用的姿态。

当前不是。当前控制读取 `sensors.margAttitude500Hz`，而 `evvgcCF.c` 写的是 `sensors.evvgcCFAttitude500Hz`，且当前没有调用证据。

问题六：四元数归一化和角度归一化是不是同一件事。

不是。四元数归一化是让四元数长度保持为 1；角度归一化是把弧度角包裹到 `[-PI, PI]`。二者都叫归一化，但对象和目的不同。

问题七：`PITCH_OUTPUT_LIMIT_RAD` 和 `GIMBAL_LOCK_COS_THRESH` 是否已经保护了姿态输出。

当前没有。它们在 `MargAHRS.c` 中有定义，但没有调用证据。当前 Pitch 输出只在结果为 `NaN` 时被置为 0，没有看到基于这两个宏的限幅或奇异姿态保护。

问题八：姿态角不是 `NaN`，是否就表示姿态估计可信。

不能这样判断。
非 `NaN` 只能说明数值格式没有进入明显异常。
姿态是否可信还取决于第20章的坐标方向、第21章的零偏与缩放、
第22章的加速度修正，以及陀螺仪漂移和外部运动条件。
当前仓库缺少仓库外实测证据，不能仅凭有限数值断言姿态正确。

问题九：`norm <= 0.1f` 时姿态更新是否完全停止。

不是。当前源码中，`norm > 0.1f` 只包住加速度归一化、重力误差计算和 `KpAcc` 修正。
四元数增量计算、四元数归一化和欧拉角输出仍在后续执行。
因此低模长帧更准确地说是“没有加速度重力反馈”，不是“没有姿态积分”。

问题十：第一帧加速度无效时，`MargAHRSinit()` 会等待下一帧吗。

当前不会。源码在 `MargAHRSinitialized == 0` 时直接调用 `MargAHRSinit()`，随后把 `MargAHRSinitialized` 置为 1。
后续没有自动重新初始化逻辑。
所以调试姿态初始化时，要把第一帧 `ax/ay/az` 和模长作为关键证据记录下来。

问题十一：`bgc32.h` 注释写了“卡尔曼姿态角”，能否把 `margAttitude500Hz` 当作 Kalman 输出？

当前不能。注释只能作为线索，不能替代写入路径证据。当前 `margAttitude500Hz` 的实际写入点在
`MargAHRS.c` 的欧拉角输出处，仓库中没有看到与该数组直接相关的 Kalman 状态协方差、
预测矩阵或观测更新。因此本章按当前 AHRS 主线姿态角讲解。

问题十二：PID 已经保护异常 `deltaT`，是否代表姿态积分也受同样保护？

不能这样外推。`updatePID()` 内部会把异常 `deltaT` 回退到 `0.002f`，但 `MargAHRSupdate()`
计算 `halfT = dt * 0.5f` 前没有同类上下限保护。因此一次异常 `dt500Hz` 可能已经影响四元数积分，
即使后续 PID 计算阶段对 `deltaT` 做了保护。

问题十三：启动延时是否能保护第一帧 `MargAHRSinit()`。

不能。`Core/Src/main.c` 中 `MargAHRSupdate()` 位于 `startup_delay_counter` 判断之前。
启动延时的作用是等待一段 500Hz 周期后再清 PID 状态并开启部分电机控制，不会阻止第一次
`MargAHRSupdate()` 立即调用 `MargAHRSinit()`。因此初始化可靠性仍要回到第一帧 `ax/ay/az`、
加速度模长和初始化后的 `q0/q1/q2/q3` 证据。

## 13. 本章总结

本章完成了从传感器物理量到姿态角输出的主线解释。

当前项目中：

- `角度归一化` 主要服务后续控制误差、目标角逼近和电角输出。
- `179°` 与 `-179°` 这类跨界角度必须按周期最短误差处理，否则控制误差会被放大。
- `computeMotorCommands.c` 的 `wrapToPif()` 和 `pid.c` 的 `standardRadianFormat()` 都与角度归一化有关，但处理能力和使用位置不同。
- `MargAHRSupdate()` 是当前 500Hz 主线的姿态更新入口。
- `.map/.list/.su/.cyclo` 的构建产物结论统一回到第7.1节判断：它们能证明四元数状态对象、500Hz 姿态更新入口、欧拉角输出写入路径和静态分析条目进入某次 Debug 构建，但不能替代运行时栈水位、真实耗时、分支覆盖、姿态方向、幅值、延迟或闭环稳定性实测。
- `MargAHRSinit()` 用第一帧加速度估计初始 Roll/Pitch，并把初始 Yaw 置为 0。
- 当前初始化发生在 `norm > 0.1f` 加速度模长判断之前，第一帧输入有效性需要单独验证。
- `startup_delay_counter` 位于 `MargAHRSupdate()` 之后，能推迟控制使能，但不能推迟第一次 AHRS 初始化。
- 当 `norm <= 0.1f` 时，当前源码跳过加速度重力反馈，但仍继续四元数积分和欧拉角输出。
- `MargAHRSupdate()` 用修正后的 `gx/gy/gz` 更新 `q0/q1/q2/q3`。
- 四元数积分公式来自 `q_dot = 0.5 * q * omega_q`，其中 `omega_q = [0,gx,gy,gz]`。
- 一阶欧拉积分会让四元数模长产生小量偏移，例如单轴小角速度下 `||q_temp||^2 = 1 + (gz * dt / 2)^2`。
- 四元数更新后必须做长度归一化，否则离散积分误差会积累。
- `MargAHRSupdate()` 内部没有对 `dt` 做异常回退，也没有对 `dt` 做 `NaN/Inf` 清理，不能把 `updatePID()` 的 `deltaT` 保护外推到姿态积分阶段。
- 四元数异常时重置为 `[1,0,0,0]` 能阻断异常传播，但不等于姿态估计仍然可信。
- 四元数归一化保护检查了 `norm_q < 1e-10f` 和 `isnan(norm_q)`，但没有显式检查 `isinf(norm_q)`；调试时应把 `Inf` 情况单独记录。
- `sensors.margAttitude500Hz[ROLL/PITCH/YAW]` 是当前主线输出给控制模块的姿态角。
- `bgc32.h` 中“卡尔曼姿态角”的注释与当前写入路径不完全一致，教材以 `MargAHRS.c` 实际写入逻辑为准。
- Pitch 输出的 `asinf()` 输入理论上应在 `[-1,1]`；`.list` 能证明当前构建先调用 `asinf`、再做 NaN 清零，但不能证明运行时输入域永远安全。
- `mx/my/mz`、`magDataUpdate`、`PITCH_OUTPUT_LIMIT_RAD` 和 `GIMBAL_LOCK_COS_THRESH` 在当前姿态主线中没有形成已生效的修正或保护。
- `main.c` 注释中的 `Mahony/Madgwick` 不能替代函数体证据；当前未看到 Madgwick 的 `beta` 或梯度下降修正路径。
- `evvgcCF.c` 是备用互补滤波方案，当前没有进入主线。

目标回扣：

- 本章已经把角度归一化、四元数姿态解算和欧拉角计算放回当前 500Hz 主线。
- 本章已经说明 `sensors.margAttitude500Hz` 是当前控制模块读取的姿态角来源。
- 本章只闭合软件姿态输出链路，不证明 Roll/Pitch/Yaw 方向和幅值已经通过仓库外运动测试。

本章保留以下边界：

- 不把 `MargAHRS.c` 中未调用的 `wrapToPif()` 写成姿态更新阶段已生效逻辑。
- 不把 `evvgcCF.c` 写成当前主线姿态来源。
- 不把第一帧初始化执行写成第一帧加速度一定可靠。
- 不把低模长时跳过重力反馈写成整帧姿态更新停止。
- 不把四元数 NaN 保护写成已经覆盖所有非有限数值情况。
- 不把 `.su/.cyclo` 静态分析结果写成运行时栈水位、真实耗时、分支覆盖或姿态正确性证明。
- 不提前展开 PID 控制和电机输出，只说明它们为什么依赖姿态角和角度归一化。

下一章将进入 `互补滤波备用方案`，专门解释 `evvgcCF.c` 为什么存在、它与当前 `四元数姿态解算` 的差异，以及如何判断备用姿态算法是否进入项目主线。
