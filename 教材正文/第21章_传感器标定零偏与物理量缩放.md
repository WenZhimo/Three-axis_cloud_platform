# 第21章 传感器标定、零偏与物理量缩放

## 1. 本章目标

第18章完成 MPU6050 寄存器初始化，第19章完成连续读取，第20章完成安装方向矩阵处理。到这里，项目已经能得到方向修正后的 `rawAccel/rawGyro`，但这些值仍然不是姿态解算可以直接使用的物理量。

本章目标是：

- 理解 `温度补偿标定` 解决的问题：传感器零偏会随温度变化。
- 理解 `静态零偏估计` 解决的问题：即使温度不变，静止传感器也可能存在固定偏置。
- 理解 `传感器物理量缩放` 解决的问题：把 MPU6050 原始 LSB 计数转换为加速度和角速度物理量。
- 区分当前源码中“已有函数”和“当前主线实际启用”的差别。
- 明确当前 500Hz 主循环中加速度和陀螺仪的偏置扣除、缩放顺序。
- 识别读取失败、零原始值、偏置扣除和物理量缩放之间的证据边界。
- 说明方向矩阵处理后的 `rawAccel/rawGyro` 仍然处在整数 LSB 计数层。

## 2. 前置知识

本章正式前置知识包括：

- `DWT周期计数器`
- `MPU6050寄存器配置`
- `MPU6050连续数据读取`

第20章的 `IMU安装方向矩阵` 不是本章在知识依赖图中的正式前置，但当前项目运行顺序中，方向矩阵处理发生在物理量缩放之前。因此读本章时应记住一个边界：`rawAccel/rawGyro` 已经不是寄存器刚读出的直接值，而是经过方向矩阵处理后的原始整数值。

这里的“方向处理后”不等于“已经变成物理量”。当前矩阵乘法接口和中间数组仍使用 `int16_t`，所以它更像轴交换、符号修正和整数层坐标变换。

本章不展开一阶低通滤波、加速度计重力校正、四元数姿态解算或 PID 控制。它只回答一个问题：项目如何把方向处理后的 MPU6050 原始整数值变成后续算法可用的输入。

## 3. 问题背景

MPU6050 输出的加速度和角速度是带符号整数。仅靠第19章的连续读取，项目只能得到类似这样的数据：

```text
rawAccel[XAXIS].value
rawAccel[YAXIS].value
rawAccel[ZAXIS].value
rawGyro[ROLL].value
rawGyro[PITCH].value
rawGyro[YAW].value
```

这些数据不能直接交给姿态解算，原因至少有三类。

第一，传感器存在零偏。静止不动时，陀螺仪理论角速度应接近 0，但实际可能有固定偏移。加速度计在静止时主要测到重力，也可能因为安装、器件误差和方向修正带来偏移。

第二，偏置会随温度变化。MPU6050 内部提供温度输出，项目中保留了用冷机和热机两组数据拟合温度偏置的函数。

第三，原始整数不是物理单位。加速度计量程为 ±4g 时，`8192 LSB` 对应约 `1g`；陀螺仪量程为 ±500 dps 时，`65.5 LSB` 对应 `1 deg/s`。项目必须把 LSB 转成后续算法使用的单位。

还有一个工程边界：原始数组为 0 并不能单独证明传感器处于零运动状态。当前读取失败路径会把 `rawAccel/rawGyro` 清零，随后主循环仍会执行偏置扣除和缩放。

## 4. 核心概念

### 4.1 温度补偿偏置

项目把温度补偿建模为一次线性函数：

```text
bias = slope * temperature + intercept
```

对应的配置字段位于 `eepromConfig_t`：

```c
accelTCBiasSlope[3]
accelTCBiasIntercept[3]
gyroTCBiasSlope[3]
gyroTCBiasIntercept[3]
```

运行时计算出的温度补偿偏置保存在：

```c
accelTCBias[3]
gyroTCBias[3]
```

这里的 `TC` 可以理解为 temperature compensation。它不是滤波后的传感器输出，而是准备从原始值中扣除的温度相关偏置。

### 4.2 静态零偏

静态零偏估计的基本思想是：让传感器保持静止，连续读取很多次，把结果求平均，作为当前启动状态下的固定偏置估计。

项目中 `MPU6050_ComputeStaticError()` 采样 5000 次。它累加加速度和陀螺仪数据，并最终写入：

```c
accelRTBias[3]
gyroRTBias[3]
```

需要特别注意当前主线的使用差异：

- `gyroRTBias` 会在 `Core/Src/main.c` 的 500Hz 主循环中扣除。
- `accelRTBias` 会被计算出来，但当前 500Hz 主循环没有从 `sensors.accel500Hz` 中扣除它。

因此教材不能笼统写成“静态零偏同时修正加速度和陀螺仪主线输入”。更准确地说：当前源码计算了加速度和陀螺仪静态偏置，其中陀螺仪静态偏置进入主循环缩放链路；加速度静态偏置的主线使用关系需要后续结合滤波和姿态章节继续核对。

### 4.3 物理量缩放

物理量缩放把原始整数转换为后续算法需要的浮点值。当前项目主要使用两个宏：

```c
#define ACCEL_SCALE_FACTOR 0.00119708f
#define GYRO_SCALE_FACTOR  0.00026646f
```

`ACCEL_SCALE_FACTOR` 对应 `(1 / 8192) * 9.8065`，适配 `ACCEL_FS_4G` 下的加速度计灵敏度。它把原始计数转换到接近 `m/s^2` 的量纲。

`GYRO_SCALE_FACTOR` 对应 `(1 / 65.5) * pi / 180`，适配 `GYRO_FS_500DPS` 下的陀螺仪灵敏度。它把原始计数转换为 `rad/s`。

当前项目缩放因子的来源可以拆成三层：

| 项目配置 | 官方灵敏度 | 项目缩放目标 | 项目宏 |
| --- | --- | --- | --- |
| `ACCEL_FS_4G` | `8192 LSB/g` | `LSB -> m/s^2` | `ACCEL_SCALE_FACTOR` |
| `GYRO_FS_500DPS` | `65.5 LSB/(deg/s)` | `LSB -> rad/s` | `GYRO_SCALE_FACTOR` |

加速度缩放推导为：

```text
1 LSB = (1 / 8192) g
1 g ~= 9.8065 m/s^2
ACCEL_SCALE_FACTOR = 9.8065 / 8192 ~= 0.00119708
```

陀螺仪缩放推导为：

```text
1 LSB = (1 / 65.5) deg/s
1 deg = pi / 180 rad
GYRO_SCALE_FACTOR = (1 / 65.5) * (pi / 180) ~= 0.00026646
```

这些推导依赖第18章确认的当前初始化实参。若未来把量程改成 `ACCEL_FS_2G` 或 `GYRO_FS_250DPS`，缩放因子也必须同步调整。

还要区分两套缩放入口：

- `Core/Src/main.c` 中的 500Hz 主循环使用 `ACCEL_SCALE_FACTOR` 和 `GYRO_SCALE_FACTOR`。
- `MPU6050_Get_Physical_Data()` 使用 `g_accel_sens` 和 `g_gyro_sens` 直接做除法。

这两套路径的单位目标不同。主循环把加速度转成接近 `m/s^2`，把陀螺仪转成 `rad/s`，
并且会扣除当前链路中的偏置项。`MPU6050_Get_Physical_Data()` 则把加速度除以 LSB/g 灵敏度，
得到接近 `g` 的值；把陀螺仪除以 LSB/(deg/s) 灵敏度，得到接近 `deg/s` 的值。
它没有扣除 `accelTCBias`、`gyroTCBias` 或 `gyroRTBias`，也不是当前 500Hz 主循环的实际入口。

### 4.4 偏置单位一致性

偏置扣除必须满足一个基本原则：被扣除的量和原始量必须在同一单位下。

当前项目中的几个偏置单位并不完全相同：

| 变量 | 当前生成位置 | 单位层级 | 当前主循环是否扣除 |
| --- | --- | --- | --- |
| `accelTCBias` | `MPU6050_ComputeTemError()` | 原始 LSB 计数 | 是 |
| `gyroTCBias` | `MPU6050_ComputeTemError()` | 原始 LSB 计数 | 是 |
| `gyroRTBias` | `MPU6050_ComputeStaticError()` | 原始 LSB 计数 | 是 |
| `accelRTBias` | `MPU6050_ComputeStaticError()` | 已乘 `ACCEL_SCALE_FACTOR` | 否 |

这解释了为什么当前 `main.c` 可以直接从 `rawGyro` 中扣除 `gyroRTBias`，却没有直接从 `rawAccel` 中扣除 `accelRTBias`。
如果把已经缩放到物理单位的 `accelRTBias` 直接从原始 LSB 中扣除，就会发生单位混用。
若未来要让加速度静态零偏进入主线，应先决定它以 LSB 保存，还是以 `m/s^2` 保存，并让扣除位置与单位保持一致。

### 4.5 读取有效性边界

物理量缩放公式默认输入来自一次有效的 MPU6050 读取。当前 `MPU6050_Read_And_Process()` 在读取失败时会把 `rawAccel` 和 `rawGyro` 清零并返回，但函数本身不向 500Hz 主循环返回成功或失败状态。

因此，缩放链路中出现 `rawAccel == 0` 或 `rawGyro == 0` 时，需要区分两种情况：

| 现象 | 可能含义 | 需要的证据 |
| --- | --- | --- |
| 原始数组接近 0 | 传感器读数本身接近 0 | 读取成功状态、连续样本、静止条件 |
| 原始数组被清零 | I2C 读取失败后的保护路径 | `MPU_Read_Reg()` 返回值、断点或日志 |

如果读取失败后进入主循环，公式会变成：

```text
accel = (0 - accelTCBias) * ACCEL_SCALE_FACTOR
gyro  = (0 - gyroRTBias - gyroTCBias) * GYRO_SCALE_FACTOR
```

这两个结果不一定等于物理零输入。本章只能证明源码存在该路径，不能证明实际运行中发生过该路径。

### 4.6 方向矩阵后的整数层

第20章介绍的方向矩阵发生在缩放之前。
当前矩阵乘法签名使用 `int16_t matrixC[]`、`int16_t matrixA[]` 和 `int16_t matrixB[]`。
中间数组 `straightAccelData` 和 `rotatedAccelData` 也都是 `int16_t`。

这意味着当前方向处理适合表达整数轴交换和符号翻转。它没有表达非整数安装误差、比例误差或软铁校正这类浮点校准模型。
所以本章看到的 `rawAccel/rawGyro`，应理解为“方向修正后的 LSB 计数”，而不是“校准完成后的物理量”。

## 5. 工作原理

当前项目的传感器处理顺序可以分成启动阶段和 500Hz 运行阶段。

启动阶段：

1. `Core/Src/main.c` 调用 `init_eepromConfig(true)`，把温度补偿斜率和截距初始化为 0。
2. `Core/Src/main.c` 使能 DWT 周期计数器，为微秒延时提供基础。
3. `Core/Src/main.c` 调用 `orientIMU()`，设置安装方向矩阵。
4. `Core/Src/main.c` 调用 `MPU6050_Init(ACCEL_FS_4G, GYRO_FS_500DPS, DLPF_CFG_260HZ)`。
5. `MPU6050_Init()` 根据量程参数设置内部灵敏度变量，并调用 `MPU6050_ComputeStaticError()`。
6. `MPU6050_ComputeStaticError()` 读取 5000 次数据，估计静态偏置。

当前主线中，`mpu6050Calibration()` 被注释掉，没有在开机流程中运行。因此，除非读者手动恢复这段调用或通过其他方式写入斜率和截距，否则温度补偿模型的配置值保持默认 0。

500Hz 运行阶段：

1. `MPU6050_Read_And_Process()` 连续读取 14 字节并执行方向矩阵处理。
2. `Core/Src/main.c` 用 `rawAccel` 扣除 `accelTCBias` 后乘 `ACCEL_SCALE_FACTOR`。
3. `Core/Src/main.c` 用 `rawGyro` 扣除 `gyroRTBias` 和 `gyroTCBias` 后乘 `GYRO_SCALE_FACTOR`。
4. 缩放后的结果写入 `sensors.accel500Hz` 和 `sensors.gyro500Hz`。
5. 后续 `MargAHRSupdate()` 使用这些缩放后的值。

这里的运行阶段还缺少一个状态量：`MPU6050_Read_And_Process()` 没有把本次读取是否成功传回主循环。主循环只能继续使用被该函数写入后的全局数组，所以调试时必须把“读取成功”与“缩放公式正确”分开验证。

温度补偿标定的数学结构是两点线性拟合。若两次温度采样得到：

```text
(T1, B1)
(T2, B2)
```

其中 `T` 是温度，`B` 是某一轴偏置，则斜率和截距为：

```text
slope = (B2 - B1) / (T2 - T1)
intercept = B2 - slope * T2
bias(T) = slope * T + intercept
```

源码中的 `tempDiff > 0.5f` 是一个防护条件：温差太小时不计算新斜率，避免除以过小数值放大噪声。
这不是 MPU6050 官方规则，而是项目源码中的工程保护阈值。
还要注意 `mpu6050Calibration()` 函数入口会先把四组温度补偿斜率和截距清零；如果后续 `tempDiff <= 0.5f`，函数只打印温差过小警告，不会恢复旧系数，也不会写入新的有效系数。因此手动启用温度标定时，必须同时记录温差是否越过阈值，否则一次失败标定可能让运行时温度补偿模型停留在零系数状态。

## 6. STM32实现机制

本章没有引入新的外设，但依赖第10章已经讲过的 DWT 周期计数器。

`Drivers/CustomDrivers/Src/mpu6050Calibration.c` 中的 `DWT_Delay_us()` 通过 `DWT->CYCCNT` 等待指定微秒数：

```c
uint32_t start = DWT->CYCCNT;
uint32_t ticks = us * (SystemCoreClock / 1000000);
while ((DWT->CYCCNT - start) < ticks);
```

`Drivers/CustomDrivers/Src/mpu6050.c` 中的 `MPU6050_ComputeStaticError()` 也调用 `DWT_Delay_us(1000)`，让 5000 次采样之间保持约 1ms 间隔。

因此，如果调试时发现静态零偏估计卡住，应先确认 `Core/Src/main.c` 中已经执行：

```c
CoreDebug->DEMCR |= CoreDebug_DEMCR_TRCENA_Msk;
DWT->CYCCNT = 0;
DWT->CTRL |= DWT_CTRL_CYCCNTENA_Msk;
```

当前主线中，这段 DWT 使能发生在 `MPU6050_Init()` 之前，满足静态零偏估计的运行前提。

启动性能上，`MPU6050_ComputeStaticError()` 至少包含 5000 次 `DWT_Delay_us(1000)`，仅延时下限约为：

```text
5000 * 1000 us = 5 s
```

这还不包含 5000 次 I2C 连续读取、温度补偿计算和循环开销。
因此，当前 `MPU6050_Init()` 不是一个瞬时初始化函数，它还包含一次启动标定流程。
如果调试时觉得开机等待较长，应先确认这段静态零偏估计是否正在执行。

### 7.1 构建产物证据边界

除了源码调用点，本章还可以用当前 Debug 构建产物核对“哪些符号和路径已经进入最终镜像”。这里的构建产物指：

- `Debug/Three-axis_cloud_platformV2.map`
- `Debug/Three-axis_cloud_platformV2.list`

`.map` 文件可以证明以下对象和函数被链接进当前最终镜像：

- `.data.accelTCBias`
- `.bss.gyroTCBias`
- `.bss.accelRTBias`
- `.bss.gyroRTBias`
- `.text.MPU6050_ComputeStaticError`
- `.text.MPU6050_ComputeTemError`
- `.text.DWT_Delay_us`

这说明温度补偿偏置数组、静态零偏数组、静态零偏估计函数、温度补偿计算函数和微秒延时函数不是只停留在源码文本中，而是已经进入当前 Debug 版本的可执行内容。

`.list` 文件可以进一步证明当前主线的指令路径：

- `MPU6050_Init()` 中存在对 `MPU6050_ComputeStaticError()` 的 `bl` 调用。
- `MPU6050_ComputeStaticError()` 中存在对 `rawAccel/rawGyro` 与 `accelTCBias/gyroTCBias` 的累加计算。
- `MPU6050_ComputeStaticError()` 末尾存在对 `gyroRTBias` 和 `accelRTBias` 的写入。
- `Core/Src/main.c` 的 500Hz 主循环中，`sensors.accel500Hz` 按 `(rawAccel - accelTCBias) * ACCEL_SCALE_FACTOR` 写入。
- `Core/Src/main.c` 的 500Hz 主循环中，`sensors.gyro500Hz` 按 `(rawGyro - gyroRTBias - gyroTCBias) * GYRO_SCALE_FACTOR` 写入。
- `Core/Src/main.c` 中的 `mpu6050Calibration()` 调用仍处于注释状态。

还要特别区分 `mpu6050Calibration.o` 的构建状态和 `mpu6050Calibration()` 的执行状态。`.map` 中可以看到 `LOAD ./Drivers/CustomDrivers/Src/mpu6050Calibration.o`，并且 `DWT_Delay_us` 以最终地址进入镜像；但 `.text.mpu6050Calibration` 位于 `0x00000000` 输入段，没有最终可执行地址。结合 `.list` 中 `Core/Src/main.c` 的注释调用，只能说温度标定源码文件参与了构建，其中被引用的 `DWT_Delay_us` 被保留；不能说 `mpu6050Calibration()` 当前已经进入开机主线。

`.su/.cyclo` 文件还能补充静态栈用量和圈复杂度边界：

- `MPU6050_ComputeStaticError()`：静态栈 72 字节，圈复杂度 3。
- `MPU6050_ComputeTemError()`：静态栈 16 字节，圈复杂度 1。
- `MPU6050_Get_Physical_Data()`：静态栈 24 字节，圈复杂度 1。
- `DWT_Delay_us()`：静态栈 24 字节，圈复杂度 2。
- `mpu6050Calibration()`：静态栈 96 字节，圈复杂度 7。
- `main()`：静态栈 56 字节，圈复杂度 7。

这里也要分层解释。`MPU6050_ComputeStaticError()` 和 `MPU6050_ComputeTemError()` 既有 `.map/.list`
最终链接与调用证据，也有 `.su/.cyclo` 静态分析证据；它们可以支撑本章对启动静态零偏和温度补偿计算路径的分析。
`MPU6050_Get_Physical_Data()` 与 `mpu6050Calibration()` 虽然也有 `.su/.cyclo` 条目，但当前主循环没有调用前者，
后者也没有最终可执行地址，所以这些数字只能说明相关源函数被编译器分析过，不能证明它们正在当前主线运行。
`main()` 的静态栈和圈复杂度则只能作为 500Hz 主循环所在函数的整体入口证据，不能单独拆成缩放公式本身的运行耗时。

这些构建产物不能证明以下内容：

- 5000 次静态采样期间传感器真实保持静止。
- I2C 每次读取都成功。
- 计算出的偏置值足够准确。
- `sensors.accel500Hz` 和 `sensors.gyro500Hz` 的物理单位精度已经通过实测闭环验证。
- `mpu6050Calibration()` 的斜率和截距会被持久化到 Flash/EEPROM 并在下次启动恢复。
- `.su/.cyclo` 中的静态栈和圈复杂度等于运行时最大栈水位、真实标定耗时或 500Hz 分支耗时。

因此，本章把 `.map/.list/.su/.cyclo` 作为“进入构建、调用路径和静态分析产物”的证据，而不是作为
“运行时标定质量、硬件姿态、真实耗时、栈水位或 EEPROM 持久化已经成立”的证据。后者仍需要断点记录、
串口日志、波形或仓库外实测数据支持，缺少证据时保持【待验证】。

### 7.2 官方资料、源码与构建产物分层

本章的缩放和标定结论不能只靠单一证据覆盖，适合拆成四层阅读：

- 官方资料层：TDK/InvenSense 手册给出量程对应的标称灵敏度和温度换算关系。例如在 `ACCEL_FS_4G` 下，加速度计灵敏度为 `8192 LSB/g`；在 `GYRO_FS_500DPS` 下，陀螺仪灵敏度为 `65.5 LSB/(deg/s)`；温度输出可按 `TEMP_OUT / 340 + 36.53` 换算为摄氏度。它能证明器件数据手册中的标称关系，不能单独证明当前工程确实使用这些配置。
- 源码配置层：`Drivers/CustomDrivers/Inc/mpu6050.h` 中的 `ACCEL_SCALE_FACTOR` 和 `GYRO_SCALE_FACTOR` 把上述标称灵敏度进一步换成项目使用的 `m/s^2` 与 `rad/s`；`TEMP_SENS` 与 `TEMP_ROOM_OFFSET` 则对应运行温度补偿公式。`Drivers/CustomDrivers/Src/mpu6050.c` 中 `MPU6050_ComputeTemError()` 使用温度公式计算 `accelTCBias/gyroTCBias`，`MPU6050_ComputeStaticError()` 计算 `accelRTBias/gyroRTBias`，`Core/Src/main.c` 的 500Hz 主循环再把 `rawAccel/rawGyro` 写入 `sensors.accel500Hz/sensors.gyro500Hz`。这一层能证明项目源码写了哪些公式和路径。
- 构建产物层：`.map` 能证明 `MPU6050_ComputeStaticError()`、`MPU6050_ComputeTemError()`、偏置数组和原始数据对象进入当前 Debug 镜像；`.list` 能证明 `MPU6050_Init()` 调用静态零偏估计、静态零偏估计调用读取和温度补偿、主循环执行加速度和陀螺仪缩放赋值；`.su/.cyclo` 只能补充静态栈用量和圈复杂度边界。这一层能证明“当前构建中存在并被引用”，不能证明“现场采样一定有效”。
- 运行硬件层：5000 次静态样本是否全部有效、传感器是否真正静止、I2C 是否持续成功、温度补偿参数是否来自可靠标定、结果是否持久化到下次启动、缩放后的物理量是否满足闭环控制精度，都需要运行日志、断点记录、波形、姿态输出或 Flash/EEPROM 写入读取证据。当前仓库没有这些证据时继续标为【待验证】。

因此，`8192`、`65.5`、`340` 和 `36.53` 属于官方标称公式证据；`ACCEL_SCALE_FACTOR`、`GYRO_SCALE_FACTOR`、`MPU6050_ComputeTemError()`、`MPU6050_ComputeStaticError()` 和 500Hz 主循环赋值属于源码与构建证据；静止条件、读取成功率、持久化和闭环物理精度属于运行硬件证据。把三者混在一起，会把“公式存在”和“系统已经测准”误写成同一件事。

## 8. 代码分析

### 8.1 config.c中的默认温度模型

`init_eepromConfig(true)` 中把温度补偿参数初始化为 0：

```c
eepromConfig.accelTCBiasSlope[XAXIS] = 0.0f;
eepromConfig.accelTCBiasIntercept[XAXIS] = 0.0f;
eepromConfig.gyroTCBiasSlope[ROLL] = 0.0f;
eepromConfig.gyroTCBiasIntercept[ROLL] = 0.0f;
```

其余轴也是同样的默认值。这意味着当前开机后，如果不运行温度标定、不加载非易失参数，温度补偿计算得到的偏置通常会是 0。

第17章已经说明，当前只验证到 `eepromConfig_t` 是全局配置结构，不能把它直接扩展为已经实现的 EEPROM 持久化系统。本章同样遵守这个边界。

还要注意 `Drivers/CustomDrivers/Src/mpu6050.c` 中全局变量初始化：

```c
float accelTCBias[3] = {1.0f, 0.0f, 0.0f};
```

这个初值不是 `eepromConfig` 默认值。当前启动路径中，`MPU6050_ComputeStaticError()` 每次循环都会调用 `MPU6050_ComputeTemError()`。
若 `eepromConfig` 中斜率和截距均为 0，`accelTCBias` 会被重新计算为 0。
因此分析当前主线时应以运行后的 `MPU6050_ComputeTemError()` 结果为准，不能只看全局初值。

### 8.2 mpu6050Calibration当前未进入主线

`Core/Src/main.c` 中成功初始化 MPU6050 后，温度校准调用处于注释状态：

```c
//mpu6050Calibration();
printf(">>> 已跳过温度校准，开始输出数据。\r\n\r\n");
```

因此，`mpu6050Calibration()` 是项目中存在的标定函数，但不是当前开机主线默认会执行的函数。

如果手动启用它，它会执行两轮采样：

- 第一轮采集冷机数据。
- 中间等待一段升温时间。
- 第二轮采集热机数据。
- 用两组温度和偏置计算斜率、截距。

源码中 `sampleRate = 1000` 的含义不是 1000Hz 采样率变量，而是传给 `DWT_Delay_us(sampleRate)`
的 1000 微秒延时。`numberOfSamples = 2000`，所以每一轮采样仅延时下限就是：

```text
2000 * 1000 us = 2 s
```

两轮采样合计约 4 秒延时下限，中间还有 `HAL_Delay(10000)`，也就是 10 秒等待。
再加上 4000 次 I2C 读取和计算开销，温度标定若被手动启用，启动阶段至少会增加十几秒。
这仍不是源码注释中写的 2 分钟。

温度标定开始时还会设置：

```c
mpu6050Calibrating = true;
```

第19章已经看到 `SysTick_Handler()` 只有在 `mpu6050Calibrating == false` 时才递增帧计数并置位
`frame_500Hz`。因此若温度标定真的进入主线，它会抑制 500Hz 帧触发，避免标定阶段和实时控制阶段同时读传感器。
当前主线跳过该函数，所以这个互斥行为只是源码潜在路径，不是当前开机默认现象。

源码中有以下需要记录的边界：

- `mpu6050Calibration()` 函数入口会先清零 `eepromConfig.accelTCBiasSlope/Intercept` 和 `gyroTCBiasSlope/Intercept`。如果后续 `tempDiff <= 0.5f`，新斜率和截距不会被计算，运行时结构体会保留这次清零后的值。
- 注释写着等待 2 分钟，但实际 `HAL_Delay(10000)` 是 10000ms，也就是约 10 秒。
- `mpu6050Calibration()` 中温度换算使用 `+35.0f`，而 `MPU6050_ComputeTemError()` 中使用 `+36.53f`。这两个偏移常数不一致，是否符合当前硬件和手册公式需要【待验证】。
- 冷机和热机阶段都把 `rawAccel[ZAXIS].value` 减去 `8192`，这默认了当前 Z 轴重力约等于 1g 且量程为 ±4g。
  这个处理是否与第20章 `case 10` 后的实际重力轴一致，需要静止姿态记录【待验证】。
- 函数计算出斜率和截距后只写入运行时 `eepromConfig` 结构；当前仓库没有看到随即写入 Flash/EEPROM 的持久化调用。
  因此不能把一次运行中的标定结果自动等同于下次上电仍然存在。

温度偏移常数不一致不只是“显示温度差 1.53°C”。两点拟合中，如果冷机和热机温度都统一加同一个常数，`tempDiff` 不变，所以斜率 `k` 不变；但截距 `b = bias - k*T` 会随温度零点移动。若标定阶段用 `T35 = raw/340 + 35.0` 计算截距，而运行阶段用 `T36 = raw/340 + 36.53` 代入，则同一原始温度下 `T36 = T35 + 1.53`，运行补偿会多出约 `k * 1.53` 的偏置量。只有在斜率接近 0 时，这个误差才近似可以忽略。

### 8.3 MPU6050_ComputeTemError的运行含义

`MPU6050_ComputeTemError()` 从 `rawMPU6050Temperature.value` 计算温度：

```c
float temp = ((float)rawMPU6050Temperature.value / 340.0f) + 36.53f;
```

这个公式与 MPU6050 手册中的温度换算形式一致：

```text
temperature_C = TEMP_OUT / 340 + 36.53
```

然后根据 `eepromConfig` 中的斜率和截距计算：

```c
accelTCBias[axis] = slope * temp + intercept;
gyroTCBias[axis]  = slope * temp + intercept;
```

当前 500Hz 主循环没有每帧调用 `MPU6050_ComputeTemError()`。它在 `MPU6050_ComputeStaticError()` 的 5000 次采样中被调用。

因此当前主线中，温度补偿偏置主要在启动静态估计阶段更新。进入 500Hz 主循环后，`accelTCBias/gyroTCBias` 不会在 `Core/Src/main.c` 中随每次新温度读取重新计算。

### 8.4 MPU6050_ComputeStaticError的采样逻辑

`MPU6050_Init()` 完成寄存器配置和灵敏度记录后调用：

```c
MPU6050_ComputeStaticError();
```

`MPU6050_ComputeStaticError()` 内部执行 5000 次：

```c
MPU6050_Read_And_Process();
MPU6050_ComputeTemError();
accelSum[axis] += rawAccel[axis].value - accelTCBias[axis];
gyroSum[axis]  += rawGyro[axis].value  - gyroTCBias[axis];
DWT_Delay_us(1000);
```

这说明静态零偏估计建立在第19章读取和第20章方向矩阵之后。它估计的是“方向处理后的原始整数值”的偏置，而不是寄存器直出坐标下的偏置。

采样结束后：

```c
accelRTBias[axis] = accelSum[axis] / 5000.0f * ACCEL_SCALE_FACTOR;
gyroRTBias[axis]  = gyroSum[axis]  / 5000.0f;
```

阅读源码时还会看到一段被 `/* ... */` 包住的旧写法，其中曾经把 `accelSum[axis]` 本身改写为缩放后的均值。那段不是当前执行路径。本章分析时只以上面未注释的赋值语句为准，这一点对判断 `accelOneG` 的单位层级很重要。

这里有一个单位差异：`accelRTBias` 被乘以 `ACCEL_SCALE_FACTOR`，更接近物理单位；`gyroRTBias` 保持原始计数单位，因为主循环会先用原始陀螺仪值扣除它，再乘 `GYRO_SCALE_FACTOR`。

当前主循环只实际扣除了 `gyroRTBias`，没有扣除 `accelRTBias`。所以 `accelRTBias` 虽然被计算，但当前主线用途并不等同于 `gyroRTBias`。

从单位一致性看，`gyroRTBias` 保留在原始 LSB 计数层，所以可以用于：

```text
(rawGyro - gyroRTBias - gyroTCBias) * GYRO_SCALE_FACTOR
```

`accelRTBias` 已经乘过 `ACCEL_SCALE_FACTOR`，如果要在主线中使用，更自然的位置应是缩放之后：

```text
((rawAccel - accelTCBias) * ACCEL_SCALE_FACTOR) - accelRTBias
```

当前源码没有执行这一步。本章只记录这个单位边界，不修改算法。

### 8.4.1 静态零偏单位层级的构建证据

把这一段和构建产物对照时，要区分“对象进入镜像”“公式进入镜像”和“现场样本有效”三件事。

当前源码可以写成两个求和层：

```text
S_accel[axis] = sum(rawAccel[axis] - accelTCBias[axis])
S_gyro[axis]  = sum(rawGyro[axis]  - gyroTCBias[axis])
```

`Drivers/CustomDrivers/Src/mpu6050.c` 在循环结束后写入：

```text
accelRTBias[axis] = S_accel[axis] / 5000 * ACCEL_SCALE_FACTOR
gyroRTBias[axis]  = S_gyro[axis]  / 5000
```

因此，`accelRTBias` 是“均值后再缩放”的物理量层，`gyroRTBias` 是“均值后的原始 LSB 计数层”。这一点不是从变量名推断出来的，而是由赋值语句中的 `ACCEL_SCALE_FACTOR` 是否参与计算决定的。

当前 Debug 构建产物能为这个边界提供静态证据：

| 证据位置 | 能证明的内容 | 不能证明的内容 |
|---|---|---|
| `Debug/Three-axis_cloud_platformV2.map` 中的 `.bss.accelRTBias`、`.bss.gyroRTBias` | 两个运行时零偏数组作为全局对象进入当前镜像 | 数组中的值来自真实静止有效样本 |
| `Debug/Three-axis_cloud_platformV2.list` 中 `accelSum += rawAccel - accelTCBias`、`gyroSum += rawGyro - gyroTCBias` | 静态估计求和路径进入当前构建 | 5000 次读取都成功，或采样期间云台确实静止 |
| `.list` 中 `accelRTBias = accelSum / 5000.0f * ACCEL_SCALE_FACTOR`、`gyroRTBias = gyroSum / 5000.0f` | 两个零偏数组处在不同单位层级 | 这种单位设计已经通过闭环实测验证 |
| `.list` 中主循环 `sensors.accel500Hz` 和 `sensors.gyro500Hz` 赋值 | 主线扣除了 `gyroRTBias`，没有扣除 `accelRTBias` | 加速度静态零偏不会影响未来其他保留路径 |

所以，本章可以把“`gyroRTBias` 在主循环中按 LSB 层扣除”写成当前构建事实；也可以把“`accelRTBias` 已经被计算但未进入 500Hz 加速度主线”写成源码与构建事实。至于静态估计的质量、读取失败比例、样本是否保持静止，以及这些偏置是否来自 EEPROM 恢复，都仍然需要运行时或硬件证据，当前只能保留为【待验证】。

### 8.5 方向矩阵后的整数计数层

`MPU6050_Read_And_Process()` 中的方向处理使用整数数组：

```c
int16_t straightAccelData[3];
int16_t rotatedAccelData[3];
int16_t straightGyroData[3];
int16_t rotatedGyroData[3];
```

矩阵乘法接口同样是整数层：

```c
void matrixMultiply(uint8_t aRows, uint8_t aCols_bRows, uint8_t bCols,
                    int16_t matrixC[], int16_t matrixA[], int16_t matrixB[]);
```

方向处理完成后，结果写回全局原始数组：

```c
rawAccel[axis].value = (int16_t)rotatedAccelData[axis];
rawGyro[axis].value  = (int16_t)rotatedGyroData[axis];
```

因此，第20章之后、第21章缩放之前的数据仍是整数 LSB 计数。源码中的 `(float)` 或 `(int16_t)` 类型转换不能证明这里已经进行了浮点物理校准。

这个实现的工程优点是简单、快、适合三轴安装方向的正交变换；代价是无法表达非整数角度的安装误差补偿。若未来要补偿小角度机械安装误差，需要重新设计矩阵数据类型和验证方法。

#### 8.5.1 偏置参数与方向矩阵编号绑定

第20章说明当前 `orientIMU()` 会强制选择 `case 10`。本章再补一层标定边界：温度补偿偏置、启动静态零偏和 500Hz 主循环缩放公式，都不是在 MPU6050 寄存器直出的芯片坐标上独立成立，而是在 `MPU6050_Read_And_Process()` 写回后的软件坐标上成立。

把读取和标定路径写成符号更容易看清。设 MPU6050 寄存器读出的三轴原始向量为 `v_raw`，方向矩阵为 `R`，方向处理后的原始计数为：

```text
v_sw = R * v_raw
```

当前静态估计和主循环使用的都是 `v_sw`。例如陀螺仪静态偏置估计可简化写成：

```text
b_gyro,R = mean(v_sw - b_TC,R)
         = mean(R * v_raw - b_TC,R)
```

500Hz 主循环再使用：

```text
gyro = (v_sw - b_gyro,R - b_TC,R) * GYRO_SCALE_FACTOR
```

这里下标 `R` 的含义是：这个偏置值属于某一个方向矩阵处理后的软件轴系。若未来把 `orientationMatrix` 从当前 `case 10` 改成其它方向编号，`rawAccel/rawGyro` 的轴含义和符号可能随之改变，旧的 `accelTCBias`、`gyroTCBias`、`gyroRTBias` 和调试记录就不能默认继续解释为同一轴的偏置。

| 参数或记录 | 当前绑定的轴系 | 直接复用到新方向矩阵的风险 |
|---|---|---|
| `accelTCBias[axis]` | `MPU6050_ComputeTemError()` 计算后，在方向处理后的 `rawAccel[axis]` 上扣除 | 轴交换或符号改变后，旧温度偏置可能扣到错误轴或错误符号 |
| `gyroTCBias[axis]` | 在方向处理后的 `rawGyro[ROLL/PITCH/YAW]` 上扣除 | 更换方向矩阵后，旧温度漂移斜率和截距不一定对应新软件轴 |
| `gyroRTBias[axis]` | `MPU6050_ComputeStaticError()` 对方向处理后的 `rawGyro` 求均值得到 | 新方向矩阵下直接扣除会把旧软件轴偏置当成新软件轴偏置 |
| `accelRTBias[axis]` | 当前由方向处理后的 `rawAccel` 求和并缩放得到，但未进入加速度主线 | 虽未进入当前主线，未来启用时仍必须确认方向编号一致 |
| 静止重力和单轴转动记录 | 记录的是当时 `orientationMatrix` 下的变量响应 | 方向编号改变后，旧记录只能作为历史对照，不能直接证明新配置正确 |

从纯数学上看，如果两个方向矩阵都是有效旋转矩阵，并且已知旧偏置确实是同一个物理偏置向量的表达，那么可以推导坐标变换关系：

```text
b_new = R_new * R_old^T * b_old
```

但当前仓库没有提供“历史偏置采集时的方向编号、有效样本、持久化记录和变换迁移流程”这一整套证据。因此教材不能建议直接把旧标定值按上式迁移到新方向矩阵；更稳妥的工程结论是：方向矩阵编号改变后，温度补偿、静态零偏和方向动作记录都应重新验证，缺少记录时保持【待验证】。

### 8.6 读取失败后的缩放边界

`MPU6050_Read_And_Process()` 的读取失败路径为：

```c
if (MPU_Read_Reg(REG_ACCEL_XOUT_H, I2C2_Buffer_Rx, MPU_DATA_LENGTH) != HAL_OK) {
    memset(rawAccel, 0, sizeof(rawAccel));
    memset(rawGyro, 0, sizeof(rawGyro));
    return;
}
```

但 500Hz 主循环在调用该函数后没有读取返回值，因为该函数返回类型是 `void`。所以主循环仍会执行：

```c
sensors.accel500Hz[XAXIS] =
    ((float)rawAccel[XAXIS].value - accelTCBias[XAXIS]) * ACCEL_SCALE_FACTOR;

sensors.gyro500Hz[ROLL] =
    ((float)rawGyro[ROLL].value - gyroRTBias[ROLL] - gyroTCBias[ROLL]) * GYRO_SCALE_FACTOR;
```

如果本次读取失败，`rawAccel/rawGyro` 被清零后再进入偏置扣除，输出可能是负的偏置补偿值，而不是真实的零加速度或零角速度。
因此，验证缩放正确性时不能只看 `sensors` 是否有数值，还要证明参与本次计算的原始样本来自有效 I2C 读取。

这个边界也影响启动静态估计。
`MPU6050_ComputeStaticError()` 循环内调用 `MPU6050_Read_And_Process()` 后继续累加当前 `rawAccel/rawGyro`。
如果采样期间存在读取失败，均值会混入失败路径产生的 0 值。
实际是否发生，需要断点、日志或 I2C 错误计数证明【待验证】。

这个影响可以用一个简化模型估算。假设静止时某一轴真实方向处理后原始值近似为 `r`，5000 次采样中有 `f` 次读取失败并被清零，则均值项大约变成：

```text
bias_est ~= ((5000 - f) * r + f * 0) / 5000
         = (1 - f/5000) * r
```

因此失败样本会把估计值拉向 0。对陀螺仪而言，`gyroRTBias` 偏小会让后续静止角速度仍残留一部分偏置；对加速度而言，虽然当前 `accelRTBias` 没进入主循环，但该估计仍会影响任何后续尝试使用它的路径。教材不能只写“采样 5000 次更稳定”，还必须说明 5000 次样本本身要有效。

### 8.7 accelOneG的边界

`MPU6050_ComputeStaticError()` 末尾还计算：

```c
accelOneG = sqrt(SQR(accelSum[XAXIS]) + SQR(accelSum[YAXIS]) + SQR(accelSum[ZAXIS]));
```

`accelOneG` 有一个容易被名字误导的单位生命周期。`Drivers/CustomDrivers/Src/mpu6050.c` 顶部把它初始化为：

```c
float accelOneG = 9.8065;
```

这个初值看起来像 `1g = 9.8065 m/s^2`。但 `MPU6050_ComputeStaticError()` 结束时又把它覆盖为 `accelSum` 的模长。此时它不再是 `m/s^2` 常量，而是 5000 次静态采样累计后的 LSB 计数量级。

`Drivers/SRC/Src/MargAHRS.c` 中确实存在一个保留函数会使用它：

```c
void calculateAccConfidence(float accMag)
{
    accMag /= accelOneG;
    ...
}
```

但当前全工程调用搜索只发现 `calculateAccConfidence()` 的定义，没有发现它被 `MargAHRSupdate()` 或 500Hz 主线调用。因此，本章只能说：`accelOneG` 被保留函数引用，但当前没有证据表明它影响主线姿态更新。

需要注意，当前源码这里使用的是累加量 `accelSum`，不是除以 5000 后的均值，也不是已经缩放后的三轴值。这个计算是否符合保留函数未来的归一化意图，需要在第22章和第23章结合 AHRS 代码继续验证，暂记为【待验证】边界。

可以做一个量级检查帮助读者发现问题边界。若静止时某一轴接近 `8192 LSB`，5000 次累加量级约为：

```text
5000 * 8192 = 40 960 000
```

如果要得到平均原始计数，应先除以 5000；如果要得到物理重力加速度，还应再乘
`ACCEL_SCALE_FACTOR`，结果才会接近 `9.8065 m/s^2`。
当前 `accelOneG` 直接对 `accelSum` 求模长，所以它的量级更接近“5000 次原始计数累积模长”，
不应在本章直接解释成真实 `1g` 物理量。

把这个关系写成公式更清楚。设静态采样次数 `N = 5000`，温度补偿后的某次加速度原始向量为 `a_lsb[n]`，则：

```text
S = sum(a_lsb[n])
accelOneG = ||S||
accelRTBias = (S / N) * ACCEL_SCALE_FACTOR
```

如果要把 `accelOneG` 转回平均重力物理量，应再执行：

```text
g_est = (accelOneG / N) * ACCEL_SCALE_FACTOR
```

在当前 `ACCEL_SCALE_FACTOR = 9.8065 / 8192` 下，若 `accelOneG ~= 40 960 000`，则 `g_est ~= 9.8065 m/s^2`。这说明 `accelOneG` 的数值本身不是 1g；只有经过“除以采样数、再乘缩放因子”后，才回到物理重力加速度量级。

因此，如果未来要把 `calculateAccConfidence(norm)` 接入 `MargAHRSupdate()`，还必须先统一单位。`MargAHRSupdate()` 当前的 `norm = sqrt(ax*ax + ay*ay + az*az)` 来自 `sensors.accel500Hz`，单位目标接近 `m/s^2`；而静态估计后的 `accelOneG` 是累计 LSB 模长。直接相除会把两个不同单位层级混在一起。

### 8.8 主循环中的物理量缩放

当前主线真正送入姿态解算的是 `Core/Src/main.c` 中的赋值：

```c
sensors.accel500Hz[XAXIS] =
    ((float)rawAccel[XAXIS].value - accelTCBias[XAXIS]) * ACCEL_SCALE_FACTOR;

sensors.gyro500Hz[ROLL] =
    ((float)rawGyro[ROLL].value - gyroRTBias[ROLL] - gyroTCBias[ROLL]) * GYRO_SCALE_FACTOR;
```

三轴加速度使用相同形式：扣除 `accelTCBias`，乘 `ACCEL_SCALE_FACTOR`。

三轴陀螺仪使用相同形式：扣除 `gyroRTBias` 和 `gyroTCBias`，乘 `GYRO_SCALE_FACTOR`。

这说明当前主线的实际处理顺序是：

```text
连续读取
-> 安装方向矩阵
-> 加速度扣除温度偏置，陀螺仪扣除静态偏置和温度偏置
-> 物理量缩放
-> sensors.accel500Hz / sensors.gyro500Hz
-> MargAHRSupdate()
```

但要再次强调：加速度主线只扣除了 `accelTCBias`，没有扣除 `accelRTBias`。

把三轴写入拆开看，当前 `sensors.accel500Hz` 的单位目标接近 `m/s^2`，`sensors.gyro500Hz` 的单位目标是 `rad/s`。
这与 `MargAHRSupdate()` 中陀螺仪积分需要弧度每秒的使用方式一致。
但加速度是否严格达到物理精度，还需要静止重力模长、单轴翻转和外部基准验证。

### 8.8.1 500Hz缩放链路的变量层级

为了避免把“变量名相似”误读成“单位层级相同”，可以把当前 500Hz 主线拆成下面几个可核查的变量层级。
这个表只描述 `Core/Src/main.c` 中实际写入 `sensors` 的路径；`Debug/Three-axis_cloud_platformV2.list` 中也能看到同一组赋值和随后对 `MargAHRSupdate()` 的调用。

| 变量或表达式 | 当前单位层级 | 源码依据 | 证据边界 |
| --- | --- | --- | --- |
| `rawAccel[axis].value` | 方向矩阵处理后的 `int16_t` 原始计数 | `MPU6050_Read_And_Process()` 读取 14 字节后执行方向矩阵，并把结果写回 `rawAccel` | 能证明缩放前已经经过软件方向变换；不能证明安装方向与真实机械坐标完全一致 |
| `accelTCBias[axis]` | 加速度原始计数层偏置 | `MPU6050_ComputeTemError()` 按温度模型写入，主循环公式在乘比例因子前扣除它 | 能证明当前公式扣除了温度补偿偏置；不能证明温度模型参数已经由实测标定 |
| `ACCEL_SCALE_FACTOR` | `LSB -> m/s^2` 的比例因子 | `mpu6050.h` 中定义为 `(1/8192) * 9.8065` | 能证明源码按 `ACCEL_FS_4G` 的标称灵敏度换算；不能证明传感器实际灵敏度误差为 0 |
| `sensors.accel500Hz[axis]` | 送入 AHRS 的加速度物理量层 | `((float)rawAccel[axis].value - accelTCBias[axis]) * ACCEL_SCALE_FACTOR` | 能证明当前加速度主线没有扣除 `accelRTBias`；不能证明该简化对姿态结果的影响大小 |
| `rawGyro[axis].value` | 方向矩阵处理后的 `int16_t` 原始计数 | `MPU6050_Read_And_Process()` 把 `rotatedGyroData` 写回 `rawGyro` | 能证明软件坐标变换进入当前镜像；不能证明陀螺仪实际轴向安装误差已经完全消除 |
| `gyroRTBias[axis]` 与 `gyroTCBias[axis]` | 陀螺仪原始计数层偏置 | `MPU6050_ComputeStaticError()` 写 `gyroRTBias`，`MPU6050_ComputeTemError()` 写 `gyroTCBias` | 能证明主循环在缩放前同时扣除两类陀螺仪偏置；不能证明静态样本全部有效或 EEPROM 已恢复历史标定 |
| `GYRO_SCALE_FACTOR` | `LSB -> rad/s` 的比例因子 | `mpu6050.h` 中定义为 `(1/65.5) * pi/180` | 能证明源码按 `GYRO_FS_500DPS` 的标称灵敏度换算；不能证明真实零偏和比例因子误差已经校准 |
| `sensors.gyro500Hz[axis]` | 送入 AHRS 的角速度物理量层 | `((float)rawGyro[axis].value - gyroRTBias[axis] - gyroTCBias[axis]) * GYRO_SCALE_FACTOR` | 能证明当前陀螺仪主线是“先扣 LSB 偏置、再换算 rad/s”；不能证明运行时角速度一定无漂移 |

这个拆分说明：`accelRTBias` 被计算并进入 `.map/.list` 证据链，但它不是当前 `sensors.accel500Hz` 公式的一部分；
`MPU6050_Get_Physical_Data()` 也进入了最终镜像，但不是当前 500Hz 主循环的缩放入口。
因此，教材可以把上表写成“当前构建事实”，但不能把它进一步写成“硬件已经标定正确”或“EEPROM 持久化已经生效”，后两者仍应保留为【待验证】。

### 8.9 MPU6050_Get_Physical_Data未进入当前缩放路径

`Drivers/CustomDrivers/Src/mpu6050.c` 中还存在：

```c
void MPU6050_Get_Physical_Data(float *ax, float *ay, float *az,
                               float *gx, float *gy, float *gz)
```

它使用 `g_accel_sens` 和 `g_gyro_sens` 把原始值除以灵敏度。这个函数可以帮助理解量程与灵敏度的关系，但当前 `Core/Src/main.c` 的 500Hz 主循环没有调用它。

教材在分析当前运行主线时，应以 `Core/Src/main.c` 中的缩放表达式为准。

两条路径的差异可以写成表格：

| 路径 | 加速度单位目标 | 陀螺仪单位目标 | 是否扣温度偏置 | 是否扣运行零偏 |
| --- | --- | --- | --- | --- |
| `MPU6050_Get_Physical_Data()` | `g` | `deg/s` | 否 | 否 |
| `main.c` 500Hz 主循环 | `m/s^2` | `rad/s` | 是 | 陀螺仪扣 `gyroRTBias` |

这解释了为什么不能把两个函数的输出直接拿来做数值相等比较。若要比较，应先统一单位和偏置处理。
例如 `MPU6050_Get_Physical_Data()` 的 `gx` 若为 `1 deg/s`，对应主循环的陀螺仪单位约为：

```text
1 deg/s * pi / 180 ~= 0.0174533 rad/s
```

而加速度 `1g` 对应主循环约为：

```text
1g * 9.8065 ~= 9.8065 m/s^2
```

此外，`g_accel_sens` 和 `g_gyro_sens` 由 `MPU6050_Init()` 根据传入量程更新。
如果跳过初始化或传入非法量程，默认值仍可能停留在 `ACCEL_SENS_2G` 和 `GYRO_SENS_250`。
当前主线已经调用 `MPU6050_Init(ACCEL_FS_4G, GYRO_FS_500DPS, ...)`，但这个函数不是当前 500Hz 缩放入口。

## 9. 调试方法

调试第21章内容时，建议按顺序观察。

第一步，确认 DWT 已启用。

- 在 `Core/Src/main.c` 的 DWT 使能代码后设置断点。
- 确认 `DWT->CTRL` 中计数器使能位已经打开。
- 再进入 `MPU6050_Init()`，避免 `DWT_Delay_us()` 等待异常。

第二步，确认当前是否运行温度标定。

- 查看 `Core/Src/main.c` 中 `mpu6050Calibration()` 是否仍为注释。
- 如果仍然注释，调试记录中应写明“当前主线跳过温度校准”。
- 不要把 `mpu6050Calibration.c` 中的算法存在误写成开机必然执行。
- 若手动启用该函数，应记录 `mpu6050Calibrating` 是否置为 `true`，以及 SysTick 是否因此停止置位 500Hz 帧。
- 同时记录两轮 2000 点采样、中间 10 秒等待、`tempDiff` 是否大于 `0.5f`，以及计算出的斜率/截距是否被持久化保存。

第三步，观察静态零偏估计。

- 在 `MPU6050_ComputeStaticError()` 入口设置断点。
- 观察 `samples` 从 0 到 4999 的循环。
- 观察 `gyroRTBias[ROLL/PITCH/YAW]` 在循环结束后的值。
- 同时记录 `accelRTBias`，但注意它当前没有在主循环加速度缩放中被扣除。

第四步，观察温度补偿偏置。

- 在 `MPU6050_ComputeTemError()` 中观察 `rawMPU6050Temperature.value`、`temp`、`accelTCBias` 和 `gyroTCBias`。
- 若 `eepromConfig` 中斜率和截距仍为 0，偏置计算结果通常也会是 0。
- 如果手动启用了温度标定，应记录冷机、热机温度和 `tempDiff`。

第五步，验证物理量缩放。

- 在 `Core/Src/main.c` 的 `sensors.accel500Hz` 和 `sensors.gyro500Hz` 赋值处设置断点。
- 用调试器读取 `rawAccel/rawGyro`、`accelTCBias/gyroTCBias`、`gyroRTBias`。
- 手算一轴结果，与 `sensors` 中的值对照。
- 不要把 `MPU6050_Get_Physical_Data()` 的输出直接和 `sensors` 比较；前者是 `g` / `deg/s`，
  后者目标是 `m/s^2` / `rad/s`，且偏置扣除也不同。

第六步，验证读取状态边界。

- 在 `MPU6050_Read_And_Process()` 中观察 `MPU_Read_Reg()` 返回值。
- 若返回失败，记录 `rawAccel/rawGyro` 被清零后的主循环计算结果。
- 不要把清零后的 `rawAccel/rawGyro` 解释为传感器真实静止。
- 若没有断点、日志或错误计数，读取失败是否发生保持【待验证】。

第七步，验证单位一致性。

- 确认 `gyroRTBias` 与 `rawGyro` 都是 LSB 计数后再相减。
- 确认 `accelRTBias` 已经是乘过 `ACCEL_SCALE_FACTOR` 的量。
- 不要把 `accelRTBias` 直接从 `rawAccel` 中扣除。
- 若尝试启用加速度静态偏置扣除，应先选择 LSB 层扣除或物理量层扣除，并写清验证方案。

第八步，观察启动耗时。

- 记录进入 `MPU6050_ComputeStaticError()` 和退出该函数的时间。
- 理论延时下限约 5 秒，实际还包括 I2C 读取和计算。
- 若启动耗时与预期差异较大，应检查 DWT、I2C 读取失败和 5000 次循环是否完整执行。

调试记录建议：

- 记录 DWT 启用状态、温度标定是否进入主线、静态零偏采样次数和偏置数组结果。
- 对缩放计算，应记录原始值、温度补偿项、运行时零偏项、比例系数和 `sensors` 写入值。
- 温度补偿记录应分开写算法存在、当前是否调用、斜率/截距是否为零和温度来源。
- 温度公式差异、`accelOneG` 计算意图和真实物理量精度缺少外部依据时保持【待验证】。
- 对 `accelOneG`，应分别记录初值、静态估计覆盖后的数值、单位层级、`calculateAccConfidence()` 是否被调用。
- 对单位记录，应明确每个偏置处于 LSB 层还是物理量层。
- 对单位对比，应明确当前值来自 `MPU6050_Get_Physical_Data()` 还是 `main.c` 500Hz 缩放路径。
- 对缩放验证，应至少手算一轴加速度和一轴陀螺仪，避免只看宏名判断单位正确。
- 对读取状态，应把 `HAL_OK`、失败清零路径和本次 `sensors` 计算结果放在同一张记录表中。
- 对方向矩阵，应记录矩阵输入、输出和最终 `rawAccel/rawGyro`，证明它仍是整数计数层。
- 对温度标定，应记录运行时参数是否写入 `eepromConfig`，以及是否存在 Flash/EEPROM 持久化证据。

## 10. 常见问题

问题一：当前项目开机时会自动执行温度校准吗。

当前不会。`main.c` 中 `mpu6050Calibration()` 被注释掉，并打印“已跳过温度校准”。温度校准函数存在，但不是当前主线默认执行路径。

问题二：既然温度校准被跳过，`MPU6050_ComputeTemError()` 还有意义吗。

它仍会在 `MPU6050_ComputeStaticError()` 中被调用。
但如果 `eepromConfig` 的斜率和截距都是默认 0，那么计算出的温度偏置通常仍为 0。
真正的温度补偿效果依赖有效的斜率和截距来源。
当前仓库没有证明这些系数来自已保存标定结果。

问题三：静态零偏估计是否修正了加速度和陀螺仪。

当前源码会计算 `accelRTBias` 和 `gyroRTBias`，但主循环只扣除 `gyroRTBias`。加速度主循环只扣除 `accelTCBias`。因此不能把两者说成完全相同地进入了主线。

问题四：`ACCEL_SCALE_FACTOR` 和 `GYRO_SCALE_FACTOR` 为什么和初始化量程有关。

因为项目初始化时使用 `ACCEL_FS_4G` 和 `GYRO_FS_500DPS`。这两个量程对应的灵敏度分别是 `8192 LSB/g` 和 `65.5 LSB/(deg/s)`，缩放因子正是围绕这两个值构造的。

如果初始化量程变了，而这两个缩放宏没有同步变化，物理量会整体按比例错误。
例如加速度从 ±4g 改为 ±2g 后，灵敏度会变成 `16384 LSB/g`，继续使用 `8192 LSB/g` 的缩放会把加速度估大约一倍。

问题五：为什么不直接使用 `MPU6050_Get_Physical_Data()`。

当前主循环没有调用这个函数，而是直接在 `Core/Src/main.c` 中用缩放宏写入 `sensors.accel500Hz` 和 `sensors.gyro500Hz`。教材分析当前主线时以实际调用路径为准。

问题六：温度公式为什么有两个偏移常数。

`mpu6050Calibration()` 中使用 `+35.0f`，`MPU6050_ComputeTemError()` 中使用 `+36.53f`。二者不一致，需要结合 MPU6050 手册或实测数据确认，当前标记为【待验证】。
从两点拟合角度看，统一平移温度轴不会改变斜率，但会改变截距。若标定和运行使用不同温度零点，运行补偿值会出现 `slope * 1.53` 量级的系统偏移。

问题七：静态零偏估计一定能得到正确零偏吗。

不能保证。
源码能证明 `MPU6050_ComputeStaticError()` 会采样并求均值，
但零偏质量依赖传感器静止、姿态稳定、温度条件和读取数据有效。
这些条件属于仓库外实测证据，当前教材只能提示读者记录并验证。
如果设备在采样期间移动，均值会把运动量也吸收到偏置估计中。

问题八：为什么 `accelRTBias` 已经计算出来却不直接扣除。

因为当前 `accelRTBias` 保存时已经乘过 `ACCEL_SCALE_FACTOR`，处在物理量层；
而 `main.c` 中加速度扣偏发生在原始 LSB 层。
源码没有把它放到缩放后再扣除，所以本章不能声称加速度静态偏置已经进入当前主线。

问题九：温度公式应该用 `+35.0f` 还是 `+36.53f`。

MPU6050 手册公式使用 `+36.53`。
当前 `MPU6050_ComputeTemError()` 也使用 `+36.53f`，但 `mpu6050Calibration()` 使用 `+35.0f`。
教材只能把不一致记录为源码边界；是否需要改动固件，需要硬件实测或明确维护决策。

问题十：`rawAccel/rawGyro` 为 0 是否表示传感器真实输入为 0。

不能直接这样判断。当前读取失败路径会把 `rawAccel/rawGyro` 清零，然后主循环继续执行偏置扣除和缩放。
因此 `raw=0` 至少可能表示读取失败后的保护值。只有同时证明 `MPU_Read_Reg()` 返回 `HAL_OK`，才能把该样本作为有效传感器读数继续分析。

问题十一：方向矩阵处理后是否已经完成物理校准。

没有。当前方向矩阵处理仍使用 `int16_t` 数组和整数矩阵乘法，结果写回 `rawAccel/rawGyro`。
它解决的是坐标方向问题，不解决量纲缩放、温度漂移、静态零偏质量或非整数安装误差补偿。

问题十二：`MPU6050_Get_Physical_Data()` 的输出能否直接和 `sensors.gyro500Hz` 比较。

不能直接比较。
`MPU6050_Get_Physical_Data()` 的陀螺仪结果接近 `deg/s`，而当前主循环的
`sensors.gyro500Hz` 目标是 `rad/s`。
前者也没有扣除 `gyroRTBias` 和 `gyroTCBias`。
若要比较，必须先统一偏置处理和角速度单位。

问题十三：温度标定算出的斜率和截距是否会自动保存到下次上电。

当前不能这样断言。
`mpu6050Calibration()` 会把结果写入运行时 `eepromConfig` 结构，但本章没有在仓库中看到该函数末尾
调用 Flash/EEPROM 写入接口。
因此当前只能证明运行时结构被更新，不能证明持久化已经完成。

问题十四：为什么 `accelOneG` 的量级需要单独验证。

因为当前代码对 5000 次 `accelSum` 的累加量直接求模长。
若静止重力轴约为 `8192 LSB`，累加量级约为 `40 960 000`，它不是平均值，也不是 `m/s^2`。
若要恢复到平均物理重力量级，需要 `(accelOneG / 5000) * ACCEL_SCALE_FACTOR`。
当前 `calculateAccConfidence()` 会引用 `accelOneG`，但没有调用证据；所以不能说它已经影响当前 AHRS 主线。

问题十四补充：为什么不能直接把 `calculateAccConfidence(norm)` 接到当前 AHRS。

因为当前 `MargAHRSupdate()` 中的 `norm` 来自 `sensors.accel500Hz`，单位目标接近 `m/s^2`。
静态估计后的 `accelOneG` 则是 5000 次 LSB 累加模长。
二者单位层级不同，直接相除会把物理量层和累计原始计数层混用。
如果未来要启用加速度置信度，应先决定 `accelOneG` 保存 `m/s^2`、平均 LSB，还是累计 LSB，并同步修改调用入口。

## 13. 本章总结

本章完成了从方向处理后的原始整数值到传感器物理输入的分析。

当前项目中：

- `mpu6050Calibration()` 提供冷机/热机两点温度标定，但在 `Core/Src/main.c` 中被注释掉。
- `init_eepromConfig(true)` 默认把温度补偿斜率和截距设为 0。
- `MPU6050_Init()` 会调用 `MPU6050_ComputeStaticError()`，启动阶段采样 5000 次估计静态偏置。
- `MPU6050_ComputeStaticError()` 会计算 `accelRTBias` 和 `gyroRTBias`，但当前主循环只扣除 `gyroRTBias`。
- 当前加速度缩放公式是 `(rawAccel - accelTCBias) * ACCEL_SCALE_FACTOR`。
- 当前陀螺仪缩放公式是 `(rawGyro - gyroRTBias - gyroTCBias) * GYRO_SCALE_FACTOR`。
- `ACCEL_SCALE_FACTOR` 来自 `9.8065 / 8192`，`GYRO_SCALE_FACTOR` 来自 `(pi / 180) / 65.5`。
- 官方资料能支撑 `8192 LSB/g`、`65.5 LSB/(deg/s)`、`TEMP_OUT / 340 + 36.53` 等标称换算关系；项目源码和 Debug 构建产物只能进一步证明当前工程写入并链接了这些缩放、温度补偿和静态零偏路径，不能替代运行实测。
- `gyroRTBias` 保持 LSB 层，可在原始陀螺仪值上扣除；`accelRTBias` 已处在物理量层，当前没有进入加速度主线。
- `MPU6050_Read_And_Process()` 读取失败时会清零 `rawAccel/rawGyro`，但主循环没有本次读取状态返回值。
- 方向矩阵处理后的 `rawAccel/rawGyro` 仍是整数 LSB 计数，不是浮点物理量。
- 温度补偿偏置、启动静态零偏和方向动作记录都绑定到当时的方向矩阵软件轴系；方向编号改变后不能默认复用旧标定值。
- `MPU6050_Get_Physical_Data()` 存在，但不是当前 500Hz 主循环的实际缩放入口。
- `MPU6050_Get_Physical_Data()` 与主循环缩放路径的单位体系不同，前者接近 `g` / `deg/s`，
  后者目标是 `m/s^2` / `rad/s`，且偏置扣除项不同。
- `mpu6050Calibration()` 若被手动启用，包含两轮 2000 点采样和一次 10 秒等待，并通过
  `mpu6050Calibrating` 抑制 500Hz 帧触发。
- 温度标定阶段和运行补偿阶段若使用不同温度偏移常数，斜率可能不变，但截距和最终补偿值会出现系统差异。
- 手动启用 `mpu6050Calibration()` 时，函数入口会先清零温度补偿系数；若温差不足未进入 `tempDiff > 0.5f` 分支，本次运行中的补偿系数会保持为 0。
- 静态零偏估计没有读取成功计数，失败清零样本会把均值拉向 0；实际是否发生需要运行证据。
- 温度标定结果当前只证明写入运行时 `eepromConfig`，没有证明已经持久化到下次上电。
- `accelOneG` 当前由 5000 次累加量直接求模长，量级不应直接解释成真实 `1g` 物理量。
- `accelOneG` 被 `calculateAccConfidence()` 引用，但该函数当前没有调用证据，不能写成 500Hz AHRS 主线已启用。
- 若要把 `accelOneG` 转回平均重力物理量，需要使用 `(accelOneG / 5000) * ACCEL_SCALE_FACTOR`。
- 当前 `.map/.list` 能证明偏置数组、静态估计函数、温度补偿函数、主循环缩放赋值和 `MPU6050_ComputeStaticError()` 调用已经进入 Debug 构建，但不能证明运行时标定质量、I2C 成功率或 EEPROM 持久化。
- 当前 `.su/.cyclo` 能补充静态零偏、温度补偿、保留物理量转换、保留温度标定和 `main()` 的静态栈与圈复杂度边界，
  但不能替代运行时栈水位、真实标定耗时或当前主线执行证据。

目标回扣：

- 本章已经区分温度补偿标定、静态零偏估计和物理量缩放三类问题。
- 本章已经说明哪些函数存在、哪些路径当前启用，以及 500Hz 主循环中的实际扣偏和缩放顺序。
- 本章已经补充读取失败、清零保护值和整数方向矩阵对缩放结论的影响。
- 本章只闭合传感器输入整理目标，不证明物理量精度已经通过仓库外实测。
- 本章不把 `.su/.cyclo` 静态分析结果写成最终可执行路径、运行耗时或栈水位证明。

本章保留以下待验证边界：

- 温度标定函数中的等待时间注释和实际 `HAL_Delay(10000)` 不一致。
- 温度换算中的 `+35.0f` 与 `+36.53f` 不一致，且可能通过截距影响运行补偿值。
- 温度标定入口清零系数后，若温差不足导致新系数没有写入，是否会覆盖原本可用的运行时补偿参数，需要现场启用路径和持久化策略证据。
- `accelOneG` 当前由累加量计算；它只进入未调用的 `calculateAccConfidence()` 保留函数，若未来启用需先统一单位。
- 冷机/热机温度差、传感器静止条件和缩放后物理量精度需要仓库外实测。
- I2C 读取失败是否在实际运行中出现，以及失败样本是否进入静态估计或 500Hz 主循环，需要运行证据。
- 当前整数方向矩阵是否足以覆盖真实机械安装误差，需要结合装配测量和姿态输出验证。
- 温度标定斜率/截距是否被保存并在下次启动加载，需要 Flash/EEPROM 写入与读取证据。
