# 第21章 传感器标定、零偏与物理量缩放

## 1. 本章目标

第18章完成 MPU6050 寄存器初始化，第19章完成连续读取，第20章完成安装方向矩阵处理。到这里，项目已经能得到方向修正后的 `rawAccel/rawGyro`，但这些值仍然不是姿态解算可以直接使用的物理量。

- 明确当前 500Hz 主循环中加速度和陀螺仪的偏置扣除、缩放顺序。

## 3. 问题背景

MPU6050 输出的加速度和角速度是带符号整数。仅靠第19章的连续读取，项目只能得到类似这样的数据：

```text
rawAccel[XAXIS].value
rawAccel[YAXIS].value
rawAccel[ZAXIS].value
rawGyro[ROLL].value
rawGyro[PITCH].value
rawGyro[YAW].value
```

这些数据不能直接交给姿态解算，原因有三类。

第一，传感器存在零偏。静止不动时，陀螺仪理论角速度应接近 0，但实际可能有固定偏移。加速度计在静止时主要测到重力，也可能因为安装、器件误差和方向修正带来偏移。

第二，偏置会随温度变化。MPU6050 内部提供温度输出，项目中保留了用冷机和热机两组数据拟合温度偏置的函数。

第三，原始整数不是物理单位。加速度计量程为 ±4g 时，`8192 LSB` 对应约 `1g`；陀螺仪量程为 ±500 dps 时，`65.5 LSB` 对应 `1 deg/s`。项目必须把 LSB 转成后续算法使用的单位。

## 4. 核心概念

### 4.1 温度补偿偏置

项目把温度补偿建模为一次线性函数：

```text
bias = slope * temperature + intercept
```

对应的配置字段位于 `eepromConfig_t`：

```c
accelTCBiasSlope[3]
accelTCBiasIntercept[3]
gyroTCBiasSlope[3]
gyroTCBiasIntercept[3]
```

运行时计算出的温度补偿偏置保存在：

```c
accelTCBias[3]
gyroTCBias[3]
```

这里的 `TC` 可以理解为 temperature compensation。它不是滤波后的传感器输出，而是准备从原始值中扣除的温度相关偏置。

### 4.2 静态零偏

静态零偏估计的基本思想是：让传感器保持静止，连续读取很多次，把结果求平均，作为当前启动状态下的固定偏置估计。

项目中 `MPU6050_ComputeStaticError()` 采样 5000 次。它累加加速度和陀螺仪数据，并最终写入：

```c
accelRTBias[3]
gyroRTBias[3]
```

需要特别注意当前主线的使用差异：

- `gyroRTBias` 会在 `Core/Src/main.c` 的 500Hz 主循环中扣除。
- `accelRTBias` 会被计算出来，但当前 500Hz 主循环没有从 `sensors.accel500Hz` 中扣除它。

因此教材不能笼统写成“静态零偏同时修正加速度和陀螺仪主线输入”。更准确地说：当前源码计算了加速度和陀螺仪静态偏置，其中陀螺仪静态偏置进入主循环缩放链路；加速度静态偏置的主线使用关系需要后续结合滤波和姿态章节继续核对。

### 4.3 物理量缩放

物理量缩放把原始整数转换为后续算法需要的浮点值。当前项目主要使用两个宏：

```c
#define ACCEL_SCALE_FACTOR 0.00119708f
#define GYRO_SCALE_FACTOR  0.00026646f
```

`ACCEL_SCALE_FACTOR` 对应 `(1 / 8192) * 9.8065`，适配 `ACCEL_FS_4G` 下的加速度计灵敏度。它把原始计数转换到接近 `m/s^2` 的量纲。

`GYRO_SCALE_FACTOR` 对应 `(1 / 65.5) * pi / 180`，适配 `GYRO_FS_500DPS` 下的陀螺仪灵敏度。它把原始计数转换为 `rad/s`。

## 5. 工作原理

当前项目的传感器处理顺序可以分成启动阶段和 500Hz 运行阶段。

启动阶段：

1. `Core/Src/main.c` 调用 `init_eepromConfig(true)`，把温度补偿斜率和截距初始化为 0。
2. `Core/Src/main.c` 使能 DWT 周期计数器，为微秒延时提供基础。
3. `Core/Src/main.c` 调用 `orientIMU()`，设置安装方向矩阵。
4. `Core/Src/main.c` 调用 `MPU6050_Init(ACCEL_FS_4G, GYRO_FS_500DPS, DLPF_CFG_260HZ)`。
5. `MPU6050_Init()` 根据量程参数设置内部灵敏度变量，并调用 `MPU6050_ComputeStaticError()`。
6. `MPU6050_ComputeStaticError()` 读取 5000 次数据，估计静态偏置。

当前主线中，`mpu6050Calibration()` 被注释掉，没有在开机流程中运行。因此，除非读者手动恢复这段调用或通过其他方式写入斜率和截距，否则温度补偿模型的配置值保持默认 0。

500Hz 运行阶段：

1. `MPU6050_Read_And_Process()` 连续读取 14 字节并执行方向矩阵处理。
2. `Core/Src/main.c` 用 `rawAccel` 扣除 `accelTCBias` 后乘 `ACCEL_SCALE_FACTOR`。
3. `Core/Src/main.c` 用 `rawGyro` 扣除 `gyroRTBias` 和 `gyroTCBias` 后乘 `GYRO_SCALE_FACTOR`。
4. 缩放后的结果写入 `sensors.accel500Hz` 和 `sensors.gyro500Hz`。
5. 后续 `MargAHRSupdate()` 使用这些缩放后的值。

## 6. STM32实现机制

本章没有引入新的外设，但依赖第10章已经讲过的 DWT 周期计数器。

`Drivers/CustomDrivers/Src/mpu6050Calibration.c` 中的 `DWT_Delay_us()` 通过 `DWT->CYCCNT` 等待指定微秒数：

```c
uint32_t start = DWT->CYCCNT;
uint32_t ticks = us * (SystemCoreClock / 1000000);
while ((DWT->CYCCNT - start) < ticks);
```

`Drivers/CustomDrivers/Src/mpu6050.c` 中的 `MPU6050_ComputeStaticError()` 也调用 `DWT_Delay_us(1000)`，让 5000 次采样之间保持约 1ms 间隔。

因此，如果调试时发现静态零偏估计卡住，应先确认 `Core/Src/main.c` 中已经执行：

```c
CoreDebug->DEMCR |= CoreDebug_DEMCR_TRCENA_Msk;
DWT->CYCCNT = 0;
DWT->CTRL |= DWT_CTRL_CYCCNTENA_Msk;
```

当前主线中，这段 DWT 使能发生在 `MPU6050_Init()` 之前，满足静态零偏估计的运行前提。

## 8. 代码分析

### 8.1 config.c中的默认温度模型

`init_eepromConfig(true)` 中把温度补偿参数初始化为 0：

```c
eepromConfig.accelTCBiasSlope[XAXIS] = 0.0f;
eepromConfig.accelTCBiasIntercept[XAXIS] = 0.0f;
eepromConfig.gyroTCBiasSlope[ROLL] = 0.0f;
eepromConfig.gyroTCBiasIntercept[ROLL] = 0.0f;
```

其余轴也是同样的默认值。这意味着当前开机后，如果不运行温度标定、不加载非易失参数，温度补偿计算得到的偏置通常会是 0。

第17章已经说明，当前只验证到 `eepromConfig_t` 是全局配置结构，不能把它直接扩展为已经实现的 EEPROM 持久化系统。本章同样遵守这个边界。

### 8.2 mpu6050Calibration当前未进入主线

`Core/Src/main.c` 中成功初始化 MPU6050 后，温度校准调用处于注释状态：

```c
//mpu6050Calibration();
printf(">>> 已跳过温度校准，开始输出数据。\r\n\r\n");
```

因此，`mpu6050Calibration()` 是项目中存在的标定函数，但不是当前开机主线默认会执行的函数。

如果手动启用它，它会执行两轮采样：

- 第一轮采集冷机数据。
- 中间等待一段升温时间。
- 第二轮采集热机数据。
- 用两组温度和偏置计算斜率、截距。

源码中有两个需要记录的边界：

- 注释写着等待 2 分钟，但实际 `HAL_Delay(10000)` 是 10000ms，也就是约 10 秒。
- `mpu6050Calibration()` 中温度换算使用 `+35.0f`，而 `MPU6050_ComputeTemError()` 中使用 `+36.53f`。这两个偏移常数不一致，是否符合当前硬件和手册公式需要【待验证】。

### 8.3 MPU6050_ComputeTemError的运行含义

`MPU6050_ComputeTemError()` 从 `rawMPU6050Temperature.value` 计算温度：

```c
float temp = ((float)rawMPU6050Temperature.value / 340.0f) + 36.53f;
```

然后根据 `eepromConfig` 中的斜率和截距计算：

```c
accelTCBias[axis] = slope * temp + intercept;
gyroTCBias[axis]  = slope * temp + intercept;
```

当前 500Hz 主循环没有每帧调用 `MPU6050_ComputeTemError()`。它在 `MPU6050_ComputeStaticError()` 的 5000 次采样中被调用。

因此当前主线中，温度补偿偏置主要在启动静态估计阶段更新。进入 500Hz 主循环后，`accelTCBias/gyroTCBias` 不会在 `Core/Src/main.c` 中随每次新温度读取重新计算。

### 8.4 MPU6050_ComputeStaticError的采样逻辑

`MPU6050_Init()` 完成寄存器配置和灵敏度记录后调用：

```c
MPU6050_ComputeStaticError();
```

`MPU6050_ComputeStaticError()` 内部执行 5000 次：

```c
MPU6050_Read_And_Process();
MPU6050_ComputeTemError();
accelSum[axis] += rawAccel[axis].value - accelTCBias[axis];
gyroSum[axis]  += rawGyro[axis].value  - gyroTCBias[axis];
DWT_Delay_us(1000);
```

这说明静态零偏估计建立在第19章读取和第20章方向矩阵之后。它估计的是“方向处理后的原始整数值”的偏置，而不是寄存器直出坐标下的偏置。

采样结束后：

```c
accelRTBias[axis] = accelSum[axis] / 5000.0f * ACCEL_SCALE_FACTOR;
gyroRTBias[axis]  = gyroSum[axis]  / 5000.0f;
```

这里有一个单位差异：`accelRTBias` 被乘以 `ACCEL_SCALE_FACTOR`，更接近物理单位；`gyroRTBias` 保持原始计数单位，因为主循环会先用原始陀螺仪值扣除它，再乘 `GYRO_SCALE_FACTOR`。

当前主循环只实际扣除了 `gyroRTBias`，没有扣除 `accelRTBias`。所以 `accelRTBias` 虽然被计算，但当前主线用途并不等同于 `gyroRTBias`。

### 8.5 accelOneG的边界

`MPU6050_ComputeStaticError()` 末尾还计算：

```c
accelOneG = sqrt(SQR(accelSum[XAXIS]) + SQR(accelSum[YAXIS]) + SQR(accelSum[ZAXIS]));
```

这个变量在后续 `Drivers/SRC/Src/MargAHRS.c` 中会被使用。本章只确认它在静态估计函数中被赋值，不展开姿态解算中的使用效果。

需要注意，当前源码这里使用的是累加量 `accelSum`，不是除以 5000 后的均值，也不是已经缩放后的三轴值。这个计算是否符合后续归一化意图，需要在第22章和第23章结合 AHRS 代码继续验证，暂记为【待验证】边界。

### 8.6 主循环中的物理量缩放

当前主线真正送入姿态解算的是 `Core/Src/main.c` 中的赋值：

```c
sensors.accel500Hz[XAXIS] =
    ((float)rawAccel[XAXIS].value - accelTCBias[XAXIS]) * ACCEL_SCALE_FACTOR;

sensors.gyro500Hz[ROLL] =
    ((float)rawGyro[ROLL].value - gyroRTBias[ROLL] - gyroTCBias[ROLL]) * GYRO_SCALE_FACTOR;
```

三轴加速度使用相同形式：扣除 `accelTCBias`，乘 `ACCEL_SCALE_FACTOR`。

三轴陀螺仪使用相同形式：扣除 `gyroRTBias` 和 `gyroTCBias`，乘 `GYRO_SCALE_FACTOR`。

这说明当前主线的实际处理顺序是：

```text
连续读取
-> 安装方向矩阵
-> 加速度扣除温度偏置，陀螺仪扣除静态偏置和温度偏置
-> 物理量缩放
-> sensors.accel500Hz / sensors.gyro500Hz
-> MargAHRSupdate()
```

但要再次强调：加速度主线只扣除了 `accelTCBias`，没有扣除 `accelRTBias`。

### 8.7 MPU6050_Get_Physical_Data未进入当前缩放路径

`Drivers/CustomDrivers/Src/mpu6050.c` 中还存在：

```c
void MPU6050_Get_Physical_Data(float *ax, float *ay, float *az,
                               float *gx, float *gy, float *gz)
```

它使用 `g_accel_sens` 和 `g_gyro_sens` 把原始值除以灵敏度。这个函数可以帮助理解量程与灵敏度的关系，但当前 `Core/Src/main.c` 的 500Hz 主循环没有调用它。

教材在分析当前运行主线时，应以 `Core/Src/main.c` 中的缩放表达式为准。

## 9. 调试方法

调试第21章内容时，建议按顺序观察。

第一步，确认 DWT 已启用。

- 在 `Core/Src/main.c` 的 DWT 使能代码后设置断点。
- 确认 `DWT->CTRL` 中计数器使能位已经打开。
- 再进入 `MPU6050_Init()`，避免 `DWT_Delay_us()` 等待异常。

第二步，确认当前是否运行温度标定。

- 查看 `Core/Src/main.c` 中 `mpu6050Calibration()` 是否仍为注释。
- 如果仍然注释，调试记录中应写明“当前主线跳过温度校准”。
- 不要把 `mpu6050Calibration.c` 中的算法存在误写成开机必然执行。

第三步，观察静态零偏估计。

- 在 `MPU6050_ComputeStaticError()` 入口设置断点。
- 观察 `samples` 从 0 到 4999 的循环。
- 观察 `gyroRTBias[ROLL/PITCH/YAW]` 在循环结束后的值。
- 同时记录 `accelRTBias`，但注意它当前没有在主循环加速度缩放中被扣除。

第四步，观察温度补偿偏置。

- 在 `MPU6050_ComputeTemError()` 中观察 `rawMPU6050Temperature.value`、`temp`、`accelTCBias` 和 `gyroTCBias`。
- 若 `eepromConfig` 中斜率和截距仍为 0，偏置计算结果通常也会是 0。
- 如果手动启用了温度标定，应记录冷机、热机温度和 `tempDiff`。

第五步，验证物理量缩放。

- 在 `Core/Src/main.c` 的 `sensors.accel500Hz` 和 `sensors.gyro500Hz` 赋值处设置断点。
- 用调试器读取 `rawAccel/rawGyro`、`accelTCBias/gyroTCBias`、`gyroRTBias`。
- 手算一轴结果，与 `sensors` 中的值对照。

调试记录建议：

- 记录 DWT 启用状态、温度标定是否进入主线、静态零偏采样次数和偏置数组结果。
- 对缩放计算，应记录原始值、温度补偿项、运行时零偏项、比例系数和 `sensors` 写入值。
- 温度补偿记录应分开写算法存在、当前是否调用、斜率/截距是否为零和温度来源。
- 温度公式差异、`accelOneG` 计算意图和真实物理量精度缺少外部依据时保持【待验证】。

## 10. 常见问题

问题一：当前项目开机时会自动执行温度校准吗。

当前不会。`main.c` 中 `mpu6050Calibration()` 被注释掉，并打印“已跳过温度校准”。温度校准函数存在，但不是当前主线默认执行路径。

问题二：既然温度校准被跳过，`MPU6050_ComputeTemError()` 还有意义吗。

它仍会在 `MPU6050_ComputeStaticError()` 中被调用。
但如果 `eepromConfig` 的斜率和截距都是默认 0，那么计算出的温度偏置通常仍为 0。
真正的温度补偿效果依赖有效的斜率和截距来源。
当前仓库没有证明这些系数来自已保存标定结果。

问题三：静态零偏估计是否修正了加速度和陀螺仪。

当前源码会计算 `accelRTBias` 和 `gyroRTBias`，但主循环只扣除 `gyroRTBias`。加速度主循环只扣除 `accelTCBias`。因此不能把两者说成完全相同地进入了主线。

问题四：`ACCEL_SCALE_FACTOR` 和 `GYRO_SCALE_FACTOR` 为什么和初始化量程有关。

因为项目初始化时使用 `ACCEL_FS_4G` 和 `GYRO_FS_500DPS`。这两个量程对应的灵敏度分别是 `8192 LSB/g` 和 `65.5 LSB/(deg/s)`，缩放因子正是围绕这两个值构造的。

问题五：为什么不直接使用 `MPU6050_Get_Physical_Data()`。

当前主循环没有调用这个函数，而是直接在 `Core/Src/main.c` 中用缩放宏写入 `sensors.accel500Hz` 和 `sensors.gyro500Hz`。教材分析当前主线时以实际调用路径为准。

问题六：温度公式为什么有两个偏移常数。

`mpu6050Calibration()` 中使用 `+35.0f`，`MPU6050_ComputeTemError()` 中使用 `+36.53f`。二者不一致，需要结合 MPU6050 手册或实测数据确认，当前标记为【待验证】。

问题七：静态零偏估计一定能得到正确零偏吗。

不能保证。
源码能证明 `MPU6050_ComputeStaticError()` 会采样并求均值，
但零偏质量依赖传感器静止、姿态稳定、温度条件和读取数据有效。
这些条件属于仓库外实测证据，当前教材只能提示读者记录并验证。
如果设备在采样期间移动，均值会把运动量也吸收到偏置估计中。

## 13. 本章总结

本章完成了从方向处理后的原始整数值到传感器物理输入的分析。

当前项目中：

- `mpu6050Calibration()` 提供冷机/热机两点温度标定，但在 `Core/Src/main.c` 中被注释掉。
- `init_eepromConfig(true)` 默认把温度补偿斜率和截距设为 0。
- `MPU6050_Init()` 会调用 `MPU6050_ComputeStaticError()`，启动阶段采样 5000 次估计静态偏置。
- `MPU6050_ComputeStaticError()` 会计算 `accelRTBias` 和 `gyroRTBias`，但当前主循环只扣除 `gyroRTBias`。
- 当前加速度缩放公式是 `(rawAccel - accelTCBias) * ACCEL_SCALE_FACTOR`。
- 当前陀螺仪缩放公式是 `(rawGyro - gyroRTBias - gyroTCBias) * GYRO_SCALE_FACTOR`。
- `MPU6050_Get_Physical_Data()` 存在，但不是当前 500Hz 主循环的实际缩放入口。

目标回扣：

- 本章已经区分温度补偿标定、静态零偏估计和物理量缩放三类问题。
- 本章已经说明哪些函数存在、哪些路径当前启用，以及 500Hz 主循环中的实际扣偏和缩放顺序。
- 本章只闭合传感器输入整理目标，不证明物理量精度已经通过仓库外实测。

本章保留三个待验证边界：

- 温度标定函数中的等待时间注释和实际 `HAL_Delay(10000)` 不一致。
- 温度换算中的 `+35.0f` 与 `+36.53f` 不一致。
- `accelOneG` 当前由累加量计算，后续需要结合姿态解算章节验证其影响。

下一章将进入 `滤波入口与重力校正`，在本章已经得到的 `sensors.accel500Hz` 和 `sensors.gyro500Hz` 基础上，分析项目如何进一步处理加速度和姿态修正。
