# 第22章 滤波入口与重力校正

## 1. 本章目标

第21章已经把方向处理后的 `rawAccel/rawGyro` 转换为 `sensors.accel500Hz` 和 `sensors.gyro500Hz`。进入姿态解算之前，项目还出现了两类容易混在一起的处理：一类是 `一阶低通滤波`，另一类是 `加速度计重力校正`。

本章目标是：

- 理解一阶低通滤波的基本目的：抑制快速变化和测量噪声。
- 定位 `firstOrderFilter.c` 中滤波器参数初始化和滤波公式。
- 明确当前主线只调用 `initFirstOrderFilter()`，没有在 500Hz 加速度处理路径中调用 `firstOrderFilter()`。
- 理解 `加速度计重力校正` 的最小原理：用加速度测得的重力方向修正陀螺仪积分漂移。
- 定位 `MargAHRSupdate()` 中加速度归一化、重力方向估计和 `KpAcc` 比例修正。
- 区分 MPU6050 内部 DLPF、项目软件一阶低通和 AHRS 重力反馈三条不同路径。

## 3. 问题背景

传感器进入姿态解算前有两个现实问题。

第一，原始传感器数据即使已经缩放，也会有噪声。如果直接把每一帧的快速抖动交给控制算法，后续姿态和 PID 都可能出现不必要的高频变化。一阶低通滤波用于让信号变化更平滑。

第二，陀螺仪适合短时间积分，但有漂移。加速度计能观测重力方向，但容易受线性加速度影响。云台静止或缓慢运动时，加速度向量主要反映重力方向，因此可以用它对陀螺仪积分方向做修正。

当前项目的实际情况比较微妙：

- `firstOrderFilter.c` 中确实实现了一阶低通滤波器。
- `Core/Src/main.c` 也在启动阶段调用了 `initFirstOrderFilter()`。
- 但当前 500Hz 主循环没有调用 `firstOrderFilter()` 处理 `sensors.accel500Hz`。
- `Core/Src/main.c` 调用 `MPU6050_Init(..., DLPF_CFG_260HZ)`，硬件 DLPF 配置属于 MPU6050 初始化路径。
- 当前真正进入主线的是 `MargAHRSupdate()` 里的加速度重力校正。

因此教材必须避免两种误写：不能把“滤波器模块存在”写成“当前加速度数据已经经过该软件滤波器”；也不能把“软件滤波未调用”写成“传感器链路完全没有任何滤波配置”。

还要注意源码注释和真实调用之间的边界。`Core/Src/main.c` 在调用
`MPU6050_Read_And_Process()` 前有“初步处理（如滤波）”的注释，但注释本身不是调用证据。
当前应以函数调用关系为准：这一行调用的是 MPU6050 读取与方向矩阵处理路径，不是
`firstOrderFilter()`。

## 4. 核心概念

### 4.1 一阶低通滤波

一阶低通滤波器的作用是保留较慢变化，削弱较快变化。它常用于传感器数据平滑、指令平滑和微分项噪声抑制。

项目中的滤波器状态结构是：

```c
typedef struct firstOrderFilterData
{
    float gx1;
    float gx2;
    float gx3;
    float previousInput;
    float previousOutput;
} firstOrderFilterData_t;
```

其中：

- `gx1/gx2/gx3` 是滤波系数。
- `previousInput` 是上一帧输入。
- `previousOutput` 是上一帧输出。

### 4.2 时间常数tau

`Drivers/SRC/Src/config.c` 中设置了多个滤波时间常数，例如：

```c
eepromConfig.accelX500HzLowPassTau = 0.1f;
eepromConfig.accelY500HzLowPassTau = 0.1f;
eepromConfig.accelZ500HzLowPassTau = 0.1f;
```

`tau` 越大，滤波越平滑，但响应越慢；`tau` 越小，响应越快，但噪声抑制更弱。

连续时间的一阶低通滤波器常写成：

```text
H(s) = 1 / (tau * s + 1)
```

其中 `s` 是拉普拉斯变量，`tau` 是时间常数。它表达的是：输入中的低频成分更容易通过，
高频成分会被削弱。

项目中的 `firstOrderFilter.c` 不是直接运行连续时间公式，而是把它离散化为可在 500Hz
采样循环中执行的差分方程。当前系数形式对应 Tustin，也叫双线性变换的一阶低通写法：

```text
a = 2 * tau * fs

y[n] = 1 / (1 + a) * x[n]
     + 1 / (1 + a) * x[n-1]
     - (1 - a) / (1 + a) * y[n-1]
```

和项目变量对应关系为：

```text
gx1 = 1 / (1 + a)
gx2 = 1 / (1 + a)
gx3 = (1 - a) / (1 + a)
```

所以 `firstOrderFilter()` 中的 `- gx3 * previousOutput` 不是额外多出来的一项，
而是差分方程反馈项的符号约定。

### 4.3 加速度计重力校正

加速度计在只受重力影响时，测得的是重力方向。姿态算法内部也能根据当前姿态状态估计一个重力方向。

如果这两个方向不一致，说明仅靠陀螺仪积分得到的姿态方向正在偏离重力观测。项目用叉乘形式计算误差：

```text
exAcc, eyAcc, ezAcc
```

然后把误差乘以 `eepromConfig.KpAcc`，加回陀螺仪角速度：

```c
gx += exAcc * eepromConfig.KpAcc;
gy += eyAcc * eepromConfig.KpAcc;
gz += ezAcc * eepromConfig.KpAcc;
```

这一步不是低通滤波，而是姿态估计中的反馈修正。

### 4.4 三种容易混淆的处理

本章标题里有“滤波入口”，但当前项目里至少有三种容易混用的概念：

| 名称 | 发生位置 | 当前主线证据 | 作用边界 |
| --- | --- | --- | --- |
| MPU6050 DLPF | 传感器内部 `REG_CONFIG` | `MPU6050_Init(..., DLPF_CFG_260HZ)` | 影响芯片输出的数据路径 |
| 软件一阶低通 | `firstOrderFilter()` | 仅初始化，当前未处理 `sensors.accel500Hz` | 平滑软件变量 |
| 加速度重力校正 | `MargAHRSupdate()` | 500Hz 主循环实际调用 | 用方向误差修正陀螺仪输入 |

这三者不能互相替代。硬件 DLPF 是否达到预期带宽，需要寄存器写入成功、手册参数和运行证据；软件一阶低通是否进入主线，需要函数调用证据；重力校正是否改善姿态，需要姿态日志或外部基准【待验证】。

## 5. 工作原理

当前项目中，第22章相关路径分为启动初始化和 500Hz 运行两部分。

启动阶段：

1. `Core/Src/main.c` 调用 `init_eepromConfig(true)`。
2. `Drivers/SRC/Src/config.c` 设置加速度滤波、指令滤波和姿态指令滤波的时间常数。
3. `Core/Src/main.c` 调用 `initFirstOrderFilter()`。
4. `initFirstOrderFilter()` 根据时间常数计算 `firstOrderFilters[]` 中多个滤波器的系数和初值。
5. `Core/Src/main.c` 调用 `MPU6050_Init(ACCEL_FS_4G, GYRO_FS_500DPS, DLPF_CFG_260HZ)`。
6. `MPU6050_Init()` 把第三个实参写入 `REG_CONFIG`，这是 MPU6050 内部 DLPF 配置路径。

500Hz 运行阶段：

1. `Core/Src/main.c` 调用 `MPU6050_Read_And_Process()`。
2. `Core/Src/main.c` 把原始整数缩放为 `sensors.accel500Hz` 和 `sensors.gyro500Hz`。
3. 当前代码没有调用 `firstOrderFilter()` 处理这些加速度值。
4. `Core/Src/main.c` 调用 `MargAHRSupdate()`。
5. `MargAHRSupdate()` 对输入加速度向量求模并归一化。
6. `MargAHRSupdate()` 根据当前姿态状态计算估计重力方向 `vx/vy/vz`。
7. `MargAHRSupdate()` 用加速度观测方向与估计方向计算误差 `exAcc/eyAcc/ezAcc`。
8. 误差乘以 `KpAcc` 后修正 `gx/gy/gz`。

本章只讲第5至第8步的重力校正，不展开后续四元数积分和欧拉角输出。

## 6. STM32实现机制

本章处理完全发生在 CPU 软件层，没有新增 STM32 外设。

严格说，软件一阶低通和 AHRS 重力校正在 STM32 CPU 上执行；MPU6050 DLPF 则发生在传感器芯片内部。STM32 只通过 I2C 在初始化阶段写入 `REG_CONFIG`，并不能在后续 500Hz 循环里证明 DLPF 的实时效果。

一阶低通滤波器使用普通浮点运算。STM32F103RCTx 是 Cortex-M3 内核，没有硬件 FPU，因此这些浮点运算由编译器和运行库生成软件浮点代码。当前项目仍采用这种实现，说明滤波和 AHRS 代码是软件算法层，不依赖特殊外设。

一阶滤波器的系数初始化发生在系统启动阶段，调用顺序位于 `Core/Src/main.c`：

```c
init_eepromConfig(true);
PWM_Motor_Init();
initFirstOrderFilter();
initPID();
```

加速度重力校正发生在 500Hz 主循环内。`MargAHRSupdate()` 的输入 `dt500Hz` 来自第21章之后的实时循环时间计算，它决定后续姿态更新步长。本章不展开实时循环调度，后面第26章会专门讲。

从性能角度看，如果 `firstOrderFilter()` 被调用，一次调用主要包含 3 次浮点乘法、2 次浮点加减
和 2 次状态写回，计算量较小。真正需要重点关注的是 `MargAHRSupdate()`，因为它包含 `sqrt`、
倒数、归一化、叉乘、四元数更新和四元数归一化等软件浮点计算。仓库中 `Core/Src/main.c`
已经维护 `executionTime500Hz`，因此是否满足 500Hz 实时预算，应以 DWT 或运行日志记录为准；
没有实测数据时只能写成【待验证】。

源码里还有一个性能审查细节：加速度模长使用 `sqrt(...)`，四元数模长使用 `sqrtf(...)`。
在 C 语言中 `sqrt` 是 double 版本，可能引入 float 到 double 再回到 float 的转换成本；
但实际代价取决于编译选项和链接的数学库，需要通过反汇编、map 文件或 DWT 实测确认【待验证】。

## 8. 代码分析

### 8.1 firstOrderFilters数组

`Drivers/SRC/Src/firstOrderFilter.c` 定义：

```c
firstOrderFilterData_t firstOrderFilters[NUMBER_OF_FIRST_ORDER_FILTERS];
```

`Drivers/SRC/Inc/firstOrderFilter.h` 中 `NUMBER_OF_FIRST_ORDER_FILTERS` 为 9，并定义了多个索引：

```c
ACCEL_X_500HZ_LOWPASS
ACCEL_Y_500HZ_LOWPASS
ACCEL_Z_500HZ_LOWPASS
PITCH_RATE_POINTING_50HZ_LOWPASS
PITCH_ATT_POINTING_50HZ_LOWPASS
```

部分 Roll/Yaw 指令滤波初始化代码在当前文件中被注释掉。教材不能把所有 9 个滤波器都说成当前已经完整初始化并进入主线。

### 8.2 initFirstOrderFilter的系数计算

以 X 轴加速度滤波器为例：

```c
a = 2.0f * eepromConfig.accelX500HzLowPassTau * 500.0f;
gx1 = 1.0f / (1.0f + a);
gx2 = 1.0f / (1.0f + a);
gx3 = (1.0f - a) / (1.0f + a);
```

这里的 `500.0f` 对应 500Hz 采样链路。`accelX500HzLowPassTau` 来自 `config.c` 默认配置。

代入当前默认值 `tau = 0.1s` 和 `fs = 500Hz`：

```text
a = 2 * 0.1 * 500 = 100

gx1 = 1 / (1 + 100) = 1 / 101 ~= 0.00990099
gx2 = 1 / (1 + 100) = 1 / 101 ~= 0.00990099
gx3 = (1 - 100) / (1 + 100) = -99 / 101 ~= -0.980198
```

因为运行公式是：

```c
output = gx1 * input + gx2 * previousInput - gx3 * previousOutput;
```

所以在这个默认配置下，上一帧输出的实际系数是 `-gx3 ~= +0.980198`。这意味着该滤波器
一旦被调用，会强烈依赖历史输出，信号会比较平滑，但阶跃响应也会更慢。这里仍然只是
滤波器系数初始化结论，不能推出当前 500Hz 加速度主线已经使用该输出。

Z 轴滤波器的初值是：

```c
previousInput  = -9.8065f;
previousOutput = -9.8065f;
```

这个初值假设 Z 轴初始接近一个重力加速度方向。但当前第20章和第21章已经指出，项目的实际安装方向、Z 轴正负约定和缩放链路存在需要结合仓库外实测继续验证的边界。因此这里不能单独证明硬件静止时 Z 轴一定为 `-9.8065f`，只能说明滤波器初值这样设置。

### 8.3 firstOrderFilter公式

`firstOrderFilter()` 的输出公式是：

```c
output = gx1 * input + gx2 * previousInput - gx3 * previousOutput;
```

随后更新历史状态：

```c
previousInput = input;
previousOutput = output;
```

这是一种离散一阶滤波实现。它依赖上一帧输入和上一帧输出，所以调试时不能只看当前输入；必须同时观察 `previousInput/previousOutput`。

如果用第8.2节的默认 X 轴系数做一次手算，假设：

```text
input = 9.8065
previousInput = 0
previousOutput = 0
```

则第一帧输出为：

```text
output = 0.00990099 * 9.8065
       + 0.00990099 * 0
       - (-0.980198) * 0
       ~= 0.09709
```

这说明一阶滤波器若从零状态接收一个接近 1g 的阶跃输入，第一帧不会立刻跳到
`9.8065`。它会逐步靠近输入值。调试时要把这个现象和“传感器本身读数错误”区分开。
但再次强调：当前加速度主线没有调用 `firstOrderFilter()`，所以这个手算用于理解函数，
不能作为当前 AHRS 输入已经变平滑的运行证据。

### 8.4 当前主线未调用firstOrderFilter

全工程搜索当前调用点，能看到：

- `Core/Src/main.c` 调用 `initFirstOrderFilter()`。
- `Drivers/SRC/Src/firstOrderFilter.c` 定义 `firstOrderFilter()`。

但当前 500Hz 主循环没有调用：

```c
firstOrderFilter(...)
```

因此，本章对 `一阶低通滤波` 的结论必须谨慎：

- 项目存在通用一阶低通滤波器。
- 项目启动时初始化了滤波器系数。
- 当前主线的加速度输入没有经过 `firstOrderFilter()`。
- 第28章会再讲 PID D 项内部的独立一阶低通形式，那不是这里这个 `firstOrderFilter()` 函数的直接调用。

### 8.4.1 MPU6050硬件DLPF边界

第18章已经分析过 MPU6050 初始化。这里需要把它和软件一阶低通再分开一次。

`Core/Src/main.c` 当前调用：

```c
MPU6050_Init(ACCEL_FS_4G, GYRO_FS_500DPS, DLPF_CFG_260HZ);
```

`Drivers/CustomDrivers/Src/mpu6050.c` 中把第三个实参写入 `REG_CONFIG`：

```c
status = MPU_Write_Reg(REG_CONFIG, dlpf);
```

所以当前项目可以证明“初始化路径尝试配置 MPU6050 内部 DLPF”。但这不是 `firstOrderFilter()`，也不是 `MargAHRSupdate()` 的加速度重力校正。

还要注意，`Drivers/SRC/Src/config.c` 中存在：

```c
eepromConfig.dlpfSetting = DLPF_CFG_184HZ;
```

但当前 `main.c` 没有把 `eepromConfig.dlpfSetting` 作为 `MPU6050_Init()` 的第三个实参，而是硬编码传入 `DLPF_CFG_260HZ`。
因此，不能把配置对象里的 `DLPF_CFG_184HZ` 写成当前真实硬件 DLPF 设置。

本章只记录这条边界：软件一阶低通未进入 500Hz 加速度主线，不等于 MPU6050 初始化没有 DLPF 配置；硬件 DLPF 配置存在，也不等于软件滤波器输出已经进入 AHRS。

### 8.5 MargAHRSupdate的加速度输入

`Core/Src/main.c` 调用：

```c
MargAHRSupdate(sensors.gyro500Hz[ROLL],
               sensors.gyro500Hz[PITCH],
               sensors.gyro500Hz[YAW],
               sensors.accel500Hz[XAXIS],
               sensors.accel500Hz[YAXIS],
               sensors.accel500Hz[ZAXIS],
               0, 0, 0, false, dt500Hz);
```

这说明当前 `MargAHRSupdate()` 使用的是第21章缩放后的加速度和陀螺仪。磁力计输入均为 0，`magDataUpdate` 为 `false`，所以本章不讲磁力计修正。

### 8.6 加速度归一化

`MargAHRSupdate()` 首先计算加速度模长：

```c
norm = sqrt(ax*ax + ay*ay + az*az);
```

若 `norm > 0.1f`，则执行：

```c
normR = 1.0f / norm;
ax *= normR;
ay *= normR;
az *= normR;
```

这一步把加速度向量变成单位方向向量。换句话说，重力校正关心的是方向，而不是加速度大小本身。

如果 `norm <= 0.1f`，当前函数会跳过加速度归一化和重力误差修正，但后续四元数仍会用陀螺仪
输入继续更新。结合第19章的读失败清零路径看，读失败或异常小的加速度向量可能让本帧退化为
“仅陀螺更新”。这能避免除以接近零的模长，但不能把该帧标记为有效重力观测。

### 8.7 估计重力方向与误差

`MargAHRSupdate()` 根据当前姿态状态计算估计重力方向：

```c
vx = 2.0f * (q1q3 - q0q2);
vy = 2.0f * (q0q1 + q2q3);
vz = q0q0 - q1q1 - q2q2 + q3q3;
```

随后计算误差：

```c
exAcc = vy * az - vz * ay;
eyAcc = vz * ax - vx * az;
ezAcc = vx * ay - vy * ax;
```

这组式子可以理解为“估计重力方向”和“测得加速度方向”之间的方向误差。完整姿态数学推导留到第23章，本章只需要知道：误差越大，说明当前姿态状态与加速度观测越不一致。

用向量写法可以更清楚：

```text
a_meas = [ax, ay, az] / ||[ax, ay, az]||
v_est  = [vx, vy, vz]
e      = v_est x a_meas
```

叉乘展开后就是：

```text
exAcc = vy * az - vz * ay
eyAcc = vz * ax - vx * az
ezAcc = vx * ay - vy * ax
```

如果 `v_est` 和 `a_meas` 方向相同，叉乘结果接近零，说明加速度观测不需要明显修正
当前姿态。如果二者存在夹角，叉乘方向给出修正轴，叉乘模长随夹角增大而增大。
这也是 Mahony 类非线性互补滤波中常见的重力参考方向误差构造方式。

这里还有一个工程边界：归一化后的加速度只保留方向，不保留模长异常信息。当云台快速平移、
强振动或受到冲击时，加速度方向可能不再主要代表重力方向。当前 `MargAHRSupdate()` 没有
调用 `calculateAccConfidence()` 动态降低 `KpAcc`，所以强线性加速度下的修正可靠性需要
结合实测日志标记为【待验证】。

### 8.8 KpAcc比例修正

误差被乘以 `eepromConfig.KpAcc` 后加到陀螺仪输入：

```c
gx += exAcc * eepromConfig.KpAcc;
gy += eyAcc * eepromConfig.KpAcc;
gz += ezAcc * eepromConfig.KpAcc;
```

`config.c` 中默认：

```c
eepromConfig.KpAcc = 0.98f;
eepromConfig.KiAcc = 0.02f;
```

但当前 `MargAHRSupdate()` 只使用 `KpAcc`。文件中虽然定义了 `exAccInt/eyAccInt/ezAccInt` 和 `kiAcc` 相关变量，但当前主线没有用 `KiAcc` 做加速度积分校正。不能把 `KiAcc` 写成当前已生效参数。

这个设计有一个可以教学化的权衡：

- 只使用 `KpAcc`，实现简单，反馈路径短，不会因为错误加速度观测持续积分而形成额外偏置。
- 不使用 `KiAcc`，意味着当前主线没有通过加速度误差在线估计并抵消陀螺零偏的积分环节。
- 如果未来启用 `KiAcc`，必须同时处理积分限幅、加速度可信度和运动状态判断，否则线性加速度
  可能被误当成重力误差并累积到陀螺修正中。

因此本章把当前方案称为“加速度比例重力校正”更准确；把它写成完整 PI 型 Mahony 修正会
超过当前仓库证据。

### 8.9 calculateAccConfidence当前未进入主线

`MargAHRS.c` 中存在：

```c
void calculateAccConfidence(float accMag)
```

它会使用 `accelOneG` 计算加速度置信度。但当前全工程没有发现 `calculateAccConfidence()` 的调用点。

第21章留下的 `accelOneG` 边界在这里得到进一步澄清：`accelOneG` 确实被 `MargAHRS.c` 中的保留函数使用，但该函数当前没有进入 `MargAHRSupdate()` 主线。

## 9. 调试方法

调试第22章内容时，建议按顺序进行。

第一步，确认滤波器初始化。

- 在 `Core/Src/main.c` 的 `initFirstOrderFilter()` 调用处设置断点。
- 单步进入 `Drivers/SRC/Src/firstOrderFilter.c`。
- 观察 `firstOrderFilters[ACCEL_X_500HZ_LOWPASS].gx1/gx2/gx3`。
- 对照 `eepromConfig.accelX500HzLowPassTau = 0.1f` 计算系数。

第二步，确认当前是否调用滤波函数。

- 在 `firstOrderFilter()` 函数入口设置断点。
- 运行主循环，如果断点没有命中，记录“当前 500Hz 主线未调用该函数”。
- 不要仅凭 `initFirstOrderFilter()` 已运行就判定加速度已滤波。

第三步，确认 MPU6050 DLPF 配置路径。

- 在 `Core/Src/main.c` 中确认 `MPU6050_Init()` 的第三个实参。
- 在 `MPU6050_Init()` 中确认 `REG_CONFIG` 写入使用的是 `dlpf` 实参。
- 对照 `config.c` 中的 `eepromConfig.dlpfSetting`，记录它当前没有接入该实参。
- 不能把硬件 DLPF 配置写成软件 `firstOrderFilter()` 已启用。

第四步，观察 `MargAHRSupdate()` 输入。

- 在 `Core/Src/main.c` 调用 `MargAHRSupdate()` 前设置断点。
- 记录 `sensors.accel500Hz` 和 `sensors.gyro500Hz`。
- 确认输入来自第21章缩放链路。

第五步，观察加速度归一化。

- 在 `MargAHRSupdate()` 中观察 `norm`。
- 如果 `norm <= 0.1f`，重力校正分支不会执行。
- 如果 `norm > 0.1f`，观察归一化后的 `ax/ay/az`。
- 若前序 I2C 读取失败并导致原始量被清零，应把本帧记录为“重力观测无效或待验证”，
  不能把姿态继续更新误读为加速度修正仍然有效。

第六步，观察重力误差和修正量。

- 观察 `vx/vy/vz`、`exAcc/eyAcc/ezAcc`。
- 观察 `gx/gy/gz` 在 `KpAcc` 修正前后的变化。
- 如果 `KpAcc` 改为 0，理论上这一步对陀螺仪输入不再产生加速度修正。

第七步，分离四类结论。

- “系数已经初始化”只需要 `initFirstOrderFilter()` 和 `gx1/gx2/gx3` 证据。
- “函数输出符合预期”需要单步进入 `firstOrderFilter()` 或写临时测试调用。
- “滤波结果进入主线”必须证明 `firstOrderFilter()` 的返回值被传入后续 AHRS 或控制链路。
- “硬件 DLPF 已配置”必须证明 `MPU6050_Init()` 实参和 `REG_CONFIG` 写入路径。

这四类结论不能互相替代。当前仓库可以稳定支持第一类和第四类结论；
第二类可以通过调试临时验证；第三类在当前 500Hz 加速度路径中没有调用证据。

调试记录建议：

- 记录滤波器初始化调用、配置 `tau`、系数 `gx1/gx2/gx3` 和当前主线调用搜索结果。
- 对重力校正，应记录 AHRS 输入、加速度模长、归一化结果、误差项和修正前后陀螺仪值。
- 若 `firstOrderFilter()` 未命中断点，应写成“初始化存在、主线调用未确认”，不能写成已滤波。
- 加速度修正效果、噪声改善和姿态稳定性需要运行日志或外部姿态基准，缺失时标记为【待验证】。
- 对硬件 DLPF，应记录 `DLPF_CFG_260HZ`、`REG_CONFIG` 写入和 `eepromConfig.dlpfSetting` 未接入的边界。

## 10. 常见问题

问题一：当前项目是否已经对加速度使用 `firstOrderFilter()`。

当前没有发现主循环调用 `firstOrderFilter()` 的证据。项目初始化了滤波器参数，但当前加速度主线直接把 `sensors.accel500Hz` 传给 `MargAHRSupdate()`。

问题二：`initFirstOrderFilter()` 有意义吗。

有意义，它说明项目保留了通用一阶滤波器配置和状态初始化。
后续如果恢复或新增调用，就可以直接使用这些滤波参数。
但当前教材必须区分“初始化存在”和“运行时使用”。
初始化滤波器不等于当前加速度或姿态指令已经经过该函数滤波。

问题二补充：源码注释写了“如滤波”，是否可以证明已经滤波。

不能。注释可以提示作者意图，但教材审查要以调用关系为准。当前 500Hz 主线调用
`MPU6050_Read_And_Process()` 后直接缩放并传入 `MargAHRSupdate()`，没有调用
`firstOrderFilter()`。

问题三：`KpAcc` 和 `KiAcc` 是否都生效。

当前 `MargAHRSupdate()` 只使用 `KpAcc` 修正 `gx/gy/gz`。`KiAcc` 在配置中存在，但当前加速度积分校正没有进入主线。

问题四：`calculateAccConfidence()` 是否影响当前姿态更新。

当前没有调用证据。它使用 `accelOneG` 计算置信度，但不在当前 `MargAHRSupdate()` 主线中。

问题五：加速度归一化是不是物理量缩放。

不是。物理量缩放已经在第21章完成。这里的归一化是把加速度向量变为单位方向向量，用于比较重力方向。

问题六：本章是否已经讲完姿态解算。

没有。本章只讲加速度重力校正入口。四元数积分、归一化和欧拉角输出在第23章展开。

问题七：加速度重力校正是不是等同于一阶滤波。

不是。
一阶滤波是对信号做平滑，通常关注噪声和响应速度的折中。
本章的重力校正是把加速度方向与四元数估计的重力方向比较，
再用误差修正陀螺仪积分。
二者都能改善姿态估计，但作用对象和数学含义不同。

问题八：当前没有调用 `firstOrderFilter()`，是否表示项目完全没有滤波。

不能这样写。
当前 500Hz 加速度软件路径没有调用 `firstOrderFilter()`，这是软件滤波主线结论。
但 MPU6050 初始化时传入了 `DLPF_CFG_260HZ`，并在 `MPU6050_Init()` 中写入 `REG_CONFIG`。
所以更准确的说法是：软件一阶低通没有进入当前加速度主线；传感器内部 DLPF 配置属于另一路硬件初始化证据。

问题九：`eepromConfig.dlpfSetting = DLPF_CFG_184HZ` 是否是当前硬件 DLPF 设置。

当前不能这么说。
`main.c` 调用 `MPU6050_Init()` 时直接传入 `DLPF_CFG_260HZ`，没有把 `eepromConfig.dlpfSetting` 传进去。
因此当前真实初始化实参以 `main.c` 为准，配置对象字段只能说明存在一个尚未接入当前调用路径的配置值。

本章完成了第21章传感器输入到第23章姿态解算之间的过渡。

当前项目中：

- `firstOrderFilter.c` 实现了通用一阶低通滤波器。
- `Core/Src/main.c` 启动阶段调用 `initFirstOrderFilter()`。
- 当前 500Hz 主循环没有调用 `firstOrderFilter()` 处理加速度输入。
- `main.c` 中“如滤波”的注释不能替代 `firstOrderFilter()` 调用证据。
- MPU6050 初始化路径写入了 `REG_CONFIG`，当前 `main.c` 实参是 `DLPF_CFG_260HZ`。
- `eepromConfig.dlpfSetting = DLPF_CFG_184HZ` 当前没有接入 `MPU6050_Init()` 实参。
- 当前默认 `tau = 0.1s`、`fs = 500Hz` 时，一阶低通系数为 `gx1 = gx2 ~= 0.00990099`、
  `gx3 ~= -0.980198`，上一帧输出的实际系数约为 `+0.980198`。
- `MargAHRSupdate()` 是当前主线加速度重力校正的实际入口。
- `MargAHRSupdate()` 对加速度向量归一化，计算估计重力方向 `vx/vy/vz`，再计算 `exAcc/eyAcc/ezAcc` 修正陀螺仪输入。
- 当加速度模长 `norm <= 0.1f` 时，重力修正分支跳过，后续姿态更新退化为陀螺仪主导。
- `exAcc/eyAcc/ezAcc` 可以写成 `v_est x a_meas`，即估计重力方向与测得加速度方向的叉乘误差。
- 当前只使用 `KpAcc` 做比例修正，`KiAcc` 未进入加速度积分校正主线。
- `calculateAccConfidence()` 存在，但当前没有调用证据。

目标回扣：
