# 质量审查清单

输入来源：

- 教材设计/前三阶段基础包验收记录.md
- 教材编写/统一章节模板.md
- 教材编写/术语与命名规范.md

使用约束：

- 每轮新增或修改教材正文前，必须先执行本清单。
- 若发现高优先级问题，必须先修复，再继续扩写。
- 检查结果记录到 `教材编写/迭代记录.md`。

---

# P0 事实与项目理解

检查项：

- 是否存在与项目源码不一致的描述。
- 是否存在与 CubeMX 配置不一致的外设描述。
- 是否存在把未启用外设写成项目主线的情况。
- 是否存在把 HAL 库能力误写成项目业务使用的情况。

通过标准：

- 所有结论都能追溯到项目文件、函数、变量或配置项。

---

# P1 依赖与知识断层

检查项：

- 是否出现未登记知识点。
- 是否出现未建立依赖关系的知识点。
- 是否出现未映射章节的知识点。
- 是否跳过前置知识直接解释高级内容。
- 是否违反 `知识点/教学顺序.md`。

通过标准：

- 每个正式知识点都能按“知识点总表 -> 知识依赖图 -> 学习优先级 -> 教学顺序 -> 教材章节”追溯。

---

# P2 逻辑与章节结构

检查项：

- 章节是否只解决一个核心问题。
- 章节之间是否有自然过渡。
- 理论主线和项目主线是否同时存在。
- 是否有重复讲解或章节粒度失衡。

通过标准：

- 本章目标、前置知识、项目应用、调试方法和总结相互闭合。

---

# P3 项目证据与工程背景

检查项：

- 是否明确项目位置。
- 是否列出相关模块、相关代码和运行流程。
- 是否标明调试观察点。
- 证据不足处是否标记【待验证】。

通过标准：

- 读者能从章节内容回到具体项目文件。

---

# P4 原理、推导与实现机制

检查项：

- 是否解释问题背景。
- 是否解释核心概念。
- 是否解释工作原理。
- 是否解释 STM32 实现机制。
- 数学或控制内容是否有必要推导。

通过标准：

- 章节不是只给结论、只讲 API 或只贴项目行为。

---

# P5 调试、验证与实践

检查项：

- 是否包含调试方法。
- 是否包含常见问题。
- 是否包含实践任务。
- 是否包含思考题。
- 实践任务是否只使用已学前置知识。

通过标准：

- 读者知道看什么、怎么判断、出错时从哪里定位。

---

# P6 表达与统一性

检查项：

- 是否使用统一术语。
- 文件名、函数名、变量名是否与项目一致。
- 是否使用统一章节结构。
- 是否出现泛泛表达或空话。

通过标准：

- 同一概念在全书只使用一个正式名称。

---

# P7 新增内容扩展

检查项：

- 新增内容是否已经通过 P0 至 P6 检查。
- 新增内容是否存在项目证据。
- 新增内容是否需要先补知识库。

通过标准：

- 只有高优先级问题清零后，才允许新增扩展内容。

---

# 每轮审查输出格式

每轮必须记录：

- 【本轮审查范围】
- 【发现的问题】
- 【问题优先级】
- 【修复内容】
- 【新增内容】
- 【更新的知识点映射】
- 【更新的章节映射】
- 【仍待解决的问题】
- 【下一轮优化目标】
