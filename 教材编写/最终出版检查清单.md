# 最终出版检查清单

本清单用于 Book Refinement 阶段收尾前的最终审查。

使用原则：

- 不以新增章节、扩展知识点或继续加长正文为默认目标。
- 只有发现明确的重复讲解、术语漂移、证据边界混淆、学习路径断点或格式不一致时，才继续修改正文。
- 若检查项只能带来很小的措辞收益，且不会影响学习、查阅或维护，应保留现状。

---

## 1. 结构完整性

检查项：

- `教材正文/` 是否仍保持第01章至第34章的既有目录结构。
- 每章开头和结尾是否保留上一章、下一章导航。
- 每章是否包含 `本章速查：`、`【必须掌握】`、`【工程深化】`、`【拓展阅读】` 和 `【证据与验证】`。
- 调试章节或调试小节是否按“现象 -> 可能原因 -> 定位方法 -> 验证步骤 -> 解决方案 -> 经验总结”组织。

通过标准：

- 没有缺章、漏导航或阅读层次缺失。
- 不再因为单个长句存在就默认重构。

---

## 2. 唯一知识来源

检查项：

- 是否存在同一知识点在多个章节完整重复讲解。
- 外设基础、时钟、PSC、HAL/CMSIS、PID、滤波、姿态解算等核心知识点是否有明确主讲章节。
- 其它章节是否使用简要回顾和跨章节引用，而不是重新完整展开。

通过标准：

- 每个知识点只有一个完整讲解位置。
- 重复内容只保留为必要回顾、证据引用或项目上下文。

---

## 3. 证据边界

检查项：

- 源码、`.ioc`、`.map`、`.list`、`.su`、`.cyclo`、HAL/CMSIS 和官方资料是否分层表述。
- `.map/.list` 是否只写成构建或静态证据，不直接写成运行成功。
- `.su/.cyclo` 是否只写成静态栈估计和圈复杂度，不换算成真实最大栈水位或真实耗时。
- 硬件实测、USB 主机枚举、EEPROM 持久化、电机相序、温升和长期稳定性是否继续保留 `【待验证】`。

通过标准：

- 构建证据、运行证据和硬件证据不混写。
- 缺少现场记录时，不把软件路径写成硬件已经验证。

---

## 4. 术语与格式

检查项：

- 文件名、函数名、变量名、寄存器名和英文缩写是否与项目源码一致。
- `Debug/Three-axis_cloud_platformV2.map`、`.list`、`.su`、`.cyclo` 等构建产物引用是否使用统一写法。
- 表格、代码块、实践任务、思考题和章节固定块是否保持一致风格。
- Markdown 文件是否保持 UTF-8 无 BOM 和 CRLF。

通过标准：

- 同一概念在全书只使用一个正式名称。
- 没有格式漂移、编码漂移或章节固定块缺失。

---

## 5. 学习路径

检查项：

- STM32 初学者是否可以只读【必须掌握】形成主线理解。
- 嵌入式开发者是否可以通过【必须掌握】和【工程深化】定位工程实现。
- 算法读者是否能快速进入姿态解算、滤波、PID 和控制章节。
- 维护人员是否能通过调试、证据与验证、构建产物边界和硬件诊断章节定位问题。

通过标准：

- 教材既能顺序学习，也能作为工程参考书查阅。
- 不同读者不会被证据审计内容打断主线阅读。

---

## 6. 收尾判断

继续修改的条件：

- 发现事实错误、证据边界错误或明显知识缺失。
- 发现同一知识点重复完整讲解。
- 发现章节结构、术语或导航不一致。
- 发现读者必须自行拆解的高密度段落，且重构能明显降低认知负担。

停止逐章微改的条件：

- 剩余问题主要是“句子较长但结构清楚”。
- 修改只能带来轻微措辞收益。
- 继续编辑会增加篇幅、引入重复，或削弱现有证据边界。

最终结论：

- 若本清单通过，应停止 Book Refinement 的逐章正文微改，转入最终校对、发布准备或硬件实测补证。
